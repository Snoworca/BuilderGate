# 실시간 프로토콜과 상태 흐름

## 인증 흐름

### 1. 비밀번호 로그인

`POST /api/auth/login`

가능한 결과:

- 비밀번호 오류면 `401`
- TOTP가 꺼져 있으면 즉시 JWT 발급
- TOTP가 켜져 있으면 `202`와 `tempToken`, `nextStage: "totp"`
- `localhostPasswordOnly` 또는 `twoFactor.externalOnly` 우회 조건이면 localhost에서는 TOTP 생략 가능

### 2. TOTP 검증

`POST /api/auth/verify`

- `tempToken`과 `otpCode`를 검증
- 성공 시 JWT 발급
- 실패 시 시도 횟수 차감
- 현재 프런트 UI는 6자리 코드를 전제로 동작

### 3. 세션 유지

- 프런트 `useHeartbeat()`가 기본 15분마다 refresh 시도
- 실패가 누적되면 토큰을 지우고 로그아웃 처리

## REST와 WebSocket의 책임 분리

| 채널 | 책임 |
|---|---|
| REST | 로그인, 설정 저장, 워크스페이스/탭 CRUD, 파일 API, telemetry 조회 |
| WebSocket | 터미널 입출력, snapshot, status, cwd, 워크스페이스 브로드캐스트 |

즉 구조 변경은 대체로 REST가 시작하고, 실시간 반영은 WS가 뿌립니다.

## WebSocket 연결 흐름

1. 프런트가 `/ws?token=<jwt>`로 연결
2. 서버가 업그레이드 시 토큰 검증
3. 성공하면 `connected { clientId }` 전송
4. 프런트는 이 `clientId`를 REST 요청의 `x-client-id` 헤더에 붙임
5. 서버는 해당 client가 만든 workspace/tab/grid 이벤트를 자기 자신에게 다시 브로드캐스트하지 않음

## 세션 구독과 snapshot handshake

현재 구조에서 가장 중요한 실시간 계약은 snapshot ack 흐름입니다.

### 1. 구독

클라이언트가:

`{ type: "subscribe", sessionIds: [...] }`

를 보내면 서버는:

- `subscribed`
- 필요하면 `screen-snapshot`

를 전송합니다.

### 2. `screen-snapshot`

현재 snapshot 메시지는 다음 개념을 포함합니다.

- `sessionId`
- `replayToken`
- `seq`
- `cols`, `rows`
- `mode`
- `data`
- `truncated`
- `source`
- `windowsPty`

`mode`는 현재 `authoritative` 또는 `fallback`입니다.

### 3. 클라이언트 적용 후 ack

프런트 `TerminalContainer`는 snapshot을 실제 xterm에 적용한 뒤:

`{ type: "screen-snapshot:ready", sessionId, replayToken }`

를 보냅니다.

### 4. ack 전 output queueing

서버는 ack 전까지 live output을 per-client/per-session queue에 적재합니다. ack가 오면 queue를 flush합니다.

### 5. resize 중 snapshot refresh

resize 후 replay pending 상태라면 서버는 quiet window와 headless drain을 본 뒤 새 snapshot을 다시 보냅니다.

이 흐름이 현재 workspace bounce, reconnect, tab/grid 전환 안정성의 핵심입니다.

## 터미널 I/O 흐름

### 입력

`TerminalView` -> `TerminalContainer` -> `WebSocketContext.send()`

메시지:

- `input`
- `resize`

서버는 이를 `SessionManager.writeInput()` / `SessionManager.resize()`로 전달합니다.

### 출력

PTY 출력은 현재 경로에서:

1. `SessionManager`
2. headless terminal write
3. `WsRouter.routeSessionOutput()`
4. 클라이언트 `onOutput`

순으로 흐릅니다.

### 상태

현재 서버가 보내는 세션 상태는 실질적으로:

- `running`
- `idle`
- `session:exited`

입니다.

프런트는 `session:exited`를 받으면 런타임 상태를 `disconnected`로 승격합니다.

## 워크스페이스 상태 흐름

### 서버 원본

`WorkspaceService.state`

- `workspaces`
- `tabs`
- `gridLayouts`

### 프런트 메모리 상태

`useWorkspaceManager`

- `workspaces`
- `tabs`
- `gridLayouts`
- `activeWorkspaceId`

### 브로드캐스트 이벤트

현재 정의된 주요 이벤트:

- `workspace:created`
- `workspace:updated`
- `workspace:deleted`
- `workspace:reordered`
- `tab:added`
- `tab:updated`
- `tab:removed`
- `tab:reordered`
- `grid:updated`

## 현재 쓰이지 않거나 부분적으로만 쓰이는 이벤트

- `workspace:deleting`: 서버 emit는 있으나 현재 프런트 handler 없음
- `session:error`: 타입만 있고 현재 서버 emit 없음
- `tab:disconnected`: 타입과 프런트 handler는 있으나 현재 서버 emit 없음
- `grid:updated`: 프런트 state는 갱신하지만 메인 그리드 UI는 localStorage 기반이어서 실사용 효과가 제한적

## 상태 저장 위치 구분

| 위치 | 저장 내용 |
|---|---|
| `server/data/workspaces.json` | 워크스페이스, 탭, `lastCwd`, 서버 쪽 `gridLayouts` |
| `server/data/totp.secret` | 암호화된 TOTP secret |
| 브라우저 `localStorage` | JWT, active workspace, mosaic layout, grid ratios, terminal snapshots, font size |
| 브라우저 메모리 | WS `clientId`, session handler registry, workspace handler registry |
