# 프런트엔드 구조

## 현재 앱 조합

루트 렌더 트리는 다음 순서입니다.

1. `StrictMode`
2. `AuthProvider`
3. `App`
4. `App` 내부 `AuthGuard`
5. 인증 통과 후 `WebSocketProvider`
6. 실제 앱 셸 `AppContent`

중요한 사실:

- React Router는 없습니다
- 화면 전환은 `App.tsx`의 `screen: 'workspace' | 'settings'` 상태로 처리합니다
- 앱의 중심은 "워크스페이스 + 터미널 runtime 유지"입니다

## 주요 상태 관리자

### `AuthContext`

- 로그인
- 2FA 검증
- 로그아웃
- 토큰 refresh
- `localStorage` 토큰 복원
- `auth-expired` 이벤트 처리

### `WebSocketContext`

- 단일 WebSocket 연결 유지
- reconnect backoff
- 세션별 handler 등록
- 워크스페이스 이벤트 handler 등록
- `connected.clientId`를 받아 API 레이어의 `x-client-id`와 연결
- 구독 해제 grace window 동안 메시지 buffer 유지

### `useWorkspaceManager`

현재 앱의 실질적인 중심 상태입니다.

주요 책임:

- `GET /api/workspaces` 초기 로드
- `active_workspace_id` 복원
- 워크스페이스/탭 CRUD
- WS 이벤트로 상태 동기화
- 탭의 runtime status/cwd 반영
- old session snapshot removal marker 처리

주의할 점:

- 서버에서 받은 `gridLayouts`를 들고는 있지만, 메인 그리드 UI는 이 값을 적극 사용하지 않습니다

### `useResponsive`

- `768px` 미만을 모바일로 취급
- 모바일에서는 사이드바를 드로어로 표시
- 모바일에서 그리드 모드는 자동으로 탭 모드로 되돌립니다

### `useMosaicLayout`, `useLayoutMode`, `useFocusHistory`

- 워크스페이스별 그리드 트리 복원
- equal / focus / auto 레이아웃 모드 관리
- focus 대상 추적
- `localStorage` 저장

## 현재 프런트 구조의 핵심: terminal runtime 분리

최근 구조에서 가장 중요한 변화는 "터미널 runtime 소유권"과 "화면에 보이는 위치"를 분리한 것입니다.

### `TerminalRuntimeProvider`

- runtime host registry 보유
- 각 탭이 현재 어느 사각형에 붙어야 하는지 추적
- layout invalidation 지원

### `TerminalHostSlot`

- tab 모드나 grid 모드가 원하는 위치에 host slot을 배치
- DOM rect를 측정해 runtime 레이어에 전달
- 이 컴포넌트 자체가 xterm 인스턴스를 만들지는 않음

### `TerminalRuntimeLayer`

- 실제 `TerminalContainer`들을 한곳에서 렌더
- host가 없거나 숨겨졌을 때는 offscreen parking
- `disconnected` 탭은 runtime entry를 만들지 않음

### `TerminalContainer`

- 세션 단위 WS subscribe 수명주기
- `screen-snapshot` 수신, 중복/역행 snapshot 무시
- snapshot 적용 후 `screen-snapshot:ready` ack 전송
- live output, status, cwd 전달
- imperative handle로 `focus`, `fit`, `sendInput`, `replaceWithSnapshot` 노출

### `TerminalView`

- xterm.js 인스턴스 실제 소유
- 폰트 크기와 local snapshot 관리
- `ResizeObserver`와 fit
- 브라우저 복사/붙여넣기 처리
- helper textarea focus 관리

결론적으로 현재는 tab/grid 전환이 같은 세션의 xterm runtime을 재사용하는 구조입니다.

## 현재 마운트된 주요 컴포넌트

| 영역 | 주요 컴포넌트 |
|---|---|
| 인증 | `AuthGuard`, `LoginForm`, `TwoFactorForm` |
| 헤더 | `Header` |
| 워크스페이스 | `WorkspaceSidebar`, `WorkspaceItem`, `MobileDrawer` |
| 탭 모드 | `WorkspaceTabBar`, `TerminalHostSlot`, `MetadataRow` |
| 그리드 모드 | `MosaicContainer`, `MosaicTile`, `MosaicToolbar` |
| 설정 | `SettingsPage` |
| 보조 UI | `ContextMenu`, `ConfirmModal`, `DisconnectedOverlay` |

## 탭 모드 구조

- 현재 active workspace의 active tab만 화면에 보입니다
- 하지만 실제 runtime은 `wm.tabs` 전체를 기준으로 살아 있을 수 있습니다
- `MAX_ALIVE_WORKSPACES = 0`이라 LRU eviction은 현재 비활성
- 즉 워크스페이스 전환 시 runtime을 적극적으로 재사용합니다

## 그리드 모드 구조

`MosaicContainer`는 `react-mosaic-component` 위에 현재 제품 규칙을 덧씌운 래퍼입니다.

주요 동작:

- 이진 트리 leaf는 tab id
- 탭 수가 바뀌면 equal 모드로 재구성
- stale tab id 감지 시 즉시 재빌드
- `equal`, `focus`, `auto` 모드 지원
- focus/auto 비율은 `localStorage` 기반
- splitter drag 이후 layout refresh 강제

현재 그리드 관련 중요한 사실:

- 현재 UI는 server `gridLayouts`보다 local `mosaic_layout_<workspaceId>`를 주로 사용
- `workspaceApi.updateGrid()`는 훅에 있지만 메인 경로에서 적극 사용되지 않습니다

## 현재 보이는 기능과 실제 컨텍스트 메뉴

터미널 컨텍스트 메뉴 빌더는 현재 아래 동작만 만듭니다.

- 새 세션
- 세션 닫기
- 복사
- 붙여넣기

즉 오래된 pane split 메뉴를 전제로 코드를 읽으면 현재 UI를 잘못 이해하게 됩니다.

## 설정 화면의 역할

`SettingsPage`는 두 종류의 값을 다룹니다.

### 서버 런타임 값

- 인증 세션 시간
- 비밀번호 교체
- TOTP on/off와 issuer/accountName
- CORS
- 터미널 기본값
- idle delay
- 파일 제한

### 브라우저 로컬 값

- `grid_auto_focus_ratio`
- `grid_focus_ratio`

즉 설정 화면 하나 안에 "서버에 저장되는 값"과 "브라우저 localStorage만 바뀌는 값"이 함께 있습니다.
