# Technical Decisions

## ADR-2601: Option A를 `patch-package` 기반 vendor patch로 시작한다

- 상태: Accepted
- 이유:
  - 현재 저장소에 vendor patch 인프라가 없다.
  - 1차 구현은 fork보다 `patch-package`가 시작 비용이 낮다.
  - `frontend/package.json`에 `patch-package` 의존성과 `postinstall` wiring을 함께 넣어 clean install 재현성을 확보해야 한다.
  - 구체 경로:
    - devDependency: `patch-package`
    - script: `"postinstall": "patch-package"`
    - patch file: `frontend/patches/react-mosaic-component+6.1.1.patch`

## ADR-2602: v1 reorder semantics는 slot index move로 정의한다

- 상태: Accepted
- 이유:
  - 사용자 예시 `1 -> 5 => [2,3,4,5,1]`를 단순하게 만족시킬 수 있다.
  - before/after split semantics를 추가하면 명세가 불필요하게 복잡해진다.

## ADR-2603: primary pointer drag만 reorder를 시작한다

- 상태: Accepted
- 이유:
  - 우클릭은 기존 컨텍스트 메뉴와 충돌한다.
  - desktop에서는 좌클릭 drag로 시작하고 좌클릭 release를 drop으로 본다.
  - 현재 앱의 상호작용 모델과 가장 자연스럽다.

## ADR-2604: patch는 runtime 산출물까지 반영해야 한다

- 상태: Accepted
- 이유:
  - 현재 앱은 vendor package의 `lib/`와 CSS 산출물을 실제 runtime에서 사용한다.
  - `src/`만 수정하면 실행 결과가 바뀌지 않을 수 있다.
  - 따라서 patch 작성 기준은 `lib/*.js`, `lib/index.d.ts`, 루트 CSS이고 `src` 수정은 참고용으로만 둔다.
  - 이 저장소에서는 vendor package build를 다시 돌려 `src -> lib` 재생성하지 않는 것을 기본 원칙으로 한다.

## ADR-2605: library는 reorder contract, app은 mode semantics를 담당한다

- 상태: Accepted
- 이유:
  - `equal/focus/auto/none` 의미는 현재 앱 계층에 이미 존재한다.
  - vendor patch에 mode 규칙까지 밀어 넣으면 결합도가 과해진다.
  - 단, vendor는 full mode 대신 app이 주입하는 `reorderEnabled` boolean만 받아 reorder-vs-split 분기를 수행한다.

## ADR-2606: v1에서는 root/head/tail sentinel을 도입하지 않는다

- 상태: Accepted
- 이유:
  - slot index semantic만으로 첫 칸/마지막 칸 이동을 표현할 수 있다.
  - sentinel은 UX/명세 복잡도를 올린다.

## ADR-2607: 검증은 app E2E 중심으로 가져간다

- 상태: Accepted
- 이유:
  - 현재 저장소는 Playwright E2E 경로가 이미 있다.
  - vendor package 자체 unit test 체인을 이 repo에서 직접 돌리는 비용이 크다.

## ADR-2608: none 모드는 topology와 split ratio를 최대 보존한다

- 상태: Accepted
- 이유:
  - 사용자 요구는 기존 mode semantics 보존을 전제로 한다.
  - 따라서 `none`은 balanced/equal rebuild가 아니라, manual layout을 최대 유지하는 reorder 정책이 필요하다.

## ADR-2609: root drop path도 reorder contract에 포함한다

- 상태: Accepted
- 이유:
  - vendor library는 `Mosaic.tsx`와 `RootDropTargets.tsx`를 통해 root-level drop infrastructure를 가진다.
  - v1에서는 root drop을 reorder 삽입으로 해석하지 않고, reorder mode에서 비활성화/no-op 처리하는 정책을 명시해야 한다.

## ADR-2610: invalid drop은 explicit restore path를 사용한다

- 상태: Accepted
- 이유:
  - 현재 vendor의 invalid drop branch는 hide 복구를 split reset 경로에 의존한다.
  - Option A에서 단순 no-op로 바꾸면 dragged window가 숨겨진 채 남을 수 있다.
  - reorder branch는 pre-drag snapshot 또는 동등한 restore token으로 원래 tree를 복원해야 한다.

## ADR-2611: vendor patch는 neutral reorder tree만 생성한다

- 상태: Accepted
- 이유:
  - vendor layer에는 현재 app `layoutMode` 정보가 없다.
  - mode별 canonicalization 여부를 vendor로 밀어 넣으면 app state가 vendor API로 새게 된다.
  - 따라서 vendor는 shape-preserving neutral reorder tree만 만들고, `equal/focus/auto/none` 후처리는 app layer가 담당한다.
