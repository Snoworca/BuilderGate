# Phase 2: Vendor Reorder DnD Patch

## Objective

`react-mosaic-component` 내부에서 full-cell target과 reorder payload/update 경로를 구현한다.

## Target Files

- `frontend/node_modules/react-mosaic-component/src/Mosaic.tsx`
- `frontend/node_modules/react-mosaic-component/src/RootDropTargets.tsx`
- `frontend/node_modules/react-mosaic-component/src/internalTypes.ts`
- `frontend/node_modules/react-mosaic-component/src/MosaicDropTarget.tsx`
- `frontend/node_modules/react-mosaic-component/src/MosaicWindow.tsx`
- `frontend/node_modules/react-mosaic-component/src/util/mosaicUpdates.ts`
- `frontend/node_modules/react-mosaic-component/lib/*.js`
- `frontend/node_modules/react-mosaic-component/lib/**/*.d.ts`
- `frontend/node_modules/react-mosaic-component/react-mosaic-component.css`
- 필요 시 대응 `lib/` 파일

## Planned Changes

1. `Mosaic.tsx` 레벨에 `reorderEnabled` signal plumbing 추가
2. reorder drop payload 타입 추가
3. window-level full-cell drop target 렌더링
4. root-level drop path를 reorder mode에서 비활성화/no-op 처리
5. split update 대신 neutral reorder update 분기 추가
6. full-cell blue guide CSS 반영
7. desktop 좌클릭 drag/release 경로 유지
8. runtime entry(`lib`/`.d.ts`/CSS)까지 patch 반영
9. invalid drop/reset branch를 explicit restore path로 치환
10. `renderToolbar`의 기본 custom-toolbar drag wrapper 억제
11. touch path 전역 비활성화

## Implementation Checklist

- [ ] `reorderEnabled` prop/context plumbing 추가
- [ ] reorder payload 타입 추가
- [ ] full-cell target branch 추가
- [ ] `Mosaic.tsx`/`RootDropTargets.tsx` root drop path 비활성화 또는 no-op 처리
- [ ] `MosaicWindow.end()` reorder branch 추가
- [ ] `createReorderedTreePreservingShape()` 또는 동등 helper 추가
- [ ] vendor patch는 neutral shape-preserving reorder 결과만 생성
- [ ] pre-drag snapshot 또는 동등한 restore token 확보
- [ ] invalid drop/reset branch가 기존 split reset 대신 explicit restore path를 사용
- [ ] split guide CSS 제거/비활성화
- [ ] runtime에 실제 반영되는 `lib/**/*.d.ts`/JS/CSS patch 포함
- [ ] desktop 좌클릭 drag/release 계약 유지
- [ ] `renderToolbar(...)` 반환값이 reorder mode에서 기본 `connectDragSource`로 감싸지지 않음
- [ ] grip element만 drag source로 연결됨
- [ ] touch path는 reorder payload를 절대 만들지 않음

## Test

### Scenario 1: full-cell guide is shown

Given a grid cell drag in reorder mode  
When the pointer hovers a target cell  
Then the full target cell is highlighted with the blue guide

### Scenario 2: drop emits reorder semantics

Given source and target cells  
When the source is dropped on the target cell  
Then the vendor update path performs move/insert reorder rather than split insertion

### Scenario 2-1: invalid drop path restores the pre-drag tree

Given a drag that ends without a valid target cell  
When `MosaicWindow.end()` handles the invalid drop  
Then the previous split reset path is replaced by an explicit restore path that leaves the pre-drag tree intact

### Scenario 3: root drop path remains consistent

Given the reorder patch is active  
When drag state reaches root-level drop infrastructure  
Then root-level drop rendering is disabled or no-op and does not fall back to split-only behavior

### Scenario 4: toolbar surface outside the grip is not draggable

Given a custom toolbar render path  
When the user drags from the toolbar surface but not the grip  
Then reorder does not start

### Scenario 5: touch path is globally blocked

Given the vendor library still runs under `MultiBackend`/`HTML5toTouch`  
When the client uses touch input  
Then the reorder branch never emits payloads
