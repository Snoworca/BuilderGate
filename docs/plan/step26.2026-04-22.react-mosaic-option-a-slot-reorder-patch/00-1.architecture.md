# Architecture

## 핵심 문제

현재 `react-mosaic-component`는 split insertion 전용 DnD 모델이다.

- target은 `top/bottom/left/right` 4방향 drop zone
- payload는 `{path, position}`
- update는 `createDragToUpdates()`를 통한 split parent 생성

사용자 요구는 이 모델과 다르다.

- target은 셀 전체
- payload는 slot reorder 의미
- update는 split 생성이 아니라 `move/insert`
- 시작점은 그립(handle) 한정
- 적용 범위는 `equal` 모드 한정

즉, Option A는 visual patch가 아니라 **library internal DnD contract patch**다.

## 제안 구조

### 1. Vendor patch layer

`react-mosaic-component` 내부에 `equal` 모드 전용 reorder branch를 추가한다.

- split DnD branch
- reorder DnD branch

grid mode에서도 `equal`일 때만 reorder branch를 사용한다.

핵심 patch 대상:

- `Mosaic.tsx`
- `RootDropTargets.tsx`
- `internalTypes.ts`
- `MosaicDropTarget.tsx`
- `MosaicWindow.tsx`
- `util/mosaicUpdates.ts`
- `react-mosaic-component.css` 또는 대응 source style

vendor는 full `layoutMode`를 알 필요는 없지만, 최소한 다음 신호는 받아야 한다.

- `reorderEnabled: boolean`

이 신호는 app layer가 `layoutMode === 'equal'`일 때만 `true`로 주입한다.

### 2. Reorder payload

기존 `{path, position}` 대신 grid reorder에 필요한 payload를 만든다.

예:

```ts
type MosaicReorderDropData = {
  kind: 'reorder';
  path: MosaicPath;
  slotIndex: number;
};
```

v1에서는 before/after가 아니라 **slot index semantic**을 채택한다.

즉, “셀 전체에 drop” = “그 셀의 현재 slot index로 이동”.

일반 규칙:

- `target slot index`는 source 제거 전 visible leaf order에서 계산한다.
- source 제거 후, 제거 전 계산한 `target slot index`에 source를 삽입한다.
- source가 target보다 앞이면 결과적으로 source는 target 뒤쪽 slot로 이동한다.
- source가 target보다 뒤이면 결과적으로 source는 target 앞쪽 slot로 이동한다.

### 3. Tree update

library 내부 update 경로를 split insertion과 reorder insertion으로 분기한다.

- split: 기존 `createDragToUpdates()`
- reorder: 새 `createReorderedTree()` 또는 `createReorderUpdates()`

reorder 핵심:

1. visible leaf order 추출
2. source 제거
3. target slot index에 삽입
4. topology-preserving neutral reorder tree 생성

### 4. App responsibility

library가 neutral reorder 결과 tree를 만들고,
앱은 `equal` 전용 gate와 mode semantics를 후처리한다.

- `equal`: neutral reorder tree를 받은 뒤 canonical equal grid 재적용
- `none/focus/auto`: reorder branch 진입 자체를 막고 기존 동작 유지

### 5. Persistence boundary

저장/복원은 여전히 app layer에서 한다.

- vendor patch는 tree reorder semantics까지만 책임
- `useMosaicLayout`는 reordered tree와 mode를 저장/복원
- `equal` 모드에서는 persisted reordered leaf order가 canonical order source여야 한다.
- 탭 추가/삭제 시에도 `tabs.map(...)` 순서가 persisted equal reorder 결과를 덮어써서는 안 된다.

### 6. Input Policy

reorder 시작 입력은 desktop mouse 기준으로 고정한다.

- desktop start: 그립(handle)에서 좌클릭 drag
- desktop drop: 좌클릭 release
- right click: 컨텍스트 메뉴 유지

즉 Option A v1은 handle-based desktop reorder contract를 다룬다.

### 6-1. Enforcement Gate

이 제약은 문장 선언만으로 끝내지 않는다.

- app layer에서 `layoutMode === 'equal'`일 때만 reorder branch를 활성화해야 한다.
- app layer에서 handle에만 drag source를 연결해야 한다.
- app layer는 vendor에 `reorderEnabled` 신호를 명시적으로 주입해야 한다.
- vendor patch는 `reorderEnabled === true` 이면서 desktop mouse left-button drag/release일 때만 reorder payload를 생성해야 한다.
- touch/backend 경로와 non-equal mode에서는 reorder payload가 생성되지 않아야 한다.
- vendor 기본 custom-toolbar wrapper가 toolbar surface 전체를 drag source로 감싸지 않도록 억제해야 한다.
- 즉, app은 `equal`이 아닐 때 reorder drag source 자체를 연결하지 않아야 한다.
- 즉, app은 handle-only gate를 CSS가 아니라 실제 drag source wiring 수준에서 보장해야 한다.

### 7. Shape-preserving vs canonical rebuild

Option A는 tree rebuild 정책을 `equal` branch 안에서만 사용한다.

- vendor patch: 항상 topology-preserving neutral reorder tree만 생성
- `equal`: app layer에서만 canonical grid 재구성 허용
- `none/focus/auto`: reorder branch 비진입

즉 vendor layer는 mode를 몰라도 되고, app layer가 mode semantics를 결정한다.

### 8. Root drop path policy

v1에서는 root-level drop을 reorder insertion으로 해석하지 않는다.

- `RootDropTargets`는 reorder mode에서 비활성화하거나 no-op로 만든다.
- 목적은 root path에서 기존 split semantics가 새 reorder semantics를 오염시키지 않게 하는 것이다.
- 셀 전체 target reorder는 window-level drop target만 담당한다.

### 9. No-op boundary rules

- source를 자기 자신과 동일한 slot에 drop하면 reorder를 적용하지 않는다.
- pointer가 어떤 target cell에도 매칭되지 않는 상태로 release되면 reorder를 적용하지 않는다.
- 즉 commit 조건은 “유효한 target cell slot이 확정된 경우”로 한정한다.
- 현재 vendor 기본 reset 경로(`dropResult` 부재 시 split reset)는 Option A에서 그대로 쓰면 안 된다.
- 따라서 reorder branch는 drag 시작 시점의 pre-drag tree snapshot 또는 동등한 restore token을 보관해야 한다.
- `MosaicWindow.end()`의 invalid drop/reset branch는 plain no-op가 아니라, source hide를 해제하고 pre-drag tree를 복원하는 explicit restore path여야 한다.

## 권장 구조

- 단기: `patch-package`로 vendor patch 적용
- library는 reorder payload + reorder tree 생성 책임
- app은 mode별 후처리 + persistence 책임

이 분리가 가장 단순하고, 나중에 vendor fork로 승격하기도 쉽다.
