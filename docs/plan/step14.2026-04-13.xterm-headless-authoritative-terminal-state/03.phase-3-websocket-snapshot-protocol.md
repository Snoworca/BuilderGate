## 상세 구현 지침

### 권장 메시지 형태

```ts
type ClientWsMessage =
  | { type: 'subscribe'; sessionIds: string[] }
  | { type: 'unsubscribe'; sessionIds: string[] }
  | { type: 'screen:ready'; sessionId: string; snapshotId: string }
  | { type: 'input'; sessionId: string; data: string }
  | { type: 'resize'; sessionId: string; cols: number; rows: number }
  | { type: 'ping' };

type ServerWsMessage =
  | { type: 'screen:snapshot'; sessionId: string; snapshotId: string; generation: number; data: string; cols: number; rows: number; truncated: boolean; altScreenActive: boolean; replayMode: 'headless' | 'legacy' }
  | { type: 'output'; sessionId: string; data: string }
  | { type: 'subscribed'; sessions: Array<{ sessionId: string; status: string; cwd?: string; replayMode: 'headless' | 'legacy'; snapshotPending: boolean }> };
```

### ordering 규칙

1. subscribe 처리 중 live output 는 바로 전송하지 않고 queue 로 적재
2. snapshot 이 있는 세션은 `screen:snapshot` 을 먼저 보낸다
3. 그 직후 `subscribed(snapshotPending=true)` 를 보낸다
4. `screen:ready(snapshotId)` 를 기다린다
5. 올바른 ack 를 받으면 queued output flush 후 live mode 로 전환한다
6. snapshot 이 없는 세션은 `subscribed(snapshotPending=false)` 만 보낸다
7. timeout 이 나면 replay state 를 정리하되 socket 는 유지한다

### duplicate subscribe 규칙

- 같은 socket 가 이미 같은 session live 상태면:
  - 새 snapshot 금지
  - `subscribed` 만 갱신
- 같은 socket 이 `snapshotPending` 상태에서 중복 subscribe 하면:
  - 기존 pending state 유지
  - 새 `snapshotId` 발급 금지
- 다른 socket 이 같은 session 을 subscribe 하면:
  - 별도 pending state 생성

## 테스트 섹션

### 시나리오 1: snapshot 후 live output flush
- Given: snapshot 전송 직후 live output 가 들어온다
- When: 클라이언트가 정상 ack 한다
- Then: output 는 snapshot 적용 뒤에만 도착한다

### 시나리오 2: stale ack 무시
- Given: 이전 barrier 의 `snapshotId` 로 `screen:ready` 가 온다
- When: router 가 ack 를 처리한다
- Then: queued output 는 flush 되지 않는다

### 시나리오 3: duplicate subscribe idempotent
- Given: 같은 socket-session pair 가 이미 구독 중이다
- When: 다시 subscribe 한다
- Then: snapshot 은 한 번만 전송된다

### 시나리오 4: reconnect resync
- Given: socket 가 끊기고 새 socket 가 연결된다
- When: active session 을 다시 subscribe 한다
- Then: 새 barrier 와 새 snapshot 이 생성된다

### 시나리오 5: timeout cleanup
- Given: 클라이언트가 ack 를 보내지 않는다
- When: timeout 이 지난다
- Then: replay pending state 가 정리되고 누수되지 않는다
