## 상세 구현 지침

### PoC 범위

- live 서비스 경로에 바로 꽂지 않는다.
- `server/src/services/terminal/` 아래 spike 또는 isolated harness 로 시작한다.
- 입력은 recorded ANSI fixture 와 실 PTY 둘 다 사용한다.

### 직렬화 계약

기본 후보 payload:

```ts
type ScreenSnapshotMessage = {
  type: 'screen:snapshot';
  sessionId: string;
  snapshotId: string;
  generation: number;
  format: 'ansi';
  data: string;
  cols: number;
  rows: number;
  truncated: boolean;
  altScreenActive: boolean;
  cursorVisible: boolean;
};
```

고정할 항목:

- `data` 는 클라이언트가 `term.write()` 로 바로 적용 가능한 형식이어야 한다.
- `generation` 은 세션 restart/reset 시만 증가한다.
- `snapshotId` 는 subscribe barrier 단위다.
- `truncated` 는 scrollback 상한 초과를 의미한다.

### 실패 시 fallback 규칙

- headless init 실패 시 세션 전체를 죽이지 않는다.
- 세션은 `legacy replay` 모드로 강등하고 warning metric 을 남긴다.
- fallback 여부는 `subscribed` metadata 또는 별도 debug log 로 노출한다.

## 테스트 섹션

### 시나리오 1: 기본 shell 출력 snapshot
- Given: PowerShell 또는 Bash 세션이 일반 프롬프트와 명령 출력 몇 줄을 생성했다
- When: headless snapshot 을 생성한다
- Then: snapshot 이 비어 있지 않고 새 renderer 에 재적용 가능하다

### 시나리오 2: resize 이후 snapshot
- Given: 긴 줄이 wrap 되는 출력과 초기 cols/rows 가 있다
- When: PTY 와 headless 양쪽에 resize 를 적용한 뒤 snapshot 을 만든다
- Then: snapshot 결과가 새 크기 기준으로 일관된다

### 시나리오 3: alt-screen/TUI
- Given: `vim`, `less`, `codex` 같은 alternate-screen 패턴이 있다
- When: 화면 전환 중 snapshot 을 생성한다
- Then: alt-screen 여부와 복구 가능성을 명시적으로 판단할 수 있다

### 시나리오 4: headless init 실패
- Given: headless 인스턴스 생성 또는 serializer 가 실패한다
- When: 세션을 계속 유지한다
- Then: 세션은 fallback 모드로 동작하고 process 는 살아 있어야 한다

### 시나리오 5: 경계값 - 긴 scrollback
- Given: `pty.maxBufferSize` 근처의 긴 출력이 있다
- When: snapshot 을 반복 생성한다
- Then: 직렬화 시간과 payload 크기가 측정되고 상한을 넘으면 truncate 정책이 발동한다
