## 제안 상태 머신

```text
idle
  -> subscribing
  -> applying_snapshot
  -> awaiting_live
  -> live
  -> resyncing
  -> error
```

### 전이 규칙

- mount: `idle -> subscribing`
- `screen:snapshot` 수신: `subscribing -> applying_snapshot`
- snapshot write 완료 + ack 송신: `applying_snapshot -> awaiting_live`
- 첫 live output 또는 subscribed complete: `awaiting_live -> live`
- WS reconnect: `live -> resyncing -> subscribing`
- session error/disconnect: `* -> error`

### fallback 정책

- 기본값: `browserSnapshotFallback = false`
- true 일 때만 아래 조건에서 보조 복원 허용:
  - 서버가 `replayMode = legacy` 인 세션
  - 서버 snapshot 이 비어 있고 session 이 idle 인 경우
- 서버 snapshot 이 존재하면 local snapshot 을 절대 적용하지 않는다

## 상세 구현 지침

### `TerminalView` 역할 축소

- 유지:
  - xterm 인스턴스 생성
  - `write()`, `clear()`, `fit()`, `focus()`
  - `applySnapshot(data)` / `replaceWithSnapshot(data)`
- 제거 또는 축소:
  - `localStorage` primary save/restore
  - `beforeunload/pagehide` snapshot persist
  - `restorePendingRef` 를 browser snapshot 기준으로 잡는 로직

### `TerminalContainer` 책임 강화

- subscribe 와 ack orchestration
- pending output buffer 관리
- reconnect 감지 시 resubscribe
- current snapshot barrier 와 active sessionId 추적
- `screen:snapshot` 와 `subscribed` 의 도착 순서가 바뀌어도 deadlock 없이 처리

## 테스트 섹션

### 시나리오 1: 최초 mount snapshot 적용
- Given: session subscribe 직후 `screen:snapshot` 이 온다
- When: 컨테이너가 snapshot 을 적용한다
- Then: 터미널은 clear 후 snapshot 만 보이고 이후 live output 으로 전환된다

### 시나리오 2: reconnect 후 resync
- Given: WS 재연결이 발생했다
- When: session 이 다시 subscribe 된다
- Then: 기존 browser snapshot 없이 서버 snapshot 으로 복구된다

### 시나리오 3: snapshot 없는 live session
- Given: 서버가 빈 snapshot 또는 live-only subscribed 를 보낸다
- When: 컨테이너가 상태를 전이한다
- Then: deadlock 없이 live 상태로 들어간다

### 시나리오 4: fallback disabled
- Given: local snapshot 이 남아 있다
- When: 서버 snapshot 이 도착한다
- Then: local snapshot 은 무시된다

### 시나리오 5: snapshot 적용 중 live output 도착
- Given: snapshot write 도중 `output` 이벤트가 온다
- When: 컨테이너가 buffer 를 쌓는다
- Then: ack 이후 순서대로 flush 된다
