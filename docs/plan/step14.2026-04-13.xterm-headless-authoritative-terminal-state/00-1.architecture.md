# ProjectMaster `@xterm/headless` 전환 아키텍처

## 1. 아키텍처 목표

- 브라우저 `localStorage` snapshot 이 아니라 서버가 기억하는 terminal state 를 권위 데이터로 만든다.
- `node-pty` 는 I/O 운반 계층으로 남기고, 화면 해석과 snapshot 생성은 headless emulator 계층으로 분리한다.
- subscribe, reconnect, grid remount, workspace switch 를 모두 같은 `snapshot gate -> live stream` 규약으로 처리한다.
- 기존 `history` tail replay 는 최종적으로 retire 하되, migration 기간에는 feature flag 뒤에 남긴다.

## 2. 시스템 컨텍스트

```text
Shell Process
  -> node-pty
    -> SessionManager
      -> HeadlessTerminalState
        -> SnapshotSerializer
          -> WsRouter snapshot gate
            -> WebSocketContext
              -> TerminalContainer state machine
                -> TerminalView renderer
```

### 현재 대비 변경점

- `SessionManager` 내부 세션 데이터에 `HeadlessTerminalState` 가 추가된다.
- `WsRouter`는 `history/history:ready` 대신 `screen:snapshot/screen:ready` 기반 barrier 를 사용한다.
- `TerminalView`는 화면 복원 책임을 잃고, `clear -> apply snapshot -> live output render`만 담당한다.
- `useWorkspaceManager`, `App.tsx`, `MosaicContainer`는 mount/unmount 를 허용하되 snapshot handoff 로 일관성을 유지한다.

| 컴포넌트 | 책임 | 현재 파일 | 목표 변경 |

## 4. 제안 데이터 모델

### 4.1 세션 런타임

```ts
interface SessionData {
  session: Session;
  pty: IPty;
  emulator: HeadlessTerminalState;
  snapshotGeneration: number;
  legacyReplayBuffer?: string;
  replayMode: 'headless' | 'legacy';
}
```

### 4.2 HeadlessTerminalState

```ts
interface HeadlessTerminalState {
  write(data: string): void;
  resize(cols: number, rows: number): void;
  createSnapshot(): TerminalScreenSnapshot;
  getMetrics(): TerminalStateMetrics;
  reset(): void;
  dispose(): void;
}

interface TerminalScreenSnapshot {
  snapshotId: string;
  generation: number;
  format: 'ansi';
  data: string;
  cols: number;
  rows: number;
  truncated: boolean;
  altScreenActive: boolean;
  cursorVisible: boolean;
}
```

### 4.3 소켓별 replay gate

```ts
interface ReplayPendingState {
  snapshotId: string;
  generation: number;
  queuedOutput: string;
  ackDeadline: NodeJS.Timeout;
}
```

필수 규칙:

- `snapshotId` 는 subscribe 마다 새로 발급한다.
- `generation` 은 세션 headless state reset/restart 때만 증가시킨다.
- `queuedOutput` 는 `screen:ready` 이전 live output 임시 적재소다.
- `snapshotId` 가 맞지 않는 `screen:ready` 는 무시한다.

## 6. 주요 시퀀스

### 6.1 최초 mount / refresh / reconnect

1. `TerminalContainer` 가 `subscribe(sessionId)` 호출
2. `WsRouter` 가 세션 구독을 등록하고 새 `snapshotId` 를 발급
3. `SessionManager` 가 `HeadlessTerminalState.createSnapshot()` 호출
4. 서버가 `screen:snapshot` 전송
5. 클라이언트가 `TerminalView.clearAndApplySnapshot()` 수행
6. 클라이언트가 `screen:ready(sessionId, snapshotId)` 전송
7. 서버가 queued live output flush 후 `output` live 모드로 전환

### 6.2 duplicate subscribe / StrictMode remount

1. 같은 socket-session pair 가 이미 live 상태면 snapshot 재전송 금지
2. 단, explicit unsubscribe 이후 재subscribe 는 새 `snapshotId` 로 처리
3. reconnect 로 socket 가 바뀌면 새 barrier 를 생성

### 6.3 grid remount / workspace switch

- tab 모드: 기존 DOM 유지, 재subscribe 없음
- grid 모드 remount: 새 `TerminalContainer` 인스턴스가 subscribe 하며 snapshot 재적용
- workspace switch: active workspace 변경은 visibility 변화가 중심이고 session state 는 유지

## 7. Phase-아키텍처 매핑

| Phase | 구현 컴포넌트 | 의존 컴포넌트 | 검증 초점 |
|-------|-------------|-------------|----------|
| 1 | Headless spike, serializer facade | 없음 | snapshot 품질, alt-screen, resize |
| 2 | SessionManager + HeadlessTerminalState | PTY Engine | lifecycle, metrics, config limits |
| 3 | WsRouter + ws-protocol | SessionManager | ordering, ack token, idempotency |
| 4 | WebSocketContext + TerminalContainer + TerminalView | WsRouter | mount/reconnect/fallback state machine |
| 5 | App + MosaicContainer + workspace flows | Terminal components | tab/grid/workspace behavior |
| 6 | Tests + logging + rollout flags | 전체 | 회귀, 관측성, rollback |

## 8. 아키텍처 위험과 제약

| 항목 | 위험 | 대응 |
|------|------|------|
| `@xterm/headless` experimental | serialize 경로가 예상과 다를 수 있음 | Phase 1 에서 단일 세션 PoC 와 fallback serializer 검증 |
| 메모리 증가 | 세션당 scrollback/state 모델 유지 비용 증가 | scrollback 상한 설정, metrics 노출, eviction 정책 검증 |
| alt-screen/TUI | snapshot 직렬화가 메인 화면과 달라질 수 있음 | TUI 시나리오를 P0 검증 항목으로 유지 |
| subscribe race | live output 가 snapshot 적용 중 섞일 수 있음 | `snapshotId` barrier 와 socket별 queue 사용 |
| browser snapshot 잔재 | poisoned snapshot 혼입 가능 | Phase 4 에서 기본 경로 제거, Phase 6 에서 완전 retire |

- [x] local snapshot 은 보조 또는 제거 대상으로 격하되어 있다.
- [x] 설정 변경, 메모리 제한, alt-screen/TUI 가 별도 리스크로 기록되어 있다.
