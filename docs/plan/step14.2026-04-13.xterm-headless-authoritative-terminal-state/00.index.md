## 조사 입력 요약

| 조사 역할 | 담당 | 초점 |
|----------|------|------|
| 조사 축 A | Schrodinger | 프런트엔드/프로토콜 통합, state machine, snapshot 우선순위 |
| 조사 축 B | Hume | 서버 세션/emulator lifecycle, subscribe barrier, 설정 경계 |
| 조사 축 C | Nietzsche | 회귀 테스트, 롤아웃, failure mode 와 cleanup 기준 |

### 조사 축 A: 프런트엔드/프로토콜 통합

- 현재 `TerminalContainer`는 `history` 수신 시 `replaceWithHistory()`, `subscribed` 수신 시 local snapshot 복원을 수행한다.
- 현재 `TerminalView`는 `@xterm/addon-serialize` + `localStorage` 기반 snapshot을 저장/복원하며, 이것이 refresh/reconnect 시 주 복구 경로다.
- `App.tsx` 기준 tab 모드는 모든 터미널을 숨김만 하고 유지하지만, grid 모드는 `MosaicContainer` 경유로 타일 단위 remount가 발생할 수 있다.
- 따라서 서버 authored state 도입 시 `초기 mount`, `WS 재연결`, `grid remount`, `workspace 전환`을 하나의 state machine으로 묶어야 한다.

### 조사 축 B: 서버 세션/에뮬레이터 통합

- 현재 `SessionManager`는 `node-pty` 출력을 `replayBuffer` 문자열에 tail 형태로 보관한다.
- 현재 `WsRouter`는 `history -> history:ready -> queued output flush` 순서만 보장하고, 화면 상태 자체는 보장하지 못한다.
- 권위 상태 도입 지점은 `SessionManager.createSession()`의 PTY lifecycle 과 `WsRouter.handleSubscribe()`의 subscribe barrier 이다.
- `pty.maxBufferSize`는 이미 런타임 변경 가능하므로, headless scrollback 상한과 즉시 재절단 정책도 같은 설정 경계에 맞춰야 한다.

### 조사 축 C: 테스트/롤아웃/운영

- 기존 server test-runner 는 `history` ordering 과 duplicate subscribe 를 검증한다.
- 기존 frontend E2E 는 refresh 후 snapshot 복원과 poisoned snapshot 우선순위만 검증한다.
- headless 전환 후에는 `alt-screen`, `resize`, `reconnect 중 output interleave`, `grid remount`, `duplicate subscribe`, `fallback off`를 새 회귀 기준으로 삼아야 한다.
