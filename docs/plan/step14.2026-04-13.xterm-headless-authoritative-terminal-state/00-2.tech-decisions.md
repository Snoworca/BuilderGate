# 기술 결정 기록

## ADR-001: 서버 권위 터미널 상태 엔진으로 `@xterm/headless` 채택

**상태**: Accepted  
**날짜**: 2026-04-13

### 컨텍스트

- 현재 프런트는 `@xterm/xterm@^6.0.0` 와 `@xterm/addon-serialize@^0.14.0` 를 사용한다.
- 현재 서버는 `node-pty@^1.0.0` 만 사용하며 화면 상태를 모르고 문자열 replay tail 만 저장한다.
- 목표는 refresh/reconnect/grid remount 에서 브라우저 snapshot 없이 현재 화면을 복원하는 것이다.

### 결정

- 1차 구현은 `node-pty + @xterm/headless` 조합으로 간다.
- `libvterm` 직접 도입은 1차 범위에서 제외한다.
- `@xterm/headless` 직렬화 경로는 Phase 1 PoC 로 검증하고, 실패 시 serializer facade 를 둔다.

### 선택지 비교

| 선택지 | 장점 | 단점 | 점수 |
|--------|------|------|------|
| `libvterm` 직접 도입 | 강력한 에뮬레이터, 언어 독립적 | Node/TS/Windows 통합 비용 큼 | 6/10 |
| 브라우저 snapshot 유지 + 서버 replay 개선 | 빠른 수정 | 권위 상태 부재, TUI/alt-screen 복구 불완전 | 4/10 |
| `@xterm/headless` | xterm 계열 일치, 현재 스택과 자연스러운 통합 | experimental, PoC 필요 | 9/10 |

### 결과

- parser mismatch 위험을 최소화한다.
- 단, Phase 1 통과 전까지는 구조 고정을 금지한다.

## ADR-002: 프로토콜은 `history` 가 아니라 `screen:snapshot` barrier 로 전환

**상태**: Accepted  
**날짜**: 2026-04-13

### 결정

- 새 메시지 계약:
  - Server -> Client: `screen:snapshot`
  - Client -> Server: `screen:ready`
  - Server -> Client: `output`
- `screen:snapshot` 은 `snapshotId`, `generation`, `format`, `cols`, `rows`, `altScreenActive` 를 포함한다.
- `screen:ready` 는 반드시 `snapshotId` 를 포함한다.
- 기존 `history/history:ready` 는 migration flag 뒤로 격리 후 제거한다.

### 이유

- `history` 는 "무슨 문자열을 tail 로 보냈는가"만 의미한다.
- `screen:snapshot` 은 "현재 권위 화면 상태"라는 의미를 명확히 준다.
- `snapshotId` 가 있어야 duplicate ack 와 stale ack 를 구분할 수 있다.

## ADR-003: 브라우저 local snapshot 은 기본 경로에서 제거하고 단계적으로 retire

**상태**: Accepted  
**날짜**: 2026-04-13

### 결정

- 기본 동작에서는 `TerminalView` 가 local snapshot 을 저장/복원하지 않는다.
- migration 기간에만 `terminal.browserSnapshotFallback` 플래그로 emergency fallback 을 둘 수 있다.
- fallback 이 켜져도 서버 snapshot 이 있으면 항상 서버 snapshot 이 우선이다.
- rollout 안정화 후 snapshot 관련 코드와 키 정리를 별도 cleanup phase 로 삭제한다.

### 결과

- poisoned snapshot 혼입 경로를 끊는다.
- 다만 migration 동안 fallback 여부를 명시적으로 관측해야 한다.

## ADR-004: duplicate subscribe 는 idempotent, reconnect 는 explicit resync

**상태**: Accepted  
**날짜**: 2026-04-13

### 결정

- 같은 socket-session pair 의 중복 subscribe 는 새 snapshot 을 보내지 않는다.
- socket 가 바뀐 reconnect, explicit unsubscribe 후 재subscribe, session restart 는 새 snapshot barrier 를 연다.
- `WsRouter` 는 `snapshotId` 와 socket identity 를 기준으로 replay state 를 관리한다.

### 결과

- React StrictMode / remount 로 인한 중복 화면 누적을 막는다.
- 실제 재연결 복구는 계속 허용된다.

## ADR-005: 테스트 전략은 server unit + protocol race + browser E2E 3층 구조

**상태**: Accepted  
**날짜**: 2026-04-13

### 결정

- server test-runner:
  - snapshot ordering
  - stale ack rejection
  - duplicate subscribe idempotency
  - resize/alt-screen/headless metrics
- protocol race test:
  - snapshot 중 live output interleave
  - reconnect 직후 queued output flush
- frontend E2E:
  - refresh
  - workspace switch
  - grid remount
  - Codex/TUI path

### 결과

- 단일 계층 테스트만으로 놓치기 쉬운 race 를 명시적으로 잡는다.

## 외부 라이브러리 및 버전 계획

| 라이브러리 | 위치 | 상태 | 용도 |
|-----------|------|------|------|
| `@xterm/headless` | server | 신규 추가 예정 | 서버 side terminal state |
| `@xterm/addon-serialize` | server | PoC 검증 후 추가 여부 확정 | headless snapshot 직렬화 |
| `@xterm/xterm` | frontend | 유지 | 렌더러 |
| `node-pty` | server | 유지 | PTY bridge |
| `ws` | server | 유지 | snapshot/live output transport |

## 계획 승인 조건

- [x] 엔진 선택 근거가 연구 보고서와 현재 코드 구조에 연결되어 있다.
- [x] 프로토콜 변경이 명명, ack 토큰, idempotency 기준까지 포함한다.
- [x] browser snapshot retirement 정책과 rollback flag 정책이 분리되어 있다.
- [x] 테스트 전략이 단위/경합/E2E 레이어로 분리되어 있다.
