# Terminal Screen Repair V2 구현 계획

## 0. 프로젝트 온보딩 컨텍스트

BuilderGate는 브라우저에서 xterm 터미널을 여러 Workspace와 React Mosaic Grid tile로 표시한다. 현재 Grid/workspace repair 경로는 화면 크기 변경 또는 workspace 이동 후 서버에 `repair-replay`를 보내고, 서버가 full `screen-snapshot`을 다시 전송하면 프런트가 `replaceWithSnapshot()`으로 `term.reset()` 후 snapshot 전체를 write한다. Codex처럼 scrollback이 긴 AI TUI에서는 이 full snapshot 재적용이 웹 터미널에서 처음부터 마지막까지 느리게 재출력되는 사용자 체감 버그로 나타난다.

이번 구현의 핵심은 `full snapshot restore`와 `screen repair`를 프로토콜과 적용 경로에서 분리하는 것이다. 재구독, 재연결, 신규 subscribe는 계속 full `screen-snapshot` handshake를 사용한다. Grid surface 가시화, workspace 이동, tile resize, 수동 middle mouse repair는 현재 viewport만 row patch로 보정하는 `screen-repair` 경로를 사용한다.

품질 기준은 SRS의 `FR-SESS-016`, `FR-SESS-017`, `FR-WS-012`, `FR-WS-013`, `FR-GRID-012`, `FR-GRID-016`, `FR-GRID-017`, `AC-020`이다. scrollback 축소, repair 비활성화, full snapshot fallback, `term.reset()` 기반 repair는 구현 범위 밖이다.

## 1. 실행 가드

`scope_freeze`는 true다. 계획 변경은 `change_log`에 요구사항 ID, 사유, 영향 파일을 남긴다.

Pre-commit gate는 frontmatter의 `pre_commit_gate`를 따른다. Windows 수동 검증과 Playwright E2E는 AGENTS.md 규칙에 따라 `https://localhost:2002`를 대상으로 실행한다.

금지 구현:

- Grid/workspace repair에서 `screen-snapshot` 재전송을 기본 경로로 쓰지 않는다.
- Grid/workspace repair에서 `TerminalView.replaceWithSnapshot()` 또는 `term.reset()`을 호출하지 않는다.
- scrollback 크기, snapshot byte budget, terminal history 보존 정책을 낮추지 않는다.
- degraded headless 상태에서 full snapshot으로 자동 전환하지 않는다. `screen-repair:rejected`로 관측 가능하게 종료한다.
- active buffer type이 normal/alternate 중 어느 것인지 payload에 포함하고, 클라이언트 active buffer와 다르면 적용하지 않는다.
- 사용자가 scrollback을 보고 있거나 IME composition 또는 터미널 입력이 진행 중이면 server request 전 단계에서 defer하고, request 이후 상태가 바뀌면 `screen-repair:failed`로 종료한다.

**implementation_steps**

1. 서버와 프런트의 `ClientWsMessage` union에 `ScreenRepairRequestMessage`, `ScreenRepairReadyMessage`, `ScreenRepairFailedMessage`를 추가한다.
2. 서버와 프런트의 `ServerWsMessage` union에 `ScreenRepairMessage`와 `ScreenRepairRejectedMessage`를 추가한다.
3. `ScreenSnapshotMessage`와 이름이 겹치지 않게 `repairToken`을 사용한다. `replayToken`은 full snapshot restore 전용으로 남긴다.
4. `ScreenRepairMessage`에 full snapshot 문자열 필드 이름인 `data`를 추가하지 않는다. patch 본문은 `viewportRows`와 `ansiPatch`로 제한한다.
5. request에는 `clientAtBottom`과 `clientBufferType`을 포함한다. 서버는 false 또는 buffer mismatch 요청을 reject할 수 있어야 한다.

**rationale**

`screen-snapshot`과 `screen-repair` token과 payload 이름을 분리해야 프런트가 repair를 snapshot replacement로 착각하지 않는다.

**pitfalls**

- `screen-repair` 서버 message와 클라이언트 request가 같은 `type`을 공유하므로 타입 narrowing 시 필드 존재로 방향을 구분한다.
- `data` 필드를 재사용하면 기존 `replaceWithSnapshot()` 경로로 연결될 위험이 크다.
- `screen-repair:failed`는 repair 적용 실패를 서버에 알리는 메시지다. 성공 ACK처럼 처리하면 실제 화면 보정 여부를 오판한다.

export interface HeadlessScreenRepairPayload {
  seq: number;
  cols: number;
  rows: number;
  bufferType: 'normal' | 'alternate';
  cursor: { x: number; y: number; hidden?: boolean };
  viewportRows: ScreenRepairRowPatch[];
  ansiPatch: string;
}

export type HeadlessScreenRepairResult =
  | { ok: true; payload: HeadlessScreenRepairPayload }
  | { ok: false; reason: 'geometry-mismatch' | 'buffer-mismatch' | 'headless-degraded' | 'generation-failed' };

export function serializeHeadlessScreenRepair(
  state: HeadlessTerminalState,
  expected: { cols: number; rows: number; bufferType: 'normal' | 'alternate' },
  maxBytes: number,
): HeadlessScreenRepairResult;

export async function getScreenRepair(
  sessionId: string,
  expected: { cols: number; rows: number; bufferType: 'normal' | 'alternate' },
): Promise<HeadlessScreenRepairResult>;
```

**implementation_steps**

1. `headlessTerminal.ts`에 viewport 전용 serializer를 추가한다. `state.terminal.buffer.active.viewportY + y`로 visible line을 읽고 `rows`개만 추출한다.
2. 각 line은 `translateToString(true)` 기반 `text`와 cell attribute 기반 ANSI writer로 `ansi`를 생성한다. `serializeAddon.serialize({ scrollback: 0 })`는 repair row patch 생성에 사용하지 않는다.
3. cell attribute 기반 ANSI writer는 foreground/background color, bold, dim, italic, underline, inverse, hidden, strikethrough를 row 안에서 변경 지점별 SGR sequence로 출력하고 row 끝에서 reset한다.
4. `ansiPatch`는 save cursor, row별 absolute cursor 이동, line clear, row write, server cursor 복원 순서로 만든다. newline을 쓰지 않고 viewport 밖으로 scroll되지 않게 한다.
5. `SessionManager.getScreenRepair()`를 async로 추가한다. `pendingHeadlessWrites > 0`이면 `headlessWriteChain` 종료를 짧은 timeout 안에서 기다린 뒤 생성한다.
6. `headlessHealth !== 'healthy'` 또는 headless 없음은 full snapshot fallback이 아니라 `{ ok:false, reason:'headless-degraded' }`를 반환한다.
7. expected cols/rows와 session cols/rows 또는 headless cols/rows가 다르면 `{ ok:false, reason:'geometry-mismatch' }`를 반환한다.
8. expected `bufferType`과 headless active buffer type이 다르면 `{ ok:false, reason:'geometry-mismatch' }`가 아니라 buffer mismatch reject reason을 반환하도록 WsRouter에 enough metadata를 전달한다.
9. 기존 `getScreenSnapshot()` 동작은 subscribe/reconnect용으로 유지한다.

**rationale**

문제의 원인은 full snapshot을 repair에도 재사용한 데 있다. 서버가 cell attribute 기반 viewport-only patch를 만들면 repair payload 크기는 rows와 cols에 비례하고 scrollback 길이에 비례하지 않으며, color/cursor 품질을 보존할 수 있다.

**pitfalls**

- `serializeHeadlessTerminal()`의 기본 호출은 scrollback 전체를 포함하므로 repair에서 호출하지 않는다.
- `serialize({ scrollback: 0 })` stream은 cursor movement, line feed, alternate buffer sequence를 섞을 수 있으므로 repair row patch 생성에 사용하지 않는다.
- headless degraded 상태에서 fallback snapshot을 보내면 같은 버그가 되살아난다.
- normal buffer와 alternate buffer를 혼동하면 TUI 화면을 잘못 덮는다.

  details?: { repairToken?: string; cols?: number; rows?: number },
): void;
```

**implementation_steps**

1. `handleMessage()` switch에 `screen-repair`, `screen-repair:ready`, `screen-repair:failed`를 추가한다.
2. `repairPendingSessions`를 `replayPendingSessions`와 별도로 둔다. 구조는 `repairToken`, `sessionId`, `queuedOutputChunks`, `queuedOutputBytes`, `timer`, `seq`, `cols`, `rows`를 가진다.
3. `screen-repair` request가 미구독 session이면 `screen-repair:rejected`를 보낸다.
4. 같은 session repair pending이면 `screen-repair:rejected` reason `pending`을 보낸다.
5. request의 `clientAtBottom`이 false면 pending을 만들지 않고 `screen-repair:rejected` reason `apply-rejected`를 보낸다.
6. `sessionManager.getScreenRepair(sessionId, { cols, rows, bufferType: clientBufferType })`를 호출한다. `ok:false`는 full snapshot fallback 없이 reject로 보낸다.
7. `ok:true`면 `screen-repair` message를 send하고 해당 session output을 ACK 전까지 queue한다.
8. `screen-repair:ready` ACK가 token과 일치하면 queue를 `output`으로 flush하고 pending을 정리한다.
9. `screen-repair:failed`가 token과 일치하면 repair 미적용 debug event를 기록하고 queued output을 원래 순서대로 flush한 뒤 pending을 정리한다.
10. ACK timeout도 queued output을 flush한 뒤 pending을 정리한다. timeout이 output을 drop하거나 영구 보류하면 실패다.
11. repair queued output은 기존 `appendQueuedOutput()`의 tail-trim 로직을 재사용하지 않는다. chunk array에 순서대로 보관하고 flush 시 join한다.
12. repair queue hard cap을 초과하면 `screen_repair_queue_overflow` debug/replay event를 기록하고 repair pending을 abort한다. abort 시 이미 queued된 output과 overflow를 유발한 output을 순서대로 즉시 flush한 뒤 이후 output은 일반 `output` 경로로 보낸다. tail-trim으로 일부 output만 보내는 동작은 실패다.
13. stale token은 debug event만 남기고 queue를 flush하지 않는다.
14. 기존 `handleRepairReplay()`는 legacy 직접 호출 테스트를 위해 유지하되 Grid/workspace 경로에서 사용하지 않는다는 관찰 이벤트를 남긴다.

**rationale**

full snapshot replay와 repair는 queue 경계가 비슷하지만 payload와 fallback 정책이 다르다. 별도 pending map을 두면 replay ACK와 repair ACK가 섞이지 않는다.

**pitfalls**

- replay pending 중 repair request가 오면 순서가 깨질 수 있다. 이 경우 `pending` 또는 `generation-failed` reject로 닫고 subscribe replay를 우선한다.

- 기존 replay queue helper는 limit 초과 시 tail-trim한다. repair queue에서 재사용하면 FR-WS-013 위반이다.

## 8. 용어집

| 용어 | 정의 |
|---|---|
| full snapshot restore | subscribe/reconnect에서 터미널 전체 상태를 복원하는 `screen-snapshot` handshake |
| screen repair | Grid/workspace layout 변화 뒤 현재 viewport만 row patch로 보정하는 비파괴 repair |
| viewport row patch | terminal rows 수를 넘지 않는 bounded row payload |
| replayToken | full snapshot restore ACK token |
| repairToken | screen repair ACK token |
| degraded headless | 서버 headless terminal이 snapshot 또는 repair를 신뢰성 있게 만들 수 없는 상태 |
| user scroll position | xterm buffer의 `viewportY`가 bottom `baseY`와 다른 상태 |
