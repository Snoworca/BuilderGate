## 1. 개요

이 계획은 `0.5.3step1`의 터미널 컨텍스트 메뉴 등록 항목 붙여넣기 구현에 대한 후속 보강 계획이다. 기존 구현은 FR-ARCH-001부터 FR-ARCH-006의 핵심 동작을 제공하지만, 3개 서브에이전트 최종 분석에서 다음 개선 후보가 발견되었다.

- 느린 `commandPresetApi.getAll()` 응답이 사용자가 닫은 메뉴를 뒤늦게 다시 열 가능성
- 데스크톱 우클릭 직후 키보드 탐색이 실제 focus 경로에서 약한 점
- 모바일 다이얼로그의 focus와 ARIA 역할 계약 보강
- 붙여넣기 E2E의 `contains` 기반 assertion false-positive 가능성
- category/control-character/viewport/mobile split coverage 보강

본 계획은 새 기능 범위를 추가하지 않는다. 이미 정의된 `0.5.3step1` 요구사항의 완료 품질을 높이는 hardening 작업만 포함한다.

## 2. 선행 조건 및 전제

- 대상 서버 검증 포트는 사용자 지시에 따라 반드시 `https://localhost:2222`다.
- 본 `0.5.3step1` hardening 실행에서는 `https://localhost:2002`로 테스트하면 안 된다.
- 이 계획의 `2222` 포트는 repository 기본 검증 포트보다 우선하는 사용자 명시 지시다.
- `frontend/playwright.config.ts`의 전역 기본값과 관계없이, 본 계획의 모든 E2E gate는 `PLAYWRIGHT_BASE_URL='https://localhost:2222'`를 명시한다.
- `2222`에서 실행 중인 BuilderGate HTTPS 서버가 없으면 E2E gate를 시작하지 말고, 해당 포트로 서버를 먼저 기동한다.
- 등록 항목 붙여넣기는 계속 실행이 아니라 terminal input gate를 통한 paste-only 동작이다.
- debug capture에 raw preset 값을 기록하지 않는다.
- 기존 메뉴 label과 시각 스타일은 변경하지 않는다.

### Phase 1 — 메뉴 열기 수명주기와 focus 안정화

#### TASK-H1-001 — pending context-menu open 무효화와 slow-load snapshot 계약 보강

**Related requirements**: FR-ARCH-004, FR-ARCH-005, FR-ARCH-006

**Files**

- 수정: `frontend/src/App.tsx`
- 수정: `frontend/src/components/Grid/MosaicContainer.tsx`
- 수정: `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts`

**Source anchors**

- `frontend/src/App.tsx:276-316`
- `frontend/src/components/Grid/MosaicContainer.tsx:587-624`
- `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts:140-181`

**Implementation guide**

1. `closeTabContextMenu`에서 `tabContextMenuOpenSeqRef.current += 1`을 먼저 수행한 뒤 `tabContextMenu.close()`와 snapshot clear를 수행한다.
2. `MosaicContainer`의 `closeContextMenu`에서도 `contextMenuOpenSeqRef.current += 1`을 먼저 수행한다.
3. `openTabContextMenu`와 `openContextMenu`의 async preset load callback은 기존 seq equality check를 그대로 사용한다. close가 seq를 증가시키면 이미 닫힌 메뉴는 다시 열리지 않는다.
4. slow-success API는 등록 항목을 누락하지 않는다. preset load가 늦게 성공하면 성공한 snapshot으로 메뉴를 한 번 연다.
5. slow-failure API는 기존처럼 console warning을 남기고 registered menu 없이 built-in 메뉴를 연다.
6. close 이후 resolve된 success/failure callback은 UI를 다시 열거나 snapshot을 바꾸지 않는다.
7. built-in 메뉴를 preset load 전에 먼저 여는 loading UI는 이 계획에서 구현하지 않는다. 해당 변경은 등록 항목 존재 시 메뉴 표시 계약을 바꾸므로 별도 SRS 변경이 선행되어야 한다.
8. 동일 로직이 Tab Mode와 Grid Mode에서 같은 동작을 하도록 local helper 함수 또는 동일 구조의 작은 함수로 작성한다.

**Rationale**

닫힌 뒤 늦게 도착한 async callback이 메뉴를 다시 여는 것은 명시적 dismiss 의도를 깨뜨린다. 반대로 등록 항목이 있는데 slow-success에서 built-in-only 메뉴를 여는 것은 FR-ARCH-005의 표시 계약을 약하게 만들 수 있으므로, 이 계획은 close invalidation과 slow-success correctness를 우선한다.

**Pitfalls**

- slow-success 응답은 등록 항목 snapshot을 포함한 메뉴를 연다.
- close 이후 resolve된 promise는 console warning 외 UI 변경을 만들지 않는다.
- duplicate contextmenu guard는 유지한다.

**Tests**

- E2E에서 `GET /api/command-presets`를 지연시키고, 메뉴를 닫은 뒤 응답을 풀어도 `.context-menu`가 다시 생기지 않는지 검증한다.
- E2E에서 `GET /api/command-presets` slow-success를 유도하고, 응답 이후 등록 항목 메뉴와 preset label이 보이는지 검증한다.
- E2E에서 slow-success 응답 전 메뉴를 닫으면, 응답 이후 등록 항목 메뉴와 built-in 메뉴가 모두 다시 생기지 않는지 검증한다.
- Grid Mode로 전환한 뒤 slow-load close/no resurrection 또는 registered paste 중 최소 1개 경로를 직접 검증한다.
- 기존 load failure stale snapshot test는 유지한다.

#### TASK-H1-002 — 데스크톱 메뉴 초기 focus와 roving keyboard navigation 보강

**Related requirements**: FR-ARCH-001, FR-ARCH-004

**Files**

- 수정: `frontend/src/components/ContextMenu/ContextMenu.tsx`
- 수정: `frontend/src/components/ContextMenu/ContextMenu.css`
- 수정: `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts`

**Source anchors**

- `frontend/src/components/ContextMenu/ContextMenu.tsx:90-196`
- `frontend/src/components/ContextMenu/ContextMenu.tsx:296-316`
- `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts:99-114`

**Implementation guide**

1. root desktop menu mount 후 첫 enabled `.context-menu-item`에 focus한다.
2. `MenuItemList` 또는 root menu container에 keyboard handler를 추가한다.
   - `ArrowDown`: 현재 menu scope의 다음 enabled item으로 focus 이동
   - `ArrowUp`: 이전 enabled item으로 focus 이동
   - `Home`: 첫 enabled item으로 focus 이동
   - `End`: 마지막 enabled item으로 focus 이동
   - `Escape`: root menu close
   - `ArrowRight`: children이 있는 item에서 submenu open
   - `ArrowLeft`: submenu 내부에서는 해당 submenu를 닫고 parent item으로 focus 복귀
3. submenu가 열리면 submenu의 첫 enabled item에 focus한다.
4. paste action 뒤에는 기존 terminal focus restore 동작을 유지한다.
5. disabled item은 `tabIndex=-1` 유지, roving 대상에서 제외한다.

**Rationale**

현재 테스트는 row를 직접 focus한 뒤 keyboard handler를 호출한다. 실제 사용자는 우클릭 직후 terminal helper textarea에 focus가 남아 있을 수 있으므로 keyboard path가 요구사항만큼 강하지 않다.

**Pitfalls**

- terminal focus restore와 menu initial focus가 충돌하지 않게 action close 이후에만 terminal focus를 복원한다.
- submenu portal이 document body에 렌더링되므로 focus scope 계산은 현재 `.context-menu` root element 내부로 제한한다.

**Tests**

- E2E에서 우클릭 직후 별도 `.focus()` 호출 없이 `ArrowDown`, `Enter`, `ArrowRight`로 submenu가 열리는지 검증한다.
- disabled `복사` item이 keyboard target이 되지 않는지 검증한다.
- paste 선택 후 xterm helper textarea focus가 복원되는 기존 assertion을 유지한다.

**Acceptance tests**

### Phase 2 — 모바일 다이얼로그 접근성 보강

#### TASK-H2-001 — mobile dialog focus, role, close restore 계약 정리

**Related requirements**: FR-ARCH-003, FR-ARCH-004

**Files**

- 수정: `frontend/src/components/ContextMenu/ContextMenu.tsx`
- 수정: `frontend/src/components/ContextMenu/ContextMenu.css`
- 수정: `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts`

**Source anchors**

- `frontend/src/components/ContextMenu/ContextMenu.tsx:354-437`
- `frontend/src/components/ContextMenu/useContextMenuHistory.ts:35-95`
- `frontend/src/components/ContextMenu/ContextMenu.css:128-184`
- `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts:288-334`

**Implementation guide**

1. mobile dialog mount 전 `document.activeElement`를 ref에 저장한다.
2. dialog mount 후 `.context-menu-dialog` 또는 첫 enabled `.context-menu-dialog-item`에 focus한다.
3. close 시 이전 focus target이 아직 document 안에 있으면 그 element로 focus를 복원한다. terminal paste action은 기존 terminal focus restore가 우선한다.
4. ARIA 역할은 둘 중 하나로 통일한다.
   - 권장: `.context-menu-dialog-list`의 `role="menu"`를 제거하고 button list로 노출한다.
   - menu role 유지 시 각 action button에 `role="menuitem"`을 추가하고 separator에 `role="separator"`를 추가한다.
5. `Tab`과 `Shift+Tab`이 dialog 내부 focusable element 사이에서 순환하도록 최소 focus trap을 추가한다.
6. `Escape`, backdrop click, browser back, header back 모두 `closeWithHistory` 또는 `backWithHistory` 경로를 사용한다.

**Rationale**

모바일 UI는 시각적으로 정상 동작하지만 modal focus와 role 계약이 약하면 키보드와 assistive tech 사용자가 경로를 잃을 수 있다.

**Pitfalls**

- Android back 처리용 synthetic history entry를 focus restore 로직에서 변경하지 않는다.
- leaf action이 paste를 수행한 뒤 dialog close와 history cleanup을 한 번만 수행하는지 유지한다.

**Tests**

- Mobile Safari E2E를 path navigation, header back, browser back root close, backdrop child close, leaf action exactly once, reopen root state로 분리한다.
- mobile leaf action 뒤 paste debug event 증가량이 정확히 1인지 유지한다.
- dialog open 직후 active element가 dialog 또는 첫 action인지 검증한다.
- `.context-menu-dialog-list`의 role 계약을 검증한다. button list 방식을 선택하면 `role="menu"`가 없는지 확인하고, menu 방식을 선택하면 각 action button에 `role="menuitem"`이 있는지 확인한다.
- `Tab`과 `Shift+Tab`을 눌러 focus가 dialog 내부에서 순환하는지 검증한다.
- backdrop, header close, browser back, leaf action close 후 이전 focus target 또는 terminal helper textarea로 focus가 복원되는지 검증한다.

### Phase 3 — 검증 false-positive 축소

#### TASK-H3-001 — exact paste assertion을 current input line 기준으로 강화

**Related requirements**: FR-ARCH-004, FR-ARCH-006

**Files**

- 수정: `frontend/tests/e2e/helpers.ts`
- 수정: `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts`

**Source anchors**

- `frontend/tests/e2e/helpers.ts:160-207`
- `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts:40-98`
- `frontend/tests/e2e/terminal-context-menu-registered-items.spec.ts:342-351`

**Implementation guide**

1. `expectTerminalScreenContains`를 유지하되, 등록 항목 paste 검증에는 새 helper `expectVisibleTerminalCurrentInputEquals(page, value)`를 사용한다.
2. helper는 `.terminal-view:visible .xterm-rows`의 text를 읽고 마지막 prompt 이후 현재 입력 구간을 추출한다.
   - PowerShell prompt pattern: `PS <cwd>>`
   - fallback: 마지막 non-empty row의 trailing text
3. assertion은 `currentInput === value` 또는 `currentInput.endsWith(value)`가 아니라 exact equality를 기본으로 한다.
4. command와 prompt payload는 공백 없는 ASCII를 사용해 parser ambiguity를 줄인다.
5. directory payload는 `C:\\Work\\...` 형태의 backslash 포함 Windows path를 유지해 directory exact paste와 `cd` 미생성 요구를 계속 검증한다.
6. debug metadata 검증은 계속 `source`, `byteLength`, `codePointCount`, `enterCount`, `controlCount`를 확인한다.

**Rationale**

`toContainText`는 화면 어딘가에 기대 문자열이 있으면 통과한다. 현재 입력 라인을 exact로 비교하면 같은 길이의 잘못된 payload나 이전 출력에 기대 문자열이 남은 경우를 걸러낼 수 있다.

**Pitfalls**

- debug event에 raw input을 기록하지 않는다.
- prompt parser가 실패하면 테스트를 성공시키지 말고 명시적으로 helper error를 throw한다.

**Tests**

- command, directory, prompt E2E 모두 current input exact equality를 사용한다.
- directory E2E는 backslash 포함 Windows path를 exact equality로 검증하고, visible terminal에 `cd `가 나타나지 않는 assertion을 유지한다.
- exact helper unit 수준이 필요하면 작은 pure parser function을 `helpers.ts` 내부 또는 `tests/e2e` 전용 util로 분리한다.
