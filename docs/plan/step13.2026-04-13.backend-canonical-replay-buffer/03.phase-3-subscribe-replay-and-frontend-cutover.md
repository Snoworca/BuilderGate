# Phase 3: Subscribe Replay And Frontend Cutover

## Objective

Switch the client recovery path from browser-primary snapshots to server-primary replay.

## Prerequisites

- Phase 1 message contract complete
- Phase 2 replay snapshot API complete

## Architecture Reference

- `history` is authoritative
- `output` remains live incremental data
- terminal must clear before applying `history`

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/ws/WsRouter.ts`
- `server/src/services/WorkspaceService.ts`
- `server/src/routes/sessionRoutes.ts`
- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/types/ws-protocol.ts`

## Planned Changes

1. `SessionManager` stops owning final direct fanout for replay-sensitive delivery and routes session output through router-controlled delivery.
2. `WsRouter.handleSubscribe()` sends `history`, keeps the socket-session pair replay-pending, and waits for `history:ready`.
3. `WsRouter` owns the replay-pending queue with:
   - cap `min(pty.maxBufferSize, 262144 characters measured with string.length)`
   - ack timeout `5000 ms`
   - batch-level `subscribed` metadata is emitted immediately after batch classification
4. Frontend session handlers gain `onHistory` and replay lifecycle support.
5. `TerminalContainer` owns a replay state machine:
   - `awaiting-history-decision`
   - `replaying-server-history`
   - `live`
6. `TerminalView` adds a promise-based `replaceWithHistory(data)` or equivalent clear-and-write completion contract.
7. Existing browser snapshot restore becomes fallback-only and must not auto-run before the server-history decision is made.

## Behavioral Rules

- history replay must clear the existing xterm buffer first
- live output that arrives while history is writing must be buffered
- after history write finishes, buffered live output must flush in order
- reconnecting terminals must not show duplicated pre-refresh content
- duplicate subscribe on the same socket-session pair must not trigger replay again
- browser snapshot fallback may run only if the server-history decision resolves to `no history available`
- replay-pending queues must be bounded and cleaned up on timeout
- `history:ready` is per session even when the original subscribe call was batched
- `unsubscribe` and socket close must clear replay-pending state immediately
- session deletion and restart must clear replay-pending state immediately
- batch-level `subscribed` is metadata-only and must not be treated as proof that replay-pending sessions are already live

## Frontend Replay Gate

- `TerminalView` mount must not immediately apply the current browser snapshot as the primary restore path.
- `TerminalContainer` must decide between:
  - server history path
  - fallback snapshot path
  - empty terminal path
- once `history` arrives, local snapshot replay is permanently skipped for that mount instance

## Async Completion Contract

- `replaceWithHistory(data)` must resolve only after xterm has finished the clear-and-write cycle
- only after that resolution may the client send `history:ready`
- only after `history:ready` may the server flush queued live output for that pair
- if replay timeout fires before `history:ready`, the client must be forced through a clean re-subscribe path rather than leaving the pair half-live
- if the user unsubscribes or the socket closes, the server must discard replay-pending backlog immediately
- if the session is deleted or restarted, the server must discard replay-pending backlog for the old session ID immediately

## Test

### Scenario 1: Refresh restores server history

Given a terminal with long scrollback
When the browser refreshes
Then the reconnected terminal clears and replays server history before showing new live output

### Scenario 2: Reconnect does not duplicate old lines

Given a mounted terminal instance that loses and regains WS connectivity
When history is received on reconnect
Then the final visible content is not duplicated by the new replay

### Scenario 3: Local snapshot is bypassed when server history exists

Given both browser snapshot data and server replay data exist
When the terminal restores
Then server history wins and the browser snapshot path does not append duplicate content

### Scenario 4: Output during history write is preserved

Given live PTY output arrives while the client is replaying server history
When replay completes
Then the buffered live chunks flush afterward in order

### Scenario 5: Boundary case with empty history

Given the server returns empty history for a live session
When the client handles subscribe
Then normal live output still appears and the terminal remains usable

### Scenario 6: Live output races history replay

Given live PTY output is emitted while a client is still replaying `history`
When the replay finishes
Then the client receives exact ordered content with no interleaving before `history:ready`

### Scenario 7: Replay timeout does not leak memory

Given a client receives `history` but never sends `history:ready`
When the replay-ack timeout expires
Then the server drops the pending queue for that pair and the system remains bounded

### Scenario 8: Session delete or restart clears replay-pending state

Given a session has a replay-pending subscriber
When the session is deleted or restarted
Then the server discards replay-pending backlog for the old session immediately
