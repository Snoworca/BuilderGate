# Backend Canonical Replay Buffer Plan

Date: 2026-04-13
Scope: lightweight but precise server-side terminal history preservation across refresh, reconnect, and workspace switching
Mode: plan only

## Goal

Make the backend the canonical owner of replayable terminal history so that:

- browser refresh no longer depends on fragile `localStorage` screen snapshots
- reconnect and workspace switching can recover long scrollback from the server
- history replay does not duplicate transient newline bursts, alternate-screen noise, or partial escape sequences

## Input Basis

- current implementation in `server/src/services/SessionManager.ts`, `server/src/ws/WsRouter.ts`, `frontend/src/components/Terminal/*`
- prior analysis in `docs/report/2026-04-13.workspace-switch-scrollback-root-cause.md`
- prior preservation notes in `docs/report/2026-04-13.terminal-scrollback-preservation.md`

## Requirements

- `BG-001` Refresh and reconnect must restore useful terminal history without requiring frontend-only snapshots.
- `FR-001` Each session must maintain a bounded canonical replay buffer on the backend.
- `FR-002` Replay buffer append must happen regardless of current WS subscriber count.
- `FR-003` Truncation must preserve valid control-sequence boundaries and avoid broken ANSI fragments.
- `FR-004` Buffer recording must define a filter policy for alternate-screen and transient full-screen noise.
- `FR-005` Subscribe flow must send explicit history replay before normal live streaming.
- `FR-006` Frontend must clear-and-replay server history without duplicating existing terminal content.
- `FR-007` Existing browser snapshot restore must be demoted to fallback-only behavior or removed from the primary path.
- `FR-008` Unit, integration, and manual regression coverage must prove refresh, reconnect, and long-output recovery.

## Assumptions

1. The requested “lightweight solution” excludes a server-side headless terminal emulator.
2. Exact pixel-for-pixel screen parity is not required; stable replayable history is the target.
3. Current `pty.maxBufferSize` should remain the primary bounded-capacity setting unless a migration becomes unavoidable.
4. Multiple concurrent viewers of the same session remain supported.

## Recommended Execution Order

1. Define the contract first.
2. Replace backend buffer semantics second.
3. Switch subscribe/replay and frontend restore third.
4. Finish with regression gates and rollout checks.

## Success Criteria

- Refresh of an active session restores server history without relying on `SerializeAddon` as the primary source.
- WS reconnect does not collapse scrollback to “recent few lines”.
- Long newline-heavy output does not create duplicated replay gaps caused by mixed local snapshot and live output.
- Bounded replay buffer remains within configured memory limits.
- Existing tab, grid, and multi-session behavior remains intact.

## Out Of Scope

- introducing a headless terminal emulator
- persistent history across full server restarts
- changing workspace persistence format in `server/data/workspaces.json`
- redesigning unrelated terminal UI
