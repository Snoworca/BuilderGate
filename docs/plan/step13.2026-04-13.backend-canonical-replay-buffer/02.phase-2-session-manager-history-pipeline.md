# Phase 2: Session Manager History Pipeline

## Objective

Implement precise backend replay recording in `SessionManager` with bounded memory and safe truncation.

## Prerequisites

- Phase 1 protocol and naming contract accepted

## Architecture Reference

- server is the source of replayable PTY output
- truncation safety matters more than byte-perfect raw retention

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/services/SessionManager.test.ts`
- `server/src/test-runner.ts`

## Planned Changes

1. Append replay content for every PTY output chunk.
2. Apply filter policy before appending to replay storage.
3. Replace naive tail slicing with safe escape-boundary truncation.
4. Track `truncated` state alongside replay content.
5. Expose a `getReplaySnapshot(sessionId)` API for `WsRouter`.

## Detailed Notes

### Recording Rule

- append on every `onData`
- do not clear replay content on subscribe
- continue broadcasting live `output` to active subscribers

### Truncation Rule

- hard cap stays bounded by `pty.maxBufferSize`
- preserve only the newest valid tail
- never leave the stored tail starting inside an incomplete CSI or OSC sequence
- when `pty.maxBufferSize` is lowered at runtime, all in-memory replay buffers must be re-truncated immediately

### Filter Rule

Initial implementation should be conservative:

- preserve normal shell output
- strip known alternate-screen enter or leave noise that destroys replay utility
- do not attempt full TUI normalization in this phase

## Test

### Scenario 1: Replay buffer survives active subscription

Given a session with an active subscriber
When new PTY output arrives
Then replay storage still records that output

### Scenario 2: Replay buffer remains bounded

Given repeated output exceeding `pty.maxBufferSize`
When replay append runs
Then stored replay length stays within the cap

### Scenario 3: Escape-boundary truncation is valid

Given output whose truncation point lands inside an ANSI escape sequence
When replay truncation runs
Then the stored replay starts at the next safe boundary

### Scenario 4: Alternate-screen noise is suppressed

Given PTY output containing known alternate-screen sequences
When replay filtering runs
Then the replay buffer omits the destructive sequences while live broadcast remains unaffected

### Scenario 5: Empty or filtered chunk is harmless

Given output that becomes empty after replay filtering
When the append helper runs
Then replay state remains consistent and no exception is thrown

### Scenario 6: Runtime cap reduction re-truncates existing sessions

Given replay buffers larger than a newly lowered `pty.maxBufferSize`
When runtime config is applied
Then every in-memory replay buffer is immediately truncated to the new safe limit
