## Test

### Scenario 1: Long-output refresh recovery

Given a session with large scrollback near the configured replay limit
When the browser refreshes
Then the newest bounded history is restored and marked truncated if needed

### Scenario 2: Workspace switch plus reconnect

Given two workspaces with active sessions
When a reconnect occurs while the user is on a different workspace
Then both sessions recover from server replay rather than collapsing to recent tail only

### Scenario 3: Tab restart clears stale history

Given a tab is restarted and receives a new session ID
When the user reopens the tab
Then old replay or snapshot content does not leak into the new session

### Scenario 4: TUI noise remains bounded

Given a tool enters alternate screen repeatedly
When replay storage records output
Then the resulting replay remains usable and does not flood the restored terminal with destructive artifacts

### Scenario 5: Boundary case at exact buffer cap

Given replay content length exactly equals `pty.maxBufferSize`
When one more chunk is appended
Then truncation keeps the newest safe tail and flags truncation correctly

### Scenario 6: Replay barrier ordering

Given the server is replaying history to a reconnecting client
When fresh PTY output arrives before the client sends `history:ready`
Then the output is queued server-side and flushed only after replay completion

### Scenario 7: Replay timeout cleanup

Given a replay-pending client never acknowledges history
When the replay timeout expires
Then the queued backlog is dropped, memory remains bounded, and a future fresh subscribe can recover from canonical replay

### Scenario 8: Session lifecycle cleanup

Given a session is deleted or restarted while replay is pending
When lifecycle cleanup runs
Then all replay-pending state tied to the old session ID is removed immediately
