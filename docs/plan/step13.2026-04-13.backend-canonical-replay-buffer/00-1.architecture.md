# Architecture

## System Context

Current state:

- `node-pty` emits raw PTY output on the server
- `SessionManager` only keeps `outputBuffer` for unsubscribed gaps
- `WsRouter` flushes that gap buffer on subscribe and clears it
- frontend `TerminalView` reconstructs history from browser-local screen snapshots

Target state:

- server owns a bounded canonical replay buffer per session
- every subscribe or reconnect can request a full replay snapshot from that buffer
- frontend treats server replay as authoritative and only uses local snapshots as fallback

## Components

1. `SessionManager`
   Responsibilities:
   - append filtered PTY output into a bounded replay buffer
   - expose a stable replay snapshot API
   - keep truncation safe across ANSI escape boundaries

2. `WsRouter`
   Responsibilities:
   - send `history` before live `output`
   - keep subscribe ordering deterministic
   - avoid replay duplication on re-subscribe

3. `WebSocketContext` / `TerminalContainer`
   Responsibilities:
   - receive explicit history payloads
   - clear terminal and replay history before resuming live output
   - buffer live output until replay write completes

4. `TerminalView`
   Responsibilities:
   - expose a replay-safe clear/write boundary
   - stop acting as the primary history store

## Proposed Data Model

```ts
interface SessionReplayState {
  content: string;
  truncated: boolean;
  savedBytes: number;
  filterMode: 'none' | 'minimal' | 'standard';
}
```

Within `SessionData`, the plan recommends replacing the current `outputBuffer` semantics with a canonical replay field:

```ts
interface SessionData {
  // existing fields...
  replayBuffer: string;
  replayTruncated: boolean;
}
```

## Protocol Shape

Add a new server-to-client message:

```ts
{ type: 'history'; sessionId: string; data: string; truncated: boolean }
```

Add a new client-to-server replay completion signal:

```ts
{ type: 'history:ready'; sessionId: string }
```

Subscribe flow target order:

1. read replay snapshot from `SessionManager`
2. send `history`
3. mark the socket-session pair as `replay-pending`
4. send one batch-level `subscribed` metadata response immediately after the batch is classified
5. wait for per-session `history:ready`
6. flush queued live output for that session
7. continue normal `output`

## Sequence Diagram

```text
Client TerminalContainer -> WsRouter: subscribe(sessionId)
WsRouter -> SessionManager: getReplaySnapshot(sessionId)
SessionManager --> WsRouter: { data, truncated, status, cwd }
WsRouter -> Client TerminalContainer: history(sessionId, data, truncated)
WsRouter -> WsRouter: mark replay-pending(socket, sessionId)
WsRouter -> Client TerminalContainer: subscribed(batch status/cwd metadata)
PTY -> SessionManager: onData(rawOutput)
SessionManager -> SessionManager: appendFilteredReplay(rawOutput)
WsRouter -> WsRouter: queue output for replay-pending(socket, sessionId)
Client TerminalContainer -> WsRouter: history:ready(sessionId)
WsRouter -> Client TerminalContainer: queued output(sessionId, chunk...)
PTY -> SessionManager: onData(rawOutput)
SessionManager -> SessionManager: appendFilteredReplay(rawOutput)
SessionManager -> WsRouter: broadcast output(sessionId, chunk)
WsRouter -> Client TerminalContainer: output(sessionId, chunk)
```

## Buffer Recording Policy

- Always append filtered PTY output, not only when there are no subscribers.
- Apply safe truncation only after append.
- Preserve bounded tail according to `pty.maxBufferSize`.
- Track whether truncation happened so the client can disclose that history is partial.

## Replay Barrier

- A socket-session pair is either `replay-pending` or `live`.
- While `replay-pending`, live chunks for that pair must queue on the server.
- The pair becomes `live` only after the client sends `history:ready` after terminal clear-and-write completion.
- Each replay-pending pair must have:
  - a bounded queued-output cap
  - a replay-ack timeout
- If `history:ready` never arrives before timeout, the server must clean up the pending queue and force the pair back through a fresh subscribe path instead of retaining unbounded memory.
- `unsubscribe(sessionId)` and socket close must tear down replay-pending state immediately, without waiting for timeout.
- session deletion and session restart must also tear down replay-pending state immediately for the affected session ID.

## Replay Queue Policy

- `replay-pending` queue cap is a fixed internal limit:
  - `min(pty.maxBufferSize, 262144 characters measured with string.length)`
- replay-ack timeout is a fixed internal limit:
  - `5000 ms`
- This phase intentionally uses fixed constants instead of adding new runtime settings.

## Duplicate Subscribe Rule

- Duplicate `subscribe` calls for the same socket-session pair are idempotent.
- If the pair is already `live` or already `replay-pending`, the router must not send `history` again.
- A fresh replay happens only for a new socket or a future explicit force-replay path.

## Batch Subscribe Semantics

- `subscribe(sessionIds[])` remains batch-based at the transport level.
- Replay state is tracked per socket-session pair, not per batch.
- One subscribe call may therefore produce a mixed batch:
  - session A enters `replay-pending`
  - session B has empty history and becomes `live` immediately
  - session C is already live and is treated idempotently
- `history:ready` is acknowledged per session, not per batch.
- `subscribed` remains a batch-level metadata response and is emitted once immediately after the subscribe batch is classified.
- `subscribed` does not mean every session is already live; it only means the batch metadata handshake is complete.

## Delivery Ownership Refactor

- Pair-level replay queues cannot be implemented while `SessionManager.broadcastWs()` directly pushes to every subscriber.
- This plan therefore requires delivery fanout to become router-mediated:
  - `SessionManager` emits session output into `WsRouter`
  - `WsRouter` decides whether each socket-session pair is `live` or `replay-pending`
  - only `WsRouter` may flush queued output for replay-pending pairs

## Risks

- Replaying raw history into an existing terminal without clearing will duplicate output.
- Truncating across escape boundaries can corrupt subsequent replay.
- Alternate-screen heavy tools can pollute replay unless filter policy is explicit.

## Architectural Decision

For the lightweight solution, the replay source of truth must move from browser snapshot strings to backend raw-history replay. That is the smallest change that still removes the current class of refresh and reconnect failures.
