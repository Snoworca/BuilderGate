# Phase 1: Buffer Model And Protocol

## Objective

Define the backend replay contract before changing behavior so that server, router, and frontend all share the same history semantics.

## Prerequisites

- understand current `SessionManager.outputBuffer` lifecycle
- confirm current `ServerWsMessage` contract

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/types/ws-protocol.ts`
- `frontend/src/types/ws-protocol.ts`
- `server/src/ws/WsRouter.ts`

## Planned Changes

1. Replace the “gap-only buffer” concept with a canonical replay buffer concept.
2. Add an explicit `history` WS message type with:
   - `sessionId`
   - `data`
   - `truncated`
3. Add an explicit `history:ready` client message so the server knows when replay is fully applied on the client.
4. Document the exact subscribe ordering and replay barrier contract.
5. Define duplicate-subscribe idempotency for the same socket-session pair.
6. Define batch subscribe semantics when a single `subscribe(sessionIds[])` contains sessions in different replay states.
7. Define replay queue cap and replay-ack timeout behavior.
   - queue cap: `min(pty.maxBufferSize, 262144 characters measured with string.length)`
   - ack timeout: `5000 ms`
   - cleanup triggers: `history:ready`, `unsubscribe`, socket close, timeout
8. Define fallback behavior when replay is unavailable:
   - empty history is allowed
   - live output continues normally
9. Define rollout invariant:
   - `history` is additive and backward-compatible only within the same frontend/backend deploy pair

## Test

### Scenario 1: Fresh subscribe receives history first

Given a session with buffered replay content
When a client subscribes to that session
Then the client receives `history` before any new live `output`

### Scenario 2: Empty session subscribe is still valid

Given a newly created session with no replay content
When a client subscribes
Then the subscribe succeeds and the client receives either empty history or no history payload, followed by normal status data

### Scenario 3: Truncated history is disclosed

Given a session whose replay buffer exceeded the configured cap
When history is requested
Then the payload sets `truncated: true`

### Scenario 4: Unsupported frontend/backend mix is rejected in tests

Given protocol types are changed on one side only
When TypeScript build or integration tests run
Then the mismatch is detected before release

### Scenario 5: Boundary case for zero-length history

Given a session with zero replay bytes
When the server builds a history payload
Then the payload remains well-formed and does not crash the client replay path

### Scenario 6: Duplicate subscribe is idempotent

Given the same socket subscribes twice to the same session
When the second subscribe is processed
Then the router does not send a second `history` payload and does not duplicate terminal content

### Scenario 7: Mixed batch subscribe remains well-defined

Given one subscribe call contains a session with history and another with empty history
When the router processes the batch
Then replay-pending and live transitions occur per session without blocking the whole batch
