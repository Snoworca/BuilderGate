# Tech Decisions

## ADR-001: Do Not Introduce A Headless Terminal Emulator In This Phase

Decision:

- keep the implementation lightweight
- use backend raw replay buffer as the authoritative recovery source

Why:

- lower implementation cost
- no new heavy dependency in `server/package.json`
- good enough for refresh/reconnect recovery

Tradeoff:

- replay remains stream-faithful, not exact rendered-screen faithful

## ADR-002: Reuse `pty.maxBufferSize` As Replay Capacity

Decision:

- keep the existing runtime-editable cap as the replay buffer limit

Why:

- avoids config migration
- matches existing operational controls in runtime settings
- keeps rollout small and reversible

Tradeoff:

- the field name remains generic rather than replay-specific

## ADR-003: Introduce An Explicit `history` WS Message

Decision:

- do not overload normal `output` for full replay snapshots

Why:

- frontend needs a clear “replace current content” signal
- reconnect and fresh subscribe become distinguishable from normal live chunks
- easier regression testing

## ADR-004: Keep Browser Snapshot Restore As Fallback Only

Decision:

- server history wins whenever available
- local snapshot restore must be skipped or replaced once `history` arrives

Why:

- mixed local snapshot plus live replay is the source of duplicated newline and cursor drift issues

## ADR-005: Apply Replay Filters Conservatively

Decision:

- start with a conservative default filter policy
- strip or suppress only the patterns already known to destroy useful replay

Initial candidates:

- alternate screen enter/leave sequences
- incomplete CSI / OSC fragments at the truncation boundary

Non-goal for this phase:

- perfectly normalizing every TUI artifact

## ADR-006: Prefer Atomic Clear-And-Replay On The Client

Decision:

- receiving `history` triggers terminal clear, replay write, then buffered live-output flush

Why:

- smallest reliable way to handle refresh and reconnect on an already-mounted terminal instance
- reuses the existing buffered-output concept already present in `TerminalView`

## ADR-007: Add An Explicit Replay Barrier

Decision:

- introduce `history:ready` from client to server
- keep socket-session pairs in a replay-pending state until the client confirms replay write completion

Why:

- without a barrier, `history` and live `output` can interleave
- terminal write completion is asynchronous in the frontend
- this is the lightest ordering guarantee that does not require full sequence-number reconciliation

## ADR-008: Duplicate Subscribe Must Be Idempotent

Decision:

- preserve the current `alreadySubscribed` intent
- do not replay history again for duplicate subscribe on the same socket-session pair

Why:

- React remount and reconnect patterns already exist
- replaying again would duplicate history

## ADR-009: Replay Queue Limits Use Fixed Internal Constants

Decision:

- use `min(pty.maxBufferSize, 262144 characters measured with string.length)` for per-pair replay-pending queue cap
- use `5000 ms` for replay-ack timeout

Why:

- avoids config sprawl in the lightweight phase
- keeps memory bounded
- keeps timeout behavior deterministic for tests
- matches the current backend buffer accounting model without introducing a byte-encoder in the lightweight phase

## ADR-010: Router Owns Final Delivery Fanout

Decision:

- move effective per-socket output delivery control into `WsRouter`
- `SessionManager` remains the producer of session output, not the final fanout policy owner

Why:

- replay-pending queues are socket-session specific
- current direct fanout from `SessionManager.broadcastWs()` cannot express per-pair gating
