### 주요 기술 결정 요약

- 저장소 공용 설정 예시는 플랫폼 중립으로 유지하고, 실제 `config.json5` 자동 생성 시점에만 OS별 PTY 기본값을 주입한다.
- 비윈도우에서는 파일에 남아 있는 Windows 전용 PTY 값이 서버 부팅을 깨지 않도록 로드 경계에서 정규화한다.
- 로드 시 정규화는 우선 런타임 메모리 기준으로 적용하고, 자동 파일 rewrite는 이번 단계 범위에 넣지 않는다.
- Settings는 capability snapshot 기준으로 Windows 전용 필드를 숨기고, 저장 validation과 UI 계약이 동일한 정책을 보도록 정렬한다.
- 이번 단계는 새로운 shell 실행 엔진을 추가하지 않는다. 대신 이미 런타임이 지원하던 `cmd`/`zsh`/`sh`를 Settings/session route 계약과 일치시키는 수준까지 정렬한다.

### 구현 흐름 요약

```text
First boot, no config.json5
  -> ensureConfigExists()
  -> copy or render example
  -> apply OS-specific PTY defaults
  -> write config.json5

Existing config.json5
  -> loadConfig()
  -> parse JSON5
  -> normalize platform-incompatible PTY values for current OS
  -> configSchema parse
  -> RuntimeConfigStore / SessionManager consume normalized config

Settings page
  -> GET /api/settings
  -> capability snapshot marks Windows-only fields unavailable on non-Windows
  -> frontend hides those controls
  -> save validation uses the same platform policy
```

## 구현 방향 요약

1. `config.json5`가 없을 때 현재 OS에 맞는 PTY 기본값으로 생성되도록 bootstrap contract를 명시한다.
2. 기존 stale config가 있어도 non-Windows 부팅이 실패하지 않도록 load-time normalization을 추가한다.
3. Windows 전용 backend/shell 정책을 한 곳에서 정의하고 Settings capability, save validation, runtime check가 재사용하도록 정리한다.
4. Settings UI에서는 non-Windows일 때 Windows 전용 옵션을 숨기고, 숨긴 값 때문에 저장이 막히는 자기모순을 없앤다.
5. 서버 테스트와 수동 검증 가이드를 통해 clean install, stale config, Windows 유지 동작을 모두 회귀 검증한다.

## 성공 기준

- 새 macOS/Linux checkout에서 `server/config.json5`가 없어도 첫 실행 시 서버가 정상 부팅한다.
- 기존 `config.json5`에 Windows 전용 PTY 값이 남아 있어도 macOS/Linux 부팅이 차단되지 않는다.
- Windows는 계속 `ConPTY` 선호 기본 정책을 사용할 수 있다.
- Settings 페이지는 non-Windows에서 Windows 전용 옵션을 표시하지 않는다.
- Settings 저장은 현재 OS와 맞지 않는 숨김 PTY 값 때문에 실패하지 않는다.
- README와 example config는 실제 bootstrap/runtime 정책과 일치한다.
- Settings와 session route는 이미 런타임이 지원하던 shell 목록을 host capability 기준으로 일관되게 노출한다.

## 범위 밖

- 새로운 shell provider 또는 실행 엔진 추가
- workspace 복구 시 서로 다른 OS 간 shell migration까지 자동 변환
- `config.json5` 로드 시 자동 rewrite 또는 migration log 파일 생성
- `dev.js` 전체 launcher 구조 개편
