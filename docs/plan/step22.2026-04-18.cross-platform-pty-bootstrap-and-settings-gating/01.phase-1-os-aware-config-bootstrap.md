## Phase 1: OS-Aware Config Bootstrap

### 목표

`server/config.json5`가 없을 때 현재 운영체제에 맞는 PTY 기본값으로 파일이 생성되도록 bootstrap contract를 확립한다.

## 선행 조건

- 없음

## 기존 코드 재사용 힌트

- `server/src/utils/config.ts`
  - `ensureConfigExists()` 흐름을 그대로 bootstrap 진입점으로 사용한다.
- `server/src/schemas/config.schema.ts`
  - 플랫폼 중립 기본값과 validation contract를 기준선으로 재사용한다.
- `server/config.json5.example`
  - 두 번째 템플릿 파일을 만들지 말고 단일 예제 파일 또는 단일 source template를 유지한다.

## 코드 중복 주의

- IMPL-001 / IMPL-007
  - OS별 템플릿 파일을 따로 두지 않는다.
- IMPL-001
  - Windows용 bootstrap 기본값을 `SessionManager`와 별도로 하드코딩하지 말고 추후 정책 helper와 연결 가능한 형태로 둔다.

## 구현 체크리스트

- [ ] Phase 1-3: `server/config.json5.example`를 플랫폼 중립 예제로 정리한다 `IMPL-001`, `IMPL-007`

## 언어+버전 주의사항

- TypeScript 5.x + Node.js ESM 환경이다.
- `config.ts`는 파일 생성과 암호화 보조 로직도 함께 가진다. bootstrap 변경이 password encryption 경로를 깨지 않게 해야 한다.

## LLM 코딩 주의사항

- Simplicity: 새 config 렌더러를 도입하지 말고 기존 bootstrap 흐름에 최소 변경으로 붙인다.
- Surgical: `config.json5.example`를 수정하더라도 다른 섹션 구조와 주석은 최대한 보존한다.

## 테스트 시나리오

### 정상 케이스

- Given: `server/config.json5`가 없다
  When: Windows에서 서버 bootstrap이 수행된다
  Then: 생성된 config는 Windows용 PTY 기본값을 가진다

- Given: `server/config.json5`가 없다
  When: macOS/Linux에서 서버 bootstrap이 수행된다
  Then: 생성된 config는 non-Windows용 PTY 기본값을 가진다

### 예외 케이스

- Given: `config.json5.example`는 존재하지만 일부 PTY field가 누락됐다
  When: bootstrap이 수행된다
  Then: 생성 결과는 스키마 기본값과 bootstrap 정책을 조합해 유효 config를 만든다

### 경계값 테스트

- Given: password encryption이 필요한 auth config가 함께 있다
  When: missing config bootstrap 후 load가 이어진다
  Then: bootstrap PTY 변경이 기존 password 처리와 충돌하지 않는다
