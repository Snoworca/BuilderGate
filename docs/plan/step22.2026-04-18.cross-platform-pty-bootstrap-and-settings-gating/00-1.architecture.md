# Architecture

## 1. 현재 구조 요약

현재 PTY 정책은 다음처럼 분산되어 있다.

1. `server/src/utils/config.ts`가 `config.json5`가 없으면 `config.json5.example`을 그대로 복사한다.
2. `server/src/index.ts`는 부팅 초기에 `sessionManager.assertRuntimePtyCapabilities()`를 호출한다.
3. `server/src/services/SessionManager.ts`는 non-Windows에서 `useConpty=true` 또는 `windowsPowerShellBackend !== 'inherit'`면 즉시 예외를 던진다.
4. `server/src/services/RuntimeConfigStore.ts`와 `SettingsService.ts`는 Settings capability와 validation을 별도로 들고 있다.
5. `frontend/src/components/Settings/SettingsPage.tsx`는 capability를 기준으로 일부 필드를 숨기지만, 저장 payload와 backend validation 계약은 완전히 일치하지 않는다.

그 결과:

- 저장소 기본 설정이 Windows 편향이면 macOS/Linux 부팅이 바로 깨진다.
- 기존 stale config를 가진 사용자는 예제 파일을 고쳐도 계속 부팅 실패를 겪을 수 있다.
- Settings UI와 backend validation 사이에 미묘한 자기모순이 남아 있다.

## 2. 목표 구조

### 2.1 최초 생성 경로

```text
boot
  -> ensureConfigExists()
    -> source example/template load
    -> applyPlatformBootstrapDefaults(platform)
    -> write config.json5
```

핵심:

- 저장소 공용 예제는 중립적이어야 한다.
- 실제 생성되는 `config.json5`만 OS 기준으로 PTY 기본값을 받는다.

### 2.2 기존 파일 로드 경로

```text
loadConfig()
  -> read config.json5
  -> JSON5 parse
  -> normalizePtyConfigForPlatform(raw, platform)
  -> configSchema.parse()
  -> runtimeConfigStore / sessionManager / settingsService
```

핵심:

- stale Windows PTY 값이 non-Windows 부팅을 더 이상 깨면 안 된다.
- normalization은 우선 메모리 기준으로만 적용하고 자동 rewrite는 하지 않는다.

### 2.3 Settings 노출/저장 경로

```text
GET /api/settings
  -> buildFieldCapabilities(platform, policy)
  -> frontend hides unsupported fields

PATCH /api/settings
  -> validatePatch(platform, policy)
  -> runtime/store/session update
```

핵심:

- capability와 validation이 같은 PTY 정책 소스를 재사용해야 한다.
- non-Windows에서는 Windows 전용 옵션이 보이지 않고, 숨겨진 값이 저장을 막지 않아야 한다.

## 3. 권장 책임 분리

| 컴포넌트 | 책임 | 비고 |
|---|---|---|
| `config.ts` | bootstrap 생성 + 파일 로드 정규화 | 최초 생성/기존 파일 모두 처리 |
| `pty platform policy helper` | 플랫폼별 허용 backend/shell/기본값 정의 | 중복 정책 제거 핵심 |
| `SessionManager` | 최종 defensive validation + spawn 해석 | 상위 normalization 실패 시에도 방어 |
| `RuntimeConfigStore` | UI에 노출할 capability snapshot 생성 | helper 재사용 |
| `SettingsService` | save validation과 apply summary | helper 재사용 |
| `SettingsPage` | capability 기반 숨김/힌트/UI state | backend contract 소비자 |

## 4. 권장 정책

### 4.1 Bootstrap 기본값

| 플랫폼 | `useConpty` | `windowsPowerShellBackend` | `shell` |
|---|---|---|---|
| Windows | `true` | `"conpty"` 또는 `"inherit"` 정책 중 선택 | `"auto"` |
| macOS/Linux | `false` | `"inherit"` | `"auto"` |

권장:

- Windows에서는 사용자가 선호한 `ConPTY` 기본 정책을 유지한다.
- non-Windows에서는 Windows PTY 전용 필드를 중립값으로 생성한다.

### 4.2 Load normalization

| 플랫폼 | 입력 | 정규화 결과 |
|---|---|---|
| non-Windows | `useConpty=true` | `false` |
| non-Windows | `windowsPowerShellBackend='conpty'/'winpty'` | `'inherit'` |
| non-Windows | `shell='powershell'/'wsl'/'cmd'` | `'auto'` |

이 정규화는 부팅 안정성 목적이다.

### 4.3 Settings capability

- non-Windows:
  - `pty.useConpty` hidden
  - `pty.windowsPowerShellBackend` hidden
  - shell options는 최소 `auto`, `bash`를 유지
- Windows:
  - `winpty` probe 실패 시 가용 옵션 축소 가능
  - 축소된 옵션과 저장 validation은 동일 계약을 공유

## 5. 실패 정책

### 5.1 missing config

- missing config는 failure가 아니라 auto-bootstrap 경로로 처리한다.
- bootstrap 후에도 invalid file이 생기지 않아야 한다.

### 5.2 stale config on non-Windows

- stale Windows PTY 값은 startup hard-fail 대신 normalization 대상으로 본다.
- 다만 완전한 shell migration까지 자동으로 해석하지는 않는다. 이번 단계는 부팅 안정성과 settings self-block 해소가 목적이다.

### 5.3 save path

- 사용자가 볼 수 없는 Windows 전용 필드가 save를 막아서는 안 된다.
- 필요하면 save payload를 변경 필드 기준으로 줄이거나, validation 허용값과 snapshot values를 정렬한다.

## 6. 테스트 구조

| 레이어 | 목표 | 권장 방식 |
|---|---|---|
| config loader 테스트 | missing/stale config bootstrap 및 normalization | `server/src/test-runner.ts` |
| SessionManager 테스트 | non-Windows defensive validation과 spawn policy | `server/src/test-runner.ts` |
| settings snapshot 테스트 | capability hidden / validation alignment | `server/src/test-runner.ts` |
| frontend 확인 | non-Windows capability에서 Windows fields 미렌더링 | 필요 시 경량 UI 검증 또는 수동 검증 |
| 문서 검증 | README/example가 실제 정책과 일치 | manual review |

## 7. 제약

| 제약 | 영향 |
|---|---|
| `config.json5.example`는 저장소 공용 파일이다 | OS 편향 값을 그대로 두면 다시 회귀한다 |
| `ConfigFileRepository`는 patch-in-place 성격이다 | 로드 시 자동 rewrite는 신중해야 한다 |
| `buildPatch()`는 현재 PTY 전체 객체를 보낸다 | 숨겨진/legacy 값 때문에 save가 꼬일 수 있다 |
| shell 지원 표면이 코드/설정/UI에서 완전히 같지 않다 | 이번 단계는 PTY bootstrap 중심으로 정렬 범위를 제한해야 한다 |

## 8. Phase-Architecture Mapping

| Phase | 구현 컴포넌트 | 결과 |
|---|---|---|
| Phase 1 | `config.ts`, `config.json5.example`, README | OS-aware bootstrap contract 확립 |
| Phase 2 | `config.ts`, 정책 helper, `SessionManager` | stale config도 안전하게 로드되는 runtime contract |
| Phase 3 | `RuntimeConfigStore`, `SettingsService`, `SettingsPage`, 관련 타입 | Settings capability/save/UI 계약 정렬 |
| Phase 4 | `test-runner`, 문서/검증 가이드 | 회귀 방지와 운영 설명 정리 |
