# BuilderGate EXE localStorage quota 조사 1/3

## 1. 요약 결론

BuilderGate EXE 실행 중 `Failed to execute 'setItem' on 'Storage': Setting the value of 'cws_auth_token' exceeded the quota`가 발생하는 직접 지점은 프런트엔드 인증 토큰 저장 경로다. 다만 현재 JWT는 약 207자 수준이라 `cws_auth_token` 자체가 quota를 채우는 원인으로 보기는 어렵다. 더 큰 원인은 동일 origin의 `terminal_snapshot_*` 값들이 누적되어 localStorage 여유 공간을 소진한 뒤, 이후 작은 인증 토큰 저장도 실패하는 구조다.

현재 `tokenStorage.setToken()`은 quota 예외를 복구하지 않고 그대로 throw한다. 반면 `TerminalView`는 단일 snapshot content 2,000,000자 제한만 있고 `terminal_snapshot_*` 전체 사용량 cap, LRU 정리, quota 실패 후 재시도 전략이 없다. 따라서 해결은 인증 저장 경로에 복구성 cleanup을 추가하고, 터미널 snapshot 저장 경로에는 사전 예산 관리와 quota 발생 시 오래된 snapshot 정리 후 1회 재시도를 넣는 방향이 적합하다.

## 2. 조사 근거

### 2.1 `cws_auth_token` 저장 경로

| 파일 | 라인 | 현재 동작 | 판단 |
|---|---:|---|---|

| `frontend/src/contexts/AuthContext.tsx` | 154-190 | 일반 로그인 성공 시 `tokenStorage.setToken(response.token, response.expiresIn)` 호출 | 예외는 catch되어 login 실패 error가 됨 |

권고: `tokenStorage.setToken()`을 quota 회복의 1차 방어선으로 만든다. 인증 토큰은 작은 값이므로 여기서 `terminal_snapshot_*`만 정리하고 재시도하면 사용자가 로그인 화면에 갇히는 문제를 줄일 수 있다.

### 2.2 JWT 크기

| 파일 | 라인 | 현재 동작 |
|---|---:|---|
| `server/src/services/AuthService.ts` | 108-126 | payload `{ sub:'admin', iat, exp, jti }`를 HS256으로 sign |
| `server/src/utils/constants.ts` | 80-88 | 기본 session 30분, 최대 24시간, JWT algorithm `HS256` |
| `server/src/routes/authRoutes.ts` | 132-176, 267-293 | login, verify, refresh 응답에 `token`과 `expiresIn` 반환 |
| `server/src/services/BootstrapSetupService.ts` | 84-90 | bootstrap password 설정 후 동일 token 발급 |

동일 payload 구조와 HS256으로 샘플을 생성하면 JWT 길이는 207자였다. `cws_auth_expires` 값은 epoch millisecond 문자열이라 일반적으로 13자 정도다. 따라서 인증 데이터 총량은 key 이름과 storage overhead를 포함해도 수백 바이트 수준이다. quota 초과 메시지에 `cws_auth_token`이 표시되는 것은 "그 key가 커서"가 아니라 "origin storage가 이미 가득 찬 뒤 그 key를 쓰다가 실패"했다는 의미로 보는 것이 맞다.

## 5. 실패 메커니즘

1. 각 터미널은 idle 2초 후, unmount, `beforeunload`, `pagehide`에서 `terminal_snapshot_${sessionId}`를 저장한다.
2. 단일 snapshot content는 최대 2,000,000자로 제한되지만, 여러 세션 snapshot 총량 제한은 없다.
3. localStorage quota가 소진된 상태에서 로그인, bootstrap, 2FA verify, refresh가 `cws_auth_token`을 쓰면 DOMException이 발생한다.
4. `AuthContext`는 이 예외를 일반 실패로 catch하지만, 정리 후 재시도하지 않으므로 사용자는 인증을 완료할 수 없다.
5. 명시적으로 tab close/restart를 하지 않은 오래된 snapshot은 남아 있을 수 있어 EXE 재실행 후에도 같은 origin quota 상태가 유지된다.

## 6. 구현 권고

### M-1: quota 판별과 snapshot 정리 유틸 추가

대상 후보: `frontend/src/utils/terminalSnapshot.ts` 또는 신규 `frontend/src/utils/storageQuota.ts`.

권고 내용:
- `isQuotaExceededError(error: unknown): boolean`을 추가한다.
- localStorage 전체 key를 순회해 `terminal_snapshot_*`와 `terminal_snapshot_remove_*`를 찾는다.
- `terminal_snapshot_*` 값은 JSON parse 후 `savedAt` 기준으로 오래된 순서를 계산한다.
- JSON 파싱 실패, schema 불일치, 빈 content, 현재 sessionId와 맞지 않는 명백한 오염 값은 정리 후보에 포함한다.
- `cws_auth_token`, `cws_auth_expires`, `active_workspace_id`, `mosaic_layout_*`, `tab_state_*`는 인증 복구 목적의 자동 정리 대상에서 제외한다.

### M-2: `tokenStorage.setToken()`에 정리 후 1회 재시도 추가

대상: `frontend/src/services/tokenStorage.ts`.

권고 내용:
- token/expires 저장을 하나의 transaction처럼 취급한다.
- 첫 저장 실패 시 `cws_auth_token`과 `cws_auth_expires`를 모두 제거해 partial auth state를 남기지 않는다.
- 실패가 quota이면 `terminal_snapshot_*` 정리 유틸을 호출하고 token/expires 저장을 1회 재시도한다.
- 재시도 실패 시 두 auth key를 다시 제거하고, 기존처럼 caller가 실패를 알 수 있도록 예외를 throw한다.
- console warning에는 정리한 snapshot 수, 정리 전후 대략 byte/char 사용량, 재시도 결과를 남긴다.

이 변경은 인증 흐름의 UX를 바꾸는 것이 아니라 기존 성공 조건을 quota 상황에서 회복하는 보강이다. 단, 복구 실패 시 오류를 숨기지 않아야 한다.

### M-3: `TerminalView` snapshot 저장에 LRU budget과 quota retry 추가

대상: `frontend/src/components/Terminal/TerminalView.tsx`, `frontend/src/utils/terminalSnapshot.ts`.

권고 내용:
- 단일 snapshot cap 2,000,000자는 localStorage 총량 대비 크다. cap을 유지하더라도 전체 `terminal_snapshot_*` budget을 별도로 둔다.
- 저장 전 현재 session snapshot을 제외한 오래된 snapshot을 budget 안에 들어올 때까지 정리한다.
- `localStorage.setItem(getTerminalSnapshotKey(sessionId), ...)`가 quota로 실패하면 가장 오래된 `terminal_snapshot_*`를 추가 삭제하고 1회 재시도한다.
- 현재 저장하려는 snapshot이 너무 크면 기존처럼 저장을 건너뛰되, "이 snapshot 때문에 auth가 막히지 않도록" 이전 snapshot을 유지할지 삭제할지 정책을 명시한다. 권고는 이전 snapshot 유지보다 aggregate budget 우선이다.
- `terminal_snapshot_remove_*` 마커는 작은 값이지만, 대응되는 session snapshot이 없거나 오래된 마커는 정리 대상에 포함한다.

### M-4: 관측 가능성 유지

권고 내용:
- quota 복구가 발생하면 `console.warn('[storageQuota] ...')`로 관측 가능하게 남긴다.
- 의미 있는 실패를 삼키지 않는다. token 저장 복구 실패는 auth flow에 실패로 노출되어야 한다.
- terminal snapshot 저장 실패는 terminal 동작 자체를 중단하지 않되, debug 로그에 quota 여부와 정리 결과를 남긴다.

## 7. 회귀 테스트 위치와 시나리오

### 7.1 인증 저장 회귀

권장 위치: `frontend/tests/e2e/auth-bootstrap.spec.ts`.

근거:
- 기존 `TC-2301`은 bootstrap 성공 후 `localStorage.getItem('cws_auth_token')`을 검증한다.
- 기존 `TC-2304`는 stale token clear를 검증한다.
- 같은 파일에 quota 회복 케이스를 추가하면 인증 저장 계약이 한 곳에 모인다.

테스트 제안:
- `TC-2306`: localStorage에 오래된 `terminal_snapshot_old`와 최신 `terminal_snapshot_new`를 심은 뒤, `Storage.prototype.setItem`을 monkey patch해 `cws_auth_token` 첫 저장에서 `QuotaExceededError`를 던지게 한다. 구현은 snapshot 정리 후 재시도해야 하며, 최종적으로 `cws_auth_token`은 저장되고 오래된 snapshot은 삭제되어야 한다.
- `TC-2307`: 정리 가능한 `terminal_snapshot_*`가 없는 상태에서 `cws_auth_token` 저장이 계속 quota 실패하면 `cws_auth_token`과 `cws_auth_expires`가 partial로 남지 않아야 한다.

실제 quota를 채우는 테스트는 브라우저/엔진별 한계가 달라 느리고 불안정하다. 회귀 테스트는 `QuotaExceededError`를 합성해 deterministic하게 검증하는 편이 적합하다.

### 7.2 터미널 snapshot quota 회귀

권장 위치: `frontend/tests/e2e/terminal-authority.spec.ts` 또는 신규 `frontend/tests/e2e/local-storage-quota.spec.ts`.

근거:
- `terminal-authority.spec.ts`는 이미 `terminal_snapshot_${sessionId}`를 직접 심고 server snapshot 우선/old session 무효화를 검증한다.
- quota만 독립적으로 다룰 경우 신규 `local-storage-quota.spec.ts`가 더 읽기 쉽다.

테스트 제안:
- `TC-LS-01`: 오래된 `terminal_snapshot_*`가 여러 개 있을 때 새 snapshot 저장 중 quota가 발생하면 가장 오래된 snapshot을 제거하고 현재 snapshot 저장을 1회 재시도한다.
- `TC-LS-02`: 손상된 snapshot JSON은 load 또는 cleanup 중 제거되고 정상 snapshot만 정렬/보존된다.
- `TC-LS-03`: `terminal_snapshot_remove_*` 마커가 오래되었거나 대응 snapshot이 없으면 cleanup 대상이 된다.

### 7.3 검증 실행

Playwright 설정은 `frontend/playwright.config.ts` 8-16에서 `baseURL: 'https://localhost:2002'`와 HTTPS 무시를 사용한다. webServer도 `node dev.js`로 2002 포트를 사용한다. AGENTS 규칙에 따라 수동 검증과 E2E는 `https://localhost:2002`를 대상으로 해야 한다.

권장 명령:

```powershell

Push-Location frontend
npx playwright test tests/e2e/auth-bootstrap.spec.ts --project="Desktop Chrome"
npx playwright test tests/e2e/terminal-authority.spec.ts --project="Desktop Chrome"
Pop-Location
```

## 8. 구현 순서

1. `terminalSnapshot.ts` 또는 storage quota 유틸에 quota 판별, snapshot 목록화, 오래된 snapshot 정리 함수를 추가한다.
2. `tokenStorage.setToken()`을 partial state 제거, quota cleanup, 1회 재시도 구조로 바꾼다.
3. `TerminalView.saveSnapshot()`과 `persistBufferedOutput()`에 같은 cleanup/retry 정책을 적용한다.
4. auth bootstrap E2E에 `cws_auth_token` quota 회복 테스트를 추가한다.
5. terminal snapshot E2E에 오래된 snapshot LRU 정리와 손상 snapshot 정리 테스트를 추가한다.
6. `npm --prefix frontend run build`와 관련 Playwright E2E를 최종 재실행한다.
7. Phase 완료 시 AGENTS 규칙에 따라 코드 리뷰 루프를 수행한다.

## 9. 리스크와 결정 필요 지점

- 전체 snapshot budget 값을 결정해야 한다. 권고는 "auth/login을 막지 않는 것"을 최우선으로 두고 보수적으로 시작하는 것이다.
- `mosaic_layout_*`와 `tab_state_*`까지 자동 정리할지는 별도 결정이 필요하다. 현재 조사 범위에서는 terminal snapshot만 자동 정리 대상으로 권고한다.
- 실제 브라우저 localStorage quota는 엔진/환경별로 다를 수 있으므로 테스트는 quota를 물리적으로 채우기보다 `QuotaExceededError` 합성 방식이 안정적이다.
- 토큰 저장 실패 메시지를 사용자에게 별도로 보여줄지 여부는 UI 변경이므로, 이번 계획에서는 console 관측성과 기존 auth error 경로 유지까지만 권고한다.
