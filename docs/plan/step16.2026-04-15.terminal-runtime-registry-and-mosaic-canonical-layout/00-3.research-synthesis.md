# 연구 취합

## 조사 축 요약

### 조사 1. 렌더 트리와 수명주기

핵심 발견:

- tab mode와 grid mode는 서로 다른 React subtree다.
- 같은 `tab.id/sessionId`라도 모드 전환 시 React는 same-instance move가 아니라 unmount/mount로 처리한다.
- `TerminalView`는 xterm 생성과 dispose를 mount/unmount 수명주기에 강하게 묶고 있다.

### 조사 2. 세션/WS/authority 계약

핵심 발견:

- 서버는 세션당 여러 subscriber를 허용하지만,
- 프런트는 같은 브라우저 안에서 세션당 handler 하나를 사실상 가정한다.
- 따라서 같은 세션을 tab/grid 양쪽에서 동시에 live consumer로 두는 설계는 현재 계약과 충돌한다.
- 안전한 방향은 "하나의 consumer를 이동시키는 구조"다.

### 조사 3. 아키텍처 후보

비교 결과:

- 현재 구조 유지 + handoff 개선: 단기 대응은 가능하지만 근본 해법이 아님
- registry/host-layer: 가장 현실적인 주 경로
- `react-mosaic` canonical layout: 장기적으로 매우 유효하지만 단독 해법은 아님
- pure DOM reattach: 가능하지만 초기 리스크가 큼

## 취합 결론

### 결론 1. 이번 문제의 본체는 `mosaic` 유무가 아니다

핵심은 누가 터미널 인스턴스를 소유하는가다.

현재는:

- tab path가 자기 터미널을 소유
- grid path가 자기 터미널을 소유

새 구조는:

- registry가 터미널을 소유
- tab/grid는 host만 제공

이어야 한다.

### 결론 2. `react-mosaic`를 버릴 필요는 없고, 오히려 더 핵심적으로 써야 한다

다만 용도가 바뀌어야 한다.

- 나쁜 사용법: tab UI를 mosaic처럼 보이게 만드는 것
- 좋은 사용법: workspace canonical layout state를 한 곳에 두는 것

### 결론 3. tab mode를 먼저 shared runtime 구조로 전환해야 한다

이유:

- blast radius가 더 작다
- 현재 tab mode는 이미 keep-alive 성격이 있다
- tab에서 안정화된 registry를 grid에 연결하는 편이 리스크가 낮다

### 결론 4. rollout은 단계적으로 해야 한다

필수 순서:

1. registry/host contract
2. tab cutover
3. grid cutover + canonical mosaic
4. lifecycle/replay hardening
5. regression/cleanup

## 계획에 반영된 설계 문장

> `react-mosaic`를 tab mode의 시각 컴포넌트로 확대하는 것이 아니라, workspace의 정규 레이아웃 모델로 승격하고, 실제 터미널 재사용은 registry/host-layer가 담당하게 만든다.
