# 터미널 런타임 레지스트리 및 Mosaic 정규 레이아웃 계획

Date: 2026-04-15  
Scope: tab/grid 모드 전환 시 터미널을 재생성하지 않고, 세션당 단일 터미널 런타임을 유지하면서 `react-mosaic`를 workspace 정규 레이아웃 모델로 승격하는 대규모 구조 변경  
Mode: plan only

## 목표

현재 tab 모드와 grid 모드는 서로 다른 렌더 트리에서 같은 세션의 터미널을 각각 소유한다. 이 구조 때문에 모드 전환은 사실상 `unmount -> subscribe -> snapshot -> ack -> live output flush` handoff가 되고, 여기서 검은 화면, duplicate replay, resize churn, TUI redraw corruption이 반복된다.

이번 계획의 목표는 다음과 같다.

- 세션당 단 하나의 live terminal runtime만 유지한다.
- tab/grid는 terminal instance를 직접 소유하지 않고 host slot만 제공한다.
- workspace 레이아웃의 정본은 `react-mosaic` tree로 통일한다.
- tab mode는 canonical tree를 파괴하지 않는 projection으로 동작한다.
- 현재 headless authority / replay token / snapshot fence 계약은 유지한다.

## 요구사항

- `BG-001` tab/grid 모드 전환은 같은 세션의 xterm runtime을 재생성하지 않아야 한다.
- `BG-002` workspace 전환, grid 진입, tab 복귀 시 검은 화면 대기 시간이 구조적으로 줄어야 한다.
- `BG-003` Codex 같은 TUI 세션에서 duplicate snapshot/replay 체감이 더 이상 모드 전환의 기본 결과가 아니어야 한다.
- `FR-001` frontend는 세션당 단일 mounted terminal consumer만 갖는다.
- `FR-002` terminal runtime과 layout host를 분리하는 registry 계층이 도입되어야 한다.
- `FR-003` tab mode는 canonical mosaic tree를 파괴하지 않는 projection이어야 한다.
- `FR-004` grid mode는 canonical mosaic tree를 직접 렌더하되, terminal JSX를 직접 소유하지 않고 host slot만 렌더해야 한다.
- `FR-005` 기존 tab bar UX는 유지되어야 한다.
- `FR-006` focus, selection, paste, fit, resize, metadata row 동작은 host 이동 후에도 유지되어야 한다.
- `FR-007` restart/delete/orphan recovery 시 registry와 workspace state가 함께 정리되어야 한다.
- `FR-008` 현재 headless authority, replay token, queued output flush 계약은 깨지지 않아야 한다.
- `FR-009` `Workspace-2` 같은 stale mosaic layout 복구 경로도 새 구조에서 유지되어야 한다.
- `FR-010` Codex `1부터 500까지 종 방향으로 출력` 수동 시나리오가 최종 검증 항목에 포함되어야 한다.

## 가정

1. 서버 authority 모델은 계속 `SessionManager` headless snapshot 기반으로 유지한다.
2. 이번 작업의 본체는 frontend 구조 변경이며, 서버는 계약 보강과 lifecycle cleanup 범위까지만 포함한다.
3. `react-mosaic`는 계속 사용한다. 단, tab UI 전체를 mosaic처럼 보이게 만드는 것은 목표가 아니다.
4. 대상 환경은 `https://localhost:2002`이다.

## 권장 실행 순서

1. terminal runtime을 view tree 밖으로 끌어내는 registry/host contract부터 고정한다.
2. tab mode를 shared host 기반으로 먼저 전환한다.
3. 같은 contract 위에서 grid mode와 canonical mosaic tree를 통합한다.
4. focus, selection, restart/delete, replay lifecycle hardening을 붙인다.
5. 자동/수동 회귀와 rollout 문서로 마감한다.

## 성공 기준

- 같은 세션이 tab/grid 전환만으로 `TerminalView` dispose/recreate 경로를 타지 않는다.
- 같은 sessionId에 대한 frontend WS handler는 한 시점에 정확히 하나만 live하다.
- grid 진입 시 검은 화면만 오래 보이는 현상이 재현되지 않는다.
- tab 복귀나 workspace 왕복이 snapshot/replay duplication의 기본 원인이 아니게 된다.
- persisted mosaic tree는 tab mode 사용 중에도 보존된다.
- Codex `1..500` 세로 출력 시나리오가 tab/grid 전환과 새로고침에서 안정적이다.

## 범위 밖

- `node-pty` 교체
- 서버 재시작 후 full runtime persistence
- `react-mosaic` 라이브러리 자체 교체
- tab UX를 grid UX와 동일하게 만드는 시각적 재설계
