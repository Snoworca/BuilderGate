# Phase 5. Regression Rollout And Cleanup

## 목적

새 구조를 ship 가능한 상태로 닫는다. 자동 회귀, 수동 Codex 검증, 관측성, old path 제거, 롤백 경계까지 확정한다.

## 범위

- 자동 테스트 확대
- 수동 시나리오 문서화
- old tab/grid direct-render path 제거
- observability 보강
- rollout/rollback 기준 문서화
- feature flag 및 persisted state rollback 규칙 확정

## Cutover Switch

이번 작업의 전환 스위치는 하나로 고정한다.

- 이름: `terminal_runtime_registry_enabled`
- 위치: frontend runtime config 또는 localStorage-backed dev flag
- 의미:
  - `false`: old direct-render path 사용
  - `true`: registry/host-layer path 사용

shadow mode나 복수 스위치는 두지 않는다. dual path는 코드 공존 상태를 뜻할 뿐, 운영 스위치는 이 플래그 하나만 사용한다.

## 대상 파일

- frontend Playwright 테스트
- server test runner
- telemetry/debug capture 관련 파일
- old rendering path 제거 대상 파일

## 구현 체크리스트

1. tab/grid 왕복, hidden workspace, refresh, restart, delete, stale layout 회귀를 자동화한다.
2. Codex 수동 시나리오를 final gate로 고정한다.
3. old direct-render path를 제거하기 전 `terminal_runtime_registry_enabled` 단일 switch를 유지한다.
4. runtime registry/host 구조를 기준으로 telemetry를 재정렬한다.
5. 다음 관측 항목을 추가한다.
   - sessionId당 live consumer 수
   - host slot 미부착 runtime 수
   - restart 후 orphan runtime 수
   - replay ack age
   - tab/grid 전환 중 runtime recreate count
   - sessionId당 subscribe count
   - host attach count
6. rollout 절차를 단계화한다.
   - enable 조건
   - disable 조건
   - 즉시 rollback 절차
   - persisted layout/localStorage 정리 규칙
   - old path 최종 제거 시점

## Enable 절차

1. `terminal_runtime_registry_enabled=true`로 올린다.
2. 브라우저 새로고침 후 `live consumer 수 == 1`, `runtime recreate count == 0`인지 확인한다.
3. 아래 수동 시나리오를 통과시킨다.
   - tab <-> grid 왕복
   - hidden workspace 복귀
   - restart/delete
   - Codex `1..500`
4. 30분 이상 실사용 관측에서 아래 임계치를 넘지 않으면 enable 유지한다.

## Disable 조건

아래 중 하나라도 발생하면 즉시 `terminal_runtime_registry_enabled=false`로 내린다.

- sessionId당 live consumer 수가 1을 초과
- mode 전환만으로 runtime recreate count가 증가
- restart 후 orphan runtime 수가 1 이상으로 남음
- replay ack age가 5초를 지속 초과
- duplicate redraw, 장시간 검은 화면, blank gap이 재현

## Rollback 절차

1. `terminal_runtime_registry_enabled=false`
2. 브라우저 새로고침
3. registry in-memory state 폐기 확인
4. 새 구조 전용 localStorage key가 있으면 제거
5. old direct-render path에서 핵심 수동 시나리오 재확인

## Persisted State 규칙

- canonical mosaic layout 저장 포맷은 이번 작업에서 강제 변경하지 않는다.
- registry state는 기본적으로 in-memory만 사용한다.
- 새 구조 전용 localStorage key를 도입하면 prefix를 `terminal_runtime_registry_`로 통일하고 rollback 시 제거한다.

## Old Path 제거 시점

- 모든 자동 테스트 green
- final-validation 체크리스트 green
- enable 상태에서 수동 Codex 시나리오 안정화 확인
- 위 조건을 만족한 뒤에만 old direct-render path를 제거한다

## 테스트

- `npm run build`
- `server/npm test`
- Playwright authority/reuse 회귀
- 수동 `codex` 시나리오

## 종료 조건

- old path 제거 후에도 핵심 시나리오가 green이다.
- 사용자 관찰 이슈였던 검은 화면, duplicate replay, grid 진입 blank terminal이 회귀하지 않는다.
- 운영 전환 기준과 rollback 기준이 명확하다.
- observability만으로 runtime recreate나 subscribe churn을 식별할 수 있다.

## 리스크

- old path 제거가 늦으면 유지비가 커지고, 너무 빠르면 회귀 원인 분리가 어려워진다.
