# 아키텍처

## 문제 프레임

현재 구조는 workspace마다 세션 목록은 공유하지만, 실제 터미널 렌더 트리는 모드별로 갈라져 있다.

현재 고위험 경로:

1. tab mode에서는 `App`이 `TerminalContainer`를 직접 렌더한다.
2. grid mode에서는 `MosaicContainer`가 `renderTerminal(tab)`로 다시 `TerminalContainer`를 렌더한다.
3. 모드 전환 시 하나의 subtree는 사라지고, 다른 subtree가 새로 뜬다.
4. 새 subtree는 `subscribe -> screen-snapshot -> ack` 경로를 다시 탄다.
5. 그 사이 queued output, resize, focus, restorePending 경계가 흔들린다.

## 목표 아키텍처

### 1. Terminal Runtime Registry

새 구조의 핵심은 `sessionId -> runtime entry` 레지스트리다.

중요 규칙:

- registry canonical key는 항상 `sessionId`다.
- `tabId -> sessionId`는 workspace state가 관리하는 별도 매핑이다.
- 같은 `tabId`라도 restart 이후 `sessionId`가 바뀌면 기존 runtime entry를 재사용하지 않는다.
- host slot은 `tabId`를 통해 entry를 찾을 수 있지만, entry 정체성은 `sessionId`가 결정한다.

runtime entry가 가져야 할 것:

- `sessionId`
- `runtimeGeneration`  
  registry-local monotonic integer. 새 runtime entry가 생성될 때마다 증가하며, restart로 `sessionId`가 바뀌면 반드시 새 `runtimeGeneration`을 부여한다.
- live `TerminalContainer`/`TerminalView` handle
- 현재 host slot id
- visibility/focus intent
- layout-independent metadata
- cleanup status

workspace state가 계속 가져야 할 것:

- `tabId`
- `tabId -> sessionId`
- `activeTabId`
- tab bar 순서
- canonical mosaic tree

레지스트리는 "터미널을 어디에 보여줄지"와 "터미널이 존재하는가"를 분리한다.

### 2. Host Slot Layer

tab/grid는 terminal JSX를 직접 렌더하지 않는다. 대신 slot을 선언한다.

slot 종류:

- active tab slot
- grid pane slot
- hidden parking slot

slot의 책임:

- host DOM ref 제공
- host rect 변경 알림
- active/focused intent 제공
- pointer/focus delegation 제공

### 3. Canonical Mosaic Layout

workspace 레이아웃의 정본은 `react-mosaic` tree다.

규칙:

- grid mode는 canonical tree를 그대로 렌더한다.
- tab mode는 canonical tree를 파괴하지 않는다.
- tab mode는 canonical tree와 별도로 "현재 어떤 tab가 전면 projection인지"만 바꾼다.

즉 tab mode는 layout reset이 아니라 view projection이다.

### 4. Single Consumer Contract

frontend는 세션당 live WS consumer를 하나만 유지한다.

따라서:

- runtime registry entry만 `subscribeSession(sessionId, handlers)`를 소유한다.
- host slot은 WS를 모른다.
- tab/grid 전환은 handler 교체가 아니라 host ownership 이동이어야 한다.
- host detach/attach는 `unsubscribe` 또는 새 `subscribe`를 의미하지 않는다.
- `screen-snapshot:ready` ack ownership도 registry entry만 가진다.
- pending replay token이 있는 동안 host 이동이 일어나도 replay pending state는 유지된다.

### 5. Server Authority Boundary

서버는 기존 역할을 유지한다.

- `SessionManager`: headless authority, snapshot cache, degraded fallback
- `WsRouter`: subscribe, replay token, queued output flush

이번 구조 변경은 서버 권위 모델을 바꾸지 않는다. 대신 frontend가 같은 세션의 consumer를 반복 생성하는 일을 줄여 서버 replay 경계를 덜 밟게 만든다.

## Restart / Lineage Transition Model

restart는 같은 tab에 새로운 session이 붙는 이벤트다.

| 상황 | tabId | sessionId | 기대 동작 |
|------|-------|-----------|-----------|
| 일반 전환 | 유지 | 유지 | 같은 runtime entry, 같은 `runtimeGeneration` 유지 |
| host 이동 | 유지 또는 미변경 | 유지 | host만 바뀌고 WS/replay state 유지 |
| tab restart | 유지 | 변경 | old runtime entry 폐기, 새 `sessionId`와 새 `runtimeGeneration`으로 재생성 |
| workspace delete | 제거 | 제거 | tab mapping과 runtime entry 모두 cleanup |
| orphan recovery | 불명 또는 제거 | 제거 | runtime entry 강제 cleanup |

이 표는 Phase 4의 lifecycle 테스트 기준이 된다.

## Tab Order And Canonical Tree Policy

tab bar 순서와 canonical mosaic tree는 동일 개념이 아니다.

- tab bar 순서의 진실원은 workspace tabs 배열이다.
- canonical mosaic tree는 split topology의 진실원이다.
- tab reorder는 tab bar 순서를 바꾸지만, 기존 split topology를 강제로 재작성하지 않는다.
- stale recovery나 unassigned tab append가 필요할 때만 현재 tab 순서를 사용해 leaf를 보충한다.
- tab mode는 canonical tree를 변경하지 않는다.

## 상태 소유권

### Workspace State

workspace state가 계속 소유해야 할 것:

- `activeWorkspaceId`
- `activeTabId`
- `viewMode`
- persisted mosaic tree
- tab metadata

### Runtime Registry

registry가 새로 소유해야 할 것:

- terminal runtime existence
- host attachment
- WS consumer ownership
- live focus target
- selection/paste/fit imperative handle

## 마이그레이션 원칙

1. 새 registry와 host slot은 기존 tab mode 옆에 병행 도입한다.
2. tab mode가 새 구조로 안정화되기 전까지 grid cutover를 시작하지 않는다.
3. grid mode cutover 전에 canonical mosaic tree와 tab projection 규칙을 확정한다.
4. old path 제거는 Phase 4 이후에만 한다.

## 주요 리스크

- host 이동 중 focus/selection이 깨질 수 있다.
- `TerminalView`를 host-rebind 가능 구조로 바꾸는 과정에서 xterm dispose 규칙이 불명확해질 수 있다.
- tab projection과 canonical tree가 따로 놀면 persisted layout이 망가질 수 있다.
- registry cleanup이 약하면 restart/delete/orphan 경로에서 zombie runtime이 남을 수 있다.

## 아키텍처 결정 요약

1. 핵심 문제는 `mosaic` 유무보다 terminal ownership이다.
2. registry/host-layer를 먼저 만든다.
3. `react-mosaic`는 canonical layout state로 승격하되, tab UI 전체를 mosaic로 치환하지는 않는다.
4. 서버 authority 계약은 유지하고, frontend handoff 횟수를 줄이는 방향으로 접근한다.
