# 기술 결정

## ADR-001 `react-mosaic`를 tab UI 자체로 사용하지 않는다

결정:

- tab mode의 시각 표현은 계속 tab bar + active terminal projection을 유지한다.
- `react-mosaic`는 canonical layout tree와 grid rendering에 사용한다.

이유:

- tab UX는 split layout UX와 다르다.
- 탭 순서, 빠른 전환, 색상, rename, shell add UX를 mosaic로 흡수하려 하면 오히려 복잡해진다.
- 문제의 본체는 tab UI가 아니라 terminal ownership 분리다.

## ADR-002 세션당 단일 frontend consumer를 유지한다

결정:

- 같은 브라우저 내에서 같은 sessionId의 live WS handler는 하나만 허용한다.

이유:

- 현재 `WebSocketContext`가 사실상 그 계약을 전제로 한다.
- 같은 세션을 tab/grid 양쪽에서 동시에 subscribe하는 구조는 이번 리팩터링의 목표가 아니다.
- 중복 subscribe는 duplicate snapshot/replay churn의 직접 원인이었다.

## ADR-003 registry/host-layer를 pure DOM reattach보다 우선한다

결정:

- 초기 구현은 registry + host slot + shared runtime 계층으로 간다.
- pure DOM reattach는 내부 구현 수단이 될 수 있으나, 최상위 설계 용어로 채택하지 않는다.

이유:

- pure DOM reattach는 리스크가 크고 디버깅이 어렵다.
- registry/host-layer는 현재 `terminalRefsMap`과 imperative handle 구조를 자연스럽게 확장한다.

## ADR-004 canonical mosaic tree는 tab mode에서 파괴하지 않는다

결정:

- tab mode 진입 시 persisted mosaic tree를 single leaf로 바꾸지 않는다.
- tab mode는 active projection만 바꾼다.

이유:

- 모드 전환이 layout reset이 되면 grid 복귀 품질이 떨어진다.
- stale recovery와 persisted tree 보존이 충돌한다.

## ADR-005 tab cutover를 grid cutover보다 먼저 한다

결정:

- Phase 1~2에서 tab mode를 shared runtime 구조로 안정화한다.
- grid는 그 다음에 같은 runtime을 host slot으로 재사용하게 전환한다.

이유:

- 현재 tab mode는 이미 absolute stack keep-alive 경향을 가지고 있다.
- 가장 작은 blast radius로 registry를 먼저 증명할 수 있다.

## ADR-006 서버는 authority model을 유지한다

결정:

- `SessionManager` headless authority와 `WsRouter` replay token 계약은 유지한다.

이유:

- 현재 불안정성의 핵심은 서버 authority 자체보다 frontend handoff 횟수와 lifecycle churn이다.
- 서버 모델까지 동시에 바꾸면 원인 분리가 어렵다.

## ADR-007 old path 제거는 회귀 검증 후에만 한다

결정:

- 새 registry/host path를 병행 도입한 뒤, 충분한 회귀 검증 후 old tab/grid direct-render path를 제거한다.

이유:

- 대규모 구조 변경이므로 롤백 여지가 필요하다.
- Codex, PowerShell, grid stale recovery, restart/delete까지 확인 후 제거해야 한다.
