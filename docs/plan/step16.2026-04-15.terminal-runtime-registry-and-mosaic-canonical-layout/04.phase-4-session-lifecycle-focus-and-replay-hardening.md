# Phase 4. Session Lifecycle Focus And Replay Hardening

## 목적

새 구조가 들어간 뒤 lifecycle과 replay 경계를 다시 닫는다. terminal runtime 재사용은 구조적으로 유리하지만, restart/delete/orphan recovery와 focus/paste/input 같은 주변 계약을 끝까지 맞춰야 한다.

## 범위

- restart/delete/orphan cleanup
- active session lineage invalidation
- focus/selection/paste/input hardening
- replay/snapshot 경계 회귀 검증

## 구현 체크리스트

1. registry cleanup이 restart/delete/orphan recovery와 연결되게 한다.
2. old session lineage와 new session lineage가 섞이지 않게 한다.
3. restart 시 `tabId`는 유지, `sessionId`는 교체, `runtimeGeneration`은 증가라는 transition model을 그대로 구현한다.
4. registry entry만 subscribe/ack ownership을 가진다는 규칙을 lifecycle 전 경로에 유지한다.
5. focus, blur, selection, paste, sendInput 경로가 host 이동 후에도 안정적으로 유지되게 한다.
6. visibility/host reassignment가 resize noise로 이어지지 않게 한다.
7. replay token, queued output flush, snapshot ack 계약이 새 구조에서도 그대로 유지됨을 검증한다.

## 테스트

- restart 후 old runtime cleanup 테스트
- deleteWorkspace/checkOrphanTabs와 registry cleanup 연계 테스트
- hidden -> visible -> grid -> tab 연속 전환 focus 회귀 테스트
- replay lineage regression 테스트
- pending replay token 도중 host 이동 또는 tab/grid 전환이 ack ownership을 깨지 않는 테스트
- restart 이후 같은 tabId에서 `runtimeGeneration`이 증가하고 old generation이 재활성화되지 않는 테스트

## 종료 조건

- lifecycle cleanup이 완전하다.
- 새 구조가 기존 replay 보호장치를 깨지 않는다.
- input/paste/focus 회귀가 없다.

## 리스크

- host 이동 중 stale focus가 남아 ghost input이 생길 수 있다.
- registry cleanup 누락 시 zombie runtime 또는 stale subscriber가 남을 수 있다.
