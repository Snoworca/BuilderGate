# Phase 1. Runtime Registry And Host Contract

## 목적

터미널 재활용의 핵심 전제인 `세션당 단일 runtime` 계약을 먼저 도입한다. 이 단계에서는 tab/grid 둘 다 완전히 갈아엎지 않고, registry와 host slot이라는 새 중간 계층을 만드는 데 집중한다.

## 범위

- terminal runtime registry 설계 및 도입
- host slot registration API 설계
- terminal imperative handle ownership 이동
- 기존 path와 병행 가능한 feature flag 또는 compatibility layer 마련

## 구현 체크리스트

1. runtime registry canonical key를 `sessionId`로 고정한다.
2. `tabId -> sessionId` 매핑은 workspace state에 남기고 registry와 분리한다.
3. runtime entry가 `sessionId`, lineageId, current host, imperative handle, focus state를 보유하게 한다.
4. registry entry만 `subscribeSession`과 `screen-snapshot:ready` ack ownership을 갖게 한다.
5. host slot 컴포넌트 또는 hook을 도입한다.
6. host attach/detach가 WS state 변화로 해석되지 않게 한다.
7. 기존 `terminalRefsMap` 사용처를 registry adapter로 우회시킨다.
8. registry가 없는 old path와 coexist 가능한 안전장치를 둔다.
9. `TerminalView`가 host 이동을 견딜 수 있도록 lifecycle 분해 계획을 반영한다.

## 테스트

- registry가 sessionId별로 단 하나의 runtime만 유지하는 단위 테스트
- host attach/detach가 중복 runtime 생성 없이 일어나는 테스트
- imperative API(`focus`, `sendInput`, `getSelection`, `fit`)가 registry 경유로 유지되는 테스트
- pending replay token이 있는 상태에서 host 이동이 subscribe/unsubscribe churn을 만들지 않는 테스트
- restart로 `tabId`는 유지되고 `sessionId`만 바뀌는 경우 old/new runtime이 섞이지 않는 테스트

## 종료 조건

- 같은 sessionId에 대해 registry entry가 하나만 생성된다.
- host slot이 바뀌어도 old path처럼 즉시 dispose/recreate하지 않는다.
- host 이동만으로 WS subscribe state가 흔들리지 않는다.
- Phase 2가 registry 기반으로 tab mode를 실제 전환할 수 있을 만큼 API가 안정된다.

## 리스크

- registry가 너무 크면 디버깅이 어려워진다.
- host ownership 경계가 모호하면 double focus, stale handle 문제가 생길 수 있다.
