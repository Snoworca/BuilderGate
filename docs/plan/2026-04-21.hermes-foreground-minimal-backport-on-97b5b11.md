# Hermes Foreground Minimal Backport Plan on `97b5b11`

작성일: 2026-04-21  
대상 브랜치: `research/apply-95820f1-on-97b5b11`  
기준 베이스: `97b5b11291eba0c01e4b6158f87c02524ec3aad8`

## 목적

`95820f165001014166f19b2b39814c388c49ad8e`의 Hermes foreground idle 설계를, 현재 안정적인 WSL/bash 부트스트랩을 유지한 채 최소 범위로 백포트한다.

이번 백포트의 1차 목표는 두 가지다.

1. `hermes` 실행 후 사용자 입력 대기 상태를 `idle`로 유지한다.
2. `e5c95ed` 계열의 WSL rcfile 회귀를 다시 들여오지 않는다.

## 현재 베이스의 핵심 문제

현재 [SessionManager.ts](../../../server/src/services/SessionManager.ts) 기준으로 상태 붕괴 지점은 두 곳이다.

- heuristic output 경로
  - non-echo output이면 즉시 `running`
  - 코드: `onData()` 내부 `isEchoOutput()` 분기
- input 경로
  - Enter 입력만으로 heuristic에서 즉시 `running`
  - 코드: `writeInput()` 내부 `if (hasEnter && data.detectionMode === 'heuristic')`

즉, 현재 모델은 `foreground app`이라는 개념이 없고, `shell prompt vs running process`의 2단 구조만 가진다.

## 절대 가져오지 말아야 할 것

### 1. `e5c95ed`의 `bash --rcfile` 전환

현재 베이스의 WSL/bash 경로는 아래 구조다.

- bash: `BASH_ENV` 기반 hook
- wsl: `wsl.exe` 기본 시작
- zsh: 별도 env 주입 없이 runtime hook injection

이 구조는 사용자가 확인한 WSL 회귀 전 상태와 맞닿아 있다. 따라서 아래 변경은 금지한다.

- `buildShellEnv()`의 `BUILDERGATE_BASH_RCFILE_MODE` 도입
- `resolveShell()`의 `--rcfile` 기반 bash/wsl 실행
- `bash-osc133.sh`의 rcfile bootstrap 체인

### 2. `760be3a`의 generic foreground fallback 전체

`760be3a`는 detector 미부착 foreground app의 repaint를 generic하게 분류하려고 시도했다. 그러나 이 층은 범위가 넓고 `Codex/Claude` 입력 중 redraw, telemetry, prompt chrome 같은 추가 문제를 끌어들였다.

이번 백포트의 목표는 Hermes만 안정화하는 것이다. 따라서 아래는 1차 범위에서 제외한다.

- generic foreground fallback
- detector 미부착 앱까지 포괄하는 repaint-only 분류
- Codex/Claude 전용 또는 범용 prompt chrome 분류

## 최소 백포트 범위

### Phase A. 상태 모델 추가

`SessionManager`에 아래 필드를 추가한다.

- `derivedState`
- `foregroundDetectorRegistry`
- `inputBuffer`
- `pendingForegroundAppHint`
- `lastSubmittedCommand`

그리고 helper를 추가한다.

- `createForegroundDetectorRegistry()`
- `ensureDerivedState()`
- `transitionToShellPrompt()`
- `beginForegroundProcess()`
- `beginForegroundActivity()`
- `syncStatusFromDerivedState()`

### Phase B. Hermes detector 연결

`onData()` 흐름을 아래 순서로 바꾼다.

1. OSC133 parse
2. shell heuristic parse
3. foreground hint 채택
4. detector inspect
5. derived state -> external status sync

단, detector가 없거나 Hermes hint가 없으면 기존 shell 중심 경로를 유지한다.

### Phase C. input path 분리

`writeInput()`에서 해야 할 일:

- input buffer 추적
- 제출 명령 추출
- `detectForegroundAppHint(command)`로 `hermes` hint 저장
- `hermes` submit이면 즉시 `running`이 아니라 `foreground_app + waiting_input`

즉, Hermes 한정으로는 “Enter 입력 = running” 규칙을 끊는다.

### Phase D. shell prompt 복귀 유지

prompt/CWD 감지는 계속 shell prompt 복귀 신호로 사용한다.

- bash / WSL bash: 기존 구조 유지
- zsh: 기존 `precmd()` CWD hook 유지
- PowerShell: 기존 prompt hook 유지

즉, 일반 셸 세션 회귀를 줄이기 위해 shell signal을 버리지 않는다.

## Hermes 검출의 cross-shell 가능성 검토

## 결론

`Hermes detector` 자체는 WSL 전용이 아니다. 직접 `hermes` 명령을 입력하는 한, `bash`, `zsh`, macOS 환경에서도 검출 가능하다.

다만 강도는 셸마다 다르다.

### 1. bash (Linux/macOS/WSL)

지원 가능성이 높다.

이유:

- `shellType='bash'`
- 현재 베이스도 `writeInput()`에서 제출 명령을 볼 수 있다
- 이후 detector 구현의 `detectForegroundAppHint(command)`는 shell-agnostic하다
- bash는 현재도 `BASH_ENV` 기반 OSC133 경로가 있어 output 관측이 상대적으로 풍부하다

따라서 아래는 그대로 잡힌다.

- `hermes`
- `/usr/local/bin/hermes`
- `~/bin/hermes`
- `env FOO=1 hermes`
- `command hermes`

### 2. zsh (macOS 포함)

지원 가능하다. 다만 bash보다 약하다.

이유:

- detector는 shell type에 강하게 의존하지 않고, command hint + Hermes output 패턴을 본다
- 현재 베이스도 zsh에서 `precmd()` 기반 CWD hook은 있다
- 사용자가 직접 `hermes`를 입력하면 `writeInput()` 기반 hint로 검출 가능하다

제약:

- 현재 베이스에는 zsh용 OSC133 integration이 없다
- 따라서 `bash`처럼 command-start marker를 풍부하게 얻기 어렵다
- history recall / command echo 기반 보조 힌트는 bash보다 약할 수 있다

즉, zsh/macOS에서의 1차 지원 범위는 아래로 정의하는 것이 안전하다.

- 직접 `hermes` 명령을 입력한 경우: 지원
- path로 직접 실행한 경우: 지원
- alias/function/wrapper로 우회 실행한 경우: 1차 범위 제외

### 3. PowerShell / cmd

이번 범위에서 Hermes detector를 막을 이유는 없다. 다만 현재 사용자 이슈의 중심은 WSL/bash/zsh 계열이므로 1차 검증 우선순위는 낮다.

## detector 규칙의 shell 의존성

`HermesForegroundDetector`의 핵심 신호는 Hermes 출력 자체다.

- welcome text
- tip text
- banner
- status frame
- CPR warning
- repaint-only ticker

이 신호들은 `bash`, `zsh`, macOS 여부보다 Hermes가 출력하는 ANSI/text 패턴에 의존한다.

즉, detector의 본질은 cross-shell로 재사용 가능하다.

shell 차이는 detector보다 “hint를 얼마나 빨리 확보하느냐”와 “shell prompt 복귀를 얼마나 강하게 잡느냐”에서 생긴다.

## 권장 구현 순서

1. `ForegroundAppDetector.ts`, `HermesForegroundDetector.ts`를 현재 브랜치에 추가
2. `SessionManager`에 `derivedState`와 Hermes hint/input buffer만 최소 이식
3. `writeInput()`에서 Hermes submit만 `waiting_input`으로 처리
4. `onData()`에서 Hermes detector observation을 derived state에 반영
5. 기존 bash/zsh/powershell shell prompt 복귀 경로는 유지
6. WSL 부트스트랩 파일은 건드리지 않음

## 테스트 전략

사용자 요구대로 “테스트가 곧 스펙”으로 잡아야 한다.

### 반드시 추가할 통합 테스트

- bash에서 `hermes` submit 직후 `idle`
- bash에서 Hermes welcome/status bootstrap 출력 후 `idle`
- bash에서 Hermes semantic output 시 `running`
- zsh에서 `hermes` submit 직후 `idle`
- zsh에서 prompt 복귀 후 다시 `shell_prompt`로 수렴
- WSL bash에서 기존 컬러/PATH 경로를 건드리지 않았음을 보장하는 회귀
- 일반 bash command는 기존대로 `running -> idle`
- PowerShell prompt redraw는 기존대로 `idle`

### 이번 단계에서 피할 테스트

- detector 클래스만 떼어낸 단독 unit test

이유:

- 사용자 요구는 detector 자체의 정교함보다 실제 세션 통합 동작 보장에 있다
- 기존에 문제가 된 것도 detector 단위가 아니라 `SessionManager` 통합 상태 전이였다

### 주의 사항

- Hermes detector를 넣더라도 기존 Hermes harness 기반 동작을 망가뜨리지 말 것
- 특히 WSL/bash의 기존 정상 동작을 위해 `bash-osc133.sh`, `buildShellEnv()`, `resolveShell()`의 현재 기동 경로를 보존할 것
