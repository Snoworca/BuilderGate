# Phase 3: Router Snapshot Handshake And Client Cutover

## Objective

Cut the realtime path over from raw history replay to authoritative `screen-snapshot` delivery with tokenized replay gates.

## Prerequisites

- Phase 2 completed
- `SessionManager.getScreenSnapshot(sessionId)` is available

## Target Files

- `server/src/ws/WsRouter.ts`
- `server/src/types/ws-protocol.ts`
- `frontend/src/types/ws-protocol.ts`
- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`

## Planned Changes

1. Replace `history` replay semantics with `screen-snapshot`.
2. Create a replay gate before snapshot delivery and attach a unique replay token.
3. Send exactly one deterministic recovery-start message per subscribe path:
   - `mode: 'authoritative'` when the server snapshot is healthy
   - `mode: 'fallback'` when the session is degraded and the browser may use the temporary fallback path
4. Queue live output per socket-session pair until the client acknowledges the matching token.
5. Reject stale acknowledgements:
   - wrong token
   - old token after a newer subscribe
   - ack after unsubscribe or socket close
6. Use a fixed replay gate policy:
   - queue cap: `min(pty.maxSnapshotBytes, 262144 characters measured with string.length)`
   - ack timeout: `5000 ms`
7. Send `screen-snapshot` before `subscribed` for any new replay cycle so `subscribed` never becomes the restore trigger.
8. Update the frontend mount flow so that:
   - server snapshot is attempted first
   - local snapshot restore is fallback-only
   - live output remains buffered until snapshot application completes
9. Keep `subscribed` as metadata only:
   - status
   - cwd
   - no implication that the session is already live

## Test

### Scenario 1: First subscribe receives authoritative snapshot before live output

Given a session with an authoritative server snapshot
When a client subscribes
Then `screen-snapshot` arrives before any live `output`

### Scenario 2: Reconnect uses a new replay token

Given a client disconnects and reconnects to the same session
When the router creates a new replay cycle
Then the new cycle uses a new token and does not accept the old token

### Scenario 3: Stale acknowledgement is ignored

Given a newer replay token has already replaced an older one
When the older token is acknowledged later
Then the router does not release the newer replay gate

### Scenario 4: Fallback path is explicit

Given the session is marked degraded or no authoritative snapshot is available
When the router sends `screen-snapshot` with `mode: 'fallback'`
Then the frontend uses the documented fallback path instead of mixing both primary and fallback restores

### Scenario 5: Boundary case for empty authoritative snapshot

Given a valid session with an empty current screen
When the router sends `screen-snapshot`
Then the client still completes the replay cycle without hanging
