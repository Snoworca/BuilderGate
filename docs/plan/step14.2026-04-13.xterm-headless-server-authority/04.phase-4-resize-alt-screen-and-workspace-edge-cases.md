# Phase 4: Resize, Alt-Screen, And Workspace Edge Cases

## Objective

Close the lifecycle gaps that usually break terminal fidelity: resize races, alternate-screen behavior, delete/restart cleanup, and workspace-driven session replacement.

## Prerequisites

- Phase 3 completed
- server snapshot handshake works end to end

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/services/WorkspaceService.ts`
- `server/src/ws/WsRouter.ts`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `server/src/test-runner.ts`

## Planned Changes

1. Finalize resize ordering:
   - update PTY size
   - update headless size
   - invalidate snapshot cache
   - keep router ordering coherent if resize happens during replay
2. Define authoritative alternate-screen behavior:
   - current active screen is what gets serialized
   - normal-screen and alt-screen are not merged artificially
3. Ensure cleanup on:
   - `deleteSession`
   - `deleteMultipleSessions`
   - `deleteTab`
   - `restartTab`
   - `deleteWorkspace`
   - socket close
   - unsubscribe
4. Document server-restart behavior:
   - `WorkspaceService.checkOrphanTabs()` recreates fresh sessions
   - no stale authoritative state crosses a full server restart in this phase
5. Ensure old session IDs cannot leak queued output or authoritative state into restarted tabs.

## Implementation Checklist

- [ ] make resize invalidate the authoritative snapshot cache

## Test

### Scenario 1: Resize before refresh restores the resized screen

Given a user resizes the terminal and then refreshes
When the server provides the next authoritative snapshot
Then the client restores the resized geometry and current content, not a stale pre-resize screen

### Scenario 2: Alt-screen reconnect preserves the current active screen

Given a TUI is running inside alternate-screen
When the browser reconnects
Then the authoritative snapshot shows the current active alt-screen view

### Scenario 3: Delete during replay cleans all state

Given a session has a replay-pending client
When the session is deleted or the workspace is deleted
Then replay gates, queued output, and headless terminal state are all cleared

### Scenario 4: Restart invalidates the old session lineage

Given a tab is restarted while an older replay token still exists
When the new session is created
Then the old session token and queued output can no longer affect the new session ID

### Scenario 5: Boundary case for rapid resize bursts

Given a drag-resize produces many size updates
When the system processes them quickly
Then the final authoritative snapshot reflects the last accepted geometry without leaking memory or replay order
