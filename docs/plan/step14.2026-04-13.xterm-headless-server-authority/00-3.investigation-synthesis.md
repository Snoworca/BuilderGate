# Investigation Synthesis

This document records the three investigation tracks that were combined into the final plan.

## Track 1: Backend Lifecycle And Authority Model

Files inspected:

- `server/src/services/SessionManager.ts`
- `server/src/services/WorkspaceService.ts`
- `server/src/services/SettingsService.ts`
- `server/src/types/config.types.ts`
- `docs/report/2026-04-13.server-side-terminal-emulation-research.md`

Findings:

- `SessionManager` is already the owner of PTY creation, PTY output handling, resize, delete, and runtime PTY config application.
- `WorkspaceService` defines the real cleanup call sites for delete, restart, and orphan recovery, so any authoritative terminal state must be cleaned there indirectly through `SessionManager`.
- the current `replayBuffer` string in `SessionManager` is tightly coupled to `pty.maxBufferSize`, which is not an appropriate long-term model for headless scrollback state.
- `SettingsService` currently treats `pty.maxBufferSize` as an immediate runtime field, so the headless migration needs an explicit config migration instead of a silent semantic swap.

Planning items:

- keep headless ownership in `SessionManager`
- define cleanup on delete, restart, workspace delete, orphan recovery, and shutdown
- define new scrollback and snapshot limits in config and settings

## Track 2: Protocol And Frontend Boundary

Files inspected:

- `server/src/ws/WsRouter.ts`
- `server/src/types/ws-protocol.ts`
- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`

Findings:

- the current router uses `history` plus `history:ready`, but replay release is keyed only by `sessionId`.
- `WebSocketContext` and `TerminalContainer` already have a replay gate shape, so the migration can reuse that structure while upgrading the contract.
- `TerminalView` still owns browser-local snapshots and can restore them before the server becomes authoritative if the contract is not changed carefully.
- the authoritative snapshot must arrive before live `output`, and the client must acknowledge completion with a value stronger than only `sessionId`.

Planning items:

- replace `history` with `screen-snapshot`
- add replay tokens and stale-ack rejection
- make server snapshot first and browser snapshot fallback-only

## Track 3: Tests, Operability, And Risk Gates

Files inspected:

- `server/src/test-runner.ts`
- `server/src/services/SessionManager.test.ts`
- `frontend/tests/e2e/*`
- `server/src/services/SettingsService.ts`
- `docs/report/2026-04-13.server-side-terminal-emulation-research.md`

Findings:

- current tests cover lightweight replay ordering, but not authoritative screen fidelity, alt-screen, or tokenized replay.
- the research report explicitly treats PoC quality for alt-screen, resize, and serialization as the first hard gate.
- rollout without observability would make it hard to diagnose degraded headless sessions, oversized snapshots, or stuck replay gates.

Planning items:

- add PoC tests before production refactor
- add unit, protocol, and E2E coverage for alt-screen, resize, delete, restart, and stale acknowledgements
- add logs or metrics for snapshot latency, snapshot bytes, replay queue size, timeout count, and degraded sessions

## Combined Conclusion

The plan must do five things together:

1. prove `@xterm/headless` can serialize the current screen acceptably enough for this project
2. move state ownership into `SessionManager`
3. change the WS contract to server-first `screen-snapshot` with replay tokens
4. update delete, restart, resize, and workspace lifecycle paths to clear authoritative state correctly
5. add enough observability and regression coverage that the migration can be rolled out safely
