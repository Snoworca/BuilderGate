# Phase 2: SessionManager Headless Lifecycle

## Objective

Move terminal state authority into `SessionManager` so each session owns a PTY plus a healthy or degraded headless terminal.

## Prerequisites

- Phase 1 passed
- final message contract and config migration direction are frozen

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/types/config.types.ts`
- `server/src/schemas/config.schema.ts`
- `server/src/services/SettingsService.ts`
- `server/src/services/RuntimeConfigStore.ts`
- `server/src/services/SessionManager.test.ts`

## Planned Changes

1. Extend `SessionData` with:
   - headless terminal instance
   - current `cols` and `rows`
   - screen sequence counter
   - cached serialized snapshot
   - health state
2. On PTY output:
   - write bytes into the headless terminal first
   - invalidate cached snapshot
   - increment the sequence counter
   - route live output through the router
3. Add `getScreenSnapshot(sessionId)` that returns:
   - `seq`
   - `cols`
   - `rows`
   - `data`
   - `truncated`
   - health metadata if needed for fallback decisions
4. Replace raw replay-limit semantics with headless settings:
   - `pty.scrollbackLines`
   - `pty.maxSnapshotBytes`
   - default apply scope:
     - `pty.scrollbackLines`: `new_sessions`
     - `pty.maxSnapshotBytes`: `immediate`
5. Define degraded-session behavior:
   - if headless write or serialize fails, mark the session degraded
   - clear cached authoritative snapshot
   - keep PTY live output working
   - expose enough state for router and frontend fallback decisions

## Test

### Scenario 1: Output updates the authoritative snapshot

Given a running session with PTY output
When `SessionManager` later serializes that session
Then the authoritative snapshot includes the latest screen state

### Scenario 2: Clean cache avoids duplicate serialization work

Given a session with no new output since the last snapshot
When `getScreenSnapshot(sessionId)` is called twice
Then the second call reuses cached snapshot data rather than rebuilding it unnecessarily

### Scenario 3: Degraded headless state stays bounded

Given a session whose headless write or serialize path throws
When `SessionManager` processes that failure
Then the session is marked degraded, cached snapshot data is cleared, and PTY output can still continue live

### Scenario 4: Runtime config migration is explicit

Given a settings patch that changes raw replay-era fields
When the new configuration model is applied
Then the result is either migrated explicitly or rejected, not silently reinterpreted

### Scenario 5: Boundary case for maximum snapshot size

Given a session whose serialized screen exceeds `pty.maxSnapshotBytes`
When `getScreenSnapshot(sessionId)` runs
Then the response is truncated or rejected according to the documented limit and never grows unbounded
