# Architecture

## System Context

Current state:

- `SessionManager` owns PTY creation and keeps a raw `replayBuffer` string in `server/src/services/SessionManager.ts`
- `WsRouter` sends `history` plus queued live `output` in `server/src/ws/WsRouter.ts`
- the browser still reconstructs visible state with local snapshot logic in `frontend/src/components/Terminal/TerminalView.tsx`

Target state:

- each session owns a PTY plus one `@xterm/headless` terminal
- the headless terminal is the authoritative screen and scrollback model
- `WsRouter` sends a serialized `screen-snapshot` from that headless state before any live output
- the browser clears and applies the authoritative snapshot, then resumes live output

## Components

1. `SessionManager`
   Responsibilities:
   - create and dispose the headless terminal with the PTY
   - write PTY output into the headless terminal
   - track current geometry, screen sequence, health, and cached serialized snapshot
   - expose `getScreenSnapshot(sessionId)` for router delivery

2. `WsRouter`
   Responsibilities:
   - create a replay gate per socket-session pair
   - emit `screen-snapshot` with a per-delivery replay token
   - queue live output until the matching `screen-snapshot:ready` arrives
   - reject stale acknowledgements and duplicate subscribe races

3. `WorkspaceService`
   Responsibilities:
   - continue owning tab and workspace CRUD
   - ensure delete, restart, and orphan-recovery paths call the new session cleanup correctly
   - never leave stale replay tokens or headless state behind when session IDs change

4. `WebSocketContext` / `TerminalContainer`
   Responsibilities:
   - route `screen-snapshot` and replay token acknowledgements
   - apply server snapshot before any local restore path
   - keep local snapshot restore as an explicit fallback only while the migration is staged

5. `TerminalView`
   Responsibilities:
   - expose `replaceWithSnapshot(data)` or equivalent clear-and-write boundary
   - keep live-output buffering while the replay gate is active
   - stop acting as the primary history store

## Proposed Data Model

Within `SessionManager`, the current `SessionData` grows from raw replay ownership to headless-state ownership:

```ts
interface SessionSnapshotCache {
  seq: number;
  cols: number;
  rows: number;
  data: string;
  truncated: boolean;
  generatedAt: number;
  dirty: boolean;
}

interface SessionData {
  session: Session;
  pty: IPty;
  headless: HeadlessTerminal;
  cols: number;
  rows: number;
  screenSeq: number;
  headlessHealth: 'healthy' | 'degraded';
  snapshotCache: SessionSnapshotCache | null;
  // existing idle/CWD fields remain
}
```

Within `WsRouter`, replay-pending state becomes tokenized:

```ts
interface ReplayGateState {
  replayToken: string;
  snapshotSeq: number;
  queuedOutput: string;
  timer: NodeJS.Timeout;
}
```

## Memory Model

- `pty.maxBufferSize` is not a sufficient long-term meaning for headless state because the new authority is no longer a raw string tail.
- The plan therefore introduces:
  - `pty.scrollbackLines`: line-based headless scrollback retention
  - `pty.maxSnapshotBytes`: maximum serialized snapshot payload
- `SessionManager` keeps only one cached serialized snapshot per session.
- `WsRouter` keeps one bounded replay queue per socket-session pair while that pair is pending.
- If a session becomes `degraded`, cached snapshot data is cleared so stale state is never replayed as authoritative.

## Snapshot And Replay Model

Recommended server message:

```ts
{
  type: 'screen-snapshot',
  sessionId: string,
  replayToken: string,
  seq: number,
  cols: number,
  rows: number,
  mode: 'authoritative' | 'fallback',
  data: string,
  truncated: boolean,
  source: 'headless'
}
```

Recommended client acknowledgement:

```ts
{
  type: 'screen-snapshot:ready',
  sessionId: string,
  replayToken: string
}
```

Why the token is required:

- `server/src/ws/WsRouter.ts` currently keys replay release only by `sessionId`
- that is sufficient for the lightweight replay plan but not for overlapping subscribe cycles
- a replay token prevents an old acknowledgement from releasing a newer snapshot gate

Why `mode` is required:

- the client must always receive one deterministic recovery-start message for a subscribed session
- `mode: 'authoritative'` means the payload is the server source of truth
- `mode: 'fallback'` means the session is degraded or cannot currently provide an authoritative snapshot, so the client may use the temporary browser fallback path before acknowledging the replay token

## Sequence Diagram

```text
Client -> WsRouter: subscribe(sessionId)
WsRouter -> WsRouter: create replay gate with replayToken
WsRouter -> SessionManager: getScreenSnapshot(sessionId)
SessionManager -> SessionManager: serialize cached or fresh headless state
SessionManager --> WsRouter: { seq, cols, rows, data, truncated, mode }
WsRouter -> Client: screen-snapshot(sessionId, replayToken, seq, ...)
WsRouter -> Client: subscribed(status, cwd, metadata)
PTY -> SessionManager: onData(output)
SessionManager -> SessionManager: headless.write(output), mark cache dirty, seq++
SessionManager -> WsRouter: routeSessionOutput(sessionId, output, seq)
WsRouter -> WsRouter: queue output while gate is pending
Client -> WsRouter: screen-snapshot:ready(sessionId, replayToken)
WsRouter -> Client: flush queued output in order
WsRouter -> WsRouter: mark pair live
```

## Resize Semantics

- `frontend/src/components/Terminal/TerminalContainer.tsx` continues to send resize intent.
- `SessionManager.resize(sessionId, cols, rows)` becomes the only place that updates:
  - PTY size
  - headless terminal size
  - cached snapshot invalidation
  - current geometry metadata
- if resize arrives while a replay gate is pending, the gate remains pending and the eventual queued output flush must reflect the resized session, not the stale pre-resize snapshot
- No raw replay truncation logic may survive as the source of truth after this migration.
- Phase 1 must explicitly verify whether headless resize plus serialization preserves wrapped content acceptably enough for this project.

## Alternate-Screen Semantics

- The authoritative snapshot is the current active screen, including alternate-screen if the application is inside it.
- The plan does not attempt to send both normal-screen scrollback and alternate-screen content simultaneously.
- Phase 1 must verify:
  - entry into alt-screen
  - redraw while inside alt-screen
  - exit back to normal screen
  - reconnect while alt-screen is active
- If headless tracking of alt-screen is not good enough, the migration must stop before production cutover.

## Session Lifecycle Integration

Creation:

- `server/src/services/SessionManager.ts#createSession`
- create PTY
- create headless terminal with matching `cols`, `rows`, and scrollback settings
- attach PTY output handler that writes into headless before routing live output

Deletion:

- `server/src/services/SessionManager.ts#deleteSession`
- kill PTY
- dispose headless terminal
- clear cached snapshot
- ask `WsRouter` to clear replay state for that session

Restart:

- `server/src/services/WorkspaceService.ts#restartTab`
- old session ID must fully dispose PTY, headless state, and replay gates before the new session is created
- the new session starts with a fresh headless terminal and fresh replay token lineage

Workspace delete:

- `server/src/services/WorkspaceService.ts#deleteWorkspace`
- all tabs in the workspace must call the same full session cleanup path

Orphan recovery:

- `server/src/services/WorkspaceService.ts#checkOrphanTabs`
- recreated sessions after server restart are fresh sessions with empty headless state
- stale browser-local snapshots from the old session ID must not be treated as authoritative

## Replay Gate Policy

- snapshot delivery is always followed by one replay gate per socket-session pair
- queue cap per replay gate:
  - `min(pty.maxSnapshotBytes, 262144 characters measured with string.length)`
- replay acknowledgement timeout:
  - `5000 ms`
- cleanup triggers:
  - matching `screen-snapshot:ready`
  - unsubscribe
  - socket close
  - session delete
  - session restart
  - timeout expiry
- timeout expiry clears the gate and queued output for that pair and requires a fresh subscribe cycle before another snapshot may be trusted

## Package Structure

The plan intentionally avoids a large class explosion. The minimum expected module impact is:

- keep ownership in `server/src/services/SessionManager.ts`
- extend `server/src/ws/WsRouter.ts`
- extend `server/src/types/ws-protocol.ts` and `frontend/src/types/ws-protocol.ts`
- add at most one focused helper if the serialization path needs isolation

## Architectural Risks

- `@xterm/headless` is experimental, so serialization behavior must be proven before refactor depth increases.
- Headless serialization can become expensive if done per subscribe without caching.
- A token-less replay gate would reintroduce stale acknowledgement races.
- Reusing `pty.maxBufferSize` without a config migration would blur raw replay and headless-state semantics.

## Architectural Decision

This migration should keep `SessionManager` as the session owner and `WsRouter` as the delivery owner, but replace raw replay strings with a headless terminal plus cached authoritative snapshots. That is the smallest architecture change that can plausibly solve refresh, reconnect, resize, and alt-screen together.
