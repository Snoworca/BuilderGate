# Xterm Headless Server Authority Plan

Date: 2026-04-13
Scope: migrate ProjectMaster from raw replay and browser-local snapshots to `@xterm/headless` as the authoritative server-side terminal state
Mode: plan only

## Goal

Make the server the owner of terminal screen state so that:

- refresh, reconnect, and workspace switching restore the actual terminal screen instead of a fragile string tail
- alternate-screen and TUI workloads survive browser lifecycle events
- resize, delete, restart, and replay ordering are handled by one consistent lifecycle model

- research report: `docs/report/2026-04-13.server-side-terminal-emulation-research.md`

## Requirements

- `BG-001` Refresh and reconnect must restore the authoritative server screen state for active sessions.
- `BG-002` Tab mode, grid mode, and multi-workspace flows must keep current behavior while changing the replay model.
- `BG-003` Server memory, replay latency, and degraded-emulator cases must remain bounded and observable.
- `FR-001` Each session must own one `@xterm/headless` instance for authoritative screen and scrollback state.
- `FR-002` `SessionManager` must create, resize, serialize, degrade, and dispose the headless terminal with the PTY lifecycle.
- `FR-003` The replay protocol must send `screen-snapshot` before live `output`.
- `FR-004` Replay acknowledgement must use a per-delivery token so stale acknowledgements cannot release a newer replay gate.
- `FR-005` Live PTY output must queue per socket-session pair while that pair is replay-pending.
- `FR-006` The frontend must clear and apply the server snapshot before resuming live output.
- `FR-007` Browser `localStorage` snapshots must become fallback-only during rollout and removable after validation.
- `FR-008` Resize must update PTY size and headless size with one coherent session contract.
- `FR-009` Alternate-screen and TUI current state must be recoverable on refresh and reconnect.
- `FR-010` Session delete, tab restart, workspace delete, socket close, unsubscribe, and orphan-session recovery must clear all replay and emulator state.
- `FR-011` The runtime settings model must replace the raw character-buffer assumption with explicit headless scrollback and snapshot limits.
- `FR-012` Unit, integration, and E2E coverage must prove ordering, resize, alt-screen, cleanup, and fallback behavior.

## Assumptions

1. Full persistence of terminal screen state across server process restart is still out of scope for this migration.
2. The browser will remain an `@xterm/xterm` renderer, not the source of truth.
3. `@xterm/headless` is accepted as an experimental dependency, but only after Phase 1 proves a workable serialization path.
4. A temporary rollout fallback is acceptable, but the steady-state target is server-authoritative replay.

## Recommended Execution Order

1. Prove the headless serialization path and freeze the protocol.
2. Move session lifecycle ownership into `SessionManager`.
3. Cut over the router and frontend to `screen-snapshot` with replay tokens.
4. Finish lifecycle edge cases: resize, alt-screen, delete, restart, orphan recovery.
5. Close with regression, observability, rollout gates, and cleanup of legacy paths.

## Success Criteria

- Refresh of a live terminal restores the current server screen, not just a recent text tail.
- A reconnect during Codex-style or full-screen terminal output keeps the current view stable.
- No replay gate can be released by a stale acknowledgement from an older subscribe cycle.
- Resize before refresh produces a snapshot that matches the new server-side geometry.
- Session delete, restart, and workspace delete do not leak headless terminals or replay-pending buffers.
- Memory cost remains bounded by explicit scrollback and snapshot limits rather than accidental string growth.

## Out Of Scope

- cross-process or multi-node terminal state replication
- durable terminal state persistence across full server restart
- redesigning unrelated terminal UI
- replacing `node-pty` itself
