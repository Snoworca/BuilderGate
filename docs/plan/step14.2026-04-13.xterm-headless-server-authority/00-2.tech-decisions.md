# Technical Decisions

## ADR-001: Use `@xterm/headless` as the first migration target

Decision:

- adopt `@xterm/headless` for server-side screen authority

Why:

- the frontend already uses the xterm family
- the research report recommends the same parser family to reduce mismatch risk
- the current stack is Node.js and TypeScript, not a native sidecar stack

Not chosen:

- keep raw replay as the long-term fix
- move directly to `libvterm`

## ADR-002: Keep `node-pty` as transport, not as the state model

Decision:

- `node-pty` remains the PTY process and byte transport
- headless xterm becomes the state interpreter and snapshot source

Why:

- `node-pty` does not solve scrollback, wrap, cursor, erase, or alt-screen state by itself
- the current code already routes PTY bytes through `SessionManager`

## ADR-003: Replace `history` semantics with `screen-snapshot` semantics

Decision:

- the authoritative server payload should be named for what it is: a screen snapshot, not a raw text history tail

Why:

- `server/src/ws/WsRouter.ts` currently sends `history`, but the target is no longer raw replay
- the new contract must make server authority explicit and reduce ambiguity during rollout

Migration note:

- one deploy pair may temporarily support both message names internally
- the steady-state target is `screen-snapshot` plus `screen-snapshot:ready`

## ADR-004: Use replay tokens, not only `sessionId`, for replay acknowledgement

Decision:

- every authoritative snapshot delivery gets a unique replay token
- replay gates use fixed bounded defaults until evidence justifies new settings:
  - queue cap: `min(pty.maxSnapshotBytes, 262144 characters)`
  - ack timeout: `5000 ms`

Why:

- stale acknowledgements must not release newer replay gates
- duplicate subscribe, reconnect races, and StrictMode remount patterns are easier to reason about with explicit tokens

## ADR-005: Replace raw buffer configuration with explicit headless-state configuration

Decision:

- introduce `pty.scrollbackLines`
- introduce `pty.maxSnapshotBytes`
- deprecate `pty.maxBufferSize` instead of silently reusing it with a different meaning

Why:

- raw character tails and headless scrollback are different resources
- current settings in `server/src/services/SettingsService.ts` assume a raw string buffer

Planning requirement:

- define migration and apply-scope rules in the settings pipeline
- default apply scope:
  - `pty.scrollbackLines`: `new_sessions` unless Phase 1 proves safe runtime mutation for active headless terminals
  - `pty.maxSnapshotBytes`: `immediate`

## ADR-006: Keep browser-local snapshots as temporary fallback only

Decision:

- browser-local snapshots remain allowed only as a temporary fallback when:
  - the server is still on the old path during rollout, or
  - a session is explicitly marked `degraded`

Why:

- `frontend/src/components/Terminal/TerminalView.tsx` currently stores snapshots for recovery
- removing that path in the same phase as the server cutover increases rollback risk

Steady-state target:

- remove local snapshots from the primary recovery path after validation

## ADR-007: Prefer cached serialization over per-subscribe full serialization

Decision:

- `SessionManager` caches the last serialized headless snapshot and invalidates it on output and resize

Why:

- multiple viewers or fast reconnects would otherwise serialize the same screen repeatedly
- caching keeps the expensive step under one owner

## ADR-008: Stop the migration if PoC disproves alt-screen or resize quality

Decision:

- Phase 1 is a hard gate, not a soft recommendation

Why:

- the entire migration exists to solve state fidelity problems
- if headless serialization cannot preserve the current screen well enough, the project should not commit to a deeper refactor
