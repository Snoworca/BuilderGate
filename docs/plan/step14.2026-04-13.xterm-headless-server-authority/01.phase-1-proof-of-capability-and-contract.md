# Phase 1: Proof Of Capability And Contract

## Objective

Prove that `@xterm/headless` can provide a usable serialized current-screen snapshot in this codebase before invasive lifecycle refactors begin.

## Prerequisites

- read `docs/report/2026-04-13.server-side-terminal-emulation-research.md`
- confirm current replay and snapshot behavior in `SessionManager`, `WsRouter`, and `TerminalView`

## Target Files

- `server/package.json`
- `server/src/services/SessionManager.ts`
- `server/src/types/ws-protocol.ts`
- `frontend/src/types/ws-protocol.ts`
- `server/src/test-runner.ts`
- `server/src/services/SessionManager.test.ts`

## Planned Changes

1. Add `@xterm/headless` and the serializer dependency path required for Node-side snapshot generation.
2. Create a minimal production-adjacent spike inside the server test surface that proves:
   - headless terminal creation
   - PTY-like output writes
   - resize
   - alternate-screen entry and exit
   - serialization of the current screen
3. Freeze the target WS contract:
   - `screen-snapshot`
   - `screen-snapshot:ready`
   - replay token
4. Define the config migration from raw replay limits to headless scrollback and snapshot limits.
5. Decide whether serialization is synchronous enough to remain inside the subscribe path without a worker or async lock.

## Test

### Scenario 1: Snapshot of a normal shell screen is serializable

Given a headless terminal with ordinary shell output
When the server serializes the current screen
Then the result is non-empty, deterministic, and reusable as a replay payload

### Scenario 2: Resize updates the serialized snapshot

Given a wrapped line in the headless terminal
When the server resizes the terminal and serializes again
Then the snapshot reflects the new geometry instead of the old wrap layout

### Scenario 3: Alternate-screen entry is observable

Given an application that enters alt-screen
When the headless terminal receives the escape sequence
Then the current-screen snapshot represents the active alt-screen, not stale normal-screen tail content

### Scenario 4: Unsupported serializer path stops the migration

Given the serializer path cannot run in Node or produces unusable output
When Phase 1 validation runs
Then the migration does not proceed to lifecycle refactors

### Scenario 5: Boundary case for empty current screen

Given a newly created headless terminal with no output
When the server serializes it
Then the result is valid and does not crash the future snapshot replay path
