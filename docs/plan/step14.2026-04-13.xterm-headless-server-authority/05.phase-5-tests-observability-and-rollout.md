## Objective

Prove the migration is safe enough to ship and provide the minimum observability required to debug headless-session failures in production.

5. Publish a completion report with known limitations and rollback notes.

## Test

### Scenario 1: Refresh after long output restores the authoritative screen

Given a long-running terminal with deep scrollback
When the user refreshes
Then the current server screen is restored without collapsing to only recent lines

### Scenario 2: Hidden workspace reconnect remains correct

Given one workspace is hidden while another is active
When the WebSocket reconnects
Then both visible and hidden subscribed sessions recover through the server-authoritative path

### Scenario 3: Degraded session is observable

Given the headless path fails for one session
When the system falls back
Then logs or counters clearly show the degraded session and replay mode

### Scenario 4: Kill switch rollback is clean

Given the authoritative path must be disabled temporarily
When the kill switch is applied
Then the system rolls back to the documented fallback behavior without half-enabled protocol state

### Scenario 5: Boundary case for many concurrent replay-pending sessions

Given many clients subscribe to many sessions at once
When replay gates are active concurrently
Then queue growth remains bounded and observable

### Scenario 6: Codex-driven long vertical output survives workspace switches

Given a session terminal launches `codex`
When the operator enters the Korean prompt `1부터 300까지 종 방향으로 출력하시오`
And a second workspace is opened and used while the first workspace becomes hidden
And the user switches back and forth between those workspaces
Then the Codex session must retain its scrollback and current visible position
And the hidden workspace terminal must not collapse to only recent lines
And the browser refresh path must recover the same current screen through the server-authoritative snapshot path

## Required E2E Procedure

The migration is not considered validated unless at least one E2E or manual validation run uses this exact style of workload:

1. Open a terminal session and launch `codex`.
2. Submit the prompt:
   `1부터 300까지 종 방향으로 출력하시오`
3. Wait until the terminal visibly contains a long vertical list that extends beyond the current viewport.
4. Open another workspace and interact with a different terminal there.
5. Switch back to the `codex` workspace and verify:
   - previously printed lines are still scrollable
   - the terminal has not collapsed to only a recent tail
6. Refresh while the `codex` workspace is active and verify the same screen is restored.
7. Repeat the workspace switch after refresh and verify the scrollback remains intact.
