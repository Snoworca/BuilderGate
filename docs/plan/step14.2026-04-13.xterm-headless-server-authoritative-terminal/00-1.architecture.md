# Architecture

## 시스템 컨텍스트

현재 구조는 `node-pty -> SessionManager -> WsRouter -> browser xterm` 이다.  
Step 14 의 목표 구조는 `node-pty -> @xterm/headless -> authoritative snapshot + live output -> browser xterm` 이다.

```text
Shell Process
  -> node-pty
    -> SessionRuntime
      -> HeadlessTerminalRuntime (@xterm/headless)
      -> SnapshotSerializer
      -> WsRouter
        -> screen-snapshot / live output
          -> TerminalContainer
            -> TerminalView (@xterm/xterm)
```

## 현재 코드 기준 제약

### 서버

- `SessionData` 에는 문자열 tail `replayBuffer` 만 있다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [26-31]
- PTY output 은 `appendReplayOutput` 후 즉시 WS 로 브로드캐스트된다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [224-229]
- resize 는 PTY 만 resize 하고 끝난다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [319-324]
- subscribe 는 `history` 와 `history:ready` barrier 를 사용하지만, payload 는 화면 상태가 아니라 문자열 replay 다. [WsRouter.ts](../../../server/src/ws/WsRouter.ts) [154-166], [188-195]

### 프런트

- `TerminalView` 는 local snapshot 저장/복구를 유지한다. [TerminalView.tsx](../../../frontend/src/components/Terminal/TerminalView.tsx) [165-190], [281-301]
- `TerminalContainer` 는 `history` 가 안 오면 브라우저 snapshot 을 적용한다. [TerminalContainer.tsx](../../../frontend/src/components/Terminal/TerminalContainer.tsx) [69-82]
- `WebSocketContext` 는 session별 handlers 를 라우팅하지만, snapshot lifecycle 전용 상태는 없다. [WebSocketContext.tsx](../../../frontend/src/contexts/WebSocketContext.tsx) [22-29], [131-150]

## 목표 컴포넌트

### 1. `HeadlessTerminalRuntime`

세션당 하나씩 유지되는 서버 권위 터미널 상태 객체.

필드:

- `terminal`: `@xterm/headless` 의 `Terminal`
- `serializer`: Node-safe serializer 경로
- `cols`, `rows`
- `mode`: `off | shadow | primary | degraded`
- `snapshotCache`
- `snapshotDirty`
- `metrics`
- `geometryLease`

역할:

- PTY 출력 적용
- resize 적용
- snapshot 생성
- alt-screen/TUI 상태 보존
- serialize 실패 시 degraded 전환

### 2. `SessionRuntime`

기존 `SessionData` 를 확장한 서버 세션 런타임.

추가 필드:

- `headlessRuntime`
- `screenMode`
- `snapshotRevision`
- `lastSnapshotAt`
- `lastSerializeDurationMs`
- `lastSnapshotBytes`
- `degradedReason`

### 3. `SnapshotTransport`

현재 `history` 중심 프로토콜을 `screen-snapshot` 중심으로 전환한다.

새 메시지:

- `screen-snapshot`
- `screen-ready`
- `screen-state`
- `screen:degraded`

유지 메시지:

- `output`
- `status`
- `cwd`
- `session:error`

### 4. `GeometryLeaseManager`

동일 세션에 여러 subscriber 가 붙는 경우, 누가 PTY/headless resize 권한을 갖는지 결정한다.

규칙:

- 활성 포커스 클라이언트가 우선 owner
- owner 가 없으면 visible subscriber 중 최신 heartbeat/interaction client 가 owner
- passive viewer 는 viewport fit 만 하고 PTY resize 는 보내지 않음
- owner disconnect 시 lease 이전

## 상태 모델

### 세션 상태

```text
created
  -> shadow-warming
  -> primary-ready
  -> degraded
  -> terminated
```

### 소켓-세션 쌍 상태

```text
detached
  -> subscribing
  -> snapshot-pending
  -> live
  -> detached
```

규칙:

- `snapshot-pending` 동안 live `output` 은 queue 하되 bounded 해야 한다.
- `screen-ready` ACK 없이는 live flush 금지.
- ACK timeout 시 해당 pair 는 degraded reconnect path 로 떨어지고 queue 를 비운다.

## 시퀀스 다이어그램

### A. 정상 출력

```text
PTY output
  -> SessionManager receives raw chunk
  -> HeadlessTerminalRuntime.write(raw chunk)
  -> snapshotDirty = true
  -> WsRouter.routeSessionOutput(output)
  -> live subscribers receive output
```

### B. 새 subscribe / reconnect

```text
client subscribe
  -> WsRouter resolves session + replay state
  -> HeadlessTerminalRuntime.getSnapshot()
  -> send screen-snapshot
  -> queue live output for that ws/session pair
  -> client replaces terminal with snapshot
  -> client sends screen-ready
  -> WsRouter flushes queued live output
```

### C. resize owner 전환

```text
viewer B becomes active
  -> GeometryLeaseManager transfers owner from A to B
  -> next resize from B updates PTY + headless terminal
  -> passive viewers only fit local viewport
```

## 성능 envelope

이 문서의 수치는 구현 전 acceptance budget 이다.

- 단일 세션 `10k` scrollback snapshot serialize p95: `<= 150ms`
- reconnect 후 `screen-ready` 직전까지 p95: `<= 500ms`
- idle session 추가 RSS 목표: `<= 6MB`
- active session 추가 RSS 목표: `<= 10MB`
- `25` 동시 세션에서 event loop lag p95: `<= 50ms`
- ACK timeout rate: `<= 0.1%`

## 아키텍처 제약

- 브라우저 local snapshot 은 권위 경로에서 제거한다.
- shadow 모드에서는 headless state 를 생성하되 사용자 복구 경로는 바꾸지 않는다.
- `SessionManager` 는 PTY 와 headless state 둘 다 소유하지만, transport 세부는 `WsRouter` 가 맡는다.
- 공개 `/health` 엔드포인트에는 민감한 세션 데이터 노출 금지. 운영 메트릭은 별도 보호 경로로 제공한다.

## 핵심 아키텍처 리스크

1. Node-safe serializer 경로가 실제로 안정적이지 않을 수 있다.
2. alt-screen/TUI 가 raw output 과 snapshot 간에 다르게 보일 수 있다.
3. 다중 subscriber resize ownership 을 명시하지 않으면 화면 점프가 계속 발생한다.
4. headless snapshot serialize 비용이 예상보다 크면 shadow 단계에서 즉시 드러나야 한다.

## 아키텍처 승인 조건

- PoC 단계에서 `@xterm/headless` + serializer feasibility 가 확정된다.
- geometry lease 와 ACK barrier 가 문서상 모호하지 않다.
- degraded mode 에서 live output 손실 없이 복구 기능만 줄어드는 경로가 있다.
