# Tech Decisions

## ADR-1401: 서버 권위 상태 엔진으로 `@xterm/headless` 를 채택한다

결정:

- 1차 구현은 `@xterm/headless` 를 기준으로 간다.

근거:

- 현재 프런트가 이미 `@xterm/xterm@^6.0.0` 을 사용한다. [frontend/package.json](../../../frontend/package.json) [12-25]
- `@xterm/headless` 는 Node.js 에서 실행 가능한 headless terminal 이고, 원격 서버의 terminal state 추적 용도를 공식 문서가 직접 언급한다.
- xterm 공식 문서는 같은 계열 parser 를 공유하므로 VT/CSI/OSC 해석 mismatch 를 줄일 수 있다.

제약:

- 공식 README 는 package 자체를 experimental 로 표시한다.
- headless addon 은 Node-safe 여야 하며 DOM API 를 쓰면 안 된다.

보완:

- Step 14 는 곧바로 primary cutover 하지 않고 `off -> shadow -> primary` 3단계로 간다.

## ADR-1402: snapshot payload 는 "문자열 tail" 이 아니라 "권위 screen snapshot" 이다

결정:

- 기존 `history` replay 를 대체하는 새로운 `screen-snapshot` 프로토콜을 도입한다.

근거:

- 현재 `replayBuffer` 는 화면 상태가 아니라 tail 문자열이다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [26-31]
- 현재 alt-screen enable/disable 은 replay 경로에서 제거되고 있다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [58-60]
- 새로고침 후 "한참 밑에서 시작" 문제는 브라우저 snapshot 과 live output 이 섞이면서 개행이 중복되는 구조에서 나왔다. [TerminalView.tsx](../../../frontend/src/components/Terminal/TerminalView.tsx) [173-183], [252-257]

## ADR-1403: 브라우저 local snapshot 은 보조 수단이며, 최종적으로 제거 대상이다

결정:

- `primary` 모드에서 브라우저 local snapshot restore 는 기본적으로 비활성화한다.
- shadow/off 모드에서만 fallback 또는 비교용으로 제한한다.

근거:

- 현재 `TerminalContainer` 는 history 가 없을 때 local snapshot 을 적용한다. [TerminalContainer.tsx](../../../frontend/src/components/Terminal/TerminalContainer.tsx) [69-82]
- 이 경로가 current canonical source 를 흐리게 만든다.

## ADR-1404: geometry ownership 을 명시적으로 도입한다

결정:

- 세션마다 하나의 geometry lease owner 를 둔다.

근거:

- 현재 WS 계층은 동일 세션에 여러 subscriber 를 허용한다. [WsRouter.ts](../../../server/src/ws/WsRouter.ts) [23-24], [142-145]
- 현재 resize 메시지는 누가 보냈는지와 무관하게 PTY 에 바로 반영된다. [WsRouter.ts](../../../server/src/ws/WsRouter.ts) [201-203]
- headless state 도입 후 이 모호성은 더 큰 화면 불일치로 이어진다.

## ADR-1405: serializer feasibility 를 1차 게이트로 둔다

결정:

- Node-side 에서 `@xterm/addon-serialize` 또는 동등 경로가 안정적으로 작동하는지 PoC 를 Phase 1 종료 조건으로 둔다.

근거:

- 프런트는 이미 `@xterm/addon-serialize@^0.14.0` 을 사용한다. [frontend/package.json](../../../frontend/package.json) [13-15]
- headless README 는 addon 이 Node-safe 여야 한다고 명시한다.

Fallback:

- serializer 가 부적합하면 headless buffer introspection + custom serializer 대안을 별도 spike 로 분기한다.
- 이 경우 Step 14 전체 primary cutover 는 보류한다.

## ADR-1406: 장애 시 degraded mode 로 떨어지고 PTY 자체는 계속 살아 있어야 한다

결정:

- headless write 또는 snapshot serialize 가 실패해도 PTY live output 은 중단하지 않는다.
- 세션은 `degraded` 로 마킹하고, reconnect 복원만 fallback 경로를 탄다.

근거:

- 사용자는 "세션이 끊기지 않는 것" 을 우선 기대한다.
- 운영 리스크는 복구 품질 저하보다 세션 강제 종료가 더 크다.

## ADR-1407: 관측성이 준비되기 전에는 `primary` 기본 전환 금지

결정:

- 최소 메트릭, 로그, timeout 알림이 없으면 primary rollout 금지.

필수 메트릭:

- active headless sessions
- degraded sessions
- snapshot serialize latency
- snapshot bytes
- queued live bytes during ACK barrier
- `screen-ready` timeout count
- geometry lease transfer count

## ADR-1408: rollout 은 session-level feature flag 로 제어한다

결정:

- 전역 설정 + 세션별 override 를 모두 둔다.

상태:

- `off`: 기존 경로만 사용
- `shadow`: headless 업데이트 + 측정만 수행, 사용자 복구는 기존 경로
- `primary`: `screen-snapshot` 이 권위 경로

이유:

- 실시간 세션 제품에서 한 번에 전환하면 rollback 비용이 크다.

## ADR-1409: 테스트 전략은 4계층으로 분리한다

결정:

- 단위: server runtime, serializer, protocol
- 통합: `SessionManager + WsRouter + fake ws + fake pty`
- E2E: refresh/reconnect/workspace/grid/TUI smoke
- benchmark/ops drill: latency, memory, rollback

근거:

- 현재 `server/src/test-runner.ts` 는 빠른 단위/통합 harness 를 이미 제공한다. [test-runner.ts](../../../server/src/test-runner.ts) [25-39]
- 현재 Playwright 자산은 로그인, pane persistence, terminal visible 확인 정도는 재사용 가능하다. [helpers.ts](../../../frontend/tests/e2e/helpers.ts) [4-16]

## ADR-1410: 운영 메트릭은 public health 와 분리한다

결정:

- `/health` 는 계속 단순 공개 health 로 유지한다.
- 터미널 메트릭은 인증된 내부 경로로 분리한다.

근거:

- 현재 `/health` 는 공개 경로다. [index.ts](../../../server/src/index.ts) [142-149]
- 세션별 상태, snapshot bytes, degraded reason 은 공개 health 에 실으면 안 된다.
