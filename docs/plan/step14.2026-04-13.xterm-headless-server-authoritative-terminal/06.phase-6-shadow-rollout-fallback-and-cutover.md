# Phase 6: Shadow Rollout, Fallback, Cutover

## Objective

실사용 세션을 깨지 않으면서 `off -> shadow -> primary` 순서로 안전하게 전환하고, 이상 시 즉시 rollback 가능하도록 만든다.

## Prerequisites

- Phase 5 observability 준비 완료
- shadow 기준 mismatch / latency / memory 리포트 확보

## Architecture Ref

- 현재 replay 경로가 이미 WS barrier 를 가지고 있다는 점 [WsRouter.ts](../../../server/src/ws/WsRouter.ts) [154-195]
- 현재 브라우저 snapshot fallback 구조 [TerminalContainer.tsx](../../../frontend/src/components/Terminal/TerminalContainer.tsx) [69-82]

## Rollout 단계

1. `off`
   - 기존 경로만 사용
2. `shadow`
   - headless state 구축, metrics 수집, 사용자 복구는 기존 경로 유지
3. `primary` for canary
   - 일부 세션만 `screen-snapshot` 적용
4. `primary` default
   - 신규 세션 기본 전환
5. cleanup
   - 구형 replay/local snapshot 권위 경로 제거

## Rollback 원칙

- flag 전환만으로 복구 가능해야 한다.
- rollback 은 PTY 세션 자체를 죽이지 않아야 한다.
- rollback 후에도 live output, input, resize 는 그대로 동작해야 한다.

## Test

### 정상 시나리오 1

Given shadow 모드에서 mismatch rate 와 latency 가 기준 이하일 때  
When canary 세션에 primary 를 켜면  
Then refresh/reconnect 복구가 서버 snapshot 기준으로 동작해야 한다

### 정상 시나리오 2

Given primary 세션이 정상 동작 중일 때  
When 운영자가 flag 를 shadow 로 내리면  
Then 새 reconnect 부터 구형 경로로 즉시 돌아가야 한다

### 예외 시나리오 1

Given primary rollout 직후 ACK timeout rate 가 급증할 때  
When rollback 을 실행하면  
Then 세션 종료 없이 복구 경로만 내려가야 한다

### 예외 시나리오 2

Given 특정 shell 조합에서만 TUI mismatch 가 반복될 때  
When 세션 모드를 override 하면  
Then 해당 shell 또는 세션만 `off` 로 격리 가능해야 한다

### 경계값 시나리오

Given primary default 직전 shadow 데이터가 임계치 근처일 때  
When 운영 승인 여부를 판단하면  
Then mismatch, latency, memory 세 지표가 모두 기준 이하여야 한다
