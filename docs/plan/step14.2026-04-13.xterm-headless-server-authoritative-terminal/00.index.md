# Step 14: `@xterm/headless` 서버 권위 터미널 상태 전환 계획

작성일: 2026-04-13  
프로젝트: `ProjectMaster`  
범위: 새로고침, 재연결, 워크스페이스 전환, 긴 scrollback, alt-screen/TUI, 다중 클라이언트, 운영/롤백

## 목표

현재의 문자열 tail replay와 브라우저 `localStorage` snapshot 혼합 복구를 폐기하고, 서버가 세션별 터미널 화면 상태를 권위 데이터로 유지하도록 전환한다.

이번 계획의 성공 기준은 다음과 같다.

- 새로고침 후 복원이 "최근 몇 줄"이 아니라 서버 권위 snapshot 기준으로 이뤄진다.
- `\n`, `\r\n`, cursor drift, alt-screen 전환 같은 화면 불일치가 구조적으로 줄어든다.
- `SessionManager` 가 PTY 출력과 resize를 headless terminal state 에 동기 반영한다.
- 프런트는 브라우저 snapshot 이 아니라 서버 `screen-snapshot` 을 우선 적용한다.
- 장애 시 `off -> shadow -> primary` 플래그만으로 즉시 롤백 가능하다.

## 배경

현재 구현의 제약은 코드에서 직접 확인된다.

- 서버는 문자열 tail 기반 `replayBuffer` 만 유지한다. [SessionManager.ts](../../../server/src/services/SessionManager.ts)
  - `replayBuffer` 모델: `SessionData.replayBuffer` [26-31]
  - PTY 출력 누적: `appendReplayOutput` 호출 지점 [186-229]
  - resize 시 PTY 만 변경, 화면 모델은 없음 [319-324]
- 서버는 subscribe 시 `history` 를 보내고 `history:ready` 이후 live output 을 푼다. [WsRouter.ts](../../../server/src/ws/WsRouter.ts)
  - subscribe/replay barrier [126-167]
  - replay pending queue [240-295]
- 프런트는 여전히 브라우저 snapshot 저장/복구 코드를 가지고 있다. [TerminalView.tsx](../../../frontend/src/components/Terminal/TerminalView.tsx)
  - `SerializeAddon` + localStorage snapshot [165-190]
  - 브라우저 snapshot restore [281-301]
  - history 수신 시 xterm reset 후 write [303-327]
- `TerminalContainer` 는 서버 history 가 오지 않으면 브라우저 snapshot fallback 을 시도한다. [TerminalContainer.tsx](../../../frontend/src/components/Terminal/TerminalContainer.tsx) [62-82]

이 구조는 "출력 스트림 재생" 에는 적합하지만 "렌더된 화면 상태 복원" 에는 적합하지 않다.

## 조사 트랙 요약

이번 계획은 세 가지 조사 트랙을 기준으로 취합했다.

1. 런타임/상태 모델 트랙
   - `SessionManager`, `WsRouter`, `node-pty`, `@xterm/headless` 결합 방식
   - resize, alt-screen, cursor, erase, wrap, scrollback 보존 방식
2. 프런트/프로토콜 트랙
   - `history` 중심 프로토콜을 `screen-snapshot` 중심으로 바꾸는 절차
   - `TerminalContainer`, `TerminalView`, `WebSocketContext` cutover
3. 테스트/운영 트랙
   - 성능 예산, 메모리 압박, 관측성, rollout, fallback, rollback
   - 현재 테스트 자산과 부족한 race/perf/ops 검증 보강

## 요구사항 묶음

### 기능 요구사항

- `FR-1401`: 세션마다 서버 권위 터미널 상태를 유지한다.
- `FR-1402`: 새 subscribe/reconnect 시 권위 snapshot 을 전송한다.
- `FR-1403`: alt-screen/TUI 상태를 복원 경로에 포함한다.
- `FR-1404`: resize 는 PTY 와 headless state 에 동시에 반영된다.
- `FR-1405`: 동일 세션의 중복 subscribe 와 재연결은 idempotent 하게 처리된다.
- `FR-1406`: 장애 시 live output 은 계속 전달되고 복구 기능만 degraded 모드로 떨어진다.

### 비기능 요구사항

- `NFR-1401`: snapshot serialize p95 <= 150ms (`10k` scrollback, 단일 세션 기준)
- `NFR-1402`: reconnect 후 첫 화면 복원 p95 <= 500ms
- `NFR-1403`: headless 추가 RSS 는 idle session 당 목표 `<= 6MB`, active session 당 목표 `<= 10MB`
- `NFR-1404`: 25개 동시 세션까지는 이벤트 루프 지연 p95 `<= 50ms`
- `NFR-1405`: flag 한 번으로 `primary -> shadow -> off` 롤백 가능

### 운영 요구사항

- `OBS-1401`: snapshot bytes, serialize latency, ack timeout, degraded session 수를 수집한다.
- `OBS-1402`: per-session geometry owner, snapshot mode, last snapshot age 를 진단 가능해야 한다.
- `OPS-1401`: shadow 모드에서 결과를 비교 수집하고, mismatch 를 로그/메트릭으로 노출한다.

## 품질 게이트

- 아키텍처 일관성: 서버 권위 상태와 브라우저 fallback 경계가 명시된다.
- 테스트 가능성: 단위, 통합, E2E, benchmark, 운영 drill 이 모두 정의된다.
- 운영 가능성: 메트릭, 경고, fallback, rollback, degraded behavior 가 문서화된다.
- 구현 가능성: 현재 코드 경로와 수정 지점이 명확하다.

## 초기 리스크

- `@xterm/headless` 는 공식적으로 experimental 이다.
- headless addon 은 Node-safe 여야 한다.
- 현재 `SessionManager` 는 alt-screen 제어 시퀀스를 replay 경로에서 제거한다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [58-60]
- 현재 resize 는 PTY 에만 반영되고 서버 화면 모델이 없다. [SessionManager.ts](../../../server/src/services/SessionManager.ts) [319-324]
- 현재 `/health` 는 단순 상태만 제공하므로 터미널 전용 운영 메트릭이 없다. [index.ts](../../../server/src/index.ts) [142-149]
