## 권장 성능 예산

- serialize p95 <= `150ms`
- reconnect first-render p95 <= `500ms`
- idle session RSS overhead 목표 <= `6MB`
- active session RSS overhead 목표 <= `10MB`
- 25 concurrent sessions event loop lag p95 <= `50ms`
- snapshot ack timeout rate <= `0.1%`

## 운영 리스크와 대응

| 리스크 | 증상 | 대응 |
| --- | --- | --- |
| serializer CPU 급증 | reconnect 지연, event loop lag 증가 | snapshot cache + throttle + shadow에서 측정 |
| 메모리 압박 | RSS 증가, GC 지연 | session cap 경고, oversized session degrade |
| ACK timeout 누적 | reconnect 후 빈 화면 | queue drop + 재subscribe 허용 + alert |
| geometry owner 흔들림 | resize 점프 | owner lease 로그 + transfer debounce |
| experimental API 변경 | 빌드/런타임 불안정 | 버전 고정 + release notes 확인 gate |

## Test

### 정상 시나리오 1

Given shadow 모드에서 10개 세션이 idle 상태일 때  
When metrics endpoint 를 조회하면  
Then active headless session 수, RSS, serialize latency 요약이 노출되어야 한다

### 정상 시나리오 2

Given reconnect 부하 bench 를 실행할 때  
When `screen-snapshot` serialize 와 ACK roundtrip 을 측정하면  
Then p95 지표가 예산과 비교 가능해야 한다

### 예외 시나리오 1

Given 메모리 경고 임계치를 넘을 때  
When 새 세션이 shadow/primary 로 진입하려고 하면  
Then 운영 로그와 경고가 발생하고 필요 시 해당 세션은 `off` 또는 `degraded` 로 떨어져야 한다

### 예외 시나리오 2

Given diagnostics endpoint 가 인증되지 않은 요청을 받을 때  
When 접근을 시도하면  
Then 민감한 세션 메트릭이 노출되지 않아야 한다

### 경계값 시나리오

Given 25개 동시 세션과 1개 burst output 세션이 함께 있을 때  
When benchmark 를 수행하면  
Then event loop lag, snapshot latency, memory overhead 가 동시에 측정돼야 한다
