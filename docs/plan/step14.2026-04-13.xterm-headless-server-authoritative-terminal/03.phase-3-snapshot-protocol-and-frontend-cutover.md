# Phase 3: Snapshot 프로토콜과 프런트 cutover

## Objective

기존 `history/history:ready` 경로를 `screen-snapshot/screen-ready` 중심으로 치환하고, 브라우저 local snapshot 을 권위 경로에서 제거한다.

## Prerequisites

- Phase 2 완료
- geometry lease 와 degraded mode 정의 완료

## Test

### 정상 시나리오 1

Given reconnect 된 클라이언트가 있고 서버 snapshot 이 준비되어 있을 때  
When `screen-snapshot` 을 받고 `screen-ready` 를 보내면  
Then queued live output 이 snapshot 뒤에만 flush 되어야 한다

### 정상 시나리오 2

Given shadow 모드가 아니라 primary 모드일 때  
When 새 subscribe 가 발생하면  
Then 브라우저 local snapshot 은 적용되지 않고 서버 snapshot 만 적용되어야 한다

### 예외 시나리오 1

Given `screen-ready` ACK 가 timeout 될 때  
When barrier timeout 이 발생하면  
Then 해당 pair 의 queue 는 정리되고 이후 재subscribe 가 가능해야 한다

### 예외 시나리오 2

Given stale revision 을 기준으로 늦게 도착한 ACK 가 있을 때  
When 서버가 이를 처리하면  
Then 최신 snapshot/live state 를 오염시키지 않아야 한다

### 경계값 시나리오

Given snapshot bytes 가 운영 경고 한계 직전까지 커졌을 때  
When subscribe/reconnect 가 발생하면  
Then snapshot apply 시간과 queued live bytes 가 측정되어야 한다

### 테스트 코드 템플릿

```ts
async function testSnapshotAckBarrier(): Promise<void> {
  const harness = createWsSnapshotHarness();

  harness.subscribe('session-1');
  harness.emitPty('session-1', 'after-snapshot');

  assert.equal(harness.clientMessages[0].type, 'screen-snapshot');
  assert.equal(harness.clientMessages.some(m => m.type === 'output'), false);

  harness.sendScreenReady('session-1');
  assert.equal(harness.clientMessages.at(-1)?.type, 'output');
}
```

### 회귀 테스트 실행 조건

- [ ] poisoned browser snapshot 이 서버 snapshot 보다 우선되지 않음
- [ ] duplicate subscribe 가 snapshot 을 중복 재생하지 않음
- [ ] stale ACK 가 live queue 를 잘못 flush 하지 않음
- [ ] 새 세션이 오래된 browser snapshot 을 상속하지 않음
