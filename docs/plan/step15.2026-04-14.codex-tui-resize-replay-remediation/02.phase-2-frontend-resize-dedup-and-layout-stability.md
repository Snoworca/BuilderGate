# Phase 2: Frontend Resize Dedup And Layout Stability

## Objective

Stop the frontend from sending avoidable resize noise during workspace visibility changes.

## Prerequisites

- Phase 1 telemetry is available so resize traffic changes can be measured

## Target Files

- `frontend/src/App.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/MetadataBar/MetadataRow.tsx`
- `frontend/tests/e2e/terminal-authority.spec.ts`

## Planned Changes

1. Introduce last-sent geometry tracking in `TerminalView`.
2. Deduplicate `onResize(term.cols, term.rows)` across:
   - initial mount
   - visibility return
   - `ResizeObserver`
3. Remove one redundant visible-transition fit path if it is no longer needed.
4. Stabilize the terminal viewport so `MetadataRow` does not cause geometry churn during workspace switching.
5. Keep the change surgical: no unrelated terminal UI redesign.

## Implementation Checklist

- [ ] store last-sent `cols/rows` in `TerminalView`
- [ ] skip resize send when geometry is unchanged
- [ ] collapse duplicate fit/resize sources to one primary path
- [ ] make metadata height stable across visible/invisible transitions
- [ ] add regression coverage for repeated visibility toggles

## Test

### Scenario 1: Visibility toggle does not resend identical geometry

Given a terminal returns to visibility without real geometry change  
When the resize pipeline settles  
Then no duplicate resize message is sent

### Scenario 2: Metadata visibility does not change terminal rows spuriously

Given the metadata bar becomes visible for the active tab  
When the user switches workspaces rapidly  
Then the xterm viewport height remains stable unless the outer container actually changed size
