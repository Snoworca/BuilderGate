# Phase 1: Reproduction And Telemetry Baseline

## Objective

Freeze a trustworthy reproduction path and add enough lineage telemetry to prove whether a future fix actually removed duplicate redraw rather than merely hiding it.

## Prerequisites

- existing step14 authority path is green on `server npm test` and `frontend npm run build`

## Target Files

- `server/src/index.ts`
- `server/src/ws/WsRouter.ts`
- `server/src/services/SessionManager.ts`
- `server/src/test-runner.ts`
- `frontend/tests/e2e/terminal-authority.spec.ts`

## Planned Changes

1. Add recent replay-event ring buffer telemetry on the server.
2. Capture `resize_requested`, `resize_skipped`, `snapshot_sent`, `snapshot_refreshed`, `ack_ok`, `ack_stale`, `output_queued`, `output_flushed`.
3. Expose bounded recent events through `/api/sessions/telemetry`.
4. Add one reproducible server-side test that simulates noisy duplicate resize input.
5. Add one regression harness entry for the Codex workspace-bounce scenario, even if the final assertion still depends on later phases.

## Implementation Checklist

- [ ] add bounded replay event type and ring buffer storage
- [ ] log resize requests with current and requested geometry
- [ ] expose recent replay events from telemetry endpoint
- [ ] add test coverage for telemetry retention and event ordering
- [ ] document the baseline reproduction script

## Test

### Scenario 1: Noisy duplicate resize becomes observable

Given the same geometry is requested repeatedly  
When the server records replay events  
Then telemetry must clearly show the repeated input and how it was handled

### Scenario 2: Replay lineage can be traced end to end

Given one session enters replay-pending state  
When snapshot, ack, and queued output events occur  
Then telemetry must show the lineage by `sessionId`, `replayToken`, and `snapshotSeq`
