# Investigation Synthesis

## Evidence Summary

### Confirmed by direct code inspection

- tab-mode workspace switching usually keeps the same terminal instance alive
- the visible transition still triggers multiple layout and resize paths
- `SessionManager.resize()` currently treats all resize requests as meaningful
- `WsRouter.refreshReplaySnapshots()` creates fresh replay lineage whenever resize mutates state

### Confirmed by user reproduction

- Codex printing a long vertical range shows duplicated status/prompt blocks during fast workspace bounce
- large blank gaps grow between visible content regions
- the corruption is much more obvious with Codex than with a plain shell command

### Confirmed by multi-agent review

- server-side: no-op resize handling is missing
- frontend-side: visibility + metadata + fit + observer creates resize churn
- TUI-side: snapshot/live-output boundaries are not frame-safe
- lifecycle-side: duplicate subscribe is not the primary explanation in tab mode
- test-side: current observability is too aggregate to explain lineage cleanly

## Root Cause Model

### Primary Root Cause

Visibility switching creates redundant resize traffic, and the server currently converts that noise into real PTY and replay mutations.

### Secondary Root Cause

When replay is active, authoritative snapshot generation is not tightly fenced to drained headless output, so a TUI redraw frame can be split across snapshot and queued output.

### Non-Primary Cause

Duplicate subscription is not the leading cause for the user's reported tab-mode reproduction.

## Priority Order

1. Remove redundant resize generation.
2. Make server resize idempotent.
3. Fence authoritative snapshot generation behind drained writes.
4. Harden TUI-specific replay and fallback behavior.
5. Expand telemetry and Codex-based regression coverage.

## What Will Not Work

- only increasing debounce without deduplicating geometry
- only adding more replay token checks while leaving resize churn intact
- relying on browser-local snapshots as the primary path for Codex TUI recovery

## Planning Implication

The implementation must be staged. Fixing replay lineage first without fixing resize noise would leave the most common user-visible trigger intact. Fixing frontend noise alone without snapshot fencing would still leave Codex vulnerable on refresh/reconnect and genuine resize boundaries.
