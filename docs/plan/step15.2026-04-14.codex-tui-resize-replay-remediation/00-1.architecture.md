# Architecture

## Problem Frame

The current architecture already treats the server as the terminal authority, but the visibility path still leaks frontend layout churn into the PTY lifecycle.

Current high-risk chain:

1. Workspace becomes visible.
2. The terminal wrapper changes from `display:none` to `display:flex`.
3. `MetadataRow` is mounted only for the visible tab, changing the terminal viewport height.
4. `TerminalContainer` and `TerminalView` both trigger `fit()` and `onResize()`.
5. The server handles each resize as a real geometry mutation.
6. Codex TUI redraws its footer/status region.
7. Replay/snapshot boundaries can capture part of that redraw and append the rest later.

This plan separates the problem into two layers:

- layout-originated noise that should never mutate PTY state
- genuine replay lineage that must be TUI-safe when mutation is real

## Target Architecture

### Frontend Geometry Contract

The frontend becomes responsible for sending resize only when geometry has materially changed.

Required properties:

- one source of truth for the last sent `cols/rows`
- deduplication before any `send({ type: 'resize' })`
- no redundant visible-transition `fit()` path
- stable terminal viewport height across `MetadataRow` show/hide

### Server Resize Contract

The server becomes responsible for treating no-op resize as a hard no-op.

Required properties:

- unchanged `cols/rows` must not call `pty.resize()`
- unchanged `cols/rows` must not increment `screenSeq`
- unchanged `cols/rows` must not invalidate snapshot cache
- unchanged `cols/rows` must not trigger `refreshReplaySnapshots()`

### Snapshot Fence Contract

An authoritative snapshot must reflect a drained headless state, not a partially applied redraw frame.

Required properties:

- snapshot generation waits for prior headless writes to settle
- the lineage between `screenSeq`, `snapshotSeq`, and queued output is explicit
- refresh/replay uses one coherent boundary for TUI redraw workloads

## Dataflow Before

1. UI visibility changes
2. multiple `fit()` paths execute
3. multiple resize messages can be emitted even when geometry is unchanged
4. server mutates replay lineage anyway
5. TUI redraw is triggered
6. snapshot/live output boundary may split one visual frame

## Dataflow After

1. UI visibility changes
2. layout remains stable or converges through one controlled fit path
3. resize is emitted only if `cols/rows` actually changed
4. server rejects unchanged resize without touching lineage
5. authoritative snapshot is created only from drained headless state
6. replay refresh is reserved for meaningful geometry or lineage changes

## Main Risks

- over-suppressing resize and leaving a real geometry mismatch unresolved
- moving snapshot generation behind drain in a way that raises latency too much
- preserving current refresh behavior while tightening replay boundaries

## Architecture Decision Summary

1. Fix the resize noise first because it explains the user-visible failure with the least risk.
2. Then harden snapshot fencing because Codex TUI still needs replay-safe sequencing after genuine resize.
3. Do not pursue wider UX changes unless geometry cannot be stabilized surgically.
