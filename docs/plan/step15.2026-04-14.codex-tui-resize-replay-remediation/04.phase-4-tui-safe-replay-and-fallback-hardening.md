# Phase 4: TUI-safe Replay And Fallback Hardening

## Objective

Harden the replay path so Codex-style redraw workloads do not reappear as duplicate prompt/status blocks after genuine replay boundaries such as refresh, reconnect, or meaningful resize.

## Prerequisites

- Phase 3 snapshot fencing is in place

## Target Files

- `server/src/ws/WsRouter.ts`
- `server/src/services/SessionManager.ts`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `server/src/test-runner.ts`
- `frontend/tests/e2e/terminal-authority.spec.ts`

## Planned Changes

1. Re-check replay refresh rules so only meaningful lineage changes trigger refresh.
2. Ensure fallback behavior for TUI-heavy sessions does not amplify duplicate redraw artifacts.
3. Prefer last good authoritative state over empty fallback where feasible.
4. Tighten restore sequencing so `replaceWithSnapshot()` and resumed live output are easier to reason about.

## Implementation Checklist

- [ ] audit refresh triggers after Phase 3 changes
- [ ] reduce replay refreshes caused only by UI noise
- [ ] define TUI-safe fallback preference order
- [ ] add regression for duplicated footer/status redraw after refresh or reconnect
- [ ] verify existing hidden-workspace and restart lineage tests still pass

## Test

### Scenario 1: Refresh does not duplicate Codex footer redraw

Given a Codex session is actively redrawing its footer  
When refresh or replay restore occurs  
Then the footer/status block appears once in the restored terminal

### Scenario 2: Fallback path does not compound blank gaps

Given a session enters fallback mode  
When recovery is applied  
Then the terminal does not accumulate large blank vertical regions beyond the underlying workload behavior
