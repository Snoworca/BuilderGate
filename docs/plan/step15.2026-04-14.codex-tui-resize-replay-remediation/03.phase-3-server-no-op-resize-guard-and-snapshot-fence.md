# Phase 3: Server No-op Resize Guard And Snapshot Fence

## Objective

Make server-side geometry handling idempotent and prevent authoritative snapshots from being serialized across a partially drained TUI redraw frame.

## Prerequisites

- Phase 2 frontend resize noise is reduced

## Target Files

- `server/src/services/SessionManager.ts`
- `server/src/ws/WsRouter.ts`
- `server/src/test-runner.ts`

## Planned Changes

1. Add a hard no-op resize guard in `SessionManager.resize()`.
2. Ensure no-op resize does not:
   - call `pty.resize()`
   - increment `screenSeq`
   - dirty snapshot cache
   - trigger `refreshReplaySnapshots()`
3. Fence authoritative snapshot generation behind drained headless writes.
4. Make the snapshot lineage explicit enough that queued output is never flushed for content already represented by the snapshot.

## Implementation Checklist

- [ ] add no-op resize early return
- [ ] add tests for unchanged geometry behavior
- [ ] wait for relevant headless write drain before serialization
- [ ] preserve bounded latency and failure handling
- [ ] add regression for snapshot/live-output overlap on TUI-style redraw sequences

## Test

### Scenario 1: No-op resize is a true no-op

Given a session already has geometry `X`  
When resize `X` arrives again  
Then PTY and replay lineage remain unchanged

### Scenario 2: Snapshot is taken after headless drain

Given a burst of TUI redraw output is still draining  
When an authoritative snapshot is requested  
Then serialization waits for the relevant drain boundary instead of capturing a partial frame
