# 기술 결정 기록 (Architecture Decision Records)

## ADR-001: WebSocket 라이브러리로 ws 선택

**상태**: Accepted
**날짜**: 2026-03-27

### 컨텍스트
BuilderGate의 SSE+HTTP 통신을 WebSocket으로 전환할 때 라이브러리 선택이 필요하다.

### 선택지 비교

| 선택지 | 장점 | 단점 | 점수 |
|--------|------|------|------|
| socket.io | 룸/네임스페이스 내장, 자동 재연결, TypeScript 코어 | 프로토콜 오버헤드 15-20%, 폴링 폴백 불필요, 번들 +10.4kB | 5/10 |
| **ws** ✓ | 경량, 성능 2-3배, 프론트 번들 0kB, 업계 표준 | 재연결/멀티플렉싱 직접 구현 (~60줄) | 9/10 |
| uWebSockets.js | 최고 성능 (ws 3-8배) | Express 비호환, 전면 재작성 필요 | 3/10 |

### 결정
**ws** 라이브러리를 사용한다. xterm.js 공식 예제, code-server, ttyd 등 웹 터미널 프로젝트의 사실상 표준이다.

### 결과
- 서버: `npm install ws @types/ws`
- 프론트엔드: 추가 의존성 없음 (네이티브 WebSocket API)

## ADR-002: 하이브리드 아키텍처 (WS + REST)

**상태**: Accepted
**날짜**: 2026-03-27

### 컨텍스트
모든 HTTP 엔드포인트를 WS로 전환할지, 일부만 전환할지 결정 필요.

### 결정
**실시간 스트리밍만 WS, CRUD는 REST 유지**한다. 35개 REST 엔드포인트 중 5개만 WS로 전환.

### 근거
- CRUD 작업은 요청-응답 패턴이 자연스러움 (HTTP 상태 코드, 캐싱)
- 파일 API는 바이너리 전송이 있어 REST가 적합
- curl, Postman 등 개발 도구와의 호환성 유지

## ADR-003: CWD 추적을 서버 Push로 전환

**상태**: Accepted
**날짜**: 2026-03-27

### 컨텍스트
현재 CWD를 3초(TerminalContainer) + 5초(useCwd) 폴링으로 추적. 세션당 ~20 req/min 발생.

### 결정
서버에서 `fs.watchFile(cwdFilePath)`로 변경 감지 → WS `{ type: "cwd" }` 메시지로 push한다.

### 근거
- 폴링 제거: 8세션 × 20 req/min = 160 req/min → 0 req/min
- 실시간성: 3-5초 지연 → 즉시 감지 (~1초 이내)
- `fs.watchFile`은 Windows/Linux/macOS 모두 지원, stat 기반 폴링이라 안정적

## ADR-004: 좀비 세션 감지 메커니즘

**상태**: Accepted
**날짜**: 2026-03-27

### 컨텍스트
현재 SSE 에러 시 탭 상태가 disconnected로 전환되지 않아 좀비 터미널 발생.

### 결정
3단계 좀비 감지:
1. **서버 → 클라이언트**: PTY exit 시 `{ type: "session:exited" }` 메시지 전송
2. **구독 응답**: subscribe 시 존재하지 않는 세션은 `status: "error"` 로 응답
3. **WS 연결 끊김**: 프론트엔드가 모든 세션을 일시 중단, 재연결 후 subscribe로 상태 확인
