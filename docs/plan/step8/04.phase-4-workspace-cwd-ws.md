## LLM 코딩 주의사항

- **Simplicity**: CWD 감시에 `fs.watchFile`(Node.js 내장)을 사용. chokidar는 불필요. `fs.watchFile`은 stat 폴링 기반이라 크로스 플랫폼에서 안정적이며, 세션당 1개 파일만 감시하므로 성능 영향 없음
- **Surgical**: workspaceRoutes.ts에서 REST 엔드포인트(CRUD)는 수정하지 않는다. broadcast 호출만 WsRouter로 위임. SSE 엔드포인트는 아직 제거하지 않음 (Phase 5)

## 상세 구현 지침

### 4-1. WsRouter 워크스페이스 broadcast

```typescript
// 모든 WS 클라이언트에 브로드캐스트 (세션 구독과 무관)
broadcastAll(event: string, data: object, excludeClientId?: string): void {
  const msg = JSON.stringify({ type: event, data });
  for (const [ws, meta] of this.clients) {
    if (meta.clientId !== excludeClientId && ws.readyState === 1) {
      ws.send(msg);
    }
  }
}
```

### 4-2. workspaceRoutes broadcast 전환

```typescript
// 변경 전 (workspaceRoutes.ts)
broadcast('workspace:created', workspace, clientId);

// 변경 후
wsRouter.broadcastAll('workspace:created', workspace, clientId);
```

workspaceRoutes에서 `wsRouter` 인스턴스에 접근할 수 있도록, Express의 `app.set('wsRouter', wsRouter)` 또는 DI를 통해 전달한다.

### 4-3. 자기 메시지 제외

현재 REST 요청의 `x-client-id` 헤더로 자기 메시지를 제외한다. WS 전환 후에도 REST로 CRUD를 수행하므로 동일 패턴 유지:

```typescript
// REST 엔드포인트에서 (변경 없음)
const clientId = req.headers['x-client-id'] as string | undefined;
wsRouter.broadcastAll('workspace:created', workspace, clientId);
```

프론트엔드는 REST 요청 시 `x-client-id` 헤더에 WS clientId를 포함:
```typescript
// api.ts의 authFetch에 clientId 추가
headers: { ...getAuthHeaders(), 'x-client-id': wsClientId }
```

### 4-5. CWD 파일 감시

```typescript
// SessionManager.ts — createSession 내부에 추가
if (cwdFilePath) {
  fs.watchFile(cwdFilePath, { interval: 1000 }, () => {
    try {
      const cwd = fs.readFileSync(cwdFilePath, 'utf8').trim();
      if (cwd && cwd !== lastCwd) {
        lastCwd = cwd;
        this.broadcastWs(sessionId, 'cwd', { cwd });
      }
    } catch { /* 파일 없음 — 무시 */ }
  });
}

// deleteSession에서 감시 해제
fs.unwatchFile(cwdFilePath);
```

### 4-7. CWD 폴링 제거

**TerminalContainer.tsx**: `useEffect`의 `setInterval(poll, 3000)` 블록 전체 제거. Phase 3에서 subscribeSession의 `onCwd` 핸들러가 이미 CWD 업데이트를 처리.

**useCwd.ts**: `setInterval(fetchCwd, 5000)` 블록 제거. WS의 `cwd` 메시지가 useWorkspaceManager의 `updateTabCwd`를 호출하도록 연결.

## 테스트 시나리오

### 정상 케이스
- **Given**: WS 연결 + 워크스페이스 화면
  **When**: REST로 새 탭 추가 `POST /api/workspaces/:id/tabs`
  **Then**: WS로 `{ type: "tab:added", data: {...} }` 수신, 그리드에 새 셀 표시

- **Given**: WS 연결 + 터미널에서 `cd /tmp` 실행
  **When**: 서버가 cwdFilePath 변경 감지
  **Then**: WS로 `{ type: "cwd", sessionId, cwd: "/tmp" }` 수신, UI 업데이트

### 예외 케이스
- **Given**: 2개 브라우저 탭에서 동일 워크스페이스 열람
  **When**: 탭 A에서 워크스페이스 이름 변경
  **Then**: 탭 B에만 `workspace:updated` 수신 (탭 A는 제외)

### 경계값 테스트
- **Given**: CWD가 매우 빠르게 변경 (cd 연속 실행)
  **When**: fs.watchFile의 1초 interval 내 여러 번 변경
  **Then**: 마지막 CWD만 전송 (interval 기반이므로 자연스럽게 디바운싱)
