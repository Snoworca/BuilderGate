# Phase 3: 터미널 입출력 WS 전환

## 목표

터미널의 핵심 입출력을 SSE+HTTP에서 WebSocket으로 전환한다. SessionManager의 SSE broadcast를 WS broadcast로 교체하고, TerminalContainer가 useWebSocket 구독을 사용하도록 변경한다. 좀비 세션 문제를 동시에 해결한다.

## LLM 코딩 주의사항

- **Simplicity**: SessionManager에 WS 전용 broadcast를 추가하되, SSE broadcast는 아직 제거하지 않는다. Phase 5에서 제거. 두 경로가 공존해도 문제없음 (SSE 클라이언트가 없으면 빈 루프)
- **Surgical**: TerminalContainer.tsx에서 useSSE 호출을 교체할 때, 핸들러 인터페이스(onOutput, onStatus, onError)는 동일하게 유지. CWD 폴링도 아직 유지 (Phase 4에서 제거)

## 상세 구현 지침

### 3-1. WsRouter subscribe 핸들러

```typescript
// 서버: subscribe 메시지 처리
private handleSubscribe(ws: WebSocket, sessionIds: string[]) {
  const results: Array<{sessionId: string; status: string; cwd?: string}> = [];

  for (const id of sessionIds) {
    // 구독 등록
    if (!this.sessionSubscribers.has(id)) this.sessionSubscribers.set(id, new Set());
    this.sessionSubscribers.get(id)!.add(ws);
    this.clientSubscriptions.get(ws)!.add(id);

    // 세션 존재 여부 확인
    const session = this.sessionManager.getSession(id);
    if (session) {
      results.push({ sessionId: id, status: session.status, cwd: ... });
      // 버퍼된 출력 전송 (SSE의 addSSEClient 패턴)
      this.sessionManager.flushBufferToWs(id, ws);
    } else {
      results.push({ sessionId: id, status: 'error' });  // 좀비 감지!
    }
  }

  ws.send(JSON.stringify({ type: 'subscribed', sessions: results }));
}
```

### 3-3. SessionManager WS broadcast

**파일**: `server/src/services/SessionManager.ts`에 메서드 추가

```typescript
// 기존 broadcast(SSE)와 병행하는 WS broadcast
broadcastWs(sessionId: string, event: string, payload: object): void {
  const subscribers = this.wsRouter?.getSubscribers(sessionId);
  if (!subscribers) return;
  const msg = JSON.stringify({ type: event, sessionId, ...payload });
  for (const ws of subscribers) {
    if (ws.readyState === 1) ws.send(msg);
  }
}
```

### 3-7. TerminalContainer 전환

**파일**: `frontend/src/components/Terminal/TerminalContainer.tsx`

```typescript
// 변경 전
useSSE(sessionId, {
  onOutput: (data) => terminalRef.current?.write(data),
  onStatus: (status) => onStatusChange(sessionId, status),
  onError: (message) => console.error('Session error:', message),
  onAuthError,
});

// 변경 후
const ws = useContext(WebSocketContext);
useEffect(() => {
  const unsubscribe = ws.subscribeSession(sessionId, {
    onOutput: (data) => terminalRef.current?.write(data),
    onStatus: (status) => onStatusChange(sessionId, status),
    onError: (message) => {
      console.error('Session error:', message);
      onStatusChange(sessionId, 'disconnected');  // 좀비 세션 해결!
    },
    onCwd: (cwd) => onCwdChange?.(sessionId, cwd),
  });
  return unsubscribe;
}, [sessionId]);
```

## 테스트 시나리오

### 정상 케이스
- **Given**: WS 연결 + 세션 1개 구독
  **When**: 서버 PTY에서 "hello" 출력
  **Then**: WS로 `{ type: "output", sessionId, data: "hello" }` 수신, xterm에 표시

- **Given**: WS 연결 + 세션 구독
  **When**: 프론트엔드에서 `ws.send({ type: "input", sessionId, data: "ls\n" })`
  **Then**: 서버 PTY에 "ls\n" 입력, 이후 출력이 WS로 돌아옴

### 예외 케이스 (좀비 세션)
- **Given**: 서버 재시작 후 WS 재연결
  **When**: subscribe에 이전 세션 ID 포함
  **Then**: `{ type: "subscribed", sessions: [{ sessionId, status: "error" }] }` 수신 → 탭이 disconnected로 전환

### 경계값 테스트
- **Given**: 8개 세션 동시 구독
  **When**: 모든 세션에서 동시 출력
  **Then**: 각 세션의 output이 올바른 터미널에 라우팅 (sessionId 기반 분리)
