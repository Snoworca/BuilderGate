# BuilderGate 네이티브 기본 데몬 모드 기술 결정 기록

## ADR-8-001: PM2를 완전히 제거하고 native daemon으로 대체

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 맥락 | 현재 `tools/start-runtime.js`, `stop.js`, `tools/build-daemon-exe.js`, README에 PM2 설치/시작/중지 경로가 있다. |
| 결정 | production start/stop/build/docs에서 PM2 호출과 dependency를 제거한다. |
| 근거 | 사용자는 BuilderGate 실행파일 자체가 기본 daemon으로 동작하기를 요구했다. PM2 fallback은 SRS에서 제외됐다. |
| 영향 | launcher, stop utility, sentinel, readiness, state store가 자체 구현되어야 한다. |

## ADR-8-002: `--foreground`를 canonical, `--forground`를 alias로 지원

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | 문서 대표 명령은 `--foreground`를 사용하되 `--forground`도 동일하게 처리한다. |
| 근거 | SRS가 사용자 원문 호환 alias를 Must로 요구한다. |
| 검증 | CLI unit test에서 두 옵션 모두 mode `foreground`를 반환해야 한다. |

## ADR-8-003: Runtime path는 공통 resolver로 고정

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | `tools/daemon/runtime-paths.js`를 만들고 start/stop/sentinel/build 검증이 같은 경로 계약을 사용한다. |
| 근거 | `dist/bin`, config, server entry, bundled Node, state/log 경로가 분기되면 packaged/source 동작이 깨진다. |
| 검증 | packaged/source/env override/unit test로 path contract를 고정한다. |

## ADR-8-004: Strict config는 production native runtime에서만 강제

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | `NODE_ENV=production`이고 native launcher/foreground 경로인 경우 기존 config parse/schema/read/cert/TOTP fatal은 fallback 없이 throw한다. config 파일이 없을 때의 bootstrap template 생성은 유지한다. |
| 근거 | SRS는 meaningful error를 default config로 숨기는 것을 금지하지만 `dev.js` 개발 편의 fallback은 범위 밖이다. |
| 검증 | invalid JSON5/schema config가 daemon/foreground에서 non-zero exit와 fatal preflight state를 만든다. |

## ADR-8-005: Readiness는 identity 기반으로 판정

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | `/health` 200만으로 성공하지 않고 `pid`와 `startAttemptId` 또는 `stateGeneration`을 확인한다. |
| 근거 | target port에 이전 instance 또는 무관 서버가 응답하는 false-positive를 막아야 한다. |
| 검증 | 무관 `/health` 서버가 200을 반환해도 launcher는 timeout/port conflict로 실패해야 한다. |

## ADR-8-006: TOTP preflight는 기존 `TOTPService` 확장으로 구현

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | secret 생성/로드/BASE32/QR URI 정책은 `TOTPService`를 재사용하고, `suppressConsoleQr` 옵션만 추가한다. |
| 근거 | QR preflight를 별도 구현하면 secret 암호화와 validation 중복이 생긴다. |
| 검증 | daemon parent는 QR/manual key를 출력하고 app child는 suppress env로 QR을 출력하지 않는다. |

## ADR-8-006A: Launcher TOTP preflight adapter는 compiled server ESM helper를 동적 import

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | CommonJS launcher의 `tools/daemon/totp-preflight.js`는 `server/dist/services/daemonTotpPreflight.js`를 동적 import한다. 이 helper는 `server/dist/index.js`를 import하지 않고 config path와 secret path를 인자로 받아 side effect 없이 실행한다. |
| 근거 | TOTP secret 암호화/검증/QR 정책은 서버 서비스와 같아야 하지만, launcher가 서버 entrypoint를 import하면 서버가 시작되는 side effect가 생길 수 있다. |
| 검증 | secret absent/existing 양쪽에서 QR/manual key/issuer/accountName 출력이 동일하고, preflight failure는 app/sentinel orphan 없이 startup failure가 된다. |

## ADR-8-007: Stop은 signal-only가 아니라 internal shutdown protocol을 사용

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | server에 `performGracefulShutdown(reason)` 공통 함수를 만들고 loopback+token internal route가 이를 호출한다. |
| 근거 | Windows에서 POSIX signal만으로 workspace/CWD flush 완료를 정의할 수 없다. |
| 검증 | shutdown route response의 flush result, workspace JSON, `/health` 비응답을 모두 확인한다. |

## ADR-8-008: Sentinel은 packaged physical entry/source launcher reexec

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | packaged sentinel은 bundled Node로 `tools/daemon/sentinel-entry.js`를 실행하고, source sentinel은 `process.execPath tools/start-runtime.js --internal-sentinel`를 사용한다. |
| 근거 | `pkg` self-reexec는 packaged argument/bootstrap 해석 차이로 sentinel startup이 불안정했다. 물리 entry 파일을 배포본에 포함하면 app child와 같은 bundled Node 실행 계약을 재사용할 수 있고, PID command line 검증도 안정적이다. |
| 검증 | stop PID 검증에서 source `--internal-sentinel` 또는 packaged `tools/daemon/sentinel-entry.js` marker가 없는 무관 프로세스를 sentinel로 인정하지 않는다. build output 검증은 sentinel entry와 모든 daemon runtime dependency 파일을 필수로 확인한다. |

## ADR-8-009: 문서 검증은 required/forbidden pattern 테스트로 자동화

| 항목 | 결정 |
| --- | --- |
| 상태 | Accepted |
| 결정 | README/root와 `dist/bin/README.md`에서 required keywords와 forbidden PM2 patterns를 테스트한다. |
| 근거 | 이 요구사항은 반복적으로 깨지기 쉬운 운영 계약이므로 문서도 회귀 테스트 대상이어야 한다. |
| 검증 | `--foreground`, `--forground`, `BuilderGateStop`, `config.json5`, `QR`, `dist/bin`, `native daemon` 또는 `네이티브 데몬` 포함 및 production PM2 패턴 0개. |
