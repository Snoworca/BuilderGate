# BuilderGate 네이티브 기본 데몬 모드 아키텍처

## 1. 아키텍처 개요

BuilderGate production runtime을 PM2 외부 프로세스 관리자에서 자체 native daemon 구조로 전환한다. `tools/start-runtime.js`와 packaged `BuilderGate.exe`는 동일한 런처 계약을 공유하고, daemon mode에서는 app child와 sentinel child를 detached로 시작한다.

```text
User Console
  |
  v
Launcher Parent (BuilderGate.exe or tools/start-runtime.js)
  |-- CLI/path/config strict preflight
  |-- optional TOTP QR preflight
  |-- daemon state write
  |-- spawn app child
  |-- spawn sentinel child
  |-- readiness identity check
  v
Detached Runtime
  |-- App Child: server/dist/index.js
  |-- Sentinel Child: BuilderGate.exe self-reexec or node tools/start-runtime.js --internal-sentinel
  |-- Runtime files: runtime/buildergate.daemon.json, runtime/*.log
```

## 3. 데이터 흐름

### 3.1 Daemon Start

1. Launcher가 CLI와 runtime path를 결정한다.
2. Config preflight가 config 없음 bootstrap과 기존 config strict failure를 구분한다.
3. 기존 active state가 있으면 같은 실행 계약은 idempotent success, 다른 계약은 conflict failure로 종료한다.
4. TOTP enabled이면 parent가 같은 `TOTP_SECRET_PATH`로 QR/manual key를 출력한다.
5. Launcher가 app child와 sentinel child를 detached spawn하고 state를 `starting` 또는 `running`으로 갱신한다.
6. Readiness는 `/health` body/header 또는 IPC ready message에서 `pid/startAttemptId/stateGeneration`을 확인해야 성공한다.
7. Phase 3의 sentinel은 최소 internal mode marker와 `status=stopping` 감지 종료만 보장하고, abnormal restart policy는 Phase 6에서 확장한다.

### 3.2 Foreground Start

1. `--foreground` 또는 `--forground`가 있으면 sentinel을 만들지 않는다.
2. App server stdout/stderr를 현재 콘솔에 연결한다.
3. Active daemon state를 기록하지 않는다.
4. 종료는 Ctrl+C/SIGINT/SIGTERM과 `performGracefulShutdown()`이 처리한다.

### 3.3 Native Stop

1. Stop utility가 state를 읽고 schema와 PID identity를 검증한다.
2. State를 `stopping`으로 atomic update한다.
3. Sentinel을 먼저 종료해 app 자동 재시작을 막는다.
4. Stop utility가 loopback HTTPS shutdown route에 token을 전달한다.
5. Server는 `performGracefulShutdown()` 완료 후 flush result를 반환하고 process exit를 예약한다.
6. Stop utility가 `/health` 비응답과 workspace JSON flush evidence를 확인한 뒤 stopped state를 기록한다.

## 4. Daemon State

`<BIN_DIR>/runtime/buildergate.daemon.json`은 상태별 union이다. `fatalStage=preflight`에서는 PID/heartbeat 필드가 null 또는 생략될 수 있고, config parse/schema 실패가 port 해석 전에 발생하면 `port=null`이어야 한다.

핵심 필드:

- `status`: `starting`, `running`, `stopping`, `stopped`, `fatal`
- `appPid`, `sentinelPid`
- `launcherPath`, `serverEntryPath`, `serverCwd`, `nodeBinPath`, `configPath`, `totpSecretPath`
- `port`, `argvHash`, `shutdownToken`, `startAttemptId`, `stateGeneration`
- `restartCount`, `lastRestartAt`, `lastExitCode`, `heartbeatAt`, `fatalReason`, `fatalStage`

## 5. 보안 경계

- `shutdownToken`은 CLI 인자로 전달하지 않는다.
- shutdown route는 socket remote address가 loopback인지 직접 검증하고 forwarding header를 신뢰하지 않는다.
- token, password, JWT, OTP code, TOTP secret 평문은 로그에 남기지 않는다.
- state 전체를 로그에 출력하지 않고 민감 필드는 마스킹한다.
- PID 검증 실패 시 kill fallback을 수행하지 않는다.

## 6. 변경 금지 경계

- UI 시각 요소, 로그인 UX, WebSocket/SSE payload, session status flow는 변경하지 않는다.
- `dev.js` 개발 실행은 daemon mode로 바꾸지 않는다.
- `twoFactor.totp.enabled` 같은 신규 중첩 schema를 도입하지 않는다.
- `--replace` 또는 자동 stop-and-replace는 이번 범위에 포함하지 않는다.
