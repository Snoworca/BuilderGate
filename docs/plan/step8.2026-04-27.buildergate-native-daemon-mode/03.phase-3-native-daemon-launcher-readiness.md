# Phase 3: Native Daemon Launcher, PM2 제거, Readiness Identity

## Phase 3: Native daemon launcher, PM2 제거, readiness identity

### 목표

무인자 기본 실행을 PM2 없는 native daemon으로 전환한다. 동일 daemon start는 idempotent success로 처리하고, 다른 실행 계약은 stop 선행 요구로 거부한다. Readiness는 `/health` 200이 아니라 identity로 검증한다. 이 Phase에서 최소 sentinel spawn/marker/state 계약도 함께 구현해 이후 stop Phase가 sentinel을 검증할 수 있게 한다.

## 선행 조건

- [ ] Phase 1 완료
- [ ] Phase 2 foreground branch 완료
- [ ] daemon state store와 runtime path resolver 사용 가능

## Target Files

| 파일 | 변경 |
| --- | --- |
| `tools/start-runtime.js` | PM2 install/start/status/delete branch 제거, default daemon 연결 |
| `tools/daemon/launcher.js` | daemon start orchestration |
| `tools/daemon/readiness.js` | `/health` or IPC readiness identity polling |
| `tools/daemon/process-info.js` | spawned app/sentinel identity metadata 수집 기반 |
| `tools/daemon/sentinel.js` | 최소 internal sentinel mode, marker, `status=stopping` 감지 종료 |
| `server/src/index.ts` | `/health`에 `pid/startAttemptId/stateGeneration` body/header 제공 |
| `server/src/test-runner.ts` 또는 daemon tests | idempotent/conflict/readiness false-positive tests |

## 기존 코드 재사용 힌트

- `ensureDependenciesAndBuild()`는 daemon 실행 전 build artifact 준비에 재사용한다.
- `resolvePort()`의 CLI/config/default 우선순위는 유지한다.
- `withLocalBinPath()`와 local Node discovery는 app child env 구성에 재사용하되 PM2 interpreter 목적은 제거한다.

## 코드 중복 주의

- PM2 제거와 daemon start 구현을 동시에 하되 legacy PM2 app cleanup을 native daemon scope로 가져오지 않는다.
- readiness cleanup은 “이번 launcher가 spawn한 app/sentinel”만 대상으로 해야 한다. 기존 무관 프로세스를 종료하지 않는다.

## 언어+버전 주의사항

- HTTPS self-signed `/health` polling은 certificate 검증을 우회할 수 있어야 하지만 loopback 대상만 사용한다.
- Detached child의 stdio는 log file로 연결하고 parent가 성공 전 종료하지 않도록 readiness await를 보장한다.

## LLM 코딩 주의사항

- Simplicity: Phase 3에서는 sentinel restart 정책을 완성하지 않는다. 다만 daemon state와 stop 검증을 위해 최소 sentinel spawn/marker/stopping 감지는 구현한다.
- Surgical: PM2 문자열 제거 중 README 변경은 Phase 8에서 한다. 코드와 build dependency 제거는 Phase 3/7에서 한다.
- Safety: 기존 active daemon이 다르면 자동 stop-and-replace를 구현하지 않는다.

## 테스트 시나리오

### 정상 케이스

- Given: source production build artifact와 valid config가 있다.
  When: `node tools/start-runtime.js -p 2002`를 실행한다.
  Then: parent는 readiness identity 확인 후 0으로 종료하고 app child는 `/health`에 응답한다.

- Given: 같은 daemon이 이미 running이다.
  When: 같은 options로 start한다.
  Then: 새 child 없이 idempotent success를 반환하고 기존 appPid/sentinelPid를 유지한다.

### 예외 케이스

- Given: target port에 무관한 서버가 `/health` 200을 반환한다.
  When: daemon start를 실행한다.
  Then: identity mismatch로 success 처리하지 않는다.

- Given: existing active daemon state가 다른 port를 가진다.
  When: 새 start를 실행한다.
  Then: 자동 교체하지 않고 stop 안내와 non-zero exit를 반환한다.

- Given: `--reset-password`와 `--bootstrap-allow-ip`가 daemon start와 함께 전달된다.
  When: launcher가 app child를 spawn한다.
  Then: password reset은 resolved config에 선반영되고 bootstrap allowlist는 env로만 전달된다.

### 경계값 테스트

- Given: readiness timeout 30초가 경과한다.
  When: app identity가 확인되지 않는다.
  Then: 이번 attempt child만 cleanup하고 무관 프로세스는 건드리지 않는다.
