# Phase 4: TOTP Daemon Preflight와 QR 중복 억제

## Phase 4: TOTP daemon preflight와 QR 중복 억제

### 목표

TOTP enabled daemon 실행에서 parent가 detach 전에 QR/manual entry key를 출력하고, app child는 같은 secret path를 사용하되 QR을 중복 출력하지 않게 한다. Initial startup failure와 settings hot-swap warning 정책을 분리한다.

## 선행 조건

- [ ] Phase 1 strict config와 path resolver 완료
- [ ] Phase 3 daemon launcher env 구성 완료
- [ ] `TOTPService` 현재 secret path 주입 동작 확인

## Target Files

| 파일 | 변경 |
| --- | --- |
| `server/src/services/TOTPService.ts` | `suppressConsoleQr` 또는 동등 옵션 추가 |
| `server/src/services/daemonTotpPreflight.ts` | 신규. compiled server ESM에서 side-effect-free QR preflight 제공 |
| `server/src/services/twoFactorRuntime.ts` | env path/suppress와 initial startup fatal policy 반영 |
| `server/src/index.ts` | startup TOTP failure를 fatal로 전달, env secret path 사용 |
| `tools/daemon/totp-preflight.js` | parent QR preflight helper |
| `tools/daemon/launcher.js` | daemon preflight와 app child suppress env 연결 |
| `server/src/test-runner.ts` | TOTP suppress/startup fatal/hot-swap tests |

## 기존 코드 재사용 힌트

- `TOTPService`의 secret create/load/BASE32 validation/QR printing을 재사용한다.
- `CryptoService` machine key 정책을 preflight에서도 동일하게 사용한다.
- `reconcileTotpRuntime()`의 settings hot-swap warning 정책은 유지하고 initial startup path만 fatal로 분리한다.
- `tools/daemon/totp-preflight.js`는 `server/dist/services/daemonTotpPreflight.js`를 동적 import해 실행한다. 이 helper는 server top-level `index.js`를 import하지 않고 config path/secret path를 인자로 받는다.

## 코드 중복 주의

- QR URI 생성, issuer/accountName 기본값, manual key 출력 형식을 launcher에서 별도 복제하지 않는다.
- `twoFactor.totp.enabled` 같은 신규 schema를 만들지 않는다. 조건은 `config.twoFactor.enabled === true`다.

## 언어+버전 주의사항

- `TOTPService`는 TypeScript ESM class다. CommonJS launcher는 `server/dist/services/daemonTotpPreflight.js`를 동적 import하는 방식으로 고정한다.
- secret path는 Windows backslash와 POSIX slash 모두에서 path normalization을 테스트한다.

## LLM 코딩 주의사항

- Simplicity: QR preflight 성공만 위해 TOTP 등록 UI나 auth flow를 변경하지 않는다.
- Surgical: `authRoutes`의 TOTP login/verify contract는 변경하지 않는다.
- Security: secret 평문은 manual entry key로 콘솔 출력되는 요구사항 외 로그 파일에 남기지 않는다.

## 테스트 시나리오

### 정상 케이스

- Given: `twoFactor.enabled: true`이고 secret 파일이 없다.
  When: daemon mode start를 실행한다.
  Then: parent console에 QR/manual key가 출력되고 app child는 같은 `TOTP_SECRET_PATH`를 사용한다.

- Given: `twoFactor.enabled: true`이고 기존 secret 파일이 있다.
  When: daemon mode start를 실행한다.
  Then: parent console에 QR, manual entry key, issuer, accountName이 detach 전에 출력된다.

- Given: app child env에 `BUILDERGATE_SUPPRESS_TOTP_QR=1`이 있다.
  When: TOTP runtime을 initialize한다.
  Then: secret은 로드되지만 console QR은 출력되지 않는다.

### 예외 케이스

- Given: corrupted TOTP secret 파일이다.
  When: daemon preflight를 실행한다.
  Then: startup은 실패하고 app/sentinel child를 남기지 않는다.

### 경계값 테스트

- Given: settings 화면에서 TOTP issuer 변경 중 secret corruption이 발생한다.
  When: hot-swap runtime refresh가 실패한다.
  Then: previous service를 유지하고 warning을 반환한다.
