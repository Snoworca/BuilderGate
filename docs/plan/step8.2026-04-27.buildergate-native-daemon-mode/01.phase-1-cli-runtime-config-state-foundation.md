## Phase 1: CLI, runtime path, strict config, daemon state 기반

## 기존 코드 재사용 힌트

- `tools/start-runtime.js`의 `parseBootstrapAllowIps`, `resetPasswordInConfigContent`, `resetPasswordInConfigFile`, `resolvePort` 기존 의미를 유지한다.
- `server/src/utils/config.ts`의 `getConfigPath()`, `loadConfigFromPath()`, `configSchema`를 strict config의 기준으로 사용한다.
- `server/src/utils/config.ts`의 top-level default export side effect는 launcher preflight에서 직접 재사용하지 않는다. `loadConfigFromPathStrict(configPath, platform)` 또는 동등한 side-effect-free strict loader API를 추출하고, `tools/daemon/config-preflight.js`는 이 API를 compiled server helper로 호출한다.
- `tools/build-daemon-exe.js`의 `OUTPUT_DEFAULT = dist/bin` 정책을 runtime path resolver 테스트의 packaged 기준으로 재사용한다.

## 코드 중복 주의

- `BIN_DIR`, `SERVER_DIR`, `SERVER_ENTRY`, `NODE_BIN`, `CONFIG_PATH`, `STATE_PATH`, `LOG_PATH`를 start/stop/build/sentinel에 각각 복사하지 않는다.
- config port parsing regex를 새 strict config 판단 기준으로 확장하지 않는다. 가능하면 서버 config loader 또는 동일 schema helper를 재사용한다.
- state schema validation과 token masking은 stop/sentinel에서 다시 만들지 않도록 공통 모듈에 둔다.

## 언어+버전 주의사항

- Launcher는 CommonJS 파일이다. ESM import를 직접 섞지 말고 `require` 또는 동적 import 경계를 명확히 둔다.
- Server TypeScript는 `module: NodeNext`/ESM 패턴을 유지한다.
- Node.js 20.x 기준 API를 사용하되 pkg `node18` target 호환성을 깨는 API는 피한다.

## LLM 코딩 주의사항

- Simplicity: Phase 1에서는 실제 daemon spawn을 구현하지 않는다. parser/path/state/config 기반만 만든다.
- Surgical: 기존 `--reset-password`와 bootstrap allowlist 의미를 바꾸지 않는다.
- Safety: strict config는 production native runtime에 한정하고 `dev.js` 개발 편의 fallback은 건드리지 않는다.

## 테스트 시나리오

### 정상 케이스

- Given: `BuilderGate.exe --foreground --port 2002 --bootstrap-allow-ip 127.0.0.1`
  When: CLI parser를 실행한다.
  Then: mode는 `foreground`, port는 `2002`, allowlist는 `127.0.0.1`이다.

- Given: packaged resolver 입력이 `process.pkg=true`, `process.execPath=C:\dist\bin\BuilderGate.exe`다.
  When: runtime path를 resolve한다.
  Then: `CONFIG_PATH=C:\dist\bin\config.json5`, `SERVER_DIR=C:\dist\bin\server`, `STATE_PATH=C:\dist\bin\runtime\buildergate.daemon.json`이다.

### 예외 케이스

- Given: existing `config.json5`가 invalid JSON5다.
  When: production strict mode에서 load한다.
  Then: default config fallback 없이 오류를 throw한다.

- Given: corrupt state file이다.
  When: state store가 읽는다.
  Then: 명확한 parse error를 반환하고 PID kill 같은 side effect는 없다.

### 경계값 테스트

- Given: port `1023`, `65536`, `1024`, `65535`.
  When: CLI parser가 검사한다.
  Then: 앞의 두 값은 실패하고 뒤의 두 값은 성공한다.

- Given: `buildergate`, `start.bat`, `start.sh`가 start interface로 launcher를 호출한다.
  When: parser와 wrapper smoke를 실행한다.
  Then: `IR-8-001`의 source/package start interface가 같은 옵션 계약을 사용한다.

- Given: config file A와 B가 있고 `BUILDERGATE_CONFIG_PATH`가 A를 가리킨다.
  When: `--reset-password`를 실행한다.
  Then: A의 `auth.password`만 빈 문자열이 되고 B는 변경되지 않는다.

- Given: `--bootstrap-allow-ip 127.0.0.1,10.0.0.8`를 daemon 또는 foreground로 실행한다.
  When: app child env를 검사한다.
  Then: `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS`가 설정되고 config 파일에는 저장되지 않는다.

- Given: `--help`를 실행한다.
  When: help text를 검사한다.
  Then: 기본 daemon, `--foreground`, `--forground`, stop 명령, `dist/bin`, config path 정책이 포함된다.

- Given: preflight fatal state이고 port 해석 전 실패다.
  When: state schema를 validate한다.
  Then: `port=null`과 PID field 생략이 valid다.

- Given: state file에 `shutdownToken`이 포함된다.
  When: state store가 파일을 쓴다.
  Then: 가능한 OS에서 owner read/write 권한이 적용되고, 불가능한 환경은 skip reason을 테스트 결과에 남긴다.
