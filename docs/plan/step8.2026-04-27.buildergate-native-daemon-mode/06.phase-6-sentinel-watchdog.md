# Phase 6: Sentinel Watchdog와 Fatal/Restart 정책

## Phase 6: Sentinel watchdog와 fatal/restart 정책

### 목표

PM2의 감시 역할을 native sentinel로 대체한다. Sentinel은 abnormal app exit만 제한적으로 재시작하고, fatal startup과 stop 중에는 재시작하지 않는다.

## 선행 조건

- [ ] Phase 1 daemon state/log 기반 완료
- [ ] Phase 3 app child daemon start와 최소 sentinel marker/stopping 감지 완료
- [ ] Phase 5 stop utility가 최소 sentinel을 검증하고 중지할 수 있음

## 기존 코드 재사용 힌트

- Phase 3의 app child spawn helper를 sentinel restart에서도 재사용한다.
- Phase 1 state store의 atomic update와 log masking을 sentinel heartbeat/restart 기록에 재사용한다.
- Phase 5 PID validator의 sentinel marker 검증을 stop과 공유한다.

## 코드 중복 주의

- sentinel이 app spawn env를 별도로 구성하지 않도록 launcher helper를 공유한다.
- fatal classification을 sentinel 내부 문자열 matching만으로 만들지 말고 launcher/app startup failure result와 state를 기준으로 분류한다.

## 언어+버전 주의사항

- Detached self-reexec에서는 사용자가 보는 CLI와 internal mode가 섞이지 않도록 `--help` 출력과 internal dispatch 순서를 테스트한다.
- Restart window 계산은 system clock jump에 취약하지 않도록 timestamp list 또는 monotonic 기준을 고려한다.

## LLM 코딩 주의사항

- Simplicity: OS service/systemd/launchd 등록은 구현하지 않는다.
- Surgical: stop utility의 `stopping` state와 충돌하지 않도록 sentinel loop가 state를 매번 재확인한다.
- Safety: fatal startup 재시작 루프는 사용자 PC를 불필요하게 점유하므로 반드시 차단한다.

## 테스트 시나리오

### 정상 케이스

- Given: app child가 abnormal exit한다.
  When: sentinel이 exit를 감지한다.
  Then: backoff 후 app이 재시작되고 restart fields가 갱신된다.

### 예외 케이스

- Given: invalid production config다.
  When: daemon start 또는 sentinel restart가 발생한다.
  Then: sentinel은 무한 재시작하지 않고 fatal state를 기록한다.

### 경계값 테스트

- Given: 10분 내 5회 초과 restart가 발생한다.
  When: sentinel이 다음 restart를 판단한다.
  Then: 더 이상 재시작하지 않고 fatal 또는 stopped 상태로 기록한다.

- Given: state가 `stopping`이다.
  When: app child가 종료된다.
  Then: sentinel은 재시작하지 않고 종료한다.
