# Phase 5: PID 검증, Internal Shutdown, Native Stop

## Phase 5: PID 검증, internal shutdown, native stop

### 목표

PM2 stop을 native stop utility로 대체한다. Stop은 daemon state와 PID identity를 검증하고, sentinel을 먼저 중지한 뒤 loopback+token internal shutdown route로 workspace/CWD flush 완료를 확인한다.

- [ ] Phase 1 daemon state 기반 완료
- [ ] Phase 3 daemon running state와 readiness 완료
- [ ] `WorkspaceService.forceFlush()`와 `SessionManager.stopAllCwdWatching()` 현재 동작 확인

## 기존 코드 재사용 힌트

- `WorkspaceService.snapshotAllCwds()`와 `WorkspaceService.forceFlush()`를 shutdown completion의 실제 구현으로 재사용한다.
- 기존 signal handler의 timer cleanup 순서를 `performGracefulShutdown()`으로 이동한다.
- `server/src/middleware/debugCaptureGuards.ts`의 loopback 판단 패턴은 참고하되 forwarding header를 신뢰하지 않는 별도 shutdown guard를 명시한다.

## 코드 중복 주의

- signal handler와 internal shutdown route가 flush 로직을 각각 가지면 안 된다.
- PID validator를 stop utility 내부 private function으로만 만들면 sentinel cleanup과 start conflict 검증에서 중복된다.

## 언어+버전 주의사항

- Windows process command/cwd/start time 조회는 PowerShell/WMI/Node fallback이 필요할 수 있다. 조회 불가능 OS에서는 SRS의 fallback 조건을 따른다.
- HTTPS shutdown request는 self-signed certificate를 허용하되 대상은 `127.0.0.1`이어야 한다.

## LLM 코딩 주의사항

- Simplicity: kill fallback은 graceful failure로 명확히 표시하고 성공처럼 숨기지 않는다.
- Surgical: auth middleware, public API, Settings API에 shutdown token을 노출하지 않는다.
- Safety: PID 검증 실패 시 절대 종료 명령을 실행하지 않는다.

## 테스트 시나리오

### 정상 케이스

- Given: daemon state가 valid이고 app/sentinel PID 검증이 통과한다.
  When: `node stop.js`를 실행한다.
  Then: sentinel이 먼저 중지되고 app은 internal shutdown으로 flush 후 종료된다.

### 예외 케이스

- Given: state appPid가 무관 프로세스를 가리킨다.
  When: stop을 실행한다.
  Then: 해당 PID를 종료하지 않고 validation failure를 반환한다.

- Given: shutdown token이 없거나 invalid다.
  When: internal route를 호출한다.
  Then: shutdown이 거부되고 `/health`는 계속 응답한다.

- Given: `performGracefulShutdown()`이 10초 안에 완료되지 않는다.
  When: stop utility를 실행한다.
  Then: stop은 graceful failure와 non-zero exit를 반환하고 kill fallback을 성공으로 숨기지 않는다.

- Given: internal shutdown route가 비활성인 실행 조건이거나 flush가 실패한다.
  When: stop utility가 shutdown route를 호출한다.
  Then: `404` 또는 `500` response를 구분해 출력하고 app 상태를 검증한다.

### 경계값 테스트

- Given: foreground process가 실행 중이고 active daemon state는 없다.
  When: stop utility를 실행한다.
  Then: foreground는 계속 `/health`에 응답하고 stop은 not running을 출력한다.
