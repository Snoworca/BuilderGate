# Phase 2: Terminal Snapshot Storage Budget

## 목표

terminal snapshot이 localStorage 전체 quota를 장기적으로 독점하지 않도록 aggregate budget, LRU cleanup, quota retry를 추가한다.

## 변경 대상

| 파일 | 변경 |
|---|---|
| `frontend/src/components/Terminal/TerminalView.tsx` | snapshot 저장 전 budget 적용, quota retry |
| `frontend/src/utils/terminalSnapshot.ts` | snapshot metadata/listing/eviction helper |
| `frontend/tests/e2e/terminal-authority.spec.ts` 또는 신규 `frontend/tests/e2e/local-storage-quota.spec.ts` | snapshot cleanup 회귀 테스트 |

## 구현 상세

1. 기존 `SNAPSHOT_MAX_CONTENT_LENGTH = 2_000_000`는 단일 snapshot cap으로 유지 또는 보수적으로 조정한다.
2. 별도 aggregate budget 상수를 둔다.
3. 저장 전 현재 session snapshot을 제외한 오래된 snapshot을 budget 안에 들어오도록 삭제한다.
4. `localStorage.setItem(getTerminalSnapshotKey(sessionId), payload)`가 quota로 실패하면 오래된 snapshot을 추가 삭제하고 1회 재시도한다.
5. 손상된 JSON, schema 불일치, `savedAt`이 없는 오래된 형식은 삭제 우선순위를 높인다.

## LRU 정렬 기준

| 조건 | 정렬 |
|---|---|
| JSON parse 실패 | 가장 먼저 삭제 |
| `savedAt` 없음 | 오래된 항목으로 취급 |
| 현재 sessionId | 마지막까지 보존 |
| `terminal_snapshot_remove_*` marker | 오래된 marker부터 삭제 |

## 테스트

| ID | 시나리오 |
|---|---|
| TC-028-2A | 오래된 snapshot 여러 개가 budget을 초과하면 가장 오래된 항목부터 삭제 |
| TC-028-2B | 현재 저장하려는 session snapshot은 다른 session보다 우선 보존 |
| TC-028-2C | 손상된 snapshot JSON은 cleanup 중 제거 |
| TC-028-2D | quota retry 후 새 snapshot 저장 성공 |

## 완료 조건

- terminal snapshot 저장이 auth token 저장 실패를 유발할 정도로 누적되지 않는다.
- quota 실패는 warning으로 관측되며 terminal 기능을 중단하지 않는다.
- 기존 terminal authority/server replay 테스트가 회귀 없이 통과한다.
