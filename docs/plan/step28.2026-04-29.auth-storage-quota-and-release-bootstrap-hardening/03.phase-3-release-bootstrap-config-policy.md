# Phase 3: Release Bootstrap Config Policy

## 목표

공식 GitHub Release artifact가 항상 bootstrap-safe `config.json5`를 포함하도록 빌드 정책과 CI 검사로 고정한다.

## 현재 확인 결과

`v0.3.0` release asset은 정상이다. Windows ZIP 내부 루트 `config.json5`는 `auth.password: ""`, `jwtSecret: ""`이며 `server/config.json5`는 포함되지 않았다.

하지만 `tools/build-daemon-exe.js`의 현재 정책은 `server/config.json5`가 존재하면 산출물 `config.json5`로 그대로 복사한다. clean GitHub runner에서는 낮은 위험이지만, dirty local build나 self-hosted runner에서는 비밀번호가 산출물에 섞일 수 있다.

## 변경 대상

| 파일 | 변경 |
|---|---|
| `tools/build-daemon-exe.js` | release-safe config 생성 옵션 또는 profile 정책 |
| `tools/daemon/build-daemon-exe.test.js` | source config 존재 시 release profile은 bootstrap template 사용 검증 |
| `.github/workflows/release.yml` | archive 전 config inspection step 추가 |
| `README.md` 또는 release 문서 | fresh extraction과 기존 config 재사용 차이 명시 |

## 구현 상세

1. `copyRuntimeConfigFile()`에 `configPolicy` 옵션을 추가한다.
2. 기본 release profile은 `bootstrap-template` 정책을 사용한다.
3. 기존 source config 복사는 명시 옵션 `--include-user-config` 또는 local-only policy에서만 허용한다.
4. release workflow는 archive 전에 target output의 `config.json5`를 검사한다.
5. 검사 항목은 다음과 같다.

```text
auth.password == ""
auth.jwtSecret == ""
config file exists at artifact root
server/config.json5 is absent
```

## 테스트

| ID | 시나리오 |
|---|---|
| TC-028-3A | `server/config.json5`에 non-empty password가 있어도 release build output은 `auth.password: ""` |
| TC-028-3B | local explicit include option을 쓰면 기존 호환 정책대로 source config 복사 |
| TC-028-3C | release workflow inspection은 non-empty password artifact를 실패 처리 |
| TC-028-3D | macOS app runtime resources config도 bootstrap-safe |

## 완료 조건

- 공식 release build와 local release-equivalent build에서 user config 유출 가능성이 차단된다.
- CI가 artifact upload 전에 bootstrap config 계약을 검증한다.
- 사용자가 기존 폴더에 덮어쓴 경우와 fresh extraction의 차이가 문서화된다.
