# Architecture

## 현재 흐름

```text
Browser auth flow
  -> AuthContext login/bootstrap/2FA/refresh
  -> tokenStorage.setToken()
  -> localStorage.setItem('cws_auth_token')
  -> quota full이면 DOMException
  -> auth flow failure

Terminal view
  -> SerializeAddon.serialize()
  -> localStorage.setItem('terminal_snapshot_<sessionId>', large JSON)
  -> per snapshot 2,000,000 char cap
  -> total localStorage budget 없음
```

```text
GitHub Release build
  -> npm run build:<target>
  -> tools/build-daemon-exe.js
  -> dist/bin/<profile>-<version>/config.json5
  -> release workflow archives that directory
  -> packaged runtime reads EXE-neighbor config.json5
```

## 목표 흐름

```text
Auth token save
  -> attempt token/expires save
  -> if QuotaExceededError
       clear partial auth keys
       evict recoverable terminal snapshot cache
       retry exactly once
  -> if still failed
       clear partial auth keys
       throw observable error
```

```text
Terminal snapshot save
  -> build snapshot payload
  -> enforce per-snapshot cap
  -> enforce aggregate terminal snapshot budget
  -> attempt save
  -> if QuotaExceededError
       evict oldest terminal snapshots
       retry exactly once
  -> if still failed
       keep terminal running and log warning
```

```text
Release first-run
  -> release build produces bootstrap-safe config.json5
  -> artifact inspection rejects non-empty auth.password
  -> fresh extraction starts packaged runtime
  -> GET /api/auth/bootstrap-status
  -> setupRequired: true, requesterAllowed: true for localhost
```

## 컴포넌트 책임

| 컴포넌트 | 현재 책임 | 변경 책임 |
|---|---|---|
| `frontend/src/services/tokenStorage.ts` | token/expires direct localStorage read/write | quota cleanup retry, partial auth state 방지 |
| `frontend/src/utils/terminalSnapshot.ts` | snapshot key 생성/삭제 marker | snapshot listing, LRU metadata, quota cleanup utility |
| `frontend/src/components/Terminal/TerminalView.tsx` | terminal snapshot 저장/복원 | aggregate budget 적용, quota retry |
| `tools/build-daemon-exe.js` | runtime files와 config 복사 | release-safe config policy와 검증 |
| `.github/workflows/release.yml` | target build/archive/release upload | artifact config inspection 추가 |
| `tools/daemon/launcher.js` | foreground/daemon child launch | packaged internal mode 진입 안정화 |
| `tools/start-runtime.js` | CLI parsing과 launcher 연결 | packaged foreground internal app env 전달 검증 |

## 데이터 보존 원칙

- 인증 복구를 위해 자동 삭제할 수 있는 데이터는 recoverable cache인 `terminal_snapshot_*`, `terminal_snapshot_remove_*`로 제한한다.
- `cws_auth_token`, `cws_auth_expires`, workspace id, mosaic layout, tab state, font setting은 auth quota cleanup에서 삭제하지 않는다.
- terminal snapshot은 서버 권위 상태가 아니라 재표시 편의 캐시이므로 오래된 항목부터 삭제 가능하다.
- 삭제와 재시도는 관측 가능하게 `console.warn`으로 남기되, 사용자의 terminal interaction 자체를 중단하지 않는다.

## 릴리즈 bootstrap 원칙

- 공식 release artifact는 개발자 로컬 config를 포함하면 안 된다.
- release artifact의 `config.json5`는 배포 직후 bootstrap 상태여야 한다.
- 사용자가 기존 폴더에 덮어쓰는 경우의 기존 `config.json5` 보존은 release asset 문제가 아니므로, 문서와 smoke test에서 fresh extraction 기준을 명확히 한다.
- packaged executable의 내부 app/sentinel 실행은 CLI arg가 아니라 명시적 env mode로 진입하는 편이 pkg 런타임에서 더 안정적이다.
