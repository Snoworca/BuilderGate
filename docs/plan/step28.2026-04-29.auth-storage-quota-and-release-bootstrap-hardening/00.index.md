# Step 28: Auth Storage Quota And Release Bootstrap Hardening

## 목표

`Failed to execute 'setItem' on 'Storage': Setting the value of 'cws_auth_token' exceeded the quota` 오류를 재현 가능하게 막고, GitHub Release에서 내려받은 EXE의 최초 실행이 항상 비밀번호 초기 설정 상태로 진입하는지 검증 가능한 계약으로 고정한다.

## 조사 요약

| 영역 | 결론 | 근거 |
|---|---|---|
| `cws_auth_token` quota 오류 | JWT 자체가 큰 것이 아니라 동일 origin localStorage가 이미 가득 찬 상태에서 작은 token 저장이 실패한 것이다 | `AuthService` JWT payload는 `sub`, `iat`, `exp`, `jti`만 포함한다 |
| 주요 용량 소비 후보 | `terminal_snapshot_*`가 세션별 최대 2,000,000자까지 저장되며 전체 budget/LRU 정리가 없다 | `TerminalView.tsx`, `terminalSnapshot.ts` |
| 최신 GitHub Release 자산 | `v0.3.0` Windows asset의 루트 `config.json5`는 `auth.password: ""`, `jwtSecret: ""`로 bootstrap-ready 상태다 | GitHub Release API 및 ZIP 직접 검증 |
| 릴리즈 최초 실행 불일치 후보 | 기존 daemon/기존 추출 폴더/환경 변수 config override 또는 packaged internal entrypoint 실행 문제가 원인일 수 있다 | packaged `BuilderGate.exe --foreground`에서 `--internal-app` module 해석 실패 확인 |
| 구조적 패키징 위험 | 빌드 시 `server/config.json5`가 있으면 현재 스크립트가 산출물 `config.json5`로 그대로 복사한다 | `tools/build-daemon-exe.js` |

## 최종 성공 기준

- EXE 또는 source 실행 환경에서 localStorage가 terminal snapshot으로 가득 차도 login/bootstrap/refresh token 저장이 복구된다.
- 복구 가능한 snapshot이 없거나 재시도도 실패하면 auth key가 부분 저장되지 않고 오류가 관측 가능하게 남는다.
- terminal snapshot 저장은 localStorage 전체 quota를 장기간 독점하지 않는다.
- release artifact의 `config.json5`는 최초 배포 기준 `auth.password: ""`, `jwtSecret: ""` 상태임이 자동 검증된다.
- fresh release extraction에서 서버가 새 config를 사용하고 `/api/auth/bootstrap-status`가 초기 비밀번호 설정 필요 상태를 반환한다.
- packaged foreground와 daemon mode가 `--internal-app`/sentinel 진입 문제 없이 동작한다.

## 범위 밖

- browser storage를 IndexedDB로 전면 이전하는 대형 마이그레이션
- localStorage에 저장된 모든 사용자 레이아웃/설정 데이터의 통합 garbage collection
- 원격 IP bootstrap 정책 확대 또는 reverse proxy trust model 변경
- updater 기능 구현
