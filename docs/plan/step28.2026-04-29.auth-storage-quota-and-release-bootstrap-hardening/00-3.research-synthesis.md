# Research Synthesis

## Subagent 1: frontend storage quota

결론: `cws_auth_token`은 원인이 아니라 실패 지점이다. 현재 JWT는 매우 작고, quota를 채우는 유력 후보는 `terminal_snapshot_*` 대형 JSON이다. `tokenStorage.setToken()`에는 quota cleanup/retry가 없고, `TerminalView`에는 전체 localStorage budget이 없다.

핵심 경로:

| 파일 | 의미 |
|---|---|
| `frontend/src/services/tokenStorage.ts` | `cws_auth_token`, `cws_auth_expires` 직접 저장 |
| `frontend/src/contexts/AuthContext.tsx` | login/bootstrap/2FA/refresh 후 token 저장 |
| `frontend/src/components/Terminal/TerminalView.tsx` | idle, unmount, pagehide 시 terminal snapshot 저장 |
| `frontend/src/utils/terminalSnapshot.ts` | `terminal_snapshot_*`, `terminal_snapshot_remove_*` prefix |

권고: auth 저장 경로에는 recoverable snapshot cleanup 후 1회 retry를 넣고, terminal snapshot 저장 경로에는 aggregate budget과 LRU cleanup을 넣는다.

## Subagent 2: GitHub Release bootstrap

결론: 최신 `v0.3.0` GitHub Release asset 자체는 bootstrap-ready 상태다. Windows ZIP 안의 루트 `config.json5`에는 `auth.password: ""`, `jwtSecret: ""`가 포함되어 있었고, `server/config.json5`는 artifact에 없었다.

확인된 asset:

| Asset | SHA-256 |
|---|---|
| `BuilderGate-win-amd64-0.3.0.zip` | `230e49ac351cc8e682216d402f137787b67970a06c8ea8ba5cbd3e95813c58c9` |
| `BuilderGate-linux-amd64-0.3.0.tar.gz` | `d334cfbe6eb19aafeb0b94cc58c3fbcd5c5b5c3a817442287c221baca6890417` |
| `BuilderGate-linux-arm64-0.3.0.tar.gz` | `d93cae8af84dd6ead2622625adae5ba341a7ef963567ea4f841792505a79dab7` |
| `BuilderGate-macos-arm64-0.3.0.zip` | `650f759ad655f61b0921c365505a3efe7e13a7bdc55cba2673873e827b763da5` |
| `BuilderGate-win-arm64-0.3.0.zip` | `3de0f6866746f3bdd899575766bc42c98a86eaf9d0d5c2b936089be82fb5d32d` |

남은 위험:

- `tools/build-daemon-exe.js`는 source `server/config.json5`가 있으면 산출물에 복사한다.
- 기존 추출 폴더를 재사용하거나 기존 daemon이 떠 있으면 새 release의 config가 사용되지 않을 수 있다.
- `BUILDERGATE_CONFIG_PATH`가 기존 config를 가리키면 fresh asset과 다른 결과가 난다.

## Subagent 3: plan structure

결론: 기존 `docs/plan/stepN.YYYY-MM-DD.slug` 구조와 phase/verification 문서 분리를 따른다. Playwright 및 수동 검증은 프로젝트 규칙에 따라 `https://localhost:2002`를 기준으로 한다.

## 직접 추가 검증: packaged internal entrypoint

fresh release Windows ZIP에서 `config.json5`가 bootstrap-ready인 것은 확인됐지만, packaged foreground 실행 중 다음 오류가 관찰되었다.

```text
Error: Cannot find module '<release-dir>\\--internal-app'
```

이는 packaged child command가 `BuilderGate.exe --internal-app` 형태로 실행될 때 pkg 런타임이 `--internal-app`를 내부 모드 플래그가 아니라 module path처럼 해석할 수 있음을 보여준다. `tools/start-runtime.js`에는 `process.pkg && BUILDERGATE_INTERNAL_MODE === 'app'` 경로가 이미 있으므로, packaged child launch를 env 중심으로 바꾸는 계획이 타당하다.

## 현재 판단

사용자가 경험한 "GitHub release에서 다운로드 후 첫 실행했는데 비밀번호 초기화가 되지 않음"은 release asset의 config 내용 문제라기보다 다음 중 하나일 가능성이 높다.

1. 기존 2001/2002 daemon이 계속 떠 있어서 새 EXE가 아니라 기존 서버에 접속했다.
2. 기존 추출 폴더나 기존 `config.json5`를 재사용했다.
3. `BUILDERGATE_CONFIG_PATH`가 기존 config를 가리켰다.
4. packaged internal entrypoint 실패로 새 서버가 정상 부팅하지 못했고, 브라우저는 stale 서버나 실패 상태를 보았다.

따라서 구현은 frontend quota recovery와 release bootstrap policy뿐 아니라 packaged runtime first-run smoke까지 포함해야 한다.
