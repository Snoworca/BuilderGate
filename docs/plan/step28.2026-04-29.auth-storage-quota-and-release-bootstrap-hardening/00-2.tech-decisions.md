# Technical Decisions

## ADR-001: `cws_auth_token` 오류의 1차 복구 대상은 terminal snapshot cache로 제한한다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | token은 수백 바이트 수준이지만 `terminal_snapshot_*`는 세션별 대형 JSON을 저장한다. |
| 결정 | auth 저장 중 quota가 발생하면 `terminal_snapshot_*`, `terminal_snapshot_remove_*`만 정리하고 1회 재시도한다. |
| 영향 | 로그인/초기 설정 성공 가능성이 높아지고, 사용자 설정/레이아웃 데이터 삭제를 피한다. |

## ADR-002: auth token/expires 저장은 partial state를 남기지 않는다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | token 저장 성공 후 expires 저장 실패 또는 반대 순서 실패가 발생하면 불완전한 인증 상태가 남을 수 있다. |
| 결정 | `setToken()` 실패 시 항상 `cws_auth_token`, `cws_auth_expires`를 제거한 뒤 재시도 또는 throw한다. |
| 영향 | 실패가 명확해지고 stale token 기반의 애매한 상태를 줄인다. |

## ADR-003: quota 회귀 테스트는 실제 quota 소진 대신 합성 DOMException을 우선한다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | browser별 localStorage quota는 환경 차이가 크고 실제로 채우는 테스트는 느리고 불안정하다. |
| 결정 | Playwright에서 `Storage.prototype.setItem` monkey patch로 `QuotaExceededError`를 합성한다. |
| 영향 | deterministic한 회귀 테스트를 빠르게 유지할 수 있다. |

## ADR-004: release build는 bootstrap-safe config를 기본 정책으로 둔다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | 현재 공식 release asset은 정상이나, 빌드 시 `server/config.json5`가 존재하면 산출물에 그대로 복사되는 구조적 위험이 있다. |
| 결정 | release profile에서는 source user config를 복사하지 않고 bootstrap template을 생성한다. 로컬 config 포함은 명시 옵션으로만 허용한다. |
| 영향 | dirty local build/self-hosted runner에서도 비밀번호가 릴리즈 산출물에 섞이는 위험을 차단한다. |

## ADR-005: packaged internal mode는 env 중심으로 진입한다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | `BuilderGate.exe --foreground` 검증 중 child `BuilderGate.exe --internal-app`가 `--internal-app`를 모듈 경로로 해석하는 문제가 관찰되었다. |
| 결정 | packaged child process는 args 없이 같은 EXE를 실행하고 `BUILDERGATE_INTERNAL_MODE=app|sentinel` env로 내부 모드를 선택한다. |
| 영향 | pkg 런타임의 argv/module 해석 차이를 피한다. process discovery 로직은 env/state/log 기반 보강이 필요할 수 있다. |

## ADR-006: 최초 실행 검증은 fresh extraction을 기준으로 자동화한다

| 항목 | 결정 |
|---|---|
| 상태 | Proposed |
| 맥락 | 기존 실행 폴더, 기존 daemon, `BUILDERGATE_CONFIG_PATH` override가 있으면 release asset이 정상이어도 bootstrap이 뜨지 않을 수 있다. |
| 결정 | release smoke는 임시 새 디렉터리에 ZIP을 풀고, 기존 2001/2002 listener가 없는 상태에서 `/api/auth/bootstrap-status`를 확인한다. |
| 영향 | 사용자의 실제 문제를 재현할 때 stale process와 stale config 여부를 분리해 판단할 수 있다. |
