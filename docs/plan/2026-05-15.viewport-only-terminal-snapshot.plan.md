# Viewport-Only Terminal Snapshot Implementation Plan

## 1. 개요

이 계획은 긴 Codex 대화가 쌓인 터미널에서 브라우저 새로고침, WebSocket 재연결, 같은 session 재구독이 발생할 때 과거 scrollback 전체가 다시 xterm에 주입되는 문제를 줄인다. 서버 `screen-snapshot`과 브라우저 `terminal_snapshot_*` fallback 모두 current viewport 복원 payload만 다루며, 서버 scrollback lazy loading은 범위 밖이다.

스펙 기준은 `FR-SESS-007`, `FR-SESS-008`, `FR-SESS-015`, `FR-SESS-016`, `FR-SESS-017`, `FR-SESS-018`, `FR-WS-005`, `FR-WS-006`, `FR-WS-007`, `AC-009`, `AC-017`, `AC-020`, `AC-023`이다. 특히 `AC-023`은 reload/reconnect/resubscribe 후 최신 viewport marker는 보여야 하지만, 과거 scrollback 전체 조회를 보장하지 않는다고 명시한다.

### Feasibility 요약

| 영역 | 난이도 | 판단 |
|---|---|---|
| 서버 headless snapshot | Medium | `@xterm/addon-serialize` 호출 옵션을 viewport-only로 고정하고 UTF-8 byte cap을 적용하면 영향면이 좁다. |
| degraded fallback | High | 현재 `degradedReplayBuffer`가 tail scrollback을 보낼 수 있어 테스트 기대값까지 함께 바꿔야 한다. |
| 브라우저 localStorage fallback | High | 기존 schema v1 payload가 이미 full scrollback일 수 있으므로 schema v2 전환과 legacy restore 금지가 필요하다. |
| E2E 검증 | Medium | `screen-snapshot.data`, `terminal_snapshot_*`, visible DOM을 각각 검사해야 full replay를 놓치지 않는다. |

Infeasible 항목은 없다.

## 2. 범위와 비범위

### 구현 범위

- 서버 authoritative `screen-snapshot`을 viewport-only payload로 직렬화한다.
- snapshot byte limit은 UTF-8 byte 기준으로 판정한다.
- degraded/fallback 경로는 scrollback tail을 snapshot처럼 보내지 않는다.
- 브라우저 localStorage terminal snapshot은 schema v2 viewport-only payload만 저장하고 복원한다.
- legacy schema v1 snapshot은 복원하지 않고 제거한다.
- reload, WebSocket reconnect, resubscribe, replay refresh, Grid repair 경로 회귀 테스트를 추가한다.

### 구현하지 않는 범위

- 서버 scrollback lazy loading API.
- 사용자가 과거 scrollback 위치로 스크롤해 둔 상태까지 정확히 복원하는 range snapshot.
- WebSocket `screen-snapshot` required field 추가.
- UI 레이아웃, 아이콘, 레이블, 색상 변경.
- session `idle`/`running` 판정 로직 변경.

## 3. 프로젝트 온보딩 컨텍스트

BuilderGate는 Express/WebSocket/node-pty 서버와 React/xterm.js 프런트엔드로 구성된 로컬 터미널 작업 환경이다. 서버는 `SessionManager`에서 PTY 출력과 headless xterm 상태를 관리하고, `WsRouter`는 subscribe/replay ACK handshake와 output/input queue를 관리한다. 프런트엔드는 `TerminalContainer`가 WebSocket snapshot을 받고, `TerminalView`가 xterm 인스턴스와 localStorage fallback snapshot을 다룬다.

이번 작업의 핵심 경로는 다음과 같다.

| 경로 | 역할 |
|---|---|
| `server/src/utils/headlessTerminal.ts` | headless xterm snapshot, screen repair serialization |
| `server/src/services/SessionManager.ts` | snapshot cache, degraded state, observability |
| `server/src/ws/WsRouter.ts` | subscribe/resubscribe replay, ACK 대기, refresh replay |
| `server/src/test-runner.ts` | 서버 회귀 테스트 runner |
| `frontend/src/utils/terminalSnapshot.ts` | localStorage snapshot key, quota, eviction 유틸 |
| `frontend/src/components/Terminal/TerminalView.tsx` | xterm serialize/restore와 local fallback |
| `frontend/src/components/Terminal/TerminalContainer.tsx` | `screen-snapshot` 적용 경로 |
| `frontend/tests/e2e` | HTTPS 2002 대상 Playwright 검증 |

검증 대상 URL은 `https://localhost:2002`이며, Vite dev server `http://localhost:2003`을 직접 대상으로 삼지 않는다.

## 4. 용어집

| 용어 | 정의 |
|---|---|
| viewport-only snapshot | 현재 터미널 viewport를 재구성하는 데 필요한 ANSI payload. normal buffer scrollback 전체를 포함하지 않는다. |
| authoritative snapshot | 서버 headless xterm 상태에서 생성한 `screen-snapshot` payload. |
| fallback snapshot | 서버 headless snapshot이 없거나 비어 있을 때 사용하는 복구 경로. 이 계획에서는 검증된 local schema v2 viewport payload만 허용한다. |
| degraded headless | headless xterm write, resize, serialize 실패로 서버가 authoritative snapshot을 만들 수 없는 상태. |
| replay handshake | 서버가 `screen-snapshot`을 보내고 클라이언트 `screen-snapshot:ready` ACK 전까지 output/input을 큐잉하는 흐름. |
| screen repair | Grid repair 등에서 full snapshot 대신 active viewport row patch를 주고받는 복구 경로. |

#### TASK-P1-001. Make headless serialization viewport-only and byte-bounded

**REQ**: `FR-SESS-007`, `FR-SESS-008`, `FR-SESS-016`, `FR-SESS-018`, `AC-009`, `AC-023`

**Feasibility**: Medium. Existing serializer already accepts `ISerializeOptions`; the change is contract hardening, byte accounting, and tests.

**Owns**

- `server/src/utils/headlessTerminal.ts`
- `server/src/test-runner.ts`

**Reads**

- `server/src/services/SessionManager.ts`
- `server/src/types/ws-protocol.ts`

**Source Anchors**

- `docs/srs/buildergate.srs.md:211-222`
- `docs/srs/buildergate.srs.md:653-667`
- `server/src/utils/headlessTerminal.ts:131-152`
- `server/src/utils/headlessTerminal.ts:154-205`
- `server/src/test-runner.ts:164-168`
- `server/src/test-runner.ts:3867-3895`

**Signatures**

```ts
export const VIEWPORT_ONLY_SERIALIZE_OPTIONS: ISerializeOptions = { scrollback: 0 };

export function serializeHeadlessTerminal(
  state: HeadlessTerminalState,
  maxSnapshotBytes: number,
  options?: ISerializeOptions,
): SerializedHeadlessSnapshot;
```

**Implementation Guide**

1. Add `VIEWPORT_ONLY_SERIALIZE_OPTIONS` in `headlessTerminal.ts`.
2. In `serializeHeadlessTerminal`, call `state.serializeAddon.serialize(options ?? VIEWPORT_ONLY_SERIALIZE_OPTIONS)`.
3. Keep alternate buffer serialization enabled. Do not pass `excludeAltBuffer`.
4. Replace `serialized.length > maxSnapshotBytes` with `Buffer.byteLength(serialized, 'utf8') > maxSnapshotBytes`.
5. Return `data: ''` and `truncated: true` on byte cap exceed, matching the existing oversized snapshot behavior.
6. Add `testHeadlessSnapshotViewportOnlyLongScrollback` in `server/src/test-runner.ts`.
7. Register the test near the existing headless snapshot tests.

**Reference Pattern**

- `serializeHeadlessScreenRepair` already reads only active viewport rows: `server/src/utils/headlessTerminal.ts:154-205`.
- `testHeadlessScreenRepairViewportOnly` already checks old lines are absent after long output: `server/src/test-runner.ts:3897-3925`.

**Rationale**

The server `screen-snapshot` handshake is still useful for fast restore, but the payload must no longer reconstruct normal buffer scrollback. A byte cap based on JS string length is not enough when multibyte characters are present.

**Pitfalls**

- Do not use `serialize()` without options in any server snapshot path.
- Do not change screen repair behavior while editing snapshot serialization.
- Do not trim ANSI escape sequences manually; rely on xterm serializer output and byte cap.

**Test Instructions**

- Success: write 500 numbered lines, snapshot contains latest viewport marker and excludes the first marker.
- Failure: force a small `maxSnapshotBytes`, assert `truncated === true` and `data === ''`.
- Edge: include multibyte Korean text and assert cap uses `Buffer.byteLength`.
