# Phase 3: 서버 통합 — index.ts + TwoFactorService + services/index.ts

## 목표

TOTPService를 서버 엔트리포인트에서 초기화하고, TwoFactorService에 `stage` 파라미터를 추가한다.
기존 COMBO-1(비밀번호만), COMBO-2(이메일 OTP)가 계속 정상 작동하는지 확인한다.

## 선행 조건

- Phase 1 완료 (타입/스키마 정의)
- Phase 2 완료 (TOTPService 구현)

## LLM 코딩 주의사항

- **Simplicity**: `totpService` 초기화는 기존 `twoFactorService` 초기화 패턴을 그대로 복사 + 수정. 추상화/팩토리 불필요.
- **Surgical**: `TwoFactorService`에서 `createPendingAuth`의 `stage` 파라미터만 추가. 내부 이메일 OTP 로직은 건드리지 않음.
- 배너 수정 시 기존 배너 라인 길이(이미 고정 폭) 유지. 문자 수 맞춰 공백 조정.

### Phase 3-4: index.ts — 배너 2FA 상태 문구 (NFR-401)

배너에서 2FA 상태 라인을 찾아 동적으로 생성:
```typescript
// 2FA + TOTP 상태 문구 생성
function get2FAStatusLine(config: Config): string {
  const emailEnabled = config.twoFactor?.enabled && !!(config.twoFactor.email && config.twoFactor.smtp);
  const totpEnabled = config.twoFactor?.enabled && config.twoFactor?.totp?.enabled;

  if (!config.twoFactor?.enabled) return '2FA:  Disabled';
  if (emailEnabled && totpEnabled) return '2FA:  Enabled (Email OTP + TOTP)';
  if (totpEnabled) return '2FA:  Enabled (TOTP only)';
  return '2FA:  Enabled (Email OTP)';
}
```

### Phase 3-5: TwoFactorService.ts — createPendingAuth 리팩토링

기존 `createPendingAuth(email: string): Promise<PendingAuth>` 시그니처를 완전히 교체:
> **중요**: 기존 메서드는 이메일 전송까지 수행함. 새 설계에서는 OTP 생성만 담당하고 이메일 전송은 `sendOTP()`로 분리.

```typescript
// 변경 전 (기존)
async createPendingAuth(email: string): Promise<PendingAuth>

// 변경 후 (신규)
createPendingAuth(stage: TwoFAStage = 'email'): { tempToken: string; otp: string } {
  this.cleanupExpiredOTPs();
  if (this.otpStore.size >= MAX_OTP_STORE_SIZE) this.removeOldestOTP();

  const otp = this.generateOTP();
  const tempToken = crypto.randomUUID();
  
  const otpData: OTPData = {
    otp,
    email: this.config.email ?? '',
    expiresAt: Date.now() + (this.config.otpExpiryMs || 300000),
    attempts: 0,
    stage,             // [신규] stage 필드
  };
  
  this.otpStore.set(tempToken, otpData);
  return { tempToken, otp };
}
```

### Phase 3-5b: TwoFactorService.ts — 신규 메서드 5개

```typescript
// 이메일 전송 (실패 시 throw — FR-501 폴백용)
async sendOTP(otp: string): Promise<void> {
  const email = this.config.email;
  if (!email) throw new Error('[2FA] No email configured');
  const sent = await this.sendOTPEmail(email, otp);
  if (!sent) throw new Error('[2FA] Failed to send OTP email');
}

// smtp + email 설정 여부 (COMBO 결정 로직용)
hasEmailConfig(): boolean {
  return !!(this.config.email && this.config.smtp);
}

// OTP 데이터 조회 (authRoutes verify 핸들러용)
getOTPData(tempToken: string): OTPData | undefined {
  return this.otpStore.get(tempToken);
}

// OTP 데이터 삭제 (인증 성공 또는 최대 시도 초과 시)
invalidatePendingAuth(tempToken: string): void {
  this.otpStore.delete(tempToken);
}

// stage 변경 (COMBO-4 중간 단계 전환용)
updateStage(tempToken: string, stage: TwoFAStage): void {
  const otpData = this.otpStore.get(tempToken);
  if (otpData) { otpData.stage = stage; }
}
```

## 테스트 시나리오

### 정상 케이스
- Given: `twoFactor.enabled = false`, `twoFactor.totp.enabled = false`
  When: 서버 시작
  Then: 배너에 "2FA:  Disabled" 출력, `totpService = undefined`

### 예외 케이스
- Given: `twoFactor.enabled = true`, `twoFactor.totp.enabled = true`, `data/totp.secret` 없음
  When: 서버 시작
  Then: TOTPService 초기화 시 파일 자동 생성, QR 코드 출력, 서버 정상 기동

- Given: `twoFactor.enabled = true`, `twoFactor.totp.enabled = true`, `data/totp.secret` 손상/복호화 실패
  When: 서버 시작 → `totpService.initialize()` throw → catch 블록 진입
  Then: 서버는 계속 기동, 단 `totpService.isRegistered() === false`
  → 이후 `POST /api/auth/login` 시 503 반환 (FR-401): "TOTP is enabled but not configured..."
  ⚠️ 이 시나리오를 Phase 3 테스트로 직접 검증: initialize()가 throw해도 서버 기동 중단이 되지 않도록 try/catch 블록 확인

### 경계값 테스트
- Given: 기존 이메일 OTP 설정 (twoFactor.enabled = true, smtp 있음, totp 없음)
  When: 서버 시작 후 `POST /api/auth/login`
  Then: 기존 이메일 OTP 플로우 그대로 동작 (하위 호환 확인)
