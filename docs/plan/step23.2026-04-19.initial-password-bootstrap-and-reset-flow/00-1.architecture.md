# Architecture

## 1. 현재 구조 요약

현재 인증/설정 부트스트랩은 다음과 같은 한계를 가진다.

1. `server/src/utils/config.ts`는 `config.json5`가 없으면 `config.json5.example`을 복사한다.
2. 예제 파일에는 가짜 기본 비밀번호 문자열이 들어 있고, 이 값은 첫 실행 시 자동 암호화된다.
3. `server/src/services/AuthService.ts`는 비밀번호가 없으면 단지 “인증 실패” 상태로만 동작한다.
4. `server/src/routes/authRoutes.ts`는 bootstrap state를 별도로 노출하지 않고, 프런트는 단순 로그인 폼만 제공한다.
5. `tools/start-runtime.js`는 포트와 빌드/pm2만 다루며 인증 리셋 기능이 없다.

결과적으로:

- 기본 비밀번호 하드코딩 제거가 어렵다.
- 비밀번호 미설정 상태가 사용자 친화적으로 표현되지 않는다.
- 운영자가 비밀번호를 잊었을 때 웹 bootstrap 화면으로 되돌리는 절차가 없다.

## 2. 목표 구조

### 2.1 Config bootstrap

```text
No config.json5
  -> ensureConfigExists()
  -> render built-in JSON5 template
  -> auth.password = ""
  -> bootstrap.allowedIps = []
  -> write config.json5
```

핵심은 “예제 파일은 문서 예시로만 남기고, 실제 실행 bootstrap source는 코드 내부 템플릿”으로 분리하는 것이다.

### 2.2 Server bootstrap contract

```text
GET /api/auth/bootstrap-status
  -> password missing?
  -> requester allowed?
  -> return {
       setupRequired,
       requesterAllowed,
       allowPolicy: 'localhost' | 'allowlist' | 'denied'
     }
```

```text
POST /api/auth/bootstrap-password
  -> password missing?
  -> requester allowed?
  -> validate password + confirm
  -> encrypt and persist auth.password
  -> authService.updateRuntimeConfig()
  -> issue JWT
```

### 2.3 Frontend auth branching

```text
AuthGuard unauthenticated
  -> AuthContext boot check
  -> if setupRequired && requesterAllowed
       render BootstrapPasswordForm
     else
       render LoginForm
```

### 2.4 Operational reset flow

```text
start.sh/start.bat --reset-password
  -> tools/start-runtime.js
  -> rewrite only auth.password = ""
  -> optional runtime override: --bootstrap-allow-ip 192.168.0.10
  -> pm2 start with env override
```

## 3. 권장 책임 분리

| 컴포넌트 | 책임 | 비고 |
|---|---|---|
| `config.ts` | 초기 JSON5 템플릿 생성, empty password canonicalization | 파일 생성 책임 |
| `AuthService` | `hasConfiguredPassword()`, `requiresBootstrapSetup()` 노출 | 인증 상태 판단 책임 |
| `authRoutes.ts` | bootstrap status / bootstrap password route 제공 | public bootstrap 계약 |
| `SettingsService` | 기존 비밀번호 변경 흐름 유지 | bootstrap과 일반 설정 변경 분리 |
| `AuthContext` | bootstrap 상태 조회, 성공 시 auth state 반영 | 프런트 분기 책임 |
| `BootstrapPasswordForm` | 비밀번호/확인 입력과 클라이언트 검증 | 초기 setup 전용 UI |
| `tools/start-runtime.js` | reset-password, temporary allowlist flag 처리 | 운영 도구 책임 |

## 4. 권장 데이터 흐름

### 4.1 First run, localhost bootstrap

```text
Server first boot
  -> config generated with empty password
  -> browser opens /
  -> frontend requests bootstrap-status
  -> setupRequired=true, requesterAllowed=true
  -> BootstrapPasswordForm displayed
  -> user submits password + confirm
  -> password encrypted and persisted
  -> JWT issued
  -> app enters authenticated state
```

### 4.2 Remote bootstrap via allowlist

```text
Operator runs:
  start.sh --reset-password --bootstrap-allow-ip 192.168.0.50

Server boot
  -> auth.password empty
  -> runtime allowlist includes 192.168.0.50
  -> requester from 192.168.0.50 sees bootstrap form
  -> all other remote IPs denied
```

### 4.3 Normal login after setup

```text
Password already configured
  -> bootstrap-status returns setupRequired=false
  -> LoginForm remains active
  -> existing /api/auth/login flow unchanged
```

## 5. IP access policy

권장 정책은 다음 우선순위로 평가한다.

1. `127.0.0.1`, `::1`, `::ffff:127.0.0.1`는 항상 허용
2. `bootstrap.allowedIps`에 명시된 주소 허용
3. 런처 env override `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS` 추가 허용
4. 그 외 요청은 bootstrap denied

주의:

- `X-Forwarded-For`는 신뢰 프록시 정책이 확정되기 전까지 기본적으로 사용하지 않는다.
- CORS가 아니라 서버측 requester IP 검사로 보안 결정을 내려야 한다.

## 6. 비밀번호 canonical state

외부 입력은 `null`, `""`, 미존재를 모두 허용할 수 있으나 내부 저장과 런타임 비교는 하나로 통일해야 한다.

권장 canonical form:

```json5
auth: {
  password: ""
}
```

이 방식의 장점:

- 현재 `AuthConfig.password: string` 타입과 스키마를 크게 흔들지 않는다.
- 런처 리셋, bootstrap status, runtime config 비교가 단순해진다.
- “정확히 비어 있음” 상태가 명확하다.

## 7. bootstrap 성공 응답 권장안

초기 비밀번호 설정 성공 시 응답은 `success only`보다 `auto-login`이 더 자연스럽다.

권장 응답:

```json
{
  "success": true,
  "token": "...",
  "expiresIn": 1800000
}
```

이유:

- 사용자가 bootstrap 후 다시 로그인 폼에 같은 비밀번호를 입력할 필요가 없다.
- bootstrap requester는 이미 localhost/allowlist로 제한된 trusted requester다.
- 프런트 `AuthContext`는 기존 login success branch를 재사용할 수 있다.

## 8. 실패 정책

### 8.1 bootstrap denied

- 상태 API는 `setupRequired=true`, `requesterAllowed=false`를 반환한다.
- 프런트는 비밀번호 설정 폼 대신 “이 주소에서는 초기 설정 불가” 안내를 표시한다.

### 8.2 bootstrap route misuse

- 이미 비밀번호가 설정된 상태에서 호출되면 `409` 또는 `400` 계열로 명시적으로 거절한다.
- 비밀번호/확인 불일치, 최소 길이 미달 등은 validation error로 반환한다.

### 8.3 reset-password file mutation failure

- 런처는 pm2 start 전에 즉시 실패해야 한다.
- 부분 수정 후 손상 위험을 막기 위해 “auth.password path만” 안전하게 찾아 바꾸는 helper가 필요하다.

## 9. 테스트 구조

| 레이어 | 목표 | 권장 방식 |
|---|---|---|
| Server unit/integration | empty password, bootstrap status, IP policy, password persistence | `server/src/test-runner.ts` |
| Frontend E2E | setup form 노출, confirm mismatch, bootstrap success auto-login | Playwright |
| Launcher smoke | `--reset-password`, `--bootstrap-allow-ip` 옵션 동작 | shell/batch manual smoke |
| Regression | 기존 login, 2FA, settings password rotation 유지 | existing auth/settings tests |
