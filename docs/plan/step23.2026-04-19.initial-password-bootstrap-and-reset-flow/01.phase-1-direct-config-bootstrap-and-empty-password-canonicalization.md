# Phase 1: Direct Config Bootstrap And Empty-Password Canonicalization

## 목표

`config.json5`가 없을 때 더 이상 `config.json5.example`을 복제하지 않고, 서버가 직접 bootstrap 가능한 JSON5 템플릿을 생성하도록 만든다. 동시에 비밀번호 미설정 상태를 `auth.password: ""`로 정규화한다.

## 대상 파일

- `server/src/utils/config.ts`
- `server/src/types/config.types.ts`
- `server/src/schemas/config.schema.ts`
- 신규 helper 후보: `server/src/utils/configTemplate.ts`
- `server/config.json5.example`
- `README.md`

## 구현 항목

1. `config.ts`에 built-in JSON5 template renderer 또는 dedicated template helper를 추가한다.
2. 최초 생성 템플릿의 `auth.password`는 빈 값으로 둔다.
3. bootstrap allowlist를 위한 `bootstrap.allowedIps: []` 설정 섹션을 타입/스키마에 추가한다.
4. 로드 시 `auth.password === null` 또는 미존재 입력을 `""`로 정규화하는 helper를 추가한다.
5. password auto-encryption은 non-empty plaintext일 때만 실행되도록 유지한다.
6. `config.json5.example`는 문서 예시로 유지하되 “실행 시 복제 source가 아님”을 명시한다.

## 코드 재사용/주의

- 현재 `ensureConfigExists()`는 example file 복제에 의존한다. 이 지점이 직접 bootstrap renderer로 바뀌어야 한다.
- 현재 password encryption logic은 `auth.password`가 truthy일 때만 동작한다. empty canonical state를 넣어도 부작용이 없어야 한다.
- 기존 OS-aware PTY bootstrap 로직은 유지되어야 하므로, 새 template renderer도 platform-aware defaults를 적용해야 한다.

## 테스트

- `config.json5`가 없는 temp dir에서 `loadConfigFromPath()` 호출 시 파일이 직접 생성되는지 확인
- 생성된 파일의 `auth.password`가 empty state인지 확인
- `password: null` 또는 누락 입력을 로드했을 때 validated config에서 `""`로 정규화되는지 확인
- plaintext non-empty password는 기존처럼 암호화되는지 회귀 확인

## 완료 조건

- 최초 생성 시 가짜 기본 비밀번호 문자열이 더 이상 파일에 들어가지 않는다.
- 비밀번호 미설정 상태가 런타임에서 명확히 감지 가능하다.
- 기존 cross-platform PTY bootstrap 동작이 깨지지 않는다.
