# Phase 3: Frontend First-Password Setup UX

## 목표

unauthenticated 사용자가 최초 접속했을 때, 서버 bootstrap status에 따라 로그인 폼 대신 초기 관리자 비밀번호 설정 폼을 보게 만든다.

## 대상 파일

- `frontend/src/contexts/AuthContext.tsx`
- `frontend/src/components/Auth/AuthGuard.tsx`
- `frontend/src/components/Auth/LoginForm.tsx`
- 신규 컴포넌트 후보:
  - `frontend/src/components/Auth/BootstrapPasswordForm.tsx`
- `frontend/src/services/api.ts`
- `frontend/src/types/*`
- `frontend/tests/e2e/*`

## 구현 항목

1. `authApi.getBootstrapStatus()`와 `authApi.bootstrapPassword()`를 추가한다.
2. `AuthContext`에 bootstrap state를 추가한다.
   - checking bootstrap
   - setupRequired
   - requesterAllowed
   - bootstrapError
3. `AuthGuard`는 unauthenticated 상태에서:
   - setupRequired && requesterAllowed -> `BootstrapPasswordForm`
   - setupRequired && !requesterAllowed -> denied 안내 UI
   - else -> 기존 `LoginForm`
4. `BootstrapPasswordForm`을 추가한다.
   - password
   - confirmPassword
   - client-side mismatch validation
   - loading/error state
5. bootstrap 성공 시:
   - JWT 저장
   - authenticated state 전환
   - 일반 앱 화면으로 진입
6. bootstrap 완료 후 새로고침해도 일반 로그인/인증 상태 계약이 유지되도록 한다.

## UX 가이드

- headline은 “초기 관리자 비밀번호 설정”처럼 명확해야 한다.
- requester denied 상태에서는 왜 이 장치에서 설정할 수 없는지 설명한다.
- confirm mismatch, too short, server denied 등 에러 메시지를 구분한다.
- bootstrap 성공 후 다시 로그인 폼을 보지 않도록 auto-login 흐름을 유지한다.

## 테스트

- setupRequired=true, requesterAllowed=true 상태에서 bootstrap form이 보이는지
- confirm mismatch 시 submit 차단되는지
- bootstrap 성공 시 authenticated 화면으로 진입하는지
- setupRequired=true, requesterAllowed=false 상태에서 denied 안내만 보이는지
- 일반 로그인 흐름이 여전히 작동하는지

## 완료 조건

- 사용자가 비밀번호 미설정 상태를 “로그인 실패”가 아니라 “초기 설정 필요”로 인식할 수 있다.
- bootstrap 완료 후 불필요한 이중 입력 없이 앱으로 진입한다.
