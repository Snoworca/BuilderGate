# Phase 2: Bootstrap Auth API And Server-Side Access Policy

## 목표

서버가 비밀번호 미설정 상태를 명시적으로 표현하고, 허용된 requester만 초기 비밀번호를 설정할 수 있는 bootstrap auth contract를 제공한다.

## 대상 파일

- `server/src/services/AuthService.ts`
- `server/src/routes/authRoutes.ts`
- `server/src/index.ts`
- 신규 helper 후보:
  - `server/src/utils/bootstrapAccessPolicy.ts`
  - `server/src/services/BootstrapSetupService.ts`
- `server/src/types/auth.types.ts`
- `server/src/test-runner.ts`

## 구현 항목

1. `AuthService`에 `hasConfiguredPassword()` 또는 동등한 method를 추가한다.
2. requester IP 평가 helper를 추가한다.
   - localhost 기본 허용
   - `bootstrap.allowedIps`
   - env override `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS`
3. `GET /api/auth/bootstrap-status` route를 추가한다.
4. `POST /api/auth/bootstrap-password` route를 추가한다.
   - password
   - confirmPassword
   - validation
   - bootstrap allowed check
   - password missing check
5. 성공 시:
   - 비밀번호 암호화 저장
   - `AuthService.updateRuntimeConfig()` 호출
   - JWT 발급
6. 이미 비밀번호가 설정된 뒤의 bootstrap 호출은 명시적으로 차단한다.
7. audit log 및 bootstrap attempt log를 남긴다.
8. bootstrap route 전용 rate limit 또는 보호 장치를 검토/추가한다.

## 권장 응답 계약

### `GET /api/auth/bootstrap-status`

```json
{
  "setupRequired": true,
  "requesterAllowed": true,
  "allowPolicy": "localhost"
}
```

### `POST /api/auth/bootstrap-password`

```json
{
  "success": true,
  "token": "...",
  "expiresIn": 1800000
}
```

## 코드 재사용/주의

- 기존 `/api/auth/login` route는 bootstrap state를 직접 처리하지 않고 유지한다.
- settings 기반 password rotation과 bootstrap password setup은 분리해야 한다.
- `X-Forwarded-For` 신뢰 여부가 확정되지 않았으므로 requester IP는 기본적으로 direct socket/IP 기준으로 처리한다.

## 테스트

- password missing 상태에서 bootstrap-status가 setupRequired를 반환하는지
- localhost requester는 허용, 임의 remote IP는 기본 차단되는지
- allowlist IP는 허용되는지
- bootstrap-password 성공 시 encrypted password 저장 + runtime update + JWT issuance가 되는지
- 비밀번호가 이미 설정된 뒤에는 bootstrap route가 다시 열리지 않는지
- 기존 `/api/auth/login`과 2FA 흐름이 회귀 없이 유지되는지

## 완료 조건

- 서버만으로도 “초기 설정 필요” 상태를 판정할 수 있다.
- 허용된 requester만 bootstrap을 완료할 수 있다.
- bootstrap 성공 직후 새 비밀번호로 즉시 인증 상태가 성립한다.
