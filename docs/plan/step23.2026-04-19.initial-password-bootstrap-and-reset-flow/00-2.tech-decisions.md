# Tech Decisions

## TD-001: `config.json5`는 코드에서 직접 생성한다

### 결정

`config.json5.example` 복제를 bootstrap source로 사용하지 않고, 서버 코드에서 직접 JSON5 템플릿을 렌더링한다.

### 이유

- 가짜 기본 비밀번호 문자열을 bootstrap source에서 제거할 수 있다.
- 실행 bootstrap과 문서 예제를 분리할 수 있다.
- 플랫폼별 기본값, bootstrap section, 향후 운영 플래그를 코드로 일관되게 제어할 수 있다.

### 영향

- `config.json5.example`는 문서/예시 전용으로 남는다.
- `ensureConfigExists()` 구현이 템플릿 렌더러를 호출하도록 바뀐다.

## TD-002: 비밀번호 미설정 canonical state는 `""`이다

### 결정

`null`, 누락, 빈 문자열 입력은 모두 허용하되, 내부 저장과 런타임 표현은 `auth.password: ""`로 정규화한다.

### 이유

- 기존 `AuthConfig.password: string` 계약을 크게 바꾸지 않는다.
- 비교/검증/리셋 로직이 단순해진다.
- 설정 파일 mutation helper가 명확해진다.

### 영향

- 스키마는 string 기반 유지, normalize helper만 추가한다.
- `--reset-password`는 결과적으로 `""`를 기록한다.

## TD-003: bootstrap route는 public이지만 강한 게이트를 가진다

### 결정

bootstrap status / password setup route는 auth 없이 접근 가능하되, 다음 조건이 모두 만족될 때만 성공한다.

1. 현재 비밀번호가 비어 있음
2. requester IP가 localhost 또는 allowlist에 포함됨

### 이유

- 로그인 이전에 초기 설정이 필요하므로 auth-required route로 만들 수 없다.
- 대신 “one-time + requester gate”로 보안 경계를 세워야 한다.

### 영향

- route 내부에 bootstrap policy helper가 필요하다.
- audit log와 전용 rate limit 추가가 필요하다.

## TD-004: bootstrap allowlist는 config + runtime override 조합으로 간다

### 결정

정적 정책은 `bootstrap.allowedIps`에 두고, 운영 편의용 임시 허용은 런처 env override `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS`로 추가한다.

### 이유

- 영구 allowlist와 일시적 운영 override를 분리할 수 있다.
- 사용자가 요청한 “실행 옵션으로 IP를 입력” 요구를 만족한다.
- 설정 파일을 항상 덮어쓰지 않아도 된다.

### 영향

- `tools/start-runtime.js`는 `--bootstrap-allow-ip`를 parse 해서 env에 주입한다.
- 서버는 config allowlist + env allowlist를 합쳐서 평가한다.

## TD-005: bootstrap 성공 시 JWT를 즉시 발급한다

### 결정

초기 비밀번호 설정 성공 응답은 단순 success가 아니라 JWT와 expiresIn을 포함한다.

### 이유

- bootstrap 후 다시 로그인 폼으로 돌아가는 이중 입력을 제거한다.
- `AuthContext.login()` 성공 흐름과 유사하게 구현할 수 있다.
- bootstrap requester는 이미 trusted IP gate를 통과했다.

### 영향

- 프런트는 bootstrap success 후 즉시 authenticated state로 전이한다.
- 서버는 password persistence + runtime update + token issue를 한 transaction-like 흐름으로 묶어야 한다.

## TD-006: `--reset-password`는 파일의 `auth.password`만 바꾼다

### 결정

`--reset-password`는 `config.json5`의 `auth.password`만 빈 값으로 바꾸고, 다른 설정은 건드리지 않는다.

### 이유

- 사용자가 정확히 요구한 동작이다.
- 운영 중 포트, PTY, 2FA, workspace 등 다른 설정을 보존해야 한다.

### 영향

- 런처에 section-aware config mutation helper가 필요하다.
- 실패 시 pm2 시작 전에 즉시 중단해야 한다.

## TD-007: 일반 로그인/설정 기반 비밀번호 변경 흐름은 유지한다

### 결정

bootstrap은 “최초 설정 / 리셋 후 재설정” 전용이고, 기존 settings password rotation과 `/api/auth/login` 흐름은 유지한다.

### 이유

- 기존 설정 화면과 테스트 자산을 최대한 재사용한다.
- bootstrap을 일반 비밀번호 변경 기능으로 확장하면 정책 경계가 흐려진다.

### 영향

- `SettingsService`는 여전히 current/new/confirm 기반 rotation을 담당한다.
- bootstrap route와 settings route는 목적이 다르므로 별도 validation을 유지한다.

## TD-008: 부트스트랩 화면은 LoginForm 대체 모드로 노출한다

### 결정

별도의 앱 경로를 만들지 않고, `AuthGuard`가 unauthenticated 상태에서 bootstrap 여부에 따라 `LoginForm` 또는 `BootstrapPasswordForm`을 선택 렌더링한다.

### 이유

- 현재 인증 진입점 구조와 잘 맞는다.
- 첫 접속 UX가 단순하다.
- deep link 처리와 route 증가를 피할 수 있다.

### 영향

- `AuthContext`에 bootstrap status state가 추가된다.
- `AuthGuard` 분기가 하나 더 생긴다.
