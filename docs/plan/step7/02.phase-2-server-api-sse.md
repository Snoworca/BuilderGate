## Phase 2: Server REST API + Workspace SSE

### 목표
WorkspaceService를 HTTP로 노출하는 REST API와, 다중 클라이언트 동기화를 위한 Workspace SSE 엔드포인트를 구현한다.

### 선행 조건
- Phase 1 완료 필요 (WorkspaceService, 타입, 에러코드)

### 기존 코드 재사용 힌트
- **sessionRoutes.ts**: `server/src/routes/sessionRoutes.ts` — 라우트 구조 패턴: `Router()` + try/catch + `AppError` + `res.json(data)`. SSE 라우트: `initSSE(res)` → `req.on('close', cleanup)`
- **SSE 유틸**: `server/src/utils/sse.ts` — `initSSE(res)`, `sendSSE(res, event, data)`, `sendSSEComment(res, comment)` 3개 함수 재사용
- **Auth 미들웨어**: `server/src/middleware/authMiddleware.ts` — `createAuthMiddleware(authService)` 반환값을 라우터에 적용. SSE는 `?token=` 쿼리 파라미터 지원
- **index.ts**: `server/src/index.ts` — `setupRoutes()` 함수에 `app.use('/api/workspaces', authMiddleware, workspaceRoutes)` 1줄 추가

### 언어+버전 주의사항
- Express Router 패턴: `router.get('/', handler)` — async handler는 try/catch 필수
- SSE: `req.on('close', ...)` 정리 패턴 필수 (메모리 누수 방지)
- PUT /api/workspaces/order를 GET /api/workspaces/:id 보다 **먼저** 등록 (Express 라우팅 우선순위 — `order`가 `:id`로 매칭되는 것 방지)
- SSE 자기 자신 제외: 각 SSE 클라이언트에 고유 ID 부여, broadcast 시 발신자 ID 비교

### LLM 코딩 주의사항
- Simplicity: 라우트 핸들러는 얇게 유지 — 비즈니스 로직은 WorkspaceService에 위임. 라우트에서 직접 JSON 파일 접근 금지. 미들웨어 체인 추가 금지 (auth만 사용)
- Surgical: 기존 sessionRoutes.ts, fileRoutes.ts, authRoutes.ts 수정 금지. index.ts에는 라우트 마운트 1줄 + WorkspaceService 초기화만 추가

### 테스트 시나리오

#### 정상 케이스
- Given: 인증된 사용자
  When: POST /api/workspaces { name: "Test" }
  Then: 201 응답, Workspace 객체 반환, SSE workspace:created 이벤트 발행 (다른 클라이언트에만)

- Given: 탭 2개가 있는 Workspace
  When: DELETE /api/workspaces/:id
  Then: 200 응답, 소속 PTY 세션 2개 종료 (SessionManager.deleteMultipleSessions), SSE workspace:deleted 이벤트 발행

#### 예외 케이스
- Given: 인증 토큰 없음
  When: 임의 API 호출
  Then: 401 Unauthorized (authMiddleware)

- Given: 존재하지 않는 Workspace ID
  When: PATCH /api/workspaces/nonexistent
  Then: 404 WORKSPACE_NOT_FOUND

- Given: Workspace 1개만 남은 상태
  When: DELETE /api/workspaces/:id
  Then: 409 LAST_WORKSPACE

#### 경계값 테스트
- Given: SSE 클라이언트 2개 연결 (A, B)
  When: 클라이언트 A가 POST /api/workspaces/:id/tabs
  Then: 클라이언트 B에 tab:added 이벤트 수신, 클라이언트 A에는 미수신

- Given: Workspace에 탭 8개 존재 (maxTabsPerWorkspace=8)
  When: POST /api/workspaces/:id/tabs
  Then: 409 TAB_LIMIT_EXCEEDED

- Given: 전체 PTY 세션 32개 (maxTotalSessions=32)
  When: 아무 Workspace에서 POST /api/workspaces/:id/tabs
  Then: 409 SESSION_LIMIT_EXCEEDED
