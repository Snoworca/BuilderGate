## Phase 7: Server Persistence + Restart Recovery + Cross-Device

### 목표
WorkspaceService의 JSON 저장 신뢰성을 강화(atomic write, backup, debounce)하고, 서버 재시작 복구 프로토콜과 크로스 디바이스 접속을 완성한다.

### 선행 조건
- Phase 6 완료 필요 (전체 UI 기능 동작)
- Phase 4의 disconnected 탭 UI (FR-7207) 구현 완료 필요

### 기존 코드 재사용 힌트
- **WorkspaceService (Phase 1 결과)**: 기존 stub(`writeFile` 단순 호출)을 atomic write + backup + debounce로 강화. 기존 CRUD 메서드 시그니처 변경 없이 내부 flush 로직만 교체
- **SessionManager.hasSession()**: Phase 1에서 추가한 메서드로 orphan 탭 감지
- **server/src/index.ts**: `process.on('SIGINT')`, `process.on('SIGTERM')` 핸들러 추가 위치

### 구현 체크리스트
- [ ] Phase 7-1: WorkspaceService에 debounce flush 도입 — `setTimeout`/`clearTimeout` 기반, 변경 발생 후 5초 대기 후 flush. Workspace/Tab 생성/삭제 시에는 debounce 무시하고 즉시 flush. 그리드 리사이즈 드래그 중에는 저장 억제 (드래그 종료 시 1회 flush) `FR-7702`
- [ ] Phase 7-2: Atomic write 구현 — `workspaces.json.tmp`에 `fs.promises.writeFile()` 후 `fs.promises.rename()`으로 교체. tmp 파일은 같은 디렉토리에 생성 (Windows cross-device rename 방지). 파일 생성 시 `mode: 0o600` `FR-7702`
- [ ] Phase 7-3: Backup 관리 — flush 전 기존 `workspaces.json`을 `workspaces.json.bak`으로 복사. 최근 1개만 유지. JSON 파싱 실패 시 `.bak`에서 복원 시도 `FR-7702, FR-7704`
- [ ] Phase 7-4: 서버 재시작 복구 프로토콜 — 서버 시작 시 JSON 로드 → 각 `WorkspaceTab.sessionId`에 대해 `SessionManager.hasSession(id)` 확인 → 미존재 시 탭 상태를 `disconnected`로 마킹 → SSE `tab:disconnected` 이벤트 발행 `FR-7706`
- [ ] Phase 7-5: Graceful shutdown — `server/src/index.ts`에 `process.on('SIGINT/SIGTERM')` 핸들러 추가. debounce 대기 중인 변경을 즉시 flush 후 종료 `FR-7702`
- [ ] Phase 7-6: 레이아웃 복원 검증 — 브라우저 새로고침 시 `GET /api/workspaces` → 전체 상태 복원, SSE 재연결, 각 터미널 SSE 스트림 재연결 `FR-7705`
- [ ] Phase 7-7: 크로스 디바이스 동기화 검증 — 다른 브라우저에서 접속 시 동일 레이아웃, SSE로 실시간 변경 반영 `FR-7705`

### 참조 스펙
- `FR-7702`: 서버 동시성 모델 — in-memory canonical, 5초 debounce, atomic write, backup
- `FR-7704`: JSON 파일 손상 복구 — `.bak` 복원 → 빈 초기 상태 fallback
- `FR-7705`: 레이아웃 복원 (재접속 + 크로스 디바이스)
- `FR-7706`: 서버 재시작 복구 프로토콜
- `FR-7207`: 탭 세션 비정상 종료 처리

### 언어+버전 주의사항
- `fs.promises.rename()` — Windows에서 cross-device rename 실패 가능, tmp 파일을 반드시 같은 디렉토리에 생성
- `setTimeout`/`clearTimeout` — debounce 구현. `lodash.debounce` 금지
- `process.on('SIGINT')`, `process.on('SIGTERM')` — Node.js graceful shutdown

### LLM 코딩 주의사항
- Simplicity: debounce는 단순 `setTimeout` + `clearTimeout` 패턴 (3줄). lodash.debounce, p-debounce 등 외부 라이브러리 금지. atomic write는 `writeFile` + `rename` 2줄
- Surgical: WorkspaceService의 기존 CRUD 메서드 시그니처 변경 금지. 내부 `save()` 메서드만 Phase 1의 stub에서 atomic write로 교체. `server/src/index.ts`에는 SIGINT/SIGTERM 핸들러만 추가

### 테스트 시나리오

#### 정상 케이스
- Given: 탭 순서 드래그 변경 3회 (1초 간격)
  When: 마지막 변경 후 5초 경과
  Then: workspaces.json에 1회만 쓰기 (debounce), 내용은 최종 상태

- Given: 서버 재시작 (PTY 전부 소멸)
  When: 클라이언트 재접속
  Then: Workspace/탭 레이아웃 복원, 모든 탭 disconnected 상태, 에러 오버레이 + 재시작 버튼 표시

#### 예외 케이스
- Given: workspaces.json.tmp 쓰기 완료 직전 서버 크래시 (rename 미완료)
  When: 서버 재시작
  Then: 원본 workspaces.json 정상 로드 (tmp는 무시), 시작 시 잔여 .tmp 파일 삭제

- Given: workspaces.json 손상, .bak은 정상
  When: 서버 시작
  Then: .bak에서 복원, 로그에 "JSON 손상 — 백업에서 복원" 경고

- Given: workspaces.json과 .bak 모두 손상
  When: 서버 시작
  Then: 빈 초기 상태로 시작 (Workspace 1개), 로그에 "모든 데이터 손상 — 초기화" 경고

#### 경계값 테스트
- Given: Workspace 생성 (즉시 flush 대상)
  When: 생성 직후 서버 SIGTERM
  Then: 재시작 시 해당 Workspace 존재 (즉시 flush + SIGTERM flush)

- Given: graceful shutdown (SIGTERM) 중 debounce 대기 변경 + 진행 중 write 없음
  When: SIGTERM 수신
  Then: 즉시 flush 1회 후 종료 (데이터 유실 없음)

- Given: Workspace 10개, 탭 총 30개 상태에서 서버 재시작
  When: 클라이언트 접속
  Then: 30개 탭 모두 disconnected 표시, 각 탭에 재시작 버튼
