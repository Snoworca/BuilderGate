## Phase 3: Frontend State Management — Hooks + API Service

### 목표
프론트엔드의 Workspace 상태 관리 계층을 구축한다. 서버 API 호출, SSE 이벤트 처리, 클라이언트 상태 관리를 담당하는 훅과 서비스를 구현한다.

### 선행 조건
- Phase 2 완료 필요 (서버 API + SSE 엔드포인트)

### 기존 코드 재사용 힌트
- **api.ts**: `frontend/src/services/api.ts` — `authFetch(url, options)` + `parseError(response)` 패턴으로 workspaceApi 추가
- **useSession.ts**: `frontend/src/hooks/useSession.ts` — CRUD + 상태 관리 구조 참고. `{ sessions, activeSessionId, createSession, deleteSession, ... }` 반환 패턴
- **useSSE.ts**: `frontend/src/hooks/useSSE.ts` — `EventSource` + `?token=` 쿼리 패턴. handlers를 `useRef`로 안정화하는 패턴 참고
- **useTabManager.ts**: `frontend/src/hooks/useTabManager.ts` — `reorderTabs(fromIndex, toIndex)`, `closeOtherTabs` 로직 참고

### 구현 체크리스트
- [ ] Phase 3-1: `frontend/src/services/api.ts`에 workspaceApi 객체 추가 — getWorkspaces, createWorkspace, updateWorkspace, deleteWorkspace, reorderWorkspaces, addTab, updateTab, deleteTab, reorderTabs, updateGrid, restartTab. `activeWorkspaceId`는 ADR-002에 따라 서버 호출 없이 `localStorage` 직접 저장 `FR-7705`
- [ ] Phase 3-2: `frontend/src/types/workspace.ts` 생성 — Workspace, WorkspaceTab, WorkspaceTabRuntime (`status: 'running'|'idle'|'disconnected'`, `cwd: string` 추가), GridLayout, WorkspaceState 타입 `섹션 4.1, 4.2`
- [ ] Phase 3-3: `frontend/src/hooks/useWorkspaceSSE.ts` 생성 — `GET /api/workspaces/stream` EventSource 연결, workspace:*/tab:*/grid:* 이벤트 처리, 상태 업데이트 콜백 연결 `섹션 5.2`
- [ ] Phase 3-4: `frontend/src/hooks/useWorkspaceManager.ts` 생성 — 핵심 상태 관리 훅. workspaces, tabs(WorkspaceTabRuntime[] — **모든 Workspace의 탭을 하나의 배열로 관리**), gridLayouts, activeWorkspaceId(localStorage), totalSessionCount(`useMemo(() => tabs.length, [tabs])` — 전체 Workspace 합산), CRUD 메서드, 탭 관리 메서드, 뷰 모드 전환 `FR-7101~FR-7108, FR-7201~FR-7206`
- [ ] Phase 3-5: useWorkspaceManager에 탭 닫기 후 활성 탭 전환 정책 구현 — 우측 인접 → 좌측 인접 → null (빈 상태) `FR-7205`
- [ ] Phase 3-6: useWorkspaceManager에 8색 팔레트 자동 할당 로직 구현 — `colorIndex = workspace.colorCounter++ % 8`. 삭제 후에도 카운터 복귀 안 함 `FR-7202`

### 언어+버전 주의사항
- React 18 + TypeScript
- `useCallback`으로 메서드 메모이제이션 (리렌더링 최적화)
- `useRef`로 SSE 핸들러 안정화 (useSSE.ts 패턴 참고)
- `localStorage` 접근은 try/catch로 래핑 (private browsing 모드 대응)
- `totalSessionCount`는 `useMemo(() => tabs.length, [tabs])`로 계산

### LLM 코딩 주의사항
- Simplicity: useWorkspaceManager 단일 훅으로 모든 상태 관리. Context/Redux 도입 금지. useSession + useTabManager 패턴과 동일한 수준의 복잡도 유지. 상태 정규화 라이브러리 금지
- Surgical: 기존 useSession.ts, useSSE.ts, useTabManager.ts 수정 금지. 새 훅만 추가. api.ts에는 workspaceApi 객체만 추가 (기존 sessionApi, authApi 변경 금지)

### 테스트 시나리오

#### 정상 케이스
- Given: useWorkspaceManager 마운트
  When: 초기 로드
  Then: GET /api/workspaces 호출 → workspaces, tabs, gridLayouts 상태 설정, loading → false

- Given: Workspace에 탭 3개 (인덱스 0,1,2), 탭 1이 활성
  When: 탭 1 닫기
  Then: 탭 2가 활성화 (우측 인접), tabs 배열에서 탭 1 제거

- Given: SSE 연결 활성, 다른 클라이언트에서 탭 추가
  When: tab:added 이벤트 수신
  Then: useWorkspaceManager의 tabs 배열에 새 탭 추가 (API 호출 없이 상태 직접 업데이트)

#### 예외 케이스
- Given: 서버 API 호출 실패 (네트워크 에러)
  When: createWorkspace 호출
  Then: error 상태 설정, 낙관적 업데이트 없음 (서버 확인 후에만 상태 변경)

- Given: SSE 연결 끊김
  When: 네트워크 복구
  Then: SSE 재연결 + GET /api/workspaces로 전체 상태 재동기화 (delta 아닌 full refresh)

#### 경계값 테스트
- Given: 탭 1개만 있는 Workspace
  When: 마지막 탭 닫기
  Then: activeTabId = null, 빈 상태

- Given: 탭 8개 (colorCounter = 8)
  When: 탭 1개 삭제 후 새 탭 추가
  Then: 새 탭 colorIndex = 0 (8 % 8), colorCounter = 9

- Given: totalSessionCount = 32 (전체 Workspace 합산)
  When: 아무 Workspace에서 addTab 시도
  Then: SESSION_LIMIT_EXCEEDED 에러 처리, UI에서 + 버튼 비활성화 근거
