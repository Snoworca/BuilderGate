# Tech Decisions (ADR) — Step 7: CMUX-Style Workspace Pivot

## ADR-001: 서버 상태 관리 — In-Memory + JSON Flush

**결정:** WorkspaceService는 in-memory 상태를 canonical source로 유지하고, 디스크 JSON은 persistence layer로만 사용한다.

**이유:**
- Node.js 단일 스레드 모델에서 in-memory 변경은 순차적 → race condition 없음
- API 응답이 메모리에서 직접 → 파일 I/O 없이 빠른 응답
- debounce flush로 빈번한 디스크 쓰기 방지

**구현:** 5초 debounce + Workspace/Tab 생성/삭제 시 즉시 flush. Atomic write (tmp + rename).

## ADR-002: activeWorkspaceId — 클라이언트 로컬 상태

**결정:** `activeWorkspaceId`는 서버에 저장하지 않고 클라이언트 `localStorage`에만 유지한다.

**이유:**
- 멀티 클라이언트 환경에서 한쪽이 active를 변경하면 다른 쪽이 강제 전환되는 문제 방지
- 각 클라이언트가 독립적인 뷰를 가져야 함

## ADR-003: Workspace SSE — 별도 엔드포인트

**결정:** `GET /api/workspaces/stream`을 별도 SSE 엔드포인트로 추가. 기존 세션 SSE와 분리.

**이유:**
- 세션 SSE는 PTY output 전용 (고빈도, 세션별)
- Workspace SSE는 메타데이터 변경 (저빈도, 전역)
- 레이어 분리가 아키텍처적으로 깔끔

---

## ADR-004: Grid Mode — CSS Grid 기반

**결정:** 기존 SplitPane(바이너리 트리)를 CSS Grid 기반 격자로 대체.

**이유:**
- 바이너리 트리는 N개 터미널 동시 배치에 비효율적
- CSS Grid는 `grid-template-columns/rows`로 비율 기반 배치 네이티브 지원
- `ResizeObserver`와 자연스럽게 통합

---

## ADR-005: 탭 색상 — 단조 증가 카운터

**결정:** 탭 색상 할당은 `colorCounter++ % 8`로 단조 증가. 삭제 후에도 카운터 복귀하지 않음.

**이유:**
- 색상 재사용 시 "이전 탭과 같은 색"이 되어 혼란 유발
- 8개 상한이므로 8색이면 충분
- 단조 증가가 구현 단순
