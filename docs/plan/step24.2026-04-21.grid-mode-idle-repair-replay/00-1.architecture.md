# Architecture

## 핵심 설계

현재 수동 복구 경로는 사실상 다음 순서다.

1. grid host 크기 변화
2. `fit()`으로 새 cols/rows 계산
3. server `resize()`
4. server replay refresh
5. frontend `replaceWithSnapshot()`로 authoritative 화면 재구성

이번 구현은 이 과정을 **geometry mutation 없이 재현**하는 방향으로 설계한다.

## 제안 구조

### Server

- resize와 별개인 `repair replay` 진입점 추가
- 특정 session에 대해 현재 authoritative snapshot을 다시 보내되:
  - replay token 발급
  - pending replay 상태 생성
  - ACK 후 `session:ready`
  - queued output 경계 재사용

즉, resize가 만든 replay lifecycle을 geometry mutation 없이 재사용하는 서버 진입점을 만든다.

### Frontend

- TerminalContainer가 `status=idle`을 감지
- grid mode에서만 동작
- 짧은 quiet window 후 한 번만 repair replay 요청
- snapshot 수신 시 기존 `replaceWithSnapshot()` 경로로 화면 재구성
- 재적용 후 필요하면 `fit()` 동반

## 왜 이 방향인가

- 단순 React rerender는 복구력이 약하다.
- synthetic resize는 효과는 강하지만 SIGWINCH 부작용이 크다.
- 현재 구조에서 가장 안전한 재사용 자산은 authoritative snapshot replay 경로다.

## 주요 리스크

- repair replay가 output burst와 겹치면 queued output 중복/누락이 생길 수 있다.
- idle 전환 직후 곧바로 다시 running이 되는 경우 불필요한 replay가 발생할 수 있다.
- grid mode 판정 범위가 넓으면 tab mode 회귀가 생길 수 있다.
