# Phase 1: Server Repair Replay Protocol

## Objective

resize 없이도 authoritative snapshot replay를 시작할 수 있는 서버 진입점을 추가한다.

## Prerequisites

- 없음

## Target Files

- `server/src/ws/WsRouter.ts`
- `server/src/services/SessionManager.ts`
- `server/src/types/ws-protocol.ts`
- `frontend/src/types/ws-protocol.ts`
- `server/src/test-runner.ts`

## Planned Changes

1. client가 특정 session에 대해 repair replay를 요청할 수 있는 WS message 추가
2. server가 pending replay 상태를 생성하고 현재 snapshot을 재전송하는 경로 추가
3. ACK 후 `session:ready`와 queued output flush 경계를 기존 경로와 동일하게 유지
4. resize 기반 refresh와 repair replay가 lineage 측면에서 구분 가능하도록 telemetry 보강

## Implementation Checklist

- [ ] repair replay client message 타입 추가
- [ ] `WsRouter`에 repair replay handler 추가
- [ ] 현재 snapshot을 replay pending 상태로 재전송하는 helper 추가
- [ ] replay telemetry에 repair origin 구분값 추가
- [ ] ACK/queued output/session:ready 경계 테스트 추가

## Test

### Scenario 1: repair replay starts without geometry change

Given a subscribed session with no resize  
When client requests repair replay  
Then server sends a new authoritative snapshot and waits for ACK

### Scenario 2: output stays queued during repair replay

Given output arrives while repair replay is pending  
When snapshot ACK arrives  
Then queued output flushes once and `session:ready` follows
