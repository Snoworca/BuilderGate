# Grid Mode Idle Repair Replay Plan

Date: 2026-04-21  
Scope: Grid mode TUI 세션이 줄바꿈/라인 배치가 깨졌을 때, 사용자의 수동 리사이즈가 주는 복구 효과를 `idle` 진입 후 1회 repair replay로 재현한다  
Mode: plan only

## Goal

Grid mode에서만 발생하는 TUI 깨짐을 `idle` 진입 후 한 번의 안전한 repair replay로 복구한다.

핵심 목표:

- geometry mutation 없이 수동 리사이즈와 유사한 복구 효과 재현
- grid mode 한정 범위로 시작
- 기존 resize replay, authoritative snapshot, ACK 경로 재사용
- tab mode 및 일반 shell 흐름 회귀 방지

## Requirements

- `IMPL-001` Grid mode idle 진입 후 1회 repair replay를 수행할 수 있어야 한다.
- `IMPL-002` repair replay는 geometry mutation 없이 현재 authoritative snapshot 재적용을 우선 사용해야 한다.
- `IMPL-003` repair replay는 `idle + quiet window` 조건에서만 동작해야 한다.
- `IMPL-004` tab mode 및 일반 shell 세션에는 기본적으로 영향을 주지 않아야 한다.
- `IMPL-005` 기존 resize replay ACK/queued output 경계와 충돌하지 않아야 한다.
- `IMPL-006` Codex/Hermes류 TUI의 grid corruption 시나리오를 회귀 테스트와 수동 검증 절차로 고정해야 한다.

## Assumptions

1. 문제의 핵심 복구 효과는 단순 rerender가 아니라 authoritative snapshot 재구성에 가깝다.
2. 현재 `refreshReplaySnapshots()`는 pending replay 상태에서만 동작하므로, repair replay 전용 진입점이 필요하다.
3. codex와 hermes 모두 “grid mode의 TUI 복구”라는 증상 기반 범주로 다루고, 1차 구현에서 앱별 detector는 필수 조건으로 두지 않는다.

## Recommended Execution Order

1. 서버에 resize 없이도 사용할 수 있는 repair replay 진입점을 만든다.
2. 프런트에서 `grid mode + idle + quiet window` 조건에 맞춰 one-shot repair trigger를 건다.
3. 회귀 테스트와 실기기 검증으로 복구 효과와 부작용을 확인한다.

## Success Criteria

- Grid mode TUI 깨짐이 idle 진입 후 한 번의 자동 복구로 정상화된다.
- tab mode와 일반 shell 세션의 기존 replay/input 흐름은 유지된다.
- queued output / replay token / session:ready 경계가 깨지지 않는다.
- synthetic resize 없이도 대부분의 깨짐이 복구된다.

## Out Of Scope

- synthetic resize를 기본 복구 전략으로 채택
- 일반 non-grid 세션 전체에 자동 repair 적용
- codex/hermes 외 모든 foreground app 일반화
