# Frontend Build Error Remediation Plan

Date: 2026-04-13
Scope: `frontend` TypeScript build blockers remaining after terminal/WS fixes
Mode: analysis and implementation planning only

## Goal

Restore a clean `npm run build` for the current frontend by resolving the remaining strict TypeScript errors in:

- `src/components/Grid/MosaicContainer.tsx`
- `src/components/Grid/MosaicTile.tsx`
- `src/components/Header/Header.tsx`
- `src/components/MetadataBar/MetadataRow.tsx`
- `src/components/Workspace/WorkspaceItem.tsx`
- `src/components/Workspace/WorkspaceTabBar.tsx`
- `src/utils/contextMenuBuilder.ts`

## Assumptions

1. The preferred fix is the smallest behavior-preserving change that matches the current runtime intent.
2. Existing strict compiler settings in `frontend/tsconfig.app.json` are correct and should not be relaxed.
3. `Header` is intended to display the active terminal cwd.
   Reason: `App.tsx` still passes `activeCwd`, `Header.tsx` already computes `displayCwd`, and `Header.css` already defines `.header-cwd` styles.
4. `MetadataRow.isOdd` is currently dead API and should be removed unless a real alternating-row requirement is reintroduced.

## Current Errors

- `TS7006` / `TS2345` in `MosaicContainer.tsx`
  `setMosaicTree(prev => ...)` is valid React state-setter usage, but `useMosaicLayout` exports an overly narrow setter type.
- `TS6133` in `MosaicTile.tsx`
  `onAdd` and `availableShells` are declared but unused.
- `TS6133` in `Header.tsx`
  `displayCwd` is declared but unused.
- `TS6133` in `MetadataRow.tsx`
  `isOdd` is declared but unused.
- `TS6196` in `WorkspaceItem.tsx`
  `WorkspaceTabRuntime` import is unused.
- `TS6133` in `WorkspaceTabBar.tsx`
  `ContextMenuItem` import is unused.
- `TS2322` / `TS2741` in `contextMenuBuilder.ts`
  separator items do not match the current `ContextMenuItem` type.

## Root Cause Synthesis

Three independent analysis tracks converged on the same structure:

1. Shared type model drift
   `ContextMenu` runtime supports separator-only rows, but `ContextMenuItem` still models all entries as label-bearing items.

2. Hook contract drift
   `useMosaicLayout` internally uses a real React state setter, but its exported type blocks the functional updater form already used by `MosaicContainer`.

3. UI/API residue under strict TypeScript
   Several props and imports are leftovers from prior iterations. With `noUnusedLocals` and `noUnusedParameters` enabled, those residues are now hard build blockers.

## Recommended Execution Order

1. Phase 1: Shared context-menu typing and strict-import cleanup
   Why first: it removes the cross-cutting type defect that affects multiple files.

2. Phase 2: Grid layout contract cleanup
   Why second: it resolves the only non-trivial setter mismatch and removes dead grid props without changing grid behavior.

3. Phase 3: Header/Metadata alignment and final build verification
   Why third: it finishes the remaining dead-value errors and validates the intended header behavior in the UI.

## Success Criteria

- `cd frontend && npm run build` exits successfully.
- No remaining compiler errors mention the seven target files in this plan.
- Context menus still render normal actions, separators, and submenus correctly.
- Grid layout still responds correctly in equal/focus/auto modes.
- Header cwd rendering is either restored intentionally or removed intentionally with no dead prop left behind.
- Metadata rows still render identically in both tab mode and grid mode.

## Out of Scope

- Large refactors in `MosaicContainer` effect dependencies or layout heuristics
- UX redesign of the header or metadata bar
- Compiler or lint rule relaxation
- Server-side changes
