# Phase 1: Context Menu And Strict Typing

## Objective

Fix the shared `ContextMenuItem` type model so separator rows are type-safe, then remove the strict-mode import residue that is currently failing the build.

## Target Files

- `frontend/src/components/ContextMenu/ContextMenu.tsx`
- `frontend/src/utils/contextMenuBuilder.ts`
- `frontend/src/components/Workspace/WorkspaceItem.tsx`
- `frontend/src/components/Workspace/WorkspaceTabBar.tsx`

## Root Cause

The renderer already supports two runtime shapes:

- actionable rows with `label`
- separator rows with only `separator: true`

The current `ContextMenuItem` interface models only the first shape and treats `separator` as an optional add-on. That forces unsafe casts and breaks builder code that emits real separator objects.

## Planned Changes

1. Replace `ContextMenuItem` with a discriminated union.
   Recommended shape:
   - `ContextMenuActionItem` with required `label`
   - `ContextMenuSeparatorItem` with `separator: true`
   - `type ContextMenuItem = ContextMenuActionItem | ContextMenuSeparatorItem`

2. Narrow separator rows before accessing action-only fields.
   `MenuItemList` should continue branching on `item.separator`.
   `MenuItemRow` should only accept the action-item branch.

3. Remove separator casts from builders and consumers.
   Use plain separator literals instead of `as ContextMenuItem`.

4. Remove unused strict-mode imports.
   - Delete `WorkspaceTabRuntime` import from `WorkspaceItem.tsx`
   - Delete unused `ContextMenuItem` import from `WorkspaceTabBar.tsx`

## Why This Shape

- Making `label` optional would hide malformed actionable rows.
- Keeping casts would preserve the same defect and let future callers reintroduce the problem silently.
- A discriminated union matches actual runtime behavior and gives the compiler real protection.

2. Confirm no remaining errors in:
   - `ContextMenu.tsx`
   - `contextMenuBuilder.ts`
   - `WorkspaceItem.tsx`
   - `WorkspaceTabBar.tsx`
3. Manual smoke test:

## Risks

- Medium: `ContextMenuItem` is shared across several menu producers.
- Low: removing unused imports is behavior-free.
