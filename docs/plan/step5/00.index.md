## Overview

This plan implements the Step 5 runtime settings page without destabilizing the current BuilderGate codebase. The work is split into a server track and a frontend track so runtime configuration, persistence, UI navigation, and validation can be delivered incrementally.

| Area | Current Stack | Step 5 Goal |
| --- | --- | --- |
| Backend | Node.js, Express, TypeScript | Editable runtime settings snapshot, validated PATCH API, hot-apply for supported keys |
| Frontend | React, TypeScript, existing single-shell app | Header entry point, in-app settings screen, typed form flow |
| Persistence | `server/config.json5` | Preserve comments/order while updating editable values only |
| Security | Session-based auth, 2FA service, encrypted secrets | Write-only secret handling, password rotation, audit-safe responses |

## Dependency Graph

```text
Phase 1 -> Phase 2 -> Phase 3
Phase 2 -> Phase 4 -> Phase 5
Phase 3 + Phase 5 -> Phase 6
```

- Phase 1 defines runtime-editable data structures and service update seams.
- Phase 2 adds safe persistence and API contracts on top of those seams.
- Phase 3 wires hot-apply behavior and hardens rollback behavior.
- Phase 4 introduces the settings entry point and page shell.
- Phase 5 builds the editable UI and PATCH flow on top of the server API.
- Phase 6 validates end-to-end behavior and release readiness.

## Quality Gates

- Every phase must leave the app buildable and keep legacy workspace navigation intact.
- API contracts must use an explicit editable whitelist; unsupported keys must fail closed.
- `server/config.json5` updates must preserve comments and remain valid JSON5.
- Secret values must never be returned in plaintext by the API.
- Frontend must not introduce a router or reset active workspace state when entering settings.
- Release approval requires all verification documents and [final-validation.md](C:/Work/git/_Snoworca/ProjectMaster/docs/plan/step5/final-validation.md) to be completed.

## Execution Notes

- Step 5 should be implemented in order. Do not start UI forms before the server payload contract is stable.
- Use fixture-backed tests for config persistence before touching the real file writer path.
- Keep server and frontend plans separate in execution, but validate them together in Phase 6.
- If runtime application proves unsafe for a specific key, move that key to the excluded list rather than forcing hot-apply support.
