# Technical Decisions

## BuilderGate - Step 5: Runtime Settings Page

## ADR-001: Use a Dedicated Runtime Config Store

- Status: Accepted
- Decision: Add a focused `RuntimeConfigStore` instead of letting routes and services keep reading mutable values from the global config object.
- Rationale: Step 5 needs explicit runtime-safe state, apply-scope metadata, and a controlled editable whitelist.
- Consequence: Existing startup wiring changes only where services need runtime updates.

## ADR-002: Expose Only a Whitelisted Editable Snapshot

- Status: Accepted
- Decision: Backend returns and accepts only supported runtime-editable keys.
- Rationale: `config.json5` contains restart-required, internal, and secret values that must not become general UI controls.
- Consequence: Step 5 can claim "most configurable runtime-safe settings" without exposing unsafe controls.

## ADR-003: Auth Routes Must Use Runtime Service Accessors

- Status: Accepted
- Decision: Auth and 2FA routes obtain service instances through runtime accessors rather than fixed bootstrap captures.
- Rationale: 2FA enable or disable must apply to subsequent logins immediately.
- Consequence: Minor bootstrap and route registration changes are required.

## ADR-004: Replace Static CORS Wiring With a Runtime Delegate

- Status: Accepted
- Decision: Express CORS middleware will use a delegate backed by `RuntimeConfigStore`.
- Rationale: `security.cors.*` is editable in Step 5 and should affect later requests without restart.
- Consequence: CORS settings remain validated server-side and excluded if a safe delegate cannot be implemented.

## ADR-005: Preserve `config.json5` Structure Via Section-Aware Patching

- Status: Accepted
- Decision: Implement a config repository that patches editable sections while preserving comments and ordering.
- Rationale: A naive serialize-and-overwrite path would destroy operator comments and make manual maintenance harder.
- Consequence: Additional fixture-based tests are required for persistence.

## ADR-006: Keep Frontend Navigation Inside the Existing Shell

- Status: Accepted
- Decision: Add a settings screen state inside `App.tsx` instead of introducing a router.
- Rationale: The current app is already a single-shell workflow; Step 5 only needs one extra authenticated screen.
- Consequence: Header and shell components need small but careful updates to preserve workspace state.

## ADR-007: Secrets Are Write-Only in the UI

- Status: Accepted
- Decision: Passwords and SMTP secrets are never returned from the API and are only sent when changed.
- Rationale: Browsers do not need existing secret values to edit runtime configuration safely.
- Consequence: UI uses empty secret inputs plus "configured" indicators, and PATCH payloads include secrets only when entered.

## ADR-008: Split Delivery Into Server Track, Frontend Track, and Integration

- Status: Accepted
- Decision: Sequence work as runtime store, API/persistence, runtime apply, UI shell, UI forms, then integration.
- Rationale: The UI should not be built against unstable contracts, and runtime apply should be hardened before release.
- Consequence: Phase ordering is mandatory for predictable delivery.

## ADR-009: AGENTS Guardrails Are Release Criteria

- Status: Accepted
- Decision: Preserve codebase stability, scope discipline, and surgical change sets as explicit completion criteria.
- Rationale: The repository already contains unrelated local edits; Step 5 must not disturb them.
- Consequence: Final validation includes dirty-worktree awareness and scope review.

## Testing Strategy

### Server

- Add fixture-based tests for editable whitelist extraction and config patching.
- Test `GET /api/settings` redaction behavior and `PATCH /api/settings` validation failures.
- Cover runtime apply seams for auth duration, 2FA hot-swap, session updates, file-manager updates, and dynamic CORS behavior.

### Frontend

- Test header entry, screen switching, and workspace state preservation.
- Test form serialization, dirty-state warnings, secret field handling, and field-level validation.
- Verify that unsupported or platform-specific fields are hidden when not applicable.

### Cross-Stack

- Exercise full settings load and save flow against the real API contract.
- Confirm runtime changes affect later requests, new sessions, or new logins according to scope.
- Confirm invalid saves do not corrupt `config.json5` or leave runtime state partially applied.

## Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| JSON5 comments or ordering are lost | Operator-facing config drift | Use section-aware patcher and fixture snapshots |
| Hot-swapped 2FA does not affect auth routes | Security behavior mismatch | Route accessors + integration tests for next login flow |
| Dynamic CORS behaves differently than startup config | Broken browser access | Add dedicated runtime tests and fail closed on invalid values |
| Existing sessions are broken by runtime updates | User disruption | Limit changes to new sessions when required and document scopes |
| Secrets leak in API responses or logs | Security incident | Write-only secret fields, redaction, and response tests |
| Dirty worktree is overwritten | User trust and merge risk | Confine changes to Step 5 plan docs during planning and scope-check implementation later |

## Dependency Guidance

- Avoid new runtime dependencies unless current codebase utilities cannot support safe config patching.
- If a helper library becomes necessary, prefer a narrowly scoped dev dependency for tests before adding runtime weight.
- Reuse existing modal, form, and API patterns on the frontend where they already exist.

## Exit Conditions

- Editable whitelist matches actual runtime-safe keys consumed by the current codebase.
- Backend can load and save settings safely without restart for supported fields.
- Frontend exposes a complete settings workflow from header button to save confirmation.
- Verification documents are filled with evidence from tests and manual checks.
- Final validation confirms Step 5 did not expand into unrelated refactors.
