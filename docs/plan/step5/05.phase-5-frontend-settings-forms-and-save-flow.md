# Phase 5 Plan

## Frontend Settings Forms And Save Flow

## Objective

Implement the editable settings UI, validation, and save flow on top of the Step 5 server API. This phase turns the settings shell into a complete operator workflow for runtime-safe configuration changes.

## Prerequisites

- Phase 4 completed
- `PATCH /api/settings` contract stable from Phase 2 and Phase 3
- Apply-scope metadata available from backend snapshot

## Requirement Mapping

| Step 5 Need | Phase 5 Output |
| --- | --- |
| Password can be changed | password change section and validation |
| Most runtime-safe config can be edited | sectioned forms for whitelist fields |
| Restart-required settings are not shown | capability-driven rendering |
| Save flow is understandable and safe | dirty tracking, validation, apply feedback |

## Implementation Checklist

- Finalize `frontend/src/types/settings.ts`
- Implement editable controls in `SettingsPage.tsx`
- Implement `patchSettings()` in settings API client
- Add client validation and dirty tracking
- Add unsaved-change confirmation flow
- Surface apply-scope hints and save result feedback
- Add targeted UI tests and manual verification notes

## Detailed Guidance

### 1. Settings Sections

Render focused sections instead of a raw JSON editor:

- Authentication
- Two-Factor Authentication
- CORS
- Terminal Defaults
- Session And File Manager

Each section should be driven by backend capabilities so unsupported fields do not appear.

### 2. Password And Secret Inputs

Authentication section:

- current password
- new password
- confirm new password
- session duration

Two-factor section:

- SMTP password input left blank on load
- `configured` indicator if the backend reports an existing encrypted value

Rules:

- secret inputs are never prefilled
- empty secret inputs mean "keep existing secret"
- mismatched password confirmation blocks submit

### 3. Draft State And Minimal PATCH Payloads

Track:

- loaded snapshot
- editable draft values
- dirty fields
- validation errors
- save status

On submit:

- construct a minimal PATCH payload containing changed editable keys only
- omit unchanged arrays and unchanged booleans
- include password fields only when rotation is requested

### 4. Field Types

Use simple controls aligned with current codebase patterns:

- text inputs for host, email, term name
- numeric inputs for durations, dimensions, max counts, ports
- checkboxes or toggles for booleans
- textarea or token-list editor for string arrays such as origins, blocked extensions, blocked paths
- read-only badges for apply scope

The UI does not need a new form framework unless the current stack already has one. Keep it lightweight.

### 5. Save Feedback

After a successful save:

- refresh the snapshot from the response payload
- clear dirty state
- show concise per-scope feedback such as:
  - applied immediately
  - applies to new logins
  - applies to new sessions

On failure:

- keep the draft intact
- render field-level or section-level error messages where possible

### 6. Unsaved Changes Protection

Reuse `ConfirmModal` for:

- leaving settings with unsaved changes
- returning to workspace with dirty form state
- optionally reloading a failed fetch if local edits would be lost

### 7. Mobile And Layout Behavior

Settings should remain usable on narrow viewports:

- stack section cards vertically
- keep primary save action visible near the top and bottom if needed
- prevent long array fields from breaking horizontal layout

## Test Scenarios

- Secret fields are empty on initial load while configured flags show existing state.
- Minimal PATCH payload includes only changed fields.
- Password confirmation mismatch blocks submit.
- Unsupported or unavailable platform fields are hidden.
- Leaving the page with unsaved changes opens confirmation modal.
- Successful save refreshes values and clears dirty state.
- Failed save preserves draft values and displays the error.

## Regression Checklist

- Header navigation still works after form logic is added.
- Workspace return path does not lose open tabs or sessions.
- Existing auth-expired flow still works when settings API returns 401.
- No unsupported settings appear in the UI.

## Definition Of Done

- The settings page supports all approved Step 5 editable fields.
- Password rotation and SMTP secret updates follow write-only handling.
- Dirty tracking, validation, and save feedback are complete.
- Restart-required settings remain absent from the UI.
