# Phase 6 Plan

## Integration, Regression, And Release

## Objective

Validate that the Step 5 server and frontend tracks work together end to end, that runtime changes apply according to scope, and that the implementation remains narrowly scoped and safe for the existing codebase.

## Prerequisites

- Server Phases 1 through 3 completed
- Frontend Phases 4 and 5 completed
- Verification templates available under `docs/plan/step5/verification`

## Requirement Mapping

| Step 5 Need | Phase 6 Output |
| --- | --- |
| Settings workflow works end to end | integrated manual and automated verification |
| Runtime-safe changes apply correctly | cross-stack scope validation |
| Existing behavior is not broken | regression pass |
| Codebase remains protected | final diff and scope review |

## Implementation Checklist

- Run backend regression checks
- Run frontend regression checks
- Execute cross-stack settings scenarios
- Validate `server/config.json5` persistence quality
- Complete all verification templates
- Complete `final-validation.md`
- Confirm final diff remains Step 5 scoped

## Detailed Guidance

### 1. Backend Regression

Confirm:

- auth login, logout, refresh, and status still work
- session creation, resize, delete, and SSE output still work
- file manager operations still enforce path and size safety rules
- HTTPS startup and SSL behavior remain unchanged

### 2. Frontend Regression

Confirm:

- workspace loads after authentication
- sidebar, tab bar, terminal, file manager, and viewer workflows still function
- mobile sidebar and header actions still behave correctly
- auth-expired flow still redirects users back to login

### 3. Cross-Stack Integration

Execute the scenarios in [integration-test-guide.md](C:/Work/git/_Snoworca/ProjectMaster/docs/plan/step5/integration-test-guide.md), especially:

- settings load
- password rotation
- 2FA enable and disable
- PTY default updates for new sessions
- file-manager updates for later requests
- CORS updates for later browser requests
- invalid save rollback behavior

### 4. Config File Validation

Inspect `server/config.json5` after representative saves:

- JSON5 remains parseable
- comments and operator-facing structure are preserved
- only intended fields changed
- backup file exists when expected

### 5. Scope And Safety Review

Before release:

- review `git diff --stat` or equivalent
- ensure no unrelated refactors slipped in
- confirm restart-required settings are not exposed
- confirm secrets are not present in logs, snapshots, or responses

## Test Scenarios

- Full open-edit-save-return workflow succeeds from the header settings button.
- Changing `auth.durationMs` affects later logins only.
- Changing password requires the current password and updates later login behavior.
- Enabling 2FA changes later login flow to OTP verification.
- Disabling 2FA restores direct login flow.
- Changing PTY defaults affects only newly created sessions.
- Changing file-manager limits affects later file operations immediately.
- Changing CORS allowlist affects later preflight requests.
- Invalid save attempts do not corrupt runtime state or `config.json5`.

## Regression Checklist

- Existing workspace session state survives visiting settings.
- Existing API consumers remain compatible.
- No newly exposed secrets or restart-only controls are visible.
- Dirty worktree outside Step 5 files remains untouched.

## Definition Of Done

- Step 5 passes integrated server, frontend, and cross-stack checks.
- Verification templates are completed with evidence.
- Final validation document is completed and reviewed.
- Release decision is based on documented pass or fail criteria, not informal confidence.
