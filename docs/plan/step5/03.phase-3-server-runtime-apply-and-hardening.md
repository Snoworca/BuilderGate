# Phase 3 Plan

## Server Runtime Apply And Hardening

## Objective

Turn persisted Step 5 settings into live runtime behavior for the supported subset of keys. This phase wires the new settings service into bootstrap and service flows, then hardens rollback and regression behavior.

## Prerequisites

- Phase 2 completed
- Settings API contract stable
- Runtime update seams from Phase 1 implemented

## Requirement Mapping

| Step 5 Need | Phase 3 Output |
| --- | --- |
| Runtime-safe settings apply without restart | service update wiring |
| 2FA changes affect later logins | runtime service accessors |
| CORS changes affect later requests | dynamic CORS delegate |
| Failures do not leave partial runtime state | rollback and apply summary |

## Implementation Checklist

- Refactor bootstrap to use runtime service accessors
- Apply settings changes after successful persistence
- Hot-swap `TwoFactorService` when related settings change
- Replace static CORS options with a runtime-backed delegate
- Apply PTY, session, auth duration, and file-manager changes to live services
- Add rollback protection for runtime apply failures
- Add targeted integration tests

## Detailed Guidance

### 1. Bootstrap Runtime Owners In `server/src/index.ts`

Current bootstrap builds `AuthService`, optional `TwoFactorService`, auth middleware, and `FileService` with fixed startup config. Step 5 should keep that structure but add runtime owners:

- `runtimeConfigStore`
- `settingsService`
- `getAuthService()`
- `getTwoFactorService()`
- `getFileService()`

Avoid a broad inversion-of-control refactor. Keep bootstrap readable and local.

### 2. Refactor Auth Routes For Runtime Access

`createAuthRoutes` currently receives fixed `authService` and `twoFactorService` instances. Introduce a runtime accessor variant so login and verify endpoints always use the latest services after Step 5 saves.

Expected effect:

- password validation uses updated password immediately after save
- 2FA enable or disable affects the next login attempt
- token issuance uses the latest session duration for new logins and refreshes

### 3. Apply Runtime Changes By Owner

Map supported fields to owners:

- `auth.password`, `auth.durationMs` -> `AuthService.updateRuntimeConfig`
- `twoFactor.*` -> recreate or disable `TwoFactorService`
- `security.cors.*` -> update runtime CORS snapshot
- `pty.*`, `session.idleDelayMs` -> `sessionManager.updateRuntimeConfig`
- `fileManager.*` -> `fileService.updateConfig`

Treat apply scopes explicitly:

- `immediate`: file-manager and CORS behavior
- `new_logins`: auth duration, password, 2FA
- `new_sessions`: PTY defaults and idle behavior

### 4. Replace Static CORS Options

Current code calls `app.use(cors(corsOptions))` with startup values. Replace this with a delegate that reads the latest CORS snapshot on each request.

Requirements:

- preserve current fallback behavior for empty origins in development
- use runtime values for credentials and max age
- fail closed on invalid origin data

### 5. Rollback And Failure Semantics

If runtime application fails after persistence:

- capture a pre-apply runtime snapshot
- attempt to restore service state from the snapshot
- surface a clear apply summary in the API response or error path

The preferred ordering is:

1. validate patch
2. persist config safely
3. apply runtime changes
4. on apply failure, restore runtime state and report failure

Do not attempt full file rollback unless necessary. The initial goal is runtime consistency plus clear operator feedback. A follow-up restore of the config file may be added if testing shows it is required.

## Test Scenarios

- Auth duration change affects next login token lifetime.
- Password rotation invalidates old password on the next login attempt.
- Enabling 2FA causes later logins to return `requires2FA`.
- Disabling 2FA restores direct login flow without restart.
- PTY default cols, rows, shell, and backend apply to newly created sessions only.
- Updated idle delay affects later idle transitions without breaking active sessions.
- File manager changes affect later directory and file validation requests immediately.
- CORS changes affect later preflight and API requests.
- Simulated runtime apply failure restores prior in-memory state.

## Regression Checklist

- Health route and HTTPS startup behavior remain unchanged.
- Existing sessions stay connected after unrelated settings saves.
- Auth middleware still validates previously issued tokens correctly.
- Route registration order remains predictable and readable.

## Definition Of Done

- Supported settings apply at runtime according to documented scopes.
- Auth routes consume live service references rather than stale captures.
- Dynamic CORS works for later requests.
- Runtime apply failures do not leave services in a partially updated state.
