# Architecture Design

## BuilderGate - Step 5: Runtime Settings Page

## System Context

Step 5 adds an authenticated settings screen that can edit most runtime-safe values from `server/config.json5` and the application password. The current codebase is a single-process server plus a React frontend without route-based page transitions. The architecture must therefore introduce runtime configuration seams without destabilizing startup-time service wiring.

## Existing Constraints

| Constraint | Current Observation | Architectural Consequence |
| --- | --- | --- |
| Frontend shell | Main UI is a single app shell, not a routed SPA | Settings must be another in-app screen, not a router migration |
| Config access | Many services read startup config directly or cache constructor values | A runtime store and service update methods are required |
| Auth routes | Auth and 2FA flows use service instances created during bootstrap | Auth routes need runtime accessors for hot-swapped services |
| Session behavior | Session timeout derives from current config usage | Future sessions must see new values without breaking active sessions |
| File manager behavior | File validation limits are read from config-backed service state | File service needs runtime update support for new requests |
| CORS | Express CORS behavior is wired at startup | Use a delegate function backed by runtime settings |
| Config persistence | `config.json5` contains comments and is hand-editable | Writer must patch editable values while preserving structure/comments |

## Target Backend Architecture

### Core Components

1. `RuntimeConfigStore`
   - Holds the current editable runtime snapshot.
   - Exposes immutable read models for routes and services.
   - Knows which keys are editable, excluded, secret, or platform-specific.

2. `ConfigFileRepository`
   - Reads `server/config.json5`.
   - Applies a section-aware patch to editable keys only.
   - Validates the merged document and writes a backup before replacement.

3. `SettingsService`
   - Coordinates validation, secret handling, persistence, and runtime application.
   - Produces an `EditableSettingsSnapshot` for `GET /api/settings`.
   - Applies a `SettingsPatchRequest` and returns runtime apply results.

4. `settingsRoutes`
   - Authenticated HTTP surface for fetching and updating settings.
   - Never returns secret plaintext values.

5. Runtime-aware service seams
   - `AuthService.updateRuntimeConfig(...)`
   - `SessionManager.updateRuntimeConfig(...)`
   - `FileService.updateConfig(...)`
   - `TwoFactorService` recreation and teardown contract
   - Dynamic CORS delegate supplied from runtime store

### Service Access Pattern

Auth-related routes should not capture a stale `TwoFactorService` instance. Replace fixed service references with a runtime accessor object, for example:

- `getAuthService()`
- `getTwoFactorService()`
- `getSettingsService()`

This keeps route registration stable while allowing Step 5 to swap 2FA behavior after a successful save.

## Target Frontend Architecture

### Shell Strategy

- Keep the existing app shell and workspace state.
- Add a lightweight screen state such as `type AppScreen = 'workspace' | 'settings'`.
- Add a header settings button in the top-right area before or near logout.
- Render `SettingsPage` in the same authenticated shell instead of introducing React Router.

### Settings Page Responsibilities

- Fetch the current settings snapshot from `GET /api/settings` on entry.
- Build sectioned forms around the editable whitelist only.
- Track dirty state and build minimal PATCH payloads.
- Warn before leaving with unsaved changes.
- Show per-field apply scope so users know which changes affect current or future sessions.

## Data Contracts

### `EditableSettingsSnapshot`

Contains:

- metadata for editable fields, excluded groups, and apply scopes
- current non-secret values
- boolean flags such as whether SMTP password is configured
- platform-specific availability flags where needed

Must exclude:

- restart-required values
- JWT secrets
- raw encrypted secrets
- any unsupported nested objects not consumed by the current runtime

### `SettingsPatchRequest`

Contains only changed keys, for example:

- `auth.currentPassword`
- `auth.newPassword`
- `auth.durationMs`
- `twoFactor.enabled`
- `twoFactor.smtp.host`
- `security.cors.allowedOrigins`
- `pty.defaultCols`

### Apply Scope Model

Use a field-level scope such as:

- `immediate`
- `new_logins`
- `new_sessions`

This model appears both in backend metadata and in the UI.

## Sequence Flows

### Load Settings

1. User opens settings screen from the header.
2. Frontend requests `GET /api/settings`.
3. Settings API returns editable values plus metadata and secret presence flags.
4. Frontend renders forms using the returned capability map.

### Save Settings

1. User submits a minimal PATCH payload.
2. Server validates current password requirement, field types, and whitelist membership.
3. `ConfigFileRepository` merges patch into `config.json5` and writes a backup.
4. `SettingsService` applies runtime-safe changes.
5. API returns updated snapshot and apply result summary.

### 2FA Toggle Hot-Swap

1. PATCH changes `twoFactor.enabled` or SMTP settings.
2. Server persists encrypted secret if supplied.
3. `SettingsService` replaces or disables the active `TwoFactorService`.
4. Next login attempt uses the new 2FA behavior without a restart.

### CORS Runtime Apply

1. PATCH changes allowed origins, methods, or headers.
2. Runtime store updates the CORS snapshot.
3. Express CORS delegate evaluates the new values on subsequent requests.

## Change Safety Boundaries

- Do not introduce a general config editor that exposes unsupported keys.
- Do not refactor the whole app to React Router for one settings page.
- Do not rewrite all services to depend on a new DI container.
- Do not serialize the entire JSON5 file with a naive `JSON.stringify`.
- Do not return secret plaintext values to the browser.
- Do not make restart-required settings appear editable.

## Architecture Readiness Checklist

- Editable whitelist defined and reviewed against real runtime consumers.
- Hot-apply owners identified for every supported field.
- Secret handling path separated from normal value serialization.
- Frontend screen transition preserves current workspace state.
- Config persistence path preserves file validity and comments.
- Rollback path defined for runtime-apply failure after persistence attempt.
