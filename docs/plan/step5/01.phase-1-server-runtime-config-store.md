# Phase 1 Plan

## Server Runtime Config Store

## Objective

Introduce the minimal backend seams required to describe editable runtime settings without changing external behavior yet. This phase defines the editable whitelist, runtime metadata, and service update hooks that later phases will use.

## Detailed Guidance

### 1. Define Settings Types

Create `server/src/types/settings.types.ts` with:

- `EditableSettingsKey`
- `EditableSettingsSnapshot`
- `EditableSettingsValues`
- `SettingsPatchRequest`
- `FieldApplyScope`
- `FieldCapability`

The editable model should include these groups:

- `auth.durationMs`
- password change pseudo-fields for API use only
- `twoFactor.enabled`
- `twoFactor.email`
- `twoFactor.otpLength`
- `twoFactor.otpExpiryMs`
- `twoFactor.smtp.host`
- `twoFactor.smtp.port`
- `twoFactor.smtp.secure`
- `twoFactor.smtp.auth.user`
- `twoFactor.smtp.auth.password` as write-only
- `twoFactor.smtp.tls.rejectUnauthorized`
- `twoFactor.smtp.tls.minVersion`
- `security.cors.allowedOrigins`
- `security.cors.credentials`
- `security.cors.maxAge`
- `pty.termName`
- `pty.defaultCols`
- `pty.defaultRows`
- `pty.useConpty`
- `pty.maxBufferSize`
- `pty.shell`
- `session.idleDelayMs`
- `fileManager.maxFileSize`
- `fileManager.maxDirectoryEntries`
- `fileManager.blockedExtensions`
- `fileManager.blockedPaths`
- `fileManager.cwdCacheTtlMs`

Explicitly exclude:

- `server.port`
- `ssl.*`
- `logging.*`
- `auth.maxDurationMs`
- `auth.jwtSecret`
- `fileManager.maxCodeFileSize`
- `bruteForce.*` until there is a runtime consumer

### 2. Add `RuntimeConfigStore`

Responsibilities:

- Materialize an editable snapshot from the validated config singleton.
- Expose metadata for the frontend, including apply scope and write-only flags.
- Normalize platform-specific capabilities such as `pty.useConpty` on non-Windows.
- Provide helper methods:
  - `getSnapshot()`
  - `getEditableValues()`
  - `getFieldCapabilities()`
  - `isEditable(path)`
  - `mergeEditablePatch(patch)`

### 3. Add Runtime Update Hooks

Update current services without changing their public responsibilities:

- `AuthService`
  - add `updateRuntimeConfig(next: Pick<AuthConfig, 'password' | 'durationMs'>)`
  - re-decrypt and replace cached password when changed
  - keep `jwtSecret` immutable for Step 5

- `SessionManager`
  - replace `readonly IDLE_DELAY_MS` and `readonly MAX_BUFFER_SIZE` with mutable runtime values
  - add `updateRuntimeConfig({ idleDelayMs, pty })`
  - use latest values for new timers and new buffer truncation

- `FileService`
  - add `updateConfig(next: FileManagerConfig)`
  - keep current cache map, but use new limits for later operations

- `TwoFactorService`
  - document teardown requirement such as `destroy()` or equivalent pending-auth cleanup
  - if no teardown is needed, state it explicitly and keep recreation localized to later phases

### 4. Apply Scope Definitions

Capture scope metadata now so later API and UI work stay aligned:

- `auth.durationMs`: `new_logins`
- password change: `immediate` for validation path after save
- `twoFactor.*`: `new_logins`
- `security.cors.*`: `immediate`
- `pty.*`: `new_sessions`
- `session.idleDelayMs`: `new_sessions`
- `fileManager.*`: `immediate`

## Test Scenarios

- Snapshot excludes restart-required and unsupported keys.
- Secret fields appear as write-only capability metadata, not plaintext values.
- Platform-specific fields are marked unavailable when the runtime cannot honor them.
- `AuthService.updateRuntimeConfig` changes later password validation and token duration behavior.
- `SessionManager.updateRuntimeConfig` affects new idle timers and later buffer truncation without crashing existing sessions.
- `FileService.updateConfig` applies new validation limits to later file operations.

## Regression Checklist

- Existing login flow remains unchanged before any settings endpoint is introduced.
- Existing session creation still uses current config defaults.
- File manager operations behave exactly as before with unchanged config values.
- No route registration or API contract changes are visible yet.

## Definition Of Done

- Runtime-editable whitelist is encoded in one backend location.
- Runtime consumers have focused update hooks with tests.
- Excluded keys are documented and cannot leak into later snapshots.
- No external behavior changes beyond new internal seams.
