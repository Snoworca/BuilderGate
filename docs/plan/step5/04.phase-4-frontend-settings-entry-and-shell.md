# Phase 4 Plan

## Frontend Settings Entry And Shell

## Objective

Add the settings entry point to the authenticated UI and introduce a settings screen shell that coexists with the existing workspace layout. This phase stops before full form editing; it only establishes navigation, loading, and state preservation.

## Prerequisites

- Phase 2 server API available for `GET /api/settings`
- Review current shell in `frontend/src/App.tsx`
- Review current header in `frontend/src/components/Header/Header.tsx`
- Review existing modal reuse pattern in `frontend/src/components/Modal/ConfirmModal.tsx`

## Requirement Mapping

| Step 5 Need | Phase 4 Output |
| --- | --- |
| Settings page reachable from top-right | header settings button |
| Settings lives inside the app | in-shell screen switch |
| Existing workspace remains intact | preserved session and tab state |
| Settings screen can load data | initial fetch and loading shell |

## Implementation Checklist

- Update `frontend/src/components/Header/Header.tsx`
- Update `frontend/src/components/Header/Header.css`
- Add screen state to `frontend/src/App.tsx`
- Add `frontend/src/components/Settings/SettingsPage.tsx`
- Add `frontend/src/components/Settings/SettingsPage.css`
- Add `frontend/src/services/settingsApi.ts` or equivalent typed client module
- Add `frontend/src/types/settings.ts`
- Render loading, error, and empty-state shell for settings

## Detailed Guidance

### 1. Header Entry Point

Add a settings button in the header's right-side action group so it appears in the top-right corner alongside logout.

Requirements:

- desktop and mobile layouts remain stable
- button is accessible with clear label text such as `Settings`
- current path display still truncates correctly
- logout button behavior is unchanged

### 2. App Screen State In `App.tsx`

The current app renders the workspace directly after authentication. Introduce a local screen state:

- `workspace`
- `settings`

Behavior:

- default screen is `workspace`
- clicking the header settings button opens `settings`
- closing the settings page returns to `workspace`
- existing `useSession`, `useTabManager`, `useCwd`, and modal state stay mounted unless there is a clear reason to unmount them

Preserving mounted workspace state is preferred because Step 5 should not reset active sessions, tabs, or viewer state when users inspect settings.

### 3. Add `SettingsPage` Shell

Create a page-level component responsible for:

- fetching `GET /api/settings` on entry
- showing load, error, and refresh states
- rendering section containers even before full edit controls are complete
- exposing `onBack` and later `onSaved` hooks

The initial shell can render read-only sections or placeholders in this phase. Form editing is Phase 5.

### 4. Add Typed Client Module

Create `frontend/src/services/settingsApi.ts` or extend `frontend/src/services/api.ts` with a separate export that follows the existing auth fetch pattern.

Minimum methods:

- `getSettings()`
- optionally `patchSettings()` stub for the next phase

Keep token handling consistent with current `authFetch` behavior so 401 responses still trigger re-authentication.

### 5. Layout And Styling

Reuse the existing dark shell visual language:

- align spacing with current header and content areas
- avoid introducing a second global page chrome
- keep the page readable on mobile widths
- reserve space for future per-section forms and action buttons

## Test Scenarios

- Header shows a settings button in the top-right area.
- Clicking the button switches the content area to settings.
- Returning from settings restores the prior workspace view without losing session or tab state.
- Settings page requests the snapshot once on entry and renders loading or error UI.
- 401 from settings API triggers the same auth-expired handling as other protected endpoints.

## Regression Checklist

- Session selection, tab switching, and terminal rendering behave the same in workspace mode.
- Header mobile menu still works.
- Logout still works from the header.
- No router dependency or route URL migration is introduced.

## Definition Of Done

- Authenticated users can open and leave a settings screen from the header.
- Workspace state remains intact across the transition.
- Settings page shell can load current settings data.
