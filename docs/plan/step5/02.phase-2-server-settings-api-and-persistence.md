# Phase 2 Plan

## Server Settings API And Persistence

## Objective

Add an authenticated settings API and a safe persistence layer for `server/config.json5`. This phase exposes runtime-editable values to the frontend and persists validated changes while preserving JSON5 comments and structure.

## Prerequisites

- Phase 1 completed
- Runtime whitelist and apply scopes finalized
- Existing config loading behavior reviewed in `server/src/utils/config.ts`
- Existing schema validation reviewed in `server/src/schemas/config.schema.ts`

## Requirement Mapping

| Step 5 Need | Phase 2 Output |
| --- | --- |
| Settings page can load current values | `GET /api/settings` |
| Settings page can save runtime-safe edits | `PATCH /api/settings` |
| Password and secrets can be changed safely | validated secret-aware patch path |
| Restart-required settings stay out of scope | whitelist enforcement in API and repository |

## Implementation Checklist

- Add `server/src/services/ConfigFileRepository.ts`
- Add `server/src/services/SettingsService.ts`
- Add `server/src/routes/settingsRoutes.ts`
- Mount settings routes behind existing auth middleware
- Implement `GET /api/settings`
- Implement `PATCH /api/settings`
- Add config backup, merge validation, and rollback rules
- Add tests for redaction, validation, and backup behavior

## Detailed Guidance

### 1. Create `ConfigFileRepository`

This repository should centralize all Step 5 writes to `server/config.json5`.

Required behavior:

- read the raw JSON5 file from the same path used by `server/src/utils/config.ts`
- patch only editable sections
- preserve comments and surrounding formatting as much as possible
- validate merged content with `JSON5.parse` and `configSchema.parse`
- write a `.bak` file before replacing the source
- support dry-run validation for tests

Avoid reusing the startup-time password auto-encryption helper directly. That helper is read-path oriented and limited to fixed password locations. Step 5 needs broader section-aware patching with explicit control over which secrets are being updated.

### 2. Create `SettingsService`

Responsibilities:

- build the `EditableSettingsSnapshot` for `GET /api/settings`
- validate incoming `SettingsPatchRequest`
- require current password when changing `auth.password`
- enforce new password confirmation and minimum validation at the API boundary
- encrypt any new secret values using `CryptoService`
- pass the merged values to `ConfigFileRepository`
- return updated snapshot plus apply-result metadata

Recommended helper methods:

- `getSettingsSnapshot()`
- `validatePatch(patch)`
- `savePatch(patch, actorContext)`
- `redactResponse(snapshot)`

### 3. Route Contract

Create `server/src/routes/settingsRoutes.ts` with:

- `GET /api/settings`
- `PATCH /api/settings`

Response shape should include:

- `values`
- `capabilities`
- `secretState`
- `excludedSections`
- `applySummary` after PATCH

Validation rules:

- reject unknown keys
- reject restart-required keys even if structurally valid
- reject password change without correct current password
- reject 2FA enablement when required email or SMTP fields are missing
- reject invalid arrays such as malformed origins or empty blocked extensions

### 4. Mount Routes In `server/src/index.ts`

Mount settings routes with the same authenticated boundary used for session and file routes:

- create authenticated middleware once from `AuthService`
- add `/api/settings` after auth routes
- keep the route addition isolated; do not refactor unrelated bootstrap code in this phase

### 5. Secret Handling

Follow a write-only model:

- `auth.password` never returned
- `twoFactor.smtp.auth.password` never returned
- include flags such as `passwordConfigured` and `smtpPasswordConfigured`
- only persist secret fields when the PATCH request explicitly includes new values

## Test Scenarios

- `GET /api/settings` returns editable values and metadata without secret plaintext.
- `PATCH /api/settings` rejects unsupported keys with a 400-class response.
- Password change fails when current password is missing or incorrect.
- 2FA enablement fails if required email or SMTP fields are incomplete.
- Config repository creates a backup before writing.
- Invalid merged config is rejected before replacing the file.
- JSON5 comments remain present in fixture snapshots after a successful patch.

## Regression Checklist

- Existing `/api/auth/*` and `/api/sessions/*` behavior remains unchanged.
- Startup config loading still succeeds from an untouched `config.json5`.
- API does not expose raw encrypted values.
- Save failures do not partially write the config file.

## Definition Of Done

- Authenticated settings API exists and is type-safe.
- Persistence layer preserves JSON5 validity and comments.
- Password and SMTP secret updates are encrypted before persistence.
- Unsupported fields fail closed.
