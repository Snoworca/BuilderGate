## Phase 3: README/docs 정책과 최종 검증 정리

### 목표

README와 docs policy가 새 build 계약을 정확히 설명하게 하고, 구현 완료 후 `npm run build` 전체 대상 검증 또는 대체 가능한 build smoke 검증을 수행할 수 있게 최종 검증 경로를 정리한다.

## 선행 조건

- Phase 1 완료.
- Phase 2 완료.

## 기존 코드 재사용 힌트

- `tools/daemon/docs-policy.js`의 required/forbidden pattern 체계를 유지한다.
- `tools/daemon/docs.test.js`의 packaged README discovery 방식을 확장한다.
- `README.md`의 native daemon build/run 섹션을 기존 구조 안에서 수정한다.

## 코드 중복 주의

- README에 같은 build script 표를 여러 곳에 중복 작성하지 않는다. 한 섹션을 기준으로 명령과 산출물 구조를 정리한다.
- docs test에서 target 목록을 새로 하드코딩하더라도 `win-amd64`, `linux-amd64`, `win-arm64`, `linux-arm64`, `macos-arm64` 한 곳에 모아 관리한다.

## 구현 체크리스트

- [ ] Phase 3-1: README의 `npm run build` 설명을 전체 지원 대상 5개 생성으로 변경한다. `AC-014`, `AC-015`
- [ ] Phase 3-2: README에 개별 script `build:windows-amd64`, `build:linux-amd64`, `build:windows-arm64`, `build:linux-arm64`, `build:macos-arm64`를 주 표기로 문서화한다. `FR-8-013`
- [ ] Phase 3-3: README에 macOS는 ARM64만 지원하며 `macos-arm64`는 raw 실행파일과 `BuilderGate.app`을 함께 제공한다고 명시한다. `FR-RUN-013`
- [ ] Phase 3-4: README에 기존 `build:daemon-*` ARM64 scripts가 호환 alias임을 짧게 남긴다. `FR-8-013`
- [ ] Phase 3-5: `tools/daemon/docs.test.js`의 packaged README 후보에 `dist/bin/win-amd64`, `dist/bin/linux-amd64`, `dist/bin/win-arm64`, `dist/bin/linux-arm64`, `dist/bin/macos-arm64`를 포함한다. `AC-014`, `AC-015`
- [ ] Phase 3-6: docs policy가 PM2 forbidden pattern을 계속 거부하는지 확인한다. `FR-RUN-012`
- [ ] Phase 3-7: 최종 검증에서 `node --test tools/daemon/build-daemon-exe.test.js tools/daemon/docs.test.js`, `npm run test:daemon`, `npm run test:docs`, `git diff --check`를 실행한다. `TEST-8-017`
- [ ] Phase 3-8: 시간이 허용되면 `node tools/build-daemon-exe.js --required-amd64 --skip-runtime-install`로 빠른 build smoke를 수행하고, release 준비 시 `npm run build` 전체 5개 대상 빌드를 수행한다. `AC-015`

## 언어+버전 주의사항

- README와 Markdown 문서는 UTF-8로 유지한다.
- docs test는 `node:test` CommonJS 패턴을 유지한다.
- PowerShell/Unix 명령 예시는 기존 README 스타일을 따른다.

## LLM 코딩 주의사항

- Simplicity: README build 섹션만 수정하고 기능 설명 전체를 재작성하지 않는다.
- Surgical: PM2 금지 문구 검증을 약화하지 않는다.
- User Contract: `npm run build`가 전체 지원 대상이라는 사용자 결정을 README와 tests에 일관되게 반영한다.

## 테스트 시나리오

### 정상 케이스

- Given: root README가 갱신되어 있다.
  When: `npm run test:docs`를 실행한다.
  Then: native daemon required keywords와 build target 설명이 통과하고 PM2 forbidden pattern이 없다.

- Given: `dist/bin/win-amd64/README.md`와 `dist/bin/linux-amd64/README.md`가 존재한다.
  When: docs test를 실행한다.
  Then: packaged README 정책 검증 대상에 포함된다.

- Given: `dist/bin/macos-arm64/README.md`가 존재한다.
  When: docs test를 실행한다.
  Then: macOS ARM64 packaged README도 정책 검증 대상에 포함된다.

### 예외 케이스

- Given: README에 `npm run build`가 ARM64 3개만 생성한다고 남아 있다.
  When: review 또는 docs test를 수행한다.
  Then: SRS 불일치로 수정한다.

- Given: README에 production PM2 guidance가 다시 추가된다.
  When: `npm run test:docs`를 실행한다.
  Then: forbidden pattern으로 실패한다.

### 경계값 테스트

- Given: 아직 build output이 없어 packaged README가 존재하지 않는다.
  When: docs test를 실행한다.
  Then: root README 정책은 검증되고, packaged README 검증은 기존 정책에 따라 적어도 하나의 packaged README가 있을 때 수행된다. 구현 중 필요하면 테스트 전제와 메시지를 새 target 구조에 맞게 조정한다.
