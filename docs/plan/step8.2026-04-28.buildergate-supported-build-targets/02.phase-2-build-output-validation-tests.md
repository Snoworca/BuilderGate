## Phase 2: 산출물 검증과 build unit test 강화

### 목표

각 대상별 배포본이 SRS의 산출물 계약을 충족하는지 `validateBuildOutput()`과 `tools/daemon/build-daemon-exe.test.js`로 검증한다. 특히 `server/dist/public/index.html`, PM2 부재, 플랫폼별 icon artifact, macOS `BuilderGate.app` bundle을 명시적으로 보호한다.

## 선행 조건

- Phase 1 완료.
- `TARGET_PROFILES`, `ALL_SUPPORTED_TARGETS`, npm scripts가 정렬되어 있어야 한다.

## 기존 코드 재사용 힌트

- `validateBuildOutput(outputDir, { platform })` 구조를 유지한다.
- `getExecutableNames(platform)`를 통해 app/stop executable 이름을 판별한다.
- `createBuildOutputFixture()` test helper를 확장해 frontend public asset과 플랫폼별 icon fixture를 만든다.
- `createMacAppBundle()` 기존 테스트를 macOS output validation과 연결한다.

## 코드 중복 주의

- 플랫폼별 required path 검증을 별도 validator 함수로 과도하게 분리하지 않는다. `validateBuildOutput()` 안에서 명확한 required list 구성으로 충분하다.
- PM2 부재 검증은 기존 `server/node_modules/pm2` 검사와 README forbidden pattern 검사를 재사용한다.

- [ ] Phase 2-4: macOS `BuilderGate.app` 내부 구조(`Contents/Info.plist`, `Contents/MacOS/BuilderGate`, `Contents/Resources/BuilderGate.icns`, `Contents/Resources/runtime/buildergate`) 검증을 유지/강화한다. `FR-RUN-013`

## 언어+버전 주의사항

- `node:test`와 `node:assert/strict` 기존 패턴을 유지한다.
- filesystem fixture는 `fs.writeFileSync(..., 'utf8')`를 사용한다.
- 경로 비교는 Windows/Linux 모두 통과하도록 `path.join()`과 정규식 `[\\\\/]` 패턴을 사용한다.

## LLM 코딩 주의사항

- Simplicity: 실제 build를 수행하는 통합 테스트를 unit test에 넣지 않는다. fixture 기반 validator test로 빠르게 보호한다.
- Surgical: `copyRuntimeFiles()`, `buildExe()`, `ensureTargetNodeRuntime()` 흐름은 산출물 검증에 필요한 범위 외에는 변경하지 않는다.
- Safety: Linux 검증 완화는 `.ico`/`.icns`를 금지하는 것이 아니라 필수에서 제외하는 것이다.

## 테스트 시나리오

### 정상 케이스

- Given: Windows output fixture에 exe, stop exe, `server/dist/public/index.html`, bundled `node.exe`, `BuilderGate.svg`, `BuilderGate.ico`, README, config가 있다.
  When: `validateBuildOutput(outputDir, { platform: 'win32' })`를 실행한다.
  Then: 통과한다.

- Given: Linux output fixture에 `buildergate`, `buildergate-stop`, `server/dist/public/index.html`, bundled `node`, `BuilderGate.svg`, README, config가 있고 `.ico`/`.icns`는 없다.
  When: `validateBuildOutput(outputDir, { platform: 'linux' })`를 실행한다.
  Then: 통과한다.

- Given: macOS output fixture에 raw executables, `BuilderGate.icns`, `BuilderGate.app` bundle, runtime, README, config가 있다.
  When: `validateBuildOutput(outputDir, { platform: 'darwin' })`를 실행한다.
  Then: 통과한다.

### 예외 케이스

- Given: output fixture에서 `server/dist/public/index.html`이 없다.
  When: `validateBuildOutput()`을 실행한다.
  Then: frontend asset 누락 오류로 실패한다.

- Given: Windows output fixture에서 `BuilderGate.ico`가 없다.
  When: `validateBuildOutput(..., { platform: 'win32' })`를 실행한다.
  Then: Windows icon artifact 누락 오류로 실패한다.

- Given: macOS output fixture에서 `BuilderGate.app` 또는 `Contents/Resources/BuilderGate.icns`가 없다.
  When: `validateBuildOutput(..., { platform: 'darwin' })`를 실행한다.
  Then: app bundle 누락 오류로 실패한다.

- Given: `server/node_modules/pm2/package.json`이 있다.
  When: `validateBuildOutput()`을 실행한다.
  Then: PM2 runtime dependency 오류로 실패한다.

### 경계값 테스트

- Given: `copyIconAssets()`가 Linux output에 `.ico`/`.icns`를 추가로 생성했다.
  When: Linux output validation을 실행한다.
  Then: 추가 파일 존재와 무관하게 통과한다.
