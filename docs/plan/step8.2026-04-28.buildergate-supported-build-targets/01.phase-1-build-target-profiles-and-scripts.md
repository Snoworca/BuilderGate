## Phase 1: 빌드 대상 프로필과 npm script 계약 정렬

### 목표

`tools/build-daemon-exe.js`가 전체 지원 대상 5개를 명명된 profile로 해석하고, `package.json`의 npm scripts가 SRS와 사용자 결정에 맞는 build entry를 제공하도록 한다.

## 선행 조건

- 최신 SRS의 `FR-RUN-012`, `FR-RUN-013`, `AC-015` 확인.
- 기존 `build:daemon-exe` 단일 output 계약을 변경하지 않는다는 결정 확인.

## 기존 코드 재사용 힌트

- `tools/build-daemon-exe.js`의 `resolveTargetSpec()`와 `resolveBuildTargets()`를 재사용한다.
- `platformFromPkgTarget()`, `archFromPkgTarget()`, `profileNameFor()`를 유지하되 `amd64` profile alias 처리를 추가한다.
- `parseArgs()`의 기존 `--target`, `--targets`, `--profile`, `--all-arm64` 흐름을 확장한다.
- `package.json`의 기존 `build:daemon-exe`는 그대로 유지한다.

## 코드 중복 주의

- target metadata resolver를 별도 파일로 새로 만들지 않는다. 기존 `tools/build-daemon-exe.js` 내부 resolver를 확장한다.
- npm script 이름 매핑을 README와 테스트에 따로 하드코딩하기 전에 `package.json`을 기준으로 검증한다.

## 구현 체크리스트

- [ ] Phase 1-1: `ARM64_TARGET_PROFILES`를 일반화해 `TARGET_PROFILES`로 전환하고 `win-amd64`, `linux-amd64`, `win-arm64`, `linux-arm64`, `macos-arm64`를 포함한다. `FR-RUN-012`, `FR-RUN-013`
- [ ] Phase 1-2: `TARGET_PROFILE_ALIASES`에 `windows-amd64`, `windows-arm64`, `mac-arm64`, `darwin-arm64`를 포함한다. `FR-RUN-013`
- [ ] Phase 1-3: `REQUIRED_AMD64_TARGETS`, `ALL_ARM64_TARGETS`, `ALL_SUPPORTED_TARGETS`를 분리한다. `FR-RUN-012`, `FR-RUN-013`
- [ ] Phase 1-4: `parseArgs()`에 `--all-supported`와 `--required-amd64`를 추가하고 기존 `--all-arm64`는 유지한다. `AC-015`
- [ ] Phase 1-5: help output에 전체 지원 대상, 필수 amd64 대상, 추가 ARM64 대상을 구분해 출력한다. `AC-014`, `AC-015`
- [ ] Phase 1-6: `module.exports`에 새 target constants를 노출해 테스트가 직접 검증할 수 있게 한다. `TEST-8-017`
- [ ] Phase 1-7: `package.json`에서 `build`와 `build:daemon-all`이 `node tools/build-daemon-exe.js --all-supported`를 호출하게 한다. `AC-015`
- [ ] Phase 1-8: `build:windows-amd64`, `build:linux-amd64`, `build:windows-arm64`, `build:linux-arm64`, `build:macos-arm64` scripts를 추가한다. `FR-8-013`
- [ ] Phase 1-9: 기존 `build:daemon-win-arm64`, `build:daemon-linux-arm64`, `build:daemon-mac-arm64`, `build:daemon-macos-arm64` scripts는 새 ARM64 scripts의 호환 alias로 유지한다. `FR-8-013`

## 언어+버전 주의사항

- Node.js 20.x에서 실행되지만 pkg target은 `node18-*`이므로 런처/packager runtime에서 Node 20 전용 API에 과도하게 의존하지 않는다.
- CommonJS 파일이므로 `require`/`module.exports` 패턴을 유지한다.
- `var` 사용 금지, 기존 스타일대로 `const` 우선 및 필요 시 `let`을 사용한다.

## LLM 코딩 주의사항

- Simplicity: target profile metadata만 확장하고 build pipeline 전체를 재작성하지 않는다.
- Surgical: `build:daemon-exe`와 `parseArgs([])`의 단일 기본 target 동작을 바꾸지 않는다.
- Compatibility: 기존 ARM64 script는 제거하지 말고 alias로 유지한다.

## 테스트 시나리오

### 정상 케이스

- Given: `parseArgs(['--all-supported'])`
  When: `resolveBuildTargets()`를 호출한다.
  Then: `win-amd64`, `linux-amd64`, `win-arm64`, `linux-arm64`, `macos-arm64`가 이 순서로 반환되고 output은 `dist/bin/<profile>`이다.

- Given: `parseArgs(['--required-amd64'])`
  When: `resolveBuildTargets()`를 호출한다.
  Then: `win-amd64`, `linux-amd64`만 반환된다.

- Given: 루트 `package.json`
  When: scripts를 검사한다.
  Then: `build`와 `build:daemon-all`은 `--all-supported`를 호출하고 개별 5개 script가 각 profile을 호출한다.

### 예외 케이스

- Given: `parseArgs(['--profile', 'windows-amd64'])`
  When: `resolveBuildTargets()`를 호출한다.
  Then: alias가 `win-amd64`로 정규화된다.

- Given: `parseArgs(['--profile', 'macos-amd64'])`
  When: `resolveBuildTargets()`를 호출한다.
  Then: macOS amd64는 정의되지 않은 profile이므로 명시 pkg target이 아닌 이상 지원 profile로 처리하지 않는다.

### 경계값 테스트

- Given: `parseArgs([])`
  When: `resolveBuildTargets()`를 호출한다.
  Then: 기존 단일 기본 `dist/bin` output 동작이 유지된다.
