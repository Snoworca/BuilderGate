# Verification Hardening Addendum

작성일: 2026-04-29
근거: 5개 서브에이전트 검증 결과와 코드 재확인

## 최종 보강 판정

기존 진단은 유효하다. BuilderGate에는 실제 입력 손실을 만들 수 있는 hard drop 경로가 있다. 단, 계획 문서는 "한글 IME 누락의 확정 원인"처럼 읽히면 안 된다. 올바른 목표는 다음이다.

> 앱이 관측한 입력은 전달, bounded queue, explicit reject 중 하나로 끝나야 한다. 앱이 관측하지 못한 물리 키 입력은 손실로 단정하지 않고 xterm/browser/IME 계층의 residual risk로 분리한다.

## 수정된 핵심 계약

### 1. `inputSeqStart`/`inputSeqEnd` 단일 소유자

`inputSeqStart`/`inputSeqEnd`는 `TerminalContainer`의 `InputSequencer`가 단 한 번 부여한다.

- `TerminalView`는 `captureSeq`와 `compositionSeq`만 가진다.
- `TerminalView` queue entry에는 `inputSeqStart`/`inputSeqEnd`를 넣지 않는다.
- `TerminalView`가 flush한 raw `data`는 `TerminalContainer.handleInput()`으로 들어가고, 여기서 sequence range가 부여된다.
- 단일 input은 `inputSeqStart === inputSeqEnd`다.
- coalesced input은 포함된 첫/마지막 logical input의 range를 사용한다.
- `WebSocketContext`, `WsRouter`, `SessionManager`는 전달받은 sequence range를 보존만 한다.
- server가 legacy client로부터 sequence 없는 input을 받으면 sequence 없이 queue/write하되, sequence 기반 ordering을 새로 만들지 않는다.

이 규칙은 중복 sequence, coalescing range 혼선, replay queue ordering 오류를 막기 위한 강제 계약이다.

### 2. Transient barrier와 terminal closed state 분리

queue 가능한 상태는 짧고 회복 가능한 barrier로 제한한다.

Queue 가능:
- `restore-pending`
- `replay-pending`
- `initial-geometry-pending`
- `repair-server-not-ready`
- `ws-reconnecting-short`

Queue 금지, 즉시 explicit reject:
- `terminal-hidden`
- `terminal-disposed`
- `session-exited`
- `session-missing`
- `server-error`
- `auth-expired`
- `workspace-or-session-changed`
- `ws-closed-without-reconnect`

`captureAllowed = term && visible` 같은 단순 공식은 금지한다. 구현은 반드시 `captureState`와 `transportBarrierReason`을 별도로 계산해야 한다.

### 3. Enter 포함 입력의 timeout 안전정책

Enter 포함 input은 오래 지연되면 명령 실행 시점이 사용자의 시각적 문맥과 어긋난다.

- 정상 ACK 또는 짧은 transient barrier 해소 시에는 FIFO flush 가능.
- ACK timeout, reconnect timeout, visibility/context mismatch에서는 Enter 포함 input을 flush하지 않는다.
- timeout 경로에서는 `input_rejected` 또는 `transport_input_rejected`에 `reason=timeout-enter-safety`를 기록한다.
- Enter 없는 printable input도 TTL 초과 시 reject한다.

### 4. IME 회귀 방지 원칙

과거 한글 IME 문제의 재발을 막기 위해 아래는 금지한다.

- Space, Backspace, Enter를 IME 상태에서 직접 `onInput()`으로 수동 주입하지 않는다.
- `compositionend.data`, `compositionupdate.data`, `beforeinput.data`를 곧바로 PTY로 전송하지 않는다.
- helper textarea를 composition 중 `disabled`로 바꾸지 않는다.
- xterm CompositionHelper의 delayed textarea read를 우회하지 않는다.
- fallback injection은 기본값에서 절대 전송하지 않는다. telemetry가 xterm `onData` 손실을 입증하고, 중복 입력 없음이 수동 검증된 뒤 별도 flag에서만 허용한다.

### 5. Telemetry 타입과 프라이버시 경계

기존 debug/replay detail 타입은 primitive 값만 허용한다. 따라서 gate state와 metadata는 중첩 객체로 넣지 않고 flat field로 기록한다.

허용 예:

```ts
{
  inputByteLength: 9,
  inputCodePointCount: 3,
  inputGraphemeCount: 3,
  inputHasHangul: true,
  gateCaptureAllowed: true,
  gateTransportReady: false,
  gateRestorePending: true
}
```

금지 예:

```ts
{
  data: "가나다",
  gateState: { captureAllowed: true }
}
```

Telemetry 범위:
- localhost debug capture: 원문 없는 event tape, composition metadata, safe control preview만 허용.
- 일반 authenticated observability: input 원문, safe preview, key 이름 중 printable 가능성이 있는 값은 금지. byte/count/boolean/reason/seq만 허용.
- 모든 byte budget은 `Buffer.byteLength(data, 'utf8')` 또는 browser `TextEncoder` 기준으로 계산한다. JS `string.length`를 byte로 부르지 않는다.

Keydown privacy:
- printable `KeyboardEvent.key`와 printable `KeyboardEvent.code`는 저장하지 않는다.
- control/navigation allowlist에 포함된 키만 `safeKeyName`으로 기록한다.
- IME sentinel 판단에 필요한 `keyCode=229`는 기록 가능하지만, 다른 printable key code sequence를 사용자 입력 복원에 충분한 형태로 저장하면 안 된다.

### 6. Frontend deterministic test 전략

현재 `frontend/package.json`에는 unit test runner가 없다. Phase 문서는 "Unit test"를 기존 인프라처럼 가정하면 안 된다.

구현자는 아래 중 하나를 먼저 선택해야 한다.

1. queue/sequencer/IME transaction을 pure TypeScript helper로 분리하고, frontend에 명시적 test script를 추가한다.
2. test runner 추가가 범위 밖이면 Playwright debug hook으로 TTL/FIFO/overflow/Enter-safety를 결정적으로 검증한다.

둘 중 하나 없이 TTL/FIFO/coalescing을 "테스트 완료"로 처리하면 안 된다.

### 7. `input:rejected` 수신 경로

서버가 `input:rejected`를 보내는 경우 프론트는 이를 무시하면 안 된다.

- `WebSocketContext`는 `input:rejected`를 session handler 또는 terminal debug capture로 전달해야 한다.
- UI 표시는 이번 범위에서 추가하지 않는다.
- debug capture에는 `inputSeqStart`, `inputSeqEnd`, `reason`, `sourceLayer`를 기록한다.
- coalesced input이면 `inputSeqStart`/`inputSeqEnd`, 단일 input이면 두 값이 같은 sequence range를 기록한다.

### 8. Feature flag 기본값

Phase별 기본값:

- Phase 1: `observe`, 그리고 mode plumbing/test override 구현
- Phase 2~5: 기본값은 `observe`, 자동/수동 검증에서는 `queue`를 명시적으로 활성화
- Phase 6: 전체 검증 통과 후 기본값을 `queue`로 승격 가능
- `strict`: 별도 릴리스 판단 전까지 기본값 금지

설정 위치:
- 서버: env 또는 내부 runtime config로 `BUILDERGATE_INPUT_RELIABILITY_MODE`
- 프론트: 서버가 내려주는 runtime mode를 우선하고, localhost debug에서는 localStorage override 허용
- 프론트/서버 mode가 충돌하면 더 보수적인 mode를 사용한다.

## 구현자가 반드시 지켜야 할 순서

1. Phase 1에서 원문 없는 계측과 hard drop 위치 분류를 먼저 만든다.
2. Phase 2에서 capture/transport를 분리하되, `captureState`가 closed/error인 경우 queue하지 않는다.
3. Phase 3에서 server replay input queue를 추가하되, timeout Enter flush는 금지한다.
4. Phase 4에서 `InputSequencer`를 `TerminalContainer` 단일 소유로 구현한다.
5. Phase 5에서 IME transaction을 추가하되, fallback 전송은 기본값에서 observe만 한다.
6. Phase 6에서 실제 Windows Korean IME 수동 검증을 통과하기 전까지 queue mode를 기본값으로 승격하지 않는다.
