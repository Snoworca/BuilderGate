# Protocol Contracts and Canonical Reasons

작성일: 2026-04-29

## 목적

Phase 구현 중 reason 문자열, WebSocket 입력 검증, reconnect 상태 mapping이 흩어지지 않도록 단일 계약을 정의한다. 구현자는 이 문서를 source of truth로 사용한다.

## Input Reliability Mode

```ts
type InputReliabilityMode = 'observe' | 'queue' | 'strict';
```

기본값은 `observe`다. `strict`는 이번 계획에서 기본값으로 승격하지 않는다.

## Transport State

```ts
type TerminalInputBarrierReason =
  | 'none'
  | 'restore-pending'
  | 'replay-pending'
  | 'initial-geometry-pending'
  | 'repair-server-not-ready'
  | 'ws-reconnecting-short';

type TerminalInputClosedReason =
  | 'none'
  | 'terminal-hidden'
  | 'terminal-disposed'
  | 'session-exited'
  | 'session-missing'
  | 'server-error'
  | 'auth-expired'
  | 'workspace-or-session-changed'
  | 'ws-closed-without-reconnect';

type ReconnectState = 'connected' | 'reconnecting' | 'disconnected';
```

현재 `WebSocketContext` status가 `connected | reconnecting | disconnected`이므로 그대로 사용한다. `closed`라는 별도 reconnect state 문자열은 쓰지 않는다.

Mapping:
- `connected` -> `barrierReason='none'`
- `reconnecting` + short reconnect TTL 안 -> `barrierReason='ws-reconnecting-short'`
- `disconnected` 또는 reconnect TTL 초과 -> `closedReason='ws-closed-without-reconnect'`

`TerminalContainer`는 WebSocket status를 소비해 `setInputTransportState()`에 전달해야 한다.

## Session Generation

`sessionGeneration`은 `TerminalContainer`가 소유하는 context guard다. WebSocket protocol field가 아니며 client/server epoch로 해석하지 않는다.

규칙:
- `TerminalContainer` mount 시 `1`로 시작한다.
- `sessionId` 또는 workspace/tab identity가 바뀌면 증가한다.
- terminal instance dispose/recreate 시 증가한다.
- session exited, session missing, server error, auth expired를 관측하면 증가한다.
- reconnect가 `ws-reconnecting-short` TTL을 초과해 `ws-closed-without-reconnect`로 바뀌면 증가한다.
- queued input은 enqueue 당시 `sessionGeneration`을 저장하고, flush 시 현재 값과 다르면 `reason='context-changed'`로 reject한다.

Reconnect TTL owner:
- `TerminalContainer`가 `reconnectStartedAt`을 ref로 가진다.
- WebSocket status가 `reconnecting`으로 바뀌는 순간 값이 없으면 현재 시간을 저장한다.
- status가 `connected`로 돌아오면 값을 지운다.
- status가 `disconnected`가 되거나 `now - reconnectStartedAt > 3000 ms`이면 `closedReason='ws-closed-without-reconnect'`로 전환하고 queued Enter를 포함한 pending input을 flush하지 않는다.

## Input Message Shape

```ts
type ClientInputMessage = {
  type: 'input';
  sessionId: string;
  data: string;
  inputSeqStart?: number;
  inputSeqEnd?: number;
  metadata?: InputDebugMetadata;
};
```

`clientReadyEpoch`는 이번 계획 범위에서 제거한다. epoch 소유권과 reset 규칙이 별도로 필요하면 후속 계획에서 정의한다.

## Ingress Validation

`WsRouter`는 client input message를 type cast로 신뢰하지 않는다. queue/write 전에 아래를 검증한다.

- `sessionId`: non-empty string.
- `data`: string, UTF-8 byte length가 input queue budget 이하.
- `inputSeqStart`/`inputSeqEnd`:
  - 둘 다 없으면 legacy input으로 허용.
  - 하나만 있으면 reject reason `invalid-sequence`.
  - 둘 다 있으면 `Number.isSafeInteger`.
  - `inputSeqStart >= 1`.
  - `inputSeqEnd >= inputSeqStart`.
  - `inputSeqEnd - inputSeqStart + 1 <= MAX_INPUT_SEQUENCE_SPAN`.
  - `MAX_INPUT_SEQUENCE_SPAN = 1024`.
- `metadata`:
  - client metadata는 신뢰하지 않는다.
  - server는 byte length, Enter 여부, printable/control count처럼 가능한 필드는 `data`에서 재계산한다.
  - client-provided metadata는 아래 whitelist만 telemetry correlation 용도로 복사한다.
  - nested object, raw printable key/data, unknown field는 폐기한다.

Client metadata whitelist:
- `captureSeq`: safe integer.
- `compositionSeq`: safe integer.
- `clientObservedByteLength`: safe integer.
- `clientObservedCodePointCount`: safe integer.
- `clientObservedGraphemeCount`: safe integer.
- `clientObservedGraphemeApproximate`: boolean.
- `clientObservedHasHangul`: boolean.
- `clientObservedHasCjk`: boolean.
- `clientObservedHasEnter`: boolean.

String-valued client metadata는 복사하지 않는다. `inputClass`, `reason`, `barrierReason` 같은 문자열은 server가 자체 계산하거나 canonical enum에서 선택한다.

Validation 실패는 PTY write를 하지 않고 `input:rejected` 또는 replay telemetry로 남긴다. 원문은 기록하지 않는다.

## Canonical Reject Reasons

```ts
type InputRejectedReason =
  | 'timeout'
  | 'timeout-enter-safety'
  | 'queue-overflow'
  | 'context-changed'
  | 'session-missing'
  | 'session-closed'
  | 'server-error'
  | 'auth-expired'
  | 'transport-closed'
  | 'invalid-sequence'
  | 'invalid-payload'
  | 'mode-observe-only';
```

사용 규칙:
- `timeout`: Enter 없는 input이 TTL/timeout 정책상 실행되지 않은 경우.
- `timeout-enter-safety`: Enter 포함 input이 timeout 경로에서 실행 차단된 경우.
- `queue-overflow`: byte budget 또는 entry budget 초과.
- `context-changed`: visibility/session/workspace/generation mismatch.
- `session-missing`: target session 없음.
- `session-closed`: session exited 상태.
- `server-error`: session 또는 transport 관련 server error 상태.
- `auth-expired`: 인증 만료 또는 missing token.
- `transport-closed`: WebSocket close/disconnect 또는 전송 불가.
- `invalid-sequence`: sequence range validation 실패.
- `invalid-payload`: message shape 또는 data validation 실패.
- `mode-observe-only`: observe mode라서 실제 queue/flush 대신 would-event만 기록한 경우. 서버가 client에 보낼 필요는 없고 debug/telemetry reason으로만 사용 가능.

Telemetry event kind는 `input_rejected`처럼 계층별 event name을 사용할 수 있지만, `reason` 값은 위 문자열만 쓴다.

Closed state to reject reason mapping:
- `terminal-hidden` -> `context-changed`
- `terminal-disposed` -> `context-changed`
- `session-exited` -> `session-closed`
- `session-missing` -> `session-missing`
- `server-error` -> `server-error`
- `auth-expired` -> `auth-expired`
- `workspace-or-session-changed` -> `context-changed`
- `ws-closed-without-reconnect` -> `transport-closed`

SendResult to reject reason mapping:
- 이 mapping은 `TerminalContainer`가 reconnect status/TTL을 먼저 평가한 뒤, queue 불가로 판정된 경우에만 적용한다.
- `not-open`이라도 `reconnectState='reconnecting'`이고 TTL 안이면 reject하지 않고 `ws-reconnecting-short` barrier로 queue한다.
- `not-open` -> `transport-closed`
- `stale-socket` -> `transport-closed`
- `missing-token` -> `auth-expired`

## Canonical Would Events

Observe mode에서 동작 변경 없이 경로를 볼 때는 아래 event name을 사용한다.

- frontend capture: `terminal_input_would_queue`, `terminal_input_would_reject`
- transport: `transport_input_would_queue`, `transport_input_would_reject`
- server replay: `replay_input_would_queue`, `replay_input_would_reject`

Would event는 정상 release에서 오래 유지할 수 있지만, 원문 저장 금지와 UTF-8 byte length 기준은 동일하게 적용한다.
