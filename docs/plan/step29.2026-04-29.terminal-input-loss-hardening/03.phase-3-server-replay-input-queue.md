# Phase 3: Server Replay Input Queue

## Phase 3 목표

`WsRouter`의 replay pending 중 input hard drop을 제거한다. 서버는 output과 마찬가지로 input도 bounded queue에 저장하고, ACK 또는 timeout 정책에 따라 flush/reject를 명시해야 한다.

## 구현 범위

### Phase 3-1: WS protocol 타입 확장

대상:
- `server/src/types/ws-protocol.ts`
- `frontend/src/types/ws-protocol.ts`

`ClientWsMessage.input` 확장:

```ts
| {
    type: 'input';
    sessionId: string;
    data: string;
    inputSeqStart?: number;
    inputSeqEnd?: number;
    metadata?: InputDebugMetadata;
  }
```

초기 호환:
- sequence range 등은 optional로 둔다.
- 기존 client가 `{type:'input', sessionId, data}`만 보내도 동작한다.
- 단일 input은 `inputSeqStart === inputSeqEnd`로 보낸다.
- coalesced input은 포함된 첫/마지막 logical input의 sequence range로 보낸다.
- noncanonical `inputSeq` 단일 필드는 허용하지 않는다. 이 baseline에는 해당 field가 없으므로 legacy normalization을 추가하지 않는다.

`ServerWsMessage`에 optional explicit reject 추가:

```ts
| {
    type: 'input:rejected';
    sessionId: string;
    inputSeqStart?: number;
    inputSeqEnd?: number;
    reason: InputRejectedReason;
  }
```

프론트가 아직 UX를 표시하지 않더라도 debug capture와 테스트에서 관측 가능해야 한다.
`WebSocketContext`는 이 메시지를 unknown으로 버리면 안 된다. session handler 또는 terminal debug capture에 `server_input_rejected`로 전달한다.
Reason 문자열은 `00-4.protocol-contracts.md`의 `InputRejectedReason`을 따른다.

### Phase 3-1b: WS ingress validation

대상:
- `WsRouter` message parsing and `handleInput`

작업:
- `ClientWsMessage` type cast만 믿지 않는다.
- `sessionId`, `data`, `inputSeqStart`, `inputSeqEnd`, `metadata`를 `00-4.protocol-contracts.md`의 Ingress Validation 규칙대로 검증한다.
- invalid payload는 PTY에 쓰지 않고 `reason='invalid-payload'` 또는 `reason='invalid-sequence'`로 reject/telemetry 처리한다.
- server는 가능한 metadata를 input `data`에서 재계산하고, client metadata는 `00-4.protocol-contracts.md`의 Client metadata whitelist에 있는 필드만 correlation 용도로 보존한다.
- nested object, raw printable key/data, unknown metadata field는 폐기한다.

### Phase 3-2: ReplayPendingState 확장

대상:
- `server/src/types/ws-protocol.ts`
- `server/src/ws/WsRouter.ts`

추가:

```ts
queuedInputs: QueuedReplayInput[];
queuedInputBytes: number;
```

budget:
- `MAX_REPLAY_QUEUED_INPUT_BYTES = 64 * 1024`
- `MAX_REPLAY_QUEUED_INPUT_AGE_MS = 3000`
- byte 계산은 `Buffer.byteLength(data, 'utf8')`를 사용한다.

### Phase 3-3: `handleInput` queue 전환

현재:
- pending이면 `input_blocked` 기록 후 return.

변경:
- pending이면 `queuedInputs`에 append.
- telemetry:
  - `input_queued`
  - `input_queue_overflow`
  - `input_rejected`
- session missing이면 reject.
- legacy input에 sequence range가 없으면 server가 새 sequence를 만들지 않고 sequence 없이 queue한다.
- telemetry에는 input 원문을 넣지 않는다.

Mode 분기:
- `observe`: 기존 drop 동작을 유지할 수 있지만 `replay_input_would_queue` 또는 `replay_input_would_reject` telemetry에 UTF-8 byte/count/reason을 남긴다. 원문은 금지한다.
- `queue`: replay pending input을 `queuedInputs`에 append하고 ACK 후 flush한다.
- `strict`: queue 불가/overflow/timeout/context mismatch를 explicit reject한다.
- 어떤 mode에서도 timeout Enter safety는 지켜야 하며, Enter 포함 input을 timeout에서 flush하지 않는다.

### Phase 3-4: ACK flush 순서 구현

대상:
- `handleScreenSnapshotReady`
- `consumeReplayPendingForPair`

`consumeReplayPendingForPair()` 반환값에 `queuedInputs` 포함.

ACK 정상 경로:
1. queued output flush.
2. pending state 제거.
3. queued input 중 TTL 유효한 것만 `SessionManager.writeInput()`으로 순서대로 전달.
4. expired input은 `input_rejected` telemetry.
5. `session:ready` 전송.

중요:
- queued input write는 `session:ready`보다 먼저 수행해 ready 이후 새 input과 순서가 뒤섞이지 않게 한다.

### Phase 3-5: timeout/refresh/stale token 정책

Timeout:
- 현재 timeout은 pending clear 후 `session:ready`를 보낸다.
- 변경 후 timeout에서도 queued input을 정책대로 처리해야 한다.
- Enter 포함 input은 flush하지 않고 `reason='timeout-enter-safety'`.
- Enter 없는 TTL 유효 input은 flush하고 `input_flushed_timeout`.
- TTL 초과 input은 `reason='timeout'`으로 reject.

Refresh:
- `refreshReplaySnapshots()`는 token과 snapshot을 갱신하고 `queuedOutput=''`로 만든다.
- 변경 후 `queuedInputs`는 보존해야 한다.

Stale ACK:
- token mismatch ACK는 input queue를 flush하면 안 된다.
- active pending state가 남아 있으면 active token에서만 flush한다.

Disconnect:
- websocket close 시 queued input은 flush하지 않는다.
- `input_rejected` telemetry에 `reason='transport-closed'`를 남길 수 있으면 남긴다.

## 코드 중복 주의

- `appendQueuedOutput()` 패턴을 참고하되, input은 원문이 민감하므로 telemetry에 원문을 넣지 않는다.
- `sendTo()`는 open socket만 전송하므로 reject message 실패 가능성도 telemetry로 남긴다.

## 테스트 시나리오

### Server unit tests

기존 `testWsRouterBlocksInputWhileReplayPending`를 교체한다.

새 테스트:

1. `testWsRouterQueuesInputWhileReplayPendingAndFlushesAfterAck`
   - subscribe로 replay pending 생성.
   - pending 중 input `"가나다ABC\r"` 전송.
   - ACK 전 `writeInput` 없음.
   - ACK 후 `writeInput`이 정확히 한 번, 같은 data로 호출.
2. `testWsRouterPreservesInputQueueAcrossReplayRefresh`
   - pending 중 input queue.
   - `refreshReplaySnapshots()`.
   - 새 token ACK.
   - input flush 확인.
3. `testWsRouterDoesNotFlushInputForStaleAck`
   - old token ACK.
   - `writeInput` 없음.
4. `testWsRouterRejectsExpiredReplayInputOnTimeout`
   - fake timers 또는 내부 timeout helper로 TTL 초과 상황 검증.
5. `testWsRouterQueuedInputOverflowIsObservable`
   - budget 초과 시 일부 reject 또는 all reject 정책 확인.
6. `testWsRouterRejectsEnterInputOnReplayTimeout`
   - Enter 포함 input은 TTL 안이어도 timeout path에서 `writeInput`되지 않는다.
7. `testWsRouterRejectsInvalidInputPayload`
   - non-string `data`, empty `sessionId`, oversized UTF-8 payload가 `writeInput`되지 않고 `reason='invalid-payload'`로 reject된다.
8. `testWsRouterRejectsInvalidInputSequenceRange`
   - one-sided sequence, non-safe integer, `start < 1`, `end < start`, span > 1024가 `reason='invalid-sequence'`로 reject된다.
9. `testWsRouterSanitizesClientInputMetadata`
   - nested metadata, unknown fields, raw printable string fields, arbitrary string values are not copied to telemetry.
   - whitelisted numeric/boolean fields만 보존된다.
10. `testWsRouterEmitsInputRejectedForRealServerScenarios`
   - routable open WebSocket 상황의 invalid payload, invalid sequence, session missing, overflow, timeout-enter-safety에서 실제 server `input:rejected` message가 전송되는지 확인한다.
   - message routing이 불가능한 malformed/unaddressable frame만 telemetry-only를 허용한다.

### Frontend/Playwright tests

1. `input:rejected` routing
   - debug capture를 켠다.
   - 실제 server reject scenario로 `input:rejected`를 발생시킨다. test-only injection은 라우터 단위 보조 검증으로만 허용하고, 유일한 통과 기준으로 쓰지 않는다.
   - `server_input_rejected` debug event가 `inputSeqStart`, `inputSeqEnd`, `reason`과 함께 기록되는지 확인한다.
   - frontend unit runner를 추가하지 않는다면 Playwright debug hook으로 검증한다.

## Phase 3 완료 게이트

- [ ] replay pending 중 input hard drop이 없다.
- [ ] 기존 output replay ordering test가 깨지지 않는다.
- [ ] ACK/timeout/refresh/stale token 모두 input queue 정책이 명시되어 있다.
- [ ] timeout에서 Enter 포함 input은 flush되지 않는다.
- [ ] invalid payload/sequence가 PTY에 쓰이지 않는다.
- [ ] unsafe client metadata가 telemetry에 복사되지 않는다.
- [ ] real server reject scenario에서 `input:rejected`가 발생한다.
- [ ] `inputSeqStart`/`inputSeqEnd` protocol shape가 frontend/server에 모두 반영된다.
- [ ] `input:rejected`가 프론트 debug capture까지 전달된다. 이 검증은 Playwright 또는 명시적 frontend test runner에서 수행한다.
- [ ] frontend/server WS protocol type이 일치한다.
- [ ] `npm --prefix server run test`

## LLM 구현 주의

- ReplayPendingState를 수정하면 fake harness type도 함께 수정할 것.
- timeout 처리에서 timer cleanup과 pending map deletion 순서를 잘못 잡으면 duplicate flush가 생긴다.
- input queue flush는 반드시 idempotent하게 설계할 것.
