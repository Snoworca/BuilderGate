# Phase 5: IME Transaction and Repair Deferral

## Phase 5 목표

Phase 1~4 이후에도 IME boundary에서 손실 또는 조합 취소가 관측될 경우, IME 조합을 transaction으로 보호한다. 이 Phase는 가장 민감하므로 telemetry 없이 추측으로 진행하지 않는다.

## 구현 범위

### Phase 5-1: IME transaction state machine

대상:
- `TerminalView`
- 새 helper 후보: `frontend/src/utils/imeTransaction.ts`

상태:

```ts
type ImeState = 'idle' | 'composing' | 'committing' | 'settling';
```

전이:
- `compositionstart`: `idle -> composing`, `compositionSeq++`
- `compositionupdate`: `composing` 유지
- `xterm.onData` 관측 중 `isComposingRef=true` 또는 `imeState=composing`: `composing -> settling`
- `compositionend`: `composing -> committing` 단, 같은 `compositionSeq`에서 이미 xterm data가 관측되었으면 즉시 `settling`
- `compositionend` 후 final `xterm.onData` 관측: `committing -> settling`
- settle deadline 30~50 ms 후: `settling -> idle`
- timeout without any xterm data in the same `compositionSeq`: telemetry `ime_commit_without_xterm_data`

주의:
- 이 상태 머신은 xterm native path를 보호하기 위한 것이며 committed text를 직접 전송하기 위한 장치가 아니다.
- xterm은 Space/Enter/Arrow keydown으로 composition을 finalize하면서 `compositionend`보다 먼저 committed data를 emit할 수 있다. 이 경우를 실패로 기록하거나 fallback 후보로 삼으면 안 된다.
- `inputSeqStart`/`inputSeqEnd`는 여기서 만들지 않는다. composition과 capture 추적에는 `compositionSeq`/`captureSeq`만 사용한다.
- 모든 IME timer는 generation-safe해야 한다. `compositionend` clear timer, settle deadline timer, fallback observation timer, deferred repair timer는 예약 시점의 `compositionSeq`와 `sessionGeneration`을 캡처하고, 실행 시 둘 중 하나라도 현재 값과 다르면 no-op 처리한다.
- stale timer가 새 composition의 `imeState`를 `idle`로 되돌리거나 deferred repair를 실행하면 안 된다.

### Phase 5-2: composition 중 readiness close 지연

대상:
- `syncInputReadiness`
- repair replay 요청 지점
- snapshot restore/replace boundary

정책:
- `imeState !== idle`이면 transient repair/restore/snapshot/replay barrier에 한해 helper textarea disabled/readOnly/capture close를 지연한다.
- closed/error/session/auth/workspace state는 IME deferral 대상이 아니다. 즉시 capture close 또는 explicit reject를 수행하고 queued input을 flush하지 않는다.
- repair replay는 composition settle 이후로 defer한다.
- deferred repair 실행도 예약 시점의 `compositionSeq`와 `sessionGeneration`이 현재 값과 일치할 때만 허용한다.
- resize measurement는 가능하되 input capture를 끊는 side effect를 금지한다.

Telemetry:
- `ime_repair_deferred`
- `ime_capture_close_deferred`
- `ime_settled`

### Phase 5-3: beforeinput/input event tape 강화

목표:
- xterm `onData`가 실제로 안 오는지 확인한다.
- `insertCompositionText`를 중심으로 기록하고, 브라우저가 제공하는 경우 `insertFromComposition`, `deleteCompositionText`도 metadata로만 기록한다.

주의:
- 기본 동작은 여전히 xterm `onData`를 source of truth로 둔다.

### Phase 5-4: guarded fallback 설계

조건:
- `compositionend` 또는 `beforeinput(insertFromComposition)`이 있었고,
- 같은 `compositionSeq`에서 composition 중 또는 compositionend 후 xterm `onData`가 전혀 없고,
- helper textarea value 또는 event metadata에서 committed text length가 확인되며,
- 같은 compositionSeq에서 이미 전송된 sequence range가 없을 때만,
- debug/feature flag 아래에서 fallback 전송한다.

기본값:
- fallback injection은 처음에는 `observe`만 한다.
- 실제 전송은 수동 검증과 telemetry가 중복 없음으로 확인된 뒤 `queue`/`strict` mode에서만 켠다.
- Phase 5 완료 조건에 fallback 전송 활성화는 포함하지 않는다. 기본 완료 상태는 fallback observe-only다.

## 테스트 시나리오

### Synthetic E2E

- `compositionstart -> compositionupdate -> compositionend -> Space -> next composition` 순서에서 input sequence range가 단조 증가한다.
- Space/Enter/Arrow keydown으로 xterm data가 compositionend보다 먼저 emit되는 경우 `ime_commit_without_xterm_data`가 기록되지 않는다.
- composition 중 repair replay 요청이 발생하면 `ime_repair_deferred`가 기록되고, settle 후 replay가 실행된다.
- `compositionend` 후 final `xterm_data_emitted`가 있으면 fallback은 실행되지 않는다.
- composition 중 ArrowLeft/ArrowRight로 커서를 이동한 뒤 한글을 입력해도 fallback injection이 실행되지 않는다.
- composition 직후 Space는 xterm native path로만 처리되고, 직전 문자 앞에 공백이 삽입되는 회귀가 없어야 한다.
- 빠른 `compositionend -> compositionstart` 연속 상황에서 이전 settle timer가 새 composition을 `idle`로 만들지 않는다.
- 빠른 `compositionend -> compositionstart` 중 deferred repair timer가 새 composition 중 실행되지 않는다.
- xterm delayed textarea read를 반영하기 위해 synthetic composition test는 `compositionend.data`만 dispatch하지 않는다. helper textarea value를 실제로 갱신한 뒤 xterm이 `setTimeout(0)`으로 읽는 경로를 기다리고 `xterm_data_emitted` 또는 WS input frame을 검증한다.
- 가능한 경우 terminal echo 또는 debug capture의 server input까지 확인한다. composition event가 발생했다는 사실만으로 성공 처리하지 않는다.

### Manual

Windows Korean IME:
- "안녕하세요" 빠르게 입력 후 Space.
- "코딩을 합시다" 빠르게 입력 후 Enter.
- "빌더게이트테스트" 중간 Backspace 수정.
- "가나다라마바사" 입력 후 ArrowLeft 3회, 한글 추가 입력, Space.
- "테스트" 입력 직후 Space를 빠르게 10회 반복.
- 10회 반복 중 누락 0회.

## LLM 구현 주의

- IME fallback은 중복 입력 위험이 크다. telemetry 없이 전송 기능을 켜지 말 것.
- composition event의 `data`를 신뢰해 직접 전송하지 말 것.
- xterm 내부 CompositionHelper의 지연 판독 설계를 우회하지 말 것.
