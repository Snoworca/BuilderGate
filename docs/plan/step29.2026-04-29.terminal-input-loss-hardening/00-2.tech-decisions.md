# Technical Decisions

## TD-001: xterm native IME pipeline을 우선한다

결정: Space, Backspace, Enter의 기본 입력은 xterm native path에 맡긴다. custom key handler는 Ctrl+C selection copy, Ctrl+V 중복 방지, 관측 이벤트만 담당한다.

근거:
- xterm custom key handler는 기본 처리보다 먼저 실행된다.
- xterm CompositionHelper는 한국어 조합의 특수성을 고려해 `compositionend` 후 textarea 값을 지연 판독한다.
- BuilderGate의 과거 한글 IME 문제도 수동 Space 전송과 조합 완료 순서가 섞이면서 발생했다.

대안:
- Space/Backspace를 직접 `onInput()`으로 보내는 방식.
- 기각 이유: IME composition order를 우회하고 중복/누락 회귀 가능성이 높다.

## TD-002: helper textarea는 transient transport barrier 때문에 disabled 하지 않는다

결정: restore/replay/geometry/serverReady 같은 transient 상태에서는 helper textarea를 `disabled`로 만들지 않는다.

근거:
- xterm은 `disableStdin`을 `readOnly`로 반영한다.
- `disabled`는 focus와 IME composition session에 더 강한 영향을 준다.
- 빠른 한글 입력에서 조합 중 disabled가 되면 음절 단위 손실 가능성이 있다.

제약:
- hidden/disconnected/unmounted terminal은 여전히 capture를 막아야 한다.
- 이 결정은 "항상 입력 가능"이 아니라 "보이는 terminal의 transient transport barrier는 capture와 분리"라는 뜻이다.

## TD-003: 입력은 delivery-or-explicit-reject 계약을 따른다

결정: 앱이 관측한 입력의 silent drop은 금지한다. 입력은 아래 중 하나가 되어야 한다.

- 즉시 전송
- bounded queue 후 flush
- TTL/overflow/context mismatch로 explicit reject telemetry

근거:
- 사용자가 보고한 증상은 silent drop이기 때문에, 원인을 고치지 못해도 적어도 관측 가능해야 한다.
- Enter가 오래 지연되어 뒤늦게 실행되는 것도 위험하므로 무한 queue는 금지한다.

제약:
- 브라우저/IME/xterm이 이벤트 자체를 만들지 않은 물리 키 입력은 앱이 reject할 수 없다.
- 이 경우는 helper textarea event tape와 residual risk로 분류한다.

## TD-004: 서버 replay pending input은 queue한다

결정: `WsRouter.handleInput()`은 replay pending 중 input을 버리지 않고 `ReplayPendingState.queuedInputs`에 저장한다.

근거:
- 현재 서버는 output은 queue하지만 input은 버린다.
- snapshot replay는 화면 동기화 barrier이지 사용자 입력을 버리는 barrier가 아니다.

순서:
1. ACK 수신
2. queued output flush
3. pending 해제
4. queued input write
5. `session:ready` 전송

`session:ready`보다 queued input write가 먼저인 이유:
- ready 이후 새로 들어오는 입력과 queued input의 순서를 보장하기 위해서다.

Timeout 정책:
- Enter 포함 input은 timeout에서 flush하지 않고 `reason='timeout-enter-safety'`로 reject한다.
- Enter 없는 TTL 유효 input만 timeout flush를 허용한다.
- TTL 초과 input은 `reason='timeout'`으로 reject한다.

## TD-005: WebSocket send 실패는 호출자가 알아야 한다

결정: `WebSocketContext.send()`는 성공 여부를 반환하거나, 최소한 input 전송 경로가 socket closed/reconnecting 상태를 알 수 있어야 한다.

근거:
- 현재 `send()`는 socket이 열려 있지 않으면 no-op이다.
- reconnect 중 빠른 타이핑 손실을 막으려면 outbox로 되돌릴 수 있어야 한다.

## TD-006: telemetry는 원문을 저장하지 않는다

결정: printable/Hangul 원문은 debug capture에 저장하지 않는다.

기록 허용:
- UTF-8 byte length
- code point count
- grapheme count
- Hangul/CJK 여부
- inputSeqStart/inputSeqEnd
- control count
- safe control preview

근거:
- terminal input은 password/token/secret일 수 있다.
- 문제 분석에는 원문이 아니라 event sequence와 byte/count/phase가 필요하다.

## TD-007: IME fallback injection은 마지막 수단이다

결정: `beforeinput/input` 기반 committed text 복구 전송은 기본 구현에 넣지 않는다. Phase 5에서도 telemetry가 xterm `onData` 자체 손실을 증명한 경우에만 debug-guarded fallback으로 설계한다.

근거:
- xterm CompositionHelper는 이미 browser별 IME 차이를 흡수하려고 설계되어 있다.
- fallback이 잘못되면 중복 입력이 더 치명적이다.

금지:
- `compositionend.data`, `compositionupdate.data`, `beforeinput.data`를 곧바로 PTY로 전송하지 않는다.
- Space/Backspace/Enter를 IME 상태에서 직접 `onInput()`으로 주입하지 않는다.
- fallback injection은 기본값에서 전송하지 않는다.

## TD-008: Playwright만으로 최종 통과를 선언하지 않는다

결정: Playwright 자동화는 재현 가능한 boundary/race를 검증하고, Windows 한국어 IME 수동 검증을 최종 release gate로 둔다.

근거:
- Playwright synthetic composition은 실제 Windows IME candidate/composition pipeline과 동일하지 않다.
- 사용자가 경험한 현상은 실제 키보드 빠른 타이핑에서 발생한다.

## TD-009: `inputSeqStart`/`inputSeqEnd`는 TerminalContainer가 단독 소유한다

결정: `inputSeqStart`/`inputSeqEnd`는 `TerminalContainer`의 `InputSequencer`에서만 부여한다.

근거:
- `TerminalView` queue, transport outbox, server replay queue가 각각 sequence를 만들면 ordering과 dedup 판단이 깨진다.
- `TerminalView`는 xterm capture 계층이며, 같은 data가 queue 후 flush될 수 있으므로 transport sequence를 부여하기에 이르다.

정책:
- `TerminalView`: `captureSeq`, `compositionSeq`만 기록.
- `TerminalContainer`: `inputSeqStart`, `inputSeqEnd` 부여. 단일 chunk는 두 값이 같다.
- `WebSocketContext`, `WsRouter`, `SessionManager`: 전달받은 sequence 보존만 수행.
- legacy input에 sequence가 없으면 server가 새 sequence를 만들지 않는다.

## TD-010: queue 가능한 barrier를 명시적으로 제한한다

결정: `captureAllowed = term && visible` 같은 단순 공식은 금지하고, `captureState`와 `transportBarrierReason`을 별도로 둔다.

Queue 가능:
- restore/replay/initial geometry/repair-induced server-not-ready/short reconnect.

Queue 금지:
- hidden/disposed/exited/server error/auth expired/session changed/closed socket without reconnect.

근거:
- 오류 또는 다른 session으로 이동한 뒤 queued Enter가 늦게 실행되는 것이 입력 누락보다 더 위험하다.

## TD-011: IME fallback injection은 회귀 위험으로 취급한다

결정: fallback injection은 Phase 5에서도 기본값으로 전송하지 않는다.

근거:
- 과거 한글 IME 문제는 수동 Space 전송, 조합 완료 순서, 커서 이동, 중복 전송이 섞이면서 발생했다.
- `compositionend.data`나 `beforeinput.data`를 직접 전송하면 글자 누락 대신 중복/순서 뒤틀림이 생길 수 있다.

허용 조건:
- telemetry가 xterm `onData` 미발행을 같은 compositionSeq에서 입증한다.
- fallback candidate가 이미 전송된 sequence range와 중복되지 않음이 확인된다.
- Windows Korean IME 수동 검증에서 Space/Backspace/Arrow/cursor 이동 조합이 모두 통과한다.
- 별도 feature flag에서만 활성화한다.
