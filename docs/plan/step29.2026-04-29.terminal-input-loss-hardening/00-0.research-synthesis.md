# Research Synthesis: Terminal Input Loss Hardening

작성일: 2026-04-29

## 결론

빠른 타이핑, 특히 한글 IME 입력 중 몇 글자가 무시되는 문제는 단일 원인으로 보기 어렵다. 현재 코드베이스에는 실제로 입력을 폐기하는 경로가 여러 개 있다. 한글 IME 조합 결과가 한 번의 xterm data event 또는 짧은 burst로 전달될 가능성이 있으므로, 한 번의 폐기가 사용자에게는 "몇 글자 누락"처럼 보일 수 있다. 이 가능성은 계측으로 확인해야 하며, 확정 원인으로 단정하지 않는다.

따라서 이 계획의 핵심은 "IME를 더 잘 추측한다"가 아니다. 핵심은 아래 네 가지다.

1. 입력을 생성하는 DOM/xterm 경로를 가능한 한 끊지 않는다.
2. 전송이 준비되지 않은 입력은 짧고 bounded한 큐에 보존한다.
3. 서버 replay barrier 중 입력도 silent drop하지 않는다.
4. 모든 입력 프레임에 원문 비노출 메타데이터와 sequence를 붙여 손실 위치를 추적한다.
5. 앱이 관측하지 못한 물리 키 입력은 residual risk로 분리하고, xterm/browser event tape로 추적한다.

## 코드베이스 근거

### 1. 프론트 xterm 입력 게이트가 입력을 폐기한다

- `frontend/src/components/Terminal/TerminalView.tsx`
  - `syncInputReadiness()`는 `serverReady && geometryReady && !restorePending && visible`일 때만 `inputReadyRef=true`로 둔다.
  - 같은 함수가 `term.options.disableStdin = !nextReady`와 `helperTextarea.disabled = !nextReady`를 동시에 수행한다.
  - `term.onData()`는 `inputReadyRef=false`면 입력을 `return`으로 폐기한다.
  - printable/non-ASCII 입력은 현재 `shouldRecordTerminalInputDebug()`에서 preview가 없으면 자세한 drop 이벤트도 남지 않는다.

### 2. helper textarea `disabled` 직접 토글은 xterm 기본 설계보다 강하다

로컬 `frontend/node_modules/@xterm/xterm/src/browser/CoreBrowserTerminal.ts` 기준 xterm은 `disableStdin` 변경 시 helper textarea의 `readOnly`를 토글한다. BuilderGate는 여기에 추가로 `helperTextarea.disabled`를 직접 설정한다.

IME 조합 중 textarea가 disabled 되면 브라우저 조합 세션, 포커스, composition event propagation이 깨질 수 있다. 이것은 native PowerShell에는 없는 브라우저 계층 리스크다.

### 3. xterm 내부도 `disableStdin`이면 `onData`를 만들지 않는다

로컬 `frontend/node_modules/@xterm/xterm/src/common/services/CoreService.ts` 기준 `triggerDataEvent()`는 `disableStdin`이면 즉시 반환한다. 즉 BuilderGate가 `disableStdin=true`로 만드는 순간, xterm 내부에서 생성된 입력도 `TerminalView.onData`까지 오지 않을 수 있다.

### 4. custom key handler는 xterm IME 처리보다 먼저 실행된다

로컬 `CoreBrowserTerminal.ts`와 xterm API 문서 기준 `attachCustomKeyEventHandler()`는 xterm 기본 키 처리 전에 실행된다. 현재 코드는 IME guard를 갖고 있지만, Space/Backspace 관측 이벤트 이름이 `manual_input_forwarded`로 남아 있고 죽은 수동 전송 분기가 뒤에 남아 있어 회귀 위험이 있다.

### 5. `compositionend`는 즉시 최종 문자열을 보장하지 않는다

로컬 `CompositionHelper.ts` 기준 xterm은 `compositionend()`에서 `setTimeout(0)` 뒤 helper textarea 값을 읽어 최종 입력을 보낸다. 주석에는 Chromium에서 `CompositionEvent.data`가 신뢰하기 어렵고, 한국어의 종성/다음 모음 결합 같은 케이스 때문에 마지막 `compositionupdate.data`도 충분하지 않다고 되어 있다.

BuilderGate의 `isComposingRef`도 `setTimeout(0)`으로 해제하지만, "최종 `onData` 관측 전까지 보호"하는 transaction 상태 머신은 아니다.

### 6. TerminalContainer도 입력을 폐기한다

- `frontend/src/components/Terminal/TerminalContainer.tsx`
  - repair replay 요청 시 `sessionReadyRef.current=false`로 바꾸고 `terminalRef.current?.setServerReady(false)`를 호출한다.
  - `handleInput()`은 `sessionReadyRef=false`면 `ws_input_dropped_not_ready`를 기록하고 전송하지 않는다.

### 7. WebSocketContext.send는 닫힌 소켓에서 silent no-op이다

- `frontend/src/contexts/WebSocketContext.tsx`
  - `send()`는 `ws.readyState === OPEN`일 때만 `ws.send()`를 호출하고, 아닌 경우 결과를 반환하지 않는다.
  - reconnect 중 사용자가 타이핑하면 호출자가 성공/실패를 알 수 없다.

### 8. 서버 replay pending 중 input은 명시적으로 버린다

- `server/src/ws/WsRouter.ts`
  - `handleInput()`은 `replayPendingSessions`에 session이 있으면 `input_blocked`를 기록하고 `return`한다.
  - 반면 출력은 `queuedOutput`에 보존된다.
  - 즉 현재 서버는 replay barrier 동안 output은 lossless에 가깝지만 input은 lossy다.

### 9. 기존 테스트가 현재 lossy 동작을 사양화하고 있다

- `server/src/test-runner.ts`
  - `testWsRouterBlocksInputWhileReplayPending()`는 replay pending 중 입력이 `writeInput`되지 않는 것을 기대한다.
- `frontend/tests/e2e/terminal-korean-ime.spec.ts`
  - 현재 테스트는 IME guard와 `manual_input_forwarded` 이벤트 중심이다.
  - 실제 문자열이 PTY까지 손실 없이 전달되는지, replay/restore 중 composition commit이 보존되는지는 검증하지 않는다.

## 외부 근거

### xterm.js

- xterm API 문서는 `attachCustomKeyEventHandler`가 키가 처리되기 전에 실행되며, 반환값으로 xterm 기본 처리 여부를 결정한다고 설명한다.
  - https://xtermjs.org/docs/api/terminal/classes/terminal/
- `disableStdin`은 입력 비활성화 옵션이다.
  - https://xtermjs.org/docs/api/terminal/interfaces/iterminaloptions/
- xterm flow control 문서는 `write()`가 비동기 버퍼링되고, WebSocket 계층에서는 별도 ACK/flow-control 설계가 필요하다고 설명한다.
  - https://xtermjs.org/docs/guides/flowcontrol/

### 브라우저 IME/Input Events

- MDN은 `compositionend`를 IME composition session 완료 또는 취소 시 발생하는 이벤트로 설명한다.
  - https://developer.mozilla.org/en-US/docs/Web/API/Element/compositionend_event
- W3C Input Events draft는 IME composition 중 `beforeinput` 이벤트가 여러 단계로 발생하고, composition text와 committed text가 별도 단계로 이동한다고 설명한다.
  - https://www.w3.org/TR/2016/WD-input-events-20161018/

### node-pty / ConPTY

- node-pty README는 terminal emulator와 결합해 사용하는 PTY read/write API를 제공하고, flow control 옵션을 제공한다.
  - https://github.com/microsoft/node-pty
- Microsoft CreatePseudoConsole 문서는 호출자가 사용자 입력을 수집해 UTF-8 plain text와 VT sequence가 섞인 input stream으로 serialize해야 한다고 설명한다.
  - https://learn.microsoft.com/en-us/windows/console/createpseudoconsole
- Microsoft ConPTY 소개 글은 ConPTY가 Windows에서 PTY-like pipe 기반 터미널 호스팅 모델을 제공한다는 배경을 설명한다.
  - https://devblogs.microsoft.com/commandline/windows-command-line-introducing-the-windows-pseudo-console-conpty/

## 실패 가능성 평가

단일 패치로 "무조건 잘 된다"고 말할 수 없다. 입력 경로는 브라우저 IME, xterm, React lifecycle, WebSocket reconnect, snapshot replay, node-pty/ConPTY, PowerShell/PSReadLine redraw가 모두 연결되어 있다.

따라서 성공 가능성을 높이는 유일한 실용적 접근은 다음과 같다.

1. 먼저 손실 위치를 구분할 수 있는 관측성을 추가한다.
2. 이미 확인된 silent drop 경로를 queue 또는 explicit reject로 바꾼다.
3. IME 조합 중 DOM 입력 캡처 계층을 절대 끊지 않는다.
4. 모든 변경을 phase별로 검증하고, 각 phase 후 기존 terminal/replay/grid 회귀 테스트를 다시 실행한다.
5. Playwright가 실제 Windows 한글 IME를 완전히 대체하지 못하므로, 최종 수동 검증을 release gate로 둔다.

## 검증 후 보정된 해석

- `helperTextarea.disabled`는 한글 IME 누락의 확정 원인이 아니라, 현재 코드에서 제거해야 할 강한 위험 요인이다.
- `disableStdin=true`로 인한 손실은 `TerminalView.onData`까지 도달하지 않을 수 있으므로, `xterm_data_dropped_not_ready`만으로는 모든 capture loss를 볼 수 없다.
- `manual_input_forwarded`는 현재 실제 수동 전송이 아니라 delegated event naming 문제다. 죽은 Space/Backspace 직접 전송 분기는 제거해야 하지만, 현재 누락의 직접 증거로 쓰면 안 된다.
- `inputBytes` 또는 queue budget은 JS `string.length`가 아니라 UTF-8 byte length 기준이어야 한다.
- `No silent input loss`는 "앱이 관측한 입력"에 대한 계약이다. 브라우저/IME/xterm이 이벤트 자체를 만들지 않은 물리 키 입력은 별도 residual risk다.

## 핵심 구현 원칙

- **No silent input loss for observed input**: 앱이 관측한 입력은 전달되거나, bounded TTL 이후 명시적으로 reject/telemetry 처리되어야 한다. 조용히 사라지는 경로는 금지한다.
- **xterm native first**: Space/Backspace/Enter를 직접 수동 변환하지 않는다. xterm의 IME-aware pipeline을 우선한다.
- **DOM capture와 transport 분리**: helper textarea와 xterm `onData`는 가능한 한 열어두고, 서버 전송만 `transportReady`에 따라 지연한다.
- **Short bounded queue only**: 입력 큐는 무한 재생하면 안 된다. focus/visibility/session identity/TTL/byte budget을 모두 가져야 한다.
- **Enter safety**: Enter가 포함된 queued input은 오래 지연되면 위험하다. 정상 ACK/transient barrier 해소에서는 전달하되, timeout/context mismatch/TTL 초과 시 실행하지 않고 explicit reject한다.
- **Privacy-safe telemetry**: 한글/printable 원문은 debug 로그에 저장하지 않는다. UTF-8 byte length, codepoint count, grapheme count, Hangul/CJK flag, sequence만 기록한다.
- **Protocol symmetry**: frontend/server `ws-protocol.ts`는 항상 함께 수정한다.
