# Phase 2: Frontend Non-Lossy Capture Gate

## Phase 2 목표

프론트에서 생성된 xterm 입력을 transient 상태 때문에 즉시 폐기하지 않도록 한다. 핵심은 DOM/xterm capture와 서버 transport readiness를 분리하는 것이다.

## 구현 범위

### Phase 2-1: `inputReadyRef` 의미 분리

대상:
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- terminal imperative handle 타입

새 refs:

```ts
const captureStateRef = useRef<'open' | 'transient-blocked' | 'closed'>('closed');
const captureAllowedRef = useRef(false);
const transportReadyRef = useRef(false);
const transportBarrierReasonRef = useRef<string>('none');
```

계산 원칙:

```ts
captureState =
  computeCaptureState(term, {
    visible,
    disposed,
    sessionExited,
    serverError,
    authExpired,
    sessionChanged,
    restorePending,
    replayPending,
    geometryReady,
    reconnecting,
  });

captureAllowed = captureState !== 'closed';

transportReady =
  Boolean(
    term
    && serverReadyRef.current
    && geometryReadyRef.current
    && !restorePendingRef.current
    && isVisibleRef.current
  );
```

상태 소유권:
- `TerminalContainer`가 repair replay, session error, auth/session 상태, reconnect 상태를 알고 있으므로 `captureState`에 필요한 원천 상태를 `TerminalView`로 전달한다.
- 기존 `setServerReady(boolean)`만으로는 queue 가능한 `repair-server-not-ready`와 reject해야 하는 `server-error`를 구분할 수 없다.
- Phase 2에서 imperative handle을 아래처럼 확장한다.

```ts
type TerminalInputBarrierReason =
  | 'none'
  | 'restore-pending'
  | 'replay-pending'
  | 'initial-geometry-pending'
  | 'repair-server-not-ready'
  | 'ws-reconnecting-short';

type TerminalInputClosedReason =
  | 'terminal-hidden'
  | 'terminal-disposed'
  | 'session-exited'
  | 'session-missing'
  | 'server-error'
  | 'auth-expired'
  | 'workspace-or-session-changed'
  | 'ws-closed-without-reconnect'
  | 'none';

setInputTransportState({
  serverReady: boolean;
  barrierReason: TerminalInputBarrierReason;
  closedReason: TerminalInputClosedReason;
  reconnectState?: 'connected' | 'reconnecting' | 'disconnected';
  sessionGeneration: number;
});
```

호환:
- 기존 `setServerReady(boolean)`는 내부에서 `setInputTransportState()`를 호출하는 thin wrapper로 남길 수 있다.
- 새 구현 경로는 반드시 reason 있는 API를 사용한다.
- `sessionGeneration` 소유권과 증가 규칙은 `00-4.protocol-contracts.md`를 따른다.
- `reconnectState`는 기존 WebSocket status인 `connected | reconnecting | disconnected`와 동일한 문자열을 사용한다.

Mode 분기:
- `observe`: 기존 전달 동작은 유지하되, 아래에서 제거 대상으로 지정한 hard drop/disabled 지점은 telemetry로 반드시 드러나야 한다. `observe`에서도 IME capture surface를 닫는 `helperTextarea.disabled` 직접 토글 제거는 허용된다. 이 변경은 손실 방지성 호환 수정이다.
- `queue`: transient barrier에서 queue/flush를 활성화한다.
- `strict`: queue 불가 입력을 explicit reject하며, 기본값으로 사용하지 않는다.

Queue 가능한 `transient-blocked` reason:
- `restore-pending`
- `replay-pending`
- `initial-geometry-pending`
- `repair-server-not-ready`
- `ws-reconnecting-short`

즉시 reject해야 하는 `closed` reason:
- `terminal-hidden`
- `terminal-disposed`
- `session-exited`
- `session-missing`
- `server-error`
- `auth-expired`
- `workspace-or-session-changed`
- `ws-closed-without-reconnect`

초기 구현에서는 `inputReadyRef`를 완전히 제거하지 말고 호환 alias로 유지한다.

```ts
inputReadyRef.current = transportReady;
```

### Phase 2-2: helper textarea `disabled` 직접 토글 제거

대상:
- `syncInputReadiness`
- 초기 mount 시 helper textarea 초기화
- focus restore logic

변경:
- transient transport barrier에서 `helperTextarea.disabled = true`를 하지 않는다.
- xterm option `disableStdin`도 capture를 끊는 효과가 있으므로 `captureAllowed` 기준으로만 true/false를 정한다.
- transport만 막힌 상태에서는 `disableStdin=false`, helper textarea enabled 유지.

현재 코드에서 반드시 수정할 지점:
- `syncInputReadiness()`의 `term.options.disableStdin = !nextReady`는 `captureAllowed` 기준으로 바꾼다.
- `syncInputReadiness()`의 `helperTextarea.disabled = !nextReady`는 제거하거나 closed state에서만 허용한다.
- 초기 mount의 `helperTextarea.disabled = true`는 closed state 초기화로만 제한하고, mount 후 transient barrier에서는 disabled를 유지하지 않는다.

정책:
- visible/mounted terminal: capture on.
- hidden/unmounted/exited/error: capture off.
- restore/replay/server-not-ready: capture on, transport queue.
- reconnect가 진행 중인 짧은 transport gap은 capture on, transport queue.
- 영구 closed/error/exited/workspace-or-session-changed: capture closed, queue 금지, explicit reject.
- `handleError()` 경로는 `closedReason='server-error'`를 전달한다.
- repair replay 시작 경로는 `barrierReason='repair-server-not-ready'`를 전달한다.
- sessionId가 바뀌거나 terminal instance가 dispose되면 pending queue를 reject한다.
- WebSocket status mapping과 reconnect TTL은 `00-4.protocol-contracts.md`의 Reconnect TTL owner 규칙을 따른다.

### Phase 2-3: TerminalView pending input queue 추가

대상:
- `TerminalView`

Queue entry:

```ts
interface PendingTerminalInput {
  data: string;
  metadata: TerminalInputDebugMetadata;
  queuedAt: number;
  captureSeq: number;
  sessionGeneration: number;
  containsEnter: boolean;
  barrierReason: string;
}
```

정책:
- `term.onData()`에서 `transportReadyRef=false`지만 `captureAllowedRef=true`이면 queue한다.
- queue byte budget: 64 KiB.
- TTL: 1500 ms.
- overflow 시 `terminal_input_queue_overflow` 기록 후 오래된 입력부터 explicit reject.
- `transportReady`가 true로 바뀌면 FIFO flush.
- flush 시 `queued_input_flushed` 기록 후 `onInput(data)` 호출.
- context mismatch guard:
  - sessionId 변경, terminal disposed, visibility false, blur 후 TTL 초과 시 실행하지 않는다.
- enqueue 당시 `sessionGeneration`을 저장하고 flush 시 현재 값과 다르면 `reason='context-changed'`로 reject한다.
- Enter 포함 input은 TTL 초과, closed state 전환, context mismatch에서 flush하지 않는다.
- queue byte budget은 UTF-8 byte length 기준이다.
- 이 queue는 sequence range를 만들지 않는다. flush 후 `TerminalContainer`가 `inputSeqStart`/`inputSeqEnd`를 부여한다.

Mode 분기:
- `observe`: 현재라면 drop될 지점에 `terminal_input_would_queue` 또는 `terminal_input_would_reject` telemetry를 남긴다. 단, printable/Hangul 원문은 저장하지 않는다.
- `queue`: transient barrier에서 실제 queue/flush한다.
- `strict`: queue 불가 또는 TTL 초과 입력을 explicit reject한다.

현재 코드에서 반드시 수정할 지점:
- `term.onData()`의 `if (!inputReadyRef.current) return` 경로는 제거한다.
- `shouldRecordTerminalInputDebug(payload) === false`인 printable/Hangul 입력도 not-ready 상황에서 telemetry 없이 return하면 안 된다.

### Phase 2-4: imperative `sendInput()` 정책 수정

대상:
- `TerminalView` imperative handle `sendInput`

현재:
- `inputReadyRef=false`면 drop.

변경:
- captureAllowed + transient transport barrier이면 queue.
- disconnected/hidden/context mismatch면 explicit reject event.
- Enter 포함 입력은 timeout이나 closed state에서 절대 뒤늦게 실행하지 않는다.

### Phase 2-5: custom key handler 정리

대상:
- `attachCustomKeyEventHandler`

작업:
- 현재 `manual_input_forwarded` 이벤트는 실제 수동 전송이 아니라 `delegatedToXterm: true`인 관측 이벤트다.
- 이벤트명을 `key_delegated_to_xterm`로 변경하거나, 기존 이벤트를 유지해야 한다면 `manual_input_forwarded` test 의존을 줄이고 새 이벤트를 병행한다.
- line 901 이후 Space/Backspace 수동 전송 dead branch는 제거한다.
- IME 상태에서 Space/Backspace/Enter를 직접 `onInput()`으로 보내는 코드를 새로 만들지 않는다.
- Arrow key 이동 후 한글 입력, 조합 중 Space, 조합 중 Backspace는 xterm native path를 유지한다.

현재 코드에서 반드시 수정할 지점:
- `onInput(' ')`와 `onInput('\x7f')`를 직접 호출하는 Space/Backspace dead branch는 삭제한다.
- 기존 테스트가 `manual_input_forwarded`를 기대하더라도 새 테스트 모델로 바꾼다.

### Phase 2-6: composition timer generation guard

대상:
- helper textarea `compositionstart`/`compositionend` listener

작업:
- `compositionSeqRef`를 둔다.
- `compositionstart` 때 seq를 증가시키고 `isComposingRef.current=true`로 둔다.
- `compositionend`의 `setTimeout(0)` callback은 캡처한 seq가 현재 seq와 같을 때만 `isComposingRef.current=false`로 만든다.
- 새 composition이 이전 compositionend timer보다 먼저 시작된 경우 이전 timer가 새 composition 상태를 false로 덮어쓰면 안 된다.

주의:
- `Ctrl+C` selection copy와 `Ctrl+V` paste 중복 방지 로직은 유지한다.

## 테스트 시나리오

### Unit/Component 수준

현재 frontend에는 unit test runner가 없다. 아래 항목은 둘 중 하나로 구현해야 한다.

- pure TS helper와 명시적 frontend test script를 추가해 deterministic test로 검증.
- test runner 추가가 범위 밖이면 Playwright debug hook과 fake clock/test-only barrier toggle로 결정적으로 검증.

- `transportReady=false`, `captureAllowed=true` 상태에서 `onData("abc")`가 `onInput`으로 즉시 가지 않고 queue된다.
- 이후 `transportReady=true`가 되면 `"abc"`가 정확히 한 번 flush된다.
- enqueue 후 `sessionGeneration`이 바뀌면 queued input은 flush되지 않고 `reason='context-changed'`로 reject된다.
- TTL 초과 후 flush 시도는 실행하지 않고 reject event만 남긴다.
- helper textarea는 restore/replay/server-not-ready 중에도 `disabled=false`다.
- restore/replay/server-not-ready 중 `term.options.disableStdin === false`다.
- restore/replay/server-not-ready 중 helper textarea `readOnly === false`다.
- `server-error`, `session-missing`, `workspace-or-session-changed`에서는 queue하지 않고 reject한다.
- Enter 포함 queued input은 TTL 초과 후 flush되지 않는다.

### Playwright

- synthetic composition 중 `setServerReady(false)` 또는 repair replay를 유발해도 helper textarea가 disabled 되지 않는다.
- 기존 `TC-IME-02`는 `manual_input_forwarded` 3회 기대를 제거하고 `key_delegated_to_xterm` 또는 `xterm_data_emitted` 기준으로 변경한다.
- 빠른 `"abcdef"` 입력 중 강제 `serverReady=false -> true`가 끼어도 최종 입력 sequence가 보존된다.
- Arrow로 커서를 이동한 뒤 한글 composition을 시작하고 Space를 눌러도 수동 injection event가 발생하지 않는다.

## Phase 2 완료 게이트

- [ ] TerminalView에서 visible/focused transient barrier 중 helper textarea disabled가 발생하지 않는다.
- [ ] transient barrier 중 `term.options.disableStdin === false`가 검증된다.
- [ ] transient barrier 중 helper textarea `readOnly === false`가 검증된다.
- [ ] `xterm_data_dropped_not_ready`는 closed state 또는 expired reject 외에는 발생하지 않는다.
- [ ] Space/Backspace/Enter는 xterm native path에 위임된다.
- [ ] `TerminalView`는 input sequence range를 생성하지 않는다.
- [ ] Enter 포함 queued input은 timeout/context mismatch에서 실행되지 않는다.
- [ ] paste regression이 통과한다.
- [ ] mobile scroll test에서 helper textarea availability 기대가 깨지지 않는다.

## LLM 구현 주의

- 전체 TerminalView를 대규모로 재작성하지 말 것.
- 기존 snapshot restore, buffered output, focus restore 로직을 건드릴 때는 변경 이유를 phase 문서와 commit message에 명시할 것.
- helper textarea disabled 제거가 접근성/모바일 테스트를 깨뜨릴 수 있으므로 관련 테스트를 반드시 실행할 것.
