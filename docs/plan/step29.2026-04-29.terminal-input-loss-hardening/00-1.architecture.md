# Architecture: Non-Lossy Terminal Input Pipeline

## 목표

빠른 타이핑과 한글 IME 조합 입력에서 "몇 글자가 무시됨"을 구조적으로 줄인다. 구현 목표는 "모든 브라우저/IME/ConPTY 조합에서 100% 보장"이 아니라, BuilderGate 내부에서 발생하는 silent drop 경로를 제거하고, 남는 외부 레이어 문제를 진단 가능한 상태로 만드는 것이다.

## 현재 입력 경로

```text
Windows Korean IME / Keyboard
  -> browser helper textarea
  -> xterm CompositionHelper / CoreService.triggerDataEvent
  -> TerminalView.term.onData
  -> TerminalContainer.handleInput
  -> WebSocketContext.send
  -> WsRouter.handleInput
  -> SessionManager.writeInput
  -> node-pty / ConPTY
  -> PowerShell / PSReadLine
```

## 현재 손실 지점

```text
[A] xterm CoreService
    disableStdin=true -> triggerDataEvent returns without onData

[B] TerminalView
    inputReadyRef=false -> onData return
    helperTextarea.disabled=true -> browser/IME capture itself may break

[C] TerminalContainer
    sessionReadyRef=false -> input not sent

[D] WebSocketContext
    socket not OPEN -> send silently does nothing

[E] WsRouter
    replayPending -> input_blocked and return
```

## 목표 아키텍처

```text
Keyboard / IME
  -> xterm native composition path
  -> TerminalView capture gate
       - captureState: open | transient-blocked | closed
       - captureAllowed: captureState != closed
       - does NOT close helper textarea for transient transport barriers
  -> InputSequencer
       - owned by TerminalContainer only
       - assigns inputSeqStart/inputSeqEnd
       - builds privacy-safe metadata
       - coalesces safe printable runs
  -> TransportOutbox
       - sends immediately if transportReady
       - queues briefly if replay/restore/reconnect transient
       - rejects explicitly if stale/overflow/session changed
  -> WebSocket input frame
       - type: input
       - sessionId
       - data
       - inputSeqStart
       - inputSeqEnd
       - metadata
  -> WsRouter replay input queue
       - queues while replayPending
       - flushes in order after ACK
       - emits input_queued / input_flushed / input_rejected telemetry
  -> SessionManager.writeInput
       - logs privacy-safe input event with inputSeqStart/inputSeqEnd
       - writes to PTY
```

## 새로운 상태 모델

### TerminalView

`inputReadyRef`를 단일 의미로 쓰지 않는다.

- `captureStateRef`
  - `open | transient-blocked | closed`.
  - xterm/helper textarea가 사용자 입력을 받아도 되는지와, queue 가능한 barrier인지 분리한다.
  - `open`: visible/mounted이고 즉시 전송 가능한 상태.
  - `transient-blocked`: visible/mounted이며 restore/replay/initial geometry/short reconnect처럼 곧 회복 가능한 상태.
  - `closed`: hidden/unmounted/disposed/session exited/server error/auth expired/session changed처럼 입력을 queue하면 안 되는 상태.
- `captureAllowedRef`
  - `captureStateRef.current !== 'closed'`일 때만 true.
  - `term && visible` 같은 단순 조건만으로 계산하지 않는다.
- `transportReadyRef`
  - captured input을 즉시 서버로 보낼 수 있는지.
  - server ready, geometry ready, restore pending false, ws/subscription ready가 모두 맞아야 true.
- `transportBarrierReasonRef`
  - `restore-pending | replay-pending | initial-geometry-pending | repair-server-not-ready | ws-reconnecting-short | none`.
  - queue/reject telemetry의 reason source다.
- `inputTransportStateRef`
  - `TerminalContainer`가 전달하는 상태를 저장한다.
  - `serverReady`, `barrierReason`, `closedReason`, `sessionGeneration`, `reconnectState`를 포함한다.
  - `reconnectState`는 기존 WebSocket status인 `connected | reconnecting | disconnected`를 그대로 사용한다.
  - `setServerReady(boolean)`만으로는 repair replay와 server error를 구분할 수 없으므로 Phase 2에서 imperative handle을 확장한다.
- `sessionGenerationRef`
  - `TerminalContainer`가 소유하는 local context guard다.
  - WebSocket protocol epoch가 아니며, 소유권/증가 규칙은 `00-4.protocol-contracts.md`를 따른다.
- `captureSeqRef`
  - `TerminalView` event tape와 local queue 진단용 sequence다.
  - `inputSeq`가 아니며 서버로 전송하지 않는다.
- `imeStateRef`
  - `idle | composing | committing | settling`
  - Phase 5에서 도입한다. Phase 2에서는 기존 `isComposingRef`를 유지하되 textarea disabled 제거와 queue로 보호한다.

### TerminalContainer

- `sessionReadyRef`
  - "서버 전송 가능" 신호로만 사용한다.
  - false일 때 입력을 폐기하지 않고 outbox로 넘긴다.
- `inputOutboxRef`
  - session별 short-lived queue.
  - TTL, byte budget, focus/session identity guard를 갖는다.
- `inputSeqRef`
  - client session 단위 monotonically increasing input sequence.
  - `inputSeqStart`/`inputSeqEnd`의 유일한 소유자다. `TerminalView`, `WebSocketContext`, server는 새 sequence를 만들지 않는다.

`TerminalContainer`는 queue 가능 여부를 판단하는 소유자이기도 하다.

- repair replay 시작: `barrierReason='repair-server-not-ready'`, queue 가능.
- session error: `closedReason='server-error'`, queue 금지.
- session exited/missing: `closedReason='session-exited' | 'session-missing'`, queue 금지.
- auth expired: `closedReason='auth-expired'`, queue 금지.
- reconnect short gap: Phase 4에서 `barrierReason='ws-reconnecting-short'`, queue 가능.

### WebSocketContext

- `send()`를 `boolean` 반환 또는 `sendResult` 반환으로 확장한다.
- input message 전용으로 `sendInputOrQueue`를 만들 수도 있으나, 중복을 줄이려면 `send()` 성공 여부만 먼저 노출하고 queue 정책은 `TerminalContainer`가 소유한다.

### WsRouter

`ReplayPendingState`를 확장한다.

```ts
interface ReplayPendingState {
  queuedOutput: string;
  queuedInputs: QueuedReplayInput[];
  queuedInputBytes: number;
  timer: NodeJS.Timeout;
  replayToken: string;
  snapshotSeq: number;
}

interface QueuedReplayInput {
  sessionId: string;
  data: string;
  inputSeqStart?: number;
  inputSeqEnd?: number;
  clientId: string;
  queuedAt: number;
  byteLength: number;
  containsEnter: boolean;
  metadata?: InputDebugMetadata;
}
```

## Queue 정책

### 프론트 outbox

- Byte budget: session당 64 KiB 시작.
- Max age: 기본 1500 ms, reconnect 중 최대 3000 ms.
- Enter 포함 input:
  - replay/restore/reconnect transient이면 queue 가능.
  - TTL 초과, reconnect timeout, context mismatch에서는 실행하지 않고 explicit reject telemetry.
- Blur/session change/hidden:
  - printable queue는 discard가 아니라 `input_rejected` event와 `reason='context-changed'`로 기록한다.
  - 사용자가 보던 terminal이 아니면 뒤늦게 명령이 실행되지 않아야 한다.

### 서버 replay input queue

- Byte budget: client-session pending state당 64 KiB 시작.
- Max age: 3000 ms.
- ACK 정상 수신 시:
  1. queued output을 client에 먼저 보낸다.
  2. replay pending을 해제한다.
  3. queued input을 FIFO 순서로 `SessionManager.writeInput()`에 전달한다. sequence range가 있는 경우 FIFO와 sequence range가 충돌하면 reject telemetry를 남기고 순서를 임의 재정렬하지 않는다.
  4. `session:ready`를 보낸다.
- ACK timeout 시:
  - Enter 포함 input은 flush하지 않고 `reason='timeout-enter-safety'`로 명시한다.
  - Enter 없는 TTL 유효 input만 정책적으로 flush할 수 있으며 `input_flushed_timeout` telemetry를 남긴다.
  - TTL 초과 input은 `reason='timeout'`, overflow input은 `reason='queue-overflow'`로 명시한다.
- replay refresh 시:
  - `queuedOutput`은 기존 정책대로 새 snapshot 기준으로 재시작 가능.
  - `queuedInputs`는 절대 reset하지 않는다.
  - token mismatch ACK는 input queue를 flush하지 않는다.

## 입력 metadata

원문 저장 없이 아래만 기록한다.

```ts
interface InputDebugMetadata {
  inputSeqStart?: number;
  inputSeqEnd?: number;
  byteLength: number;
  codePointCount: number;
  graphemeCount: number;
  hasHangul: boolean;
  hasCjk: boolean;
  hasEnter: boolean;
  controlCount: number;
  printableCount: number;
  inputClass: 'safe-control' | 'printable' | 'mixed-printable-control';
  safePreview: boolean;
  gateCaptureAllowed?: boolean;
  gateTransportReady?: boolean;
  gateServerReady?: boolean;
  gateGeometryReady?: boolean;
  gateRestorePending?: boolean;
  gateVisible?: boolean;
  gateComposing?: boolean;
  transportBarrierReason?: string;
}
```

`Intl.Segmenter`가 있으면 grapheme count에 사용한다. 없으면 code point count로 fallback하되 `graphemeApproximate=true`를 기록한다.

모든 telemetry/debug detail은 primitive flat field만 사용한다. `gateState` 같은 nested object는 기존 `TerminalDebugValue`/`ReplayTelemetryValue` 타입과 맞지 않으므로 금지한다.

UTF-8 byte budget과 byte telemetry는 browser에서는 `new TextEncoder().encode(data).byteLength`, Node에서는 `Buffer.byteLength(data, 'utf8')`로 계산한다. `data.length`를 byte로 기록하지 않는다.

## Explicit reject 수신 경로

서버가 `input:rejected`를 보내면 프론트는 이를 무시하지 않는다.

```text
WsRouter
  -> ServerWsMessage(input:rejected)
  -> WebSocketContext message router
  -> session handler or terminal debug capture
  -> recordTerminalDebugEvent('server_input_rejected', ...)
```

이번 범위에서 새 UI toast나 화면 문구는 추가하지 않는다. UI 변경은 AGENTS.md 규칙상 별도 사용자 승인 대상이다.

## Phase 간 안전장치

1. Phase 1은 코드 동작을 바꾸지 않고 telemetry와 재현 harness를 강화한다.
2. Phase 2는 프론트 capture 손실만 줄이고 서버 protocol은 최소 변경한다.
3. Phase 3은 서버 replay 중 hard drop을 queue로 바꾼다.
4. Phase 4는 WebSocket reconnect/no-op과 coalescing을 다룬다.
5. Phase 5는 IME transaction coordinator와 repair deferral을 추가한다. 가장 민감하므로 앞 phase의 telemetry로 필요성이 확인된 뒤 진행한다.
6. Phase 6은 전체 검증과 수동 Windows Korean IME release gate다.

## Feature flag / rollout

처음부터 모든 동작을 기본값으로 켜지 않는다.

- `inputReliabilityMode = "observe" | "queue" | "strict"`
  - `observe`: 기존 동작 유지 + captureSeq/metadata/drop telemetry.
  - `queue`: transient barriers에서 queue/flush. 자동/수동 검증에서 명시적으로 활성화한다.
  - `strict`: silent drop 금지. 전달 불가 시 explicit reject.

기본값:
- Phase 1에서 mode plumbing과 test override를 먼저 구현하고 기본값은 `observe`.
- Phase 2~5 테스트는 `queue` mode를 명시적으로 켠다.
- Phase 6 수동 검증 통과 후 기본값을 `queue`로 승격할 수 있다.
- `strict`는 별도 릴리스 판단 전까지 기본값으로 두지 않는다.

구현 위치는 UI 설정이 아니라 서버 runtime config와 localhost debug override로 시작한다. 사용자에게 노출하는 설정은 최종 안정화 후 별도 SRS가 필요하다.
