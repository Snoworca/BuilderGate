# Terminal Input Loss Hardening Plan

작성일: 2026-04-29
대상: BuilderGate terminal input pipeline
언어/스택: TypeScript, React 19, xterm.js 6, Express, ws, node-pty, Windows ConPTY

## 문제 정의

사용자가 네이티브 PowerShell에서는 발생하지 않는 빠른 타이핑 누락, 특히 한글 IME 입력 누락을 보고했다. 현재 코드에는 실제 입력 손실 가능성이 있는 hard drop 경로가 존재한다.

이 계획은 단일 패치로 성공을 가정하지 않는다. 손실 위치를 먼저 계측하고, 확인된 silent drop 경로를 단계적으로 queue/flush 또는 explicit reject로 전환한다.

## Non-Goals

- 새 터미널 렌더러 도입 금지.
- node-pty fork 또는 native ConPTY bridge는 이번 구현 범위에서 제외. Phase 6 이후에도 문제가 남을 때 별도 연구로 분리한다.
- UI 설정 화면에 새 옵션을 노출하지 않는다.
- terminal 입력 원문을 debug log에 저장하지 않는다.
- 한글 IME 문제를 해결한다는 이유로 기존 paste, Ctrl+C/Ctrl+V, grid repair, snapshot replay 계약을 깨지 않는다.

## 완료 조건

1. 앱이 관측한 input의 silent drop 경로가 없어지거나 explicit reject telemetry로 바뀐다.
2. replay pending 중 입력이 정상 ACK 후 순서대로 PTY에 전달된다.
3. helper textarea가 transient replay/restore/geometry barrier 때문에 disabled 되지 않는다.
4. 빠른 한글 입력 수동 검증에서 누락이 재현되지 않는다.
5. 기존 terminal/replay/grid/paste/keyboard 회귀 테스트가 통과한다.
6. 실패 시 원인 레이어를 client capture, transport, server replay, pty write, output redraw 중 하나로 분류할 수 있다.
7. `inputSeqStart`/`inputSeqEnd`는 `TerminalContainer`의 `InputSequencer`에서만 부여된다.
8. timeout/context mismatch에서는 Enter 포함 queued input이 뒤늦게 실행되지 않는다.

## 검증으로 보강된 필수 정책

- `TerminalView`는 input sequence range를 만들지 않고 `captureSeq`/`compositionSeq`만 기록한다.
- Queue 가능한 상태와 reject해야 하는 상태를 명시적으로 분리한다.
- `input:rejected` 서버 메시지는 프론트 debug capture까지 라우팅한다.
- Phase 2 전에 `inputReliabilityMode` plumbing과 테스트 override가 존재해야 한다.
- reason 문자열과 WebSocket input validation은 [Protocol Contracts](./00-4.protocol-contracts.md)를 따른다.
- Telemetry detail은 primitive flat field만 사용한다.
- Byte budget은 UTF-8 byte length 기준이다.
- IME fallback injection은 기본값에서 전송하지 않는다.
- 실제 Windows Korean IME 수동 검증 전에는 queue mode를 기본값으로 승격하지 않는다.
