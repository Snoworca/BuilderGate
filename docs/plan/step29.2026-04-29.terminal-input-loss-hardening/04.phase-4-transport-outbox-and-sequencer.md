# Phase 4: Transport Outbox and Sequencer

## Phase 4 목표

WebSocket reconnect 또는 session ready false 상태에서 input이 silent no-op으로 사라지지 않게 한다. 또한 너무 잘게 쪼개진 printable 입력을 짧게 coalesce하여 WS/PTY 호출 압력을 줄인다.

## 구현 범위

### Phase 4-1: WebSocketContext send 결과 반환

대상:
- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/contexts/WebSocketContext` type/interface
- 호출부 전체

변경:

```ts
type SendResult =
  | { ok: true }
  | { ok: false; reason: 'not-open' | 'missing-token' | 'stale-socket' };
```

기존 호출부는 결과를 무시해도 된다. input 호출부는 결과를 사용한다.
`SendResult.reason`은 외부 reject reason이 아니다. input reject/debug telemetry로 변환할 때는 `00-4.protocol-contracts.md`의 mapping을 사용한다.
단, `screen-snapshot:ready`, `resize`, `repair-replay`도 closed socket에서 no-op이면 replay barrier가 길어질 수 있다. Phase 4는 최소한 input path를 queue/reject하고, snapshot ACK 실패는 telemetry로 남긴다.

### Phase 4-2: TerminalContainer input outbox

대상:
- `frontend/src/components/Terminal/TerminalContainer.tsx`

역할:
- `sessionReadyRef=false` 또는 `send()` 실패 시 queue.
- `handleSessionReady`, WS connected/resubscribe 이후 flush.

Outbox entry는 반드시 enqueue 당시 `sessionGeneration`을 저장한다. Flush 시 현재 `sessionGeneration`과 다르면 `reason='context-changed'`로 reject하고 전송하지 않는다.

Queue guard:
- sessionId 일치.
- sessionGeneration 일치.
- terminal still visible 또는 active workspace/tab 일치.
- TTL 1500 ms, reconnect 시 3000 ms.
- max bytes 64 KiB.
- Enter 포함 input은 TTL 초과 시 reject.
- server-error/session-exited/session-missing/workspace-or-session-changed/auth-expired에서는 queue하지 않고 reject.
- byte budget은 UTF-8 byte length 기준.

Telemetry:
- `transport_input_queued`
- `transport_input_flushed`
- `transport_input_rejected`
- `transport_input_queue_overflow`

Mode 분기:
- `observe`: closed socket/session-not-ready에서 `transport_input_would_queue` 또는 `transport_input_would_reject`만 기록한다. 기존 동작과 비교 관측 목적이다.
- `queue`: `sessionReadyRef=false` 또는 `send()` 실패 시 outbox queue를 활성화한다.
- `strict`: queue 불가 또는 TTL/overflow/context mismatch는 explicit reject한다.

SendResult precedence:
- `TerminalContainer`는 `send()` 실패를 즉시 `transport-closed`로 매핑하지 않는다.
- 먼저 `00-4.protocol-contracts.md`의 reconnect status/TTL classification을 적용한다.
- status가 `reconnecting`이고 TTL 안이면 `barrierReason='ws-reconnecting-short'`로 queue 가능하다.
- status가 `disconnected`, TTL 초과, `missing-token`, `stale-socket`이면 canonical mapping에 따라 reject한다.

### Phase 4-3: InputSequencer

새 helper 후보:
- `frontend/src/utils/terminalInputSequencer.ts`

책임:
- `nextInputSeqRange(logicalChunkCount)`
- metadata 생성 호출
- microtask/short timer coalescing
- boundary 판단

소유권:
- `InputSequencer`는 `TerminalContainer`가 소유한다.
- `TerminalView`는 `captureSeq`만 만들고 input sequence range를 만들지 않는다.
- server는 legacy input에 대해 새 sequence range를 만들지 않는다.
- 단일 chunk는 `inputSeqStart === inputSeqEnd`.
- coalesced chunk는 `inputSeqStart`와 `inputSeqEnd`를 모두 WS message에 포함한다.

Coalescing 정책:
- printable-only chunks는 3~8 ms 안에서 합친다.
- Hangul/CJK printable도 합치되 원문 logging 금지.
- Enter, Escape, Ctrl+C, Ctrl+D, Backspace, arrow escape sequence는 boundary로 즉시 flush한다.
- paste는 기존 paste regression 때문에 별도 boundary로 취급한다. paste content는 coalescing하지 않는다.

### Phase 4-4: Frontend/server protocol에 inputSeq range 전송

대상:
- `TerminalContainer.handleInput`
- `frontend/src/types/ws-protocol.ts`
- `server/src/types/ws-protocol.ts`
- `WsRouter.handleInput`
- `SessionManager.writeInput`

작업:
- input WS message에 `inputSeqStart`, `inputSeqEnd`와 metadata를 포함한다.
- 서버 debug capture input event에 sequence range를 기록한다.
- replay queue도 sequence range를 보존한다.

### Phase 4-5: Outbox flush ordering

규칙:
- TerminalView queue flush -> TerminalContainer sequencer -> transport outbox -> WS send.
- 같은 session의 input sequence range 순서는 절대 바뀌면 안 된다.
- coalescing은 inputSeq range를 보존한다.

예:

```ts
{
  inputSeqStart: 10,
  inputSeqEnd: 14,
  data: "abcde"
}
```

서버는 range를 그대로 기록한다.

## 테스트 시나리오

### Unit

현재 frontend에는 unit test runner가 없다. 이 Phase에서 `terminalInputSequencer`를 pure helper로 추가한다면, 명시적 frontend test script를 추가하거나 Playwright debug hook으로 아래 경계를 결정적으로 검증한다.

- printable rapid input chunks가 1 frame으로 coalesce된다.
- Enter boundary는 coalesced printable 앞/뒤 순서를 보존한다.
- WebSocket not-open이면서 `ws-reconnecting-short` TTL 안이면 input이 outbox에 들어가고, open/ready 후 flush된다.
- TTL 초과 Enter-containing queue는 flush되지 않고 reject된다.
- workspace-or-session-changed/server-error/session-missing 상태에서는 queue되지 않고 reject된다.

### E2E

- WS reconnect를 인위적으로 유도한 뒤 빠른 ASCII/Hangul-like `insertText`가 사라지지 않는다.
- `terminal-paste.spec.ts`가 여전히 붙여넣기 중복 없이 통과한다.
- `terminal-keyboard-regression.spec.ts` auto-repeat Space 순서가 보존된다.

## Phase 4 완료 게이트

- [ ] `WebSocketContext.send()` silent no-op input 손실이 없다.
- [ ] inputSeq range가 browser debug, WS message, server debug에서 연결된다.
- [ ] microbatcher가 control key 순서를 바꾸지 않는다.
- [ ] inputSeq range는 `TerminalContainer.InputSequencer`에서만 생성된다.
- [ ] closed/error state에서는 outbox가 flush되지 않는다.
- [ ] paste/keyboard/grid regression 통과.

## LLM 구현 주의

- `send()` 반환 타입 변경으로 workspace/grid broadcast 호출부를 불필요하게 고치지 말 것.
- coalescing은 작게 시작한다. 복잡한 scheduler나 external dependency 도입 금지.
- 입력 큐는 React state가 아니라 ref 기반이어야 한다. 렌더링 루프에 연결하지 말 것.
