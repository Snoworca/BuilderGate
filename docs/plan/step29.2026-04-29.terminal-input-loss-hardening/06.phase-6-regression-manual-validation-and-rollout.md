# Phase 6: Regression, Manual Validation, Rollout

## Phase 6 목표

자동 테스트와 실제 Windows Korean IME 수동 검증을 모두 통과한 뒤에만 기본 동작을 `queue`로 승격할 수 있다. `strict`는 이번 계획의 기본값 승격 대상이 아니며, 별도 릴리스 판단과 추가 SRS가 필요하다.

## 자동 검증

### Server

- `npm --prefix server run test`
- 중점:
  - WsRouter replay input queue
  - ACK/timeout/refresh/stale token
  - SessionManager input debug metadata
  - resize replay refresh 회귀

### Frontend build

- `npm --prefix frontend run build`
- deterministic queue/sequencer tests
  - frontend unit test runner를 추가했다면 해당 script 실행.
  - test runner를 추가하지 않았다면 Playwright debug hook 기반 FIFO/TTL/overflow/Enter-safety test를 실행.

### Playwright

대상은 AGENTS.md 규칙대로 `https://localhost:2002`.

권장 실행:

```powershell
npm --prefix frontend exec playwright test tests/e2e/terminal-korean-ime.spec.ts --project "Desktop Chrome"
npm --prefix frontend exec playwright test tests/e2e/terminal-keyboard-regression.spec.ts --project "Desktop Chrome"
npm --prefix frontend exec playwright test tests/e2e/terminal-paste.spec.ts --project "Desktop Chrome"
npm --prefix frontend exec playwright test tests/e2e/terminal-authority.spec.ts --project "Desktop Chrome"
npm --prefix frontend exec playwright test tests/e2e/grid-equal-mode.spec.ts --project "Desktop Chrome"
```

## 수동 검증 매트릭스

### 환경

- OS: Windows 11
- Browser: Chrome 또는 Edge 최신 stable
- URL: `https://localhost:2002`
- Shell: PowerShell profile
- Backend: ConPTY default
- Password: local test password `1234`

### 입력 문장

아래를 각 10회 반복한다.

1. `안녕하세요`
2. `코딩을 합시다`
3. `빌더게이트 입력 테스트`
4. `가나다라마바사아자차카타파하`
5. `영문ABC한글DEF혼합`

### 동작 조합

- 빠르게 입력 후 Space.
- 빠르게 입력 후 Enter.
- 입력 중 Backspace 2회.
- 입력 중 ArrowLeft/ArrowRight로 커서 이동 후 한글 추가 입력.
- 한글 입력 직후 Space를 매우 빠르게 누르기.
- composition 확정이 Space/Enter/Arrow keydown에서 먼저 일어나는 케이스 반복.
- 입력 중 tab/grid 전환 후 계속 입력.
- 입력 중 창 resize.
- grid mode에서 idle repair가 일어날 수 있는 상태로 전환 후 계속 입력.

### 통과 기준

- 누락 글자 0회.
- 중복 글자 0회.
- composition 중 커서/조합창이 사라지는 현상 0회.
- Space가 직전 문자 앞에 삽입되는 현상 0회.
- Arrow 이동 후 한글이 깨지는 현상 0회.
- Space/Enter/Arrow 확정 경로가 fallback 후보로 기록되는 현상 0회.
- debug capture에서 `*_dropped_not_ready` silent drop 없음.
- reject event가 있으면 이유가 timeout/overflow/context mismatch로 명확해야 하며 정상 빠른 입력에서는 없어야 한다.

## Rollout

### 1단계: observe

- telemetry만 추가된 상태.
- 기존 동작과 비교하여 실제 손실 지점 확인.

### 2단계: queue

- transient barrier queue 활성.
- 수동 검증 통과 후 기본값 후보.
- Phase 6 완료 전에는 테스트/localhost override에서만 활성화한다.

### 3단계: strict

- silent drop 금지.
- delivery 불가 입력은 explicit reject.
- 추후 안정화 후 기본값 검토.

## Rollback

문제 발생 시 rollback 단위:

1. IME transaction/fallback 비활성.
2. microbatcher 비활성.
3. frontend outbox 비활성.
4. server replay input queue는 가능하면 유지. hard drop보다 안전하지만, 문제가 있으면 feature flag로 기존 behavior 복구 가능하게 둔다.
