# Tech Decisions

## ADR-001: 1차 수리 레이어는 replay/snapshot이 아니라 PowerShell-on-ConPTY producer 경계다

### 결정

문제의 1차 수리 레이어를 `PowerShell/PSReadLine on ConPTY`로 잡는다.

### 이유

- steady-state rapid repeat에서도 재현된다
- 이번 재현에서는 `replayEvents = 0`
- 서버 `raw_output`에 이미 redraw 조각이 보였다

### 기각한 대안

- snapshot/replay hardening만으로 해결
  - 증폭 요인은 줄일 수 있어도 producer 문제를 끊지는 못함

## ADR-002: Windows에서 PowerShell 세션은 최종적으로 winpty를 기본 backend로 사용한다

### 결정

Step18의 최종 목표는 Windows PowerShell 세션을 `winpty` 기본 backend로 운영하는 것이다.

단, Step18 1차 릴리스는 silent cutover를 하지 않는다.

- 기존 설치: `inherit`
- explicit 검증과 mixed cohort 확인 후: `winpty`

전역 `pty.useConpty`는 다른 셸의 기본값으로 유지한다.

### 이유

- 가장 문제적인 producer 조합을 직접 해체함
- 전체 구조를 바꾸지 않음
- consumer 쪽 semantic redraw filtering보다 단순하고 결정적임

### 기각한 대안

- 전 셸에서 global `useConpty=false`
  - 영향 범위가 너무 넓음
- PowerShell도 계속 ConPTY 유지 후 renderer normalization
  - semantic terminal interpretation 복잡도와 회귀 리스크가 큼

## ADR-003: 설정은 전역 boolean을 대체하지 않고 PowerShell 전용 override를 추가한다

### 결정

새 optional 설정을 추가한다.

예시:

```json5
pty: {
  useConpty: true,
  windowsPowerShellBackend: "winpty",
}
```

허용값:

- `inherit`
- `conpty`
- `winpty`

### 이유

- 기존 설정과 하위 호환 유지
- 변경 범위를 PowerShell로만 좁힘
- silent behavior flip 방지

### 운영 계약

- `pty.useConpty`: non-PowerShell Windows 세션의 기본값
- `pty.windowsPowerShellBackend`: PowerShell 전용 override

### 1차 릴리스 정책

- field는 config로 추가
- 기본값은 `inherit`
- Settings 표면은 최소한 effective PowerShell backend를 read-only로 드러내거나, `pty.useConpty`가 PowerShell에 직접 적용되지 않을 수 있음을 명시해야 한다

### 반영 범위

- schema
- config types
- settings types
- `RuntimeConfigStore`
- `SettingsService`
- `ConfigFileRepository`
- `config.json5.example`
- startup log / telemetry

## ADR-004: windowsPty 메타데이터는 per-session resolved backend를 반영해야 한다

### 결정

`SessionData.windowsPty`와 snapshot `windowsPty`는 전역 `runtimePtyConfig.useConpty`가 아니라 **해당 세션이 실제로 사용한 backend**를 반영해야 한다.

### 이유

PowerShell만 `winpty`, 나머지는 `conpty`일 수 있으므로 전역 값으로는 truthful contract를 유지할 수 없다.

## ADR-005: snapshot ACK는 restore drain 완료 뒤로 미룬다

### 결정

`screen-snapshot:ready` ACK는 `replaceWithSnapshot()` 직후가 아니라, 실제 restore pending drain이 끝난 뒤 보내도록 설계한다.

### 이유

- redraw burst 중간 프레임이 authority처럼 굳는 것을 방지
- 2차 증폭 요인 제거

### 기각한 대안

- ACK 타이밍 그대로 유지
  - producer 문제를 고친 뒤에도 잔여 깜빡임/중간 프레임 고정 위험이 남음

## ADR-006: winpty는 capability probe와 명시적 실패 정책 없이 사용하지 않는다

### 결정

`winpty` 선택 전 capability probe를 수행한다.

기본 정책:

- explicit `winpty` 요청 + probe 성공 -> winpty 사용
- explicit `winpty` 요청 + probe 실패 -> fail-fast with actionable error
- `inherit`는 기존 경로

### probe contract

- probe timing rule:
  - effective PowerShell default가 `winpty`인 서버는 startup preflight 수행
  - 그 외에는 첫 PowerShell `winpty` session 생성 직전에 lazy one-time probe 수행
- probe 대상: 실제 PowerShell launch shape와 동일한 `winpty` session spawn smoke test
- 성공 기준: PTY 생성 성공 + prompt bootstrap 시작 가능 + timeout 내 종료/정리 가능
- 실패 기준:
  - `winpty-agent.exe` 실행 실패
  - timeout 초과
  - AV/Defender 차단 흔적
  - backend metadata 기록 실패
- timeout: 짧은 고정 timeout
- cache 범위: process lifetime
- 무효화: 서버 재시작 시 재평가
- telemetry:
  - requested backend
  - effective backend
  - probe result
  - failure reason
  - fail-fast 여부

### operator semantics

- startup preflight 실패: 서버는 explicit `winpty` 정책을 fail-fast 상태로 표시
- lazy probe 실패: 해당 세션 생성은 실패하고 운영자에게 actionable error를 반환
- probe 성공 결과는 process lifetime 동안 cache한다

### 이유

- silent fallback은 root cause를 다시 숨긴 채 재도입한다
- AV/Defender 환경에서 `winpty-agent.exe` 차단 가능성이 있다
- 세션 생성 실패와 화면 결함을 혼동하지 않게 해야 한다
