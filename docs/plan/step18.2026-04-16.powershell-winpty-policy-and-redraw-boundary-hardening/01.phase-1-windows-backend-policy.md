# Phase 1: Windows Backend Policy

## Objective

PowerShell 세션을 전역 `useConpty` 값과 분리해서 backend를 결정할 수 있게 만든다.

## Design

### 새 설정 필드

`pty.windowsPowerShellBackend`

허용값:

- `inherit`
- `conpty`
- `winpty`

의도:

- `inherit`: 기존 `useConpty` 따름
- `conpty`: 강제 ConPTY
- `winpty`: 강제 winpty

### precedence and rollout

- Step18 1차 릴리스 기본값은 `inherit`
- 기존 설치는 behavior change 없음
- explicit config 또는 검증된 migration에서만 `winpty` cutover

### winpty capability policy

`winpty`를 고르기 전에 capability probe를 수행한다.

기본 동작:

1. explicit `winpty` 요청
2. probe
3. 성공 시 session spawn
4. 실패 시 actionable error + telemetry

기본 설계는 silent fallback-to-conpty를 허용하지 않는다.

### 새 결정 함수

`SessionManager.resolveWindowsPtyBackend(shellType, shellCmd)`

출력:

- `backend: 'conpty' | 'winpty'`
- `useConpty: boolean`

규칙:

1. non-Windows면 현재 경로 그대로
2. Windows + non-PowerShell이면 기존 `useConpty`
3. Windows + PowerShell이면 `windowsPowerShellBackend`

### 세션 생성 반영

`pty.spawn()`의 `useConpty`는 resolved session backend를 사용한다.

## Implementation Checklist

- config schema에 `pty.windowsPowerShellBackend` 추가
- config types에 동일 필드 추가
- settings types, capability, excluded policy 반영
- `RuntimeConfigStore` / `SettingsService` / `ConfigFileRepository` / startup log 반영
- `SessionManager`에 backend resolution 함수 추가
- `createSession()`에서 per-session `useConpty` 사용
- `runtimePtyConfig` clone/update 경로 반영
- effective backend telemetry / debug visibility 추가

## Test

- Windows PowerShell + `inherit` -> global 값 따름
- Windows PowerShell + `winpty` -> `useConpty=false`
- Windows PowerShell + `conpty` -> `useConpty=true`
- Windows bash/zsh/cmd 경로는 기존 동작 유지
- explicit `winpty` + probe 실패 -> fail-fast 검증
