# Architecture

## 현재 구조 요약

현재 세션 생성 경로:

1. `SessionManager.createSession()`
2. `resolveShell()`로 실제 셸과 `shellType` 결정
3. `pty.spawn(... useConpty: runtimePtyConfig.useConpty ...)`
4. `SessionData.windowsPty = getWindowsPtyInfo()` 저장
5. headless terminal 생성
6. WS snapshot / output / status / cwd 경로 연결

문제는 `useConpty`가 현재 **전역 boolean**이라 PowerShell과 다른 셸을 구분하지 못한다는 점이다.

## 목표 구조

핵심 변경:

- Windows에서 `PowerShell` 세션만 별도 backend policy로 분기
- `useConpty`는 전역 기본값으로 유지
- Step18 1차 릴리스에서는 PowerShell override 기본값을 `inherit`
- explicit cutover 이후 PowerShell 기본값을 `winpty`
- 나머지 셸은 기존 기본 정책 유지

## 목표 데이터 흐름

```text
createSession
  -> resolveShell
  -> resolveWindowsPtyBackend(shellType, shellCmd, config)
  -> pty.spawn(useConpty = resolvedBackend === conpty)
  -> SessionData.windowsPty = resolvedBackend
  -> createHeadlessTerminalState(windowsPty)
  -> getScreenSnapshot() / WsRouter / frontend windowsPty 적용
```

## 왜 이 구조가 맞는가

### 1. 구조를 거의 건드리지 않는다

변경되는 핵심은 세션 생성 시 backend 선택과 그 메타데이터의 truthful propagation 뿐이다.

유지되는 것:

- `SessionManager`
- `WsRouter`
- `TerminalRuntimeLayer`
- `TerminalContainer`
- `TerminalView`
- headless snapshot authority

### 2. 문제를 producer layer에서 끊는다

현재 가장 유력한 문제 producer는 `PowerShell/PSReadLine on ConPTY`다.

그렇다면 consumer 쪽에서 semantic redraw filtering을 구현하기보다, producer 조합을 바꾸는 편이 더 단순하고 결정적이다.

### 3. per-session truthful metadata가 가능하다

현재 `windowsPty` 타입은 이미 `conpty | winpty`를 표현할 수 있다.

따라서 다음이 가능하다.

- PowerShell 세션: `winpty` 또는 `conpty`
  - staged rollout의 `inherit`
  - explicit `conpty`
  - incident rollback
- bash / zsh / sh / cmd / 기타: 기존 값

## 2차 hardening 구조

PowerShell-on-winpty로 1차 producer 문제를 끊더라도, 아래 경계는 보조적으로 정리한다.

### snapshot ACK 경계

현재:

- `replaceWithSnapshot()` 완료 직후 ACK 전송

목표:

- 실제 restore drain 완료 후 ACK 전송

### snapshot replace semantics

현재:

- redraw burst 중간 프레임이 authority처럼 굳을 수 있음

목표:

- restore pending 종료 전 권위 상태 확정 금지

## 호환성 원칙

### 기존 설정과의 호환성

- 기존 `pty.useConpty` 유지
- 새 설정은 PowerShell 전용 optional override로 추가
- Step18 1차 릴리스의 기본값은 `inherit`
- 기존 설치는 silent behavior flip 없이 유지
- explicit rollout 이후에만 example/default guidance를 `winpty`로 전환

### Windows backend precedence

| 조건 | 최종 backend |
|---|---|
| non-Windows | 기존 경로 |
| Windows + non-PowerShell | `pty.useConpty` |
| Windows + PowerShell + `windowsPowerShellBackend = inherit` | `pty.useConpty` |
| Windows + PowerShell + `windowsPowerShellBackend = conpty` | `conpty` |
| Windows + PowerShell + `windowsPowerShellBackend = winpty` | `winpty` |

### interactive-ready boundary

입력 허용은 다음 조건 뒤로 미룬다.

1. session backend 결정 완료
2. first fit/resize 적용 완료
3. PowerShell prompt bootstrap 설치 완료
4. restore pending drain 완료

즉 `input-unblock`은 geometry-ready, prompt-ready, restore-ready 이후에만 열린다.

### readiness ownership

authoritative owner:

- backend session readiness는 서버가 가진다
- visible/input readiness는 프런트가 가진다

계약:

- 서버는 session bootstrap state를 가진다
  - `starting`
  - `ready`
- WS `subscribed` payload는 `ready`를 포함한다
- 세션이 subscribe 이후에 준비되면 WS `session:ready` 이벤트를 보낸다
- 프런트는 아래 조건이 모두 참일 때만 input을 연다
  - session ready
  - visible
  - geometry ready
  - restore ready

early input policy:

- barrier가 닫힌 동안 입력은 **큐잉하지 않고 차단**한다
- helper textarea / input path는 disabled 상태를 유지한다
- open / refresh / reconnect / workspace-restore 모두 같은 규칙을 따른다

### 기존 테스트와의 호환성

- 현재 `windowsPty` 관련 테스트 유지
- PowerShell 전용 backend resolution 테스트 추가
- 기존 tab/grid/runtime regression 그대로 유지
