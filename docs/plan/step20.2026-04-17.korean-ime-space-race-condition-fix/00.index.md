### 주요 기술 결정 (ADR 요약)

- **결정 1**: 1차 수정은 frontend `TerminalView`에서 수행한다.
  - 이유: 입력 문자열은 backend에서 변형되지 않고, 공통 증상이 shell 무관하게 재현되며, 실제 race는 custom key handler가 xterm IME 경계를 우회하는 데서 발생한다.
- **결정 2**: `Space` 전체 제거보다 IME guard 강화가 우선이다.
  - 이유: 기존 영문 prompt/held-space 회귀를 동시에 건드리지 않으면서, 조합 상태일 때만 xterm 네이티브 파이프라인에 위임하는 최소 수정이 가능하다.
- **결정 3**: backend/ConPTY/replay 경계는 후속 hardening 후보로 남긴다.
  - 이유: 2차 증폭 요인으로 보이지만 1차 root cause fix 없이 먼저 건드리면 범위만 커진다.

### 시스템 구조 개요

- 사용자의 한글 입력은 xterm helper textarea를 통해 브라우저 composition 이벤트와 keydown 이벤트를 통과한다.
- 현재 bug는 `TerminalView.attachCustomKeyEventHandler()`가 `Space`를 수동 `onInput(' ')`로 먼저 전송하면서 발생한다.
- xterm 내부 `CompositionHelper`는 final syllable commit을 비동기적으로 확정하므로, 조합 종료 직전 `Space`와 순서 경합이 가능하다.
- 수정 목표는 조합 중/직후 `Space`와 `Backspace`를 xterm 네이티브 처리로 위임하고, non-IME 영문 입력에 대한 기존 동작은 유지하는 것이다.

## 구현 방향 요약

1. `TerminalView`에 helper textarea 기반 composition state 보조 신호를 추가한다.
2. custom key handler에서 `ev.isComposing || ev.keyCode === 229 || isComposingRef.current` OR 가드로 IME 상태를 판단한다.
3. IME 상태일 때는 `Space`/`Backspace` 수동 전달을 금지하고 xterm 네이티브 처리로 위임한다.
4. 영문 `Space`/`Backspace` 수동 전달 회귀가 없는지 기존 keyboard regression을 유지 확인한다.
5. Playwright 디버그 이벤트 기반 IME 전용 회귀 테스트와 수동 한글 입력 매트릭스를 추가한다.

## 성공 기준

- 한글 `안녕하세요 `가 `안녕하세  요`로 깨지지 않는다.
- 한글 `코딩을 합시다`가 `코딩  을 합시다`로 깨지지 않는다.
- IME 조합 중 `Space`/`Backspace`는 `manual_input_forwarded`가 아니라 `ime_guard_delegated` 경로를 탄다.
- 영문 `Space` auto-repeat 및 plain `Backspace` 기존 회귀 테스트가 계속 통과한다.
- PowerShell과 WSL bash 모두에서 수동 재현이 사라진다.
