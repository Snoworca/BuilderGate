# Terminal Context Menu Registered Item Paste Implementation Plan

## 1. 개요

이 계획은 target `0.5.3step1`의 SRS 요구사항 `FR-ARCH-001`부터 `FR-ARCH-006`까지 구현하기 위한 작업 계약이다. 세션 터미널의 우클릭 메뉴에 등록 항목 붙여넣기 메뉴를 추가하고, 해당 메뉴는 사용자가 등록한 커맨드 라인, 디렉토리, 프롬프트 값을 터미널 입력 영역에 붙여넣는다. 이 동작은 실행이 아니다. Enter를 추가하거나 명령 실행 경로를 호출하지 않는다.

선행 작업은 공용 `ContextMenu` 컴포넌트를 다단계 메뉴와 모바일 다이얼로그 모두 처리할 수 있게 만드는 것이다. 데스크톱에서는 hover로 하위 메뉴를 열고, 모바일에서는 경로 기반 다이얼로그로 이동한다. 모든 메뉴는 화면 밖으로 나가지 않도록 배치되며, 물리적으로 화면보다 긴 목록은 스크롤을 가진다.

### Feasibility 요약

| 요구사항 | 난이도 | 판단 |
|---|---|---|
| FR-ARCH-001 | Medium | 이미 `children` 계약이 존재하므로 정규화와 depth 제한을 추가하면 된다. |
| FR-ARCH-002 | Medium | 위치 계산을 순수 함수로 분리하면 root와 submenu가 같은 규칙을 공유할 수 있다. |
| FR-ARCH-003 | High | 모바일 history back과 메뉴 경로 상태가 결합되므로 별도 hook으로 격리한다. |
| FR-ARCH-004 | High | 기존 실행 helper는 Enter를 붙이므로 새 paste-only helper와 terminal input gate API가 필요하다. |
| FR-ARCH-005 | High | 메뉴 open 시점의 최신 API snapshot과 열려 있는 동안의 안정성이 동시에 필요하다. |
| FR-ARCH-006 | Medium | load, validation, terminal ref failure를 console warning과 테스트 관측 지점으로 남긴다. |

Infeasible 항목은 없다.

## 2. 선행 조건 및 전제

- 주요 SPEC: `docs/spec/10.product-architecture.srs.md:29-368`
- 구현 대상은 프런트엔드 React 코드다. 서버 API는 기존 `commandPresetApi.getAll` 계약을 사용한다.
- 이번 실행의 수동 검증과 Playwright E2E 대상 URL은 사용자 지정 포트 `https://localhost:2222`다. 기본 BuilderGate 검증 포트가 필요한 환경에서는 `PLAYWRIGHT_BASE_URL`을 지정하지 않으면 기존 기본값 `https://localhost:2002`를 사용한다.
- 코드와 문서는 UTF-8로 읽고 쓴다.
- 기존 UI의 색, 기본 간격, 메뉴 항목 순서를 유지한다. 새 등록 항목 메뉴는 세션 터미널 메뉴의 마지막 항목으로 배치한다.
- 등록 항목의 상위 메뉴 label은 SRS 연구 결정에 따라 `등록 항목 붙여넣기`로 한다. 하위 category label은 `커맨드 라인`, `디렉토리`, `프롬프트`를 사용한다.
- `CommandPresetDialog`의 실행 버튼은 기존 실행 의미를 유지한다. 이번 기능은 우클릭 메뉴 전용 붙여넣기 경로다.
- xterm.js `Terminal.paste(data)`는 사용하지 않는다. 현재 터미널은 초기 `disableStdin: true` 상태와 bracketed paste 변형 가능성이 있어, 저장값이 no-op 되거나 escape sequence로 변형될 수 있다. 등록 항목 붙여넣기는 printable-only 검증 뒤 기존 terminal input gate에 `command-preset-paste` source로 정확한 payload를 전달한다.
- session status invariant를 변경하지 않는다. 등록 항목 붙여넣기 자체는 사용자의 입력 준비 동작이며, 별도 `running` 전환 근거가 아니다.

## 5. Phase 1 - Shared Context Menu Model And Desktop Containment

### TASK-P1-001 - Normalize nested context menu trees and enforce depth

**관련 요구사항**: `FR-ARCH-001`, `FR-ARCH-005`

**목표**: `ContextMenuItem.children` 계약을 유지하면서 root level 1부터 최대 level 5까지만 렌더링되는 정규화 계층을 만든다. 빈 category, action 없는 leaf, 시작/끝/연속 separator를 제거한다.

**구현 가이드**

1. `contextMenuModel.ts`에서 `ContextMenuItem`을 type-only import한다.
2. `normalizeContextMenuItems`는 입력 배열을 순회한다.
3. separator를 만나면 직전 출력 item이 action item일 때만 임시로 추가한다.
4. action item은 `children`을 먼저 재귀 정규화한다.
5. 현재 depth가 `maxDepth`와 같으면 child tree를 제거하고, 제거된 각 child의 label, path, stripped level을 `onDepthExceeded` callback으로 전달한다.
6. `onClick`이 없고 정규화된 `children`도 비어 있으면 해당 action item을 제거한다.
7. 최종 반환 전에 마지막 separator를 제거한다.
8. disabled action item은 제거하지 않는다. disabled item은 UI 상태를 표현한다.
9. `ContextMenu.tsx`의 렌더 직전 items에 이 정규화를 적용한다.
10. 기본 callback은 `console.debug('[ContextMenu] Stripped unsupported child menu depth', diagnostic)`를 남긴다.
11. desktop parent row는 hover와 click/keyboard activation 모두 child menu를 열 수 있게 유지한다. click은 action 실행이 아니라 submenu open만 수행한다.

**의사코드**

```ts
function normalize(items, depth) {
  const output = [];
  for (const item of items) {
    if (item.separator) {
      if (output.length > 0 && !last(output).separator) output.push(item);
      continue;
    }
    const nextChildren = depth < maxDepth
      ? normalize(item.children ?? [], depth + 1)
      : [];
    if (depth === maxDepth && item.children?.length) reportDepthExceeded(item.children, path);
    const hasAction = typeof item.onClick === 'function';
    if (!hasAction && nextChildren.length === 0) continue;
    output.push({ ...item, children: nextChildren.length > 0 ? nextChildren : undefined });
  }
  return trimTrailingSeparator(output);
}
```

7. `TerminalView`의 `submitCapturedInput`이 TerminalView gate 기준 결과만 반환하도록 확장한다. `ok: true`는 transport 최종 전송 보장이 아니라 TerminalView가 local input gate에서 수락하고 기존 `onInput`/queue path로 넘겼다는 뜻이다.
8. `pasteInput`은 `buildTerminalInputDebugPayload(data, { captureSeq: nextCaptureSeq() })`를 만든 뒤 `submitCapturedInput(data, debugInput, 'command-preset-paste')`를 호출한다.
9. `TerminalContainer`는 `pasteInput`을 child handle로 위임한다.
10. `App.tsx`에 `pasteTerminalInput(tabId, data)` callback을 추가하고 terminal ref의 `pasteInput` 결과를 반환한다.
11. `MosaicContainer` props에 `pasteTerminalInput?: (tabId, data) => TerminalPasteInputResult`를 추가하고 Grid Mode context menu도 이 prop만 사용한다.
12. context menu paste action은 result가 `ok: false`이면 `console.warn('[CommandPresetContextMenu] Paste failed', { reason, presetId, presetLabel })`를 남기고 메뉴를 닫는다.

```ts
function buildCommandPresetPastePayload(preset) {
  if (preset.value.length === 0) return fail('empty-value');
  if (/[\x00-\x1f\x7f]/.test(preset.value)) {
    return preset.kind === 'prompt' && /[\r\n]/.test(preset.value)
      ? fail('unsafe-multiline-prompt')
      : fail('unsafe-control-character');
  }
  if (preset.value.includes('\r') || preset.value.includes('\n')) {
    return preset.kind === 'prompt'
      ? fail('unsafe-multiline-prompt')
      : fail('unsafe-control-character');
  }
  return { ok: true, payload: preset.value };
}
```

**Rationale**

`buildTerminalInput`은 command와 directory에 `\r`을 붙여 실행한다. SRS는 붙여넣기와 실행을 명확히 분리하므로 paste-only helper와 terminal input gate source를 별도로 둔다. xterm `paste()`는 disabled stdin에서 no-op 되거나 bracketed paste escape를 추가할 수 있어 사용하지 않는다. `TerminalPasteInputResult`는 TerminalView local gate 결과만 표현하고, 이후 transport 단계의 delivery 확정으로 해석하지 않는다.

**함정 / 주의사항**

- `sendInput`으로 `preset.value`를 바로 보내지 않는다. 새 `pasteInput` source를 사용해 debug event와 failure reason을 구분한다.
- prompt 멀티라인 지원을 억지로 구현하지 않는다. 안전한 줄바꿈 삽입 경로가 생기기 전에는 diagnostic과 함께 거부한다.

3. 모든 명령이 통과하면 sub-agent reviewer에게 SRS, 계획, 변경 파일 목록, 테스트 결과를 전달한다.
4. reviewer가 finding을 주면 수정 후 테스트와 reviewer 검증을 반복한다.
5. reviewer가 `No findings` 또는 동등한 판정을 내리면 완료로 기록한다.

**Rationale**

이 저장소의 `AGENTS.md`는 command result만으로 완료 선언을 허용하지 않는다. 구현 후 sub-agent validation이 완료 조건이다.

**함정 / 주의사항**

- 계획 문서 자체 변경은 scope 확장이 아니다. 구현 중 SRS 범위 밖 기능이 추가되면 사용자 승인을 받은 뒤 change_log를 갱신한다.
- 테스트 failure를 skip하거나 mock으로 덮지 않는다.

**DoD**

- 모든 pre_commit_gate 명령이 통과한다.
- sub-agent reviewer가 SRS와 계획 대비 구현 누락이 없다고 판정한다.
- 발견된 개선사항은 수정, 테스트, 재검증 loop를 통과한다.

## 10. Scope Freeze Boundary

포함 범위:

- 공용 `ContextMenu`의 최대 5단계 nested menu 렌더링
- desktop menu viewport containment와 overflow scroll
- mobile dialog context menu navigation과 browser back 연동
- 세션 터미널 context menu의 등록 항목 붙여넣기 submenu
- 등록 항목 paste-only payload validation과 terminal input gate paste path
- unit test와 desktop/mobile Playwright regression

제외 범위:

- 등록 항목 CRUD UX 변경
- 서버 command preset API schema 변경
- 터미널 명령 자동 실행 기능
- prompt multiline paste 지원
- context menu 외 다른 메뉴 시스템 교체

## 11. 용어집

| 용어 | 정의 |
|---|---|
| 등록 항목 | 사용자가 저장한 command preset. kind는 `command`, `directory`, `prompt` 중 하나다. |
| 붙여넣기 | 터미널 입력 영역에 문자열을 삽입하는 동작이다. Enter 추가와 실행은 포함하지 않는다. |
| 실행 helper | `buildTerminalInput`처럼 command/directory 실행을 위해 Enter를 붙이는 기존 helper다. |
| 메뉴 level | root menu를 level 1로 계산한 depth 값이다. |
| snapshot | context menu가 열릴 때 API에서 읽은 등록 항목 배열이며, close 전까지 변경하지 않는다. |
| mobile dialog | 모바일 viewport에서 floating context menu 대신 표시되는 modal-style menu surface다. |

## 12. snoworca-coder 라우팅 힌트

- 권장 순서: Phase 1 -> Phase 2 -> Phase 3 -> Phase 4
- Phase 1과 Phase 3의 helper unit test는 서로 독립적으로 먼저 작성할 수 있다.
- Phase 2 E2E는 Phase 1 helper와 mobile rendering이 연결된 뒤 작성한다.
