# Grid Equal Mode Direction Implementation Plan

## 1. 개요

이 계획은 [buildergate.srs.md](../srs/buildergate.srs.md)의 `FR-GRID-013`, `FR-GRID-014`, `FR-GRID-015`, `AC-007`을 구현하기 위한 작업 문서다. 대상 기능은 Grid Mode의 Equal mode에서 tab 수와 visible Grid container 비율에 따라 Mosaic tree를 재구성하는 규칙이다.

작업 범위는 프런트엔드 Equal layout tree 생성, Mosaic container 연결, 회귀 테스트 추가로 제한한다. 서버 API, WebSocket repair, Terminal runtime, UI 시각 요소는 수정 대상이 아니다.

Feasibility 요약: 전체 난이도는 Medium이다. 핵심 리스크는 React Mosaic이 CSS Grid처럼 explicit empty slot을 제공하지 않는 점이다. 해결책은 logical grid를 Mosaic band tree로 표현하고 visible 빈 tile을 만들지 않는 방식이다.

## 2. 선행 조건 및 전제

- 기준 SRS: `docs/srs/buildergate.srs.md:255-274`
- 개발 서버와 E2E 대상: `https://localhost:2002`
- Workspace별 기본 tab 상한은 8개다.
- Equal mode만 대상이다. focus mode, auto mode, none mode는 기존 tree 복원과 면적 조정 계약을 유지한다.
- React Mosaic tree의 `direction: "row"`는 좌우 분할, `direction: "column"`은 상하 분할로 취급한다.

### TASK-P1-001: Add Equal grid decision model

관련 REQ-ID: `FR-GRID-013`, `FR-GRID-014`, `FR-GRID-015`

파일 경로:

- 수정: `frontend/src/utils/mosaic.ts`

메서드/함수 시그니처:

```ts
export interface EqualLayoutMetrics {
  containerWidth: number;
  containerHeight: number;
  cellWidth?: number;
  cellHeight?: number;
  targetColumns?: number;
  targetRows?: number;
  targetAspectRatio?: number;
}

export interface EqualGridSpec {
  arrangement: EqualLayoutArrangement;
  columns: number;
  rows: number;
  bandCounts: number[];
  outerDirection: 'row' | 'column';
  innerDirection: 'row' | 'column';
}

export function selectEqualGridSpec(
  tabCount: number,
  metrics?: EqualLayoutMetrics,
  fallbackArrangement?: EqualLayoutArrangement,
): EqualGridSpec;
```

참고 패턴:

- `frontend/src/utils/mosaic.ts:57-86`의 `buildEqualMosaicTree`가 기존 Equal mode의 arrangement 분기 위치다.
- `frontend/src/utils/mosaic.ts:21-37`의 `buildLinearMosaicTree`가 leaf 순서를 보존하는 재귀 tree 생성 패턴이다.
- `frontend/src/components/Terminal/TerminalView.tsx:808-825`의 cell height 측정 fallback 패턴을 xterm cell 측정 참고로 사용한다.

source_anchors:

- `frontend/src/utils/mosaic.ts:21-37`
- `frontend/src/utils/mosaic.ts:57-86`
- `frontend/src/components/Terminal/TerminalView.tsx:808-825`

구현 가이드:

1. `EqualLayoutMetrics`와 `EqualGridSpec` 타입을 `EqualLayoutArrangement` 바로 아래에 추가한다.
2. `selectEqualGridSpec`를 순수 함수로 구현한다.
3. `tabCount <= 0`이면 `buildEqualMosaicTree`와 동일하게 caller가 reject하도록 Error를 던진다.
4. target aspect ratio는 `cellWidth`와 `cellHeight`가 둘 다 0보다 클 때 `(targetColumns ?? 80) * cellWidth / ((targetRows ?? 24) * cellHeight)`로 산출한다. 측정값이 없으면 `targetAspectRatio`, 그 값도 없으면 1.6을 사용한다.
5. `tabCount <= 3`이면 `containerHeight > containerWidth`일 때 `arrangement = "cols"`, `containerWidth > containerHeight`일 때 `arrangement = "rows"`를 반환한다. 너비와 높이가 같으면 `fallbackArrangement`를 반환하고 fallback이 없을 때만 `rows`를 반환한다.
6. `tabCount >= 4`이면 `columns`와 `rows` 후보를 열거한다. 후보 조건은 `columns * rows >= tabCount`, `columns > 1`, `rows > 1`, `columns <= tabCount`, `rows <= tabCount`다.
7. 각 후보의 `tileAspectRatio = (containerWidth / columns) / (containerHeight / rows)`를 계산한다.
8. 각 후보의 score tuple을 `[aspectDistance, emptySlots, longAxisPenalty]`로 계산한다.
   - `aspectDistance = Math.abs(Math.log(tileAspectRatio / targetAspectRatio))`
   - `emptySlots = columns * rows - tabCount`
   - `longAxisPenalty = containerWidth >= containerHeight ? (columns >= rows ? 0 : 1) : (rows >= columns ? 0 : 1)`
9. score tuple을 순서대로 비교해 가장 작은 후보를 선택한다. 완전 동률이면 `fallbackArrangement`와 같은 장축 후보를 고른다.
10. baseline 결과는 다음과 일치해야 한다.
   - 가로형: 4개 `columns=2 rows=2`, 5~6개 `columns=3 rows=2`, 7~8개 `columns=3 rows=3`
   - 세로형: 4개 `columns=2 rows=2`, 5~6개 `columns=2 rows=3`, 7~8개 `columns=3 rows=3`
11. `bandCounts`는 visible 빈 tile을 만들지 않도록 실제 tab 수만 분배한다.
   - 가로형 7개: `[3, 2, 2]`
   - 가로형 8개: `[3, 3, 2]`
   - 세로형 7개: `[3, 2, 2]`
   - 세로형 8개: `[3, 3, 2]`
12. `bandCounts` 생성은 선택된 orientation을 따른다. `rows`는 row band, `cols`는 column band로 해석한다.

Rationale:

layout 판단을 `mosaic.ts`의 순수 함수로 분리하면 unit test가 DOM 없이 전체 tab 수 조합을 검증할 수 있다. `MosaicContainer`는 visible container 치수만 제공하므로 React 상태 경로가 얇게 유지된다.

함정 / 주의사항:

- `rows`는 가로형 화면의 logical row-major 배치를 의미하고, `cols`는 세로형 화면의 logical column-major 배치를 의미한다.
- `targetAspectRatio`는 xterm cell 측정값이 없는 fallback이다. cell 측정값이 있으면 SRS에 따라 80x24 기준 비율을 먼저 사용한다.
- `tabCount`는 rebuild 후의 target tab 수다. 추가 전 tab 수로 계산하면 3번째 tab 추가 방향이 틀어진다.

테스트 작성 지침:

- 테스트 파일: `frontend/tests/unit/mosaicEqualLayout.test.ts`

- 경계: square metrics에서 fallback arrangement가 유지되고, 4, 6, 7, 8개에서 baseline grid와 `bandCounts`가 SRS 결론과 일치한다.
