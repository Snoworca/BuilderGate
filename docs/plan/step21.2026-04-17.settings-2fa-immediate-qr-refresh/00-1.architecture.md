# Architecture

## 1. 현재 구조 요약

현재 2FA/QR 관련 흐름은 다음과 같다.

1. `server/src/index.ts`가 서버 시작 시 한 번만 `totpService`를 생성한다.
2. `server/src/routes/authRoutes.ts`는 `getTOTPService()` accessor가 돌려주는 현재 인스턴스를 그대로 사용한다.
3. `server/src/services/SettingsService.ts`는 설정 저장 시 `RuntimeConfigStore`, `AuthService`, `SessionManager`, `FileService`만 갱신한다.
4. `frontend/src/components/Settings/SettingsPage.tsx`는 페이지 진입 시에만 `/api/auth/totp-qr`를 호출한다.

그 결과:

- 저장 직후 backend `totpService` 상태가 바뀌지 않는다.
- 저장 직후 frontend QR 상태도 갱신되지 않는다.
- README 설명대로 “다음 로그인부터 적용”만 보이고, 현재 화면의 즉시 피드백은 없다.

## 2. 목표 구조

### 2.1 서버 런타임 갱신

```text
PATCH /api/settings
  -> SettingsService.savePatch()
    -> persist nextConfig
    -> updateTwoFactorRuntime(nextConfig, changedKeys)
      -> index.ts helper
         -> destroy old totpService (if needed)
         -> create new TOTPService(nextConfig.twoFactor, cryptoService)
         -> initialize()
         -> on failure: keep unregistered service + warning
```

### 2.2 프런트 즉시 갱신

```text
SettingsPage.save()
  -> settingsApi.patchSettings()
  -> setSnapshot / setDraft / setSummary
  -> if twoFactor fields changed:
       - enabled=false: clear QR state immediately
       - enabled=true or metadata changed: refreshTotpQr()
           -> GET /api/auth/totp-qr
           -> setTotpQR()
```

### 2.3 authRoutes 계약 유지

`authRoutes`의 소유권은 바꾸지 않는다.

- `/api/auth/login`은 최신 `totpService`를 통해 이후 로그인 흐름을 판정한다.
- `/api/auth/totp-qr`는 최신 `totpService.generateQRDataUrl()` 결과를 사용한다.
- 별도 새로운 QR 전용 설정 API는 만들지 않는다.

## 3. 권장 책임 분리

| 컴포넌트 | 책임 | 비고 |
|---|---|---|
| `index.ts` | `totpService` lifecycle owner | 기존 전역 변수 유지 |
| `SettingsService` | 저장 orchestration + callback 호출 + warning 반영 | global import 금지 |
| `TOTPService` | secret 로드/생성, QR data URL 생성, registered 상태 제공 | 기존 클래스 재사용 |
| `authRoutes` | 최신 runtime accessor 소비 | 구조 유지 |
| `SettingsPage` | 저장 후 QR refresh orchestration | raw fetch 중복 제거 필요 |
| `api.ts` | `totp-qr` helper 또는 auth API 래퍼 | auth header 재사용 |

## 4. 권장 데이터 흐름

### 4.1 Save Success

```text
User toggles 2FA / edits issuer
  -> PATCH /api/settings
  -> response.applySummary + response.warnings
  -> frontend QR refresh
  -> visible QR or unregistered message
```

### 4.2 Save Success With TOTP Init Failure

```text
User enables 2FA
  -> index.ts rebuild helper creates TOTPService
  -> initialize() fails (e.g. corrupted secret)
  -> helper returns warning, leaves service unregistered
  -> PATCH still succeeds
  -> GET /api/auth/totp-qr returns { registered: false, dataUrl: '' }
  -> UI shows warning + unregistered message
```

### 4.3 Disable 2FA

```text
User disables 2FA
  -> PATCH /api/settings
  -> helper destroys old totpService and clears in-memory runtime
  -> frontend clears QR state immediately
  -> later /api/auth/totp-qr returns 404
```

## 5. 실패 정책

### 5.1 예상 가능한 TOTP 초기화 실패

예상 가능한 TOTP 초기화 실패는 설정 저장 전체를 실패시키지 않는다.

이유:

- 현재 서버 시작 시에도 TOTP 초기화 실패는 hard fail이 아니라 warning + degraded state로 처리된다.
- `twoFactor.enabled=true but not configured` 상태는 기존 FR-401 의미와 일치한다.
- 사용자에게 QR 대신 경고와 `registered=false` 상태를 보여주는 편이 복구 경로를 더 명확히 제공한다.

### 5.2 예상 밖 런타임 callback 예외

예상 밖 callback 예외는 기존 `SettingsService.applyRuntimeConfig()` rollback 규칙을 따른다.

- `RuntimeConfigStore`, `AuthService`, `SessionManager`, `FileService` rollback 유지
- `totpService`도 `previousConfig` 기준으로 재복원

## 6. 테스트 구조

| 레이어 | 목표 | 권장 방식 |
|---|---|---|
| 서버 단위 테스트 | runtime callback 호출, warning 수집, disable/enable lifecycle | `server/src/test-runner.ts` |
| 라우트/통합 검증 | `/api/auth/totp-qr`가 최신 runtime 상태를 반영하는지 | server harness 또는 수동 smoke |
| 프런트 E2E | 저장 직후 QR이 같은 화면에서 갱신되는지 | Playwright |
| 문서 검증 | README와 계획/검증 문구 일치 | manual review |

## 7. 제약

| 제약 | 영향 |
|---|---|
| `SettingsService`는 현재 TOTP 의존성이 없다 | callback injection이 필요하다 |
| `TOTPService`는 생성자 config를 내부에 고정한다 | metadata 변경 시 재생성이 필요하다 |
| 설정 UI는 raw `fetch('/api/auth/totp-qr')` 1회 호출 구조다 | mount/save 공용 helper로 정리해야 한다 |
| AGENTS 검증 규칙은 `https://localhost:2002` 고정이다 | Playwright와 수동 검증 모두 같은 포트를 사용해야 한다 |
| README는 아직 “다음 로그인부터 적용”으로 설명한다 | 즉시 QR 갱신 semantics를 문서에 추가해야 한다 |

## 8. Phase-Architecture Mapping

| Phase | 구현 컴포넌트 | 결과 |
|---|---|---|
| Phase 1 | `index.ts`, `SettingsService`, `authRoutes` 검증 | 최신 TOTP runtime을 저장 직후 사용할 수 있는 backend contract |
| Phase 2 | `SettingsPage`, `api.ts`, settings UX state | 저장 직후 QR 재조회와 경고/disable 반영 |
| Phase 3 | `test-runner`, Playwright, README, validation docs | 회귀 방지와 공개 문서 정렬 |
