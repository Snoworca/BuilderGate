### 주요 기술 결정 요약

- `totpService` 소유권은 계속 `index.ts`에 둔다.
- `SettingsService`는 `TOTPService`를 직접 생성하지 않고 callback으로 재구성을 요청한다.
- QR 이미지는 `PATCH /api/settings` 응답에 넣지 않고, 저장 성공 후 `GET /api/auth/totp-qr`를 다시 호출한다.
- `twoFactor.*`의 인증 적용 범위 메타데이터는 `new_logins`로 유지하되, 설정 UI의 QR 미리보기는 저장 직후 갱신한다.
- TOTP secret 초기화 실패는 save hard-fail이 아니라 warning + `registered=false` 상태로 노출한다.

### 구현 흐름 요약

```text
SettingsPage Save
  -> PATCH /api/settings
    -> SettingsService.savePatch()
      -> ConfigFileRepository persist
      -> updateTwoFactorRuntime(nextConfig, changedKeys)
      -> applySummary.warnings 반영
  -> save 성공 후 QR helper 재호출
    -> GET /api/auth/totp-qr
    -> 최신 dataUrl / registered 상태 반영
```

## 구현 방향 요약

1. 백엔드에 `twoFactor.enabled`, `issuer`, `accountName` 변경 감지와 `totpService` 재구성 hook을 추가한다.
2. 저장 성공 후 프런트에서 QR 정보를 다시 가져오는 helper를 도입한다.
3. `enabled=false` 저장 시 QR 이미지를 즉시 제거하고, `enabled=true` 저장 시 새 QR 또는 unregistered 상태를 바로 보여준다.
4. warning을 save 응답과 UI 배너에 노출해 secret 손상/미등록 상태를 숨기지 않는다.
5. 서버 test-runner와 Playwright 또는 수동 검증 가이드를 함께 정리한다.

## 성공 기준

- 설정 페이지에서 2FA를 켜고 저장하면 같은 화면에서 QR 이미지가 바로 나타난다.
- 설정 페이지에서 `issuer` 또는 `accountName`을 바꾸고 저장하면 같은 화면의 QR URI와 이미지가 즉시 바뀐다.
- 설정 페이지에서 2FA를 끄고 저장하면 QR 이미지가 즉시 제거된다.
- `/api/auth/totp-qr`는 서버 재시작 없이 최신 TOTP 런타임 상태를 반영한다.
- secret 손상/미등록 상태에서는 save 이후 warning과 `registered=false` 상태가 즉시 드러난다.
- `https://localhost:2002` 기준 회귀 테스트와 수동 검증이 모두 통과한다.
