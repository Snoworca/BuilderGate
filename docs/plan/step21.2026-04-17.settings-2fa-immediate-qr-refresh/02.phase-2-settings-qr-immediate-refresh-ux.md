## Phase 2: Settings QR Immediate Refresh UX

### 목표

설정 페이지에서 2FA 저장 성공 직후 QR 이미지와 URI, unregistered 상태, warning 메시지가 즉시 갱신되도록 만든다.

## 선행 조건

- Phase 1 완료 필요

## 기존 코드 재사용 힌트

- `frontend/src/components/Settings/SettingsPage.tsx`
  - 현재 QR 상태(`totpQR`, `totpQRLoading`, `totpQRError`)와 QR 렌더링 블록을 그대로 재사용한다.
- `frontend/src/services/api.ts`
  - auth header / authFetch helper를 재사용해 `totp-qr` 호출을 정리한다.
- `server/src/routes/authRoutes.ts`
  - QR API 계약은 유지하고 프런트는 재조회만 수행한다.

## 코드 중복 주의

- IMPL-002 / IMPL-006
  - 페이지 진입 fetch와 저장 후 refresh를 서로 다른 raw fetch 경로로 복제하지 않는다.
- IMPL-005
  - warning banner와 QR error/unregistered 문구를 중복으로 경쟁시키지 않는다. 우선순위를 정한다.

## 언어+버전 주의사항

- React 19 패턴을 유지한다.
- 기존 상태 구조를 크게 바꾸지 말고 `save()`와 mount effect가 같은 helper를 공유하도록 한다.
- `fetch` 대신 helper를 사용할 때 401 만료 흐름이 기존 `authFetch`와 충돌하지 않게 맞춘다.

## LLM 코딩 주의사항

- Simplicity: settings page에 별도 state machine을 도입하지 않는다. 기존 state 몇 개와 helper 함수로 끝낸다.
- Surgical: settings UI 전체 레이아웃을 바꾸지 말고 QR section과 save success 흐름만 최소 수정한다.

## 테스트 시나리오

### 정상 케이스

- Given: 설정 페이지가 열린 상태다
  When: 2FA를 켜고 저장한다
  Then: 페이지를 닫거나 다시 열지 않아도 QR 이미지가 나타난다

- Given: 2FA가 켜져 있고 QR이 보인다
  When: issuer를 바꾸고 저장한다
  Then: QR URI와 이미지가 새 issuer 기준으로 갱신된다

- Given: 2FA가 켜져 있고 QR이 보인다
  When: 2FA를 끄고 저장한다
  Then: QR 이미지는 즉시 제거된다

### 예외 케이스

- Given: save는 성공했지만 backend warning이 반환된다
  When: 프런트가 QR을 다시 읽는다
  Then: warning banner와 `registered=false` 문구가 보이고 stale QR은 남지 않는다

- Given: 저장 후 QR fetch가 일시적으로 실패한다
  When: 네트워크 오류가 발생한다
  Then: save success 상태는 유지하고 QR section에 fetch 실패 문구가 노출된다

### 경계값 테스트

- Given: 2FA enable과 metadata 변경이 한 번에 저장된다
  When: save가 성공한다
  Then: QR refresh는 중복 호출 없이 1회만 수행되고 최종 값만 화면에 남는다
