## Phase 1: Backend TOTP Runtime Hot Apply

### 목표

설정 저장 직후 `twoFactor.enabled`, `issuer`, `accountName`, `externalOnly` 변경이 실행 중인 서버의 `totpService` 런타임에 반영되도록 만든다.

## 선행 조건

- 없음

## 기존 코드 재사용 힌트

- `server/src/index.ts`
  - `totpService` 전역 변수와 startup initialization 흐름을 그대로 소유권 기준점으로 사용한다.
- `server/src/services/SettingsService.ts`
  - 기존 `applyRuntimeConfig()` rollback 패턴을 재사용한다.
- `server/src/routes/authRoutes.ts`
  - `getTOTPService()` accessor 구조를 유지하고, 최신 인스턴스가 자연스럽게 보이게 한다.
- `server/src/services/TOTPService.ts`
  - `initialize()`, `destroy()`, `generateQRDataUrl()`을 그대로 재사용한다.

## 코드 중복 주의

- IMPL-001 / IMPL-004
  - 두 번째 TOTP singleton 레이어를 만들지 않는다.
- IMPL-005
  - startup initialization 정책과 설정 저장 정책을 서로 다르게 구현하지 않는다. warning 처리 기준을 공유한다.

## 언어+버전 주의사항

- TypeScript 5.x + Node.js ESM 코드베이스다.
- `index.ts`와 서비스 계층 간 import cycle이 생기지 않도록 callback 주입으로 끝낸다.
- `TOTPService` 생성/파기는 동기 init 경로와 예외 처리를 명확히 나눠야 한다.

## LLM 코딩 주의사항

- Simplicity: `TOTPService`를 전면 리팩터링하지 말고, lifecycle orchestration만 추가한다.
- Surgical: `authRoutes`의 login/verify 본문을 건드리지 말고 accessor가 가리키는 인스턴스만 최신화한다.

## 테스트 시나리오

### 정상 케이스

- Given: `totpService`가 비활성 상태다
  When: `twoFactor.enabled=true` patch가 저장된다
  Then: save 직후 새로운 `totpService`가 생성되고 QR endpoint가 `registered=true` 또는 `registered=false` 상태를 일관되게 반환한다

- Given: `totpService`가 활성 상태다
  When: `issuer` 또는 `accountName`이 저장된다
  Then: 기존 인스턴스가 새 config 기준으로 재생성되고 이후 QR URI가 새 메타데이터를 반영한다

- Given: `totpService`가 활성 상태다
  When: `twoFactor.enabled=false`가 저장된다
  Then: `totpService`가 제거되고 이후 `/api/auth/totp-qr`는 404를 반환한다

### 예외 케이스

- Given: secret 파일이 손상되어 있다
  When: 2FA enable 또는 metadata save가 수행된다
  Then: save는 warning과 함께 성공하고 런타임은 `registered=false` 상태로 남는다

- Given: unrelated settings patch다
  When: save가 수행된다
  Then: TOTP runtime callback은 재구성을 일으키지 않는다

### 경계값 테스트

- Given: `twoFactor.enabled`, `issuer`, `accountName`이 한 patch에 함께 들어 있다
  When: save가 수행된다
  Then: 재구성은 1회만 일어나고 최종 config 기준 인스턴스가 남는다
