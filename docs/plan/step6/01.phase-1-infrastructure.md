# Phase 1 Plan

## Infrastructure: Type Definitions, PaneTree Pure Functions, IndexedDB Module, React Hook, Unit Tests

---

## Objective

Build the foundational data layer for the tmux-style Pane Split System. This phase produces four new frontend modules and a unit test suite that the remaining five phases depend on. No visual UI or server-side changes are introduced; every artifact is a pure data/logic concern.

Deliverables:

1. `frontend/src/types/pane.types.ts` -- Type definitions and constants
2. `frontend/src/utils/paneTree.ts` -- 14 pure immutable tree-manipulation functions
3. `frontend/src/utils/paneDb.ts` -- IndexedDB schema, DB open, Promise-based transaction helpers
4. `frontend/src/hooks/usePaneDB.ts` -- React hook wrapping `paneDb.ts` with error handling
5. `frontend/src/utils/__tests__/paneTree.test.ts` -- Unit tests TC-6801 through TC-6807

### Task 1.1: Type Definitions

**File**: `frontend/src/types/pane.types.ts`

Create a new file containing all type definitions, interfaces, and constants for the Pane system. This file has zero runtime dependencies -- only TypeScript types and `const` values.

#### 1.1.1 Core Node Types

```typescript
/** Terminal leaf node -- renders one TerminalContainer */
export interface PaneLeaf {
  type: 'terminal';
  id: string;           // UUID v4
  sessionId: string;    // Backend PTY session ID
}

/** Split internal node -- divides space between two children */
export interface PaneSplit {
  type: 'split';
  id: string;           // UUID v4
  direction: Direction;
  ratio: number;        // 0.15..0.85, fraction for first child
  children: [PaneNode, PaneNode];  // exactly 2 (binary tree)
}

/** Discriminated union -- every node is either a leaf or a split */
export type PaneNode = PaneLeaf | PaneSplit;
```

Implementation notes:
- Use `type: 'terminal'` and `type: 'split'` as discriminants so `node.type === 'terminal'` narrows the type.
- `PaneSplit.children` is a **tuple** of exactly 2. TypeScript's `[PaneNode, PaneNode]` enforces this at compile time.

### Task 1.2: PaneTree Pure Functions

**File**: `frontend/src/utils/paneTree.ts`

Implement all 14 pure functions specified in SRS section 3.3. Every function:
- Accepts a `PaneNode` tree (or related inputs) and returns a new tree (immutable update).
- Must not mutate any input argument.
- Must not import React, DOM APIs, or any browser globals.
- Must be fully unit-testable in a Node.js environment.

Import only from `../types/pane.types`.

#### 1.2.1 `splitPane`

```typescript
/**
 * Split an existing PaneLeaf into a PaneSplit with the original leaf as
 * the first child and a new leaf (newSessionId) as the second child.
 *
 * @returns New tree root with the split applied.
 * @throws If paneId is not found in the tree.
 */
export function splitPane(
  root: PaneNode,
  paneId: string,
  direction: Direction,
  newSessionId: string
): PaneNode;
```

Implementation notes:
- Generate a new UUID for the PaneSplit and the new PaneLeaf (use `crypto.randomUUID()`).
- Initial ratio = 0.5.
- Recursively walk the tree. When `node.id === paneId && node.type === 'terminal'`, replace that node with a new `PaneSplit` containing the original node (deep-copied) as `children[0]` and a new `PaneLeaf { type: 'terminal', id: newId, sessionId: newSessionId }` as `children[1]`.
- If the target paneId is not found, throw an Error.
- For all non-matching nodes, shallow-copy the node and recurse into children if it is a PaneSplit.

#### 1.2.2 `closePane`

```typescript
/**
 * Remove a PaneLeaf from the tree. Its sibling replaces the parent PaneSplit.
 *
 * @returns New tree root, or null if the root itself was removed (single-leaf tree).
 */
export function closePane(
  root: PaneNode,
  paneId: string
): PaneNode | null;
```

Implementation notes:
- If `root.type === 'terminal' && root.id === paneId`, return `null`.
- Otherwise, recursively find the parent PaneSplit whose child matches `paneId`. Replace that PaneSplit with the sibling child (the child whose id is NOT paneId).
- Walk using a recursive helper that returns the transformed subtree.
- If the paneId is not found in the tree, return the tree unchanged.

#### 1.2.3 `resizePane`

```typescript
/**
 * Update the ratio of a PaneSplit node.
 * Clamps to [MIN_RATIO, MAX_RATIO].

export function resizePane(
  root: PaneNode,
  splitId: string,

```

Implementation notes:
- Clamp `ratio` to `[PANE_CONSTANTS.MIN_RATIO, PANE_CONSTANTS.MAX_RATIO]` using `Math.max(MIN, Math.min(MAX, ratio))`.
- Recursively walk; when `node.type === 'split' && node.id === splitId`, return a copy with the new ratio.

#### 1.2.4 `swapPanes`

```typescript
/**
 * Swap the positions (including sessionId) of two PaneLeaf nodes.
 */
export function swapPanes(
  root: PaneNode,
  paneIdA: string,
  paneIdB: string
): PaneNode;
```

Implementation notes:
- First, find both PaneLeaf nodes using `findPane`.
- If either is not found, return the tree unchanged.
- Perform a single recursive walk, replacing paneIdA's leaf with a copy that has paneIdB's `sessionId`, and vice versa. The `id` fields remain unchanged (they identify position); only `sessionId` is swapped.
- Alternatively, swap the entire leaf objects at their respective locations, preserving original tree position IDs.

#### 1.2.10 `getAdjacentPane`

```typescript
/**
 * Find the adjacent PaneLeaf in the given direction.
 * Uses tree-structural adjacency (not visual bounding-box -- that is for Phase 2).
 *
 * For Phase 1, this implements a simplified tree-walk algorithm:
 *   - Walk up from the target leaf to find a PaneSplit whose direction
 *     matches the requested movement axis.
 *   - Select the sibling subtree and return its nearest leaf.
 *
 * Phase 2 may override with DOM-based bounding-box logic per SRS 3.5.
 */
export function getAdjacentPane(
  root: PaneNode,
  paneId: string,
  direction: FocusDirection
): PaneLeaf | null;
```

Implementation notes:
- Map `FocusDirection` to axis: `left`/`right` --> `'vertical'`, `up`/`down` --> `'horizontal'`.
- Map `FocusDirection` to child index: `left`/`up` means the target pane should be in `children[1]` of a matching split (so the sibling is `children[0]`); `right`/`down` means the target pane should be in `children[0]` (sibling is `children[1]`).
- Walk up the tree from the target pane by collecting the path from root to the pane (list of parent splits with child-index at each level).
- Traverse the path from bottom to top. For each PaneSplit whose `direction` matches the axis:
  - Determine which child subtree the target pane is in (index 0 or 1).
  - If the direction requires moving to the other child (e.g., `right` and pane is in `children[0]`), then the adjacent pane is the nearest leaf in the sibling subtree on the correct edge.
- "Nearest leaf on the correct edge": for `right`, take the leftmost leaf of `children[1]` (recursively pick `children[0]`). For `left`, take the rightmost leaf of `children[0]` (recursively pick `children[1]`). Similarly for `up`/`down`.
- If no matching split is found on the path, return `null`.

### Task 1.3: IndexedDB Module

**File**: `frontend/src/utils/paneDb.ts`

Provide a low-level Promise-based wrapper around the native IndexedDB API. This module handles DB open, schema creation, and generic transaction helpers. It does NOT contain React hooks -- that is `usePaneDB.ts`.

#### 1.3.1 `openDB`

```typescript
import { PANE_DB } from '../types/pane.types';

let dbInstance: IDBDatabase | null = null;

/**
 * Open (or return cached) the IndexedDB database.
 * Creates Object Stores on first open via onupgradeneeded.
 */
export function openDB(): Promise<IDBDatabase>;
```

Implementation notes:
- Cache the `IDBDatabase` instance in a module-level variable to avoid re-opening on every call.
- Listen for `onversionchange` on the cached instance to close if another tab upgrades the schema.
- On `onupgradeneeded`:

```typescript
request.onupgradeneeded = (event) => {
  const db = (event.target as IDBOpenDBRequest).result;

  // paneLayouts
  if (!db.objectStoreNames.contains(PANE_DB.STORES.PANE_LAYOUTS)) {
    const store = db.createObjectStore(PANE_DB.STORES.PANE_LAYOUTS, { keyPath: 'sessionId' });
    store.createIndex('byUpdatedAt', 'updatedAt', { unique: false });
  }

  // savedLayouts
  if (!db.objectStoreNames.contains(PANE_DB.STORES.SAVED_LAYOUTS)) {
    const store = db.createObjectStore(PANE_DB.STORES.SAVED_LAYOUTS, { keyPath: 'id' });
    store.createIndex('byName', 'name', { unique: false });
  }

  // sessionMeta
  if (!db.objectStoreNames.contains(PANE_DB.STORES.SESSION_META)) {
    db.createObjectStore(PANE_DB.STORES.SESSION_META, { keyPath: 'sessionId' });
  }
};
```

- Version upgrades follow a chain pattern per SRS 4.1: check `oldVersion` and run migrations sequentially. For v1 this is just the initial creation above.

#### 1.3.2 Generic Transaction Helpers

```typescript
/**
 * Put a record into an Object Store (insert or update).
 */
export function dbPut<T>(storeName: string, data: T): Promise<void>;

/**
 * Get a record by key from an Object Store.
 */
export function dbGet<T>(storeName: string, key: IDBValidKey): Promise<T | undefined>;

/**
 * Delete a record by key.
 */
export function dbDelete(storeName: string, key: IDBValidKey): Promise<void>;

/**
 * Get all records from an Object Store.
 */
export function dbGetAll<T>(storeName: string): Promise<T[]>;
```

Implementation pattern (per ADR-602):

```typescript
export async function dbPut<T>(storeName: string, data: T): Promise<void> {
  const db = await openDB();
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, 'readwrite');
    tx.objectStore(storeName).put(data);
    tx.oncomplete = () => resolve();
    tx.onerror = () => reject(tx.error);
  });
}
```

Apply the same pattern for `dbGet`, `dbDelete`, `dbGetAll`.

#### 1.4.2 `saveLayout`

```typescript
async function saveLayout(sessionId: string, layout: PaneLayout): Promise<void>
```

Implementation notes:
- Build a `PaneLayoutRecord`: `{ sessionId, layout, updatedAt: Date.now() }`.
- Call `dbPut(PANE_DB.STORES.PANE_LAYOUTS, record)`.
- On failure, retry up to 3 times with 100ms delay between attempts.
- On final failure, log `console.warn('Failed to save pane layout after 3 attempts')`.
- If IndexedDB is unavailable, save a serialized JSON string to `localStorage` under key `pane_layout_{sessionId}`.

#### 1.4.3 `loadLayout`

```typescript
async function loadLayout(sessionId: string): Promise<PaneLayout | null>
```

Implementation notes:
- Call `dbGet<PaneLayoutRecord>(PANE_DB.STORES.PANE_LAYOUTS, sessionId)`.
- Return `record?.layout ?? null`.
- If IndexedDB is unavailable, attempt `JSON.parse(localStorage.getItem('pane_layout_' + sessionId))`.
- On any parse error, return `null`.

#### 1.4.4 `deleteLayout`

```typescript
async function deleteLayout(sessionId: string): Promise<void>
```

Implementation notes:
- Call `dbDelete(PANE_DB.STORES.PANE_LAYOUTS, sessionId)`.
- If IndexedDB is unavailable, remove `localStorage.getItem('pane_layout_' + sessionId)`.

#### 1.4.5 `savePreset`

```typescript
async function savePreset(name: string, layout: PaneLayout): Promise<string>
```

Implementation notes:
- Generate a UUID for the new preset: `const id = crypto.randomUUID()`.
- Replace all `sessionId` values in the layout tree with `PLACEHOLDER_SESSION_ID` using a recursive helper function. This ensures saved presets are reusable with fresh sessions.
- Build a `SavedLayoutRecord`: `{ id, name, layout: sanitizedLayout, isBuiltIn: false, paneCount: countPanes(layout.root), createdAt: Date.now() }`.
- Call `dbPut(PANE_DB.STORES.SAVED_LAYOUTS, record)`.
- Return `id`.
- QuotaExceededError handling: catch specifically, attempt to delete the oldest custom preset (query by `createdAt`, exclude `isBuiltIn: true`), then retry the put once. If still fails, throw with user-facing message.

#### 1.4.6 `loadPresets`

```typescript
async function loadPresets(): Promise<SavedLayoutRecord[]>
```

Implementation notes:
- Call `dbGetAll<SavedLayoutRecord>(PANE_DB.STORES.SAVED_LAYOUTS)`.
- Sort: built-in first (by preset order), then custom by `createdAt` descending.

#### 1.4.7 `deletePreset`

```typescript
async function deletePreset(id: string): Promise<void>
```

Implementation notes:
- First, fetch the record with `dbGet`. If `record.isBuiltIn === true`, throw an Error: `'Cannot delete built-in preset'`.
- Otherwise, call `dbDelete(PANE_DB.STORES.SAVED_LAYOUTS, id)`.

#### 1.4.8 `initBuiltInPresets`

```typescript
async function initBuiltInPresets(): Promise<void>
```

Implementation notes:
- For each entry in `BUILT_IN_PRESETS`, check if a record with that `id` already exists in `savedLayouts`.
- If it does NOT exist, build the layout using `buildPresetLayout(preset.type, Array(preset.paneCount).fill(PLACEHOLDER_SESSION_ID))` and write a `SavedLayoutRecord` with `isBuiltIn: true`.
- This function is idempotent and safe to call on every app load.

#### 1.4.9 `migrateFromLocalStorage`

```typescript
async function migrateFromLocalStorage(): Promise<void>
```

Implementation notes per SRS section 4.5 (FR-6407):
1. Check `localStorage.getItem('migrated_to_idb')`. If present, return immediately.
2. Scan localStorage for keys matching `tab_state_{sessionId}` pattern.
3. For each found key:
   a. Parse the JSON value. If parsing fails, skip that key (log warning).
   b. Extract the terminal tab entries. For each terminal tab with a `sessionId`, create a single `PaneLeaf`.
   c. If there is exactly one terminal tab, the `PaneLayout.root` is that leaf. If there are multiple, nest them into a simple vertical split chain.
   d. Save the constructed `PaneLayout` to IndexedDB via `dbPut(PANE_DB.STORES.PANE_LAYOUTS, ...)`.
4. Handle `child_session_ids`: read from localStorage. The Pane tree already references these session IDs, so no additional action is needed beyond constructing the tree correctly.
5. On success: `localStorage.setItem('migrated_to_idb', Date.now().toString())`.
6. Do NOT delete existing localStorage keys (SRS: "기존 localStorage 키는 삭제하지 않음").
7. On any failure during migration: log warning, do NOT set the migration flag (so it retries next load), do NOT corrupt existing localStorage data.

#### 1.4.10 Hook Initialization

```typescript
export function usePaneDB(): UsePaneDBReturn {
  const isAvailable = useMemo(() => isIndexedDBAvailable(), []);

  // Initialize built-in presets on first mount
  useEffect(() => {
    if (isAvailable) {
      initBuiltInPresets().catch(err =>
        console.warn('Failed to initialize built-in presets:', err)
      );
      migrateFromLocalStorage().catch(err =>
        console.warn('Failed to migrate from localStorage:', err)
      );
    }
  }, [isAvailable]);

  return {
    saveLayout,
    loadLayout,
    deleteLayout,
    savePreset,
    loadPresets,
    deletePreset,
    initBuiltInPresets,
    migrateFromLocalStorage,
    isAvailable,
  };
}
```

Implementation notes:
- The initialization `useEffect` runs once on mount. Both `initBuiltInPresets` and `migrateFromLocalStorage` are idempotent and safe to run every time.
- The functions themselves are module-level (not re-created per render) since they are stateless async operations over IndexedDB. `useCallback` wrapping is not needed.
- If IndexedDB open fails (SRS 4.3: "DB 열기 실패"), the `isAvailable` flag remains `true` but individual operations will catch and fall back to console warnings. A more robust approach: wrap `openDB` with a one-time check and set a module-level `dbFailed` flag if the first open fails, then all subsequent calls use localStorage fallback.

#### 1.4.11 Error Handling Summary (SRS 4.3)

| Error Scenario | Behavior |
|----------------|----------|
| IndexedDB not supported | `isAvailable` = false; all methods use localStorage fallback; console warning on first access |
| DB open failure | All methods degrade to in-memory/localStorage; StatusBar warning (surfaced via returned error state or console) |
| Save failure | Retry up to 3 times with 100ms delay; on final failure log console.warn |
| Migration failure | Log warning, do NOT set migration flag, existing localStorage preserved |
| QuotaExceededError | Auto-delete oldest custom preset, retry once; on second failure throw with user-facing message |
| Load returns corrupt data | Return `null`, log warning |
