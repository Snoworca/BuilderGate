## 2. Technology Decisions

### TD-001: Password Encryption Algorithm

| 항목 | 결정 |
|------|------|
| **Status** | Approved |
| **Context** | 설정 파일에 비밀번호를 안전하게 저장해야 함 |
| **Decision** | AES-256-GCM + PBKDF2 |
| **Rationale** | - AES-256: NIST 표준, 강력한 대칭 암호화<br>- GCM 모드: 인증된 암호화 (무결성 보장)<br>- PBKDF2: 키 유도 표준, 100,000 iterations |
| **Alternatives** | - bcrypt/argon2: 해시 전용, 복호화 불가<br>- AES-CBC: 인증 태그 없음<br>- ChaCha20-Poly1305: 호환성 낮음 |
| **Consequences** | - Node.js crypto 내장 모듈 사용 가능<br>- Salt/IV 저장 필요 (암호문에 포함) |

**Implementation Details**:
```typescript
// Format: enc(base64(salt[32] + iv[12] + authTag[16] + ciphertext))
// Key Derivation: PBKDF2-SHA256, 100,000 iterations
// Dependencies: Node.js crypto (built-in)
```

## 3. Architectural Decisions

### AD-001: Middleware Stack Order

| 항목 | 결정 |
|------|------|
| **Status** | Approved |
| **Context** | Express 미들웨어 실행 순서 결정 |
| **Decision** | Rate Limiter → CORS → Security Headers → Body Parser → Auth → Input Validator → Audit Logger → Routes |
| **Rationale** | 1. Rate Limiter: 모든 요청에 먼저 적용<br>2. CORS: Origin 검증<br>3. Security Headers: 응답 헤더 설정<br>4. Body Parser: JSON 파싱<br>5. Auth: 인증 검증<br>6. Input Validator: 입력 검증<br>7. Audit Logger: 요청 로깅<br>8. Routes: 비즈니스 로직 |

**Implementation**:
```typescript
app.use(rateLimiter);
app.use(corsMiddleware);
app.use(securityHeaders);
app.use(express.json());
app.use('/api/auth', authRoutes);  // No auth middleware for login
app.use('/api/sessions', authMiddleware, sessionRoutes);
app.use(auditLogger);
app.use(errorHandler);
```

### AD-002: Token Blacklist Strategy

| 항목 | 결정 |
|------|------|
| **Status** | Approved |
| **Context** | JWT 토큰 무효화 (로그아웃, 갱신) 처리 |
| **Decision** | 인메모리 Map + TTL 기반 자동 정리 |
| **Rationale** | - 단일 서버 환경에 적합<br>- Redis 등 외부 의존성 없음<br>- TTL로 메모리 누수 방지 |
| **Alternatives** | - Redis: 분산 환경 필요 시<br>- 데이터베이스: 오버헤드 큼<br>- 짧은 토큰 수명만: 사용성 저하 |
| **Consequences** | - 서버 재시작 시 블랙리스트 초기화<br>- 토큰 만료 전까지만 저장 |

**Data Structure**:
```typescript
// Map<jti, expiresAt>
private tokenBlacklist: Map<string, number> = new Map();
```

### AD-003: Session Ownership Model

| 항목 | 결정 |
|------|------|
| **Status** | Approved |
| **Context** | PTY 세션과 JWT 토큰의 관계 정의 |
| **Decision** | JWT jti를 세션 ownerJti로 바인딩 |
| **Rationale** | - 세션 생성 시 jti 기록<br>- 토큰 갱신 시에도 동일 jti 유지 (특별 처리)<br>- 간단하고 명확한 소유권 |
| **Alternatives** | - 별도 세션 ID: 복잡성 증가<br>- IP 바인딩: 동적 IP 문제<br>- 무제한 접근: 보안 취약 |
| **Consequences** | - 하트비트 시 jti 체인 관리 필요<br>- 로그아웃 시 세션 접근 불가 |

**Special Handling**:
```typescript
// 하트비트로 토큰 갱신 시
// - 새 토큰 발급 (새 jti)
// - 이전 jti 블랙리스트 추가
// - 세션의 ownerJti는 유지 (또는 새 jti로 업데이트)
```

## 6. Performance Considerations

### 6.1 Crypto Performance

| 연산 | 목표 시간 | 근거 |
|------|-----------|------|
| PBKDF2 (100k iter) | < 500ms | NFR-501 |
| JWT 검증 | < 10ms | NFR-502 |

### 6.2 Memory Usage

| 구조 | 예상 크기 | 정리 전략 |
|------|-----------|-----------|
| Token Blacklist | ~100 entries × 100 bytes | TTL 기반 자동 정리 |
| OTP Store | ~1000 entries × 200 bytes | 만료 시 정리, FIFO |
| Login Attempts | ~10000 entries × 100 bytes | 1시간 비활성 시 정리 |
| Rate Limit | ~10000 entries × 50 bytes | 윈도우 기반 정리 |
