# Terminal Title Auto Tab Name Implementation Plan

## 1. 개요

이 계획은 BuilderGate가 Codex, Claude Code, Hermes 같은 터미널 TUI의 Terminal Title 신호를 읽어 기본 이름 Tab의 표시 이름을 자동 갱신하도록 만든다. 사용자가 Tab 이름을 직접 편집한 뒤에는 같은 세션에서 title 신호가 계속 발생해도 사용자 이름을 보존한다.

스펙 출처는 `docs/srs/buildergate.srs.md`의 `FR-WSP-012`부터 `FR-WSP-015`, 데이터 요구사항의 `nameSource`와 `terminalTitle`, REST API의 `PATCH /api/workspaces/:wid/tabs/:tid`, 수용 기준 `AC-022`다.

권위 경로는 서버다. PTY raw output에서 `OSC 0`과 `OSC 2` title sequence를 감지하고, `SessionManager`가 `WorkspaceService`에 title을 알린다. `WorkspaceService`는 Tab이 자동 모드 대상인지 판정한 뒤 이름을 갱신하고, `WsRouter.broadcastAll("tab:updated", ...)`로 프런트엔드 상태를 동기화한다.

### Feasibility 요약

| 요구사항 | 난이도 | 판단 |
|---|---|---|
| FR-WSP-012 | Medium | PTY raw output에서 OSC title을 파싱하면 xterm 화면 가시성과 무관하게 처리 가능하다. |
| FR-WSP-013 | Medium | `nameSource: "user"`를 저장하면 수동 편집 잠금을 결정적으로 구현할 수 있다. |
| FR-WSP-014 | High | REST rename과 자동 rename 경로를 분리하고 WebSocket broadcast callback을 추가해야 한다. |
| FR-WSP-015 | Medium | sanitizer와 debounce 정책을 서버에 두면 모든 클라이언트가 같은 결과를 본다. |

Infeasible 항목은 없다.

## 3. 프로젝트 온보딩 컨텍스트

BuilderGate는 브라우저 기반 다중 터미널 작업 환경이다. 서버는 Express, WebSocket, node-pty, headless xterm을 사용해 세션과 Workspace state를 관리한다. 프런트엔드는 React, xterm.js, React Mosaic을 사용해 Tab Mode와 Grid Mode를 표시한다.

핵심 규칙은 다음과 같다.

- REST `PATCH /api/workspaces/:wid/tabs/:tid`는 사용자 수동 rename 경로다.
- 자동 rename은 이 REST 경로를 호출하지 않는다.
- 자동 rename이 발생하면 기존 `tab:updated` WebSocket event의 `changes.name` 소비자가 그대로 갱신되어야 한다.
- `OSC 133` idle detection과 새 `OSC 0/2` title detection은 목적이 다르다. 두 parser의 상태를 섞지 않는다.
- title parser는 output을 strip하지 않는다. xterm 화면에는 기존 output path가 그대로 전달된다.

## 5. Phase 1: Server title signal detection

목표: PTY raw output에서 Terminal Title 신호를 감지하고 sanitize된 title event를 `SessionManager` 내부 신호로 만들되, 화면 output과 session status 흐름은 변경하지 않는다.

### TASK-P1-001: Add OSC terminal title detector and sanitizer

관련 REQ-ID: `FR-WSP-012`, `FR-WSP-015`, `AC-022`

파일 경로:

- 신규: `server/src/utils/terminalTitle.ts`
- 수정: `server/src/test-runner.ts`

메서드/함수 시그니처:

```ts
export type TerminalTitleSource = 'osc0' | 'osc2';
export interface TerminalTitleEvent {
  source: TerminalTitleSource;
  rawTitle: string;
  title: string;
}

export type TerminalTitleCallback = (event: TerminalTitleEvent) => void;

export class TerminalTitleDetector {
  setCallback(cb: TerminalTitleCallback): void;
  process(data: string): void;
  destroy(): void;
}

export function sanitizeTerminalTitle(raw: string, maxLength?: number): string | null;
export function isDefaultTerminalTabName(name: string): boolean;
```

참고 패턴:

- `server/src/services/OscDetector.ts`의 residual buffer와 callback 패턴을 따른다.
- `server/src/utils/terminalPayload.ts`의 OSC terminator 인식 방식을 참고한다.
- `server/src/test-runner.ts`의 terminal payload truncation 테스트 등록 위치에 새 테스트를 추가한다.

source_anchors:

- `server/src/services/OscDetector.ts:34-105`
- `server/src/utils/terminalPayload.ts:42-85`
- `server/src/test-runner.ts:166-170`
- `server/src/test-runner.ts:4016-4051`

구현 가이드:

1. `TerminalTitleDetector`는 `\x1b]0;...BEL`, `\x1b]2;...BEL`, `\x1b]0;...ESC\`, `\x1b]2;...ESC\` 네 형태만 title event로 처리한다.
2. `OSC 1`과 `OSC 133`은 이 detector에서 title event로 처리하지 않는다.
3. chunk 경계에서 `ESC]0;Hel`과 `loBEL`이 나뉘어도 residual buffer를 사용해 하나의 event를 발생시킨다.
4. `process(data)`는 원본 output을 반환하거나 strip하지 않는다. 이 detector는 side-channel event만 만든다.
5. `sanitizeTerminalTitle()`은 control character, BEL, ESC, C1 OSC/ST, 줄바꿈, tab, bidi format character를 제거하거나 공백 하나로 접는다.
6. sanitize 후 빈 문자열이면 `null`을 반환한다.
7. 기본 길이 제한은 현재 Tab rename UI와 서버 검증에 맞춰 32 visible character로 둔다.
8. `isDefaultTerminalTabName()`은 `^Terminal-[1-9]\d*$`만 true로 본다. 사용자가 직접 `Terminal-1 memo`처럼 만든 이름은 default로 보지 않는다.
9. `TerminalTitleDetector`에는 `MAX_OSC_TITLE_PAYLOAD_CHARS = 4096` raw payload 상한을 둔다. residual buffer는 active OSC title payload를 이 상한보다 크게 보관하지 않는다.
10. title OSC 시작 후 terminator가 오기 전에 raw payload가 4096자를 초과하면 해당 OSC sequence는 title event 없이 폐기하고, 같은 chunk의 남은 문자열에서 다음 `ESC]0;` 또는 `ESC]2;` 시작점을 다시 찾는다.
11. over-cap sequence가 폐기된 뒤 같은 stream에 정상 길이 title OSC가 이어지면 정상 title만 event로 emit한다.
12. `destroy()`는 residual buffer와 over-cap 폐기 상태를 모두 초기화한다.

Rationale:

서버 raw parser를 두면 프런트 xterm이 unmounted 또는 hidden 상태여도 title 신호를 놓치지 않는다. output을 strip하지 않으므로 terminal rendering, scrollback, screen repair와 충돌하지 않는다.

함정 / 주의사항:

- `OscDetector`는 OSC 133을 strip한다. 새 detector는 strip하면 안 된다.
- regex 하나로 전체 스트림을 처리하면 매우 긴 OSC payload에서 메모리와 backtracking 위험이 생긴다. index scan 방식으로 구현한다.
- title payload가 너무 길면 32자 표시 이름만 반환하고 full raw title 저장은 하지 않는다.
- terminator 없는 긴 OSC payload는 32자 sanitize로 해결되지 않는다. raw residual 상한 초과 시 event 없이 sequence를 폐기해야 한다.

테스트 작성 지침:

- `server/src/test-runner.ts`에 최소 5개 테스트를 추가한다.
- 성공: OSC 0 BEL, OSC 2 ST가 sanitize된 title을 emit한다.
- 실패: OSC 1, OSC 133, 빈 title은 emit하지 않는다.
- 경계: title sequence가 두 chunk로 나뉘어도 한 번만 emit한다.
- 경계: control character와 bidi character가 제거되고 32자로 제한된다.
- 경계: unterminated `ESC]0;` payload가 4096자를 초과해도 residual buffer가 상한을 넘지 않고 event가 발생하지 않는다.
- 경계: over-cap title sequence 폐기 후 이어지는 정상 `ESC]2;RecoveredBEL` title은 `Recovered` event를 발생시킨다.

### TASK-P1-002: Emit terminal title events from SessionManager

```ts
export type TerminalTitleChangeCallback = (sessionId: string, title: string) => void;

```

참고 패턴:

- `SessionData`에 `oscDetector`를 저장하고 `deleteSession()`에서 `destroy()`하는 구조를 따른다.
- `onCwdChange()`처럼 외부 service가 SessionManager event를 구독하는 형태를 따른다.

source_anchors:

- `server/src/services/SessionManager.ts:150-187`
- `server/src/services/SessionManager.ts:300-338`
- `server/src/services/SessionManager.ts:375-382`
- `server/src/services/SessionManager.ts:801-815`
- `server/src/services/WorkspaceService.ts:37-44`

구현 가이드:

1. `SessionData`에 `terminalTitleDetector: TerminalTitleDetector`를 추가한다.
2. `createSession()`에서 detector를 생성하고 callback을 등록한다.
3. callback은 `this.terminalTitleChangeCallback?.(id, event.title)`만 수행한다. 이 단계에서 Workspace state를 직접 수정하지 않는다.
4. PTY `onData(rawData)` 초입에서 `sData.terminalTitleDetector.process(rawData)`를 호출한다.
5. `deleteSession()`에서 `terminalTitleDetector.destroy()`를 호출한다.
6. `onTerminalTitleChange()` public method를 추가한다. 마지막 등록 callback 1개를 유지하는 `onCwdChange()`와 같은 단순 계약으로 충분하다.

4. `login(page)` 후 API로 `TitleAuto-${Date.now()}` workspace와 PowerShell tab을 만든다.
5. `localStorage.active_workspace_id`를 새 workspace ID로 설정하고 reload한다.
6. terminal prompt가 준비되면 PowerShell command `[Console]::Out.Write("`e]0;E2E Auto Title`a")`를 입력한다.
7. UI의 MetadataRow 또는 workspace tabbar에서 `E2E Auto Title`이 보이는지 poll한다.
8. rapid burst 검증을 위해 PowerShell에서 `Burst 1`, `Burst 2`, `Burst Final` OSC title을 한 command 안에서 연속 출력한다.
9. debounce window 이후 `window.__buildergateTabNameUpdates`에 `Burst 1` 또는 `Burst 2`가 없고 `Burst Final`만 반영되었는지 확인한다.
10. REST `PATCH /api/workspaces/:wid/tabs/:tid`로 tab 이름을 `Manual Lock`으로 바꾼다.
11. 다시 title command `[Console]::Out.Write("`e]0;Ignored Auto Title`a")`를 입력한다.
12. GET `/api/workspaces`와 UI 양쪽에서 tab name이 `Manual Lock`으로 유지되는지 확인한다.
13. 테스트 workspace 이름 prefix를 cleanup allowlist에 추가하거나 spec 내부에서 afterEach 삭제한다.

Rationale:

서버 unit test만으로는 WebSocket `tab:updated`가 실제 React state와 표시 이름에 반영되는지 확인할 수 없다. E2E는 AC-022의 사용자 관찰 결과를 검증한다.

함정 / 주의사항:

- PowerShell escape sequence는 JavaScript 문자열과 PowerShell 문자열 escaping이 겹친다. 테스트 command literal은 실제 브라우저 입력 기준으로 작성하고, 실패 시 terminal visible text를 artifact로 남긴다.
- title update는 debounce 때문에 즉시 보이지 않을 수 있다. `expect.poll()` timeout은 5000ms 이상으로 둔다.
- E2E는 서버가 이미 `https://localhost:2002`에서 떠 있다는 전제의 기존 Playwright 정책을 따른다.

테스트 작성 지침:

- 성공: manual rename 후 title update가 이름을 바꾸지 않는다.
- 경계: 자동 title update 후 `GET /api/workspaces`의 `nameSource`가 `terminal-title`인지 확인한다.

## 11. 용어집

| 용어 | 정의 |
|---|---|
| OSC | Operating System Command escape sequence. 터미널 title, hyperlink, shell integration marker 등에 쓰인다. |
| OSC 0 | icon name과 window title을 함께 설정하는 sequence. 본 기능은 title로 해석한다. |
