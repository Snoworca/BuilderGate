# 기술 결정 기록

## ADR-001: 운영 런타임 기준점은 root가 아니라 `server`

### 결정

운영 런타임 기준 디렉터리는 `server`로 유지한다.

### 근거

- `server`는 이미 `dist/index.js`, `config.json5`, `certs`, `data`를 중심으로 동작한다.
- `WorkspaceService`와 `TOTPService` 일부 경로는 아직 `process.cwd()` 영향을 받는다.
- 루트 단일 `dist`는 이번 작업보다 큰 경로 정규화 과제가 된다.

### 결과

- `start.js`는 `cwd=server`를 강제한다.
- production public assets도 `server/dist/public`에 둔다.

## ADR-002: 프런트 빌드는 기본 `frontend/dist`, staging은 후처리 copy

### 결정

Vite `outDir`을 바로 `../server/dist/public`으로 바꾸지 않고, `frontend/dist`를 만든 뒤 후처리로 복사한다.

### 근거

- Vite 공식 문서는 root 밖 `outDir`에 대해 경고를 둔다.
- 기본 build 구성을 유지하면 개발 흐름과 충돌이 적다.
- hashed asset cleanup 문제를 build.js에서 명시적으로 제어할 수 있다.

### 결과

- build.js가 staging cleanup/copy 책임을 가진다.
- `frontend/vite.config.ts`는 1차 범위에서 최소 수정 또는 무수정으로 유지할 수 있다.

## ADR-003: production static serving은 절대 경로 + HTML navigation fallback

### 결정

`server/src/index.ts` production branch는 절대 경로 `publicDir`을 사용한 `express.static`과 HTML navigation fallback을 사용한다.

### 근거

- Express 공식 문서는 static root에 절대 경로 사용을 권장한다.
- client는 same-origin `/api`, `/ws`를 사용하므로 서버가 HTML/assets를 직접 내려주는 구조가 가장 단순하다.
- 자산 누락 시 HTML을 잘못 반환하지 않도록 asset request와 HTML request를 분리해야 한다.

### 결과

- `/api`, `/health`, `/ws`는 기존 route/upgrade 유지
- `/assets/*`, `/logo.svg`는 static middleware
- HTML GET만 `index.html`

## ADR-004: `start.js`는 thin launcher여야 한다

### 결정

`start.js`는 서버 기능을 재구현하지 않고, 환경/작업 디렉터리/시그널 전달만 담당하는 thin launcher로 만든다.

### 이유

- TLS, redirect, ws, auth, data path는 이미 `server/dist/index.js`가 책임지고 있다.
- launcher가 서버 기능을 복제하면 유지보수 포인트만 늘어난다.

### 결과

- `start.js` 책임:
  - args parse
  - port resolution
  - env setup
  - cwd setup
  - child lifecycle/exit propagation

## ADR-004A: `start.js` 포트 우선순위는 `--port > config server.port > 2222`

### 결정

`start.js`는 다음 우선순위로 포트를 결정한다.

1. CLI `--port`
2. `server/config.json5`의 `server.port`
3. 둘 다 없으면 `2222`

### 이유

- 사용자가 명시적으로 기본 fallback을 `2222`로 요구했다.
- 기존 설정 파일이 있으면 그 값을 존중하는 편이 운영 계약상 자연스럽다.
- AGENTS 검증 포트 `2002`는 validation concern이지 launcher default와 동일할 필요는 없다.

### 결과

- 기본 실행 예시는 `node start.js`
- AGENTS 검증 예시는 `node start.js --port 2002` 또는 config `server.port=2002`

## ADR-005: 1차 도입에서는 dev E2E를 유지하고 production smoke를 병행한다

### 결정

기존 `frontend/playwright.config.ts`의 `dev.js` 기반 webServer를 즉시 폐기하지 않는다.

### 이유

- production launcher 검증은 build 비용이 있어 개발 피드백 루프가 느려진다.
- 먼저 manual/smoke 절차를 분리해서 안정화하는 편이 안전하다.

### 결과

- Phase 4에서 production smoke 절차를 문서화한다.
- 이후 필요 시 별도 Playwright production config 또는 opt-in script를 추가한다.

## ADR-006: runtime path normalization은 후속 후보로 둔다

### 결정

`WorkspaceService`/`TOTPService`의 `process.cwd()` 의존 제거는 이번 변경의 필수 범위에 넣지 않는다.

### 이유

- `start.js`의 `cwd=server` 계약만으로 현재 문제를 해결할 수 있다.
- 전면 path normalization은 테스트 영향 범위가 더 넓다.

### 결과

- 이번 1차 릴리스는 launcher contract로 path stability를 보장한다.
- 후속 hardening 항목으로 별도 분리한다.

## 채택하지 않은 대안

### 대안 A: 루트 단일 `dist/`

기각 이유:

- 현재 상대경로 계약과 충돌 가능성이 높다.
- 1차 목표 대비 과도하게 크다.

### 대안 B: frontend outDir 직접 `server/dist/public`

기각 이유:

- 가능하지만 root 밖 outDir 관리가 1차 구현에서 더 조심스럽다.
- staging을 build script로 분리하는 편이 rollback/cleanup이 단순하다.

### 대안 C: `start.js`가 server code를 직접 import해서 실행

기각 이유:

- Node module mode, working directory, dist/source 선택 문제가 늘어난다.
- 기존 `server/dist/index.js` start contract를 재사용하는 편이 안전하다.
