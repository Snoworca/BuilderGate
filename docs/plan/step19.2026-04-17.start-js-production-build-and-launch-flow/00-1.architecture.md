# Architecture

## 1. 현재 구조 요약

현재 로컬 실행 경로는 다음과 같다.

1. 루트 `dev.js`가 `server` 개발 서버와 `frontend` Vite dev 서버를 같이 띄운다.
2. `server/src/index.ts`는 개발 환경에서만 Vite proxy를 붙인다.
3. production 모드에서는 `/health`, `/api`, `/ws`는 살아 있으나 `/` 정적 진입점은 없다.

근거:

- `server/src/index.ts`의 Vite proxy 분기
- `frontend/src/services/api.ts`의 relative `/api`
- `frontend/src/contexts/WebSocketContext.tsx`의 same-origin `/ws`

즉, 운영 경로에서 필요한 것은 별도 프런트 서버가 아니라 **같은 HTTPS origin에서 static HTML/assets를 서빙하는 production branch**다.

## 2. 목표 구조

### 2.1 Production Build Flow

```text
root build.js
  -> server/package.json : npm run build
  -> frontend/package.json : npm run build
  -> clean server/dist/public
  -> copy frontend/dist -> server/dist/public
```

### 2.2 Production Launch Flow

```text
node start.js
  -> resolve port
       1) --port
       2) server/config.json5 server.port
       3) 2222 fallback
  -> spawn process.execPath dist/index.js
       cwd = <repo>/server
       env.NODE_ENV = production
       env.PORT = resolved port only when override/fallback is needed
  -> server/dist/index.js
```

### 2.3 Request Routing

```text
Browser
  -> https://localhost:<resolved-port>/
       -> Express HTTPS server
          -> /api/*      : existing API routes
          -> /health     : existing health route
          -> /ws         : existing WebSocket upgrade
          -> /assets/*   : express.static(publicDir)
          -> /logo.svg   : express.static(publicDir)
          -> HTML GET    : sendFile(public/index.html)
```

## 3. 핵심 설계 원칙

### 3.1 운영 런타임 기준점은 `server`

이번 계획은 루트 단일 `dist/`로 가지 않는다.

이유:

- `server`는 이미 `config.json5`, `certs`, `data`, `dist/index.js`를 기준으로 동작한다.
- `WorkspaceService`, `TOTPService` 일부 경로는 아직 `process.cwd()`의 영향을 받는다.
- `start.js`에서 `cwd=server`를 고정하면 1차 목표를 가장 작은 변경으로 달성할 수 있다.

### 3.2 static path는 절대 경로

production static serving은 `server/dist/index.js` 기준 절대 경로를 사용한다.

권장 구조:

```text
server/dist/index.js
server/dist/public/index.html
server/dist/public/assets/*
```

사용 방식:

```text
publicDir = <absolute server dist public>
app.use(express.static(publicDir, { index: false }))
app.get(nonAssetHtmlRoutes, send index.html)
```

### 3.3 asset 404와 HTML fallback 분리

SPA fallback을 단순 catch-all로 넣으면 `/assets/missing.js`도 HTML을 돌려줄 수 있다.

따라서 production branch는:

- static middleware로 실제 자산 우선 처리
- HTML navigation으로 판단되는 GET만 `index.html`
- missing asset은 기존대로 404

로 설계한다.

### 3.4 build orchestration은 "재사용 + 조립"만 수행

루트 build script는 다음만 한다.

- 기존 `server` build script 호출
- 기존 `frontend` build script 호출
- staging cleanup/copy

다음은 하지 않는다.

- tsc/vite 세부 로직 재구현
- 별도 root dist tree 생성
- 새 번들러/워크스페이스 도입

## 4. 컴포넌트 책임

| 컴포넌트 | 책임 | 비고 |
|---|---|---|
| `server/src/index.ts` | production static serving, HTML fallback, route ordering | 기존 API/WS/TLS 유지 |
| `server/src/utils/config.ts` | server root helper 재사용 지점 | public path 계산 참고 |
| `frontend` build output | hashed JS/CSS/index.html 제공 | Vite 기본 build 유지 |
| 루트 `build.js` | build orchestration + public staging | fail-fast |
| 루트 `start.js` | thin launcher | no server logic duplication |
| README / Playwright docs | 운영 실행/검증 절차 설명 | Phase 4 |

## 5. 시퀀스

### 5.1 build sequence

```text
User
  -> node build.js
    -> npm run build (server)
    -> npm run build (frontend)
    -> validate server/dist exists
    -> remove server/dist/public safely
    -> copy frontend/dist -> server/dist/public
    -> success / fail-fast exit
```

### 5.2 start sequence

```text
User
  -> node start.js
    -> resolve port (--port > config server.port > 2222)
    -> spawn node dist/index.js with cwd=server
    -> HTTPS server starts on resolved port
    -> HTTP redirect server starts on resolved port - 1
```

### 5.3 browser sequence

```text
Browser
  -> GET /
    -> Express production branch
    -> sendFile server/dist/public/index.html
  -> GET /assets/index-*.js
    -> express.static(publicDir)
  -> POST /api/auth/login
    -> existing API route
  -> GET /ws?token=...
    -> existing upgrade -> WsRouter
```

## 6. 비기능 제약

| 제약 | 영향 |
|---|---|
| AGENTS 검증 포트는 `https://localhost:2002` | start.js 기본 fallback은 2222지만, 검증 시에는 `--port 2002` 또는 config `server.port=2002`를 사용해야 한다 |
| self-signed TLS는 server가 계속 담당 | 별도 reverse proxy 추가 금지 |
| `process.cwd()` 의존 경로 존재 | launcher contract에 `cwd=server` 필수 |
| 기존 dev flow 보존 | `dev.js`와 production path를 섞지 않는다 |
| same-origin API/WS 계약 | static hosting도 같은 origin이어야 한다 |

## 7. Phase-Architecture Mapping

| Phase | 구현 컴포넌트 | 핵심 결과 |
|---|---|---|
| Phase 1 | `server/src/index.ts`, path helper reuse | production static serving contract |
| Phase 2 | root `build.js`, staging helpers | repeatable build artifact assembly |
| Phase 3 | root `start.js` | stable launcher contract |
| Phase 4 | README, Playwright/manual docs | validation alignment |

## 8. 비목표와 후속 후보

1차 범위 밖:

- `WorkspaceService`/`TOTPService`의 모든 relative path를 server-root helper로 재작성
- root `package.json`/workspace 도입
- dev E2E 전량을 production launcher 기준으로 교체

후속 후보:

- dedicated production Playwright config 추가
- runtime path normalization helper 도입
- packaging/zip distribution 준비
