# Step 19: start.js Production Build And Launch Flow

## 목표

`dev.js` 개발 경로를 유지한 채, `node build.js` + `node start.js`로 동작하는 운영 실행 경로를 추가한다.

핵심 산출물:

- 루트 `build.js`
- 루트 `start.js`
- `server` production static serving
- `server/dist/public` 자산 staging
- `start.js` 포트 결정 규칙: `--port > config server.port > 2222`
- `https://localhost:2002` 기준 운영 smoke validation 경로

## 구현 방향 요약

이 계획은 다음 방향을 채택한다.

1. 운영 런타임 루트는 `server`로 유지한다.
2. 프런트 운영 자산은 `frontend/dist`를 빌드한 뒤 `server/dist/public`으로 staging한다.
3. production 서버는 `server/dist/public`을 절대 경로로 정적 서빙한다.
4. 루트 `start.js`는 `cwd=server`로 `dist/index.js`를 실행해 현재 `process.cwd()` 의존 경로를 깨지 않는다.
5. 기존 `dev.js`, Vite reverse proxy, 개발 E2E 흐름은 1차 변경 범위에서 유지한다.

## 성공 기준

- `node build.js`가 `server`/`frontend` 빌드를 순차 수행하고 `server/dist/public`을 만든다.
- `node start.js` 실행 시, CLI 인자와 설정 값이 모두 없으면 기본 포트 `2222`로 기동한다.
- `node start.js --port 2002` 실행 시 `https://localhost:2002/health`가 응답한다.
- `https://localhost:2002/`가 `Cannot GET /` 대신 앱 진입 HTML을 반환한다.
- `/api`, `/health`, `/ws` 경로는 기존 동작을 유지한다.
- 개발용 `node dev.js` 흐름은 깨지지 않는다.
- production smoke validation 절차가 문서화된다.

## 범위 밖

- 루트 단일 `dist/` 패키징
- npm workspace 도입
- SSR/manifest 기반 HTML 렌더링
- 기존 `process.cwd()` 의존 경로 전체 정규화
- 기존 Playwright dev flow의 완전 교체
