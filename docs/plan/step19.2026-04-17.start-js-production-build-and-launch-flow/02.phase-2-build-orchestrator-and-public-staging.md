## Phase 2: Build Orchestrator And Public Staging

### 목표

루트에서 반복 가능한 production build command를 제공하고, `frontend/dist`를 `server/dist/public`으로 안전하게 staging한다.

## 선행 조건

- [ ] Phase 1 완료
- [ ] `server` production static branch가 `server/dist/public`을 읽을 준비가 되어 있음

## 기존 코드 재사용 힌트

- `server/package.json`
  - 기존 `npm run build`가 server compile + `shell-integration` copy를 수행한다.
- `frontend/package.json`
  - 기존 `npm run build`가 Vite production build를 수행한다.
- `dev.js`
  - child process orchestration, port arg parsing, cleanup 패턴 참고 가능

## 코드 중복 주의

- IMPL-003
  - root script에서 tsc/vite 세부 커맨드를 직접 복제하지 않는다. 반드시 기존 subproject `npm run build`를 호출한다.
- IMPL-004
  - 운영 자산 staging 위치는 `server/dist/public` 하나로 통일한다. `root/dist`, `server/public`, `frontend/build` 같은 대체 산출물을 추가하지 않는다.

## 언어+버전 주의사항

- 루트에는 `package.json`의 `"type": "module"` 선언이 없다.
  - 1차 구현에서는 `build.js`를 CommonJS로 두어 기존 `dev.js`와 일관성을 맞춘다.
- Windows에서 `npm` 실행은 `npm.cmd` 또는 shell-aware 호출을 고려해야 한다.
- file operation은 절대 경로로 계산하고, delete target이 `server/dist/public` 하위인지 확인한다.

## LLM 코딩 주의사항

- Simplicity: npm workspace, task runner, monorepo tool 도입 금지.
- Surgical: 루트 build orchestration만 추가하고 각 subproject build script는 최대한 건드리지 않는다.

## 테스트 시나리오

### 정상 케이스

- Given: `server`와 `frontend` dependency가 설치되어 있다
  When: 루트에서 `node build.js`
  Then: server build 성공 후 frontend build 성공, `server/dist/public/index.html`이 생성된다

- Given: 기존 `server/dist/public`에 이전 hashed asset이 있다
  When: `node build.js`
  Then: old assets가 제거되고 새 build output만 남는다

### 예외 케이스

- Given: `server` build가 실패한다
  When: `node build.js`
  Then: 즉시 non-zero exit로 종료하고 frontend build/copy는 실행되지 않는다

- Given: `frontend` build가 실패한다
  When: `node build.js`
  Then: staging copy 없이 non-zero exit로 종료한다

- Given: cleanup target 계산이 의도 경로 밖으로 벗어난다
  When: build script가 delete를 시도한다
  Then: fail-fast로 중단한다

### 경계값 테스트

- Given: repeated builds
  When: `node build.js`를 여러 번 실행
  Then: 결과 디렉터리가 누적 오염 없이 idempotent하게 유지된다
