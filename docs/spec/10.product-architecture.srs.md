### FR-ARCH-001 — Context menu multi-level navigation

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | high |
| Tags | terminal, context-menu, desktop |
| Risk | medium |
| Stability | stable |
| Verification Method | Unit tests for menu tree filtering/depth behavior and Playwright desktop regression coverage for hover/click submenu opening. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

The shared context menu component shall render action items with child menus up to five visible menu levels. The root context menu counts as level 1, and the deepest visible child menu counts as level 5. Desktop users shall be able to open a child menu from a parent item without closing the parent menu, and the component shall omit empty child menu nodes rather than rendering empty submenus.

#### Rationale

The registered item paste workflow requires category and item submenus, and the shared component needs an explicit maximum depth to avoid unbounded UI behavior.

#### Acceptance Criteria

- [ ] AC-1: Given a desktop viewport and a context menu item with visible children, when the user hovers over or activates the parent item, then the child menu opens while the parent menu remains visible.
- [ ] AC-2: Given a nested menu tree deeper than five menu levels, when the context menu renders, then children below level 5 are stripped before rendering, any item that becomes actionless and childless after stripping is omitted, and the unsupported depth remains traceable through a debug or developer-observable path.
- [ ] AC-3: Given a menu item whose children are all hidden, empty, or separator-only, when the menu is built, then that item is omitted from the rendered menu.
- [ ] AC-4: Given separators around hidden or omitted items, when the final menu is rendered, then separators do not appear as leading, trailing, duplicated, or empty-group separators.

### FR-ARCH-002 — Context menu viewport containment

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | high |
| Tags | context-menu, viewport, desktop |
| Risk | medium |
| Stability | stable |
| Verification Method | Unit tests for geometry helper boundary cases and Playwright checks near right, left, top, and bottom viewport edges. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

The shared context menu component shall position the root menu and every child menu so that visible menu content remains within the viewport. If a menu cannot fit in its preferred direction, it shall flip or clamp the position. If content is taller than the available viewport space, the menu shall provide internal scrolling instead of extending outside the screen.

#### Rationale

Nested context menus are unusable if they can leave the screen, especially near viewport edges or with long registered item lists.

#### Acceptance Criteria

- [ ] AC-1: Given the root menu is opened near the right edge of the viewport, when it renders, then it is shifted left enough to remain visible inside the viewport margin.
- [ ] AC-2: Given a child menu would overflow to the right of its parent, when it opens, then it is rendered to the left of the parent item when there is enough space.
- [ ] AC-3: Given a menu would overflow left, top, or bottom after flipping, when it renders, then it is clamped inside the viewport margin instead of remaining off-screen.
- [ ] AC-4: Given a menu's content is taller than the available viewport height, when it renders, then the menu has a bounded max height and an internal vertical scroll area.
- [ ] AC-5: Given menu labels are longer than the available menu width, when they render, then labels do not overlap shortcuts, arrows, icons, or adjacent content.

### FR-ARCH-003 — Mobile context menu dialog navigation

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | high |
| Tags | context-menu, mobile, dialog |
| Risk | medium |
| Stability | stable |
| Verification Method | Playwright mobile coverage for dialog rendering, path updates, dialog back control, and popstate/back navigation. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

When the context menu is opened on a mobile viewport, the system shall show a modal-style dialog containing the current menu level. Items with children shall show a triangular child indicator and shall navigate into the child list when selected. The dialog shall show the current path and provide back navigation for nested levels.

#### Rationale

Hover-based submenu behavior does not work reliably on mobile, so the same menu tree needs an explicit dialog navigation contract.

#### Acceptance Criteria

- [ ] AC-1: Given a mobile viewport, when the terminal context menu opens, then the menu is rendered as a dialog instead of a position-based floating menu.
- [ ] AC-2: Given the mobile menu is at the root level, when it renders, then the header shows the current path as `메뉴`.
- [ ] AC-3: Given the user navigates into a category such as command items, when the child list renders, then the header path shows `메뉴 > 커맨드 라인` or the corresponding category path.
- [ ] AC-4: Given the mobile menu is inside a child level, when the user presses the dialog `뒤로가기` control, then the dialog returns to the previous menu level without closing the whole menu.
- [ ] AC-5: Given an Android browser back action or equivalent popstate occurs inside a child level, when it is handled, then the dialog returns one menu level; at the root level, the same action closes the dialog.
- [ ] AC-6: Given an item has no children, when the user selects it in mobile dialog mode, then its action runs once and the dialog closes.

### FR-ARCH-004 — Terminal registered item paste menu

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | high |
| Tags | terminal, context-menu, command-presets, paste |
| Risk | medium |
| Stability | stable |
| Verification Method | Unit tests for paste payload construction and Playwright terminal tests proving selected values are inserted without command execution. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

When one or more registered command preset items exist, the terminal context menu shall include a bottom-level `등록 항목 붙여넣기` menu. That menu shall group visible registered items into `커맨드 라인`, `디렉토리`, and `프롬프트` categories, and selecting a registered item shall paste the stored value into the target terminal session through a paste-safe input path without executing it. Command and directory registered items are single-line paste actions and shall reject stored values containing carriage return, line feed, NUL, or other submit/control characters rather than sending them to the terminal. Prompt registered items may contain multiple lines only when the system can deliver them through a paste-safe path that does not translate payload newlines into synthetic Enter key actions.

#### Rationale

Users need fast access to stored command lines, directories, and prompts from the terminal surface while preserving the safety requirement that menu selection does not execute terminal input.

#### Acceptance Criteria

- [ ] AC-1: Given at least one registered item exists, when the user opens a terminal context menu, then the bottom of the menu contains `등록 항목 붙여넣기` after the built-in terminal actions.
- [ ] AC-2: Given registered items exist for multiple kinds, when `등록 항목 붙여넣기` opens, then it shows only the non-empty categories among `커맨드 라인`, `디렉토리`, and `프롬프트`.
- [ ] AC-3: Given a category contains registered items, when the category submenu opens, then items are ordered by that kind's persisted `sortOrder` value.
- [ ] AC-4: Given the user selects a registered command item whose stored value contains no carriage return, line feed, NUL, or submit/control character, when the action runs, then exactly the stored command value is pasted to the target terminal input with no automatically appended Enter, carriage return, newline, or shell formatting.
- [ ] AC-5: Given the user selects a registered directory item whose stored value contains no carriage return, line feed, NUL, or submit/control character, when the action runs, then exactly the stored directory value is pasted to the target terminal input with no automatically generated `cd` command and no appended Enter.
- [ ] AC-6: Given the user selects a registered prompt item, when the action runs, then exactly the stored prompt value is pasted through a paste-safe path with no extra trailing control character; if the stored prompt contains multiline or control content that cannot be delivered without synthetic Enter submission, then the paste is rejected with an observable diagnostic and no terminal input is sent.
- [ ] AC-7: Given a registered item is selected, when the value is pasted, then the target terminal remains focused or focus is restored to the target terminal after the menu closes.

### FR-ARCH-005 — Registered item menu visibility and freshness

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | medium |
| Tags | terminal, context-menu, command-presets, visibility |
| Risk | medium |
| Stability | stable |
| Verification Method | Unit tests for registered item menu construction and Playwright tests for empty, single-kind, and multi-kind registered item states. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

The terminal context menu shall show registered item branches only when they contain at least one registered item available to paste. The system shall not show placeholder, disabled, or empty category branches for missing registered items. The registered item list shall be refreshed when the terminal context menu opens, and the opened menu shall use a stable snapshot until it is closed.

#### Rationale

Empty branches add noise and make the paste action harder to discover; predictable refresh behavior avoids mid-menu mutation and stale menus.

#### Acceptance Criteria

- [ ] AC-1: Given no command, directory, or prompt registered items exist, when the terminal context menu opens, then `등록 항목 붙여넣기` is not rendered at all.
- [ ] AC-2: Given only command registered items exist, when the registered item menu is opened, then `커맨드 라인` is shown and `디렉토리` and `프롬프트` are not shown.
- [ ] AC-3: Given only directory registered items exist, when the registered item menu is opened, then `디렉토리` is shown and empty categories are not shown.
- [ ] AC-4: Given only prompt registered items exist, when the registered item menu is opened, then `프롬프트` is shown and empty categories are not shown.
- [ ] AC-5: Given registered items changed after a previous context menu use, when the user opens a new terminal context menu, then the menu is based on the latest successfully loaded registered item list.
- [ ] AC-6: Given a registered item list is loaded and the menu is open, when the backing data changes elsewhere, then the currently open menu remains stable until it is closed and reopened.

### FR-ARCH-006 — Registered item paste error observability

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.3step1 |
| Status | planned |
| Priority | medium |
| Tags | terminal, context-menu, command-presets, observability |
| Risk | medium |
| Stability | stable |
| Verification Method | Unit tests for failure modes and Playwright or debug-capture tests proving no unintended Enter/execute side effect occurs after failures. |
| GitHub Issue | - |
| Related Docs | - |

#### Requirement

When the registered item list cannot be loaded or a paste action cannot be delivered to the target terminal, the system shall keep the error observable through a user-visible message, debug capture, log, or test-observable diagnostic. The system shall not silently fall back to executing a command or to another registered item kind.

#### Rationale

Project coding rules require meaningful protocol, state, validation, and lifecycle errors to stay traceable, and this feature directly affects terminal input safety.

#### Acceptance Criteria

- [ ] AC-1: Given loading registered items fails, when the terminal context menu opens, then the system does not show stale or fabricated registered item branches as if they were current.
- [ ] AC-2: Given loading registered items fails and previous registered item data exists in memory, when the menu is opened, then any use of cached data is explicitly observable or the registered item menu is omitted.
- [ ] AC-3: Given a registered item is selected but the target terminal cannot accept input, when the paste action fails, then the failure is observable through a user-visible or debug-observable path.
- [ ] AC-4: Given a paste payload is unsupported or invalid, when the user selects the item, then the system does not coerce it into a different command, directory, prompt, or shell execution path.
- [ ] AC-5: Given a failed paste action occurs, when the context menu closes, then no unintended terminal Enter or command execution side effect is produced.
