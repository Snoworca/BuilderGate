## 1. Scope Overview

This scope specifies bounded runtime resource limits, browser input backpressure, visible terminal output scheduling, snapshot storage budgets, terminal runtime residency, and split WebSocket transport behavior for BuilderGate terminal stability.

## 2. Scope Boundaries

### In Scope

- Runtime-configured caps for WebSocket, terminal output/input, snapshot storage, and terminal runtime residency.
- Browser WebSocket send-buffer backpressure for user input.
- Visible terminal output scheduling and authoritative recovery after overflow.
- Snapshot localStorage budget and tombstone TTL handling.
- Hidden terminal runtime LRU residency without deleting persistent workspace/tab state.
- Feature-flagged split WebSocket handshake, routing, fallback, and rollback-safe behavior.
- Non-2002 validation requirements for agent-run live checks.

### Out of Scope

- Making split WebSocket the default transport.
- Replacing xterm.js or the terminal rendering engine.
- Changing AI TUI idle/running semantic invariants.
- Using `https://localhost:2002` for agent-run validation.
- Terminating all `node.exe` processes during validation.

## 3. Assumptions and Constraints

- The default WebSocket transport mode remains `unified`.
- Split WebSocket is a feature-flagged transport mode and must remain rollback-safe.
- Browser main-thread rendering stalls are mitigated by visible output scheduling; split WebSocket alone is not sufficient.
- Agent-run live validation must use `https://localhost:2222`; health checks use `http://localhost:2221/health`.
- Raw terminal input must not be written to debug logs when backpressure or reconnect handling occurs.

#### Requirement

The browser client shall gate terminal input WebSocket messages by observing WebSocket.bufferedAmount, payload byte size, and runtime client WebSocket limits. Control and recovery messages shall continue to bypass input backpressure to avoid replay or repair deadlocks.

#### Rationale

Keyboard input delay can grow when terminal input keeps entering an already backed-up browser send queue. Input-only gating limits memory growth without blocking snapshot, repair, or subscription recovery traffic.

#### Acceptance Criteria

- [ ] AC-1: Given a terminal input payload is below configured WebSocket thresholds, when send() is called, then the payload is sent normally.
- [ ] AC-2: Given bufferedAmount or bufferedAmount plus payloadBytes exceeds the soft input threshold, when an input message is sent, then the send result is client-backpressure and the input enters a bounded retry path.
- [ ] AC-3: Given bufferedAmount exceeds the hard reconnect threshold, when an input message is sent, then the send result is client-hard-backpressure and a reconnect path is requested.
- [ ] AC-4: Given a control, subscribe, screen snapshot ACK, or screen repair ACK message is sent while the browser send buffer is high, then that control message is not blocked by input backpressure.
- [ ] AC-5: Given the retry path observes bufferedAmount below the low-water threshold and the queued input is sent, then the input barrier is cleared through the normal transport state path.
- [ ] AC-6: Given a backpressured input is queued or rejected, then raw input content is not written to debug output.

#### Requirement

Visible terminal output shall be written through a bounded scheduler that applies chunk and byte limits, preserves callback order, avoids splitting code points, and requests authoritative recovery when pending local output exceeds configured limits.

#### Rationale

Writing unbounded live output directly to xterm can monopolize the browser main thread and make keyboard input feel progressively slower. The scheduler bounds local work and shifts recovery to server-authoritative snapshot or repair paths.

#### Acceptance Criteria

- [ ] AC-1: Given visible terminal output is enqueued under the configured budget, when the scheduler drains, then it writes output in bounded chunks and preserves callback order.
- [ ] AC-2: Given output contains multibyte characters, when the scheduler splits work, then it does not split code points.
- [ ] AC-3: Given queued visible output exceeds configured chunk or byte limits, when enqueue runs, then not-yet-written local output is bounded and an overflow or recovery result is observable.
- [ ] AC-4: Given visible overflow marks a terminal stale, when authoritative snapshot or repair recovery completes, then the stale output state can be cleared without retaining an unbounded local queue.
- [ ] AC-5: Given the scheduler is reset, then stale callbacks from canceled writes do not mutate the new terminal state.

#### Requirement

Terminal snapshot persistence shall enforce runtime-configured per-snapshot size, total storage budget, maximum entry count, and removal tombstone TTL so localStorage does not grow or remain blocked indefinitely during long-running sessions.

#### Rationale

Runtime eviction and terminal remount increase snapshot usage. Snapshot storage must therefore be bounded before runtime residency eviction can safely increase restore frequency.

#### Acceptance Criteria

- [ ] AC-1: Given a snapshot exceeds the per-snapshot character limit, when save or restore parsing runs, then the oversized snapshot is rejected without expanding storage.
- [ ] AC-2: Given total snapshot storage exceeds the configured budget, when quota recovery or eviction runs, then older or corrupt entries are evicted until the budget is satisfied.
- [ ] AC-3: Given the number of snapshot entries exceeds maxEntries, when eviction runs, then older eligible entries are removed even if total character budget is not exceeded.
- [ ] AC-4: Given a removal tombstone is fresh, when a save for that session is attempted, then the tombstone blocks save for that current removal intent.
- [ ] AC-5: Given a removal tombstone is expired, legacy, malformed, or missing a valid timestamp, when snapshot storage is checked, then it does not permanently block future snapshots.

#### Requirement

BuilderGate shall limit mounted live terminal runtimes with an LRU residency policy that always pins visible terminals, excludes disconnected tabs, respects hidden runtime TTL, and applies configured maximum live workspace and terminal counts without deleting persistent workspace or tab state.

#### Rationale

Long-running sessions can accumulate xterm instances and terminal runtime objects even after the user stops viewing them. Residency limits reduce memory growth while preserving the user's saved workspaces and tabs.

#### Acceptance Criteria

- [ ] AC-1: Given a terminal tab is visible in the active view or desktop grid, when residency is computed, then that terminal remains resident even if caps are exceeded.
- [ ] AC-2: Given hidden terminals exceed maxLiveTerminals, when residency is computed, then the oldest eligible hidden terminals are excluded from the runtime layer.
- [ ] AC-3: Given hidden workspaces exceed maxLiveWorkspaces, when residency is computed, then hidden workspace runtimes are excluded while the active workspace remains resident.
- [ ] AC-4: Given a hidden terminal has not exceeded hiddenRuntimeTtlMs, when residency is computed, then it remains resident until the TTL expires.
- [ ] AC-5: Given a tab is disconnected, removed, or restarted, when residency metadata is updated, then stale runtime metadata does not keep that tab resident.

#### Requirement

The WebSocket transport shall support unified, split-shadow, and split modes. Split control connections shall receive logical group metadata and a short-lived pair token, while output connections shall be accepted only when the pair token and authenticated identity match the live control group.

#### Rationale

Splitting control/input from output/replay can reduce the chance that output backlog blocks input/control traffic, but the rollout must be feature-flagged, authenticated, and rollback-safe.

#### Acceptance Criteria

- [ ] AC-1: Given transport mode is unified, when a legacy WebSocket connects, then the connection stays compatible with the existing single-socket protocol.

- [ ] AC-3: Given an output socket presents a wrong pair token, wrong authenticated identity, missing pair fields, or an expired token, then the server rejects the output connection.
- [ ] AC-4: Given a duplicate output socket connects for the same group, then the previous output socket is closed and removed before the new output socket is used.
- [ ] AC-5: Given mode is split-shadow, then output socket connection lifecycle is exercised without moving terminal payload traffic away from the control path.

#### Requirement

In full split mode, terminal output, replay snapshot payloads, and screen repair payloads shall route through the paired output socket when it is open. Input, resize, subscribe, unsubscribe, ACK, ping, and repair request messages shall remain on the control socket. If the output socket is missing, closed, not open, over hard backpressure, over queue limit, or fails send, terminal payloads shall fall back to the control socket without dropping current, queued, replay-pending, or screen-repair-pending terminal output.

#### Rationale

Splitting sockets only helps input latency if the output lane can fail safely. Output failure must preserve terminal continuity and must not create a new memory leak or replay deadlock.

#### Acceptance Criteria

- [ ] AC-1: Given full split mode with a paired output socket, when terminal output, replay snapshot, or screen repair payload is sent, then the payload is sent through the output socket.
- [ ] AC-2: Given full split mode, when input, resize, subscribe, unsubscribe, replay ACK, repair ACK, repair failed, repair request, or ping is sent, then it stays on the control socket.
- [ ] AC-3: Given the output socket is absent or closes, when a terminal payload is routed, then the control socket is used as fallback.
- [ ] AC-4: Given output socket queue overflow, hard limit, not-open, or send-error occurs, then queued and current terminal payloads are rerouted to the control socket without unbounded accumulation.
- [ ] AC-5: Given an output socket closes before a replay snapshot ACK, then full split mode sends a fresh recovery snapshot on control and preserves pending output behind fallback or degraded recovery snapshots.
- [ ] AC-6: Given output failure occurs while screen repair output is pending, then fallback or degraded recovery snapshot ACK flushes that pending output on control.
- [ ] AC-7: Given split-shadow mode and the output socket closes, then control replay recovery is not forced.

| Type | functional |

| Status | planned |
| Priority | high |
| Tags | buildergate, stability, conpty, frontend-runtime, xterm, recovery, input, feasibility:high |

#### Requirement

The frontend terminal runtime shall apply snapshots, screen repairs, and ACK-pending recovery tail output through a bounded recovery write gate instead of large direct xterm writes, and shall release queued input only after recovery writes drain or an observable configured timeout is reached.

#### Rationale

Live visible output already uses a scheduler, but the ConPTY plan found that snapshot restore, snapshot replacement, and screen repair can still write large payloads directly to xterm. That can block the browser main thread and make input appear unavailable during heavy AI TUI redraws.

#### Acceptance Criteria

- [ ] AC-1: Given local snapshot restore, authoritative snapshot replacement, fallback snapshot replacement, or screen repair applies terminal content, when the frontend writes that content to xterm, then the write goes through a bounded recovery scheduler rather than a large direct term.write call.
- [ ] AC-2: Given the client has acknowledged a snapshot and receives output before session readiness, when that output belongs to the recovery tail, then it is routed through the recovery write gate rather than the live visible output lane.
- [ ] AC-3: Given session readiness arrives while the recovery write gate still has pending work, then queued input is not released to the PTY until the recovery gate drains or a configured recovery release timeout emits an observable release event.
