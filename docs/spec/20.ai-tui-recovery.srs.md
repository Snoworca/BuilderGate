## 1. Scope Overview

This scope specifies desktop Tools menu recovery options for interactive AI TUI commands, including persisted recovery configuration, command matching, terminal/tab icon metadata, and restart or orphan-tab resume execution.

## 2. Scope Boundaries

### In Scope

- Desktop Tools menu entry for managing AI TUI recovery options.
- Recovery option persistence, validation, default seeding, deletion, and ordering.
- Matching submitted terminal commands against configured recovery options.
- Displaying configured recovery icons next to terminal/tab names.
- Restoring shell type, last working directory, and configured recovery command arguments after tab restart or server orphan-tab recovery.
- Claude, Codex, and user-created custom command recovery option contracts.

### Out of Scope

- Mobile entry points for recovery option management.
- Reusing terminal context-menu registered item paste behavior as an execution path.
- Automatically enabling dangerous Claude permission flags by default.
- Guaranteeing PowerShell profile-only aliases unless the user registers the alias as a custom recovery option and the shell can resolve it at restore time.

## 3. Assumptions and Constraints

- Recovery option management is available only through the desktop Tools menu for this target.
- The default seed set contains Claude and Codex recovery options only.
- Custom commands such as `claudep` are user-created recovery options, not built-in defaults.
- Recovery execution is automatic only after an enabled configured option matches the previously submitted command.
- Command and argument validation must reject control characters that could alter terminal submission semantics.

#### Requirement

The backend shall persist recovery options in a dedicated recovery option store separate from command presets and terminal shortcuts. Each recovery option shall have a required command, an ordered arguments array appended to the command during restore execution, an enabled state, optional icon metadata, sort order, timestamps, and a stable id. The service shall normalize commands for matching and uniqueness and shall seed default options only when the recovery option store is first created.

#### Rationale

The feasibility report found that CommandPresetService and TerminalShortcutService are not the recovery option domain and that only their persistence patterns should be reused.

#### Acceptance Criteria

- [ ] AC-1: Given the recovery option store does not exist, when the service initializes for the first time, then it creates default enabled options for `claude` with arguments [`--continue`] and `codex` with arguments [`resume`, `--last`].
- [ ] AC-2: Given the user deletes a default Claude or Codex option, when the server restarts, then the missing default is not recreated by the seed process.
- [ ] AC-3: Given two options normalize to the same executable token, when the second option is saved, then the server rejects it as a duplicate command.
- [ ] AC-4: Given a command or argument contains NUL, carriage return, line feed, or another forbidden control character, when the payload is saved, then the server rejects it and does not modify the persisted store.
- [ ] AC-5: Given the persisted store contains invalid rows, when the service initializes, then invalid rows are omitted or quarantined through an observable diagnostic and valid rows remain available.
- [ ] AC-6: Given multiple recovery option mutations run concurrently, when they complete, then the persisted JSON remains valid and each successful mutation is reflected exactly once.

#### Requirement

Because recovery options can automatically execute terminal commands after restart or orphan-tab recovery, the system shall protect recovery option APIs with authentication, persist only validated command and argument fields, construct restore input with shell-specific escaping, and keep recovery failures observable. Dangerous Claude permission flags shall never be included in the built-in defaults and shall execute only when the user explicitly stores them in a recovery option.

#### Rationale

The feature intentionally auto-executes configured commands on recovery, so safe defaults, explicit dangerous flag opt-in, and traceable failures are required.

#### Acceptance Criteria

- [ ] AC-1: Given an unauthenticated request targets recovery option CRUD or reorder APIs, when the request reaches the server, then it is rejected by the same authentication boundary used for existing protected tool configuration APIs.
- [ ] AC-2: Given a recovery option command or argument contains a control character that can submit or alter terminal input, when validation runs, then the option is rejected and the previous persisted store remains unchanged.
- [ ] AC-3: Given restore execution constructs a command line from command and arguments, when the configured shell is PowerShell, cmd, bash, zsh, or sh, then each argument is escaped or quoted according to that shell before writing to the PTY.
- [ ] AC-4: Given the built-in default Claude option is created, when the default arguments are inspected, then `--dangerously-skip-permissions` and `--permission-mode bypassPermissions` are absent.
- [ ] AC-5: Given a user-created option includes `--dangerously-skip-permissions` or `--permission-mode bypassPermissions`, when that option is matched during restore, then the dangerous flag is sent only because it was explicitly stored in that option.
- [ ] AC-6: Given restore execution fails because a command is missing, disabled, deleted, rejected by validation, or cannot be written to the PTY, when the failure occurs, then the failure is recorded through a server log, debug event, or test-observable diagnostic.

#### Requirement

The recovery option service shall protect persisted recovery option data against interrupted writes and corrupted primary store files by using the repository's established temporary file and backup recovery pattern. When the primary recovery option store cannot be loaded, the service shall attempt recovery from a valid backup before falling back to an empty or default-created store, and unrecoverable store failures shall remain observable.

#### Rationale

The feasibility report explicitly listed .tmp/.bak recovery as required test coverage for the new RecoveryOptionService persistence path.

#### Acceptance Criteria

- [ ] AC-1: Given a recovery option mutation is persisted, when the write completes successfully, then the primary recovery option store is valid JSON and no stale temporary file is used as the source of truth.
- [ ] AC-2: Given the primary recovery option store is missing or corrupted and a valid backup store exists, when the service initializes, then it restores or loads the valid backup before seeding defaults.
- [ ] AC-3: Given the primary store is corrupted and no valid backup exists, when the service initializes, then it records an observable diagnostic and creates only the first-run default store if the service can prove no user-created valid store is recoverable.
- [ ] AC-4: Given a stale temporary recovery option file remains after an interrupted write, when the service initializes, then the temporary file does not override a valid primary or backup store.
- [ ] AC-5: Given backup recovery succeeds, when the recovery option list is returned through the API, then user-created options from the recovered backup are present and default seed-once rules are not reapplied as if the store were new.
