# MCP Session Orchestration

| Field | Value |
|---|---|

## 1. Scope Overview

This scope specifies the BuilderGate MCP control-plane for exposing terminal sessions, agent orchestration, webhook-triggered assignments, alias management, and safe cross-session input delivery to MCP-connected coding agents.

## 2. Scope Boundaries

### In Scope

- Streamable HTTP MCP listener configuration, security guards, tool discovery, tool calls, and listener rebind behavior.
- Session identity, stable session addressing, alias list/search, status update, message delivery, reply, and close workflows for terminal sessions.
- Agent command profiles, MCP-launched follower sessions, launch-time capability binding, kickoff prompt delivery, and launch compensation.
- Webhook invocation through random query keys or header keys, assignment creation, source session `"0"`, prompt limits, and redaction.
- Header Tools menu dialog surfaces for MCP server, security, agent profiles, webhooks, and live sessions.
- Audit logging, token/key separation, scope checks, observable rejection errors, and regression/E2E verification for MCP control-plane behavior.

### Out of Scope

- Replacing the terminal rendering engine, xterm.js, or existing workspace grid layout mechanics.
- Making split WebSocket the default transport or changing existing WebSocket protocol compatibility.
- Using browser JWTs as MCP or webhook actor credentials.
- Treating runtime `sessionId` as a permission credential.
- Broad OS process termination such as killing all `node.exe` processes during validation or cleanup.
- Guaranteeing automatic MCP client configuration for every external coding agent when that agent only supports manual configuration.

## 3. Assumptions and Constraints

- The active target is `0.5.5-buildergate-stability`.
- The default MCP HTTP endpoint is loopback-only.
- External MCP access requires explicit whitelist mode plus direct TLS or a trusted TLS reverse proxy.
- MCP input, WebSocket input, and restore input must converge on a shared input policy before MCP message delivery becomes available.
- Existing `WorkspaceTab.name` with `nameSource='user'` is the primary user alias source.
- `SessionManager` continues to own PTY spawn/write/terminate/finalization, and `WorkspaceService` continues to own tab lifecycle.
- Playwright/manual validation must target `https://localhost:2222`; `http://localhost:2221` is health/redirect only.

#### Requirement

The system shall run the BuilderGate MCP Streamable HTTP endpoint on an independent MCP listener that is loopback-only by default, permits external access only through explicit IP or CIDR whitelist mode with direct TLS or a trusted TLS reverse proxy, validates effective client IP, Origin, authentication, and actor scope before every MCP or webhook request, and rebinds MCP host or port by swapping only the MCP listener with rollback on failure instead of restarting the BuilderGate app process.

#### Rationale

The research identifies the MCP endpoint as a remote control surface for terminal input, session creation, and session termination. Keeping the listener independent, closed by default, and guarded by explicit TLS, whitelist, Origin, and scoped capability checks prevents accidental external exposure and keeps port changes from destabilizing the main app server.

#### Acceptance Criteria

- [ ] AC-1: Given MCP is enabled with default configuration, when the listener starts, then it binds to 127.0.0.1 and rejects non-loopback remote addresses before MCP tool dispatch.
- [ ] AC-2: Given whitelist mode is requested without direct TLS or trusted TLS proxy transportSecurity, when the config is validated, then the config is rejected and the existing MCP listener remains unchanged.
- [ ] AC-3: Given trusted TLS proxy mode is enabled, when a request arrives, then forwarded client IP headers are trusted only if the immediate remote address is in trustedProxies and the forwarded protocol is HTTPS; otherwise the request is rejected before whitelist matching.
- [ ] AC-4: Given an MCP or webhook request has a remote address, effective client IP, Origin, or token that fails policy, then the response contains an observable error code and audit id without disclosing token, key, or raw prompt material.
- [ ] AC-5: Given the MCP port or bind host changes, when rebind succeeds, then a candidate listener is health-probed, swapped active, and the old listener is drained without restarting the app HTTPS server or redirect server.
- [ ] AC-6: Given MCP rebind fails during validation, candidate bind, health probe, or persistence, then the previous listener and persisted config remain active and the failure is visible through MCP control status and audit.

### IR-MCP-001 — BuilderGate MCP tool surface contract

| Field | Value |
| --- | --- |
| Type | interface |
| Target | 0.5.5-buildergate-stability |
| Status | planned |
| Priority | critical |
| Tags | mcp, tools, json-rpc, interface, feasibility:high |
| Risk | high |
| Stability | evolving |
| Verification Method | MCP tools/list schema tests, tools/call validation tests, claim-code reuse tests, and Streamable HTTP integration tests. |
| GitHub Issue | - |
| Related Docs | docs/research/mcp/00.index.md, docs/research/mcp/03.mcp-http-webhook-interface-design.md |

#### Requirement

The system shall expose BuilderGate session orchestration through MCP tool discovery and tool calls using stable ASCII tool names for session identity, claim, list, search, message delivery, alias update, agent launch, close, self-close, reply-to-leader, and status update, while keeping Korean session aliases and prompts as input or output data rather than tool names.

#### Rationale

The research separates natural-language agent intent from the MCP tool surface. Stable tool names and explicit schemas let Codex, Claude Code, and other clients perform the same search, send, launch, reply, and close workflows without relying on localized command names.

#### Acceptance Criteria

- [ ] AC-1: Given an MCP client calls tools/list, then the response includes buildergate.session.whoami, buildergate.session.claim, buildergate.session.list, buildergate.session.search, buildergate.message.send, buildergate.session.set_alias, buildergate.session.open_agent, buildergate.session.close, buildergate.session.close_self, buildergate.message.reply_to_leader, and buildergate.session.update_status with JSON input schemas.
- [ ] AC-2: Given a session-bound actor calls buildergate.session.whoami, then the result includes sessionId, sessionKey, workspaceId, tabId, alias, agentKind, leaderSessionKey, bindingLifecycle, and agentStatus.
- [ ] AC-3: Given a legacy or manually configured session lacks a session-bound token, when Streamable HTTP initialization presents a valid one-time claim code as its Bearer bootstrap credential and that transport session calls buildergate.session.claim with the same code, then the code must be unexpired, bound to an existing live session, atomically consumed once, and exchanged for a registered session-bound actor token. Claim codes expire after five minutes, their server-side collection is pruned and limited to 256 records, and reuse, expiration, or a non-live binding is rejected with a stable denial code. Every subsequent POST carrying mcp-session-id must also present the exact Bearer used at initialization; the claim bootstrap transport continues presenting its original claim code after binding while the server authorizes through the internally minted actor token. Missing, mismatched, or browser-JWT credentials are rejected, an unbound bootstrap session may discover or call only the claim flow plus protocol initialization and ping operations, and every session capability is revalidated against its current live binding on each request. Once an authorized close begins, all registered actor tokens for that session are revoked even if later tab or process cleanup reports failure.
- [ ] AC-4: Given a tool call fails validation or policy, then the MCP response maps the failure to a stable observable error code rather than silently falling back to a different tool or target.
- [ ] AC-5: Given the MCP implementation exchanges JSON-RPC messages over Streamable HTTP, then request and response bodies are UTF-8 JSON and tool names remain ASCII even when aliases such as 빌더 게이트 or 클로드 are used in arguments.

### FR-MCP-001 — Stable session identity, alias listing, search, and alias updates

| Field | Value |
| --- | --- |
| Type | functional |

| Status | planned |
| Priority | critical |
| Tags | mcp, session-registry, session-key, alias, search, feasibility:high |
| Risk | high |
| Stability | evolving |
| Verification Method | Server registry synchronization tests, stale sessionId tests, alias ranking tests, tab rename propagation tests, and MCP list/search integration tests. |
| GitHub Issue | - |
| Related Docs | docs/research/mcp/01.requirements-and-domain-model.md, docs/research/mcp/02.current-architecture-inventory.md, docs/research/mcp/03.mcp-http-webhook-interface-design.md, docs/research/mcp/05.frontend-tools-dialog-and-alias-ux.md |

#### Requirement

The system shall maintain an MCP session registry that maps each stable sessionKey to the current runtime UUID sessionId, workspaceId, tabId, generation, alias, leaderSessionKey, bindingLifecycle, MCP connection state, and agentStatus, and shall use that registry to implement whoami, session list, session search, stale sessionId rejection, and alias updates based on WorkspaceTab.name with nameSource set to user.

#### Rationale

The current runtime sessionId changes on restart and orphan recovery. The research therefore requires a stable sessionKey and generation guard so long-running assignments, search results, and leader or follower relationships do not silently target stale sessions. Reusing WorkspaceTab.name preserves the user-visible alias source instead of introducing a second naming model.

#### Acceptance Criteria

- [ ] AC-1: Given a runtime terminal session is created, when the MCP registry binds it, then sessionId is a UUID generated by SessionManager and sessionKey remains stable for the tab across restart and orphan recovery.
- [ ] AC-2: Given a tab restart or orphan recovery replaces WorkspaceTab.sessionId, when registry reconciliation runs, then currentSessionId and generation are updated while the previous sessionId becomes stale for MCP targeting.
- [ ] AC-3: Given a tool call targets a sessionId that does not match the current binding generation, then the call fails with STALE_SESSION_ID and returns enough non-secret binding context for the caller to re-search or retarget.
- [ ] AC-4: Given buildergate.session.list is called with includeSelf true or false, then it returns live bindings with sessionId, sessionKey, alias, nameSource, workspaceId, tabId, agentKind, agentStatus, mcpConnected, and leaderSessionKey according to the includeSelf option.
- [ ] AC-5: Given the user renames the lower status-bar session name or tab name, when WorkspaceTab.name is saved with nameSource user, then MCP list and exact alias search use that value as the primary alias.
- [ ] AC-6: Given buildergate.session.set_alias succeeds, then it updates the same WorkspaceTab.name and nameSource user path used by UI rename and broadcasts tab:updated to connected clients.
- [ ] AC-7: Given buildergate.session.search receives a query such as 빌더 게이트, then exact user alias match is ranked before terminal-title names, tab prefixes, agent aliases, cwd, or recovery-command matches; zero matches, ambiguous equal-rank matches, stopped bindings, and unauthorized targets fail with explicit candidate or reason data.

### FR-MCP-002 — Shared input gateway for MCP messages and leader replies

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.5-buildergate-stability |
| Status | planned |
| Priority | critical |
| Tags | mcp, input-gateway, message-send, reply, replay-safety, feasibility:high |
| Risk | high |
| Stability | evolving |
| Verification Method | Gateway unit tests, WebSocket input regression tests, restore input regression tests, message.send and reply_to_leader MCP integration tests, and AI TUI idle invariant tests. |
| GitHub Issue | - |
| Related Docs | docs/research/mcp/01.requirements-and-domain-model.md, docs/research/mcp/02.current-architecture-inventory.md, docs/research/mcp/04.session-orchestration-and-input-safety.md, docs/research/mcp/06.security-ops-test-plan.md |

#### Requirement

The system shall route WebSocket user input, restore input, MCP message delivery, agent-command injection, kickoff prompt injection, webhook delivery, and reply-to-leader delivery through a shared SessionInputGateway that enforces byte limits, live-session checks, replay and repair policy, hidden-terminal policy, paste-only versus submit policy, audit hooks, and the AI TUI idle invariant before calling the low-level PTY write path.

#### Rationale

The research identifies direct SessionManager.writeInput calls from multiple ingress paths as the main safety risk. MCP and webhook input must not bypass replay queues, newline policy, hidden-session policy, or the existing AI TUI status invariant.

#### Acceptance Criteria

- [ ] AC-1: Given WebSocket user input, restore input, buildergate.message.send, buildergate.message.reply_to_leader, open_agent command injection, kickoff, or webhook delivery is submitted, when input is accepted, then the final PTY write occurs through SessionInputGateway rather than each caller directly invoking SessionManager.writeInput.
- [ ] AC-2: Given replay or screen-repair state is pending for a target session, when MCP or webhook input is submitted, then the gateway queues or rejects according to the configured replayPolicy and returns INPUT_REJECTED_REPLAY_PENDING or equivalent observable status when rejected.
- [ ] AC-3: Given a message uses paste-only delivery, then the gateway writes text without synthetic Enter submission and rejects carriage return, line feed, or submit semantics unless the actor has mcp:message.submit scope and the request explicitly sets submit true.
- [ ] AC-4: Given a hidden, closed, retired, stale-generation, or non-live target is selected, then the gateway rejects the delivery with a stable error code and does not call the PTY write path.
- [ ] AC-5: Given an AI TUI session receives user typing, local echo, prompt redraw, cursor movement, ticker output, or waiting-input repaint, then the shared input path preserves the existing invariant that those events do not transition the session to running.
- [ ] AC-6: Given a follower calls buildergate.message.reply_to_leader, then the gateway resolves the leaderSessionKey to a live binding and reuses message delivery; if the leader is unavailable, the follower remains open and the error is returned.

### FR-MCP-003 — Agent command profiles and MCP-launched follower sessions

| Field | Value |
| --- | --- |
| Type | functional |
| Target | 0.5.5-buildergate-stability |
| Status | planned |
| Priority | critical |
| Tags | mcp, agent-profile, open-agent, leader-follower, kickoff, feasibility:high |
| Risk | high |
| Stability | evolving |
| Verification Method | Agent profile store tests, open_agent transaction tests for env/generated-file/manual modes, ready timeout tests, kickoff gating tests, update_status tests, and MCP launch integration tests. |
| GitHub Issue | - |

#### Requirement

The system shall provide agent command profiles for MCP-launched coding agents and shall implement buildergate.session.open_agent so a leader can create a follower workspace tab, inject the selected agent command with MCP endpoint, stable sessionKey, leaderSessionKey, and scoped capability binding available before PTY spawn, wait for readiness or explicit status update, and deliver the kickoff prompt only after the agent is ready.

#### Rationale

The research found that current WorkspaceService.addTab spawns the PTY immediately, so MCP capability tokens and environment metadata must be prepared before tab creation. Agent profiles must be separate from command presets and recovery options because they are launch contracts, not UI insertion shortcuts or recovery candidates.

#### Acceptance Criteria

- [ ] AC-1: Given an admin creates or edits an agent profile, then command, args, aliases, isDefault, enabled, kickoffPrompt, and mcpClientConfigMode are stored in a dedicated agent profile store rather than in command presets or recovery options.
- [ ] AC-2: Given buildergate.session.open_agent is called, then the launch flow preallocates sessionKey, draft binding, launch attempt, and scoped capability token before creating the workspace tab and PTY session.
- [ ] AC-3: Given a profile uses env or generated-file MCP client config mode, when the PTY process is spawned, then MCP URL, sessionKey, current sessionId, leaderSessionKey, and capability token or generated config path are available at spawn time rather than attached after tab creation.
- [ ] AC-4: Given automatic config is not supported for a profile, when manual mode is used, then the response exposes a one-time claim code that is configured as the Streamable HTTP initialization Bearer credential, and the resulting claim-only transport session can bind by calling buildergate.session.claim with that same code before using session-bound tools.
- [ ] AC-5: Given an agent command is injected, then the injection uses SessionInputGateway with purpose agent-command and does not bypass the same enter, replay, and audit policy as other MCP input.
- [ ] AC-6: Given agent readiness is detected by buildergate.session.update_status or an approved foreground detector, then kickoffPrompt is delivered as a kickoff assignment through SessionInputGateway; if readiness times out, kickoff is not pasted automatically and the response reports kickoffPending true.
- [ ] AC-7: Given buildergate.session.update_status is called by a bound actor, then agentKind, agentStatus, detail, lastSeenAt, and MCP connected state update the live registry and are visible in list and search results.

- [ ] AC-3: Given agent command injection succeeds but readiness times out, then the session remains open, agentStatus remains unknown or starting, kickoffPending is true, and no automatic close or unsafe kickoff paste occurs.

- [ ] AC-5: Given buildergate.session.close_self is called without leaderSessionKey, then it fails with SELF_CLOSE_DENIED_NO_LEADER and does not close the user-created session.
- [ ] AC-6: Given buildergate.session.close_self is called by a follower with a live leaderSessionKey, then the tool returns an accepted close job before scheduling tab and PTY cleanup 250 to 1000 milliseconds after response flush.
- [ ] AC-7: Given any close path completes or fails, then registry bindingLifecycle, tab removal broadcast, process-tree cleanup evidence, and audit result remain consistent and traceable.
