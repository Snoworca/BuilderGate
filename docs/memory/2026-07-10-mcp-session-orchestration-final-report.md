# BuilderGate MCP 세션 오케스트레이션 최종 조사 보고서

## 1. 결론

요구한 기능은 구현 가능합니다. 현재 코드에는 다음 경로가 구현되어 있습니다.

- MCP 클라이언트가 자신의 UUID 세션 키와 현재 세션 ID를 조회한다.
- 다른 세션의 UUID 세션 키, 현재 세션 ID, 사용자 별칭, 상태를 목록 또는 검색으로 조회한다.
- 별칭, 세션 키, 현재 세션 ID를 사용해 다른 세션에 메시지를 붙여넣거나 Enter 제출까지 수행한다.
- Tools 메뉴의 `MCP 설정` 다이얼로그에서 보안, 에이전트 프로필, 웹훅, 세션, 감사/상태를 관리한다.
- HTTP MCP는 기본적으로 loopback에 바인딩하며, 외부 공개는 IP/CIDR 허용 목록과 TLS 조건을 함께 만족해야 한다.
- 고정 인증키를 다이얼로그에서 생성 또는 재생성하고 정확한 평문을 1회 복사할 수 있다.

## 2. 접속 및 인증 방식

기본 MCP URL은 설정을 변경하지 않았을 때 `http://127.0.0.1:3333/mcp`입니다. `http://localhost:3333`도 같은 호스트를 가리키지만, 실제 포트는 MCP 보안 탭의 저장값을 따릅니다.

고정 인증키는 MCP 보안 탭의 `고정 인증키 생성` 버튼에서 발급합니다. 서버는 평문 키를 저장하지 않고 SHA-256 해시만 저장합니다. 평문은 생성 또는 재생성 응답 직후에만 표시되며 `복사` 버튼으로 정확한 키를 복사합니다. 이후 MCP 요청은 다음 헤더를 사용합니다.

```http
Authorization: Bearer <고정 인증키 또는 세션 capability token>
```

수동 클라이언트는 세션 탭에서 발급한 일회성 연결 코드를 초기 Bearer로 사용하고, 같은 코드로 `buildergate.session.claim`을 호출해 세션에 결속합니다.

## 3. 인증 오류 원인

Claude Code에 나타난 `needs authentication`과 OAuth 오류는 BuilderGate가 OAuth discovery 서버가 아닌데 클라이언트가 인증 없는 `/mcp` 응답을 OAuth 흐름으로 해석하면서 발생했습니다. 과거 `/mcp`의 알 수 없는 요청이 JSON-RPC `Not found` 객체를 반환해 OAuth 오류 스키마와 맞지 않았고, 클라이언트는 이를 잘못된 OAuth 응답으로 표시했습니다.

현재 동작은 다음과 같이 정리되었습니다.

- 인증 없는 초기화는 `INVALID_TOKEN`으로 거부한다.
- 브라우저 관리자 JWT를 MCP 자격 증명으로 재사용하면 `CREDENTIAL_BOUNDARY_VIOLATION`으로 거부한다.
- 고정 인증키, 세션 capability token, 유효한 일회성 연결 코드만 MCP 경계에서 허용한다.
- `mcp-session-id`가 있어도 매 POST 요청에서 초기 Bearer를 정확히 다시 제시해야 한다.

## 4. 보안 설계

### 네트워크 경계

- 기본 바인드: `127.0.0.1`
- 외부 바인드: 명시적 IPv4/CIDR 허용 목록 필요
- `0.0.0.0/0` 금지
- 외부 허용 목록 모드: direct TLS 또는 신뢰 TLS 프록시 필요
- Origin, 신뢰 프록시, 전달 IP를 함께 검증

### 고정 인증키

- 32바이트 CSPRNG 기반
- `bgmcp_` 접두사
- SHA-256 해시만 영구 저장
- 동시 설정 저장 직렬화
- 재생성 즉시 이전 키와 기존 전송 세션 무효화
- 권한은 세션 목록, 검색, 붙여넣기, 제출로 제한

### 일회성 연결 코드

- 32바이트 CSPRNG 기반
- 5분 만료
- 최대 256개, 만료·사용 레코드 정리
- 실제 `live` 세션에만 사용 가능
- 1회 소비 후 재사용 금지
- 발급 토큰은 공용 토큰 저장소에 등록

### 세션 토큰

- 각 요청에서 현재 `live` 세션 바인딩을 재확인
- 인증된 닫기가 시작되면 삭제 성공 여부와 무관하게 세션 토큰 폐기
- MCP 관리 UI의 닫기도 동일한 수명주기 서비스를 사용
- 브라우저 JWT와 MCP capability 경계 분리

## 5. UI 구현 범위

상단 Tools 버튼의 컨텍스트 메뉴에 `MCP 설정` 항목이 있습니다. MCP 다이얼로그의 가시 라벨, 설명, 탭, 상태 값은 한글로 표시합니다.

보안 탭의 고정 인증키 흐름은 다음과 같습니다.

1. 키가 없으면 `고정 인증키 생성`을 표시한다.
2. 키가 있으면 `고정 인증키 재생성`을 표시한다.
3. 요청 전에 생성 또는 재생성 확인창을 표시한다.
4. 취소, 실패, 새로고침, 탭 이동, 다이얼로그 닫기 때 평문을 제거한다.
5. 지연 응답이 탭 이동이나 닫기 뒤 도착해도 평문을 다시 표시하지 않는다.
6. `복사`는 접두 설명이나 공백 없이 키 원문만 복사한다.

세션 전달 테스트 기본값은 정확히 `Hello, World!`이며 제출 모드는 문자열 뒤에 Enter를 추가해 `SessionInputGateway`로 전달합니다.

## 6. 포트 및 재기동 결과

운영 인스턴스는 PID `61568`이 TCP `2001`, `2002`, `2023`을 사용하고 있어 어떤 종료나 재기동도 수행하지 않았습니다.

테스트 인스턴스만 다음 절차로 재기동했습니다.

```text
stop.bat
start.bat --port 2222
```

`stop.bat`은 내부 종료 응답 타임아웃을 반환했지만 테스트 포트 `2221/2222`는 실제로 내려갔고 운영 PID는 유지됐습니다. 최종 테스트 서버는 PID `48304`로 기동됐으며 `https://localhost:2222/health`가 HTTP 200을 반환했습니다.

현재 테스트 서버의 MCP 저장 설정은 다음과 같습니다.

- `enabled: false`
- `port: 3333`
- `fixedAccessKeyConfigured: false`

따라서 현재 `2222` 테스트 인스턴스에서 별도 MCP 리스너는 열리지 않습니다. 기능을 실제 사용하려면 보안 탭에서 `MCP 엔드포인트 사용`을 켜고 고정 인증키를 생성한 뒤 저장해야 합니다.

과거 `MCP_PORT_REBIND_FAILED`의 원인은 테스트 MCP가 운영 프로세스가 이미 점유한 `2023`으로 재바인드하려 했기 때문입니다. 테스트 환경에서는 운영 포트를 사용하지 않아야 합니다.

## 8. 운영 및 롤백 주의사항

- 운영 PID와 `2001/2002/2023` 포트를 테스트 과정에서 건드리지 않는다.
- 테스트 서비스는 `start.bat --port 2222`, 종료는 `stop.bat`만 사용한다.
- `node.exe` 전체 종료 명령을 사용하지 않는다.
- 문제가 생기면 MCP 설정의 `enabled`를 끄거나 listener/control-plane 변경을 되돌릴 수 있다.
- 고정 인증키 재생성은 이전 키를 즉시 무효화하므로 기존 MCP 클라이언트의 Bearer를 함께 갱신해야 한다.
