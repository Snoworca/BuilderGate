# BuilderGate 프론트엔드 런타임 생명주기 장기 개선 계획

작성일: 2026-06-17

대상 범위: hidden terminal, live workspace cap, xterm write/snapshot, frontend WebSocket transport, localStorage cleanup

선행 문서: `docs/research/2026-06-17.buildergate-memory-input-lag-research.md`

## 1. 현재 문제 요약

BuilderGate의 장시간 사용 시 브라우저 메모리 증가와 입력 지연은 "활성 화면에 보이지 않는 런타임을 계속 live 상태로 유지하는 구조"가 핵심 원인이다. 선행 문서의 결론과 현재 코드 확인 결과, 프론트엔드에는 다음 누적 경로가 있다.

### 1.1 App.tsx의 workspace cap은 존재하지만 꺼져 있다

- `frontend/src/App.tsx:42`의 `MAX_ALIVE_WORKSPACES = 0`은 `0 = 제한 없음`이다.
- `aliveWorkspaceIds`와 `workspaceVisitOrder`는 LRU 의도를 담고 있지만, cap이 0이라 모든 workspace가 계속 alive 상태가 된다.
- `TerminalWorkspaceStage`에 전달되는 tabs와 tab mode의 `TerminalHostSlot` 렌더링이 모두 `MAX_ALIVE_WORKSPACES <= 0 || aliveWorkspaceIds.has(tab.workspaceId)` 조건을 따른다.
- `terminalRefsMap`은 `TerminalRuntimeLayer`에서 tab id별 ref를 추가하지만, workspace/tab 삭제 또는 cap eviction에 맞춘 명시적 pruning이 없다.

현재 상태는 "기능 플래그가 있는 무제한 live runtime"에 가깝다. 즉, cap을 설정값으로 노출하기 전에 lifecycle invariant와 pruning 정책을 먼저 명확히 해야 한다.

### 1.2 TerminalRuntimeLayer는 보이지 않는 런타임도 유지한다

- `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx`는 `tab.status !== 'disconnected'`인 tab마다 `TerminalContainer`를 유지한다.
- host가 없거나 보이지 않으면 runtime entry를 `PARKED_STYLE`로 화면 밖에 배치할 뿐, `TerminalContainer` 자체는 살아 있다.
- `TerminalContainer`는 `isVisible`을 prop으로 받지만, mount되어 있는 한 session subscription과 여러 timer/ref 상태를 유지한다.

이 구조는 workspace 전환 UX에는 유리하지만, 많은 workspace/tab을 오래 열어두면 xterm, addon, DOM listener, WebSocket subscription, snapshot debounce가 누적된다.

### 1.3 Hidden terminal도 output subscription과 xterm write를 계속 수행한다

- `TerminalContainer`의 `subscribeSession()` effect는 `isVisible`과 무관하게 열린다.
- `onOutput`은 `terminalRef.current?.write(data)`를 즉시 호출한다.
- `TerminalView.write()`는 restore pending이 아니면 `writeOutput(term, data)`로 바로 이어진다.
- `writeOutput()`은 chunk별로 `term.write()`를 호출하고, callback에서 snapshot 저장 debounce를 예약한다.

따라서 사용자가 보지 않는 tab/workspace도 output이 들어오면 xterm buffer와 snapshot 경로가 계속 동작한다. 출력량이 많은 hidden session 하나가 active terminal 입력 처리와 같은 브라우저 메인 스레드를 공유한다.

### 1.4 xterm output write scheduler가 없다

- 현재 `inFlightOutputRef`는 push/shift 배열일 뿐, queue depth, byte cap, frame budget, coalescing, latency telemetry로 쓰이지 않는다.
- `term.write()` callback이 늦어져도 다음 output을 제한하지 않는다.
- hidden 상태에서 들어온 output도 클라이언트에서 의미 있게 줄이거나 버리지 않는다.

목표는 xterm을 "무제한 sink"로 쓰지 않고, visible runtime만 bounded scheduler를 통해 쓰게 만드는 것이다.

### 1.5 TerminalHostSlot 측정이 slot 수만큼 증식한다

- `TerminalHostSlot`은 slot마다 `ResizeObserver`를 만들고 host/root를 observe한다.
- slot마다 `window.resize`와 capture `scroll` listener를 등록한다.
- `measure()`는 host/root `getBoundingClientRect()`를 매번 읽는다.
- App의 tab mode host props는 inline style/callback을 매 render 새로 만들 수 있어 measurement effect 재구독 압력을 키운다.

숨겨진 host가 많을수록 observer/listener와 layout measurement 비용이 선형 증가한다.

### 1.6 localStorage snapshot과 mosaic layout cleanup 경계가 약하다

- terminal snapshot 저장은 `setTerminalSnapshotWithQuotaRecovery()` 호출 전 eviction을 수행하고, 이 과정에서 localStorage 전체를 동기 순회한다.
- terminal snapshot removal marker는 `terminal_snapshot_remove_*` tombstone과 module-level Set에 의존한다.
- tombstone cleanup은 mounted `TerminalView` unmount cleanup에 기대는 부분이 있어, 이미 evicted/unmounted 상태였던 session의 회수 경계가 약하다.
- `useMosaicLayout`은 `mosaic_layout_<workspaceId>`를 저장하지만 workspace 삭제 시 제거 API가 없고, unmount cleanup이 삭제된 workspace layout을 다시 저장할 수 있다.

live runtime cap을 도입하면 unmount/eviction이 정상 흐름이 되므로, snapshot/mosaic cleanup도 같은 lifecycle owner에서 다뤄야 한다.

### 1.7 단일 WebSocket이 input/output/workspace event를 모두 담당한다

- `WebSocketContext`는 하나의 `/ws` 연결로 workspace events, subscribe/unsubscribe, terminal output, screen snapshot/repair, input, resize, ping/pong을 모두 처리한다.
- frontend `send()`는 `ws.bufferedAmount`를 확인하지 않고 `ws.send(JSON.stringify(msg))` 후 성공을 반환한다.
- `send` callback은 `status`에 의존하여 identity가 바뀔 수 있고, 이 변화가 terminal runtime 재평가로 전파될 여지가 있다.

사용자 아이디어인 "송수신 WS 분리"는 input path와 output flood를 물리적으로 분리할 수 있다는 점에서 타당하다. 다만 브라우저 WebSocket은 원래 full-duplex이므로, 단순히 socket을 두 개 여는 것보다 어떤 메시지군을 어느 provider가 소유하는지와 reconnect/ordering 계약을 먼저 나눠야 한다.

## 2. 목표 상태

장기 목표는 "보이는 터미널만 브라우저에서 비싼 runtime으로 유지하고, 보이지 않는 터미널은 서버 snapshot/repair를 통해 복원 가능한 parked 상태로 낮추는 것"이다.

### 2.1 명시적 runtime state

프론트엔드 tab runtime은 다음 상태 중 하나로 분류한다.

| 상태 | 대상 | 브라우저 xterm | WS terminal subscription | output write | 복원 방식 |
| --- | --- | --- | --- | --- | --- |
| `visible-live` | 현재 보이는 tab | 유지 | 유지 | 수행 | 실시간 |
| `hidden-warm` | active workspace의 숨은 tab 또는 최근 workspace | 유지 가능 | 선택적 유지 | 기본 중단 | visible 전환 시 repair |
| `parked` | cap 밖 workspace/tab | unmount | 해제 | 없음 | mount 후 snapshot/repair |
| `deleted` | 닫힌 tab/workspace | 없음 | 해제 | 없음 | snapshot/tombstone/mosaic cleanup |

핵심 invariant:

- active workspace는 항상 cap 밖으로 밀려나지 않는다.
- visible tab은 항상 `visible-live`이다.
- cap 밖 workspace는 `TerminalContainer`와 `TerminalView`가 unmount되어야 한다.
- hidden 상태에서 output을 브라우저 xterm에 계속 쓰지 않는다.
- parked에서 visible로 돌아올 때는 클라이언트 문자열 backlog가 아니라 서버 headless snapshot 또는 screen repair로 복원한다.

### 2.2 설정 기반 live workspace cap

`MAX_ALIVE_WORKSPACES` 상수는 장기적으로 설정 기반 정책으로 대체한다.

권장 기본값:

- `maxAliveWorkspaces = 2`: active workspace + 최근 workspace 1개를 live 후보로 유지한다.
- `0`: advanced/unlimited 모드로 남기되, Settings UI에서 메모리/입력 지연 위험을 명확히 표시한다.
- 최소값은 `1`: active workspace만 보장한다.

설정 저장 전략:

- 단기: 브라우저 localStorage 기반 `buildergate.frontendRuntime.maxAliveWorkspaces`로 빠르게 적용한다. 이 값은 기기별 UX preference에 가깝고 서버 재시작 없이 즉시 반영 가능하다.
- 장기: 서버 `config.json5`에 `frontendRuntime` 또는 `clientRuntime` 섹션을 추가하고, Settings API가 서버 기본값을 내려주도록 한다. 브라우저 localStorage는 "local override"로 둔다.
- Settings 표시: 기존 `SettingsPage`의 로컬 Grid 설정처럼 `Local` badge를 붙이는 단기안과, runtime-safe config subset에 포함하는 장기안을 단계별로 나눈다.

### 2.3 송수신 WS 분리 후 frontend 구조

장기 구조는 `WebSocketContext`를 하나의 거대한 provider로 두지 않고 역할별 transport로 쪼갠다.

권장 분리:

- `WorkspaceSocketProvider`: workspace/tab/grid event, connected/client group id, ping/pong, auth lifecycle.
- `TerminalReceiveSocketProvider`: session output, screen snapshot, screen repair, session ready/status/cwd 등 서버에서 브라우저로 흐르는 terminal event.
- `TerminalSendSocketProvider`: input, resize, screen-snapshot ack, screen-repair ack/fail, repair request 등 브라우저에서 서버로 흐르는 terminal command.

실행 단계에서는 두 단계로 줄일 수 있다.

1. 먼저 기존 단일 socket 내부에서 `workspace actions`와 `terminal transport` hook을 논리적으로 분리한다.
2. 그 다음 feature flag로 `terminalRx`와 `terminalTx`를 실제 별도 WebSocket으로 나눈다.

분리 후 `TerminalContainer`는 `useTerminalTransport(sessionId)`만 사용하고, `useWorkspaceManager`는 workspace event provider만 사용한다. 이렇게 해야 output flood가 workspace state update와 input send path를 동시에 흔드는 현상을 줄일 수 있다.

## 3. 단계별 Migration Plan

### Phase 0. 기준선과 관측값 고정

목표: 변경 전 누수/지연을 재현 가능한 숫자로 잡는다.

작업:

- `/api/sessions/telemetry`의 session/ws snapshot과 Chrome Performance/Memory profile을 함께 기록한다.
- frontend에 개발 전용 counters를 추가할 계획을 세운다.
  - mounted `TerminalContainer` 수
  - mounted `TerminalView` 수
  - active `TerminalHostSlot` 수
  - host observer/listener 수
  - terminal output scheduler queue bytes/depth
  - localStorage terminal snapshot entry 수와 총 char 추정치
- cap 도입 전후를 비교할 fixture를 정한다.
  - workspace 10개, workspace당 tab 4개
  - hidden workspace에서 `yes` 또는 대량 로그 출력
  - active terminal에서 keydown-to-echo latency 측정

완료 기준:

- 현재 unlimited 상태에서 hidden workspace output이 active input latency와 main thread work를 증가시키는지 비교 가능한 프로파일을 확보한다.
- 이후 phase의 성공/실패를 판단할 지표 이름을 고정한다.

### Phase 1. Runtime cap 설정화와 pruning owner 정리

목표: `MAX_ALIVE_WORKSPACES`를 상수가 아닌 runtime policy로 바꾸고, cap 적용 시 stale ref/state가 남지 않게 한다.

대상 파일:

- `frontend/src/App.tsx`
- 신규 후보: `frontend/src/utils/frontendRuntimePolicy.ts`
- 신규 후보: `frontend/src/hooks/useFrontendRuntimePolicy.ts`
- 장기 설정 확장 시:
  - `frontend/src/types/settings.ts`
  - `frontend/src/components/Settings/SettingsPage.tsx`
  - `server/src/schemas/config.schema.ts`
  - `server/src/types/config.types.ts`
  - `server/src/types/settings.types.ts`
  - `server/src/services/RuntimeConfigStore.ts`
  - `server/src/services/SettingsService.ts`

설계:

- `maxAliveWorkspaces`는 `1..N` 또는 `0(unlimited)`만 허용한다.
- LRU 계산은 `workspaceVisitOrder`와 `aliveWorkspaceIds`를 직접 여러 effect에서 분산 관리하지 않고, `wm.workspaces`, `wm.activeWorkspaceId`, policy cap을 입력으로 하는 helper로 모은다.
- `wm.workspaces`에서 사라진 workspace id는 `aliveWorkspaceIds`와 `workspaceVisitOrder`에서 즉시 제거한다.
- `wm.tabs`에서 사라진 tab id는 `terminalRefsMap.current.delete(tab.id)`에 해당하는 cleanup을 수행한다.
- cap을 낮추면 active workspace를 제외한 오래된 workspace부터 parked로 전환한다.
- cap을 높이면 새 workspace를 자동으로 살리지 않고, 사용자가 방문한 workspace부터 alive로 승격한다.

복원 전략:

- cap eviction은 서버 session 종료가 아니다. 브라우저 runtime만 unmount한다.
- evicted workspace로 돌아오면 `parked -> visible-live` 전환 중 server snapshot/repair를 먼저 요청한다.
- 기존 local snapshot은 서버 snapshot 실패 또는 headless degraded일 때 fallback으로만 사용한다.

테스트:

- cap helper unit test:
  - active workspace는 cap 초과에서도 유지된다.
  - 삭제된 workspace id가 visit order/alive set에서 제거된다.
  - cap 0은 eviction을 하지 않는다.
  - cap 1은 active workspace만 남긴다.
- App-level integration test:
  - workspace 전환 후 cap 밖 `TerminalContainer`가 unmount된다.
  - tab 삭제 후 `terminalRefsMap`에 stale tab ref가 남지 않는다.

### Phase 2. Hidden output write 중단과 visible 전환 repair

목표: hidden terminal이 xterm write와 snapshot save를 계속 수행하지 않게 한다.

대상 파일:

- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/types/ws-protocol.ts`
- 필요 시 `server/src/ws/WsRouter.ts`, `server/src/services/SessionManager.ts`

설계:

- `TerminalContainer.onOutput`에서 `isVisibleRef.current`를 확인한다.
- visible이 아니면 `terminalRef.current?.write(data)`를 호출하지 않는다.
- hidden 중 output을 클라이언트 문자열 배열에 무제한 보관하지 않는다.
- 대신 session별 `dirtyWhileHidden` flag와 마지막 output seq 또는 server screen seq만 기록한다.
- visible 전환 시 다음 순서로 복원한다.
  1. layout fit과 geometry 준비
  2. 서버에 `screen-repair` 또는 `repair-replay` 요청
  3. repair 성공 시 xterm에 patch 적용
  4. repair 실패 또는 headless degraded면 server snapshot/fallback snapshot 적용
  5. 그래도 실패하면 새 output 대기 상태로 전환

정책:

- active workspace의 hidden tab도 기본적으로 output write를 중단한다.
- 단, 사용자가 "숨은 탭도 실시간 scrollback 유지"를 켠 경우에만 hidden write를 허용한다. 이 옵션은 기본값이 아니어야 한다.
- alternate buffer, user scrolled 상태, IME active 상태에서는 기존 repair readiness 계약을 유지한다.

주의점:

- hidden 상태에서 output을 버리면 xterm scrollback은 stale이 된다. 이것은 의도된 tradeoff이며, visible 전환 repair가 제품 계약이 되어야 한다.
- `session status`와 `cwd`는 output과 달리 UI metadata이므로 hidden 상태에서도 받을 수 있다. 단 terminal output stream과 분리되면 metadata socket에서 처리한다.

테스트:

- hidden tab에 output이 들어와도 `TerminalView.write()`가 호출되지 않는다.
- hidden 중 snapshot debounce가 예약되지 않는다.
- visible 전환 시 `screen-repair`가 한 번 요청된다.
- repair 성공 후 input readiness가 회복된다.
- repair 실패 시 fallback snapshot 또는 새 output 대기 상태가 관측 가능하다.

### Phase 3. xterm output write scheduler 도입

목표: visible terminal의 output도 frame budget과 queue cap 안에서 처리한다.

대상 파일:

- 신규 후보: `frontend/src/components/Terminal/terminalOutputScheduler.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/utils/terminalDebugCapture.ts`

설계:

- scheduler는 `term.write()`를 직접 감싸고 다음 값을 기록한다.
  - queued chunks
  - queued bytes
  - oldest queued age
  - in-flight write 여부
  - last write duration
  - max observed write duration
- chunk가 빠르게 들어오면 frame 단위로 병합한다.
- queue가 cap을 넘으면 visible terminal에서도 무한 적재하지 않는다.
  - tail coalescing을 우선한다.
  - cap 초과 시 `needsRepair`를 세우고 오래된 queued output을 버린다.
  - 다음 안정 시점에 screen repair를 요청한다.
- snapshot save는 모든 write callback마다 예약하지 않고, scheduler flush 완료 또는 idle deadline에서 한 번만 예약한다.

초기 권장값:

- frame flush: `requestAnimationFrame` 기준 1회
- 최대 queued bytes: 서버 `maxSnapshotBytes`보다 낮은 클라이언트 전용 값, 예: 256 KiB
- write latency warning: 50 ms 이상
- repair trigger: queue cap 초과 또는 write callback 미완료가 1초 이상 지속

테스트:

- 1000개 작은 chunk가 들어와도 `term.write()` 호출 수가 병합된다.
- queue cap 초과 시 scheduler가 `needsRepair`를 반환한다.
- snapshot save는 flush cycle당 최대 1회 예약된다.
- scheduler dispose 후 pending callback이 상태를 갱신하지 않는다.

### Phase 4. TerminalHostSlot 측정 구조 중앙화

목표: slot별 observer/window listener를 Provider 단위 batch 측정으로 바꾼다.

대상 파일:

- `frontend/src/components/Terminal/TerminalHostSlot.tsx`
- `frontend/src/components/Terminal/TerminalRuntimeContext.tsx`
- `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx`
- `frontend/src/App.tsx`

설계:

- `TerminalHostSlot`은 DOM ref, tab id, visibility, interaction handler만 등록한다.
- `TerminalRuntimeProvider`가 단일 `ResizeObserver`와 단일 `window.resize`/`scroll` listener를 소유한다.
- 측정은 RAF batch로 수행한다.
- `isVisible=false`인 host는 기본적으로 측정 대상에서 제외한다.
- root rect는 batch당 한 번만 읽는다.
- App에서 넘기는 style/callback은 stable하게 memoize하거나 interactions ref에 저장해 measurement dependency에서 제거한다.

완료 기준:

- workspace/tab 수가 늘어도 window listener 수가 증가하지 않는다.
- visible host만 layout measurement를 수행한다.
- layout mode 변경과 mosaic resize 후 fit/repair는 기존 UX를 유지한다.

테스트:

- 여러 `TerminalHostSlot` mount 시 `window.addEventListener('resize')`가 한 번만 호출된다.
- hidden host는 measurement batch에서 제외된다.
- host unmount 후 context `hosts`와 `hostInteractionsRef`에서 제거된다.

### Phase 5. localStorage snapshot과 mosaic cleanup 정리

목표: runtime unmount가 정상 경로가 되어도 storage가 stale 상태로 쌓이지 않게 한다.

대상 파일:

- `frontend/src/utils/terminalSnapshot.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/hooks/useWorkspaceManager.ts`
- `frontend/src/hooks/useMosaicLayout.ts`
- `frontend/src/components/Grid/MosaicContainer.tsx`

Terminal snapshot 정책:

- `terminal_snapshot_remove_*` tombstone에 TTL을 적용한다.
- app startup, snapshot save 전, settings save 전 중 하나의 낮은 빈도 경로에서 expired tombstone을 정리한다.
- snapshot 저장은 평상시 `setItem`을 먼저 시도하고, `QuotaExceededError`일 때만 eviction scan을 수행하는 방향으로 바꾼다.
- eviction용 snapshot index를 도입하면 localStorage 전체 순회를 줄일 수 있다.
- cap eviction 때문에 unmount되는 경우는 "삭제"가 아니므로 snapshot removal marker를 만들지 않는다.
- tab close/workspace delete/restart old session만 removal marker를 만든다.

Mosaic layout 정책:

- `removeMosaicLayout(workspaceId)`를 export한다.
- workspace delete 시작 시 layout key를 제거한다.
- 삭제로 인한 `useMosaicLayout` unmount cleanup은 save를 skip한다.
- tab close는 workspace layout을 즉시 삭제하지 않고, 현재 `MosaicContainer` tree 재계산 후 저장한다.

테스트:

- workspace delete 후 `mosaic_layout_<workspaceId>`가 제거된다.
- delete-triggered unmount가 layout을 다시 저장하지 않는다.
- snapshot removal tombstone TTL 만료 후 cleanup된다.
- cap eviction은 terminal snapshot을 삭제하지 않는다.

### Phase 6. WebSocket transport 논리 분리

목표: 실제 socket을 나누기 전에 frontend provider와 hook 의존성을 분리한다.

대상 파일:

- `frontend/src/contexts/WebSocketContext.tsx`
- 신규 후보: `frontend/src/contexts/WorkspaceSocketContext.tsx`
- 신규 후보: `frontend/src/contexts/TerminalTransportContext.tsx`
- `frontend/src/hooks/useWorkspaceManager.ts`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/types/ws-protocol.ts`

설계:

- 기존 `useWebSocketState()`/`useWebSocketActions()`를 내부 호환 wrapper로 남긴다.
- `useWorkspaceManager`는 workspace event subscription만 사용한다.
- `TerminalContainer`는 terminal transport API만 사용한다.
- `send` callback은 React state인 `status`에 의존하지 않고 ref에서 현재 상태를 읽는다.
- `SendResult`에 backpressure reason을 추가할 수 있도록 타입을 확장한다.
  - 예: `backpressure`, `closed`, `stale-socket`, `missing-token`
- terminal send path는 `ws.bufferedAmount` high-water mark를 확인한다.

완료 기준:

- workspace event provider 변화가 `TerminalView` 생성 effect identity 변경으로 전파되지 않는다.
- terminal subscription lifecycle은 `TerminalContainer` mount/unmount와 직접 연결된다.
- 아직 물리적으로 단일 socket이어도 코드상 역할 경계가 나뉜다.

테스트:

- workspace event dispatch가 terminal transport hook consumer를 불필요하게 rerender하지 않는다.
- terminal send backpressure 시 input queue/reject 정책이 명시적으로 작동한다.
- reconnect 후 active terminal subscription만 재구독된다.

### Phase 7. 실제 송수신 WS 분리

목표: output flood와 input/control send path를 물리적으로 분리한다.

대상 파일:

- `frontend/src/contexts/TerminalTransportContext.tsx`
- `frontend/src/types/ws-protocol.ts`
- `server/src/index.ts`
- `server/src/ws/WsRouter.ts` 또는 신규 `server/src/ws/TerminalRxRouter.ts`, `TerminalTxRouter.ts`
- 테스트: `server/src/test-runner.ts`, frontend unit/e2e

권장 endpoint:

- `/ws/control`: workspace/tab/grid events, connection group id, ping/pong
- `/ws/terminal-rx`: subscribe/unsubscribe, output, screen snapshot, screen repair, status/cwd/session ready
- `/ws/terminal-tx`: input, resize, repair request, snapshot/repair ack

대안:

- `/ws/control` + `/ws/terminal` 두 개만 먼저 둔다.
- 이 경우 terminal socket은 여전히 full-duplex이지만 workspace event와 terminal flood를 분리할 수 있다.
- 이후 필요가 확인되면 terminal rx/tx를 추가로 분리한다.

frontend 변경:

- `TerminalReceiveSocketProvider`는 subscribed session set과 output handlers만 관리한다.
- `TerminalSendSocketProvider`는 input/resize/ack send queue와 backpressure만 관리한다.
- 두 socket은 같은 browser instance를 나타내는 `connectionGroupId`를 공유한다.
- server broadcast의 `excludeClientId`는 socket id가 아니라 group id 기준으로 동작해야 한다.
- terminal rx reconnect 시 active/live session만 재구독한다. parked session은 재구독하지 않는다.
- terminal tx reconnect 중 input은 기존 input reliability policy에 따라 queue/reject한다.

ordering 계약:

- input echo는 terminal rx로 돌아온다. tx send 성공이 곧 화면 반영을 의미하지 않는다.
- snapshot/repair ack는 tx로 보내되, rx에서 받은 repair token과 session generation을 함께 검증한다.
- rx가 끊겼는데 tx만 살아 있으면 input을 "보낼 수 있음"으로 보지 않는다. 화면 복원 불가 상태에서는 queue 또는 reject가 맞다.
- tx가 끊겼는데 rx만 살아 있으면 output은 볼 수 있지만 입력은 blocked 상태로 표시한다.

테스트:

- rx flood 중 tx input send latency가 증가하지 않는다.
- rx reconnect가 parked session을 재구독하지 않는다.
- tx reconnect 중 Enter input이 안전 정책에 따라 queue/reject된다.
- rx/tx 중 하나만 끊긴 partial failure 상태가 UI와 debug capture에 관측된다.

### Phase 8. 기본값 전환과 legacy 경로 축소

목표: cap과 hidden output pause를 기본값으로 켜고, unlimited live runtime을 예외 모드로 내린다.

작업:

- `maxAliveWorkspaces` 기본값을 2로 전환한다.
- hidden output write pause를 기본값으로 켠다.
- split WS는 충분한 soak 후 기본값으로 켠다.
- legacy 단일 `WebSocketContext` public API를 deprecated 처리한다.
- debug counters와 telemetry 문서를 업데이트한다.

완료 기준:

- 장시간 사용 시 mounted terminal runtime 수가 cap과 active workspace tab 수의 함수로 제한된다.
- hidden output이 active terminal 입력 지연을 만들지 않는다.
- workspace switch UX는 snapshot/repair 지연 외에는 기존과 동일하다.

## 4. 사용자 아이디어 반영 방식

### 4.1 설정 기반 cap

사용자 아이디어의 핵심은 `MAX_ALIVE_WORKSPACES`를 하드코딩하지 않고 운영자가 환경과 장비 성능에 맞춰 조절하게 하는 것이다. 이 아이디어는 반드시 반영해야 한다.

권장 적용:

- 단기: Settings의 "Grid Layout"처럼 local setting으로 `Live workspaces` 값을 제공한다.
- 중기: `config.json5`에 서버 기본값을 추가한다.
- 장기: 서버 기본값 + 브라우저 local override 구조로 둔다.

UX 세부:

- 값 `1`: 가장 보수적인 메모리 모드. active workspace만 live.
- 값 `2`: 권장 기본값. active + 최근 workspace.
- 값 `3..5`: 빠른 workspace 전환 우선.
- 값 `0`: unlimited. 기존 동작 호환용이지만 "장시간 사용 시 메모리/입력 지연 위험"을 명시한다.

복원 전략:

- cap 때문에 parked된 workspace는 "닫힌 것"이 아니며 서버 session은 유지된다.
- 복원은 server snapshot/repair 기반이다.
- local snapshot은 보조 fallback이다.
- cap 변경은 즉시 적용하되 active workspace는 절대 evict하지 않는다.

### 4.2 송수신 WS 분리

사용자 아이디어의 핵심은 대량 output이 input send path와 같은 socket/buffer/main-thread dispatch 경로를 공유하지 않게 하는 것이다. 이 아이디어도 장기 구조에 포함해야 한다.

권장 적용:

- 먼저 frontend provider/hook을 논리 분리한다.
- 다음 feature flag로 실제 socket을 `/ws/terminal-rx`와 `/ws/terminal-tx`로 나눈다.
- workspace event는 별도 `/ws/control`로 유지한다.

기대 효과:

- output flood가 input send `bufferedAmount`와 같은 socket queue를 밀지 않는다.
- terminal output reconnect와 workspace event reconnect를 독립적으로 다룰 수 있다.
- parked session은 rx subscription 대상에서 제외되어 서버 broadcast fan-out도 줄어든다.

주의점:

- socket이 늘면 인증, heartbeat, reconnect, observability 복잡도가 증가한다.
- server의 client id를 socket id 기준으로 유지하면 broadcast exclude가 깨질 수 있다. connection group id가 필요하다.
- rx/tx partial failure 상태가 생기므로 input readiness와 UI status 모델을 확장해야 한다.

## 5. UI/UX 영향

### 5.1 Workspace 전환

cap 밖 workspace로 이동하면 즉시 이전 화면이 완벽히 살아 있는 것이 아니라 복원 과정이 필요하다. UX는 다음 원칙을 따른다.

- active workspace 전환 자체는 즉시 반응한다.
- terminal 영역은 layout fit 후 snapshot/repair를 적용한다.
- 복원 중 input은 기존 input readiness gate를 통해 queue 또는 reject한다.
- 복원 실패는 조용히 다른 경로로 fallback하지 않고 debug capture/상태로 관측 가능해야 한다.

사용자에게 보일 수 있는 변화:

- 오래 방치한 workspace로 돌아올 때 100-500ms 수준의 terminal 화면 복원 지연이 생길 수 있다.
- hidden 중 발생한 scrollback 전체가 클라이언트 xterm에 그대로 남지 않을 수 있다.
- 서버 snapshot이 정상이라면 현재 viewport는 복원된다.
- headless degraded 상태에서는 fallback message 또는 새 output 대기 상태가 나타날 수 있다.

### 5.2 Settings

설정 UI는 기존 `SettingsPage`에 작은 runtime section을 추가하는 방식이 자연스럽다.

권장 필드:

- `Live workspaces`: number input, min 1, max 10, 0은 unlimited advanced option.
- `Hidden terminal output`: segmented control 또는 select.
  - `Pause and repair on focus` 기본값
  - `Keep writing while hidden` legacy/advanced
- `WebSocket transport`: initially read-only status 또는 experimental toggle.
  - `single`
  - `split terminal`
  - `split rx/tx`

단, AGENTS 지침상 UI visuals/labels/layout은 개인 판단만으로 바꾸면 안 된다. 실제 구현 시 Settings UI 변경은 별도 작업으로 분리하고, 사용자 확인 후 진행하는 것이 안전하다.

### 5.3 Debug/운영자 경험

장기적으로는 다음 값이 debug capture 또는 telemetry에 보여야 한다.

- runtime state별 tab 수: visible-live, hidden-warm, parked
- hidden 중 dropped/coalesced output bytes
- screen repair 요청/성공/실패 수
- xterm scheduler max queue bytes와 max write latency
- terminal rx/tx socket status와 bufferedAmount
- localStorage snapshot/mosaic cleanup count

## 6. 위험과 대응

### 6.1 Stale 화면과 복원 실패

위험:

- hidden output을 쓰지 않으면 visible 전환 시 화면이 stale할 수 있다.
- server headless snapshot/repair가 degraded이면 복원이 제한된다.

대응:

- server snapshot/repair를 cap 기능의 전제 dependency로 둔다.
- repair 실패 이유를 debug capture에 남긴다.
- local snapshot fallback을 유지하되, 성공처럼 숨기지 않는다.

### 6.2 입력 손실

위험:

- visible 전환 중 input을 받으면 복원 전 server state와 화면 state가 어긋날 수 있다.
- rx/tx split 후 한쪽만 끊긴 partial failure에서 input policy가 복잡해진다.

대응:

- 기존 input readiness gate를 runtime state와 rx/tx status까지 확장한다.
- restore/repair pending 중 Enter 입력은 엄격하게 queue 또는 reject한다.
- debug metadata는 inputSeq와 sessionGeneration을 계속 포함한다.

### 6.3 focus/IME 회귀

위험:

- `TerminalView`는 IME transaction, helper textarea, focus restore 로직이 복잡하다.
- unmount/remount 빈도가 늘면 focus steal이나 CJK 입력 회귀가 생길 수 있다.

대응:

- `waitForImeIdle()` 계약은 repair/snapshot 적용 전 계속 지킨다.
- parked 전환 시 focused terminal이면 focus restoration intent만 저장하고 DOM node 참조는 버린다.
- IME 관련 Playwright/manual 검증을 별도 체크리스트로 둔다.

### 6.4 WebSocket 복잡도 증가

위험:

- socket 분리는 인증, heartbeat, reconnect, server observability를 늘린다.
- client id가 socket별로 달라지면 workspace broadcast exclude와 debug correlation이 흔들린다.

대응:

- split 전에 provider 논리 분리와 backpressure result 타입을 먼저 도입한다.
- 실제 split은 feature flag로 시작한다.
- server/client 공통 `connectionGroupId`를 설계한다.

### 6.5 설정 UX 오해

위험:

- 사용자가 cap을 "세션 종료"로 오해할 수 있다.
- unlimited를 켜면 기존 지연 문제가 재현될 수 있다.

대응:

- 설정 설명은 "브라우저에 live로 유지할 workspace 수"라고 명확히 쓴다.
- 서버 session은 유지된다는 점과 복원 지연 tradeoff를 함께 표시한다.
- unlimited는 legacy compatibility로 분류한다.

## 8. 구현 순서 권장안

가장 안전한 순서는 다음이다.

1. 관측 counter와 cap policy helper를 먼저 만든다.
2. `maxAliveWorkspaces`를 local setting으로 적용하고 `aliveWorkspaceIds`/`workspaceVisitOrder`/`terminalRefsMap` pruning을 완성한다.
3. hidden output write를 중단하고 visible 전환 repair를 제품 계약으로 만든다.
4. xterm output scheduler를 도입해 visible terminal도 bounded 처리한다.
5. HostSlot measurement를 Provider batch로 중앙화한다.
6. snapshot/mosaic cleanup을 lifecycle owner에 묶는다.
7. WebSocket provider를 논리 분리한다.
8. feature flag로 실제 split WS를 적용한다.
9. soak 후 기본값을 `maxAliveWorkspaces=2`, hidden output pause enabled로 전환한다.

이 순서를 권장하는 이유는 cap과 hidden output pause만으로도 입력 지연의 가장 큰 원인을 줄일 가능성이 높고, split WS는 효과가 크지만 server/frontend protocol과 reconnect 상태 모델을 동시에 건드리는 고위험 변경이기 때문이다.

## 9. 핵심 결론

- `MAX_ALIVE_WORKSPACES`는 단순 상수 변경으로 끝내면 안 된다. 설정 기반 cap, LRU pruning, terminal ref cleanup, snapshot/mosaic cleanup이 같은 lifecycle 정책 아래 있어야 한다.
- hidden terminal은 브라우저 xterm에 output을 계속 쓰면 안 된다. hidden 중에는 dirty flag만 남기고, visible 전환 시 server snapshot/repair로 복원하는 구조가 장기적으로 맞다.
- xterm output은 bounded scheduler가 필요하다. visible terminal도 무제한 `term.write()` sink가 되어서는 안 된다.
- `TerminalHostSlot`의 observer/window listener는 Provider 단위 batch 측정으로 중앙화해야 한다.
- 송수신 WS 분리는 타당하지만, 먼저 `WebSocketContext`를 논리적으로 workspace event와 terminal transport로 나누고, 그 다음 rx/tx socket split을 feature flag로 적용해야 한다.
- 기본 UX는 `maxAliveWorkspaces=2`, hidden output pause enabled, server repair 우선 복원으로 잡는 것이 메모리 안정성과 전환 UX의 균형이 가장 좋다.
