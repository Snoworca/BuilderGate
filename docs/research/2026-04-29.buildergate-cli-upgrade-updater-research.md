# BuilderGate CLI upgrade/updater 구현 가능성 연구

> **프로젝트**: BuilderGate / ProjectMaster  
> **연구 주제**: `{실행 파일} upgrade` 명령으로 GitHub Release 기반 자동 업데이트를 구현할 수 있는지 검토  
> **작성일**: 2026-04-29  
> **결론**: 구현 가능하다. 단, Windows/NTFS에서는 실행 중인 `BuilderGate.exe`를 자기 자신이 직접 교체할 수 없으므로, `upgrade` 명령은 다운로드/검증/스테이징 후 별도 updater 스크립트 또는 updater 실행 파일을 띄우고 종료해야 한다.

---

## 1. 요구사항

사용자가 원하는 동작은 다음이다.

1. 콘솔에서 `{실행 파일} upgrade`를 입력한다.
2. 서버가 실행 중인지 확인한다.
3. 서버가 실행 중이라면 `{실행 파일} stop`으로 먼저 종료하라고 영어로 안내하고 upgrade 과정을 종료한다.
4. 서버가 종료된 상태라면 `https://github.com/Snoworca/BuilderGate/releases`에서 새 버전을 확인한다. GitHub API 사용을 권장한다.
5. 새 버전이 있다면 다운로드하고 압축을 푼 뒤 현재 설치본을 교체한다.
6. NTFS 파일 시스템을 고려해 update용 별도 스크립트나 실행 파일을 만든다.

---

## 2. 요약 결론

`BuilderGate.exe upgrade`는 현재 코드 구조에 자연스럽게 추가할 수 있다.

- CLI parser는 `tools/daemon/cli.js`에 있고, `stop`과 동일한 방식으로 `upgrade` command를 추가할 수 있다.
- command dispatcher는 `tools/start-runtime.js`에 있으며, `parsedArgs.command === 'upgrade'` 분기를 추가하면 된다.
- native daemon 실행 감지는 `tools/daemon/launcher.js`의 `inspectExistingDaemon()`과 daemon state를 재사용할 수 있다.
- packaged runtime root는 `tools/daemon/runtime-paths.js`에서 이미 `process.execPath` 기준으로 계산된다.
- 현재 릴리스 workflow는 `BuilderGate-win-amd64-<version>.zip` 같은 고정 asset 이름을 만든다.

다만 `upgrade`가 실행되는 프로세스 자체가 교체 대상인 `BuilderGate.exe`이므로, Windows에서는 현재 프로세스가 살아 있는 동안 같은 EXE 파일을 안전하게 overwrite할 수 없다. 따라서 Windows 기본 구현은 다음처럼 해야 한다.

1. `upgrade` 프로세스가 GitHub API 조회, asset 다운로드, digest 검증, 압축 해제, staging 검증을 수행한다.
2. `runtime/upgrades/<version>/apply-update.ps1` 같은 별도 updater 스크립트를 생성한다.
3. updater를 detached child로 실행하고, 현재 `BuilderGate.exe upgrade` 프로세스는 종료한다.
4. updater가 parent PID 종료를 기다린 뒤 파일을 교체한다.

## 3. 기존 코드 연결점

### 3.1 CLI parsing

현재 `tools/daemon/cli.js`는 `stop`을 첫 번째 positional command로 처리한다.

- `parseArgs(argv)` 기본 command: `start`
- `stop` command: `parsed.command = 'stop'`, `parsed.mode = 'stop'`
- help text: `Usage: ${executableName} [stop] ...`

`upgrade`도 같은 패턴으로 추가할 수 있다.

권장 parser 규칙:

- `upgrade`는 첫 번째 인자일 때만 허용한다.
- `upgrade --check` 같은 dry-run 옵션은 후속 확장으로 둘 수 있다.
- 시작 옵션인 `-p`, `--foreground`, `--reset-password`와 조합되면 명확한 오류를 낸다.

### 3.2 command dispatch

현재 `tools/start-runtime.js`는 help와 stop을 start flow 전에 처리한다.

```text
help -> stop -> ensureDependenciesAndBuild -> config preflight -> foreground/daemon start
```

`upgrade`는 `stop`과 같은 위치, 즉 build/preflight/start 전에 처리해야 한다. packaged runtime에서 upgrade가 dev dependency나 build step을 건드리면 안 되기 때문이다.

권장 순서:

```text
help -> stop -> upgrade -> start
```

### 3.3 서버 실행 감지

현재 native daemon은 `runtime/buildergate.daemon.json`에 state를 기록한다. `launcher.inspectExistingDaemon(paths)`는 다음을 확인한다.

- state가 daemon/running인지
- app PID와 sentinel PID가 살아 있는지
- `/health` identity가 state와 일치하는지

따라서 daemon mode 실행 중 여부는 이 함수를 재사용하면 된다.

추가로 고려할 점:

- foreground mode는 daemon state를 만들지 않는다.
- 사용자가 `BuilderGate.exe --foreground`로 실행 중이면 `inspectExistingDaemon()`만으로는 탐지하지 못할 수 있다.
- 기본 config port 또는 daemon state port의 `https://127.0.0.1:<port>/health`를 조회해 BuilderGate health payload를 확인하는 보조 감지가 필요하다.
- `--foreground -p <custom>`처럼 config에 없는 port로 실행한 서버는 완전 탐지가 어렵다. 이 경우까지 강하게 보장하려면 foreground 실행도 별도 foreground state를 쓰도록 런타임 계약을 확장해야 한다.

최소 구현의 실행 중 판단:

1. daemon state가 active이면 running.
2. daemon state가 unknown이면 안전을 위해 running/blocked로 간주하고 upgrade 중단.
3. config에서 port를 읽을 수 있으면 해당 port의 `/health`가 BuilderGate 응답인지 확인한다.
4. 응답이 확인되면 running으로 간주한다.
5. 아무 증거가 없으면 stopped로 간주한다.

실행 중일 때 출력할 영어 메시지는 다음처럼 명확해야 한다.

```text
BuilderGate is currently running. Please run "BuilderGate.exe stop" before upgrading.
```

non-Windows 실행 파일은 basename을 사용해 다음처럼 출력한다.

```text
BuilderGate is currently running. Please run "buildergate stop" before upgrading.
```

## 4. GitHub Release 조회

### 4.1 API 선택

공식 GitHub REST API를 사용한다.

- 최신 릴리스: `GET https://api.github.com/repos/Snoworca/BuilderGate/releases/latest`
- 릴리스 목록: `GET https://api.github.com/repos/Snoworca/BuilderGate/releases?per_page=20`

`latest` endpoint는 public repository에 대해 인증 없이 사용할 수 있다. 다만 안정적인 semver 판단을 위해 구현에서는 `releases?per_page=20`을 조회한 뒤 다음 조건으로 고르는 편이 더 안전하다.

1. `draft === false`
2. `prerelease === false`
3. `tag_name`이 `v<semver>` 또는 `<semver>` 형식
4. 현재 버전보다 높은 semver
5. 가장 높은 semver

GitHub API rate limit을 고려해 `GITHUB_TOKEN` 또는 `BUILDERGATE_GITHUB_TOKEN` 환경 변수가 있으면 Authorization header를 선택적으로 붙일 수 있다. public repo 기본 사용에는 토큰이 필요하지 않다.

권장 header:

```text
Accept: application/vnd.github+json
X-GitHub-Api-Version: 2022-11-28
User-Agent: BuilderGate-updater/<current-version>
```

### 4.2 현재 확인된 릴리스 상태

2026-04-29 조사 시점의 GitHub API 응답 기준 최신 릴리스는 `v0.2.0`이었다.

확인된 asset:

- `BuilderGate-linux-amd64-0.2.0.tar.gz`
- `BuilderGate-linux-arm64-0.2.0.tar.gz`
- `BuilderGate-macos-arm64-0.2.0.zip`
- `BuilderGate-win-amd64-0.2.0.zip`
- `BuilderGate-win-arm64-0.2.0.zip`

각 asset 응답에는 `size`, `digest`, `browser_download_url`이 포함되어 있었다. `digest`는 `sha256:<hash>` 형식으로 내려오므로 다운로드 후 무결성 검증에 사용할 수 있다.

### 4.3 asset 매칭 규칙

현재 release workflow의 asset 이름은 다음 계약이다.

| platform | arch | target | asset |
| --- | --- | --- | --- |
| win32 | x64 | win-amd64 | `BuilderGate-win-amd64-<version>.zip` |
| win32 | arm64 | win-arm64 | `BuilderGate-win-arm64-<version>.zip` |
| linux | x64 | linux-amd64 | `BuilderGate-linux-amd64-<version>.tar.gz` |
| linux | arm64 | linux-arm64 | `BuilderGate-linux-arm64-<version>.tar.gz` |
| darwin | arm64 | macos-arm64 | `BuilderGate-macos-arm64-<version>.zip` |

현재 macOS는 ARM64만 지원하므로 `darwin/x64`에서 upgrade를 실행하면 "unsupported target"으로 실패해야 한다.

## 5. 다운로드, 압축 해제, 검증

다운로드 위치는 설치 root 밖이 아니라 root 아래 `runtime/upgrades/`가 적절하다. 이 경로는 이미 runtime성 데이터가 모이는 위치이며, updater log와 staging/backup을 함께 둘 수 있다.

권장 layout:

```text
runtime/
  upgrades/
    v0.3.0/
      download/
        BuilderGate-win-amd64-0.3.0.zip
      stage/
        BuilderGate-win-amd64-0.3.0/
          BuilderGate.exe
          web/
          shell-integration/
          BuilderGate.ico
          BuilderGate.svg
          BuilderGate.icns
          README.md
          config.json5
          config.json5.example
      backup-20260429-120000/
      apply-update.ps1
      upgrade.log
```

검증 규칙:

1. HTTP status가 2xx인지 확인한다.
2. 다운로드 byte count가 asset `size`와 일치하는지 확인한다.
3. asset `digest`가 있으면 SHA-256을 계산해 일치해야 한다.
4. 압축 해제 결과가 단일 top-level directory인지 확인한다.
5. top-level directory 이름이 asset 계약과 일치하는지 확인한다.
6. 대상별 필수 파일을 확인한다.
7. 압축 경로 traversal이 없는지 검사한다.

필수 파일 예:

Windows:

- `BuilderGate.exe`
- `web/index.html`
- `shell-integration/bash-osc133.sh`
- `config.json5.example`
- `README.md`
- `BuilderGate.ico`

Linux:

- `buildergate`
- `web/index.html`
- `shell-integration/bash-osc133.sh`
- `config.json5.example`
- `README.md`

macOS:

- `buildergate`
- `BuilderGate.app`
- `web/index.html`
- `shell-integration/bash-osc133.sh`
- `BuilderGate.icns`

## 6. NTFS/Windows self-update 전략

### 6.1 왜 별도 updater가 필요한가

Windows는 실행 중인 EXE를 업데이트할 때 파일 잠금 문제가 발생한다. Microsoft `MoveFileEx`는 `MOVEFILE_REPLACE_EXISTING`, `MOVEFILE_DELAY_UNTIL_REBOOT` 같은 옵션을 제공하지만, reboot 지연 작업은 관리자 권한/재부팅 의존성이 있고 기본 자동 업데이트 UX로 쓰기에는 부담이 크다.

따라서 기본 전략은 reboot 예약이 아니라 다음이다.

1. `BuilderGate.exe upgrade`가 staging을 끝낸다.
2. 별도 updater process를 실행한다.
3. `BuilderGate.exe upgrade`가 종료된다.
4. updater가 parent PID 종료를 기다린다.
5. updater가 기존 파일을 backup으로 이동하고 새 파일을 복사한다.

### 6.2 Windows updater script 흐름

PowerShell updater는 다음 일을 한다.

1. `Wait-Process -Id <parentPid>`로 upgrade parent 종료 대기
2. daemon state가 running으로 바뀌지 않았는지 재확인
3. 교체 대상 파일/디렉터리를 backup directory로 이동
4. 새 파일/디렉터리를 stage에서 install root로 복사
5. 보존 대상은 건드리지 않음
6. 실패 시 가능한 범위에서 backup을 rollback
7. `upgrade.log`에 결과 기록

교체 대상:

- `BuilderGate.exe` 또는 `buildergate`
- `web/`
- `shell-integration/`
- `README.md`
- `BuilderGate.ico`
- `BuilderGate.svg`
- `BuilderGate.icns`
- `config.json5.example`
- macOS의 경우 `BuilderGate.app`

보존 대상:

- `config.json5`
- `runtime/`
- `certs/`
- `data/`
- `*.bak`
- 사용자 생성 파일

`config.json5`는 새 설치에만 stage의 기본 파일을 복사하고, 기존 설치에는 절대 덮어쓰지 않는다. 새 설정 예시는 `config.json5.example`만 갱신한다.

### 6.3 updater exe vs PowerShell script

초기 구현은 PowerShell script가 현실적이다.

장점:

- 별도 빌드 target을 늘리지 않아도 된다.
- 현재 `pkg` packaging 구조를 크게 바꾸지 않아도 된다.
- Windows 파일 교체와 retry loop를 명확하게 표현할 수 있다.

단점:

- 실행 정책/보안 제품에 의해 차단될 수 있다.
- UX가 콘솔 EXE 하나로 끝나는 형태보다는 덜 깔끔하다.

장기적으로는 `BuilderGateUpdater.exe`를 별도 작은 native/pkg binary로 만들 수 있다. 하지만 릴리스 asset에 updater exe를 포함해야 하므로 release workflow와 build validation 범위가 커진다.

## 7. 버전 비교

루트 `package.json`의 `version`을 현재 버전으로 사용한다. packaged binary에는 package metadata가 `pkg` snapshot에 포함되어 있으므로 `require('./package.json')` 또는 build-time constant를 사용할 수 있다.

semver parser는 외부 package 추가 없이 간단히 구현할 수 있다.

허용 형식:

- `0.2.0`
- `v0.2.0`

초기에는 prerelease를 무시한다.

비교 규칙:

1. `major`, `minor`, `patch`를 정수로 비교
2. 동일하면 새 버전 없음
3. 최신 릴리스가 현재보다 낮거나 같으면 no-op
4. malformed tag는 skip

---

## 8. 보안 및 안정성 요구

업데이트 기능은 remote binary를 내려받아 실행 파일을 교체하므로 보안 요구가 높다.

필수:

- HTTPS만 허용
- GitHub API host와 download URL host 검증
- SHA-256 digest 검증
- 압축 해제 path traversal 방지
- 설치 root 밖 파일 쓰기 금지
- 실행 중 daemon이면 중단
- `unknown` daemon state이면 중단
- 사용자 설정/데이터 보존
- 실패 시 기존 설치본을 가능한 범위에서 유지
- 모든 오류는 콘솔과 `runtime/upgrades/.../upgrade.log`에 남김

권장:

- GitHub API 응답의 `immutable`, `draft`, `prerelease` 확인
- asset `digest`가 없는 경우 checksum asset이 없으면 업데이트 중단
- 향후 코드 서명 또는 별도 manifest 도입

## 9. 구현 대상 파일

권장 변경 범위:

| 파일 | 변경 내용 |
| --- | --- |
| `tools/daemon/cli.js` | `upgrade` command parsing, help text 갱신 |
| `tools/start-runtime.js` | `parsedArgs.command === 'upgrade'` dispatch 추가 |
| `tools/daemon/upgrade.js` | GitHub API 조회, semver 비교, asset 선택, 다운로드, digest 검증, 압축 해제, updater script 생성 |
| `tools/daemon/upgrade.test.js` | 단위 테스트 |
| `package.json` | `pkg.scripts`에 `tools/daemon/upgrade.js` 추가 |
| `README.md` | `BuilderGate.exe upgrade` 사용법 추가 |
| `docs/srs/buildergate.srs.md` | 릴리스/업데이트 계약을 요구사항으로 추가할지 검토 |

처음부터 release workflow를 바꿀 필요는 없다. 현재 asset 이름과 archive 구조가 upgrade에 충분히 맞는다.

## 10. 테스트 계획

단위 테스트:

- `parseArgs(['upgrade'])`가 upgrade command를 반환한다.
- `upgrade`가 첫 번째 인자가 아니면 실패한다.
- version compare: `0.2.0 < 0.3.0`, `0.10.0 > 0.9.0`, 동일 버전 no-op.
- GitHub release filtering: draft/prerelease/malformed tag skip.
- platform/arch asset 선택.
- digest mismatch 실패.
- archive top-level directory validation.
- path traversal archive entry 차단.
- running daemon 감지 시 영어 stop 안내와 exit code 확인.
- Windows updater script가 보존/교체 목록을 올바르게 포함하는지 snapshot test.

통합 테스트:

- fake GitHub API server로 release JSON과 asset stream 제공.
- 임시 install root에서 zip 다운로드/해제/staging 검증.
- server running fixture에서 upgrade가 중단되는지 검증.
- stopped fixture에서 updater script 생성까지 검증.

수동/CI 테스트:

- Windows packaged `BuilderGate.exe upgrade` smoke.
- 실행 중 daemon에서 정확히 stop 안내 후 종료.
- stopped 상태에서 최신 버전 없음 메시지 확인.
- 더 높은 test release에서 다운로드/교체 성공 확인.
- 교체 후 `BuilderGate.exe --help`, `BuilderGate.exe -p <port>`, `/health`, `BuilderGate.exe stop` 확인.

## 11. 최종 판단

이 기능은 현재 BuilderGate 구조에서 구현 가능하다. 특히 최근 빌드 구조가 `BuilderGate.exe`, `web/`, `shell-integration/` 중심으로 정리되어 updater의 교체 대상이 명확해졌다.

가장 중요한 설계 결정은 다음이다.

1. 실행 중 서버는 자동 stop하지 않고, 사용자가 명시적으로 `{실행 파일} stop`을 수행하도록 요구한다.
2. Windows에서는 `upgrade` 프로세스가 자기 EXE를 직접 교체하지 않는다.
3. 다운로드 파일은 GitHub API의 asset metadata와 SHA-256 digest로 검증한다.
4. `config.json5`, `runtime/`, `certs/`, `data/`는 보존한다.

이 조건을 지키면 `BuilderGate.exe upgrade`는 안정적으로 도입할 수 있다.
