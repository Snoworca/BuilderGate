# Workspace/Session/Grid UI Implementation Architecture Research

Date: 2026-07-08
Status: follow-up research draft for implementation planning
Scope: workspace list reorder, session move-to-workspace UX, grid vertical columns layout, running badge color

## 1. User Requirement Interpretation

The request has four implementation-level requirements.

1. In the workspace list, holding the left mouse button, or a finger on mobile, should enter a drag state. Moving the item up/down should reorder workspaces.
2. In a session window context menu, add a "워크스페이스 이동" item. Selecting it opens an existing modal-style component and lets the user choose a target workspace. Full workspaces must appear dimmed and be unavailable. Selecting a valid workspace moves the session there and switches the UI to that workspace. Ended sessions must not be able to use this feature.
3. In grid mode, the existing top-left per-session toolbar should add a vertical-columns layout command. Selecting it should arrange all windows as equal-width tall columns, visually like `||||`.
4. The current "in progress/running" workspace badge should use an orange background rather than green.

The follow-up implementation research focuses on maintainable architecture, not direct code changes.

## 2. Research Topology

The follow-up research was split into five review roles. This section records process provenance for the report; product architecture claims are supported separately in the code-evidence sections below.

1. Architecture triage agent: bounded contexts, phase plan, cross-cutting risks.
2. Backend/API agent: `WorkspaceService`, routes, WebSocket protocol, persistence and validation.
3. Frontend state/UX agent: `App`, `useWorkspaceManager`, context menu, modal reuse.
4. Grid/drag agent: `useDragReorder`, Mosaic toolbar/layout persistence, `||||` representation.
5. Document reviewer/evaluator agent: reviewed this document for unsupported claims, missing constraints, contradictions, and implementation-risk gaps.

The fifth review required revisions around grid persistence, move error/rollback contracts, context-menu target ownership, and storage migration. Those revisions are incorporated below.

## 4. Recommended Architecture

### 4.1 Workspace List Reorder

Reuse the existing reorder persistence pipeline. The missing piece is vertical pointer math.

Recommended implementation:

- Extend `useDragReorder` with an explicit axis option, for example `axis?: 'x' | 'y'`, defaulting to `'x'`.
- For workspace list use `axis: 'y'`, `clientY`, `rect.top`, and `rect.height / 2`.
- Use explicit insertion targets from `0..items.length`, rather than the current first-position shortcut, so moving to the first or last slot is deterministic.
- Move the ghost on `top` for vertical drag and preserve horizontal ghost behavior for the tab bar.
- Add pointer cleanup for `pointercancel`, pointer ID tracking, and optional pointer capture.
- Suppress the click that follows a completed drag so a reorder does not also select the workspace unintentionally.

Server-side hardening should be included with the feature:

- Validate `workspaceIds` as an exact permutation of known workspace IDs.
- Reject duplicate, missing, unknown, or partial arrays.
- Apply the same exact-set validation to `reorderTabs`, because the same silent-ignore pattern exists there.

### 4.2 Session Move To Workspace

Do not implement move as `delete + add`, `updateTab + reorderTabs`, or frontend-only state manipulation. Move is a domain operation that must preserve:

- `tab.id`
- `sessionId`
- PTY process
- tab metadata
- lifecycle and recovery metadata

Recommended backend contract:

```ts
POST /api/workspaces/:sourceWorkspaceId/tabs/:tabId/move
body: { targetWorkspaceId: string }
```

Recommended service method:

```ts
WorkspaceService.moveTab(sourceWorkspaceId, tabId, targetWorkspaceId): Promise<MoveTabResult>
```

Suggested result/event payload:

```ts
interface MoveTabResult {
  tab: WorkspaceTab;
  sourceWorkspaceId: string;
  targetWorkspaceId: string;
  sourceActiveTabId: string | null;
  targetActiveTabId: string | null;
  sourceTabIds: string[];
  targetTabIds: string[];
}
```

Server validation rules:

- Source workspace exists.
- Target workspace exists.
- Target is not the same workspace.
- Tab exists and belongs to the source workspace.
- Target tab count is below `maxTabsPerWorkspace`; this count should mirror server behavior and include all tabs in the workspace, including stopped or disconnected tabs.
- Do not reject based on `maxTotalSessions`, because moving does not create a new session.
- Reject ended sessions when `lifecycleState === 'stopped'`, `recoverable === false`, or the live session is missing from `SessionManager`.

Recommended error/status contract:

- Unknown source/target workspace: existing `WORKSPACE_NOT_FOUND`, HTTP 404.
- Unknown or foreign tab: existing `TAB_NOT_FOUND`, HTTP 404.
- Full target workspace: existing `TAB_LIMIT_EXCEEDED`, HTTP 409.
- Same source/target workspace: add `INVALID_WORKSPACE_MOVE`, HTTP 400.
- Ended, non-recoverable, or missing live session: add `SESSION_NOT_MOVABLE`, HTTP 409.
- Invalid workspace/tab reorder payloads: add `INVALID_REORDER_PAYLOAD`, HTTP 400.

Server mutation rules:

- Move tab to target by changing `tab.workspaceId`.
- Append moved tab to target and reindex target `sortOrder`.
- Reindex source `sortOrder`.
- Set target `activeTabId = movedTab.id`.
- If moved tab was source active tab, set source active tab to a remaining sorted tab or `null`.
- Update both workspace `updatedAt`.
- Save once after the complete state transition.
- Snapshot source workspace, target workspace, and moved tab before mutation. If `save()` fails, restore the snapshots before rethrowing. This follows the existing save-failure pattern used by tab delete lifecycle handling in `WorkspaceService`.
- Broadcast a dedicated `tab:moved` event.

WebSocket strategy:

- Add `tab:moved` to both server and frontend protocol unions.
- Originating REST caller is already excluded from WS broadcasts through `x-client-id`; therefore `useWorkspaceManager.moveTab` must update local state from the REST response.
- Other clients should apply the `tab:moved` event without switching their active workspace.
- The initiating client should switch active workspace to the target after a successful move.
- Frontend handlers for `workspace:reordered`, `tab:reordered`, and `tab:moved` should defensively validate exact ID sets before replacing local order. Current partial-array handling can drop local items, so the move implementation should not expand that pattern.

### 4.3 Move UX And Modal

Use `WindowDialog`, not an ad hoc overlay. It already implements modal behavior, focus handling, and portal rendering in `frontend/src/components/dialog/WindowDialog.tsx:21-274`.

Recommended frontend shape:

- Add `workspaceApi.moveTab(sourceWorkspaceId, tabId, targetWorkspaceId)`.
- Add `moveTab` to `UseWorkspaceManagerReturn`.
- Add App state: `pendingWorkspaceMove: { tabId: string; sourceWorkspaceId: string } | null`.
- Add a concrete `WorkspaceMoveDialog` component using `WindowDialog`.
- Add optional move metadata to `BuildTerminalMenuOptions`, so both tab mode and grid mode use the same menu item.
- Pass the request handler into `MosaicContainer` so grid context menus can request the same App-owned modal.

Important UI rules:

- Use the right-click target tab, not necessarily the currently active tab. `App.tsx:388-400` currently builds menu actions using `activeTab`; move implementation should resolve `tabContextMenu.targetId` to the actual target tab for all target-sensitive actions added or touched by this work.
- Disable the menu item when the target tab is ended: `status === 'disconnected' || lifecycleState === 'stopped' || recoverable === false`.
- If disabled context menu items without an `onClick` are normalized away, provide a no-op handler so the disabled item remains visible if that is the intended UX.
- In the dialog, sort workspaces by `sortOrder`.
- Dim and disable the current workspace.
- Dim and disable full workspaces where `tabCount >= maxTabsPerWorkspace`, counting all tabs the same way the server does.
- Selecting a valid workspace should close the dialog only after the move succeeds, switch to the target workspace, and make the moved tab active there.

The current UI hardcodes `maxTabs` as `8` in multiple places. For maintainability, either centralize this frontend constant immediately or expose server workspace config later. Server remains the source of truth.

### 4.4 Grid Vertical Columns Layout (`||||`)

The safest design is not a broad new top-level `LayoutMode`. Keep the top-level mode contract as `equal | focus | auto | none`, and add a narrow equal-layout preset/submode:

```ts
type EqualLayoutPreset = 'smart' | 'columns';
```

Rationale:

- `||||` is equal layout with a forced single row of equal-width columns.
- Existing code and tests already attach behavior to top-level `equal`, including drag reorder and normalization.
- Adding another top-level mode increases blast radius across `useMosaicLayout`, `MosaicToolbar`, persistence, E2E helpers, and mode toggling.
- Persisting only the tree is not enough because equal-mode reload currently normalizes recovered equal layouts.

Recommended grid changes:

- Export a helper that creates a single-row Mosaic tree with `direction: 'row'`, preserving leaf order and equal split percentages.
- Persist optional `equalPreset` alongside `mode` and `focusTarget` in `mosaicLayoutStorage`; this requires a schema-version migration from the current version `1` shape.
- Keep `mode = 'equal'` and `equalPreset = 'columns'` for `||||`.
- Add a distinct toolbar command and data attribute, for example `data-layout-command="columns"`.
- Do not wire the button through plain `onLayoutModeChange('equal')`, because the existing handler toggles active equal back to `none`.
- Make active state distinguish normal equal from columns equal.
- On tab add/remove/reload, rebuild using the persisted preset so `||||` remains `||||`.
- Mention in tests and docs that current equal mode can already produce a single row on ultrawide screens; the missing behavior is a forced columns preset independent of viewport heuristics.
- On manual resize, preserve the existing behavior decision deliberately: either exit to `none`, or only exit columns if the user resizes. Document the chosen behavior in the implementation plan.

### 4.5 Grid Persistence And Move Layouts

Moving a tab between workspaces must also define layout behavior. The current code has localStorage-based active Mosaic persistence and a server `gridLayouts` model that is not clearly kept in sync by the current `MosaicContainer` render path.

Recommended immediate behavior:

- Treat `mosaicLayoutStorage` as the source of truth for visible grid layout mode/preset until a separate grid synchronization cleanup is planned.
- On move success, prune the moved tab from the source workspace localStorage layout, similar to `pruneMosaicLayoutForDeletedTab`, without clearing terminal snapshots because the session is still alive.
- For the target workspace, either append the moved tab to the persisted layout or rebuild the target tree according to its persisted `mode` and `equalPreset`. Prefer rebuild for `equal`/`columns` because that preserves mode semantics.
- If the target workspace has no persisted layout, let `useMosaicLayout` recover/build from the target tab list.
- Keep server `gridLayouts` consistent when this implementation touches `useWorkspaceManager.gridLayouts`; if not updated, document that runtime residency falls back to all active workspace tabs when server layout is stale or incomplete.
- Add tests that move a tab from grid source to grid target and verify stale source leaves are removed and target layout remains valid after reload.

### 4.6 Running Badge Orange

Make a narrow visual change only in `WorkspaceItem`.

Recommended value:

- Change `backgroundColor: '#22c55e'` to an orange token/value such as `#f97316`.

Do not globally replace green, because other green UI carries different meanings.

## 6. Regression Test Plan

Backend:

- Move active tab to a non-full workspace; preserve `tab.id` and `sessionId`.
- Move idle tab; no PTY create/terminate call.
- Reject target full workspace with `TAB_LIMIT_EXCEEDED`.
- Reject stopped/non-recoverable/missing-session tabs.
- Reject same workspace, unknown workspace, foreign tab.
- Verify exact HTTP statuses for new `INVALID_WORKSPACE_MOVE`, `SESSION_NOT_MOVABLE`, and `INVALID_REORDER_PAYLOAD` errors.
- Simulate save failure and verify source workspace, target workspace, and moved tab rollback.
- Reindex source and target `sortOrder`.
- Update source and target `activeTabId`.
- Broadcast `tab:moved` with expected payload.
- Reject invalid workspace/tab reorder arrays.

Frontend unit:

- `contextMenuBuilder` includes `워크스페이스 이동` when available.
- It disables move for ended tabs.
- `WorkspaceMoveDialog` dims and disables full/current workspaces.
- `useWorkspaceManager.moveTab` applies REST response locally and switches initiating client to target workspace.
- WS `tab:moved` applies state without switching other clients.
- Reorder/move event handlers reject invalid partial or duplicate ID payloads.
- `useDragReorder` keeps horizontal behavior and supports vertical target math.
- `mosaic` columns preset creates a single row with equal widths for 1-8 tabs.
- `mosaicLayoutStorage` migrates and preserves `equalPreset`.
- Move between grid workspaces removes stale source leaves and keeps target layout valid.

E2E/manual at `https://localhost:2222`:

- Desktop workspace long-press reorder up/down and reload persistence.
- Mobile/finger long-press reorder without accidental scroll/click behavior.
- Tab-mode context menu move opens modal, disables full target, moves session, switches workspace.
- Grid-mode context menu move uses the same modal.
- Ended session cannot use move.
- `||||` toolbar command arranges all grid panes as equal-width tall columns and persists across reload.
- Running count badge is orange and unrelated green states remain unchanged.

## 7. Key Risks

| Risk | Mitigation |
| --- | --- |
| Vertical workspace drag breaks tab-bar reorder | Add axis option with horizontal default and regression tests |
| Long-press drag selects/clicks workspace after drop | Suppress post-drag click and track pointer lifecycle |
| Frontend-only move loses PTY/session state | Use server atomic `moveTab` only |
| UI-only full/ended checks are bypassed | Repeat checks in `WorkspaceService.moveTab` |
| `save()` fails after partial in-memory move mutation | Snapshot and restore source workspace, target workspace, and tab |
| `tab:moved` local state diverges on originating client | Apply REST result locally because WS excludes origin client |
| Moved tabs leave stale Mosaic leaves | Prune source layout and rebuild/append target layout under the persisted preset |
| Grid `||||` reloads as normal equal grid | Persist equal preset/submode, not just tree |
| Badge color change hits unrelated status colors | Edit only `WorkspaceItem` running-count badge |

## 8. Open Decisions For Implementation

1. Expose server workspace limits to frontend now, or centralize the existing frontend `8` constant until a broader config contract is added.
2. Choose source `activeTabId` fallback after moving the active tab: first sorted remaining tab is simplest; adjacent tab matching close behavior may feel better.
3. Decide whether a manual resize exits `equalPreset: 'columns'` to `none`, mirroring current equal resize behavior.
4. Decide whether disabled "워크스페이스 이동" remains visible for ended sessions, or is hidden. The user only requires that ended sessions cannot use it; visible-disabled gives clearer feedback.
5. Decide whether this feature should also synchronize server `gridLayouts`, or keep localStorage as the visible grid layout authority and only guard runtime residency against stale server layouts.

## 9. Current Recommendation Summary

Implement the feature as additive contracts:

- Server owns session move validity and persistence.
- Frontend owns modal orchestration and initiating-client navigation.
- Existing context menu builder remains the shared source for terminal/window context menu items in tab and grid surfaces.
- Existing drag hook is extended with an axis option instead of duplicating reorder logic.
- `||||` is persisted as an equal-layout preset, not as a broad new top-level grid mode.
- Running badge orange change is scoped to the workspace running count badge.

This design keeps state transitions explicit, avoids fake session recreation, and limits UI changes to the surfaces required by the request.
