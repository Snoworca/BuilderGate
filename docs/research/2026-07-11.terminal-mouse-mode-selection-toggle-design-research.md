# 마우스 모드 감지 + "선택 모드 토글" 심층 설계 연구

Date: 2026-07-11
Scope: 프론트엔드 웹 터미널(xterm.js v6) — 전용 REQ 없음 (§9 SRS 추적 참고)
선행 문서(근본 원인): `docs/research/2026-07-11.claude-code-mouse-tracking-copy-disabled-research.md`
성격: **구현 설계 사전조사(코드 미수정)**. 실제 구현·REQ 등록은 사용자 결정 후 진행.

## 1. 배경과 목적

선행 조사에서 "Claude Code 실행 시에만 드래그 선택·Ctrl+C·우클릭 복사가 모두 막히는" 근본 원인이 **애플리케이션 마우스 트래킹 모드**(DECSET `?1000/?1002/?1003h`)임을 확정했다. 제안된 해결책 중 가장 저비용인 **Shift+드래그**를 사용자가 실제 시도한 결과 **"되었다 안 되었다" 하는 불안정성**을 보고했다.

이에 해결책 후보 (2) **"마우스 모드 감지 + 선택 모드 토글"**을 구현 설계 수준까지 심층 연구했다. 이 문서는 (a) Shift+드래그가 왜 불안정한지의 진짜 원인, (b) 마우스 모드 감지 방법, (c) 선택 강제 메커니즘 기법 비교, (d) 업계 벤치마크, (e) BuilderGate 통합 지점을 종합한다.

## 2. 조사 방법

4개 서브에이전트를 독립 축으로 병렬 투입했다(확증편향 방지를 위해 서로의 결론을 공유하지 않음). 모든 결론은 xterm.js **6.0.0** 소스(`frontend/node_modules/@xterm/xterm/src/**`) 정독 기반이며, dev 서버 미기동으로 런타임 실측은 별도 과제로 남긴다.

| Agent | 축 | 핵심 결과 |
|---|---|---|
| 1. 감지(Detection) API | xterm 공개/비공개 신호 | `onWriteParsed` + `modes.mouseTrackingMode` diff가 100% 공개 API 권장 전략. `mouseTrackingMode !== 'none'` ⟺ 선택 비활성 상태(증명 가능 등가). |
| 2. 강제(Enforcement) + Shift 규명 | 선택 강제/억제 기법, 불안정성 근본 | **진짜 주범은 "마우스 모드 재천명(re-assert) clear"**. 기법 B(프로토콜 NONE 토글 + 출력 일시정지) 1순위. |
| 3. 업계 벤치마크 | 타 터미널의 처리 방식 | 수정자 bypass가 지배적이나 웹에선 불안정. 명시적 토글(tmux copy-mode 3요소) 권장. |

## 3. 핵심 발견 — Shift+드래그 "되었다 안 되었다"의 진짜 원인 (Agent 2)

### 3.1 예상과 달랐던 전제: 이 앱은 `disableStdin: true`

`TerminalView.tsx:1703`은 xterm을 `disableStdin: true`로 생성한다. 그 결과 `@xterm/xterm` 코어의 `CoreService.triggerDataEvent`가 `onUserInput`을 발화하기 전에 조기 return한다(`common/services/CoreService.ts:63-65`). 이 앱은 마우스 리포트를 PTY로 보내지 않고 자체 입력 파이프라인(`onInput` prop + helper textarea)을 쓰기 때문이다.

> **정정 (2026-07-11 후속 우클릭-복사 조사)**: 위 문단의 "이 앱에서는 `onUserInput → clearSelection` 경로가 무력화되어 있다"는 단정은 **부정확하다.** `disableStdin`은 생성자 값(`true`)에 고정되지 않고, `syncInputReadiness`가 매 게이트 동기화마다 `term.options.disableStdin = !gate.captureAllowed`로 덮어쓴다(`TerminalView.tsx:749`). 정상 사용 구간(터미널 가시·활성)에서는 `captureAllowed=true` → `disableStdin=false`이므로 `onUserInput → clearSelection` 경로가 **실제로 살아 있다.** 그 결과 `?1003h`(모든 이동 리포트) 모드에서 선택을 끝낸 뒤 마우스를 조금만 움직여도(=우클릭하러 이동) 그 이동 리포트가 SGR 인코딩으로 `onUserInput`을 발화해 선택을 지운다 — 이것이 우클릭 '복사'가 (마우스 이동이 없는 Ctrl+C와 달리) **일관되게** 실패하는 주된 이유다(재천명 clear §3.2는 이차 기여). 이 정정은 본 문서의 결론(선택 모드 토글이 우월)을 바꾸지 않고 오히려 강화한다. 실제 채택된 저비용 수정(우클릭 복사 활성화)은 `onSelectionChange` 미러링이며 `frontend/src/utils/terminalSelectionMirror.ts` + `TerminalView.tsx` 통합으로 구현되었다.

### 3.2 진짜 주범: 마우스 모드 재천명(re-assert)에 의한 선택 파괴

Claude Code 같은 라이브 TUI는 리페인트·스트리밍·커서 갱신·포커스 이벤트마다 `?1000h/?1002h/?1003h`를 **반복 재전송**한다. xterm은 이를 받을 때마다:

1. `InputHandler`가 **무조건** `activeProtocol = 'ANY'` 설정 (이미 ANY여도 조건 검사 없음) — `common/InputHandler.ts:1930-1933`
2. setter가 **값이 안 바뀌어도 무조건** `_onProtocolChange.fire(events)` — `common/services/CoreMouseService.ts:211-217`
3. 핸들러가 events truthy면 **무조건** `SelectionService.disable()` — `browser/CoreBrowserTerminal.ts:734`
4. `disable()`의 첫 줄이 `this.clearSelection()` — `browser/services/SelectionService.ts:173-176`

→ **TUI가 마우스 모드를 재전송하는 매 순간, 진행 중이거나 완성된 선택이 `clearSelection()`으로 지워진다.** "되었다 안 되었다"는 정확히 **선택을 끝낸 뒤 복사(Ctrl+C/우클릭)하기 전에 TUI가 화면을 한 번이라도 다시 그리는지 여부**에 달려 있다. TUI가 잠깐 idle이면 선택이 살아남아 "됨", 스피너/스트리밍/커서 갱신으로 리페인트하면 재천명 → clear → "안 됨". 라이브 에이전트 TUI의 간헐적 출력과 정확히 부합한다.

### 3.3 부차 원인: shiftKey는 mousedown 단일 시점 게이트

`shouldForceSelection`(비-Mac에서 `event.shiftKey`)은 **오직 mousedown 시점에만** 검사된다(`SelectionService.ts:463-470`, `CoreBrowserTerminal.ts:786`). mousemove/mouseup에서는 재검사되지 않는다. 따라서 클릭보다 조금이라도 늦게 Shift를 누르면 선택이 아예 시작되지 않는다(사용자 타이밍 의존). 이것도 간헐적 실패에 기여하지만, 3.2의 재천명 clear가 지배적 주범이다.

### 3.4 왜 "토글"이 shiftKey 주입보다 구조적으로 우월한가

- shiftKey를 완벽히 눌러 선택을 시작해도, **3.2의 재천명 clear는 그대로 남는다.** shiftKey 주입은 3.3(타이밍)만 부분 해결할 뿐 3.2를 못 막는다.
- 진짜 안정화의 핵심은 **"선택하는 동안 마우스 프로토콜 상태를 동결(freeze)해 재천명이 `disable()/clearSelection()`을 호출하지 못하게 하는 것"**이며, 이는 매-이벤트 수정자로는 원리적으로 불가능하고 **지속 상태(토글) + 출력 일시정지**에서만 달성된다.

## 4. 감지(Detection) 설계 (Agent 1)

### 4.1 증명 가능한 등가식

`terminal.modes.mouseTrackingMode !== 'none'` ⟺ `coreMouseService.areMouseEventsActive === true` ⟺ **SelectionService가 disable된 상태**.
근거: `NONE` 프로토콜만 `events === 0`이고(`common/services/CoreMouseService.ts` Types), SelectionService의 enable/disable는 오직 `onProtocolChange`의 events 값으로만 토글된다(`browser/CoreBrowserTerminal.ts:734,738`). 즉 공개 API `mouseTrackingMode`가 선택 비활성 상태와 정확히 일치한다.

### 4.2 감지 방법 평가

| 방법 | 공개 API | 갱신 타이밍 | 신뢰성 | 버전 취약성 |
|---|---|---|---|---|
| `modes.mouseTrackingMode` 읽기 | **완전 공개** | 라이브/동기(매 접근 재계산), 자체 통지 없음 | 높음 | 낮음 |
| `onWriteParsed` + 위 값 diff | **완전 공개** | 각 write 청크 파싱 후(사실상 즉시) | 높음 | 낮음 |
| `_core.coreMouseService.onProtocolChange` | **비공개**(`_core` private) | 최조기(xterm 내부와 동일 시점) | 최고 | **높음** |
| MutationObserver(`enable-mouse-events` 클래스) | element 공개/클래스명 반공개 | 마이크로태스크 비동기 | 높음 | 중간 |
| `buffer.active.type === 'alternate'` | 공개 | 버퍼 전환 시 | **낮음(오답)** | 낮음 |

**중요**: 마우스 모드 변경을 통지하는 **공개 이벤트는 존재하지 않는다**(`onModeChange` 부재 확정). alt-screen 프록시(`buffer.active.type`)는 마우스 모드와 별개 DECSET이라 거짓 양성/음성이 있어 **1차 신호로 사용 금지**.

### 4.3 권장 감지 전략

- **1차(공개 API 전용)**: `terminal.onWriteParsed` 구독 → 매 발화 시 `terminal.modes.mouseTrackingMode`를 읽어 직전 캐시값과 diff → `'none'` ↔ 비-`'none'` 전이 시 UI 신호 emit.
- **초기화 필수**: `term.open()`/첫 write 직후 1회 값을 읽어 초기 상태를 세팅(구독 이전 조기 시퀀스로 이미 켜졌을 수 있음).
- **회피**: `_core...onProtocolChange`(비공개 취약)와 `buffer.active.type`(의미상 오답)는 1차로 쓰지 않는다.

## 5. 강제(Enforcement) 메커니즘 기법 비교 (Agent 2)

유효한 기법은 **(1) 로컬 선택 허용 + (2) 선택 수명 동안 재천명 clear 차단**을 모두 만족해야 한다. (PTY 전송 차단은 `disableStdin`이 이미 담당하므로 부차적.)

| 기법 | 실현성 | 재천명 clear 대응 | 침습성 | 버전 취약성 |
|---|---|---|---|---|
| **A** capture-phase shiftKey 주입 | 높음 | **낮음**(3.3만 해결, 3.2 미해결) | 낮음(공개 DOM) | 낮음 |
| **B** `activeProtocol='NONE'` 토글 + 출력 일시정지 | 높음 | **높음** | 중간(`_core`, setter는 공개) | 중간 |
| **C** `enable-mouse-events` 클래스 제거만 | — | **효과 없음** | 낮음 | — |
| **D** `_selectionService.enable()` 직접 호출 | 중간 | 중간(재천명 시 재-disable) | **높음**(private) | 중간 |
| **E** 자체 오버레이 선택 레이어 | 낮음(구현량 큼) | **최고**(xterm 선택 미사용→면역) | 낮음~중간 | 낮음 |

### 5.1 최유력: 기법 B — `activeProtocol='NONE'` 임시 토글 + 출력 일시정지 (1순위)

- 진입 시 프로토콜을 `NONE`으로 만들면 `onProtocolChange(0)` 발화 → `enable()` + `enable-mouse-events` 클래스 제거 + 상주 hover 리스너 제거(`CoreBrowserTerminal.ts:735-745`). 선택 서비스가 **일반(마우스 모드 아님) 터미널과 동일 상태**가 되어 Shift 없이 평범한 드래그로 선택된다.
- **필수 보완**: 선택 중 TUI가 `?1003h`를 재전송하면 다시 `activeProtocol='ANY'`로 되돌아가 clear된다. 그러므로 **선택 모드 동안 PTY 출력을 xterm에 쓰지 않고 버퍼링**해야 한다. 이 앱은 이미 `bufferedOutputRef`(`TerminalView.tsx` 출력 파이프라인)를 보유하므로 자연스럽게 붙고, "선택 중 화면 동결" UX와도 일치한다. 출력을 멈추면 재천명 자체가 발생하지 않는다.
- 접근성: `coreMouseService`는 실질 공개 멤버이고 `activeProtocol` setter도 공개. 현재 프로토콜은 공개 `term.modes.mouseTrackingMode`로 읽어 저장/복원한다. `_core` 접근이 유일한 반-비공개 지점이다.

### 5.2 대안: 기법 E — 자체 오버레이 선택 레이어 (견고성 최우선 시 2순위)

- xterm의 SelectionModel을 안 쓰므로 `clearSelection()`/재천명에 **원천 면역**. 이 앱은 `disableStdin`으로 마우스가 어차피 PTY로 안 나가니, E의 임무는 순수 로컬(좌표→셀 매핑→오버레이 하이라이트→복사)이다. `SerializeAddon`이 이미 로드되어 있어 텍스트 추출에 활용 가능.
- 비용: wide-char/emoji/wrapped-line 처리, 스크롤/트림 동기화를 직접 재구현해야 해 구현·검증량이 크다.

### 5.3 기각된 기법

- **A(shiftKey 주입) 단독**: 3.2 재천명 clear를 못 막아 부적합. 단, "Shift 없이 선택 시작"이라는 인체공학 개선으로 B와 조합은 가능.
- **C(클래스 제거)**: `enable-mouse-events` 클래스는 CSS에서 `cursor: default`만 준다(`css/xterm.css`). JS 게이팅은 이 클래스를 참조하지 않아 **무효**.
- **D(`enable()` 직접)**: private `_selectionService` 접근에 재천명 시 재-disable까지 겹쳐 B의 열등 변형.

## 6. 업계 벤치마크 (Agent 3)

| 도구 | 패턴 | 수정자/토글 | 비고 |
|---|---|---|---|
| VS Code 통합 터미널(xterm.js) | 수정자 bypass + 우클릭 커스텀 | Alt(Win/Linux), Option+`macOptionClickForcesSelection`(Mac) | 발견성 낮음, Mac 기본 off |
| tmux | **명시적 copy-mode 토글** + Shift bypass | `prefix+[` 진입 | 성숙한 레퍼런스(3요소) |
| iTerm2 / Windows Terminal / kitty / Alacritty | 수정자 bypass | Option(Mac) / Shift | Alacritty는 커서 모양 변화로 상태 표시 |
| **ttyd(xterm.js 웹)** | Shift bypass가 **브라우저 의존·불안정** → tmux mouse off 토글이 확실 | — | **우리 상황과 동일**([ttyd #218]) |
| Warp | 수정자 bypass | Shift | copy-on-select가 TUI에서 미동작(미해결) |

- **핵심 교훈**: 웹에서는 수정자 bypass 단독이 취약하다(크로스플랫폼 수정자 불일치 + 브라우저 가로채기 + 최악의 발견성). tmux `copy-mode`의 **3요소 설계 — (1) 진입 단축키, (2) 상시 시각 상태 표시, (3) 복사 후 자동 해제 —** 가 명시적 토글의 성숙한 레퍼런스다.
- **중요 제약**: xterm.js는 마우스 리포팅을 무조건 끄는 1급 public API를 **의도적으로 두지 않았다**([xtermjs discussion #4320] — 메인테이너: "unconditionally disabling mouse tracking may break app side"). 따라서 토글은 "리포팅을 영구히 끄는" 것이 아니라 **선택 수명 동안만 프로토콜 상태를 게이팅**하는 방식(§5.1 기법 B)이 안전하다.
- 참고 출처: [xtermjs/xterm.js #4329](https://github.com/xtermjs/xterm.js/issues/4329), [#4320](https://github.com/xtermjs/xterm.js/discussions/4320), [ttyd #218](https://github.com/tsl0922/ttyd/issues/218), [VS Code Terminal Basics](https://code.visualstudio.com/docs/terminal/basics), [tmux copy-mode(seanh.cc)](https://www.seanh.cc/2020/12/27/copy-and-paste-in-tmux/).

## 7. BuilderGate 통합 지점 (Agent 4)

1. **handle 3계층 위임 구조** — 신규 메서드는 세 곳에 한 줄씩:
   - 정의+구현: `TerminalView.tsx`의 `TerminalHandle`(L164-185)과 `useImperativeHandle`(L1516~). `getScreenRepairReadiness()`(pull-based, xterm 내부 상태 on-demand 조회) 선례를 그대로 따라 `getMouseTrackingMode()`/`setSelectionMode()` 추가 가능.
   - 위임: `TerminalContainer.tsx:1243-1263` (1:1 재노출).
   - 소비: `App.tsx`의 `terminalRefsMap`(L130) + 래퍼(L223-241).

## 8. 권장 설계 종합 (구현 전 사용자 결정 필요)

최소 침습·최대 안정 기준의 권장안:

1. **감지**: `onWriteParsed` + `modes.mouseTrackingMode` diff로 마우스 모드 활성 감지 → `onMouseTrackingModeChange` 콜백 emit + `containerRef`에 `mouse-tracking` CSS 클래스 토글. 마우스 모드일 때만 "선택 모드" 어포던스를 노출.
2. **선택 모드 강제**: 기법 **B**(진입 시 `activeProtocol='NONE'` + **출력 일시정지**via 기존 `bufferedOutputRef`, 종료 시 프로토콜 복원 + 버퍼 flush). 이것이 3.2 재천명 clear와 3.3 타이밍을 동시에 해결하는 유일한 저비용 경로.
3. **UX**(tmux 3요소 이식): 컨텍스트 메뉴 토글(진입) + CSS 클래스 배지(상시 시각 표시, "선택 중 화면 동결"과 시각적으로 일치) + 복사 직후 자동 해제(출력 재개).
4. **보조**: 우클릭 copyPaste 동작 강화, Shift/Alt bypass는 파워유저용 보조로 유지(단독 신뢰 금지).
5. **후순위**: 단축키 토글(action union 확장 선행), 기법 E(견고성 최우선 시 재검토).

이 설계는 "선택하는 동안 잠깐 화면이 멈추고(출력 버퍼링), 복사하면 다시 흐른다"는 예측 가능한 모델을 제공하며, xterm 자체 프로토콜 머신을 재사용해 침습을 최소화한다.

## 9. 미검증·리스크

1. **재천명 빈도 실측 미완** — Claude Code가 실제로 `?1003h`를 얼마나 자주 재전송하는지는 이 저장소 밖이라 미검증(메커니즘 자체는 소스로 확정). **실측 권장**: dev 서버(**`https://localhost:2222`**, `node dev.js --port 2222`)에서 `claude` 실행 → 텍스트 선택 후 TUI가 출력하는 순간 선택이 사라지는지 관찰. 동시에 Shift+드래그 안정성, `term.modes.mouseTrackingMode` 값도 함께 확인.
2. **기법 B의 `_core` 접근** — 반-비공개(setter는 공개). feature-detection 가드 권장. xterm 메이저 업그레이드 시 재검증 필요.
3. **출력 일시정지의 부작용** — 선택 중 화면이 멈추므로 장시간 선택 시 백프레셔/버퍼 상한 고려(기존 `terminalOutputScheduler`/resource limit과 상호작용 확인 필요).
4. **`Object.defineProperty(ev,'shiftKey',...)`(기법 A 조합 시) 브라우저 호환성** — 타깃 브라우저(특히 모바일 WebView) 확인 권장.
5. **커스텀 마우스 프로토콜 애드온** — `events=0` 프로토콜 등록 시 §4.1 등가식이 깨지나, 본 코드베이스엔 미관측.

## 10. SRS 추적 메모

- 이 기능(마우스 모드 감지 + 선택 모드 토글)을 직접 다루는 기존 Requirement는 없다(`docs/spec` 전체에 관련 REQ 부재).
- 본 문서는 **순수 설계 사전조사이며 코드 수정·SRS 변경을 수행하지 않았다.**
- 실제 구현 착수 전, 신규 REQ 등록(`/kiwi-srs`)으로 요구사항을 먼저 명세해야 한다(프론트엔드 selection/copy UX 신규 기능). 기법 선택(B vs E), 출력 일시정지 UX, 단축키 포함 여부는 사용자 결정 사항이다.
