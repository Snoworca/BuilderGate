# Mosaic 드래그 시작 시 원본 창 축소 현상 연구

Date: 2026-04-22
Scope: `react-mosaic-component`의 grid 드래그 시작 동작
Status: research only

## 질문

사용자가 창을 드래그하기 시작하면, 원본 pane이 즉시 쪼그라듭니다. 이 현상은 모드와 무관하게 발생합니다. 왜 이런 일이 일어나며, 꼭 필요한 동작입니까?

## 결론

이 현상은 vendor의 drag-start 경로가 여전히 source node에 대해 `hide()`를 호출하고 있기 때문입니다. 그 결과 drop이 완료되기 전에 live mosaic tree 자체가 변형됩니다.

이 동작은 기술적으로 필수는 아닙니다.

즉, Step26에서 drop semantics만 바꾸고, 기존 `react-mosaic`의 drag-start hide 동작은 그대로 남겨 두었기 때문에 발생한 현상입니다.

## 근본 원인

### 1. 드래그 시작 시 source node를 여전히 hide하고 있음

패치된 vendor 파일에서 drag start는 여전히 다음 경로를 탑니다.

- [MosaicWindow.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\MosaicWindow.js:275)

여기서 실제로 만들어지는 호출은 다음입니다.

- `defer(() => mosaicActions.hide(props.path))`

즉, source tile은 단순히 drag preview만 띄우는 것이 아니라, 드래그 시작 순간 live layout tree 자체가 바뀝니다.

### 2. `hide()`가 parent split 비율을 강제로 접어 버림

hide 구현은 다음에서 옵니다.

- [mosaicUpdates.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\util\mosaicUpdates.js:150)

`createHideUpdate()`는 parent의 `splitPercentage`를 `0` 또는 `100`으로 설정합니다.

그래서 source pane이 즉시 납작해지거나, 거의 보이지 않을 정도로 줄어드는 것처럼 보입니다.

### 3. Step26은 drop semantics만 바꿨고, drag-start semantics는 안 바꿨음

Step26에서 추가된 것은 다음입니다.

- reorder payload
- full-cell target
- root snapshot restore
- equal 전용 reorder gate

하지만 drag-start hide 동작은 그대로 유지했습니다.

그래서 현재 흐름은 다음과 같습니다.

1. drag 시작
2. source pane이 tree mutation으로 숨겨짐
3. preview/target이 표시됨
4. 나중에 drop 또는 restore 경로가 실행됨

즉, 사용자가 실제 이동을 완료하기도 전에 pane이 먼저 줄어드는 이유가 여기에 있습니다.

## 꼭 필요한가?

아니요.

기존 라이브러리 주석은 HTML DnD가 시작될 때 source element가 DOM에 존재해야 한다는 점만 설명합니다. source를 즉시 hide해야 한다는 뜻은 아닙니다.

따라서 현재의 shrink는 필수 조건이 아니라, 기존 구현에서 이어져 온 선택일 뿐입니다.

## 수정 옵션

## Option A: equal 모드에서만 고치기

`reorderEnabled === true`일 때는 drag start에서 `mosaicActions.hide(props.path)`를 호출하지 않도록 vendor 경로를 바꿉니다.

### 장점

- 패치 범위가 가장 작음
- 새 equal-mode reorder UX를 직접적으로 고침
- 파급 범위가 작음

### 단점

- non-equal 모드의 shrink는 그대로 남음
- 사용자는 이미 “모드와 무관하게” 문제가 있다고 보고했음

## Option B: 모든 window drag 경로에서 drag-start hide 제거

drag start 시점에는 mosaic tree를 전혀 변형하지 않습니다.

대신 다음처럼 바꿉니다.

1. source tile은 제자리에 둠
2. drag preview와 blue target은 그대로 사용
3. 성공적인 drop 시점에만 split 또는 reorder tree를 적용
4. invalid drop/reset은 진짜 no-op 또는 root-snapshot restore로 처리

### 장점

- 사용자가 보고한 문제를 모든 모드에서 해결함
- 요즘 DnD UX에 더 잘 맞음
- commit 전에 pane이 먼저 흔들리는 핵심 원인을 제거함

### 단점

- Option A보다 vendor 수정 범위가 큼
- split-mode invalid-drop 경로도 함께 수정해야 함
  이유: 기존 reset logic은 drag start 때 tree가 이미 숨겨졌다고 가정하기 때문

## Option C: 레이아웃은 유지하고, 시각만 숨기기

tree mutation 대신 source를 시각적으로만 표시합니다.

- opacity 낮추기
- outline
- ghost overlay
- content에만 `visibility: hidden`

### 장점

- 공간 배치가 안정적으로 유지됨
- 사용자가 “지금 이 pane을 끌고 있다”는 피드백을 여전히 받을 수 있음

### 단점

- UI state 관리가 늘어남
- 현재 hide/reset logic 교체는 여전히 필요함
- Option B보다 구현 요소가 더 많아질 수 있음

## 추천 방향

Option B가 맞습니다.

이유는 간단합니다.

- 사용자 불만은 모든 모드에 걸쳐 있음
- shrink의 원인은 reorder logic이 아니라 tree mutation임
- drag-start tree mutation 자체를 없애는 것이 구조적으로 가장 깨끗함

권장 구현 형태:

1. vendor `ConnectedInternalMosaicWindow`에서 drag start 시 `mosaicActions.hide(props.path)` 호출을 제거
2. `rootSnapshot` 캡처는 유지
3. 성공적인 drop 시:
   - split 모드는 `createDragToUpdates(...)`
   - reorder 모드는 `createReorderedTreePreservingShape(...)`
4. invalid drop 시:
   - 양쪽 branch 모두 `rootSnapshot` restore 또는, tree mutation이 없었다면 그냥 no-op
5. 필요하면 source에 비파괴적 drag 상태만 시각적으로 부여
   - opacity
   - outline
   - subtle ghost class

## 리스크 메모

### 1. split-mode reset 경로를 다시 설계해야 함

현재 split reset 경로는 drag start가 이미 tree를 바꿨다고 가정합니다.

- [MosaicWindow.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\MosaicWindow.js:341)

이 branch는 parent의 `splitPercentage`를 지우는 식으로 동작합니다.

drag-start hide를 제거한다면, 이 reset path도 예전 split-only 방식 대신 explicit restore 전략으로 통일하는 것이 맞습니다.

### 2. source와 preview가 동시에 보일 수 있음

대개는 허용 가능한 UX입니다. 다만 너무 산만하면 shrink 대신 source visual state를 적용하면 됩니다.

### 3. drag start 시 geometry stability 테스트가 필요함

현재 테스트는 drop 결과 검증은 잘 하고 있습니다.

이 변경을 넣을 때는 다음 회귀 테스트가 추가되어야 합니다.

1. source tile rect 측정
2. handle에서 drag 시작
3. drop 전에도 source width/height가 거의 변하지 않는지 검증

## 실무적 판단

현재 동작은 “왜 그런가”는 설명할 수 있지만, “원하는 UX인가”에는 그렇지 않습니다.

source shrink는 브라우저가 요구하는 것도 아니고, equal-mode reorder가 요구하는 것도 아니며, 사용자 요구사항에도 없습니다.

즉, upstream drag 구현에서 이어져 온 동작으로 보고 의도적으로 제거하는 것이 맞습니다.
