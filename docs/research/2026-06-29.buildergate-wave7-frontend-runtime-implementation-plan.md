# BuilderGate Wave7 Frontend Runtime Implementation Research And Plan

작성일: 2026-06-29

대상 문서: `docs/research/plug-leak/wave7.md`

## 목적

Wave7은 서버 안정화와 Settings 연결 이후, 브라우저 쪽 누적 비용을 줄이는 단계다. 핵심은 hidden terminal runtime을 무제한으로 유지하지 않고, hidden output을 xterm에 계속 쓰지 않으며, visible output과 browser WebSocket input path에 backpressure/recovery 계약을 붙이는 것이다.

이번 문서는 구현 전 조사 결과와 실행 계획이다. 코드 구현은 이 문서를 기준으로 별도 단계에서 진행한다.

## 조사 방식

요청에 따라 2단계 서브에이전트 조사를 수행했다.

- 조사 단계: 5개 에이전트가 runtime config, runtime residency, hidden output/scheduler, WebSocket backpressure, snapshot/layout cleanup을 분리 조사했다.
- 계획 단계: 5개 에이전트가 같은 축을 기준으로 구현 순서, 테스트, 위험, defer 범위를 제안했다.
- 메인 세션은 각 결과를 교차 확인하고 현재 dirty worktree diff를 확인했다.

## 현재 상태 요약

Wave7은 완전 신규 구현이 아니다. 현재 작업트리에는 이미 Wave7 일부 조각이 들어와 있다.

- `App.tsx`는 `useTerminalRuntimeResidency()`를 호출하고 `TerminalRuntimeLayer`에 `residentTabs`를 전달한다.
- runtime config getter 네 가지는 이미 있다: `getClientWsResourceLimits()`, `getTerminalResourceLimits()`, `getSnapshotResourceLimits()`, `getWorkspaceRuntimeResourceLimits()`.
- `TerminalContainer`는 `snapshot-restore` 또는 `debug-tail` 정책일 때 hidden output write를 skip할 수 있다.
- `TerminalView`는 visible output scheduler를 통과해 write하도록 일부 연결되어 있다.
- `WebSocketContext.send()`는 input message에 대해 `bufferedAmount`를 확인하고 `client-backpressure` / `client-hard-backpressure`를 반환한다.

남은 핵심 gap은 다음이다.

- `terminalRefsMap` stale entry pruning이 없다.
- Settings 저장 후 frontend runtime config를 즉시 reload/publish하는 경로가 없다.
- snapshot storage는 `getSnapshotResourceLimits()`가 아니라 상수 기반 예산을 사용한다.
- tombstone TTL cleanup, `maxEntries` eviction, mosaic layout localStorage 삭제/재저장 방지가 없다.
- hidden output recovery는 `restoreSnapshot()` 실패에도 dirty state를 clear한다.
- visible output overflow는 `restorePending`만 세우고 authoritative repair를 요청하지 않는다.
- `socket.send()` throw가 `SendResult`로 흡수되지 않는다.
- App/runtime integration, WebSocket backpressure 통합, snapshot/layout cleanup E2E가 부족하다.

## 결정 사항

### 1. 기존 dirty worktree 보호

현재 Wave7 관련 tracked dirty/untracked 파일은 사용자 또는 이전 작업 산출물로 취급한다. 구현자는 해당 파일을 수정할 수는 있지만, 기존 diff를 먼저 읽고 겹치는 hunk만 최소 변경해야 한다. `git reset`, `git checkout --`, broad revert는 금지한다.

### 2. `frontendRuntimeResidency` rollout

`stabilityModes.frontendRuntimeResidency`는 public runtime config flag로 유지한다. Wave7 구현은 `bounded` 경로를 완성하고 테스트한다.

현재 기본값은 `legacy`지만, Wave7 원문 완료 조건의 "mounted terminal runtime count is bounded by Settings"를 만족하려면 최종 완료 시 기본 제품 경로가 bounded여야 한다. 따라서 `legacy -> bounded` 전환은 optional follow-up이 아니라 Wave7 completion gate다. 구현 중간에는 staged milestone으로 `bounded` flag 경로만 검증할 수 있지만, 그 상태를 Wave7 완료로 부르지 않는다.

Settings UI에서 `stabilityModes.frontendRuntimeResidency`를 직접 노출하는 것은 보류한다. rollback은 `stabilityModes.frontendRuntimeResidency='off'` 또는 명시 config override로 둔다.

### 3. Hidden output 기본 정책

현재 기본값은 `write-hidden`이다. Wave7 seed 문서는 `write-hidden`을 explicit legacy fallback으로 정의한다. 따라서 구현 계획은 두 단계로 둔다.

- 먼저 `snapshot-restore` / `debug-tail` 경로를 정확하게 만들고 테스트한다.
- Wave7 completion gate에서는 기본 hidden output policy를 `snapshot-restore`로 전환한다. `write-hidden` 기본값 유지 상태는 staged milestone일 뿐이며 Wave7 완료가 아니다.
- `write-hidden`은 명시 설정 또는 rollback 문서가 있는 legacy fallback으로만 허용한다.

### 4. `TerminalHandle.sendInput()` 의미

현재 `TerminalHandle.sendInput()`의 `status: 'sent'`는 실제 WebSocket send 성공이 아니라 `TerminalView -> onInput` handoff 성공 의미다. Wave7에서는 이 public 의미를 바꾸지 않는다. 실제 WebSocket 성공/실패는 `WebSocketContext.SendResult`와 terminal debug event로 추적한다.

Wave7의 "must not claim input was sent" 수용 기준은 `WebSocketContext.SendResult`, `TerminalContainer`의 `ws_input_sent` debug event, queue/reject telemetry에 적용한다. `TerminalHandle.sendInput()`의 동기 handoff 결과는 delivery claim으로 문서화하거나 사용자-facing 성공으로 사용하지 않는다. 호출자가 실제 delivery 상태를 요구하는 API가 필요하면 후속 wave에서 별도 async delivery API로 설계한다.

### 5. Repair reason 확장

최소 구현은 기존 `workspace` repair reason을 재사용하고 debug payload에 `source: hidden-output-recovery | visible-output-overflow`를 남긴다. `ScreenRepairReason`에 새 enum을 추가하는 것은 frontend/server protocol 동시 변경이므로, 필요하면 후속 소규모 phase로 분리한다.

### Phase 1. Runtime Config Reload And Snapshot Limit Plumbing

목표: Settings 저장 후 frontend runtime config가 즉시 다시 읽히고, snapshot path가 runtime config 예산을 실제로 사용한다.

파일:

- `frontend/src/utils/inputReliabilityMode.ts`
- `frontend/src/components/Settings/SettingsPage.tsx`
- `frontend/src/utils/terminalSnapshot.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/services/tokenStorage.ts`
- `frontend/tests/unit/runtimeConfig.test.ts`
- `frontend/tests/unit/terminalSnapshot.test.ts`
- `frontend/tests/e2e/settings-resource-limits.spec.ts`

작업:

- runtime config reload 함수를 명시 API로 노출한다. 기존 `initializeInputReliabilityMode()`를 재사용하되 이름/의미를 명확히 할 수 있다.
- Settings PATCH 성공 후 `/api/runtime-config`를 reload하고 `runtimeConfigVersion` publish를 발생시킨다.
- reload 실패는 Settings 저장 실패로 되돌리지 않고 warning으로 표시한다.
- `terminalSnapshot.ts`에 `perSnapshotMaxChars`, `totalStorageBudgetChars`, `maxEntries`, `tombstoneTtlMs`를 받는 options를 추가한다.
- `TerminalView`는 snapshot parse/save에서 `getSnapshotResourceLimits()`를 사용한다.
- 기존 상수는 fallback default로만 남긴다.

테스트:

- Settings 저장 성공 후 runtime config fetch가 다시 호출되는지.
- 새 snapshot budget이 `getSnapshotResourceLimits()`에 반영되는지.
- `perSnapshotMaxChars` 초과 snapshot이 저장되지 않는지.
- `totalStorageBudgetChars`와 `maxEntries`가 eviction에 반영되는지.

### Phase 2. Runtime Residency Ref Cleanup

목표: evicted/disconnected/deleted terminal tab이 runtime layer와 ref map에 남지 않게 한다.

파일:

- `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx`
- 신규 후보: `frontend/src/components/Terminal/terminalRuntimeRefs.ts`
- `frontend/src/hooks/useTerminalRuntimeResidency.ts`
- `frontend/tests/unit/useTerminalRuntimeResidency.test.ts`
- 신규 후보: `frontend/tests/unit/terminalRuntimeRefs.test.ts`
- 신규 후보: `frontend/tests/e2e/runtime-residency.spec.ts`

작업:

- `ensureTerminalRef()`와 `pruneTerminalRefsMap()`를 pure helper로 분리한다.
- `TerminalRuntimeLayer`는 active resident runtime id 기준으로 `terminalRefsMap` stale entry를 prune한다.
- commit 이후 `useEffect` cleanup으로 처리해 render 중 map mutation 위험을 줄인다.
- `resolveVisibleTerminalTabIds()` 테스트를 보강한다.
- Wave7 원문에 남아 있는 `aliveWorkspaceIds` / `workspaceVisitOrder` pruning 항목은 현재 코드에서 제거되어 `useTerminalRuntimeResidency()`의 metadata 기반 정책으로 대체되었다. 구현 검증은 해당 옛 state가 재도입되지 않았고 `metadataRef`가 현재 tab id 기준으로 prune되는지 확인하는 방식으로 닫는다.

수용 기준:

- evicted tab은 `TerminalRuntimeLayer`에 mount되지 않는다.
- visible/active tab은 terminal cap보다 우선 pin된다.
- `terminalRefsMap`은 resident runtime 밖 tab id를 남기지 않는다.

### Phase 3. Snapshot Tombstone TTL And Mosaic Layout Cleanup

목표: workspace/tab 삭제 후 localStorage stale data가 다시 살아나지 않게 한다.

파일:

- `frontend/src/utils/terminalSnapshot.ts`
- `frontend/src/hooks/useMosaicLayout.ts`
- `frontend/src/hooks/useWorkspaceManager.ts`
- 신규 후보: `frontend/tests/unit/mosaicLayoutStorage.test.ts`
- `frontend/tests/unit/terminalSnapshot.test.ts`

작업:

- `cleanupExpiredTerminalSnapshotTombstones({ tombstoneTtlMs, now })` helper를 추가한다.
- corrupt/legacy tombstone은 expired로 취급하되, same-page `pendingSnapshotRemovals` Set은 localStorage TTL보다 우선한다.
- `mosaic_layout_${workspaceId}` key 생성을 중앙화한다.
- `clearMosaicLayoutForWorkspace(workspaceId)`와 save suppression guard를 추가한다.
- direct workspace delete와 WS `workspace:deleted` 경로에서 mosaic layout key를 제거한다.
- `useMosaicLayout`의 debounce save, beforeunload save, unmount save가 삭제된 workspace layout을 재생성하지 못하게 한다.

테스트:

- expired tombstone은 삭제되고 unexpired tombstone은 보존된다.
- corrupt tombstone은 cleanup 대상이다.
- workspace delete 후 `mosaic_layout_<workspaceId>`가 제거된다.
- delete-triggered unmount cleanup이 layout을 다시 저장하지 않는다.
- 같은 workspace id를 새로 유효하게 사용하면 suppression이 해제된다.

### Phase 4. Hidden Output Recovery Contract

목표: hidden output skip 이후 visible 전환 복구가 성공/실패를 구분하고 dirty state를 정확히 유지한다.

파일:

- `frontend/src/utils/terminalHiddenOutput.ts`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/tests/unit/terminalHiddenOutput.test.ts`
- 신규 후보: `frontend/tests/unit/terminalRuntimeRecoveryContract.test.ts`

작업:

- `beginHiddenOutputReplay()` / `finishHiddenOutputReplay()`를 실제 `TerminalContainer` 경로에서 사용한다.
- `restoreSnapshot()` 성공 시에만 hidden dirty state를 clear한다.
- local restore 실패 시 dirty state를 유지하고 authoritative repair를 요청한다.
- repair/snapshot 적용 성공 핸들러에서 dirty state를 clear한다.
- `debug-tail`은 xterm replay가 아니라 debug payload로만 남긴다.

수용 기준:

- `snapshot-restore` / `debug-tail` hidden output은 xterm write를 호출하지 않는다.
- local restore 실패가 성공처럼 숨겨지지 않는다.
- 복구 실패 상태는 debug event와 state로 관측 가능하다.

### Phase 5. Visible Output Scheduler Authoritative Recovery

목표: visible output overflow가 단순 drop/restorePending이 아니라 복구 요청으로 이어지게 한다.

파일:

- `frontend/src/utils/terminalOutputScheduler.ts`
- `frontend/src/utils/visibleOutputRecovery.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/tests/unit/terminalOutputScheduler.test.ts`
- `frontend/tests/unit/visibleOutputRecovery.test.ts`

작업:

- `TerminalView`에 `onVisibleOutputOverflow` prop을 추가한다.
- overflow 발생 시 중복 요청은 `visibleOutputRecovery` state machine으로 suppress한다.
- `TerminalContainer`가 기존 `requestScreenRepair()` 경로를 재사용해 authoritative repair를 요청한다.
- repair 요청이 send된 뒤 barrier를 세운다. 먼저 `restorePending`을 세워 repair readiness를 막는 순서를 피한다.
- repair apply 성공 시 scheduler reset, recovery finish, input readiness release를 수행한다.
- send 실패/timeout은 debug event로 남기고 stale 유지 또는 fail-open 여부를 명시한다. 권장 기본은 stale 유지다.

수용 기준:

- overflow 시 authoritative recovery request가 1회 이상 관측된다.
- 중복 overflow가 repair storm으로 번지지 않는다.
- recovery 성공 후 input readiness가 회복된다.
- recovery 실패가 조용히 성공으로 처리되지 않는다.

### Phase 6. Browser WebSocket Backpressure Hardening

목표: browser input send가 backpressure와 send exception을 모두 명시 실패로 다룬다.

파일:

- `frontend/src/utils/webSocketBackpressure.ts`
- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/tests/unit/webSocketBackpressure.test.ts`

작업:

- `socket.send(serializedPayload)`를 `try/catch`로 감싼다.
- throw 시 `SendResult`에 `send-failed` reason을 추가한다.
- input message send exception은 `ws_send_failed_exception` debug event로 남긴다.
- `TerminalContainer`는 `send-failed`를 기존 bounded transport outbox queue/reject 경로로 처리한다.
- control message의 backpressure bypass는 유지한다.

수용 기준:

- input message는 backpressure 상태에서 `socket.send()` 전에 실패한다.
- `socket.send()`가 throw해도 React/UI call stack이 깨지지 않는다.
- `ws_input_sent`는 실제 send 성공일 때만 기록된다.
- `TerminalHandle.sendInput()`의 기존 handoff 의미는 유지된다.

### Phase 7. Integration, E2E, And Rollout Gate

목표: staged milestone과 Wave7 completion을 분리하고, 최종 완료 조건을 실제 제품 동작 기준으로 확인한다.

자동 검증:

```powershell
node --experimental-strip-types --test frontend/tests/unit/runtimeConfig.test.ts frontend/tests/unit/settingsDraftHelpers.test.ts frontend/tests/unit/useTerminalRuntimeResidency.test.ts frontend/tests/unit/terminalRuntimeRefs.test.ts frontend/tests/unit/terminalHiddenOutput.test.ts frontend/tests/unit/terminalOutputScheduler.test.ts frontend/tests/unit/visibleOutputRecovery.test.ts frontend/tests/unit/webSocketBackpressure.test.ts frontend/tests/unit/terminalSnapshot.test.ts frontend/tests/unit/mosaicLayoutStorage.test.ts
npm --prefix frontend run typecheck
npm --prefix frontend run build
```

E2E/manual:

- 서버 시작은 `start.bat --port 2222`, 종료는 `stop.bat`를 사용한다.
- Playwright/manual target은 `https://localhost:2222`다.
- health check는 `curl -k http://localhost:2221/health`다.
- 절대 `node.exe` 전체 종료를 사용하지 않는다.

권장 E2E:

- `settings-resource-limits.spec.ts`: Settings 저장 후 runtime config reload 확인.
- `runtime-residency.spec.ts`: bounded mode에서 mounted runtime count가 제한되는지, visible tab이 유지되는지 확인.
- `grid-equal-mode.spec.ts` 확장 또는 신규 spec: workspace delete 후 `mosaic_layout_<id>` 미복원 확인.
- hidden high-output scenario: hidden output skip과 visible 복구 debug event 확인.

Rollout gate:

- Staged milestone: `stabilityModes.frontendRuntimeResidency='bounded'`와 `resourceLimits.terminal.hiddenOutputPolicy='snapshot-restore'`를 명시 설정했을 때 모든 focused test와 E2E가 통과한다.
- Wave7 completion: 기본 제품 경로에서 runtime residency가 bounded이고 hidden output 기본 정책이 `snapshot-restore`다.
- 기본값 전환이 위험하다고 판단되면 Wave7은 완료가 아니라 staged milestone으로만 기록하고, 남은 rollout blocker와 rollback 조건을 별도 문서에 남긴다.

## 후속으로 미룰 항목

- Browser Worker 도입.
- 실제 split WebSocket 전환.
- `TerminalHandle.sendInput()`을 실제 delivery success/failure API로 재정의.
- `ScreenRepairReason` protocol enum 확장.
- Settings UI에서 `stabilityModes.frontendRuntimeResidency` 직접 편집 노출.
- 서버 push/SSE 기반 runtime config invalidation.

## 최종 구현 완료 조건

- 기본 제품 경로에서 mounted terminal runtime count가 Settings/resource limit에 맞게 제한된다.
- evicted terminal tab은 `TerminalRuntimeLayer`와 `terminalRefsMap`에 남지 않는다.
- hidden output 기본 정책에서 xterm write가 발생하지 않는다. `write-hidden`은 명시 legacy fallback에서만 허용된다.
- hidden/visible overflow 복구 실패가 조용히 성공 처리되지 않는다.
- visible output overflow가 authoritative repair로 이어진다.
- browser input send는 `bufferedAmount`와 send exception을 모두 명시 실패로 처리한다.
- snapshot budget, max entries, tombstone TTL, mosaic layout cleanup이 runtime config와 workspace delete lifecycle에 맞게 동작한다.
- frontend typecheck/build, focused unit, E2E, 서브에이전트 리뷰가 통과한다.
