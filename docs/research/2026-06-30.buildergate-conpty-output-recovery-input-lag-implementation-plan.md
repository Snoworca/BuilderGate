# BuilderGate ConPTY 출력 복구 및 입력 지연 구현 계획

날짜: 2026-06-30

범위: ConPTY 사용 시 Codex 같은 대화형 TUI에서 화면 복구가 느리고, 출력이 계속 갱신될 때 입력이 어려워지는 문제를 개선하기 위한 구현 계획이다. 이 문서는 기존 연구 문서와 5개 서브에이전트의 정적 분석 결과를 취합했다.

참고 문서:

- `docs/research/2026-06-30.buildergate-conpty-output-recovery-input-lag-research.ko.md`
- `docs/research/2026-06-30.buildergate-conpty-output-recovery-input-lag-research.md`
- `docs/research/plug-leak/wave6.md`
- `docs/research/2026-06-29.buildergate-wave7-frontend-runtime-implementation-plan.md`

## 결론

현재 문제는 ConPTY 하나의 문제가 아니라 출력 폭주, WebSocket 전송 정책, 복구 중 큰 `xterm.write()`, 설정 미노출, 계측 부족이 겹친 병목이다.

테스트로 실제 사람이 느끼는 "너무 느림"과 "입력이 안 먹는 느낌"을 완전히 같은 형태로 안정 재현하기는 어렵다. 하지만 그 현상을 만드는 하위 조건은 충분히 테스트할 수 있다.

- 출력이 폭주해도 control/input 메시지가 굶지 않는가.
- WebSocket queue가 byte limit과 backpressure를 지키는가.
- snapshot, ACK-pending tail, `session:ready`, queued input release 순서가 깨지지 않는가.
- snapshot/repair가 큰 payload를 브라우저 main thread에 한 번에 밀어 넣지 않는가.
- 설정값을 바꾸면 runtime에 반영되고, 잘못된 값은 조용히 보정하지 않고 거절되는가.

따라서 구현 목표는 "체감 문제를 그대로 E2E로 재현"이 아니라 "증상을 만드는 병목을 합성 부하와 deterministic unit test로 고립하고, 실제 ConPTY soak로 증거를 보강"하는 것이다.

## 서브에이전트 조사 취합

| 에이전트 | 조사 축 | 채택한 결론 |
|---|---|---|
| 1 | Server transport/backpressure | `direct` 기본값은 즉시 바꾸지 않고, `safe-send-*` 경로에 priority-aware queue와 projected buffer 판단을 먼저 구현한다. |
| 2 | Frontend recovery/input lag | live output scheduler와 별도인 recovery write gate를 두고 snapshot/repair/tail direct write를 제거한다. |
| 3 | Settings/API/config | 이미 schema/runtime에 들어간 값 중 reserved 상태인 값을 해제하고, 새 runtime 변수는 Settings capability와 UI에 반드시 노출한다. |
| 4 | Metrics/tests/soak | 실제 ConPTY 체감 재현 대신 redraw flood, queue ordering, latency correlation, recovery ordering 테스트를 만든다. |
| 5 | Rollout/risk | 기본값 변경은 evidence 이후로 미루고, rollback은 Settings switch로 수행한다. |

## 채택 원칙

1. ConPTY를 조용히 끄거나 winpty/pipe로 우회하지 않는다.
2. 입력 경로 자체는 계속 직접성을 유지한다. 문제는 입력을 PTY로 쓰기 전의 frontend barrier, unified WebSocket, output FIFO, recovery 적용 지연이다.
3. control priority는 unrelated output backlog를 우회하는 의미다. active recovery transaction 내부에서 snapshot/tail/ready/input 순서를 바꾸면 안 된다.
4. output coalescing은 같은 session의 인접 `output` payload에만 적용한다.
5. 새로 도입되는 queue size, flush interval, timeout, threshold, mode 값은 하드코딩하지 않고 Settings 페이지에서 변경 가능해야 한다.
6. 단, runtime consumer가 없거나 단일 literal만 있는 값은 editable setting으로 열지 않는다. 예: 현재 `resourceLimits.headless.overflowPolicy = "degrade-headless"`만 가능하면 read-only 또는 deferred로 둔다.
7. 기본값 변경은 구현 완료와 분리한다. `wsSendMode: direct -> safe-send-enforce` 전환은 soak evidence 이후 별도 rollout로 판단한다.

## Phase 0: 계약 및 baseline 정리

목표: 구현 전에 현재 상태를 측정하고, SRS/설정 계약을 먼저 고정한다.

작업:

- `docs/spec` 또는 관련 Wave 문서에 ConPTY output/input 개선 요구를 추가한다.
- 최소 telemetry/debug/harness를 먼저 추가해 baseline artifact를 만든다.
- `wsSendMode=direct`, `safe-send-observe`, `safe-send-enforce` 각각에서 현재 settings snapshot, input latency, recovery-to-ready, queued bytes, coalesce count를 기록한다.
- 반복 가능한 PowerShell ANSI redraw flood harness를 만들고, baseline 결과를 `docs/research` 또는 `docs/analysis`에 저장한다.
- `/api/sessions/telemetry`에 frontend/backend aggregate를 먼저 얇게 확장한다.
- raw terminal input/output은 telemetry와 soak artifact에 저장하지 않는다는 보안 원칙을 명시한다.

완료 조건:

- baseline 수집 항목이 문서화되어 있고, 최소 1회 baseline artifact가 남아 있다.
- redraw flood harness를 동일 조건에서 다시 실행할 수 있다.
- "재현 어려운 체감"과 "테스트 가능한 병목"이 구분되어 있다.
- 이후 Phase에서 추가되는 모든 변수의 Settings 노출 위치가 결정되어 있다.

## Phase 2: Server WebSocket 전송 큐 개선

목표: output 폭주 중에도 control/input 관련 메시지가 굶지 않게 하되, terminal recovery 순서는 깨지 않는다.

작업:

- `server/src/ws/wsSendPolicy.ts`
  - 단일 `items[]` FIFO를 `controlQueue`, `terminalQueue` 또는 lane-aware deque로 변경.
  - `Array.shift()` 기반 drain을 제거하고 head index/deque 구조를 사용한다.
  - message classifier를 추가한다.
- `server/src/ws/WsRouter.ts`
  - `sendTransportMessage`, `enqueueTransportMessage`, `flushTransportQueue`를 priority-aware drain으로 변경.
  - `projectedBufferedBytes = ws.bufferedAmount + message.byteLength` 기준 high/hard 판단을 추가한다.
  - 단일 giant payload가 hard limit을 우회해 direct send 되지 않도록 한다.
  - same-session adjacent `output`만 coalesce하고, coalesce 후 byte limit을 다시 검사한다.
  - observability에 queue depth, projected overflow, priority flush count, control flush latency를 추가한다.

Lane 규칙:

- `output`, `screen-snapshot`, `screen-repair`, ACK-pending tail은 terminal lane으로 취급한다.
- `session:ready`는 control처럼 보이지만 recovery transaction 내부에서는 tail 뒤에 있어야 한다. 따라서 deferred-control 또는 recovery barrier를 둔다.
- input rejection, status, heartbeat 등 terminal payload ordering과 무관한 control은 output backlog를 우회할 수 있다.

완료 조건:

- unrelated output backlog가 있어도 control message가 굶지 않는다.
- recovery transaction 내부 ordering은 보존된다.
- projected hard limit 초과 시 observe/enforce 정책이 명확하게 동작한다.
- slow client 처리는 session 전체가 아니라 해당 client 단위로 제한된다.

## Phase 3: replay tail 및 screen repair overflow 개선

목표: recovery/replay 경로가 giant flush로 변하지 않게 하고, tail truncation을 byte-aware하게 만든다.

작업:

- `server/src/types/ws-protocol.ts`
  - replay pending state를 `queuedOutput: string` 중심에서 byte/chunk metadata 중심으로 확장한다.
  - `screenSeq`, `byteLength`, `truncatedBytes`, `truncatedChunks`를 기록한다.
- `server/src/ws/WsRouter.ts`
  - `appendQueuedOutput`, `appendScreenRepairQueuedOutput`을 byte-aware ring으로 교체한다.
  - UTF-8/Hangul/emoji와 ANSI escape sequence 경계가 깨지지 않게 기존 terminal payload helper를 확장한다.
  - screen repair overflow 시 `${queuedOutput}${data}`를 즉시 flush하지 않는다.
  - overflow 시 repair transaction을 abort하고 snapshot refresh/replay로 전환한다.
- `server/src/services/SessionManager.ts`
  - replay tail cap과 snapshot cap을 분리한다.
  - hard-coded replay queue limit을 Settings 기반으로 옮긴다.

완료 조건:

- tail truncation이 UTF-8 문자를 깨지 않는다.
- repair overflow가 giant output flush로 이어지지 않는다.
- overflow는 명시적 event와 refresh/recovery 경로로 관측된다.

## Phase 4: Frontend recovery write gate

목표: snapshot restore, authoritative snapshot replace, screen repair patch, ACK-pending tail이 브라우저 main thread를 오래 점유하지 않게 한다.

작업:

- `frontend/src/utils/terminalOutputScheduler.ts`
  - live/recovery 공용으로 쓸 수 있도록 `label`, `onDrain`, `onOverflow`, `onChunkWritten` 또는 Promise 기반 drain helper를 추가한다.
  - chunk boundary와 xterm write callback ordering을 테스트한다.
- `frontend/src/components/Terminal/TerminalView.tsx`
  - `recoverySchedulerRef`, `writeRecoveryOutput`, `whenRecoveryIdle`, `resetRecoveryGate`를 추가한다.
  - local snapshot restore, authoritative/fallback snapshot replace, screen repair `ansiPatch` direct `term.write()`를 recovery gate enqueue로 교체한다.
  - recovery gate overflow 시 stale/recovery-needed event를 남기고 서버 refresh를 요청한다.
- `frontend/src/components/Terminal/TerminalContainer.tsx`
  - snapshot apply 완료 후에만 `screen-snapshot:ready`를 보낸다.
  - ACK 이후 `session:ready` 전까지 들어오는 `output`은 ACK-tail로 간주해 recovery gate lane으로 보낸다.
  - `session:ready` 처리는 recovery gate drain 완료 또는 `recoveryReadyReleaseTimeoutMs` 이후 input outbox를 flush한다.
  - screen repair in-flight 상태에 client nonce/token을 붙이는 방안을 적용한다.
- `frontend/src/contexts/WebSocketContext.tsx`
  - client message classification을 명시화한다.
  - browser backpressure 대상은 `input` 중심으로 유지하고, control send 정책은 테스트로 고정한다.

완료 조건:

- snapshot/repair direct `term.write()` 경로가 제거된다.
- recovery 중 queued input은 recovery 완료 전 PTY로 release되지 않는다.
- recovery 완료 후 input이 unrelated live output backlog 뒤에서 굶지 않는다.
- timeout으로 release될 경우 debug/telemetry에 명시적으로 남는다.

단위 테스트:

- Server:
  - safe-send control priority under redraw flood
  - projected buffered hard/high limit
  - same-session output coalescing only
  - recovery transaction ordering under pressure
  - replay tail ring UTF-8/ANSI boundary
  - screen repair overflow refresh path
  - Settings PATCH validation and runtime apply
- Frontend:
  - recovery write gate chunking and drain callback ordering
  - snapshot/repair direct write 금지 contract
  - ACK-tail routing before `session:ready`
  - input outbox flush after recovery drain
  - Settings field render/PATCH/reload

E2E/soak:

- 대상 URL은 `https://localhost:2222`다.
- 서버 시작은 `start.bat --port 2222`, 종료는 `stop.bat`만 사용한다.
- 전체 `node.exe` 종료는 금지한다.
- PowerShell ANSI redraw flood workload를 사용한다.
- 시나리오:
  - visible terminal 출력 폭주 중 ASCII/한글 입력
  - hidden tab 출력 폭주 후 재진입
  - reload/resubscribe 후 snapshot + tail 복구
  - resize flood 중 repair/recovery
  - Settings에서 `outputCoalesceWindowMs`와 `wsSendMode` 변경 후 runtime 반영

Pass/Fail 기준:

- deterministic unit에서 output loss/reorder/duplication 0건.
- recovery ordering: snapshot apply, ACK-tail apply, `session:ready`, queued input release 순서 보존.
- `input_write_failed` 0건.
- 설명 불가능한 `server_input_rejected` 0건.
- soak 종료 후 `transportOutputQueuedBytes`, `transportControlQueuedBytes`, headless pending bytes가 baseline으로 복귀.
- session 종료 evidence에서 `remainingVerifiedDescendants = 0`.

## Phase 6: Rollout

목표: 구현과 기본값 변경을 분리해 위험을 낮춘다.

순서:

1. `wsSendMode=direct` 기본값 유지 상태에서 Settings 노출과 telemetry만 배포한다.
2. `safe-send-observe`로 바꿔 queue/coalesce/latency evidence를 수집한다.
3. 문제 없을 때 `safe-send-enforce`를 사용자 또는 운영자가 Settings에서 켠다.
4. soak evidence가 충분할 때만 기본값 변경을 별도 PR/커밋으로 검토한다.

Rollback:

- 서버 전송 문제: `stabilityModes.wsSendMode=direct` 또는 `safe-send-observe`.
- frontend recovery 문제: recovery gate를 우회하지 않는다. rollback은 recovery gate budget을 보수적으로 낮추고, overflow 시 서버 snapshot refresh를 강제하는 degraded mode로 제한한다.
- Settings 문제: 해당 field를 다시 reserved/unavailable로 돌리고 UI card를 숨긴다.
- telemetry 문제: sample interval 증가 또는 recent limit 감소.

## 채택하지 않는 제안

- ConPTY를 Codex에서 조용히 winpty/pipe로 바꾸는 방식.
- browser worker를 1차 해결책으로 삼는 방식. 현재 병목은 xterm write, recovery ordering, WebSocket queue가 먼저다.
- split WebSocket을 이번 작업의 전제조건으로 삼는 방식. 장기 후보지만 즉시 해결에는 priority-aware queue가 더 작고 검증 가능하다.
- `direct` 모드에서 `outputCoalesceWindowMs`가 효과 있는 것처럼 UI를 구성하는 방식.
- snapshot보다 tail/input을 먼저 보내는 control priority.
- raw terminal input/output을 telemetry나 soak artifact에 저장하는 방식.
- broad process kill을 rollback 또는 테스트 정리 방법으로 사용하는 방식.

## 구현 순서 요약

1. Phase 0의 최소 telemetry/debug/harness와 baseline artifact를 먼저 만든다.
2. Settings 계약과 schema/type/UI를 확장한다.
3. 서버 safe-send queue를 lane-aware 구조로 바꾸고 projected buffer 판단을 추가한다.
4. replay tail과 screen repair overflow를 byte-aware bounded 경로로 바꾼다.
5. frontend recovery write gate를 추가하고 direct write 경로를 제거한다.
6. telemetry/debug capture와 deterministic test를 완성한다.
7. `https://localhost:2222`에서 synthetic ConPTY redraw flood E2E와 soak를 수행한다.
8. evidence를 보고 기본값 변경 여부를 별도 결정한다.

## 최종 완료 판정

이 계획은 다음 조건을 모두 만족할 때 완료된 것으로 본다.

- 모든 새 runtime 변수는 Settings 페이지에서 확인/변경 가능하다.
- Settings invalid 값은 서버에서 거절되고 UI에 표시된다.
- output 폭주 중에도 input/control이 굶지 않는다는 deterministic test가 있다.
- recovery transaction ordering을 깨지 않는 test가 있다.
- snapshot/repair/tail이 chunked recovery write gate를 통과한다.
- ConPTY redraw flood E2E가 `https://localhost:2222`에서 수행된다.
- soak 종료 후 session/process cleanup evidence가 남는다.
- 기본값 변경은 구현 완료와 분리되어 evidence 기반으로만 진행된다.
