# Wave6 Junior Summary

작성일: 2026-06-29

## 한 줄 요약

Wave6는 서버 안에만 있던 리소스 제한 설정을 브라우저 Settings 화면에서 안전하게 보고 바꿀 수 있게 만든 작업이다.

## 왜 필요했나

기존에는 큐 크기, WebSocket 버퍼 한계, snapshot 저장 한계 같은 값이 `config.json5`와 서버 코드 쪽에 있었다. 값을 바꾸려면 사람이 설정 파일을 직접 열어 수정해야 했고, 어떤 값은 바꿔도 되는지, 어떤 값은 아직 숨겨야 하는지 구분하기 어려웠다.

Wave6의 목표는 다음과 같았다.

- 사용자가 선택된 리소스 제한값을 Settings 화면에서 수정할 수 있게 한다.
- 서버가 허용한 필드만 화면에 보여준다.
- 위험하거나 아직 Wave7에서 다룰 필드는 숨기고 직접 PATCH도 거부한다.
- 저장 결과가 바로 적용인지, 새 세션부터 적용인지 정확히 보여준다.
- Wave7에서 할 런타임 동작 변경은 건드리지 않는다.

## 무엇을 개선했나

### 1. Settings 화면에 Resource Limit 카드 추가

브라우저 Settings에 리소스 제한 카드가 생겼다.

대표적으로 다음 값을 볼 수 있고 수정할 수 있다.

- Headless pending output 제한
- 서버 WebSocket buffered high-water / hard-limit
- 브라우저 WebSocket input backpressure 기준
- 터미널 input queue / transport outbox 제한
- snapshot 크기와 저장 개수 제한
- workspace / terminal live runtime 제한
- hidden output tail 제한

효과:

- 운영자가 설정 파일을 직접 열지 않고도 주요 제한값을 조정할 수 있다.
- 설정 위치가 UI로 모여서 실수 가능성이 줄어든다.

### 2. 바뀐 값만 PATCH로 보냄

예를 들어 `headless.pendingOutputMaxBytes` 하나만 바꾸면 서버로도 그 값 하나만 보낸다.

효과:

- 변경하지 않은 설정을 실수로 덮어쓸 가능성이 줄어든다.
- 서버 로그와 응답에서 무엇이 바뀌었는지 추적하기 쉬워진다.

### 3. 잘못된 숫자는 클라이언트에서 먼저 막음

Settings 화면은 서버가 내려준 `min`, `max`, `step` 같은 제약을 보고 숫자를 검사한다.

예:

- 비어 있는 값
- 숫자가 아닌 값
- 정수가 아닌 값
- 최소값보다 작은 값
- 최대값보다 큰 값
- step에 맞지 않는 값

이 값들은 조용히 자동 보정하지 않고 오류로 보여준다.

효과:

- 사용자가 입력한 값이 몰래 다른 값으로 바뀌지 않는다.
- "내가 1을 넣었는데 왜 다른 값으로 저장됐지?" 같은 혼란을 줄인다.

### 4. 서버에서도 한 번 더 검증

프론트엔드 검증은 사용자 경험을 위한 1차 방어선이다. 최종 방어선은 서버다.

서버는 다음을 확인한다.

- 이 필드가 Settings에서 수정 가능한 필드인가
- 값의 범위가 맞는가
- 서로 관계가 있는 값이 말이 되는가
  - 예: hard reconnect 값은 input backpressure 값보다 커야 함
  - 예: hard limit은 high-water 값보다 커야 함
- 저장 중 문제가 나면 런타임 상태를 롤백할 수 있는가

효과:

- 브라우저를 우회해서 직접 API를 호출해도 예약 필드는 저장되지 않는다.
- 잘못된 설정이 파일에 남거나 런타임에 반쯤 적용되는 위험이 줄어든다.

### 5. 아직 열면 안 되는 필드는 숨기고 거부함

Wave6에서 일부 필드는 의도적으로 열지 않았다.

숨긴 필드 예:

- `resourceLimits.telemetry.*`
- `stabilityModes.*`
- visible output scheduler 관련 값
- terminal scrollback 값
- WebSocket control queue / output coalesce 값
- headless write lag / write batch / overflow policy 값

효과:

- 아직 Wave7에서 구현하거나 검증해야 할 동작을 사용자가 미리 켜지 못한다.
- Settings UI가 "지금 실제로 안전하게 바꿀 수 있는 값"만 보여준다.

### 6. public runtime config는 브라우저에 필요한 것만 공개

브라우저가 필요한 값은 `/api/runtime-config`로 볼 수 있지만, 서버 전용 값은 공개하지 않는다.

공개하지 않는 대표 섹션:

- server-only `headless`
- server-only `ws`
- `telemetry`

효과:

- 브라우저가 알아야 할 값과 서버 내부 값이 섞이지 않는다.
- 설정 노출 범위가 더 안전하고 명확해진다.

### 7. 저장 결과 표시를 더 정확하게 함

Settings 저장 후 apply summary가 나온다.

예:

- 즉시 적용되는 값
- 새 터미널 세션부터 적용되는 값

추가로, 저장 성공 후 다시 값을 수정하면 이전 성공 메시지가 바로 사라지도록 보강했다.

효과:

- 사용자가 보고 있는 성공 메시지가 현재 입력 상태와 어긋나지 않는다.
- invalid 입력 상태에서 이전 성공 메시지가 남아 혼동을 주는 일이 줄어든다.

## 무엇을 바꾸지 않았나

Wave6는 Settings 연결층 작업이다. 아래 런타임 동작은 Wave6에서 바꾸지 않았다.

- 프로세스 cleanup 정책
- 터미널 runtime residency 동작
- visible output scheduler 동작
- 브라우저 WebSocket backpressure 소비자 동작
- split WebSocket 동작
- broad process termination 동작

쉽게 말하면, Wave6는 엔진을 새로 튜닝한 것이 아니라 튜닝할 수 있는 안전한 조작 패널을 만든 것이다.

## 사용 흐름 예시

1. 사용자가 Settings 화면을 연다.
2. 서버가 "수정 가능한 리소스 제한 필드" 목록과 제약 조건을 내려준다.
3. 화면은 가능한 필드만 카드로 보여준다.
4. 사용자가 값을 바꾼다.
5. 프론트엔드가 숫자 범위를 먼저 확인한다.
6. 저장하면 바뀐 leaf 값만 PATCH로 보낸다.
7. 서버가 다시 검증하고 `config.json5`에 저장한다.
8. 응답에는 바뀐 키와 적용 범위가 담긴다.
9. 화면은 적용 범위를 요약해서 보여준다.

## 테스트와 검증

Wave6는 다음 검증을 거쳤다.

- Settings helper unit test
- Settings resource limit Playwright E2E
- frontend typecheck
- frontend build
- Wave6 scoped ESLint
- server build
- server focused resource limit tests
- full server test suite
- 서브에이전트 리뷰와 재리뷰

최근 stale summary 이슈 수정 후에는 Settings resource limit E2E가 4개 통과했다.

## 주니어 개발자가 기억할 핵심

- UI에서 값을 바꿀 수 있게 했다고 해서 아무 값이나 열면 안 된다.
- 클라이언트 검증은 편의를 위한 것이고, 서버 검증이 최종 방어선이다.
- PATCH는 바뀐 값만 작게 보내는 것이 안전하다.
- 성공 메시지는 현재 입력 상태와 맞아야 한다.
- Wave6의 책임은 Settings 연결층이고, 실제 런타임 소비자 변경은 Wave7의 책임이다.
