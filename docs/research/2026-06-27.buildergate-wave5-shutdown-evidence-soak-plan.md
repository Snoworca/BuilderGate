# BuilderGate Wave5 Shutdown Evidence And Server Soak Gate Research Plan

작성일: 2026-06-27

## 목적

`docs/research/plug-leak/wave5.md`의 Wave5 구현 전에 서버 shutdown, daemon stop, process cleanup evidence, server-only soak gate를 정리한다. 이 문서는 1차 조사 서브에이전트 5개와 2차 계획 서브에이전트 5개 결과를 취합한 구현 전 연구/계획이다.

Wave5의 핵심은 새 UI나 프론트엔드 runtime 조정이 아니다. Wave0-Wave4가 만든 config, process cleanup, lifecycle, bounded output, WebSocket backpressure 기반이 서버 종료와 반복 사용에서 실제로 누적을 막는지 검증 가능한 evidence로 연결하는 것이다.

## 결론

- `performGracefulShutdown()`은 현재 workspace CWD snapshot/flush만 수행하고, active session cleanup을 호출하지 않는다.
- Wave5 core 구현은 shutdown 순서를 `CWD snapshot -> workspace flush -> owned session termination -> cleanup evidence -> final workspace flush -> process exit`로 고정해야 한다.
- session cleanup mode는 새 기본값을 강제로 `enforce`로 바꾸지 않는다. 기본 shutdown은 기존 `session.processCleanup.mode` 런타임 설정을 따르고, strict soak 또는 별도 설정에서 enforce를 켜서 검증한다.
- `/api/internal/shutdown` 응답과 `stop-client.js`는 session cleanup evidence 5개 필드를 검증하고 출력해야 한다.
- `remainingVerifiedDescendants > 0`은 기본 `stop`에서도 실패로 처리한다. 단, 이 값은 verified-owned remaining process만 세어야 하며 unverified/skipped observations와 섞으면 안 된다. `sessionCleanupDegraded > 0` 또는 `sessionCleanupSkippedUnverified > 0`은 기본 `stop`에서는 경고, strict server soak gate에서는 실패로 처리한다.
- server-only soak gate는 CI 자동 gate가 아니라 문서화된 live checklist로 둔다. 자동 gate는 deterministic한 server/daemon tests에 한정한다.
- SRS에는 새 `FR-BGSTAB-013`을 만들지 않는다. Wave5는 `REL-BGSTAB-001` AC-4/AC-5와 `FR-BGSTAB-010`, `FR-BGSTAB-011`, `FR-BGSTAB-012`의 deferred/evidence 연결로 추적한다.

## 현재 상태

### 서버 shutdown

- `server/src/services/gracefulShutdown.ts`의 `performGracefulShutdown()`은 다음만 수행한다.
  - `SessionManager.stopAllCwdWatching()`
  - `WorkspaceService.snapshotAllCwds()`
  - `WorkspaceService.forceFlush()`
  - workspace JSON 재읽기 기반 flush evidence 확인
  - 전달받은 interval/timer clear
- `SessionManager.terminateAllSessions()`는 이미 존재하지만 shutdown path에 연결되어 있지 않다.
- `SessionManager.finalizeSession()` 경로는 PTY, timers, detectors, headless terminal, headless output queue, CWD temp file, replay/debug state를 정리한다.
- CWD temp file은 finalize에서 삭제될 수 있으므로 session termination은 반드시 CWD snapshot/first workspace flush 이후에 실행해야 한다.

### daemon stop

- `stop.bat`는 `node "%SCRIPT_DIR%stop.js" %*`를 호출하고 exit code를 그대로 반환한다.
- `stop.js`와 `tools/start-runtime.js stop`은 모두 `tools/daemon/stop-client.js:stopDaemon()`을 사용한다.
- `stopDaemon()` 순서:
  1. daemon state read/validate
  2. state를 `stopping`으로 원자 갱신
  3. sentinel exit 대기
  4. `POST /api/internal/shutdown`
  5. workspace flush evidence 검증
  6. `/health` nonresponse 확인
  7. state를 `stopped`로 기록
- 현재 session cleanup evidence 검증은 없다.
- sentinel은 `stopping`, `stopped`, `fatal` state를 보면 종료한다. session cleanup이나 broad kill 책임을 갖지 않는다.

### telemetry와 테스트

- `/api/sessions/telemetry`는 `sessions.cleanup`, `sessions.headlessOutput`, `ws.transport*` 관측값을 노출한다.
- 서버 테스트는 `server/src/test-runner.ts` 자체 러너에 집중되어 있고 `npm --prefix server test`로 실행된다.
- daemon Wave5 targeted tests는 다음 명령으로 묶는다.

```powershell
npm run test:daemon:wave5
```

## Acceptance Criteria Draft

- AC-W5-1: graceful shutdown 시작 시 active tab CWD snapshot, workspace flush, owned session termination, cleanup evidence collection, final workspace flush, process exit 순서가 테스트 가능해야 한다.
- AC-W5-2: `/api/internal/shutdown` response와 daemon stop result는 `sessionCleanupAttempted`, `sessionCleanupCompleted`, `sessionCleanupDegraded`, `sessionCleanupSkippedUnverified`, `remainingVerifiedDescendants`를 포함해야 한다.
- AC-W5-3: daemon stop은 session cleanup evidence schema를 검증하고 출력해야 하며, broad kill fallback을 추가하지 않아야 한다.
- AC-W5-4: `remainingVerifiedDescendants > 0`이면 default daemon stop도 graceful failure로 처리해야 한다.
- AC-W5-5: `sessionCleanupDegraded > 0` 또는 `sessionCleanupSkippedUnverified > 0`은 default daemon stop에서는 warning으로 남기고, strict server soak gate에서는 failure로 처리해야 한다.
- AC-W5-6: shutdown cleanup은 `FR-BGSTAB-011`의 verified ownership 원칙을 유지해야 하며 `taskkill /IM`, `killall`, 전체 `node.exe` 종료, process-name 기반 kill을 추가하지 않아야 한다.
- AC-W5-7: shutdown/restart/delete 검증은 `FR-BGSTAB-010`의 stopped/non-recoverable lifecycle 의미를 깨지 않아야 한다.
- AC-W5-8: live/manual validation은 `start.bat --port 2222`, `https://localhost:2222`, `http://localhost:2221/health`, `stop.bat` 기준으로 수행해야 한다.
- AC-W5-9: 실행하지 않은 live soak, WSL/bash scenario, long-running browser memory/input-lag validation은 완료로 과장하지 않고 remaining operational validation으로 기록해야 한다.

## Non-Goals

- 프론트엔드 runtime residency, hidden output, browser worker, browser WebSocket backpressure 구현은 Wave7 범위다.
- Settings UI/API로 resource limit을 조정하는 작업은 Wave6 범위다.
- split WebSocket migration 또는 split-mode live validation은 Wave5 범위가 아니다.
- WSL 내부 Linux process-tree kill 구현이나 `processTreeTerminator`의 대규모 재설계는 하지 않는다.
- `WorkspaceService`의 shutdown reason lifecycle persistence 정책을 바꾸지 않는다.
- unverified/detached descendant를 강제로 종료하지 않는다. evidence에 degraded/skipped로 남긴다.
- raw terminal input/output payload를 telemetry, log, evidence에 저장하지 않는다.
- Wave5 완료만으로 장시간 브라우저 memory/input-lag soak가 완전히 끝났다고 주장하지 않는다.

## 구현 계획

### Phase 5-1. Evidence Contract

목표: shutdown response와 stop-client가 공통으로 이해할 session cleanup evidence contract를 추가한다.

수정 후보:

- `server/src/services/gracefulShutdown.ts`
- `server/src/routes/internalShutdownRoutes.ts`
- `tools/daemon/stop-client.js`
- `server/src/test-runner.ts`
- `tools/daemon/stop-client.test.js`

서버 evidence interface:

```ts
export interface ShutdownSessionCleanupEvidence {
  sessionCleanupAttempted: number;
  sessionCleanupCompleted: number;
  sessionCleanupDegraded: number;
  sessionCleanupSkippedUnverified: number;
  remainingVerifiedDescendants: number;
}
```

`GracefulShutdownResult`에는 위 5개 필드를 top-level additive fields로 추가한다. 기존 workspace evidence fields는 유지한다.

daemon validator 후보:

```js
function getShutdownSessionCleanupEvidence(body, options = {}) {
  // returns { valid: true, evidence, strictFailureReason }
  // or { valid: false, reason }
}
```

검증 규칙:

- `body`는 object여야 한다.
- 5개 필드는 모두 존재해야 한다.
- 모든 필드는 `Number.isInteger(value) && value >= 0`이어야 한다.
- `sessionCleanupCompleted + sessionCleanupDegraded + sessionCleanupSkippedUnverified > sessionCleanupAttempted`는 invalid로 처리한다.
- `remainingVerifiedDescendants > 0`은 default stop failure다.
- `sessionCleanupDegraded > 0` 또는 `sessionCleanupSkippedUnverified > 0`은 default stop warning, strict gate failure다.

### Phase 5-2. Graceful Shutdown Sequence

목표: workspace CWD evidence를 잃지 않으면서 owned sessions를 bounded cleanup한다.

`server/src/services/gracefulShutdown.ts` 변경 계획:

- `GracefulShutdownOptions.sessionManager` Pick 범위를 다음으로 확장한다.
  - `stopAllCwdWatching`
  - `getAllSessions`
  - `terminateAllSessions`
  - `getObservabilitySnapshot`
- options 후보를 추가한다.
  - `sessionCleanupTimeoutMs?: number`
  - `sessionCleanupWaitMs?: number`
  - `sessionCleanupMode?: 'legacy' | 'observe' | 'enforce'`
- 기본 정책:
  - `sessionCleanupTimeoutMs`: 3000ms
  - `sessionCleanupWaitMs`: undefined 또는 runtime config 값을 따름
  - `sessionCleanupMode`: undefined로 두고 기존 `session.processCleanup.mode`를 따름
- helper 후보:
  - `createEmptySessionCleanupEvidence()`
  - `withTimeout()`
  - `collectSessionCleanupEvidence(before, after, targetSessionIds, timedOut)`
  - `resolveShutdownSessionCleanupReason(reason): 'shutdown'`
- `remainingVerifiedDescendants` 산출은 공개 cleanup telemetry의 `remainingDescendants`만으로 추정하지 않는다. 구현은 `SessionManager.terminateAllSessions()` 또는 cleanup result 경로에 shutdown-scoped detail을 추가해 verified-owned remaining count와 unverified/skipped count를 분리해야 한다. 이 detail이 없으면 `remainingVerifiedDescendants`는 fail-closed로 계산하지 말고 `0`으로 두면서 `sessionCleanupDegraded` 또는 `sessionCleanupSkippedUnverified`에 품질 저하를 반영한다.

권장 실행 순서:

```text
startedAt 기록
sessionManager.stopAllCwdWatching()
workspaceService.snapshotAllCwds()
workspaceService.forceFlush()
first workspace flush evidence read
clear shutdown timers
target session ids와 cleanup telemetry before 수집
sessionManager.terminateAllSessions({
  reason: 'shutdown',
  mode: options.sessionCleanupMode,
  waitMs: options.sessionCleanupWaitMs,
})를 sessionCleanupTimeoutMs로 감싸 실행
cleanup telemetry after 수집
session cleanup evidence 계산
workspaceService.forceFlush()
final workspace flush evidence read
GracefulShutdownResult 반환
```

주의:

- termination 이후에는 `snapshotAllCwds()`를 다시 호출하지 않는다.
- final workspace flush는 snapshot 재수집이 아니라 shutdown 중 발생한 lifecycle/session event persist 보장 용도다.
- timeout은 shutdown exception이 아니라 degraded evidence로 남긴다.
- workspace flush 실패는 기존처럼 shutdown failure로 처리한다.

`server/src/index.ts` 변경 계획:

- `performServerGracefulShutdown(reason)`에서 새 options를 전달한다.
- `sessionCleanupMode`는 기본적으로 전달하지 않아 runtime config를 따른다.
- log는 session cleanup summary를 포함하도록 갱신한다.
- `process.exit(0)`는 shutdown result 반환 이후 유지한다.

`SessionManager.ts` 변경 계획:

- core behavior 변경은 최소화한다.
- `terminateAllSessions()`와 `getObservabilitySnapshot().cleanup`을 재사용한다.
- broad kill이나 process-name kill 경로를 추가하지 않는다.
- shutdown reason lifecycle persistence 정책은 변경하지 않는다.

### Phase 5-3. Daemon Stop Integration

목표: `stopDaemon()`이 workspace flush evidence와 session cleanup evidence를 모두 검증한 뒤 health nonresponse로 넘어가게 한다.

`tools/daemon/stop-client.js` 변경 계획:

- `getShutdownSessionCleanupEvidence()` 추가.
- `formatShutdownSessionCleanupEvidence()` 추가.
- `getSessionCleanupStopPolicy(evidence, { strictSessionCleanupEvidence?: boolean })` 추가.
- `formatStopSuccessMessage(state, flushEvidence, sessionCleanupEvidence, warnings)`로 success output 확장.
- `stopDaemon()` result에 `sessionCleanupEvidence`와 `sessionCleanupWarnings` 추가.
- invalid evidence 또는 `remainingVerifiedDescendants > 0`이면:
  - `exitCode: 1`
  - `status: 'graceful-failure'`
  - state는 `stopping` 유지
  - `/health` nonresponse 단계로 넘어가지 않음
  - broad kill fallback 없음
- `strictSessionCleanupEvidence: true`이면 degraded/skipped도 failure로 처리한다.

`tools/daemon/sentinel.test.js` 변경 계획:

- sentinel은 session cleanup을 수행하지 않는다는 계약을 유지한다.
- `stopping` 관측 시 `exit`, `spawnApp` 미호출, `killProcess` 미호출 회귀를 보강한다.

`stop.bat` 변경 계획:

- 기본적으로 변경하지 않는다.
- 정적 검증에서 `taskkill`, `wmic`, `Stop-Process`, `killall`, broad `node.exe` 종료 패턴이 없는지 확인한다.

### Phase 5-4. Server Tests And Daemon Tests

`server/src/test-runner.ts` 추가/보강 후보:

- `performGracefulShutdown terminates sessions after workspace CWD flush`
  - 순서: `stop-watchers -> snapshot -> forceFlush#1 -> terminateAllSessions -> forceFlush#2`
- `performGracefulShutdown preserves lastCwd when termination removes cwd temp file`
  - termination stub이 cwd temp file을 삭제해도 workspace JSON에는 snapshot 값이 남아야 한다.
- `performGracefulShutdown returns session cleanup evidence`
  - attempted/completed/degraded/skipped/remaining fields를 검증한다.
- `performGracefulShutdown normalizes internal shutdown reason for SessionManager`
  - result reason은 `internal-shutdown`, session cleanup reason은 `shutdown`.
- `performGracefulShutdown degrades session cleanup on timeout and still final-flushes workspace`
  - timeout 시 result는 반환되고 degraded evidence가 증가하며 final flush가 호출된다.
- `internal shutdown route returns session cleanup evidence`
  - response JSON에 workspace evidence와 session cleanup evidence가 함께 있어야 한다.

`tools/daemon/stop-client.test.js` 추가/보강 후보:

- `stopDaemon validates shutdown session cleanup evidence`
- `stopDaemon rejects invalid shutdown session cleanup evidence`
- `stopDaemon fails graceful stop when remaining verified descendants are reported`
- `stopDaemon prints session cleanup evidence on graceful success`
- `stopDaemon treats degraded and skipped cleanup as warning by default`
- `stopDaemon treats degraded and skipped cleanup as failure in strict session cleanup mode`
- `stopDaemon never uses broad kill fallback on evidence failure`

`tools/daemon/sentinel.test.js` 보강 후보:

- `runSentinelTick exits when stop-client owns stopping state without restarting app`
- restart/backoff/readiness 중 stop이 시작되면 `spawnApp`과 `killProcess`가 호출되지 않는 기존 패턴 유지.

자동 verification command:

```powershell
npm --prefix server test
npm run test:daemon:wave5
git diff --check
```

### Phase 5-5. Server-Only Soak Checklist

자동 CI gate로 넣지 않는다. 이유:

- PowerShell, bash, WSL availability가 환경별로 다르다.
- `start.bat --port 2222`와 HTTPS reverse proxy 상태에 의존한다.
- 실패 cleanup도 test-started PID로만 처리해야 하므로 broad kill 없는 자동 복구가 어렵다.

수동/live checklist:

```powershell
.\start.bat --port 2222
curl -k http://localhost:2221/health
```

검증 URL:

- app: `https://localhost:2222`
- health: `http://localhost:2221/health`

종료:

```powershell
.\stop.bat
```

scenario:

- repeated create/delete tab with PowerShell.
- repeated create/delete tab with bash or WSL where available.
- restart tab loop verifies old session descendants disappear.
- high-output command verifies headless queue and WS queue return to baseline.
- direct session delete verifies orphan recovery does not recreate stopped tab.
- internal shutdown verifies no verified session descendants remain.

측정:

- `/api/sessions/telemetry` before/during/after.
- known app PID와 test-started session root PID 기준 descendants.
- `sessions.cleanup` counters.
- `sessions.headlessOutput.pendingBytes/pendingChunks`.
- `ws.transportQueuedClientCount`, `ws.transportOutputQueuedBytes`, `ws.transportControlQueuedBytes`.
- `TerminalObs` summary.
- log size delta.

skip policy:

- bash/WSL unavailable이면 skipped로 기록하고 coverage를 주장하지 않는다.
- long-running soak를 수행하지 못하면 completion report에 remaining operational validation으로 남긴다.

## Rollback

- server rollback: `performGracefulShutdown()`에서 session cleanup 호출을 끄고 workspace flush-only shutdown으로 되돌린다.
- daemon rollback: session cleanup evidence를 warning-only 또는 tolerant parser로 낮추되 workspace flush evidence 검증은 유지한다.
- config rollback: 기존 `session.processCleanup.mode`를 `observe` 또는 `legacy`로 낮춘다.
- optional shutdown-scoped config를 도입한 경우 `session.shutdownProcessCleanup.mode = 'off'`를 즉시 rollback path로 둔다.
- evidence fields는 additive이므로 유지 가능하다.
