# BuilderGate Long-Term Stability Improvement Plan

작성일: 2026-06-17

## 목적

이 문서는 BuilderGate 장시간 사용 시 발생하는 서버/브라우저 메모리 증가, 브라우저 키보드 입력 지연, 주기적 새로고침 필요 문제를 장기적으로 해결하기 위한 통합 개선 계획이다.

사용자가 제안한 두 아이디어를 핵심 축으로 반영한다.

1. chunk queue, output queue, cache, snapshot, live runtime의 크기를 설정으로 제한해 메모리 과사용을 막는다.
2. 보내는 WebSocket과 받는 WebSocket을 분리해 terminal output 폭주가 keyboard input 경로를 같이 막지 않게 한다.

## 핵심 진단

현재 문제는 단순한 GC 지연이나 운영체제 메모리 반환 지연만으로 설명되지 않는다. 코드상 실제 누적 지점이 있다.

1. `SessionManager` headless output queue가 byte/chunk cap 없이 증가할 수 있다.
2. 서버와 브라우저 WebSocket send 경로가 `bufferedAmount`를 보지 않는다.
3. 숨겨진 workspace/tab의 xterm runtime, output write, WS subscription, observer가 계속 살아 있다.
4. snapshot/localStorage 저장이 동기 scan과 eviction 비용을 만든다.
5. daemon/dev log와 `[TerminalObs]` 로그가 장시간 누적된다.

따라서 장기 해결은 `cap 설정`만으로도, `split WebSocket`만으로도 부족하다. 먼저 무제한 누적을 차단하고, 그 다음 입력과 출력을 물리적으로 격리해야 한다.

## 설계 원칙

1. 서버 프로세스 보호를 최우선으로 한다.
2. terminal 화면 상태를 조용히 손상시키지 않는다.
3. 느린 client는 session이 아니라 socket 단위로 격리한다.
4. hidden terminal은 live output write를 기본적으로 수행하지 않는다.
5. cap 초과는 drop, coalesce, terminate, degrade, snapshot restore 중 하나로 명시 처리한다.
6. 모든 cap 초과와 복원은 telemetry에 남긴다.
7. 설정은 서버 hard guardrail과 사용자 UX preference를 분리한다.
8. split WebSocket은 unified mode를 안정화한 뒤 feature flag로 도입한다.
9. 사용자 키 입력만으로 interactive AI TUI session status가 `running`으로 바뀌면 안 된다.

## 목표 아키텍처

### Resource Limits

`config.json5`에 `resourceLimits` 계층을 추가하고, `/api/runtime-config`로 frontend에 non-secret limit을 내려준다.

권장 상위 구조:

```json5
resourceLimits: {
  headless: {
    pendingOutputMaxBytes: 8388608,
    pendingOutputMaxChunks: 1024,
    writeLagWarnMs: 500,
    overflowPolicy: "degrade-headless",
  },
  ws: {
    serverBufferedHighWaterBytes: 8388608,
    serverBufferedHardLimitBytes: 33554432,
    perClientOutputQueueMaxBytes: 2097152,
    perClientControlQueueMaxBytes: 262144,
    outputCoalesceWindowMs: 16,
  },
  clientWs: {
    inputBackpressureBytes: 1048576,
    hardReconnectBytes: 4194304,
  },
  terminal: {
    visibleOutputQueueMaxBytes: 4194304,
    visibleOutputMaxChunks: 512,
    visibleFlushBudgetBytes: 262144,
    hiddenOutputPolicy: "snapshot-restore",
    hiddenOutputTailBytes: 0,
    inputQueueMaxBytes: 65536,
    inputQueueTtlMs: 1500,
    transportOutboxMaxBytes: 65536,
    transportOutboxTtlMs: 1500,
    scrollbackLines: 10000,
  },
  snapshots: {
    perSnapshotMaxChars: 2000000,
    totalStorageBudgetChars: 3000000,
    maxEntries: 16,
    tombstoneTtlMs: 86400000,
  },
  workspaceRuntime: {
    maxLiveWorkspaces: 3,
    maxLiveTerminals: 12,
    hiddenRuntimeTtlMs: 60000,
  },
  telemetry: {
    sampleIntervalMs: 60000,
    recentEventLimit: 256,
  },
}
```

기존 `pty.maxSnapshotBytes`, `pty.scrollbackLines`, `workspace.maxWorkspaces`, `workspace.maxTabsPerWorkspace`, `workspace.maxTotalSessions`, `logging.maxSize`, `logging.maxFiles`는 보존한다. `resourceLimits`는 runtime queue/cache/backpressure/live runtime guardrail을 담당한다.

### Split WebSocket

초기에는 단일 `/ws`를 유지하고, feature flag로 split mode를 추가한다.

권장 채널:

| 채널 | 책임 |
| --- | --- |
| unified legacy | 기존 `/ws` 단일 연결. 호환 모드로 유지 |
| control/input | input, paste, resize, subscribe/unsubscribe, snapshot ack, screen repair request/ack, ping, protocol error |
| output/replay | terminal output, screen snapshot, screen repair payload, replay payload, workspace/tab/grid broadcast |

권장 handshake:

1. frontend가 control socket을 연다.
2. 서버가 JWT를 검증하고 `clientGroupId`, `controlConnectionId`, `pairToken`, split 지원 정보를 반환한다.
3. frontend가 output socket을 열고 `clientGroupId`와 `pairToken`으로 pairing한다.
4. 서버는 subscription과 replay state를 physical socket이 아니라 `clientGroupId + sessionId + subscriptionEpoch` 기준으로 관리한다.
5. output socket backpressure나 reconnect가 발생해도 control input path는 유지한다.

## 실행 로드맵

### Phase 0: 기준선과 설정 골격

목표: enforce 없이 현재 누적 정도를 관측하고, 설정과 telemetry 기반을 만든다.

작업:

1. `server/src/schemas/config.schema.ts`와 `server/src/types/config.types.ts`에 `resourceLimits` 타입과 Zod schema를 추가한다.
2. `server/src/utils/configTemplate.ts`, `server/config.json5.example`에 기본값과 주석을 추가한다.
3. `/api/runtime-config` 응답에 frontend용 non-secret resource limits를 포함한다.
4. `SessionManagerObservability`, `WsRouterObservabilitySnapshot`에 cap 관련 gauge/counter를 추가한다.
5. frontend debug telemetry에 live runtime count, xterm write queue, snapshot save duration, WS bufferedAmount 지표를 추가한다.
6. observe mode에서는 cap 초과를 기록만 하고 기존 동작을 유지한다.

완료 기준:

- 서버 시작 시 `resourceLimits` default가 validation된다.
- `/api/runtime-config`에서 frontend cap을 확인할 수 있다.
- 현재 workload에서 headless pending, WS bufferedAmount, live xterm 수, snapshot save duration을 관측할 수 있다.

### Phase 1: Unified WebSocket과 headless queue 안전화

목표: split 도입 전 기존 단일 WS와 서버 headless path를 먼저 bounded로 만든다.

작업:

1. `SessionManager.pendingOutputChunks`를 bounded deque/write pump로 교체한다.
2. headless pending output에 byte/chunk cap을 적용한다.
3. cap 초과 시 silent drop이 아니라 `degrade-headless`로 전환하고 bounded replay tail만 유지한다.
4. `Array.shift()` 기반 처리를 cursor/deque 처리로 바꾼다.
5. `deleteSession()`, PTY `onExit`, restart cleanup에서 headless queue, write chain, listener disposer를 명시 정리한다.
6. `WsRouter.safeSend()`를 도입하고 기존 direct `ws.send()` 경로를 통합한다.
7. server-side `bufferedAmount` high/critical water를 관측하고 terminal output은 per-client queue/coalesce로 처리한다.
8. critical water 초과 시 session이 아니라 client socket만 terminate한다.

완료 기준:

- PTY output burst 중 서버 heap이 cap 위로 무한 증가하지 않는다.
- 느린 WS client가 있어도 다른 client와 session은 유지된다.
- PTY shell 내부 `exit` 후 heavy session resource가 정리된다.

### Phase 2: Frontend hidden runtime 격리

목표: 브라우저 새로고침으로만 회복되던 hidden xterm 누적을 제품 동작으로 제어한다.

작업:

1. `MAX_ALIVE_WORKSPACES = 0` 상수를 runtime config 기반 `maxLiveWorkspaces`로 대체한다.
2. `maxLiveTerminals`와 `hiddenRuntimeTtlMs`를 적용한다.
3. `aliveWorkspaceIds`, `workspaceVisitOrder`, `terminalRefsMap`을 `wm.tabs`/`wm.workspaces` 기준으로 prune한다.
4. hidden terminal은 live output write를 중단한다.
5. hidden -> visible 전환 시 server snapshot/screen repair로 복원한다.
6. `TerminalHostSlot`의 per-slot observer/window listener를 Provider 단위 batch measurement로 중앙화한다.
7. xterm output write scheduler를 추가해 visible terminal output도 queue cap, coalescing, latency telemetry를 갖게 한다.

완료 기준:

- hidden workspace에서 대량 output이 발생해도 active terminal 입력 latency가 증가하지 않는다.
- live xterm/runtime 수가 설정값 이하로 유지된다.
- workspace 전환 시 화면은 snapshot/repair로 복원되고 stale 상태가 조용히 남지 않는다.

### Phase 3: Browser input backpressure와 queue 설정화

목표: 브라우저 WS 송신 buffer가 큰 상태에서도 입력이 성공 처리되는 문제를 없앤다.

작업:

1. `WebSocketContext.send()`의 `SendResult`에 `backpressure` reason을 추가한다.
2. input 전송 전 `ws.bufferedAmount`를 확인한다.
3. `inputBackpressureBytes` 이상이면 send하지 않고 transport queue로 넘긴다.
4. `hardReconnectBytes` 이상이면 socket을 close/reconnect한다.
5. `TerminalView` input queue와 `TerminalContainer` transport outbox의 byte/TTL 상수를 runtime config로 바꾼다.
6. 입력 queue expiry timer를 입력마다 만드는 구조에서 queue별 단일 timer로 바꾼다.
7. debug payload 계산을 lazy/one-pass로 정리한다.

완료 기준:

- browser send buffer high 상태에서 입력이 `{ ok: true }`로 오판되지 않는다.
- Enter 포함 입력의 timeout-enter-safety가 유지된다.
- 일반 typing 경로에서 불필요한 regex/TextEncoder/Intl.Segmenter 중복 계산이 줄어든다.

### Phase 4: Snapshot, localStorage, layout cleanup

목표: 브라우저 저장소와 cleanup 경계를 bounded로 만든다.

작업:

1. snapshot per-entry/total/maxEntries/tombstone TTL을 runtime config로 적용한다.
2. snapshot 저장 전 매번 localStorage full scan하는 흐름을 줄이고, quota error 또는 index estimate 초과 시 eviction한다.
3. `terminal_snapshot_remove_*` tombstone에 TTL과 startup cleanup을 추가한다.
4. workspace/tab 삭제 시 snapshot, tombstone, `mosaic_layout_<workspaceId>`, terminal ref를 lifecycle owner가 정리한다.
5. deletion unmount에서는 mosaic cleanup save가 stale layout을 되살리지 않게 한다.

완료 기준:

- tab/workspace 반복 생성/삭제 후 localStorage key와 ref Map이 계속 증가하지 않는다.
- snapshot quota recovery가 입력 경로를 장시간 block하지 않는다.
- active session snapshot은 보존되고 stale/corrupt/expired entry가 먼저 제거된다.

### Phase 5: Split WebSocket infrastructure

목표: input/control과 output/replay를 실제 물리 채널로 분리한다.

작업:

1. frontend transport abstraction을 만든다. 초기는 unified implementation을 그대로 사용한다.
2. server `WsRouter`에 `channelRole: single | control | output`과 logical client registry를 추가한다.
3. 기존 `/ws`는 유지하고, split mode는 query 또는 `/ws/control`, `/ws/output` 중 하나로 feature flag 뒤에 추가한다.
4. subscription/replay state를 physical socket 기준에서 `clientGroupId + sessionId + subscriptionEpoch` 기준으로 이전한다.
5. input, resize, ack, repair request는 control로 보낸다.
6. output, snapshot, repair payload, replay는 output으로 보낸다.
7. output socket disconnect 시 control socket을 유지하고 input gate 정책을 명시한다.
8. output reconnect 시 fresh snapshot/replay를 수행한다.

결정 사항:

- 초기 구현은 path 추가보다 기존 `/ws`에 `channel=control|output` query를 쓰는 방식이 호환성이 좋다.
- 최종 운영 구조는 `/ws/control`, `/ws/output`이 더 명확하다. rollout 결과를 보고 결정한다.
- `session:ready`는 control mirror를 제공해 input gate를 빨리 열 수 있게 하되, blind input 방지 조건을 별도로 둔다.

완료 기준:

- unified mode가 계속 통과한다.
- split mode에서 output flood 중 input send latency가 개선된다.
- output socket 장애가 blind input 실행으로 이어지지 않는다.
- split fallback은 telemetry에 명시적으로 남는다.

### Phase 6: Daemon/dev log rotation과 TerminalObs 축약

목표: 로그 파일과 동기 I/O 누적을 장시간 사용 안정성 문제에서 제거한다.

작업:

1. `tools/daemon/log.js`, `tools/start-runtime.js`, `tools/daemon/launcher.js`에 rotating writer를 적용한다.
2. `appendFileSync` 기반 tee를 stream/backpressure 기반 writer로 교체한다.
3. `logging.maxSize`, `logging.maxFiles`가 실제 daemon/dev log writer에 적용되게 한다.
4. `[TerminalObs]`는 기본 summary 또는 disabled로 전환한다.
5. raw replay event 배열이나 terminal payload 전문은 일반 log에 쓰지 않는다.
6. fatal daemon state의 stale PID 정리와 `dev.js` child cleanup 대기 문제를 함께 정리한다.

완료 기준:

- 장시간 실행 후 log file이 설정된 크기와 보존 개수 안에 머문다.
- TerminalObs가 대형 JSON을 매분 출력하지 않는다.
- restart/stop 이후 stale node/vite/tsx process가 남지 않는다.

### Phase 7: Long-run validation과 기본값 전환

목표: observe/enforce 결과를 바탕으로 기본값을 안전하게 바꾼다.

작업:

1. 30분 이상 soak scenario를 만든다.
2. hidden workspace output flood와 active typing을 병행한다.
3. slow WebSocket client 시뮬레이터를 만든다.
4. unified vs split mode latency와 memory trend를 비교한다.
5. telemetry 기반으로 cap default를 조정한다.
6. `maxLiveWorkspaces`, hidden output pause, safeSend enforcement를 기본 활성화한다.
7. split WebSocket은 충분한 회귀 검증 후 default candidate로 전환한다.

완료 기준:

- keydown to echo latency가 장시간 사용 중 단조 증가하지 않는다.
- server RSS/heap, browser JS heap, live xterm count가 bounded trend를 보인다.
- output flood와 slow client 상황에서 session은 유지되고 client만 복구된다.

6. grid/tab 전환, resize, IME 입력, paste, Enter 입력 회귀 확인
7. interactive AI TUI에서 키 입력, local echo, prompt redraw, cursor movement, ticker output, waiting-for-input repaint가 session을 `running`으로 전환하지 않는지 확인
8. 실제 명령 실행과 실질 agent output은 session을 `running`으로 전환하는지 확인

## 위험과 대응

| 위험 | 영향 | 대응 |
| --- | --- | --- |
| hidden output write 중단으로 scrollback 일부 손실 | 사용자가 hidden tab history를 기대할 수 있음 | server snapshot/repair는 최종 화면 복구용으로 명확히 정의하고, history 보존은 별도 기능으로 분리 |
| headless overflow degrade | authoritative snapshot 품질 저하 | silent drop보다 명시적 degrade가 안전하다. telemetry와 UI/debug 표시 추가 |
| 느린 client terminate UX | reconnect 순간적인 깜박임 | session은 유지하고 output socket만 복구한다. snapshot restore로 화면 복원 |
| split WS 복잡도 증가 | ordering, reconnect, test matrix 증가 | unified safeSend 안정화 후 feature flag로 단계 도입 |
| blind input 위험 | output 상태를 못 본 채 input 실행 | control-ready와 output-ready를 분리하고, 위험 상태에서는 Enter 포함 입력을 queue/reject |
| 너무 낮은 cap default | 정상 대량 output에서 자주 degrade | observe mode 데이터를 기반으로 default 조정 |
| telemetry 비용 증가 | 로그/메모리 누적 재발 | aggregate counter 중심, bounded recent event, raw payload 금지 |

## 운영 롤아웃

1. **Observe release**
   - cap 초과를 기록만 한다.
   - current defaults는 대부분 기존 동작 유지.
2. **Server guardrail release**
   - headless queue, safeSend, slow client terminate를 enforce한다.
   - split WS는 아직 비활성.
3. **Frontend lifecycle release**
   - hidden output pause, live runtime LRU, xterm scheduler를 opt-in 또는 낮은 위험 default로 켠다.
4. **Split transport preview**
   - feature flag로 control/output socket을 활성화한다.
   - unified fallback 유지.
5. **Default hardening**
   - soak 결과가 안정적이면 `maxLiveWorkspaces`, hidden output pause, safeSend enforcement를 기본값으로 전환한다.
6. **Split default candidate**
   - output flood에서 input latency 개선이 충분하고 회귀가 없을 때만 split mode default 전환을 검토한다.

## 결론

사용자의 두 아이디어는 모두 효과적이다. 다만 적용 순서가 중요하다.

먼저 설정 기반 cap과 telemetry로 무제한 누적을 막아야 한다. 이 단계에서 `SessionManager` headless queue, unified WS backpressure, hidden terminal output write 중단을 처리하면 서버와 브라우저의 가장 큰 누적 원인이 줄어든다.

그 다음 보내는 WebSocket과 받는 WebSocket을 분리하면 output flood가 input path를 같이 막는 구조를 줄일 수 있다. split WS는 효과가 있지만 복잡도도 크므로, unified mode safeSend와 frontend runtime lifecycle 정리가 먼저다.

최종 목표는 "새로고침해야 회복되는 앱"이 아니라, 느린 client와 hidden output, 대량 terminal output이 발생해도 resource cap, backpressure, snapshot/repair restore, telemetry를 통해 스스로 bounded 상태로 회복하는 BuilderGate다.
