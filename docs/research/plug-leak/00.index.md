# BuilderGate Plug-Leak Wave Plan Index

작성일: 2026-06-25

## 목적

BuilderGate의 장시간 사용 지연, 세션 종료 후 PowerShell/bash/WSL 및 하위 프로세스 잔존, headless output queue 누적, WebSocket backpressure 부재를 한 번에 크게 고치지 않고 wave 단위로 나누어 구현한다. 각 wave는 독립 검증과 롤백 기준을 가진다.

## 전제

- 서버 작업을 먼저 끝낸다.
- 프론트엔드 runtime residency, hidden output, 브라우저 worker, UI 적용은 마지막 단계로 둔다.
- 원형 큐 크기, WebSocket queue limit, runtime residency limit 같은 guardrail 값은 최종적으로 브라우저 Settings 메뉴에서 변경 가능해야 한다.
- Settings persistence source of truth는 `config.json5`와 `/api/settings`다. `/api/runtime-config`는 브라우저가 읽는 public projection이다.
- `split WebSocket`은 이번 plug-leak 1차 해결의 핵심 경로가 아니다. unified `/ws`에서 bounded queue와 `safeSend`를 먼저 검증한 뒤 별도 후속으로 다룬다.
- 구현 중 broad kill은 금지한다. `taskkill /IM node.exe`, `killall`, process name 기반 kill은 사용하지 않는다.

## 5개 서브에이전트 연구 추적성

| 연구 축 | 핵심 결론 | 반영 Wave |
| --- | --- | --- |
| 세션 종료와 process tree | `deleteSession()`/workspace close는 PTY kill 이후 OS descendant 종료를 검증하지 않고, orphan recovery가 의도적 종료 세션을 재생성할 수 있다. | Wave 1, 2, 3, 5 |

| Settings/config/UI | `resourceLimits`와 `wsTransportMode`는 테스트/문서 기대와 source가 어긋나 있다. `config.json5`와 `/api/settings`를 source of truth로 복구해야 한다. | Wave 0, 6, 7 |

## Wave 순서 원칙

1. Wave 0은 반드시 가장 먼저 한다. 현재 source와 untracked stability tests/docs 사이에 `resourceLimits`, `wsTransportMode`, runtime-config export drift가 있다.
2. Wave 1은 observe-only다. 실제 강제 종료 전에 PID ownership과 cleanup dry-run evidence를 먼저 쌓는다.
3. Wave 2와 Wave 3은 세션 종료 의미를 바로잡는다. 프로세스 종료와 workspace/orphan lifecycle은 따로 보이지만 실제로는 함께 안정화해야 한다.
4. Wave 4는 output/backpressure다. headless queue와 server WS send queue를 bounded로 만든다.
5. Wave 5는 서버만으로 장기 사용 회귀를 검증한다.
6. Wave 6은 사용자가 요구한 Settings 조정 가능성을 구현한다. 서버-safe defaults와 Settings persistence가 먼저다.
7. Wave 7에서야 프론트엔드 runtime을 줄인다. 서버 누수가 잡히기 전 프론트만 줄이면 원인 분리가 어렵다.

## 공통 검증 정책

- 각 wave는 관련 unit/integration test와 최소 1회 서브에이전트 리뷰를 통과해야 한다.
- Phase 완료 리뷰 규칙에 따라 리뷰어가 findings를 내면 수정 후 재리뷰한다.
- 세션 상태 invariant는 유지한다. AI TUI에서 사용자가 타이핑하는 동안 session은 `idle`이어야 한다.
- raw terminal input/output payload는 telemetry/log에 남기지 않는다.
- Settings는 invalid 값을 silent coercion하지 않는다. unsupported/invalid value는 observable error로 거절한다.

## 완료 정의

전체 plug-leak 계획은 다음이 모두 참일 때 완료된다.

- 세션 닫기, tab delete, workspace delete, restart old session, direct session delete, natural PTY exit, daemon shutdown의 cleanup 의미가 문서화되고 테스트된다.
- verified-owned process tree는 종료되고 unverified/detached child는 degraded cleanup으로 보고된다.
- headless pending output queue는 byte/chunk cap을 가진 O(1) dequeue 구조이며 overflow는 silent drop/overwrite가 아니라 `degrade-headless`로 간다.
- 서버 WebSocket send는 `readyState`만 보지 않고 bounded queue, `bufferedAmount`, send callback error, slow-client policy를 적용한다.
- 설정 값은 `config.json5` 및 `/api/settings`에서 관리되고, 브라우저 Settings 메뉴에서 변경 가능하다.
- 프론트엔드 hidden runtime이 무제한으로 mount/subscription/write를 유지하지 않는다.
