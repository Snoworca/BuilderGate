# BuilderGate Resource Cap Configuration Plan

작성일: 2026-06-17

## 조사 목적

이 문서는 기존 누수/입력 지연 분석 문서인 `docs/research/2026-06-17.buildergate-memory-input-lag-research.md`를 바탕으로, 사용자의 아이디어 1번인 "chunk 큐나 cache 크기를 설정으로 제한하여 메모리 과사용을 방지"를 장기 개선 계획으로 구체화한다.

이번 작업은 읽기 전용 코드 조사와 계획 문서 작성만 수행했다. 코드 수정은 하지 않았다.

## 문제 요약

BuilderGate의 장시간 사용 시 메모리 증가와 입력 지연은 단일 캐시 하나의 문제가 아니라 서버 headless terminal, WebSocket 송신 버퍼, 브라우저 xterm write queue, localStorage snapshot, live workspace/runtime 유지 정책이 같이 누적되는 구조 문제다.

현재 확인된 주요 cap 공백은 다음과 같다.

| 영역 | 현재 상태 | 위험 |
| --- | --- | --- |
| 서버 headless output | `server/src/services/SessionManager.ts:2252`의 `queueHeadlessOutput()`가 `pendingOutputChunks.push(data)`와 Promise chain으로 출력마다 write 작업을 연결한다. queue byte/chunk cap이 없다. | headless write가 PTY 출력보다 느리면 서버 heap에 chunk와 Promise closure가 계속 쌓인다. `shift()` 비용도 queue가 커질수록 증가한다. |
| 서버 WebSocket 송신 | `server/src/ws/WsRouter.ts:1490`, `server/src/ws/WsRouter.ts:1708`, `server/src/services/SessionManager.ts:1977` 경로가 `readyState === OPEN`만 보고 `send()`한다. | 느린 브라우저나 막힌 socket마다 `bufferedAmount`가 커져 서버 메모리와 JSON 직렬화 비용이 증가한다. |
| 브라우저 WebSocket 송신 | `frontend/src/contexts/WebSocketContext.tsx:434`의 `send()`가 `ws.send()` 후 바로 `{ ok: true }`를 반환한다. | 브라우저 송신 버퍼가 이미 큰 상태에서도 입력이 성공 처리되어 echo/복구가 늦어진다. |
| xterm output write | `frontend/src/components/Terminal/TerminalView.tsx:1123`의 `writeOutput()`이 `inFlightOutputRef`에 push/shift만 한다. depth/bytes/latency cap이 없다. | xterm write callback이 밀릴 때 브라우저 메인 스레드 부하와 pending output이 증가한다. |
| hidden terminal output | `frontend/src/components/Terminal/TerminalContainer.tsx:1479`에서 visibility와 무관하게 `terminalRef.current?.write(data)`를 호출한다. | 보이지 않는 터미널도 xterm write, snapshot save, scrollback 증가를 계속 만든다. |
| snapshot/localStorage | `frontend/src/utils/terminalSnapshot.ts:3`에 총 3,000,000 chars, `:6`에 snapshot content 2,000,000 chars 제한은 있지만 설정화되어 있지 않고, 저장 전 localStorage 전체 scan이 반복된다. | 제한 자체는 있으나 운영자가 조정할 수 없고, eviction 비용이 입력 지연에 영향을 줄 수 있다. |
| live workspace/runtime | `frontend/src/App.tsx:42`의 `MAX_ALIVE_WORKSPACES = 0`은 무제한이다. 서버에는 `workspace.maxWorkspaces/maxTabsPerWorkspace/maxTotalSessions`가 있지만 클라이언트 live xterm 수 제한과는 다르다. | 숨겨진 workspace/tab의 xterm, observer, subscription, scrollback이 계속 살아 있다. |

반대로 이미 cap이 있는 영역도 있다.

- `frontend/src/components/Terminal/TerminalView.tsx:62`의 input queue는 `64 * 1024` bytes, TTL 1500ms다.
- `frontend/src/components/Terminal/TerminalContainer.tsx:31`의 transport outbox도 `64 * 1024` bytes, TTL 1500ms다.
- `server/src/ws/WsRouter.ts:40`의 replay queued input은 `64 * 1024` bytes, age 3000ms다.
- `server/src/schemas/config.schema.ts:62`의 server headless scrollback은 기본 1000, 최대 50000 line이다.
- `server/src/schemas/config.schema.ts:63`의 `pty.maxSnapshotBytes`는 기본 2,097,152 bytes다.

장기 개선의 핵심은 "제한이 없는 곳에 숫자를 넣는 것"만이 아니다. 각 queue/cache가 초과될 때 무엇을 보존하고 무엇을 버릴지, 그리고 그 사실을 어떻게 복구/관측할지를 명확히 해야 한다.

## 설계 원칙

1. **서버 보호가 최우선이다.** 느린 클라이언트나 브라우저 탭 하나가 PTY 세션이나 서버 프로세스 전체를 밀어 올리면 안 된다.
2. **terminal state를 조용히 손상시키지 않는다.** headless xterm에 쓰기 전 output을 drop하면 authoritative snapshot이 틀어진다. headless backlog cap 초과 시 기본 정책은 drop이 아니라 `degrade` 후 bounded fallback으로 전환해야 한다.
3. **느린 클라이언트는 session이 아니라 socket 단위로 격리한다.** WebSocket output backpressure는 기본적으로 slow client close/terminate와 reconnect snapshot restore로 해결한다.
4. **hidden terminal은 live output을 처리하지 않는다.** 보이지 않는 xterm에 output을 계속 쓰는 것은 입력 지연의 직접 원인이다. hidden runtime은 server snapshot/screen repair를 신뢰하고 client-side output tail을 기본 0으로 둔다.
5. **cap 초과는 항상 관측 가능해야 한다.** drop, coalesce, terminate, snapshot restore, reject, degrade 모두 telemetry event와 aggregate counter를 남겨야 한다.
6. **운영 guardrail과 사용자 취향 설정을 분리한다.** 서버 heap과 socket 보호용 hard cap은 env/config가 통제하고, UI 표시/scrollback/live workspace 같은 선호 설정은 서버가 내려준 상한 안에서만 local override를 허용한다.
7. **기존 계약을 보존하며 단계적으로 enforce한다.** 첫 단계는 observe/warn 가능한 설정과 telemetry부터 넣고, 위험한 영역은 allowlist 또는 feature flag로 enforce한다.
8. **입력 상태 invariant를 유지한다.** 입력 queue, WebSocket backpressure, snapshot restore 대기는 사용자의 키 입력만으로 session status를 `running`으로 바꾸면 안 된다.

| 환경변수 | 긴급 hardening, CI/perf test override, 운영 상한 강제 | 서버 시작 전에 적용되어 가장 강한 guardrail로 쓸 수 있다. | 현재는 `BUILDERGATE_INPUT_RELIABILITY_MODE`처럼 일부만 env 기반이다. 많은 개별 env는 관리가 번잡하다. |
| browser `localStorage` | localhost test override, 개인 UI 선호 | 브라우저별 tuning과 E2E test에 편하다. | 보안/운영 guardrail로 쓰면 안 된다. 서버 hard cap을 완화할 수 없어야 한다. |

### 권장 override 우선순위

서버에서 사용하는 값:

1. 코드/schema default
2. bootstrap template로 생성된 `config.json5` default
3. 사용자가 편집한 `config.json5`
4. 환경변수 override
5. Settings API를 통해 저장된 runtime config

단, 4와 5의 관계는 항목별로 나눠야 한다.

- **hard cap 항목**: env가 최종 상한이다. Settings UI는 env 상한보다 큰 값을 저장하려 하면 validation error 또는 capped warning을 반환한다.
- **운영자가 완화할 수 있는 soft cap 항목**: Settings UI가 config를 업데이트하고 즉시 반영할 수 있다. env는 기본값 override로만 쓴다.
- **client local setting**: `/api/runtime-config`의 hard bound 안에서만 적용된다. localhost/test 전용 override도 서버가 내려준 hard bound를 넘지 못한다.

이 우선순위가 필요한 이유는 브라우저 localStorage나 Settings UI가 서버 메모리 보호 장치를 느슨하게 만들면 cap의 의미가 사라지기 때문이다.

## 권장 설정 항목

새 섹션 이름은 `resourceLimits`를 권장한다. 기존 `pty.maxSnapshotBytes`, `pty.scrollbackLines`, `workspace.maxWorkspaces`, `workspace.maxTabsPerWorkspace`, `workspace.maxTotalSessions`는 유지하고, queue/backpressure/cache/live-runtime처럼 범용 resource guardrail은 `resourceLimits` 아래로 모은다.

초기 기본값은 기존 동작을 최대한 보존하되, 무제한 영역에는 보수적인 cap을 둔다. 실제 release 전에는 telemetry 기반으로 한 번 조정해야 한다.

## 초과 정책 매트릭스

| 영역 | 기본 정책 | 허용 가능한 대안 | 금지 또는 비권장 |
| --- | --- | --- | --- |
| headless pending output | `coalesce` 후 `degrade-headless` | strict 운영에서 `terminate-session` | headless write 전 output을 조용히 drop하는 정책 |
| server terminal WS output | per-client queue + coalesce, hard limit에서 client terminate | overflow 시 reconnect 유도 | 느린 client 때문에 PTY session을 kill |
| server WS control message | 작은 control queue, overflow 시 client terminate | immediate send failure logging | snapshot/control event drop |
| browser WS input send | `backpressure` reason으로 reject해서 existing queue 처리 | hard limit에서 reconnect | `ws.send()` 성공 처리 후 브라우저 buffer에 무한 누적 |
| visible xterm output | frame coalesce + queue cap, overflow 시 screen repair | repeated overflow 시 runtime remount | 표시 상태가 틀어진 채 pending queue만 drop |
| hidden xterm output | write 중단, visible 시 snapshot/screen repair | debug tail 소량 유지 | hidden terminal마다 output 문자열 누적 |
| localStorage snapshot | active 보존, 오래된/corrupt/expired tombstone eviction | quota error 시 추가 eviction | 입력 경로에서 매번 전체 scan을 오래 수행 |
| live workspace/runtime | LRU unmount, active 보존, restore on demand | maxLiveTerminals 동시 적용 | hidden xterm/observer/subscription 무제한 유지 |
| server workspace/session creation | 기존 `workspace.max*` reject | admin override | cap 초과 요청을 다른 workspace/session으로 조용히 대체 |

## 구현 계획

### Phase 0: 설정 모델과 관측 기반 정비

목표는 enforce 전에 현재 사용량과 cap 후보가 실제 workload에서 어느 정도인지 보는 것이다.

1. `server/src/schemas/config.schema.ts`와 `server/src/types/config.types.ts`에 `resourceLimits` schema/type을 추가한다.
2. `server/src/utils/configTemplate.ts`에 안전한 기본값을 추가한다.
3. `RuntimeConfigStore`, `SettingsService`, `ConfigFileRepository`는 우선 advanced 설정을 노출하지 않아도 된다. 첫 구현은 `config.json5` + `/api/runtime-config` 중심으로 충분하다.
4. `/api/runtime-config` 응답에 non-secret client resource limits를 추가한다. browser는 이 값을 읽어 input queue, xterm scheduler, snapshot budget, live workspace cap에 적용한다.
5. 환경변수 override는 초기에 두 가지 방식 중 하나를 선택한다.
   - 개별 env: `BUILDERGATE_WS_SERVER_BUFFERED_HARD_LIMIT_BYTES`처럼 명확하지만 키가 많다.
   - JSON env: `BUILDERGATE_RESOURCE_LIMITS_JSON` 하나로 관리하지만 validation error reporting이 중요하다.
6. telemetry에 현재 cap 후보 값을 포함한다.
   - session: pending headless chunks/bytes, oldest age, max observed pending bytes
   - ws: per-client `bufferedAmount`, queued output bytes, slow client count
   - client debug: xterm write queue bytes/depth, write latency, hidden output skipped count
   - snapshot: save count, eviction count, eviction ms, quota failures
   - runtime: live workspace/terminal count, evictions, restores
7. observe mode에서는 cap 초과 시 event만 기록하고 기존 동작을 유지한다. 단, 명백한 hard limit인 browser/localStorage quota error는 기존처럼 recovery한다.

### Phase 1: 서버 headless queue cap

서버 heap 보호에서 가장 중요한 단계다.

1. `pendingOutputChunks: string[]`와 `headlessWriteChain = chain.then(() => apply(data))` 구조를 write pump로 바꾼다.
2. write pump는 다음 상태를 가진다.
   - queued chunks deque
   - queued bytes
   - write in progress flag
   - oldest queued timestamp
   - dropped/degraded counters
3. enqueue 시 chunk count가 cap을 넘으면 먼저 인접 chunk를 병합한다.
4. byte cap을 넘으면 default 정책으로 headless를 degraded 처리한다.
   - `sessionData.headless` dispose
   - `headlessCloseSignal.resolve()`
   - snapshot cache + unsnapshotted output + pending tail을 `pty.maxSnapshotBytes` 기준으로 truncate
   - live WebSocket output route는 계속 수행
   - screen repair/snapshot 요청에는 degraded reason을 명확히 반환
5. write 완료 후에는 `screenSeq`를 증가시키고 실제 flushed output만 `appendUnsnapshottedOutput()`에 반영한다.
6. `Array.shift()` 대신 cursor/deque를 사용한다.
7. cleanup 경로에서 pump queue, pending Promise, close signal, debug counters를 명시적으로 초기화한다.
8. 테스트:
   - small output은 기존처럼 snapshot/replay가 유지된다.
   - chunk cap 초과는 coalesce만 발생하고 terminal state가 유지된다.
   - byte cap 초과는 degraded로 전환되고 heap이 bounded tail만 유지한다.
   - session delete/onExit/restart에서 queue 참조가 해제된다.

### Phase 2: 서버 WebSocket backpressure와 per-client output queue

느린 client 하나가 서버 process를 끌고 가는 문제를 막는다.

1. `WsRouter`에 `safeSend(ws, payload, options)`를 추가하고 `routeSessionOutput()`, `broadcastAll()`, `sendTo()`, `SessionManager.broadcastWs()`가 이를 사용하게 한다.
2. `safeSend()`는 전송 전후 다음을 기록한다.
   - encoded byte length
   - `ws.bufferedAmount`
   - message kind: terminal output, control, workspace event, debug
   - clientId, subscribed session count
3. terminal output은 client별/session별 outbox로 coalesce한다.
   - `bufferedAmount <= lowWater`이면 즉시 send
   - highWater 초과면 outbox에 append
   - outbox cap 초과 또는 hard limit 초과면 client를 close/terminate
4. control message는 output outbox와 분리한다.
   - snapshot ack/reject/status/error는 drop하지 않는다.
   - control queue cap 초과 시 client를 close하고 reconnect를 유도한다.
5. close code는 가능하면 `1013 Try Again Later`를 사용하고, reason은 `slow-client-backpressure`처럼 관측 가능한 문자열로 둔다.
6. 기존 replay/screen repair queue cap과 통합한다.
   - replay pending output은 이미 `sessionManager.getReplayQueueLimit()`를 사용한다.
   - safeSend 단계에서 다시 막히면 per-client queue 또는 close 정책으로 처리한다.
7. 테스트:
   - fake WebSocket의 `bufferedAmount`가 highWater 아래면 즉시 send
   - highWater 초과 terminal output은 coalesce queue로 이동
   - hard limit 초과는 client terminate, session 유지
   - control message overflow는 drop이 아니라 close/reject telemetry
   - `/api/sessions/telemetry`에 slow client counter와 max buffered amount가 나온다.

### Phase 3: 브라우저 WebSocket input backpressure와 transport outbox 설정화

입력 지연을 "성공으로 보이는 실패"에서 "명시적으로 queue/reject되는 상태"로 바꾼다.

1. `frontend/src/contexts/WebSocketContext.tsx`의 `SendResult`에 `backpressure` reason을 추가한다.
2. input 전송 전 `ws.bufferedAmount`를 확인한다.
   - `inputBackpressureBytes` 이상이면 `ws.send()`를 호출하지 않고 `{ ok: false, reason: 'backpressure' }`를 반환한다.
   - `hardReconnectBytes` 이상이면 socket을 close하고 reconnect를 유도한다.
3. `TerminalContainer`의 `classifyTransportQueueDecision()`은 `backpressure`를 transient queue 가능한 reason으로 처리한다.
4. `TRANSPORT_INPUT_QUEUE_BYTE_BUDGET`, `TRANSPORT_INPUT_QUEUE_TTL_MS`, `RECONNECT_INPUT_QUEUE_TTL_MS`를 runtime config로 대체한다.
5. queue expiry timer는 입력마다 timer를 늘리는 방식이 아니라 queue별 단일 timer로 바꾼다.
6. 테스트:
   - `bufferedAmount` high 상태에서 input send가 backpressure reason을 반환한다.
   - queue mode에서는 transport outbox에 들어가고 TTL 내 reconnect 후 flush된다.
   - strict mode 또는 TTL 초과 시 reject telemetry가 남는다.
   - Enter 포함 입력은 timeout-enter-safety를 유지한다.

### Phase 4: xterm output write scheduler와 hidden output 중단

브라우저 메인 스레드 부하를 줄이는 단계다.

1. `TerminalView.writeOutput()`을 scheduler로 감싼다.
   - chunk enqueue
   - frame/timer 단위 coalesce
   - in-flight write 하나 또는 제한된 개수만 허용
   - pending bytes/chunks/oldest age 측정
2. visible terminal overflow 정책은 "pending output drop + server screen repair"다.
   - client 표시가 stale임을 표시하는 내부 flag를 세운다.
   - pending queue를 비운다.
   - server authoritative snapshot/screen repair를 요청한다.
   - repair 성공 시 stale flag를 해제한다.
3. hidden terminal은 `onOutput`에서 xterm write를 호출하지 않는다.
   - `hiddenOutputPolicy=snapshot-restore`면 output data를 client에 저장하지 않는다.
   - visible 전환 시 pending server snapshot이 있으면 적용하고, 없으면 screen repair를 요청한다.
4. `inputQueueMaxBytes`, `inputQueueTtlMs`도 runtime config에서 읽는다.
5. xterm `scrollback`은 `resourceLimits.terminal.scrollbackLines`로 설정한다. 기본은 현재 값 10000으로 두고, 운영자가 낮출 수 있게 한다.
6. 테스트:
   - fake terminal write callback이 지연되어도 pending bytes가 cap 이상 증가하지 않는다.
   - visible overflow 후 screen repair가 요청된다.
   - hidden 상태에서 output을 많이 받아도 `term.write()` 호출 수가 증가하지 않는다.
   - hidden -> visible 전환 후 snapshot/screen repair로 최신 화면이 복구된다.

### Phase 5: snapshot/localStorage budget 정비

현재 localStorage budget은 숫자 상수로 존재하지만 비용과 운영 조정성이 약하다.

1. `TERMINAL_SNAPSHOT_MAX_CONTENT_LENGTH`, `TERMINAL_SNAPSHOT_STORAGE_BUDGET_CHARS`를 runtime config 기반 값으로 바꾼다.
2. 저장 전 매번 full scan하는 흐름을 줄인다.
   - snapshot index key를 유지한다.
   - 정상 저장은 우선 `setItem` 시도
   - quota error 또는 index estimate 초과 시 eviction
3. eviction 우선순위:
   - corrupt entry
   - 만료된 removal tombstone
   - 현재 session이 아닌 오래된 snapshot
   - 큰 snapshot
   - 마지막으로 현재 session 외 removal marker
4. `terminal_snapshot_remove_*`에는 TTL을 적용한다.
5. localStorage 실패는 입력 경로를 막지 않고 server snapshot restore로 degrade한다.
6. 테스트:
   - MemoryStorage 기반 unit test로 total budget, per snapshot cap, max entry cap 검증
   - quota error retry 후 저장 성공/실패 검증
   - tombstone TTL cleanup 검증
   - active session snapshot 보존 검증

### Phase 6: live workspace/tab/terminal cap

현재 `MAX_ALIVE_WORKSPACES=0`을 runtime config로 대체한다.

1. `workspaceRuntime.maxLiveWorkspaces`와 `workspaceRuntime.maxLiveTerminals`를 적용한다.
2. LRU 기준:
   - active workspace와 active tab은 evict 금지
   - 최근 전환한 workspace는 `hiddenRuntimeTtlMs` 동안 유지
   - cap 초과 시 가장 오래된 hidden workspace/tab부터 unmount
3. unmount 시 다음을 정리한다.
   - `terminalRefsMap`
   - host slot
   - WebSocket session subscription
   - xterm instance/addons/observer/listener
   - pending client-side write/input queue
4. 재활성화 시 server snapshot/screen repair로 복원한다.
5. server `workspace.maxWorkspaces/maxTabsPerWorkspace/maxTotalSessions`는 creation cap으로 유지한다. frontend live cap은 생성 제한이 아니라 mounted runtime 제한이다.
6. 테스트:
   - workspace 5개, terminal 다수 생성 후 live runtime 수가 cap 이하인지 확인
   - evicted workspace 재진입 시 snapshot restore
   - active terminal input 중 LRU evict가 발생하지 않음
   - close/delete/restart와 LRU unmount가 충돌하지 않음

### Phase 7: Settings UI와 운영 rollout

처음부터 모든 knob을 UI에 노출하지 않는다.

1. 1차 UI 노출 후보:
   - max live workspaces
   - max live terminals
   - xterm scrollback lines
   - snapshot total budget
2. 서버 hard cap 후보는 advanced 또는 config 파일 전용으로 둔다.
   - headless pending output max
   - server WS buffered hard limit
   - per-client output queue max
3. `SettingsApplySummary`는 다음 apply scope를 추가하거나 기존 scope를 명확히 재사용한다.
   - `immediate`: WS/server/client runtime에서 즉시 읽는 cap
   - `new_sessions`: xterm scrollback, PTY scrollback처럼 새 runtime/session에 적용되는 값
   - `restart_required`: env hard cap 또는 process start 시점에만 안전한 값
4. rollout 순서:
   - observe-only telemetry release
   - dev/local enforce
   - slow-client terminate enforce
   - hidden output stop enforce
   - live workspace cap default enable
5. rollback은 config/env로 가능해야 한다.
   - 예: `resourceLimits.workspaceRuntime.maxLiveWorkspaces: 0`으로 이전 무제한 유지
   - 단, server hard cap은 rollback하더라도 최소 안전 상한을 유지한다.

## 테스트와 검증 계획

이 개선은 semantic behavior를 바꾸므로 Tier 2 이상으로 다뤄야 한다.

### 단위 테스트

| 대상 | 테스트 |
| --- | --- |
| config schema | `resourceLimits` default, min/max validation, env override, invalid env warning |
| RuntimeConfigStore | snapshot에 non-secret resource limits 포함, apply scope 분류 |
| SessionManager headless pump | normal write ordering, chunk coalesce, byte overflow degrade, cleanup 후 참조 해제 |
| WsRouter safeSend | highWater queue, hardLimit terminate, terminal/control queue 분리, telemetry counter |
| WebSocketContext send | browser `bufferedAmount` high 상태에서 `backpressure` reason 반환 |
| TerminalContainer transport outbox | configured byte/TTL 적용, backpressure queue/reject, 단일 expiry timer |
| TerminalView xterm scheduler | slow write callback, queue cap overflow, screen repair trigger, hidden write skip |
| terminalSnapshot | per snapshot cap, total budget eviction, max entries, tombstone TTL, quota recovery |
| App live runtime LRU | active 보존, hidden evict, refs prune, restore request |

### 통합/E2E 테스트

모든 manual validation과 Playwright E2E는 프로젝트 규칙에 따라 `https://localhost:2002`를 대상으로 한다.

1. **hidden output isolation**
   - workspace A를 active, workspace B를 hidden으로 둔다.
   - B에서 대량 output을 발생시킨다.
   - A의 key input latency와 xterm write queue depth가 증가하지 않는지 확인한다.
   - B로 돌아왔을 때 snapshot/screen repair로 최신 화면이 복구되는지 확인한다.
2. **slow WebSocket client**
   - fake/stub WebSocket 또는 Playwright CDP throttling으로 server-side `bufferedAmount` 증가를 재현한다.
   - highWater에서는 output queue/coalesce가 발생한다.
   - hardLimit에서는 client만 close되고 session은 유지된다.
3. **browser input backpressure**
   - browser `ws.bufferedAmount`를 stub 가능한 단위 테스트로 high 상태로 만든다.
   - input send가 성공 처리되지 않고 queue 또는 reject telemetry를 남기는지 확인한다.
4. **headless backlog cap**
   - headless write를 지연시키고 PTY output을 빠르게 enqueue한다.
   - cap 초과 시 degraded로 전환되고 heap-retained pending bytes가 bounded인지 확인한다.
5. **snapshot budget**
   - localStorage에 여러 session snapshot과 removal marker를 채운다.
   - save 시 오래된 snapshot이 eviction되고 active snapshot은 보존되는지 확인한다.
6. **live runtime cap**
   - workspace/tab을 cap 이상 생성한다.
   - mounted `TerminalContainer`, xterm instance, HostSlot observer 수가 cap 이하로 유지되는지 확인한다.
7. **soak/perf smoke**
   - 10-30분 동안 output 많은 hidden session과 active typing을 병행한다.
   - JS heap, server RSS, `/api/sessions/telemetry`, debug capture를 수집한다.
   - keydown to echo latency가 시간이 지날수록 단조 증가하지 않는지 확인한다.

### 완료 기준

- 서버 RSS와 heap이 headless/WS cap 초과 시 bounded behavior를 보인다.
- slow client는 session을 죽이지 않고 client reconnect/snapshot restore로 복구된다.
- hidden output이 active terminal 입력 latency를 실질적으로 증가시키지 않는다.
- cap 초과 정책이 telemetry와 debug capture에 남는다.
- localStorage quota나 eviction이 입력 경로를 장시간 block하지 않는다.
- live xterm/runtime 수가 설정값 아래로 유지된다.

## 위험과 트레이드오프

1. **terminal scrollback 손실**
   - hidden output write를 중단하고 snapshot restore에 의존하면 client-side scrollback 전체를 보존하지 못할 수 있다.
   - 서버 snapshot은 viewport 중심이라 "최종 화면 복구"에는 강하지만 긴 history 보존에는 약하다.
2. **headless degraded 전환의 기능 저하**
   - headless backlog overflow에서 degrade하면 authoritative snapshot 품질이 떨어진다.
   - 하지만 output drop으로 잘못된 snapshot을 만드는 것보다 명시적 degrade가 낫다.
3. **느린 client terminate의 UX**
   - hard limit에서 socket을 닫으면 사용자는 reconnect/restore 지연을 볼 수 있다.
   - 대신 서버와 다른 client를 보호한다.
4. **너무 낮은 기본값의 false positive**
   - 대량 build/test output이나 긴 로그 tail에서 cap이 자주 발동할 수 있다.
   - observe-first rollout과 telemetry 기반 default 조정이 필요하다.
5. **설정 surface 증가**
   - 모든 cap을 Settings UI에 노출하면 운영자가 의미를 이해하기 어렵다.
   - advanced config와 UI 노출 항목을 분리해야 한다.
6. **runtime config 동기화**
   - frontend가 `/api/runtime-config`를 초기 load만 하면 Settings 변경 즉시 반영이 어렵다.
   - phase 1에서는 reload 필요로 두고, 이후 workspace event 또는 runtime-config-changed event를 추가할 수 있다.
7. **telemetry 자체의 비용**
   - cap 관측을 위해 event를 늘리면 로그와 debug buffer가 또 다른 누적 원인이 될 수 있다.
   - aggregate counter 중심, recent event ring bounded, raw payload 금지를 지켜야 한다.

## 핵심 결론

가장 우선순위가 높은 작업은 `SessionManager` headless pending output을 bounded write pump로 바꾸고, 서버/브라우저 WebSocket 송신에 `bufferedAmount` 기반 backpressure를 넣는 것이다. 이 두 영역은 서버 heap과 입력 지연을 직접 보호한다.

브라우저 쪽에서는 hidden terminal live output write를 중단하고, visible 전환 시 server snapshot/screen repair로 복구하는 정책이 핵심이다. `MAX_ALIVE_WORKSPACES=0`도 장기적으로는 안전하지 않으므로 `maxLiveWorkspaces=3`, `maxLiveTerminals=12` 정도의 LRU cap을 기본값으로 검증하는 것이 좋다.

기존에 이미 존재하는 `64 KiB` 입력 queue, `2,000,000/3,000,000` snapshot budget, `pty.maxSnapshotBytes=2 MiB`는 좋은 출발점이다. 다만 숫자 상수로 남겨두지 말고 `resourceLimits` 아래에서 설정화하고, 초과 정책과 telemetry를 함께 붙여야 운영 가능한 개선이 된다.
