# Claude Code 실행 시에만 드래그 선택·Ctrl+C·우클릭 복사가 모두 막히는 문제 조사

Date: 2026-07-11
Scope: 프론트엔드 웹 터미널(xterm.js) — 전용 REQ 없음 (§9 SRS 추적 참고)
관련(별개 원인): `docs/research/2026-07-06.terminal-context-menu-copy-disabled-research.md`

## 1. 증상

BuilderGate 웹 터미널에서 `claude`(Claude Code CLI/TUI)를 실행한 뒤,
Claude Code 화면 내부의 텍스트를 **마우스로 드래그하여 선택하고 Ctrl+C를 눌러도 복사되지 않는다.**
또한 **우클릭 컨텍스트 메뉴의 "복사" 항목도 비활성화(disabled)** 상태로 나타난다.
결정적으로, 이 증상은 **오직 Claude Code를 실행할 때만** 발생하며, **일반 셸 프롬프트(pwsh/bash 등)에서는 드래그 선택·복사가 정상 동작**한다.

## 2. 기존 조사와의 관계 (별개 원인임을 명확히)

`2026-07-06` 문서는 "**고빈도 출력 중** scrollback(10,000줄) 트리밍으로 이미 만들어진 선택이 **소실**되는" 레이스/타이밍 시나리오를 다룬다. 그 경우 원인은 선택이 *생겼다가 지워지는* 것이며, 조용한 세션에서는 거의 재현되지 않는다.

본 조사의 증상은 근본 원인이 다르다. Claude Code 실행 중에는 드래그해도 **선택이 애초에 생성되지 않는다.** 출력 빈도·세션 길이와 무관하며, "Claude Code일 때만"이라는 조건이 결정적이다. 두 문서는 서로 독립적인 원인을 기술한다.

## 3. 조사 방법

서로 다른 가설을 교차 검증하기 위해 4개의 서브에이전트를 병렬 투입했다(확증편향 방지를 위해 각자 증상과 조사 축만 받고, 서로의 결론을 공유하지 않은 채 독립적으로 근거를 도출).

| Agent | 초점 | 결론 요약 |
|---|---|---|
| A. xterm.js 마우스모드 ↔ 선택 상호작용 | vendored `@xterm/xterm` v6 소스 | 애플리케이션 마우스 트래킹 모드(DECSET `?1000/?1002/?1003h`)가 켜지면 xterm이 `SelectionService.disable()`을 호출, 드래그가 로컬 선택을 만들지 못하고 마우스 이벤트를 PTY로 전송. Shift+드래그는 `shouldForceSelection`로 로컬 선택 강제(비-Mac). |
| B. Claude Code TUI 제어 시퀀스 실증 | 실제 Claude Code v2.1.198을 node-pty(ConPTY)로 직접 캡처 | REPL 진입 시 `?1049h`(alt-screen) + `?1000h/?1002h/?1003h/?1006h`(마우스 트래킹)를 **실제로 방출함을 원시 바이트로 확인**. 사용자 `~/.claude/settings.json`의 `"tui": "fullscreen"`이 트리거. |
| C. 프론트엔드 Ctrl+C·복사메뉴 경로 | `frontend/src` 코드 추적 | 두 복사 경로 모두 **xterm 텍스트 선택 존재 여부라는 단일 신호**에 게이트됨. 선택이 비면 둘 다 자동으로 무력화. |
| D. 재현 검증·해결책 벤치마크 | xterm v6 소스 + 외부 이슈 + 업계 사례 | 마우스 트래킹 가설 지지(확신도 ~0.95), 대안 원인 모두 배제. Shift+드래그가 이 프로젝트 소스상 동작해야 함(외부 이슈 주장과 상충 — §7 참고). |

## 4. 종합 진단 (근본 원인)

**근본 원인 (확신도: 매우 높음, 실측 확인됨):**
Claude Code는 fullscreen TUI 모드로 실행될 때 시작 시 **애플리케이션 마우스 트래킹 모드**(DECSET `?1000h`/`?1002h`/`?1003h` + SGR 인코딩 `?1006h`)와 **대체 화면 버퍼**(`?1049h`)를 켠다. xterm.js는 이 마우스 모드를 존중하도록 설계되어 있어, 마우스 모드가 켜지면 **드래그를 로컬 텍스트 선택이 아니라 애플리케이션(PTY)으로 보낼 마우스 이벤트로 처리**한다. 그 결과 드래그해도 xterm의 텍스트 선택이 생성되지 않고, 선택이 없으므로 Ctrl+C 복사와 우클릭 컨텍스트 메뉴 "복사"가 **동시에** 무력화된다.

이것은 Claude Code **고유** 버그가 아니라, "마우스 리포팅을 켜는 모든 TUI"(vim `:set mouse=a`, less, htop, tmux mouse on, lazygit 등) 공통 현상이다. 일반 셸 프롬프트는 마우스 모드를 켜지 않으므로 정상 동작한다.

### 4.1 xterm.js 내부 메커니즘 체인 (Agent A·D, 소스 근거)

vendored 소스는 `frontend/node_modules/@xterm/xterm/src/**` (xterm.js v6).

1. TUI가 `?1000/?1002/?1003h`를 보내면 `InputHandler`가 마우스 프로토콜을 활성화한다.
   - `common/InputHandler.ts:1919-1951` — `case 1000: activeProtocol='VT200'`, `1002:'DRAG'`, `1003:'ANY'`, `1006: activeEncoding='SGR'`. DECRST 해제는 `InputHandler.ts:2160-2164`.
2. `activeProtocol` 변경 시 `onProtocolChange` 발화.
   - `common/services/CoreMouseService.ts:207-217` — `areMouseEventsActive`가 true가 되고 이벤트 발화.
3. 그 핸들러가 **선택 기능을 비활성화**하고 `enable-mouse-events` 클래스를 부여한다.
   - `browser/CoreBrowserTerminal.ts:727-739` — `element.classList.add('enable-mouse-events'); this._selectionService.disable();`
4. `SelectionService.disable()` → `_enabled=false` + 기존 선택 clear.
   - `browser/services/SelectionService.ts:173-183`.
5. 이후 mousedown 시, 비활성 상태에서 강제선택 수정자(Shift)가 없으면 **즉시 return하여 선택을 시작조차 하지 않는다.**
   - `browser/services/SelectionService.ts:462-470` — `if (!this._enabled) { if (!this.shouldForceSelection(event)) { return; } event.stopPropagation(); }`
6. 동시에 mousedown이 마우스 이벤트를 PTY로 전송한다(일반 셸엔 없는 동작).
   - `browser/CoreBrowserTerminal.ts:786` — `if (!areMouseEventsActive || shouldForceSelection(ev)) { return; } sendEvent(ev);`

즉 드래그해도 `SelectionService`의 `selectionStart`가 세팅되지 않아(`SelectionService.ts:608-610`), 텍스트 선택이 만들어지지 않는다. `term.hasSelection()`은 계속 false를 반환한다.

### 4.2 프론트엔드 인과: 왜 두 복사 경로가 **동시에** 막히는가 (Agent C, 코드 근거)

두 복사 경로 모두 동일한 단일 신호(xterm 선택 존재 여부)에 게이트되어 있다.

- **Ctrl+C 복사** — `frontend/src/components/Terminal/TerminalView.tsx:1793-1798`
  ```js
  // Ctrl+C: 텍스트 선택 시 클립보드에 복사, 미선택 시 xterm 기본 처리(SIGINT)
  if (ev.ctrlKey && !ev.altKey && !ev.metaKey && ev.key.toLowerCase() === 'c' && term.hasSelection()) {
    navigator.clipboard.writeText(term.getSelection());
    term.clearSelection();
    return false;
  }
  ```
  `term.hasSelection()`가 false이면 복사 분기를 건너뛰고, 핸들러가 `return false`를 하지 않으므로 Ctrl+C는 xterm 기본 처리로 넘어가 PTY로 `\x03`(SIGINT)이 전송된다. → "Ctrl+C를 눌러도 복사 안 됨"의 직접 원인.

- **컨텍스트 메뉴 "복사"** — `frontend/src/utils/contextMenuBuilder.ts:109-116`
  ```js
  { label: '복사', icon: '⎘', disabled: !hasSelection, onClick: () => { void onCopy(); } }
  ```
  `hasSelection`의 근원은 `TerminalView.tsx:1535` — `!!(xtermRef.current?.hasSelection() || savedRightClickSelRef.current)`. 선택이 비면 `disabled: true`. → "우클릭 복사 비활성화"의 직접 원인.

- **선택 보존 완화 코드의 무력함** — `TerminalView.tsx:2081-2088` (우클릭 mousedown 시 `savedRightClickSelRef.current = term.getSelection()`). 이 완화책은 "선택이 존재하는데 우클릭 타이밍에 지워지는" 케이스 전용이다. 선택이 애초에 비어있는 본 케이스에서는 `term.getSelection()`이 `''`를 반환해 완화 효과가 0이다.

결론: **xterm 선택이 생성되지 않으면, 앱 코드는 그 빈 선택에 대해 두 복사 경로를 동일하게 차단한다.** 앱 코드가 원인을 *추가*하는 것이 아니라, 근본 원인(마우스 모드로 선택 미생성)의 결과를 그대로 반영할 뿐이다.

### 4.3 앱 생성자 옵션이 원인을 추가하지 않음을 확인 (Agent A·D)

`TerminalView.tsx:1674-1704`의 `new Terminal({...})`에는 마우스/선택 관련 옵션(`rightClickSelectsWord`, `macOptionClickForcesSelection`, `screenReaderMode`)이 **하나도 명시되지 않아** 모두 xterm 기본값이다. 즉 억제를 완화하는 옵션도, 악화시키는 옵션도 설정돼 있지 않고, xterm 기본 동작이 그대로 적용된다. 별도 렌더러 addon 없이 `FitAddon`/`SerializeAddon`만 로드하므로(`TerminalView.tsx:1706-1709`) DOM 렌더러 기본값으로 보이나, 선택 억제 로직은 렌더러와 독립적이라 결론에 영향이 없다.

## 5. 실증 증거 (Agent B — 가장 강한 증거)

이 저장소가 사용하는 것과 동일한 node-pty(ConPTY)로 실제 설치본 Claude Code **v2.1.198**을 띄워 원시 바이트를 캡처했다.

- **REPL(fullscreen) 진입 시 방출 시퀀스** (관측됨):
  ```
  ESC[?9001h   win32-input-mode (ConPTY 부수 시퀀스)
  ESC[?1004h   focus reporting
  ESC[?2004h   bracketed paste
  ESC[?2031h   color-scheme change notification
  ESC[?1049h   ← 대체 화면 버퍼(alt-screen)
  ESC[2J       화면 클리어
  ESC[?1000h   ← 마우스 트래킹: 클릭
  ESC[?1002h   ← 마우스 트래킹: 버튼+드래그
  ESC[?1003h   ← 마우스 트래킹: 모든 이동
  ESC[?1006h   ← 마우스 좌표 SGR 확장 인코딩
  ```
  가설로 제시한 마우스 트래킹 시퀀스가 **전부 실제로 켜짐**을 확인했다(`?1005h` utf8, `?1015h` urxvt 인코딩은 미사용).

- **"왜 셸에선 정상인가"의 직접 증거**: 트러스트 프롬프트 단계(REPL 미진입)에서는 `?1004h/?2004h/?9001h/?25l`만 방출되고 **마우스/alt-screen 시퀀스가 전혀 없었다.** 마우스 모드는 REPL(fullscreen) 진입 시에만 켜진다.

- **트리거 설정 확인**: `~/.claude/settings.json`의 `"tui": "fullscreen"`. 사용자가 보고한 "최근부터" 증상 시작과, Claude Code가 최근 버전(대략 v2.1.150 전후)부터 마우스 트래킹을 기본 활성화한 것이 정합적이다.

- **외부 근거(동일 시나리오 확인)**:
  - [anthropics/claude-code #74320 — Fullscreen TUI mode breaks terminal text copy in browser-based terminals (code-server/xterm.js) due to mouse-tracking capture](https://github.com/anthropics/claude-code/issues/74320)
  - [#61936 — Mouse tracking breaks terminal right-click context menu and text selection](https://github.com/anthropics/claude-code/issues/61936)
  - [#62699 — Text cannot be copied from Claude Code's output via Ctrl+Shift+C or right-click](https://github.com/anthropics/claude-code/issues/62699)
  - [#70857 — Fullscreen(alt-screen) mode breaks terminal copy](https://github.com/anthropics/claude-code/issues/70857)
  - [Configure your terminal for Claude Code (공식 문서)](https://code.claude.com/docs/en/terminal-config)

## 6. 재현 조건

- Claude Code를 `"tui": "fullscreen"` 설정(또는 fullscreen이 기본인 최근 버전)으로 웹 터미널에서 실행 → REPL 진입.
- 이때 xterm은 마우스 트래킹 모드가 되어, 일반 드래그로는 텍스트 선택이 생기지 않는다.
- Ctrl+C → SIGINT로 전달(복사 안 됨), 우클릭 → "복사" 비활성.
- 반대로 `claude`를 종료하거나 인라인 모드/일반 셸로 돌아오면(마우스 모드 DECRST) 즉시 정상 복원.

## 7. 우회·해결책 방향 (구현 전 사용자 결정 필요 — 이번 조사에서는 미적용)

Agent A·D가 소스·업계 사례로 정리한 옵션이다. 코드 수정은 수행하지 않았다.

1. **Shift+드래그 (xterm 내장, 코드 0줄, 저비용 즉효)** — 비-Mac에서 `shouldForceSelection`이 `event.shiftKey`를 반환하므로(`SelectionService.ts:437-443`), Shift를 누른 채 드래그하면 마우스 모드에서도 로컬 선택이 강제되고, 이어서 Ctrl+C가 `hasSelection()===true`로 복사된다. **트레이드오프**: 발견성(discoverability) 문제 — 사용자 안내 필요. Mac은 Shift가 아니라 Alt+click + `macOptionClickForcesSelection` 옵션이라 크로스플랫폼 안내가 달라진다(현재 이 옵션 미설정).
2. **마우스 모드 감지 + "선택 모드" 토글** — xterm v6 공개 API `terminal.modes.mouseTrackingMode`로 감지 가능. 마우스 모드일 때 UI 배지/토글을 노출하고, 토글 ON이면 마우스 이벤트 PTY 전송을 억제해 선택을 허용. **트레이드오프**: 공개 `onModeChange` 이벤트가 없어 폴링/write 후 조회 필요. 토글 켠 동안 Claude Code 자체 마우스 UI(스크롤/클릭)가 깨진다. → **이 옵션의 구현 설계 심층 연구는 별도 문서 `docs/research/2026-07-11.terminal-mouse-mode-selection-toggle-design-research.md` 참고** (Shift+드래그 불안정성의 진짜 원인인 "마우스 모드 재천명 clear" 규명 포함).
3. **키보드 전용 "화면 복사" 커맨드** — 이미 로드된 `SerializeAddon`으로 뷰포트/버퍼 복사 버튼·단축키 제공(마우스 모드와 무관). Claude issue #74320이 명시적으로 권고. **트레이드오프**: 자유 영역 선택이 아니라 화면/행 단위.
4. **우클릭 컨텍스트 메뉴 강제 복사** — 이미 우클릭 선택 저장 코드가 반쯤 존재. 단 마우스 모드에선 선택 자체가 없어 #1/#2와 결합해야 실효.
5. **Claude Code 측 완화(설정)** — `~/.claude/settings.json`에서 `"tui": "fullscreen"` 제거(인라인 렌더링 → 마우스/alt-screen 미방출), 또는 `CLAUDE_CODE_DISABLE_MOUSE_CLICKS=1` 환경변수. **트레이드오프**: BuilderGate 코드 변경이 아니라 사용자 환경 설정 변경.

**업계 사례**: VS Code 통합 터미널은 xterm 기본(Shift)을 오버라이드해 Alt(Win/Linux)/Option+`macOptionClickForcesSelection`(Mac)로 강제 선택한다. 순정 xterm.js/ttyd/wetty는 Shift+드래그 기본값을 사용한다.

**상충점 (정직한 플래그)**: Claude issue #74320은 "'hold Shift to bypass' escape hatch가 xterm.js에 미구현"이라고 서술하나, 이 프로젝트가 vendored한 xterm.js v6.0.0 소스에는 Shift 강제선택이 명백히 구현되어 있다(`SelectionService.ts:442`, `CoreBrowserTerminal.ts:786`). 따라서 **이 프로젝트에서는 Shift+드래그가 이론상 동작해야 한다.** 이슈 작성자가 이를 몰랐거나 code-server 특유의 간섭을 겪었을 가능성이 있다. → **실서버 기동 후 `claude` 실행 중 Shift+드래그로 선택되는지 최종 경험적 검증을 권장**(§8 참고).

## 8. 미검증·불확실 지점

1. **Shift+드래그의 브라우저 실측 미완** — 소스상 성립하나(비-Mac), dev 서버 미기동으로 실제 브라우저에서 Claude Code 실행 중 Shift+드래그 → 선택 → Ctrl+C 복사가 되는지는 아직 경험적으로 확인하지 못했다. 이 한 가지만 확인하면 §7-1이 사실상 무료 해결책으로 확정된다.
2. **인라인(비-fullscreen) REPL의 마우스 방출 여부** — 캡처는 fullscreen 세트만 확정했다. 일부 외부 소스는 최근 버전이 인라인에서도 마우스 클릭을 후킹한다고 하나 직접 캡처로 확인하지 못했다. 다만 본 사용자는 `"tui": "fullscreen"`이라 진단에는 영향이 없다.
3. **Claude Code 버전 의존성** — 마우스 트래킹 기본 활성 도입 시점(대략 v2.1.150 전후)과 향후 동작 변경 가능성. 현재 캡처는 v2.1.198 기준.

## 9. SRS 추적 메모

- 이 증상을 직접 다루는 기존 Requirement는 없다(`docs/spec` 전체에 copy/clipboard/mouse-tracking 관련 REQ 부재).
- 본 문서는 **순수 조사 산출물이며 SRS 변경이나 코드 수정을 수행하지 않았다.**
- 실제 fix를 진행할 경우, 프론트엔드 selection/copy UX는 신규 REQ에 해당하므로 신규 REQ 등록(`/kiwi-srs`) 또는 `/kiwi-hot-fix` 경유 사후 동기화(`/kiwi-srs-sync`) 여부를 사용자가 먼저 결정해야 한다. §7의 어떤 옵션을 택할지도 사용자 결정 사항이다.
