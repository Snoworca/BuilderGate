## Summary

Wave6의 책임은 Wave0-Wave5에서 정의되고 서버 쪽으로 검증된 resource limit guardrail을 사용자가 브라우저 Settings에서 조정할 수 있게 만드는 연결층이다. 구현 범위는 Settings snapshot, Settings PATCH, `config.json5` persistence, apply summary, validation error surfacing, browser-safe `/api/runtime-config` projection 검증까지다.

Wave6는 Wave7의 runtime behavior를 구현하지 않는다. 즉 hidden terminal eviction, hidden output write 중단, visible output scheduler, browser WebSocket `bufferedAmount` backpressure 소비자 변경은 Wave7로 넘긴다.

핵심 결론:

- 서버 타입, schema, Settings PATCH, config persistence 경로는 이미 대부분 존재한다.
- 프론트 타입과 `settingsApi`도 준비되어 있지만 `SettingsPage`는 아직 resource cards를 렌더하지 않고 `buildPatch()`도 `resourceLimits`와 `stabilityModes`를 보내지 않는다.
- `resourceLimits.telemetry.*`는 capability unavailable이므로 Wave6 UI에서 숨긴다.
- `/api/runtime-config`는 browser-needed subset만 공개한다. `headless`, server-only `ws`, `telemetry`는 public projection에 넣지 않는다.
- `headless.writeLagWarnMs`와 `headless.writeBatchMaxBytes`는 노출 전에 실제 runtime 소비 여부를 확인해야 한다. 소비자가 없다면 unavailable/reserved로 남긴다.

## Scope

Wave6 in scope:

- Settings snapshot에 Wave6 대상 resource limit capability가 정확히 보이게 한다.
- Settings UI에서 selected resource limits를 기존 `Card`, `Field`, scope badge, validation banner, Save flow로 편집한다.
- Settings PATCH가 changed leaf만 nested partial로 전송한다.
- 서버가 valid resource values를 `config.json5`에 persist한다.
- invalid values, unknown keys, unsupported enum, inconsistent threshold relationship은 observable error로 거절한다.
- apply failure는 runtime store와 runtime consumers를 rollback한다.
- `/api/runtime-config`는 browser-public resource subset만 저장값과 일치하게 반환한다.
- reload 후 frontend runtime config getters가 저장된 public values를 읽을 수 있음을 검증한다.

Wave6 non-goals:

- process tree termination, shutdown evidence, daemon stop policy 변경
- broad kill path 추가
- App/Terminal/WebSocket runtime 소비자 동작 변경
- hidden terminal runtime eviction
- hidden output xterm write 중단
- visible output scheduler 구현
- browser WebSocket input send backpressure 구현
- split WebSocket migration/default 전환
- telemetry resource limit UI 노출
- Settings visual redesign

## Acceptance Criteria Draft

- AC-1: Given a legacy or partial `config.json5`, when Settings reads selected `resourceLimits`, then server defaults are applied through the runtime config store without exposing secrets or silently coercing invalid values.
- AC-2: Given a valid Settings PATCH for selected resource limits, when save succeeds, then `config.json5` remains the persistence source of truth and the response reports truthful `changedKeys` and `applySummary`.
- AC-3: Given invalid limits, null sections, unsupported enum values, unknown keys, or inconsistent threshold relationships, when Settings validation runs, then the request fails observably and no partial runtime or persistence update remains.
- AC-4: Given runtime apply fails after dry-run persistence succeeds, when `savePatch()` handles the failure, then runtime store and affected runtime consumers are rolled back to previous values and config file content is not committed.
- AC-5: Given `/api/runtime-config` is requested after a successful save or after reload, when browser-needed resource limits are present, then only public client-consumed fields are exposed and frontend typed getters preserve saved values or safe defaults.
- AC-6: Given headless and server WebSocket limits are changed, when new sessions or WebSocket routing use them, then `new_sessions` and `immediate` apply scopes are accurate and bounded queue/backpressure invariants remain intact.
- AC-7: Given the Settings page renders Wave6 cards, when a user edits values, then existing `Card`, `Field`, validation banner, scope badge, and Save button flow are reused with no silent UI clamping.
- AC-8: Given Wave6 implementation is complete, when focused server/frontend checks run, then resource-limit persistence, relationship rejection, rollback, runtime-config exposure, UI patch body, error rendering, and apply summary rendering are covered.

## Risks And Rollback

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Wave6 changes Wave7 runtime behavior | Cause separation fails, frontend dirty work conflicts | Limit Wave6 to Settings, persistence, projection, and tests |
| Server-only values leak through public runtime config | Public contract pollution | Keep public projection to browser-needed subset |
| Invalid values are clamped in UI | User sees false saved value | Validate form shape only; let server reject relationships |
| Runtime apply fails after dry-run persist | Runtime and config drift | Test rollback and commit file only after successful apply |
| Reserved telemetry appears editable | User can edit no-op setting | Hide unavailable capability fields |
| Legacy config insert breaks format | Server startup failure | Require JSON5 parse and schema validation in tests |

Rollback triggers:

- Settings save breaks valid existing config load.
- `/api/runtime-config` exposes server-only fields or fails frontend runtime config parsing.
- `config.json5` rendering becomes invalid or loses unrelated settings.
- apply summary reports an incorrect scope.
- resource UI sends wrong nested PATCH paths.

Rollback procedure:

1. Hide resource cards behind conditional render or feature flag.
2. Keep server defaults and existing `resourceLimits` schema.
3. Keep `/api/runtime-config` public subset conservative.
4. Revert only Wave6 UI/helper/test changes if needed; avoid touching Wave7 dirty files.
5. Re-run Settings PATCH, runtime-config, and legacy config focused tests.

## Wave7 Handoff

Wave7 should consume the public resource values after Wave6 validates persistence and reload behavior:

- `clientWs.*` for browser WebSocket input send backpressure
- `terminal.inputQueue*` and `transportOutbox*` for bounded input/outbox
- `terminal.visible*` for visible output scheduler
- `terminal.hiddenOutput*` for hidden output policy
- `snapshots.*` for snapshot/localStorage budget and tombstone TTL
- `workspaceRuntime.*` for runtime residency cap and hidden TTL
- `frontendRuntimeResidency` only when Wave7 wires App/Terminal behavior

Settings changes should alter actual runtime behavior after reload in Wave7, not as a Wave6 completion claim.
