# BuilderGate Empty Fallback Snapshot Placeholder Research

Date: 2026-07-01
Target: `0.5.5-buildergate-stability`
Scope: `BGSTAB`

## Summary

The frequent terminal line below is not shell output from PowerShell, bash, Codex, or the PTY process.

```text
[BuilderGate] Fallback snapshot unavailable. Waiting for new output...
```

It is a frontend placeholder written into xterm by `TerminalContainer` when the server sends a `screen-snapshot` message with `mode: "fallback"` and empty `data`, and local snapshot restore fails.

The issue is real and has two layers:

1. Frontend symptom: the placeholder is written into the terminal buffer repeatedly, so it looks like real terminal output.
2. Server/protocol cause: headless snapshot recovery can remain degraded or truncated-empty, and replay refresh/repair/subscribe paths can resend empty fallback snapshots with new tokens.

The safe fix is not to treat the placeholder as successful recovery. The client should acknowledge the replay token, keep recovery state pending, avoid repeated destructive terminal writes, and make the degraded state observable. Server replay timeout and degraded fallback paths also need output-preservation checks so queued output is not lost if ACK is delayed.

## Code Path

The runtime path is:

1. Server calls `SessionManager.getScreenSnapshot(sessionId)`.
2. If the headless terminal is degraded or unavailable, `SessionManager.createDegradedSnapshot()` returns an empty snapshot.
3. `WsRouter.sendSnapshotReplay()` sends a `screen-snapshot` message with `mode: "fallback"` and `data: ""`.
4. `TerminalContainer.handleScreenSnapshot()` sees fallback + empty data.
5. The client first tries `terminalRef.current.restoreSnapshot()`.
6. If local restore fails, it writes `FALLBACK_EMPTY_MESSAGE` through `replaceWithSnapshot()`.
7. The client sends `screen-snapshot:ready` after the placeholder path completes.

This is why the message appears inside the terminal. It is not emitted by the shell process.

## Why It Can Appear Often

The placeholder can repeat when any of these happen:

- Headless terminal stays degraded after `create`, `write`, `resize`, `serialize`, or queue overflow degradation.
- Oversized snapshot serialization returns a truncated empty snapshot.
- Subscribe/resubscribe requests send another snapshot.
- Repair replay sends another snapshot.
- Resize replay refresh rotates the replay token and sends another fallback snapshot while replay is pending.
- Local snapshot restore fails because there is no snapshot or geometry does not match.

The frontend duplicate guard compares snapshot content and sequence. Repeated empty fallback snapshots can arrive with different `seq` or `replayToken`, so the current guard may not suppress them.

## Risks

### User-visible noise

The placeholder is inserted into the terminal buffer, so it looks like command output. Repeating it pollutes scrollback and makes the terminal feel broken.

### Destructive buffer mutation

`replaceWithSnapshot()` resets xterm before writing the placeholder. If there is still useful local content, repeated placeholder application can make the visible state worse.

### Recovery state ambiguity

Current tests correctly prevent the placeholder from finishing visible or hidden recovery. That part should remain. The client must not mark recovery as successful just because it wrote a placeholder.

### Replay output loss on timeout

While replay is pending, server output is queued behind the snapshot ACK. ACK flushes that queued output. Timeout currently deletes pending replay state and sends `session:ready`, but the reviewed code path does not flush `pending.queuedOutput`. If placeholder application is delayed or ACK fails, output may be dropped.

## Recommended Fix Strategy

### 1. Frontend immediate mitigation

Treat empty fallback snapshots as non-destructive recovery states.

- If local snapshot restore succeeds, keep current behavior and ACK.
- If local restore fails, do not repeatedly call `replaceWithSnapshot(FALLBACK_EMPTY_MESSAGE)`.
- Prefer not writing the placeholder into xterm at all. If a user-visible signal is needed, show it through debug/status UI rather than terminal scrollback.
- If compatibility requires keeping the message, write it at most once per session/recovery generation.
- Always ACK the current replay token after the no-op/suppressed empty fallback path unless the send itself fails.
- Do not finish visible recovery, clear hidden-output recovery, or release queued input solely because the empty fallback placeholder/no-op path ran.
- Reset suppression only after real output, authoritative snapshot, non-empty fallback snapshot, local restore success, screen repair success, or session generation change.

### 2. Server/protocol reliability fix

Make fallback replay output preservation explicit.

- On replay ACK timeout, do not silently drop `pending.queuedOutput`.
- Either flush queued output in order if safe, or move it into a bounded replay tail and trigger a fresh snapshot/repair path.
- Suppress or rate-limit repeated identical empty fallback refreshes for the same client/session/sequence while no new server state exists.
- Consider sending bounded `degradedReplayBuffer` as fallback data when it is safe and not duplicative, or add an explicit degraded reason/payload field so the client can distinguish "no data exists" from "data withheld".
- Keep recovery transaction ordering: snapshot/repair/replay-tail/session-ready must not be reordered by safe-send or coalescing.

### 3. Observability

Add bounded diagnostics without durable raw terminal payloads:

- frontend: `fallback_empty_received`, `fallback_empty_local_restore_used`, `fallback_empty_placeholder_applied`, `fallback_empty_placeholder_suppressed`, `fallback_empty_ack_failed`
- server: degraded phase counts by `create`, `write`, `resize`, `serialize`, `queue-overflow`; empty fallback sent count; replay timeout count with queued output bytes; replay refresh count; degraded replay buffer bytes
- protocol: replay token churn count and repeated empty fallback suppression count

### 4. Tests

Add deterministic unit tests before E2E.

- Frontend helper/fake-terminal test: two empty fallback snapshots with failed local restore do not write the placeholder twice, but both are ACKed.
- Frontend transition test: authoritative snapshot, non-empty fallback, local restore success, real output, or repair success resets suppression.
- Frontend recovery contract: placeholder/no-op still does not finish visible recovery or clear hidden-output recovery.
- Server test: replay ACK timeout preserves or reroutes `pending.queuedOutput`.
- Server test: repeated fallback refresh does not churn tokens forever for identical empty fallback state.
- Server test: degraded snapshot diagnostics include phase/reason and degraded replay buffer size.
- E2E should be added only after unit coverage. Prefer debug-event assertions over timing-sensitive terminal text polling.

## SRS Update Decision

Three requirements are needed:

- `FR-BGSTAB-018`: Empty fallback snapshot convergence and placeholder suppression.
- `REL-BGSTAB-004`: Fallback replay output preservation and degraded snapshot convergence.
- `OBS-BGSTAB-002`: Empty fallback snapshot diagnostics and regression evidence.

These complement existing requirements:

- `FR-BGSTAB-017`: recovery write gate and queued input release barrier.
- `REL-BGSTAB-003`: byte-aware replay tail and screen repair overflow recovery.
- `OBS-BGSTAB-001`: ConPTY input-lag recovery telemetry and soak evidence.

## Implementation Order

1. Add frontend empty fallback decision helper and tests.
2. Replace repeated placeholder write with no-op or once-per-generation suppression.
3. Preserve ACK behavior and recovery-pending semantics.
4. Add server replay timeout queued-output preservation test.
5. Fix timeout output preservation and fallback refresh churn.
6. Add observability counters/debug events.
7. Run targeted frontend unit tests, server test-runner cases, SpecKiwi validation, and one subagent review.
