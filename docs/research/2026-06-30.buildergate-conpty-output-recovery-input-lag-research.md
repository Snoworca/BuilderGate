# BuilderGate ConPTY Output Recovery And Input Lag Research

Date: 2026-06-30

Scope: ConPTY mode looks visually correct, but Codex-like TUI sessions recover slowly and can become hard to type into while the agent repeatedly refreshes the screen. This document consolidates 10 read-only sub-agent investigations plus local static analysis. No code was changed as part of this research.

## Executive Summary

ConPTY itself is not the only problem. The strongest explanation is a layered bottleneck:

1. ConPTY and AI TUIs produce high-frequency redraw output. Codex-style screens emit cursor moves, erase-line sequences, title/progress updates, local echo, ticker/status redraws, and sometimes large resume output bursts.
2. The project already has many protective pieces: bounded headless output queue, server safe-send/backpressure path, viewport-only snapshots, hidden output skip/recovery, frontend visible output scheduler, browser input backpressure, and AI TUI idle heuristics.
3. Some protection is not active on the default path. The local config uses ConPTY, while schema/env defaults still keep `stabilityModes.wsSendMode = "direct"`, `session.processCleanup.mode = "observe"`, and input reliability in observe-style behavior unless explicitly configured. In `direct` mode, server WebSocket output coalescing and backpressure queueing do not materially protect slow clients.
4. The current recovery path is not full-log replay in the normal case. Server and local snapshots are viewport-only (`scrollback: 0`). The slow part is more likely large snapshot apply, ACK-pending tail flush, headless drain delay, Codex resume output, or output/control sharing one transport path.
5. Input reaches `SessionManager.writeInput()` and `pty.write()` directly once it gets through the frontend/server transport gates. The perceived "cannot type" state is caused before that point: browser main-thread rendering, restore/recovery barriers, unified WebSocket backpressure, replay ACK waiting, and server output/control FIFO behavior.

The effective fix is not to disable ConPTY by default. Keep ConPTY for interactive quality, but make output delivery bounded, chunked, measurable, and configurable. Input/control must be given priority over output. Recovery must apply snapshots and tails through the same write scheduler discipline used by live output.

## Current Relevant Implementation

### PTY and backend selection

- `server/src/services/SessionManager.ts` creates PTY sessions with `node-pty.spawn(...)` and passes `useConpty` on Windows.
- `server/src/utils/ptyPlatformPolicy.ts` makes Windows default to ConPTY and non-Windows normalize ConPTY-specific settings away.
- `server/config.json5` currently has `useConpty: true` and `windowsPowerShellBackend: "conpty"`.
- PowerShell backend can be `inherit`, `conpty`, or `winpty`, but winpty is not a guaranteed fallback because the server probes host support.

Conclusion: backend selection is deliberate. Do not silently switch Codex sessions to pipe/winpty based only on command name.

### Output path

Current live output path:

```text
ConPTY / node-pty output
  -> SessionManager pty.onData
  -> headlessOutputQueue
  -> headless xterm write commit
  -> WsRouter.routeSessionOutput
  -> server WebSocket transport send
  -> frontend WebSocketContext
  -> TerminalContainer
  -> TerminalView
  -> terminalOutputScheduler
  -> xterm.write
```

Important existing safeguards:

- `SessionManager` uses bounded headless output queue and degrades instead of silently growing without limit.
- `WsRouter` has safe-send/backpressure fields: buffered amount, queued output/control bytes, slow-client close counters, send-error counters, and output coalescing count.
- `TerminalView` live output goes through `createTerminalOutputScheduler()`, which waits for `xterm.write` callback and flushes by budget.
- `TerminalContainer` can skip hidden output and recover with snapshot/repair instead of continuously writing hidden output.

Important gaps:

- Schema default `wsSendMode` is `direct`. In that mode, `WsRouter` sends immediately and does not use the queue path that performs output coalescing.
- The server transport state has output/control byte counters, but the actual queue is a single FIFO. Control messages can still sit behind output already queued.
- Snapshot restore and screen repair can bypass the normal visible output scheduler and push large payloads into `xterm.write` directly.

### Recovery path

Normal recovery is not full history replay.

- Server headless snapshot serialization uses viewport-only semantics.
- Client local snapshot also uses viewport-only semantics.
- On subscribe/reconnect, server sends `screen-snapshot`, waits for `screen-snapshot:ready`, then flushes ACK-pending tail output and queued inputs.

Main recovery bottlenecks:

- Large authoritative snapshot apply can still block the browser even if it is viewport-only.
- ACK-pending tail output can be up to the replay queue limit and may flush as a large chunk.
- Codex `resume` may generate its own high-output replay, which is not the same as terminal replay.
- If server WS mode is `direct`, output flood can delay ACK/ready/control traffic.

### Input path

Current input path:

```text
xterm input
  -> TerminalView.submitCapturedInput
  -> TerminalContainer.handleInput
  -> TerminalInputSequencer
  -> WebSocketContext.send({ type: "input" })
  -> WsRouter.handleInput
  -> SessionManager.writeInput
  -> pty.write
```

Once input reaches `SessionManager.writeInput`, it is not placed behind headless output queue. The delay happens before that:

- browser main thread is busy rendering output or snapshot/repair;
- `restorePending`, visible-output recovery, or session-ready gates are active;
- browser `bufferedAmount` causes input backpressure;
- unified `/ws` means output/control/input share one transport;
- replay pending mode queues or rejects input until ACK/timeout;
- server safe-send enforce mode still drains one FIFO unless control and output queues are physically separated.

## Root Cause Assessment

### What is likely true

- ConPTY creates a higher fidelity stream than legacy/simple output. That includes a lot of redraw noise.
- Codex-like TUI output is not line-oriented log output. It is a stream of screen mutations.
- Browser xterm rendering is still on the main thread. No amount of server queueing can fully hide oversized `xterm.write` work.
- Current defaults and reserved settings mean the implemented guardrails may not apply to the user-visible path yet.

### What is probably not the primary cause

- Not a simple duplicate input bug. Existing xterm input path and debug events show input is generally one path.
- Not normal full scrollback replay. Current normal snapshot is viewport-only.
- Not solved by a browser worker alone. A worker can own WebSocket/JSON parsing/batching, but actual xterm rendering still happens on the main thread.

## Recommended Fix Strategy

### 1. Make server output protection active and priority-aware

Recommended changes:

- Use `safe-send-enforce` for ConPTY high-output validation and eventually as the default after soak evidence.
- Split server transport queues physically into `controlQueue` and `outputQueue`.
- Flush control first for lifecycle, ACK, input rejection, session ready, ping/pong, workspace/tab metadata, and status messages.
- Keep terminal payload ordering inside the terminal payload lane. Do not reorder snapshot/output/repair in ways that break replay correctness.
- Apply high/hard water decisions to projected size: `bufferedAmount + message.byteLength`, not only current `bufferedAmount`.
- Replace `Array.shift()` draining with head-index/deque behavior for long queues.
- Convert replay pending tail from string slicing to byte-aware seq ring metadata.
- On screen repair overflow, abort/refresh instead of concatenating a large payload and sending it anyway.

### 2. Route snapshot and repair through a recovery write gate

Live output already uses a scheduler. Snapshot and repair should use the same discipline, but not necessarily the same queue instance. A safer design is an independent recovery write gate or recovery lane so a stale/overflow state in the live output scheduler does not corrupt recovery ordering.

- authoritative snapshot apply should be chunked or scheduled through the recovery lane;
- ACK-pending tail flush should be chunked while preserving snapshot-before-tail ordering;
- screen repair `ansiPatch` should be serialized through a repair/write queue;
- `screenRepairInFlightRef` should include a request token/nonce, not only session id;
- visible output recovery should not hold input gates longer than necessary.

The goal is not to make recovery visually stale. The goal is to avoid one giant `term.write` from monopolizing the main thread.

### 3. Keep input/control ahead of output

Recommended frontend/server behavior:

- input, resize, subscribe, unsubscribe, ACK, ready, ping/pong, close/reject are control;
- output and large snapshot/repair payloads are terminal payload;
- resize should be latest-value-wins and retryable;
- replay ACK should preserve recovery ordering: snapshot and its ACK-pending terminal tail must complete before `session:ready` and queued replay input are released;
- queued replay input should then reach `SessionManager.writeInput()` before unrelated live output backlog is allowed to starve it;
- frontend should centralize control sends through one observable path.

Do not interpret control priority as permission to reorder an active snapshot/repair replay. Terminal payload ordering wins inside the recovery transaction. Control priority applies to unrelated live output backlog and lifecycle/control messages outside that transaction.

Long-term options:

- Web Worker owns WebSocket, JSON parsing, output batching, and high-priority control postMessage.
- Split socket uses separate control/output channels. The repo already has split URL/lifecycle utilities, but the active path is still unified `/ws`.

### 4. Expose output coalescing interval in Settings

User requirement: the coalescing/throttling unit should be configurable from Settings.

Current state:

- `resourceLimits.ws.outputCoalesceWindowMs` already exists in schema with `1..1000ms`, default `16`.
- It is stored by `ConfigFileRepository`.
- It is accepted by `SettingsService` schema.
- It is applied to `WsRouter` runtime config.
- It is currently reserved/unavailable in `RuntimeConfigStore` capability.
- It is not listed in `frontend/src/components/Settings/settingsDraftHelpers.ts`.
- It only has practical effect when output enters the safe-send queue path, mainly `safe-send-enforce`.

Recommended Settings change:

- Remove `resourceLimits.ws.outputCoalesceWindowMs` from the reserved Settings set.
- Add it to the "Server Backpressure" resource group as `Output coalesce window (ms)`.
- Keep constraints `1..1000ms`.
- Show apply scope as immediate.
- Document that the value applies to queued server output in safe-send mode.
- Prefer exposing it together with a controlled `stabilityModes.wsSendMode` rollout field, otherwise users may change it while `direct` mode makes it ineffective.

Suggested presets:

| Preset | Value | Use case |
|---|---:|---|
| Low latency | `16ms` | interactive typing and normal terminal use |
| Balanced | `33ms` | noisy TUI redraw, modest batching |
| High compression | `50ms` | severe redraw flood where freshness is less important |
| Custom | `1..1000ms` | debugging and host-specific tuning |

### 5. Keep ConPTY as default, use explicit fallback only

Do not silently switch Codex to another backend. Recommended policy:

- Windows interactive terminals: ConPTY remains default.
- PowerShell backend override remains user-visible and applies to new sessions.
- winpty fallback is allowed only when supported and explicitly selected.
- Pipe mode, if added, should be a separate non-interactive command runner mode, not an implicit TUI backend.
- Headless is not a backend replacement. It is a mirror/recovery layer and should degrade observably on failure.

## Test And Soak Plan

### Server unit/integration

- `WsRouter` safe-send enforce should flush control ahead of output.
- `WsRouter` should test `direct`, `safe-send-observe`, and `safe-send-enforce` separately.
- Single output payload above hard limit should not bypass protection.
- Output coalescing should merge only adjacent compatible output for the same session.
- Replay pending tail truncation should be explicit and observable.
- Screen repair overflow should trigger refresh/recovery, not a giant direct flush.
- Settings PATCH should allow `resourceLimits.ws.outputCoalesceWindowMs` after reserved removal and reject `0`/`1001`.
- Runtime config update should immediately change `WsRouter` coalescing behavior.

### Frontend unit

- Snapshot apply and repair should go through a write gate or dedicated serialized repair queue.
- Hidden output should not call xterm write while hidden.
- Visible output scheduler should honor queue and chunk budgets.
- Browser input backpressure should not log `ws_input_sent` unless send actually succeeded.
- Settings page should render and persist output coalescing interval.

### E2E

Run against `https://localhost:2222` per project rule.

Scenarios:

- high-output visible ConPTY session while typing in active terminal;
- hidden high-output tab while typing in another visible tab;
- Codex-like resume output followed by reload/resubscribe;
- workspace/tab switch during noisy TUI redraw;
- resize flood while output is pending;
- Settings change for coalescing interval and safe-send mode.

Suggested pass criteria:

- input echo p95 stays under a defined threshold for the host;
- recovery-to-input-ready p95 stays under a defined threshold;
- no `input_write_failed` in normal path;
- no unexplained `server_input_rejected`;
- no sentinel output loss, reordering, or duplication after coalescing;
- output and WS queued bytes return to baseline after soak;
- verified descendants are zero after close/restart/shutdown tests.

### Soak

Use `start.bat --port 2222` and `stop.bat`. Do not kill all `node.exe`.

Scenarios:

- 30-100 PowerShell ConPTY create/delete loops;
- restart loop on same tab;
- high-output server soak;
- hidden runtime soak;
- input latency soak with ASCII, Hangul/IME, Backspace, Enter;
- shutdown evidence soak.

Strict soak should fail on degraded/skipped cleanup where verified descendants remain. If WSL or detached descendants are skipped because identity is not safely verified, report that explicitly rather than claiming complete cleanup.

## Implementation Roadmap

### Phase A: Evidence first

- Add missing backend/input/frontend runtime metrics.
- Add a repeatable ConPTY latency and high-output soak harness.
- Record baseline with current settings.

Rollback: no runtime behavior change.

### Phase B: Server transport priority

- Make safe-send enforce path priority-aware.
- Split control/output queues.
- Use projected buffered size.
- Replace long queue `shift()` drain.
- Add tests for control starvation and coalescing.

Rollback: set `stabilityModes.wsSendMode` back to `direct` or `safe-send-observe`.

### Phase C: Frontend write-gate hardening

- Route snapshot, repair, and ACK-tail flush through a chunked/serialized recovery write gate.
- Add repair token/nonce matching.
- Add latency debug measurements around snapshot apply and repair apply.

Rollback: restore legacy direct snapshot/repair write path behind a setting.

### Phase D: Settings exposure

- Unreserve `resourceLimits.ws.outputCoalesceWindowMs`.
- Add Settings UI field.
- Prefer adding/controlling `stabilityModes.wsSendMode` at the same time or clearly marking the dependency.
- Add PATCH, persistence, runtime, and frontend Settings tests.

Rollback: reserve the field again and keep config-file-only tuning.

### Phase E: Default rollout

- Only after soak evidence, consider changing product defaults:
  - `wsSendMode` from `direct` to `safe-send-enforce`;
  - `inputReliabilityMode` from observe-style behavior to queue/reject behavior where appropriate;
  - frontend runtime residency bounded mode if not already the product default.

Rollback: setting switches, not git reset or process-name kill.

## Decision

Yes, the ConPTY problems are solvable without giving up ConPTY visual quality.

The highest-confidence fix set is:

1. activate and harden server safe-send with control priority;
2. chunk/schedule snapshot and repair writes;
3. keep input/control independent from output as much as the unified socket allows;
4. expose output coalescing interval in Settings;
5. measure ConPTY-specific input/recovery latency before changing defaults.

The user-requested configurable coalescing unit is a good fit. The repo already has the backend setting; the missing work is reserved-capability removal, Settings UI exposure, tests, and making sure the setting is meaningful by using the safe-send queue path.
