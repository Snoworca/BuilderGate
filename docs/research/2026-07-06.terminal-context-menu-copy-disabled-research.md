# 터미널 컨텍스트 메뉴 "복사" 버튼 고빈도 비활성화 조사

Date: 2026-07-06
Target: `0.5.5-buildergate-stability` (관련: `0.5.3step1` — Terminal context menu 동작, 현재 inactive/planned)
Scope: `BGSTAB` (context menu 자체는 전용 REQ 없음 — §5 SRS 추적 참고)

## 1. 증상

세션 터미널에서 텍스트를 드래그로 선택한 뒤 우클릭하면 나오는 커스텀 컨텍스트 메뉴에서, 방금 선택했음에도 "복사" 항목이 **높은 빈도로 비활성화(disabled)** 상태로 표시된다. 100% 결정적 재현은 아니며, 재현 빈도가 "높다"는 것은 어떤 조건과 상관관계가 있는 레이스/타이밍 버그임을 시사한다.

## 2. 조사 방법

서로 다른 가설을 독립적으로 검증하기 위해 3개의 서브에이전트를 병렬로 투입했다(교차 오염 방지를 위해 서로의 결론을 공유하지 않고 각자 코드베이스를 직접 조사).

| Agent | 초점 | 결론 요약 |
|---|---|---|
| A. 이벤트 타이밍 | 우클릭 → 메뉴 빌드 사이의 비동기 경로 | `hasSelection`은 우클릭 순간이 아니라 preset API 비동기 로드 + `requestAnimationFrame` 이후 "라이브 재평가"된다. 추가로 다른 탭/pane에서 포커스 없이 선택 후 우클릭 시, `TerminalRuntimeLayer`의 강제 `focus()` 호출이 mousedown보다 먼저(pointerdown 단계) 실행되어 레이스를 악화시킬 수 있음 |
| B. xterm.js 내부 selection 소실 메커니즘 | xterm.js v6 소스(`SelectionService`/`SelectionModel`) | 기존 완화 코드의 전제("우클릭 mousedown이 DOM selection을 collapse시켜 xterm이 clearSelection()한다")는 xterm.js v6 소스로 검증되지 않음(DOM 렌더러는 실제 브라우저 Selection API가 아니라 자체 오버레이 div로 선택을 그림). 대신 리사이즈(행 수 변경), 스크롤백 트리밍, 대체 화면 버퍼 전환(vim/less 등), PTY로의 사용자 입력 전송 시 xterm이 **자동으로** selection을 지운다는 사실을 소스에서 확인 |
| C. 고빈도 출력과의 상관관계 | `terminalOutputScheduler`/`terminalOutputHotPath`, xterm scrollback 트리밍 | (확신도 high) `SelectionModel.handleTrim()`이 스크롤백 트리밍 시 선택 영역이 버퍼 밖으로 밀려나면 조용히 `clearSelection()`을 호출함을 소스에서 직접 확인. scrollback은 10,000줄 고정이며, 세션이 고빈도로 출력 중일 때(`tail -f`, verbose 빌드/테스트, 폴링 스크립트 등) 선택~우클릭 사이 수백 ms 만에 선택 라인이 트리밍되어 선택이 사라짐 |

## 3. 종합 진단 (Root Cause Ranking)

세 조사 결과를 종합하면, 원인은 단일이 아니라 **여러 레이스가 중첩**되어 있으며, 우선순위는 다음과 같다.

### 3.1 (1순위, 확신도 high) xterm.js 스크롤백 트리밍에 의한 자동 selection 소실

- 근거: `frontend/node_modules/@xterm/xterm/src/browser/selection/SelectionModel.ts:123-136` `handleTrim(amount)` — 트리밍된 라인 수만큼 `selectionEnd`를 보정하다가 범위를 벗어나면 `this.clearSelection()`을 호출한다.
- `SelectionService.ts:144` — `this._bufferService.buffer.lines.onTrim(amount => this._handleTrim(amount))`로, `term.write()`가 스크롤백 한도를 넘길 때마다 실시간으로 트리거된다.
- `frontend/src/components/Terminal/TerminalView.tsx:1638` — `scrollback: 10000` 고정. 이미 10,000줄을 채운 장시간 세션(대부분의 실사용 세션)에서는 새 줄 1개당 트리밍 1개가 상시 발생한다.
- 이 소실은 **mousedown이 일어나기도 전**, 즉 사용자가 드래그로 선택을 끝낸 시점과 우클릭하는 시점 사이(사람의 반응 시간, 수백 ms~수 초)에 이미 벌어질 수 있다. 기존 완화 코드(`TerminalView.tsx:2012-2019`, `onMouseDownCapture`가 `term.getSelection()`을 저장)는 mousedown **시점 이후**의 소실만 방어하므로, mousedown 이전에 이미 비어버린 선택은 애초에 저장할 값이 없어 무력하다.
- "높은 빈도" 증상과 가장 잘 부합: 조용한 세션(사람이 명령 하나 치고 대기)에서는 거의 재현되지 않고, 로그 tail/빌드 출력처럼 초당 수십~수백 줄이 쏟아지는 세션에서 체감 빈도가 급격히 올라간다는 설명이 자연스럽다.

### 3.2 (2순위, 확신도 medium) 리사이즈로 인한 selection 소실

- 근거(Agent B): xterm.js `SelectionService`가 `bufferService.onResize`에서 `rowsChanged`일 때 `clearSelection()`을 호출한다.
- `TerminalView.tsx:2025-2058`의 `ResizeObserver` + rAF `fitAddon.fit()`이 패널 드래그(react-mosaic), 사이드바 토글 등으로 인한 미세 reflow마다 실행된다. 선택 도중 레이아웃이 변하면 selection이 사라질 수 있다.

### 3.3 (3순위, 확신도 medium) 메뉴 빌드 자체의 비동기 gap + 크로스탭 focus 레이스

- 근거(Agent A): `frontend/src/components/Grid/MosaicContainer.tsx`의 `openContextMenu`는 `commandPresetApi.getAll()`을 **await**한 뒤 `requestAnimationFrame` 콜백에서 `contextMenu.open(x, y, tabId)`를 호출하고, `contextMenuItems`는 `useMemo(() => buildMenuItems(contextMenu.targetId), [...])`로 **그 시점에** `hasSelection`을 다시 평가한다(스냅샷이 아니라 라이브 재평가). 즉 우클릭과 실제 메뉴 렌더링 사이에 원치 않는 지연 창이 하나 더 존재한다.
- 추가로 `TerminalRuntimeLayer.tsx`의 `onPointerDownCapture`가 버튼 종류와 무관하게 매번 `focusTerminal(...)`을 호출하며, 브라우저는 `pointerdown`을 `mousedown`보다 먼저 발생시키므로, 포커스가 없던 다른 pane에서 선택 후 우클릭하는 경우 이 강제 focus 전환이 선택 보존용 mousedown 캡처 핸들러보다 먼저 실행될 수 있다.

### 3.4 부가 관찰 (원인은 아니나 체감 문제와 연관)

- `writeOutputDirect`(`TerminalView.tsx:1165-1184`)는 대용량 쓰기(10,000바이트 초과) 시 `term.scrollToBottom()`을 호출한다(`TerminalView.tsx:822-853` 경유). 출력 폭주 중에는 사용자가 스크롤을 올려 과거 텍스트를 보던 중에도 뷰가 강제로 맨 아래로 스냅될 수 있어, 선택 대상 자체가 화면에서 사라지는 체감 문제를 동반한다(SelectionModel 자체를 지우진 않지만 사용자 경험상 트리밍 문제와 함께 체감 빈도를 높인다).
- `TerminalRuntimeLayer.tsx`의 `onContextMenu`/`onContextMenuCapture` 중복 등록은 `MosaicContainer.tsx`의 50ms 가드로 흡수되어 현재는 문제가 되지 않는 것으로 보이나, 취약한 패턴이다.
- 기존 완화 코드(`TerminalView.tsx:2009-2011` 주석)의 진단 전제("우클릭 mousedown이 DOM selection을 collapse시킨다")는 xterm.js v6 DOM 렌더러 소스로 확인되지 않았다(자체 오버레이 div 렌더링이며 실제 브라우저 Selection/Range API에 의존하지 않음). 이 완화책이 실제로 어떤 케이스를 막고 있는지는 재검증이 필요하다 — 다만 해가 되지는 않으므로 유지하되, 이번 조사의 1순위 원인(스크롤백 트리밍)에 대한 해결책은 아니라는 점이 중요하다.

## 4. 재현 조건 추정

- 장시간 실행되어 scrollback(10,000줄)이 이미 가득 찬 세션에서
- `tail -f`, verbose 빌드/테스트 로그, 폴링 스크립트 등 **초당 수십~수백 줄**을 계속 출력 중인 세션에서
- 사용자가 텍스트(특히 스크롤백 상단 근처의 오래된 히스토리)를 드래그 선택한 뒤
- 즉시가 아니라 약간의 지연(마우스를 우클릭 지점까지 옮기는 시간, 또는 패널 리사이즈/포커스 전환)을 두고 우클릭할 때

재현 빈도가 가장 높을 것으로 추정된다. 반대로 조용한 세션(명령 하나 실행 후 대기)에서는 거의 재현되지 않을 것으로 보이며, 이는 사용자가 보고한 "높은 빈도"라는 표현과 정합적이다.

## 5. SRS 추적 메모

- 이 증상을 직접 다루는 기존 Requirement는 없다(`docs/spec` 전체에 copy/clipboard/복사 관련 REQ 부재 확인).
- 가장 근접한 target은 `0.5.3step1`("Terminal context menu registered item paste workflow and nested menu behavior")이나 현재 `inactive/planned` 상태이며 다루는 범위도 paste/nested menu에 한정된다.
- 활성 target `0.5.5-buildergate-stability`의 `PERF-BGSTAB-002`(비동기 신원조회) 등 최근 작업은 출력 폭주 상황에서의 백엔드 안정성에 집중되어 있고, 프론트엔드 selection 보존 문제는 다루어진 적이 없다.
- 본 문서는 순수 조사 산출물이며 SRS 변경이나 코드 수정은 수행하지 않았다. 실제 fix를 진행할 경우 신규 REQ 등록(`/kiwi-srs`) 또는 `/kiwi-hot-fix` 경유 사후 동기화(`/kiwi-srs-sync`) 여부를 사용자가 먼저 결정해야 한다.

## 6. 권고 수정 방향 (구현 전 사용자 결정 필요 — 이번 조사에서는 미적용)

1. **선택 스냅샷을 더 이른 시점에 확보**: mousedown이 아니라 xterm의 `onSelectionChange` 이벤트(또는 selection이 non-empty가 될 때마다)를 구독해 "마지막으로 비어있지 않았던 선택"을 지속적으로 ref에 미러링. 스크롤백 트리밍이 mousedown 이전에 선택을 지워도, 트리밍 발생 직전 값이 이미 저장돼 있으므로 복사 가능.
2. **활성 선택이 있는 동안 트리밍 보호 또는 경고**: 스크롤백 트리밍 자체를 막을 수는 없지만(메모리 상한과 트레이드오프), 최소한 selection이 활성 상태일 때 트리밍으로 인한 clear를 감지해 텔레메트리/로그로 남기면 재현 조건 확정에 도움.
3. **리사이즈로 인한 clearSelection 영향 확인**: `ResizeObserver` → `fitAddon.fit()` 호출 빈도를 낮추거나(불필요한 미세 reflow 무시), 선택 도중 리사이즈 트리거를 지연시키는 방안 검토.
4. **메뉴 빌드 시점의 비동기 gap 축소**: `hasSelection`을 우클릭 시점에 동기적으로 스냅샷해 메뉴 items에 고정하고, preset 목록 로드는 별도로(메뉴가 이미 열린 후 비동기 갱신) 처리해 재평가 시점의 레이스를 원천 차단.
5. **기존 mousedown 캡처 완화 코드의 진단 재검증**: 실제로 어떤 케이스에서 유효한지 devtools/telemetry로 트레이싱해 확인하고, 필요 시 주석을 정확한 원인으로 갱신.

## 7. 조사 메타

- 서브에이전트 3개, 병렬 실행, 총 소요 약 8분(가장 긴 에이전트 기준), 코드 수정 없음.
- 조사 대상 주요 파일: `frontend/src/components/Terminal/TerminalView.tsx`, `frontend/src/components/Grid/MosaicContainer.tsx`, `frontend/src/components/Terminal/TerminalContainer.tsx`, `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx`, `frontend/src/utils/contextMenuBuilder.ts`, `frontend/node_modules/@xterm/xterm/src/browser/selection/SelectionModel.ts`, `frontend/node_modules/@xterm/xterm/src/browser/services/SelectionService.ts`.
