# Workspace/Session/Grid UX Research

> Project: ProjectMaster / BuilderGate
> Date: 2026-07-08
> Request: workspace list drag reorder, session move-to-workspace, grid vertical-column layout, running badge orange
> Method: 5-subagent research topology: Triage, Code Archaeology, External/Library, Risk/Alternatives, Synthesizer

---

## 1. Scope

The user requirements were interpreted as four UX changes:

1. In the workspace list, a desktop primary-button long press or mobile finger long press should allow moving workspace items up and down.
2. The terminal/session context menu should add `워크스페이스 이동`. It should open an existing modal/dialog component, show selectable workspaces, dim and disable full workspaces, move the session to the chosen workspace, and switch the screen to that workspace. Ended sessions must not support this action.
3. In grid mode, the tile toolbar at the upper-left should add a vertical-column layout option. Selecting it should arrange all session panes as equal-width tall columns, visually like `||||`.
4. The current/running badge should use an orange background instead of green.

This document is research only. No implementation changes were made.

## 3. Current Code Evidence

### Workspace reorder

The workspace list already sorts by `sortOrder` and calls the existing reorder pipeline:

- `frontend/src/components/Workspace/WorkspaceSidebar.tsx` uses `useDragReorder` and calls `onReorder(ids)`.
- `frontend/src/hooks/useWorkspaceManager.ts` exposes `reorderWorkspaces`.
- `frontend/src/services/api.ts` sends `PUT /api/workspaces/order`.
- `server/src/routes/workspaceRoutes.ts` handles `/order`.
- `server/src/services/WorkspaceService.ts` updates each workspace `sortOrder`.

The problem is the drag helper. `frontend/src/hooks/useDragReorder.ts` currently computes drop targets using `clientX` and horizontal midpoint logic, and the ghost follows `left` while keeping the original `top`. That is suitable for tab bars but wrong for a vertical workspace list.

### Session context menu and modal reuse

The terminal context menu is centralized in `frontend/src/utils/contextMenuBuilder.ts`. It currently builds:

- new session
- close session
- copy
- paste
- registered preset paste submenu

Both tab mode (`frontend/src/App.tsx`) and grid mode (`frontend/src/components/Grid/MosaicContainer.tsx`) consume this builder, so adding `워크스페이스 이동` there keeps both modes aligned.

For the modal, `frontend/src/components/dialog/WindowDialog.tsx` is the better reuse target than the older confirm modal. It already handles modal layering, `aria-modal`, focus trap, inert background, geometry, and focus restore. Existing examples include `CommandPresetDialog`.

### Missing session move API

There is no workspace-to-workspace tab/session move contract today:

- `workspaceApi` has workspace CRUD/order, tab add/update/delete/reorder/restart, and grid update, but no move method.
- `workspaceRoutes` has no move route.
- `WorkspaceService` has `addTab`, `deleteTab`, `reorderTabs`, and `restartTab`, but no method that changes a tab's `workspaceId` and reindexes both workspaces atomically.

Using `delete + add` would be wrong because it would create/terminate PTY sessions instead of moving the existing session.

### Grid layout

Grid layout mode is currently `equal | focus | auto | none` in `frontend/src/hooks/useMosaicLayout.ts`. The toolbar exposes only equal, focus, and auto in `frontend/src/components/Grid/MosaicToolbar.tsx`.

`frontend/src/utils/mosaic.ts` already models mosaic tree directions and equal layout arrangements. However, the default equal layout for 4+ sessions normally creates a grid baseline, not a guaranteed single row of tall columns. A single-row `||||` layout requires an explicit linear row tree.

Important nuance: if a one-shot `||||` command stores mode as `equal`, existing equal-mode rebuild effects can normalize the tree back to the responsive equal grid. If `||||` must behave like an active mode, it likely needs either a narrow new persisted mode or a persisted arrangement override.

### Running badge color

The green workspace running-count badge is in `frontend/src/components/Workspace/WorkspaceItem.tsx` with background `#22c55e`. The change should be scoped to this badge, not a broad replacement of all green UI states.

## 4. Consensus Findings

1. The workspace reorder storage/API path should be reused, but `useDragReorder` needs vertical orientation support.
   Evidence: workspace reorder path exists; drag target math is X-axis based.

2. Session move needs a server-side atomic API.
   Evidence: current APIs do not change `tab.workspaceId`; deleting a tab terminates the session.

3. `워크스페이스 이동` should be added via `buildTerminalContextMenuItems` and consumed by both tab mode and grid mode.
   Evidence: both `App.tsx` and `MosaicContainer.tsx` already use the same menu builder.

4. The workspace selection modal should reuse `WindowDialog`.
   Evidence: it already implements the modal behaviors this feature needs.

5. Full workspace and ended-session restrictions must be enforced by the server as well as the UI.
   Evidence: UI disabled states can be bypassed; server already owns tab capacity and lifecycle metadata.

6. `||||` is not equivalent to current responsive equal mode.
   Evidence: current equal mode may create 2x2, 3x2, or 3x3 baselines for 4-8 sessions.

7. The orange badge change should be targeted at the workspace running-count badge only.
   Evidence: other green styles represent different states such as idle/success/activity.

## 5. Open Design Decisions

### 5.1 `||||` Persistence

There are two viable approaches:

| Option | Description | Trade-off |
|---|---|---|
| One-shot command | Build a linear row tree and persist mode as `none` or explicit tree only. | Lower type/storage blast radius, but toolbar active state and tab-add behavior are less clear. |
| Narrow new mode | Add a mode such as `vertical`/`columns` that always rebuilds a linear row tree. | Better matches "mode change" UX and survives tab changes, but touches mode union, storage, tests, and migration handling. |

Recommended for implementation planning: if the user expects the toolbar icon to remain an active mode, use a narrow `vertical` mode with focused tests. If a one-time arrangement command is acceptable, use the lower-risk command approach.

### 5.2 Ended Session Definition

The UI has runtime `status: disconnected`, while the server stores lifecycle metadata such as `lifecycleState` and `recoverable`. The implementation should:

- disable the menu item for known disconnected/stopped tabs in the UI,
- reject stopped/non-recoverable tabs in the server move API,
- keep the server as the final authority.

### 5.3 Full Workspace Criteria

For moving an existing session, the target workspace tab limit is the relevant capacity check. Total session count should remain unchanged by a move, but the server should still preserve its existing resource-limit invariants.

## 6. Recommended Design

### 6.1 Workspace List Drag Reorder

Extend `useDragReorder` with an orientation option:

```ts
orientation?: 'horizontal' | 'vertical'
```

For `horizontal`, preserve current tab behavior. For `vertical`, compute the drop target with `clientY`, `rect.top`, and `rect.height / 2`, and move the ghost using `top: clientY - pointerOffset.y` while keeping the original `left`.

Also add:

- `pointercancel` cleanup,
- pointer capture/release around active drag,
- clear distinction between click selection and completed drag,
- focused regression tests to protect the existing tab bar behavior.

### 6.2 Session Move API

Add a server method equivalent to:

```ts
moveTab(sourceWorkspaceId: string, tabId: string, targetWorkspaceId: string): WorkspaceMoveResult
```

It should validate:

- source workspace exists,
- target workspace exists,
- tab exists and belongs to source,
- source and target differ,
- target workspace is not full,
- tab is not stopped/non-recoverable,
- session remains valid enough to move.

It should update atomically:

- `tab.workspaceId`,
- source workspace tab `sortOrder`,
- target workspace tab `sortOrder`,
- source `activeTabId` if the moved tab was active,
- target `activeTabId` to the moved tab,
- workspace `updatedAt`,
- persistent state.

Suggested route:

```http
POST /api/workspaces/:wid/tabs/:tid/move
Content-Type: application/json

{ "targetWorkspaceId": "..." }
```

Suggested broadcast: add a dedicated `tab:moved` event with enough data for local state updates, or return the updated full workspace state after move. A dedicated event is more incremental; full state refresh is simpler but heavier.

### 6.3 Frontend Move UX

Add `moveTab` to `workspaceApi` and `useWorkspaceManager`.

Add context menu item:

```text
워크스페이스 이동
```

Behavior:

- disabled or omitted when the target tab is ended/disconnected,
- opens `WorkspaceMoveDialog` based on `WindowDialog`,
- lists workspaces sorted by `sortOrder`,
- disables current workspace and full workspaces,
- uses dimmed visual styling for disabled workspaces,
- on selection, calls move API, updates local state, switches active workspace to target, and closes modal.

The dialog should not silently coerce an invalid target. If the server rejects the move, show a traceable error in the dialog.

### 6.4 Grid `||||` Layout

Add a helper that creates a linear row tree:

```ts
buildVerticalColumnsMosaicTree(ids)
```

Despite the UX name "vertical", the Mosaic split direction for `||||` is `row`, because the panes are arranged side by side as equal-width columns.

Implementation choices:

- For active mode semantics: add a narrow `vertical` layout mode and persist it.
- For command semantics: set the explicit tree and persist a non-auto-rebuilding state.

The toolbar should add an icon/button with a clear accessible name such as `세로 정렬`. It should preserve the existing move handle and equal/focus/auto controls.

### 6.5 Orange Running Badge

Change only the workspace running-count badge background from green to orange, for example from:

```ts
backgroundColor: '#22c55e'
```

to an existing orange/amber token such as:

```ts
backgroundColor: '#f97316'
```

Do not globally replace green values, because other green UI states have different meanings.

## 7. Risk Matrix

| Risk | Severity | Mitigation |
|---|---:|---|
| Vertical list uses horizontal drag math | High | Add orientation support and Y-axis tests. |
| Mobile long press conflicts with scroll/context menu | Medium | Add pointer cancel/capture handling and mobile E2E/manual checks. |
| Session move implemented as delete/add | High | Add atomic server move; never terminate/create PTY for move. |
| Full workspace disabled only in UI | High | Server capacity validation and visible dialog error on rejection. |
| Ended session moves through API | High | Server lifecycle validation; UI disable is advisory only. |
| Source/target active tab becomes invalid | High | Unit test source and target active-tab updates. |
| Moved tab leaves stale mosaic leaf | High | Prune/rebuild source and target layout state after move. |
| `||||` breaks layout persistence | Medium | Choose command vs mode deliberately and test reload/tab-add behavior. |
| Badge color change affects unrelated states | Medium | Edit only `WorkspaceItem` running badge. |

## 8. Verification Plan

Because this work changes runtime behavior, future implementation should be verified as Tier 2.

Recommended checks:

- Unit: `useDragReorder` horizontal regression, vertical reorder, primary-button filter, long-press threshold, `pointercancel` cleanup.
- Server: workspace reorder exact permutation validation if touched.
- Server: move success, target full, same workspace, missing workspace, missing tab, stopped/non-recoverable tab, source/target `sortOrder`, source/target `activeTabId`.
- Frontend unit: context menu item presence/disabled; modal disables full/current workspaces; dialog error rendering.
- Grid unit: `||||` helper creates equal-width linear row tree and preserves leaf order.
- E2E/manual at `https://localhost:2222`: desktop workspace drag, mobile long-press drag, move session from context menu, full workspace disabled, ended session unavailable, grid `||||`, orange running badge.

Per repository rule, manual and Playwright validation should target `https://localhost:2222`; the server should be started with `start.bat --port 2222` and stopped with `stop.bat` if needed.
