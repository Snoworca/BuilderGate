# 세션 오케스트레이션과 입력 안전성

## 설계 원칙

MCP는 세션을 "제어"하지만, PTY를 직접 소유하면 안 된다. 세션 생성/종료는 workspace lifecycle로, 입력은 공용 gateway로, 상태 관리는 registry로 분리한다.

```text
MCP tool call
  -> McpToolService
  -> policy/audit
  -> SessionAssignmentService
  -> WorkspaceSessionBindingService
  -> WorkspaceService / SessionInputGateway / SessionManager
```

이 구조를 택하는 이유:

- 기존 `SessionManager`가 이미 PTY lifecycle을 소유한다.
- 기존 `WorkspaceService`가 tab 생성/삭제/restart/orphan recovery를 소유한다.
- 기존 `WsRouter`가 replay/input safety 일부를 소유한다.
- MCP 입력, WebSocket 입력, restore 입력이 서로 다른 정책을 타면 상태 불변식이 깨진다.

## SessionInputGateway

신규 `SessionInputGateway`는 모든 입력 ingress의 단일 진입점이다.

```ts
interface SubmitInputRequest {
  sessionId: string;
  actor: InputActor;
  source: 'websocket' | 'restore' | 'mcp' | 'webhook';
  purpose: 'user-input' | 'agent-command' | 'kickoff' | 'message' | 'reply' | 'restore';
  data: string;
  submit?: boolean;
  metadata?: Record<string, unknown>;
  replayPolicy?: 'queue' | 'reject' | 'bypass-if-system';
  enterPolicy?: 'paste-only' | 'allow-submit' | 'deny-newline';
  targetGeneration?: number;
}
```

책임:

- byte limit, input queue limit, invalid session 검사.
- session ready 여부 검사.
- replay pending/repair pending이면 queue 또는 reject.
- newline/Enter submit 정책 적용.
- AI TUI foreground 입력이 `idle` invariant를 깨지 않도록 `SessionManager.writeInput` metadata 전달.
- audit hook 호출.
- rejected input에 observable error code 반환.

기존 경로 재배치:

- `WsRouter.handleInput`은 gateway 호출로 축소한다.
- `SessionManager.scheduleRestoreInput`은 gateway를 통해 restore 입력을 넣는다.
- MCP `message.send`, `open_agent`, `kickoff`, `reply_to_leader`도 gateway를 사용한다.

### Replay 상태 소유권

현재 replay/repair pending 상태는 `WsRouter`의 WebSocket client metadata 안에 있다. gateway가 세션 단위 입력 정책을 판단하려면 Phase 2에서 다음 중 하나를 먼저 구현해야 한다.

1. `SessionReplayCoordinator`를 새로 만들고 replay pending/repair pending/queued input 상태를 이 객체가 소유한다. `WsRouter`는 snapshot 전송과 ack만 담당한다.
2. 임시 단계로 `WsRouter`가 `ReplayStateProvider` 인터페이스를 제공하고 gateway가 세션의 replay pending 여부와 queue 가능 여부를 질의한다.

최종 구조는 1번이다. MCP 입력 안정성은 client별 metadata에 직접 접근하면 안 된다.

## 세션 binding

현재 `WorkspaceTab.sessionId`는 restart 시 바뀐다. MCP는 stable addressing을 위해 별도 binding을 둔다.

```text
sessionKey
  -> workspaceId
  -> tabId
  -> generation
  -> currentSessionId
```

동기화 이벤트:

- tab created: binding 생성.
- tab updated sessionId/generation: binding 갱신.
- tab deleted: binding closed/retired.
- session finalized: binding 상태 closed 또는 stopped 반영.
- orphan recovery: binding 유지, `currentSessionId` 교체.
- server startup: workspace state와 registry를 reconcile하여 누락된 binding을 backfill하고, 존재하지 않는 tab/session binding은 retired로 표시한다.

`sessionId` 직접 target 정책:

- 요청의 `sessionId`가 현재 binding의 `currentSessionId`와 일치하면 허용.
- 일치하지 않으면 `STALE_SESSION_ID`로 실패하고 새 `sessionKey/currentSessionId`를 안내한다.
- 장기 assignment는 항상 `sessionKey`를 저장한다.

## 세션 생성 흐름

`buildergate.session.open_agent` transaction:

```text
1. Actor policy check
2. AgentCommandProfile resolve
3. Preallocate sessionKey and binding draft
4. Mint session capability token
5. Prepare MCP client config and envPatch by profile mode
6. WorkspaceService.addTabWithLaunchContext(workspaceId, shell, alias, cwd, launchContext)
7. WorkspaceService passes envPatch/launch metadata into SessionManager.createSession before PTY spawn
8. Registry finalizes binding with tabId/currentSessionId/generation
9. SessionInputGateway.submitInput(command line, purpose='agent-command')
10. Wait for readiness signal
11. Inject kickoff prompt only if ready
12. Return created session info; if not ready, return kickoffPending=true
```

### PTY env injection

새 세션에서 코딩 에이전트가 자신과 리더를 알 수 있게 하려면 PTY process environment에 최소 다음 값을 넣어야 한다.

```text
BUILDERGATE_MCP_URL=http://127.0.0.1:<port>/mcp
BUILDERGATE_SESSION_ID=<current uuid sessionId>
BUILDERGATE_SESSION_KEY=<stable sessionKey>
BUILDERGATE_LEADER_SESSION_KEY=<leader sessionKey or empty>
BUILDERGATE_MCP_TOKEN=<session-bound capability token>
```

현재 `WorkspaceService.addTab()`은 즉시 `SessionManager.createSession()`을 호출해 PTY를 spawn한다. 따라서 env/token을 tab 생성 뒤에 붙이는 설계는 불가능하다. 구현은 `WorkspaceService.addTabWithLaunchContext()` 또는 기존 `addTab()`의 additive option을 추가해 spawn 전에 `envPatch?: Record<string,string>`와 launch metadata를 전달해야 한다. 이 타입은 MCP 전용이 아니라 일반 runtime option으로 둔다.

### MCP client 설정 전달

에이전트별로 MCP HTTP server 설정을 읽는 방식이 다를 수 있으므로 agent profile에 `mcpClientConfigMode`를 둔다.

- `env`: `BUILDERGATE_MCP_URL`과 `BUILDERGATE_MCP_TOKEN`을 그대로 사용한다.
- `generated-file`: 세션 전용 임시 MCP config 파일을 만들고 command args 또는 env로 경로를 넘긴다.
- `manual`: 자동 설정이 불가능한 경우 UI/응답에 one-time claim code를 표시하고, 에이전트가 `buildergate.session.claim`을 호출해 자기 세션을 연결한다.

`manual` 모드는 자동화 수준은 낮지만, 요구사항 6의 "불가능하다면 명시적으로 MCP를 호출하여 자신의 상태를 갱신" 경로와 함께 초기 호환성 확보에 유용하다.

### Agent ready 확인

우선순위:

1. 코딩 에이전트가 MCP `session.update_status({ agentStatus: 'ready' })` 호출.
2. foreground detector가 `claude`, `codex` 등 알려진 agent command를 감지.
3. timeout 후 `agentStatus='unknown'`으로 두고 kickoff 주입을 중단한다. 초기 구현에서는 ready 미확인 상태의 자동 paste fallback을 금지하고 `kickoffPending=true`를 반환한다.

자동 감지가 불가능한 경우를 요구사항이 허용하므로 explicit status update tool은 필수다.

## 메시지 전달 흐름

```text
User says: "빌더 게이트 세션에 계속 진행해 보내"
  -> Coding agent parses target="빌더 게이트", message="계속 진행해"
  -> buildergate.session.search({ query: "빌더 게이트" })
  -> one result
  -> buildergate.message.send({ target: sessionKey, message, submit: false })
  -> SessionInputGateway
  -> SessionManager.writeInput
```

정책:

- target stopped/closed이면 reject.
- target hidden이면 hidden input policy에 따라 queue 또는 reject. 초기 구현은 reject가 안전하다.
- target replay pending이면 gateway queue policy를 따른다.
- `submit=false`는 텍스트만 paste하고 Enter를 누르지 않는다.
- `submit=true`는 `mcp:message.submit` scope가 필요하다.

## 답장 흐름

Follower가 leader에게 답할 때:

```text
buildergate.message.reply_to_leader({ message: "완료했습니다." })
  -> actor binding에서 leaderSessionKey 조회
  -> leader live 여부 확인
  -> message.send로 위임
```

leader가 닫혔거나 retired이면 `LEADER_NOT_AVAILABLE`로 실패한다. 이 실패는 follower 세션을 종료하지 않는다.

## 킥오프 프롬프트

킥오프 프롬프트는 agent profile 또는 open request에서 결정한다.

적용 순서:

1. agent command가 주입된다.
2. agent ready signal을 기다린다.
3. kickoff prompt를 `purpose='kickoff'`로 입력한다.
4. kickoff 전달 audit와 assignment id를 저장한다.

주의:

- agent ready 전 무조건 prompt를 보내면 shell prompt에 그대로 입력될 수 있다.
- 초기 구현은 ready signal이 없으면 kickoff를 보류하고 UI/response에 `kickoffPending=true`를 표시하는 편이 안전하다.

## 종료 흐름

### 다른 세션 닫기

`session.close`는 destructive operation이다.

```text
policy check
  -> binding lookup
  -> WorkspaceService.deleteTab(workspaceId, tabId)
  -> registry closed
  -> broadcast tab:removed
```

현재 `WorkspaceService.deleteTab`은 tab을 non-recoverable로 저장한 뒤 `SessionManager.terminateSession`을 호출하고 tab을 제거한다. MCP close path는 이 메서드 하나만 호출해야 하며, 그 다음에 `SessionManager.terminateSession`을 다시 호출하면 안 된다.

### 자기 종료

자기 종료는 즉시 실행하면 MCP response가 끊길 수 있다. 반드시 deferred close다.

```text
close_self tool call
  -> actor binding lookup
  -> if no leaderSessionKey reject
  -> create close job
  -> return { accepted: true, closeJobId, delayMs }
  -> timer fires after response flush
  -> close target tab/session
```

수락 기준:

- leader 없는 직접 생성 세션은 `SELF_CLOSE_DENIED_NO_LEADER`.
- leader 있는 MCP-launched 세션은 response 후 tab이 사라지고 PTY process tree가 종료된다.
- 실패 시 leader에게 reply를 보낼 수 있으면 실패 메시지를 보낸다.

## 창 완전 종료 의미

요구사항의 "창을 완전히 닫아야 한다"는 현재 앱 구조에서는 workspace tab 또는 terminal pane을 닫는 것으로 해석한다. OS-level 앱 window 전체 종료가 아니라, 해당 세션 UI와 PTY process tree가 모두 제거되어야 한다.

완료 조건:

- workspace tab 제거.
- `SessionManager.getSession(sessionId)` 없음.
- process tree cleanup completed.
- WS 구독자에게 `tab:removed` 또는 session status update 전파.
- registry binding closed/retired.

## 실패와 observable error

MCP/webhook은 fallback을 조용히 수행하면 안 된다.

대표 error code:

- `UNBOUND_ACTOR`
- `INVALID_SCOPE`
- `TARGET_NOT_FOUND`
- `AMBIGUOUS_TARGET`
- `STALE_SESSION_ID`
- `TARGET_NOT_LIVE`
- `INPUT_REJECTED_REPLAY_PENDING`
- `INPUT_REJECTED_ENTER_POLICY`
- `SELF_CLOSE_DENIED_NO_LEADER`
- `MCP_PORT_REBIND_FAILED`
- `WEBHOOK_KEY_INVALID`

각 error는 audit id와 함께 반환한다.

## 테스트 포인트

- WebSocket input과 MCP input이 같은 byte/newline/replay 정책을 공유한다.
- restore input도 gateway를 통과한다.
- AI TUI foreground에서 사용자가 타이핑해도 status는 `idle`로 유지된다.
- `open_agent` 후 follower가 `whoami`로 leader를 볼 수 있다.
- `reply_to_leader`는 leader에게 paste-only 메시지를 보낸다.
- self-close는 leader 없는 세션에서 거부된다.
- self-close는 leader 있는 세션에서 response 이후 tab/session을 제거한다.
