# MCP HTTP와 웹훅 인터페이스 설계

## 기본 선택

MCP transport는 Streamable HTTP를 우선한다. 이유는 다음과 같다.

- 요구사항에 HTTP MCP, 포트 설정, 외부 IP whitelist, 웹훅이 포함되어 있다.
- 현재 앱은 이미 Express 기반이지만, MCP 포트만 재바인딩해야 하므로 독립 HTTP listener가 적합하다.
- stdio MCP는 로컬 IDE subprocess에는 단순하지만 외부 webhook, port rebind, IP whitelist 요구와 맞지 않는다.

공식 MCP 사양상 Streamable HTTP는 단일 MCP endpoint가 `POST`와 `GET`을 지원한다. Tool은 `tools/list`와 `tools/call`로 노출된다. Tool name은 ASCII 기반으로 설계하고, 한국어 alias는 tool input/output 데이터로만 사용한다.

## Listener 구조

```text
ProjectMaster process
  app HTTPS server       : https://localhost:2222
  redirect HTTP server   : http://localhost:2221
  MCP HTTP server        : http://127.0.0.1:<mcpPort>                 (loopback mode)
  MCP HTTPS/proxy server : https://<whitelisted-host>:<mcpPort>       (external whitelist mode)
    POST /mcp
    GET  /mcp
    POST /webhook/agent
    GET  /webhook/agent
```

`McpHttpServerController` 책임:

- 현재 config로 listener start.
- `rebind(nextConfig)` 실행.
- bind 실패 시 이전 listener 유지 또는 rollback.
- active port/status를 조회 가능하게 제공.
- stop 시 MCP listener만 닫고 앱 프로세스는 유지.

기본 config:

```json
{
  "enabled": true,
  "bindMode": "loopback",
  "host": "127.0.0.1",
  "port": 3333,
  "transportSecurity": "loopback-http",
  "trustedProxies": [],
  "externalWhitelist": [],
  "allowedOrigins": ["http://127.0.0.1:3333", "http://localhost:3333"]
}
```

외부 whitelist 모드:

- `bindMode='whitelist'`.
- `host='0.0.0.0'` 또는 사용자가 고른 interface address.
- `transportSecurity`는 `direct-tls` 또는 `trusted-tls-proxy`여야 한다. plain HTTP 외부 bind는 설정 검증 단계에서 거부한다.
- remote IP가 `externalWhitelist`의 IP/CIDR에 매칭되어야 한다.
- `Origin`이 비어 있거나 허용 목록에 없으면 거부한다. CLI client의 non-browser request는 Origin이 없을 수 있으므로 local loopback 또는 signed token일 때만 Origin 없음 허용을 정책으로 둔다.

`trusted-tls-proxy` 조건:

- `trustedProxies`에 정확한 proxy IP/CIDR을 등록해야 한다.
- MCP listener는 가능하면 `127.0.0.1` 또는 proxy 전용 내부 interface에 bind한다. `0.0.0.0` bind를 쓰는 경우 OS 방화벽 또는 reverse proxy 네트워크 정책으로 proxy 외 직접 접근을 차단해야 한다.
- 요청의 immediate remote address가 `trustedProxies`에 없으면 `X-Forwarded-*` 헤더를 신뢰하지 않는다.
- trusted proxy에서 온 요청도 `X-Forwarded-Proto=https` 또는 표준 `Forwarded` 헤더의 `proto=https`가 아니면 거부한다.
- proxy가 전달한 client IP만 `externalWhitelist`와 비교한다. proxy 밖에서 직접 들어온 HTTP 요청은 whitelist와 무관하게 거부한다.

## 인증 모델

브라우저 JWT는 MCP/webhook에 재사용하지 않는다.

Actor 종류:

| actor | 식별 방식 | 권한 범위 |
| --- | --- | --- |
| local UI admin | 기존 JWT | 설정 다이얼로그 REST API |
| MCP session actor | session-bound capability token | 자기 조회, 허용된 세션 검색/전달 |
| MCP leader actor | leader capability token | follower 생성, 지시, follower 종료 |
| webhook actor | random query key 또는 header key | 지정된 target/프로파일로 assignment 생성 |

토큰 원칙:

- token에는 `aud='buildergate-mcp'`, scope, exp, jti, sessionKey를 담는다.
- 저장소에는 token 원문을 저장하지 않고 hash 또는 signing secret만 보관한다.
- `sessionId`는 권한 증명이 아니다.
- 모든 실패는 audit에 actor type, source IP, reason, request id만 남긴다.

## MCP tool 목록

### `buildergate.session.whoami`

입력:

```json
{}
```

출력:

```json
{
  "sessionId": "uuid",
  "sessionKey": "stable-key",
  "workspaceId": "workspace-id",
  "tabId": "tab-id",
  "alias": "빌더 게이트",
  "agentKind": "claude",
  "leaderSessionKey": "leader-key-or-null",
  "bindingLifecycle": "live",
  "agentStatus": "ready"
}
```

정책:

- session-bound token이 없으면 `UNBOUND_ACTOR` 실패.
- legacy 세션은 `buildergate.session.claim`을 먼저 요구한다.

### `buildergate.session.claim`

입력:

```json
{
  "claimCode": "one-time-code",
  "agentKind": "claude"
}
```

용도:

- 이미 실행 중인 세션이나 MCP token이 주입되지 않은 legacy 세션을 현재 MCP actor와 연결한다.
- claim code는 UI 또는 launch coordinator가 짧은 만료 시간으로 발급한다.
- claim 성공 후 actor token은 해당 `sessionKey`에만 bind된다.
- claim code 없이 `sessionId`만 제출하는 방식은 허용하지 않는다.

### `buildergate.session.list`

입력:

```json
{
  "includeSelf": true,
  "status": ["ready", "busy", "waiting_input"],
  "includeStopped": false
}
```

출력:

```json
{
  "sessions": [
    {
      "sessionId": "uuid",
      "sessionKey": "stable-key",
      "alias": "빌더 게이트",
      "nameSource": "user",
      "workspaceId": "workspace-id",
      "tabId": "tab-id",
      "agentKind": "claude",
      "mcpConnected": true,
      "agentStatus": "ready",
      "leaderSessionKey": null
    }
  ]
}
```

### `buildergate.session.search`

입력:

```json
{
  "query": "빌더 게이트",
  "agentKind": "claude",
  "includeSelf": false
}
```

검색 순서:

1. `alias` exact match.
2. `alias` case/space normalized match.
3. tab name exact/prefix.
4. agent command alias.
5. cwd/recovery command contains match.

모호성 정책:

- 0건: `NOT_FOUND`.
- 1건: resolved target 반환.
- 2건 이상 동점: `AMBIGUOUS_TARGET`와 후보 반환.

### `buildergate.message.send`

입력:

```json
{
  "target": { "sessionKey": "stable-key" },
  "message": "계속 진행해",
  "deliveryMode": "paste",
  "submit": false
}
```

정책:

- 기본은 paste-only.
- `submit=true` 또는 message에 newline/Enter가 포함되면 actor scope와 policy gate가 필요하다.
- 내부적으로 `SessionInputGateway.submitInput`을 호출한다.
- target binding generation이 바뀐 경우 재검색을 요구한다.

### `buildergate.session.set_alias`

입력:

```json
{
  "target": { "sessionKey": "stable-key" },
  "alias": "빌더 게이트"
}
```

동작:

- `WorkspaceService.updateTab(tabId, { name: alias, nameSource: 'user' })`와 같은 의미로 저장한다.
- `tab:updated`를 broadcast한다.
- audit에 previous alias hash/preview와 new alias preview를 남긴다.

### `buildergate.session.open_agent`

입력:

```json
{
  "agentProfileId": "profile-id",
  "alias": "빌더 게이트",
  "workspaceId": "optional-workspace",
  "cwd": "optional-cwd",
  "leaderSessionKey": "optional-leader",
  "kickoffPrompt": "optional override"
}
```

동작 순서:

1. `sessionKey`, binding draft, capability token을 먼저 선할당한다.
2. agent profile의 `mcpClientConfigMode`에 따라 MCP client 설정과 launch env를 준비한다.
3. 확장된 workspace launch API로 tab과 PTY session을 만든다. 이 API는 spawn 전에 `envPatch`와 launch metadata를 `SessionManager.createSession`으로 전달해야 한다.
4. 생성된 runtime `sessionId`, `tabId`, `workspaceId`로 binding draft를 active binding으로 확정한다.
5. agent command profile의 command와 args를 `SessionInputGateway`로 주입한다.
6. foreground detector 또는 explicit `update_status`로 agent ready를 확인한다.
7. ready가 확인된 경우에만 kickoff prompt를 첫 메시지로 전달한다. timeout이면 `kickoffPending=true`로 남긴다.

실패 보상:

- 1-2단계 실패: 외부 side effect가 없으므로 요청만 실패 처리한다.
- binding draft 또는 capability token 생성 뒤 spawn 전 실패: launch attempt를 `cancelled`로 표시하고 token을 폐기하며 generated-file이 있으면 삭제한다. `McpSessionBinding.bindingLifecycle`에는 `cancelled`를 넣지 않는다.
- tab/session 생성 후 binding finalize 또는 command 주입 실패: `WorkspaceService.deleteTab(workspaceId, tabId)`를 호출해 PTY와 tab을 정리하고, token 폐기, generated-file 삭제, launch attempt `failed`와 cleanup 결과를 기록한다. active binding이 만들어졌다면 `bindingLifecycle='closed'`로 정리한다.
- agent command 주입 성공 후 ready timeout: 세션은 유지하고 `kickoffPending=true`, `agentStatus='unknown'`을 반환한다. 자동으로 세션을 닫지 않는다.
- kickoff 주입 실패: 세션은 유지하고 assignment를 `failed`로 기록하며 사용자가 재시도할 수 있게 한다.

MCP client 설정:

- `env`: 에이전트가 환경 변수 기반 MCP 설정을 지원할 때 사용한다.
- `generated-file`: Claude/Codex 등 CLI가 설정 파일을 읽는 경우, 임시 MCP config 파일을 만들고 command args/env로 경로를 전달한다.
- `manual`: 자동 설정이 불가능하면 UI에 claim code를 표시하고 에이전트가 `session.claim`을 호출하게 한다.

### `buildergate.session.close`

입력:

```json
{
  "target": { "sessionKey": "stable-key" },
  "reason": "completed",
  "closeTab": true
}
```

정책:

- destructive scope 필요.
- self target이면 `close_self`로 redirect하지 않고 명시적 거부 또는 별도 policy 적용.
- tab까지 닫는 종료는 `WorkspaceService.deleteTab` 한 경로만 호출한다. 이 메서드가 이미 `SessionManager.terminateSession`을 수행하므로 MCP layer가 다시 terminate를 호출하면 안 된다.
- runtime만 종료하는 별도 기능이 필요할 때만 dedicated close coordinator를 만들고, tab 삭제 경로와 중복되지 않게 한다.

### `buildergate.session.close_self`

입력:

```json
{
  "reason": "completed"
}
```

정책:

- `leaderSessionKey`가 없으면 `SELF_CLOSE_DENIED_NO_LEADER`.
- tool response 전송 후 250-1000ms 지연하여 close를 예약한다.
- 예약 결과와 audit id를 먼저 반환한다.

### `buildergate.message.reply_to_leader`

입력:

```json
{
  "message": "작업 완료. 테스트 통과.",
  "submit": false
}
```

정책:

- actor의 `leaderSessionKey`가 있어야 한다.
- leader binding이 live 상태여야 한다.
- 내부적으로 `message.send` 재사용.

### `buildergate.session.update_status`

입력:

```json
{
  "agentKind": "claude",
  "agentStatus": "ready",
  "detail": "waiting for prompt"
}
```

용도:

- 자동 foreground 감지가 불가능한 경우 agent가 명시적으로 상태를 갱신한다.
- status는 management system과 list/search 결과에 반영된다.

## 웹훅 설계

요구사항은 query parameter random key를 요구한다. 보안상 header token이 더 안전하지만 요구사항 충족을 위해 query key를 지원하고 다음 완화책을 적용한다.

URL 예시:

```text
http://127.0.0.1:3333/webhook/agent?key=<random-key>&prompt=<urlencoded>
```

이 URL 형식은 loopback 또는 TLS가 검증된 외부 endpoint에서만 유효하다. 외부 whitelist 모드의 plain HTTP URL은 생성하지 않는다.

권장 POST body:

```json
{
  "prompt": "작업 지시",
  "target": "빌더 게이트",
  "agentProfileAlias": "클로드",
  "mode": "open-or-send"
}
```

동작:

- key hash lookup.
- IP whitelist와 rate limit 검사.
- prompt는 size limit, safe preview, hash 생성.
- GET 요청의 `prompt` query 값은 UTF-8 기준 2048자 이하로 제한한다. 더 큰 prompt는 POST body만 허용한다.
- source session id는 `"0"`으로 기록.
- target이 있으면 search/resolve 후 send.
- target이 없으면 default agent profile로 새 session open.

로그 redaction:

- access log에서 query string 제거.
- audit에는 key id, prompt hash, prompt preview 최대 길이만 저장.
- raw prompt 원문은 audit/status/promptPreview 경로에 저장하지 않는다. assignment 추적에는 redacted preview, hash, 비밀이 제거된 metadata만 저장하며 encrypted store도 raw prompt 보관 용도로 사용하지 않는다.

## 포트 rebind 알고리즘

```text
receive requested config
  -> validate port range, host, whitelist, origin
  -> validate transportSecurity; external mode requires TLS or trusted TLS proxy
  -> create candidate listener on new host:port
  -> health probe candidate
  -> swap active controller reference
  -> drain old listener
  -> persist config
```

실패 처리:

- candidate bind 실패: 기존 listener 유지, config 저장하지 않음.
- persist 실패: candidate stop, 기존 listener 유지.
- old listener drain 실패: 새 listener는 유지하되 warning과 audit 기록.

## REST 관리 API

UI 다이얼로그는 MCP protocol이 아니라 기존 authenticated REST API를 사용한다.

- `GET /api/mcp-control/config`
- `PATCH /api/mcp-control/config`
- `GET /api/mcp-control/agents`
- `POST /api/mcp-control/agents`
- `PATCH /api/mcp-control/agents/:id`
- `DELETE /api/mcp-control/agents/:id`
- `GET /api/mcp-control/webhooks`
- `POST /api/mcp-control/webhooks`
- `POST /api/mcp-control/webhooks/:id/rotate`
- `DELETE /api/mcp-control/webhooks/:id`
- `GET /api/mcp-control/sessions`
- `GET /api/mcp-control/sessions?query=...`
- `POST /api/mcp-control/sessions/search-test`
- `POST /api/mcp-control/sessions/:sessionKey/reply-test`
- `POST /api/mcp-control/sessions/:sessionKey/close`
- `PATCH /api/mcp-control/sessions/:sessionKey/alias`

REST route는 `authMiddleware` 보호를 받지만 MCP runtime token과는 분리한다.
