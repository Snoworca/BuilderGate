# 현재 아키텍처 조사

## 요약

현재 애플리케이션은 MCP 서버를 구현하지 않는다. 대신 세션 실행, 입력, workspace tab, 복구 옵션, 설정 UI를 위한 기반은 이미 충분히 있다. 구현은 새 MCP layer가 이 기반을 직접 오염시키지 않고 재사용하도록 설계해야 한다.

핵심 현황은 다음과 같다.

- 서버는 Express/HTTPS/WS 기반이며 `/api/settings`, `/api/command-presets`, `/api/recovery-options`, `/api/sessions`, `/api/workspaces`를 mount한다.
- MCP SDK 의존성은 없다.
- PTY runtime 세션은 `SessionManager`가 `uuidv4()`로 생성한다.
- workspace tab은 `WorkspaceService`가 관리하고, tab에는 `sessionId`, `name`, `nameSource`, lifecycle, recovery metadata가 있다.
- REST input endpoint는 제거되었고, 사용자 입력은 WebSocket `input` 메시지를 통해 들어온다.
- 자동 복구 입력은 현재 `SessionManager.writeInput`을 직접 호출하므로 WebSocket replay safety를 완전히 공유하지 않는다.
- 설정 화면은 `server.port`를 편집 대상에서 제외한다. MCP 포트 변경은 별도 service/listener controller가 필요하다.

## 세션 런타임

근거 파일:

- `server/src/services/SessionManager.ts:572` - `createSession`에서 `uuidv4()`로 runtime `sessionId` 생성.
- `server/src/services/SessionManager.ts:1152` - `getAllSessions`.
- `server/src/services/SessionManager.ts:1168` - `terminateSession`.
- `server/src/services/SessionManager.ts:1250` - `writeInput`.
- `server/src/services/SessionManager.ts:1528` - `onCommandSubmitted`.
- `server/src/services/SessionManager.ts:1562` - `scheduleRestoreInput`.
- `server/src/services/SessionManager.ts:1610` - restore input이 `writeInput`을 직접 호출.
- `server/src/services/SessionManager.ts:2652` - finalizer cleanup.

`SessionManager`는 PTY lifecycle, 입력, 상태, foreground detector, cleanup을 이미 많이 소유한다. MCP 구현이 여기에 직접 tool 정책, actor 권한, 검색, audit를 넣으면 클래스가 더 커지고 테스트가 어려워진다.

권장 경계:

- `SessionManager`: PTY spawn/write/resize/terminate/finalize만 유지.
- `SessionInputGateway`: 입력 검증, replay policy, Enter policy, metadata, audit hook을 통합.
- `McpSessionControlService`: MCP tool semantics와 actor/target policy 처리.
- `WorkspaceSessionBindingService`: tab, sessionKey, currentSessionId, generation binding 관리.

## Workspace와 tab

근거 파일:

- `server/src/types/workspace.types.ts:19` - `WorkspaceTab`.
- `frontend/src/types/workspace.ts:14` - frontend `WorkspaceTab`.
- `server/src/routes/workspaceRoutes.ts:98` - tab 추가 REST route.
- `server/src/routes/workspaceRoutes.ts:124` - tab update REST route.
- `server/src/routes/workspaceRoutes.ts:144` - tab delete REST route.
- `server/src/routes/workspaceRoutes.ts:169` - tab restart REST route.
- `server/src/services/WorkspaceService.ts:331` - tab 생성 시 session 생성.
- `server/src/services/WorkspaceService.ts:402` - tab delete 시 session terminate 후 tab 제거.
- `server/src/services/WorkspaceService.ts:519` - restart tab.
- `server/src/services/WorkspaceService.ts:1338` - orphan recovery.

현재 tab은 `sessionId`를 직접 가진다. restart/orphan recovery 때 새 runtime session을 만들어 `tab.sessionId`를 교체한다. 그래서 MCP의 안정 주소로 runtime `sessionId`만 쓰면 오래 실행되는 assignment가 엉뚱한 상태가 될 수 있다.

중요한 제약:

- 현재 `WorkspaceService.addTab()`은 내부에서 즉시 `SessionManager.createSession()`을 호출해 PTY를 spawn한다. MCP-launched agent에 `BUILDERGATE_MCP_TOKEN` 같은 env를 넣으려면 tab 생성 전에 `sessionKey`와 capability token을 선할당하고, `addTabWithLaunchContext()` 같은 additive API가 spawn 전 `envPatch`를 전달해야 한다.
- 현재 `WorkspaceService.deleteTab()`은 이미 `SessionManager.terminateSession()`을 호출한 뒤 tab을 제거한다. MCP close path는 `deleteTab()`과 `terminateSession()`을 중복 호출하면 안 된다.

권장:

- tab에는 기존 `sessionId` 유지.
- MCP registry가 `sessionKey -> currentSessionId + generation + tabId` binding을 관리.
- tool input은 `sessionId`와 `sessionKey`를 모두 허용하되, `sessionId`는 현재 binding과 generation을 검증한다.

## 현재 구조에서 가장 위험한 지점

1. `SessionManager.writeInput`을 MCP가 직접 호출하면 replay/Enter/hidden terminal 정책이 흩어진다.
2. runtime `sessionId`만으로 장기 assignment target을 잡으면 restart 후 target이 깨진다.
3. 브라우저 JWT를 외부 webhook이나 MCP client에 쓰면 token 유출 시 앱 전체 권한이 노출된다.
4. 포트 rebind를 기존 `index.ts` listen 흐름에 직접 얹으면 앱 서버와 MCP 서버 lifecycle이 꼬인다.
5. self-close는 tool response 전에 자기 PTY를 죽일 수 있으므로 반드시 deferred operation이어야 한다.
6. 외부 whitelist 모드에서 plain HTTP를 열면 token, query key, prompt가 평문 노출된다. 외부 접근은 TLS 또는 trusted TLS proxy가 필수다.
7. replay/repair pending 상태는 현재 `WsRouter` client metadata에 있으므로 `SessionInputGateway` 전에 `SessionReplayCoordinator` 또는 `ReplayStateProvider`가 필요하다.
