# 보안, 운영, 테스트 계획

## 위협 모델

이 기능은 외부 또는 다른 코딩 에이전트가 터미널에 입력을 넣고, 세션을 만들고, 세션을 닫을 수 있게 한다. 따라서 단순 "조회 API"가 아니라 원격 제어면이다.

주요 위협:

- 외부 IP에서 MCP endpoint 접근.
- DNS rebinding으로 localhost MCP 호출.
- 브라우저 JWT 재사용에 따른 전체 앱 권한 탈취.
- query string webhook key가 로그, history, proxy에 남는 문제.
- prompt injection으로 newline/Enter 포함 명령 실행.
- 잘못된 sessionId로 다른 세션에 입력.
- self-close가 리더 없는 사용자 세션을 닫음.
- MCP port rebind 실패로 endpoint가 의도와 다르게 노출.
- 세션 종료 시 process tree가 남음.

## Must 정책

1. MCP 기본 bind는 `127.0.0.1`.
2. 외부 접근은 whitelist 모드에서만 허용.
3. 외부 whitelist 모드는 TLS 또는 신뢰된 TLS reverse proxy 없이는 활성화하지 않는다.
4. Origin 검증을 수행한다.
5. MCP/webhook token은 브라우저 JWT와 분리한다.
6. `sessionId`는 식별자일 뿐 권한 증명이 아니다.
7. tool마다 scope를 둔다.
8. newline/Enter submit은 paste-only보다 강한 권한이 필요하다.
9. destructive tool은 audit와 confirmation/policy gate가 필요하다.
10. webhook key는 hash 저장, query string redaction.
11. 모든 의미 있는 거부는 observable error와 audit id를 반환한다.

## Scope 모델

| scope | 허용 작업 |
| --- | --- |
| `mcp:self.read` | 자기 session 조회 |
| `mcp:sessions.list` | 세션 목록/검색 |
| `mcp:sessions.alias.write` | alias 변경 |
| `mcp:message.paste` | paste-only 메시지 전달 |
| `mcp:message.submit` | Enter 포함 제출 |
| `mcp:session.open` | 새 agent 세션 생성 |
| `mcp:session.close` | 다른 세션 종료 |
| `mcp:session.close_self` | follower 자기 종료 |
| `mcp:status.write` | 자기 agent status 갱신 |
| `mcp:webhook.invoke` | webhook assignment 생성 |

기본 session-bound token은 `mcp:self.read`, `mcp:sessions.list`, `mcp:message.paste`, `mcp:status.write`만 갖는다. leader가 follower를 만들 때 follower token에는 `mcp:message.paste`, `mcp:session.close_self`를 추가할 수 있다.

## IP와 Origin 정책

Loopback 모드:

- bind host는 `127.0.0.1`.
- remote address가 loopback이 아니면 거부.
- Origin이 있으면 `localhost`/`127.0.0.1` endpoint와 일치해야 한다.

Whitelist 모드:

- bind host는 명시 interface 또는 `0.0.0.0`.
- transport는 direct TLS 또는 trusted TLS proxy여야 한다.
- trusted TLS proxy 모드에서는 immediate remote address가 `trustedProxies`에 있어야만 forwarded header를 신뢰한다.
- `X-Forwarded-Proto` 또는 `Forwarded proto`가 `https`가 아니면 거부한다.
- proxy를 우회한 direct HTTP 접근은 whitelist와 무관하게 거부한다.
- effective client IP가 whitelist IP/CIDR에 들어야 한다. direct TLS 모드에서는 immediate remote address이고, trusted TLS proxy 모드에서는 신뢰된 proxy가 전달한 forwarded client IP다.
- proxy header는 기본적으로 신뢰하지 않는다.
- reverse proxy 지원이 필요하면 `trustedProxies`를 별도 설정으로 둔다.
- Origin 검증은 계속 수행한다.

## Webhook query key 완화책

사용자는 query parameter random key를 요구한다. 이를 허용하되 다음을 반드시 지킨다.

- key 길이는 최소 32 bytes entropy.
- 저장은 `keyId`, `keyHash`, `createdAt`, `expiresAt`, `revokedAt`.
- access log에는 path만 남기고 query는 제거.
- audit에는 key id만 남긴다.
- GET은 편의용으로 허용하되, `prompt` query 값은 UTF-8 기준 2048자 이하만 허용한다. 더 큰 prompt는 POST body만 허용한다.
- prompt size limit과 rate limit을 적용한다.
- key rotate 시 이전 key grace period는 짧게 둔다.

## Audit log

감사 이벤트 필드:

```ts
interface AuditEvent {
  id: string;
  at: string;
  actorType: 'ui-admin' | 'mcp-session' | 'webhook' | 'system';
  actorSessionKey?: string;
  sourceIp?: string;
  toolName?: string;
  action: string;
  targetSessionKey?: string;
  targetSessionId?: string;
  scope: string[];
  result: 'allowed' | 'denied' | 'failed';
  reason?: string;
  promptHash?: string;
  promptPreview?: string;
}
```

저장 금지:

- raw JWT.
- MCP token.
- webhook random key.
- raw long prompt.
- TOTP secret.

## 운영 상태

MCP 운영 상태는 UI와 API에서 조회 가능해야 한다.

상태 필드:

- enabled.
- bind host/port.
- listener status.
- active connection count.
- last rebind result.
- last error.
- rejected request counters by reason.
- webhook last used.
- audit recent events.

## 테스트 전략

이 작업은 Tier 3에 가깝다. 인증, 권한, 외부 접근, 세션 종료, 프로세스 cleanup을 다룬다. 구현 시 전체 검증은 단위, 통합, E2E, 보안 회귀를 모두 포함해야 한다.

### 서버 단위 테스트

Config:

- 기본 config는 loopback bind.
- invalid port reject.
- invalid CIDR reject.
- external whitelist mode with plain HTTP reject.
- external whitelist mode requires direct TLS or trusted TLS proxy.
- trusted proxy header accepted only from `trustedProxies`.
- trusted proxy request with non-https proto rejected.
- direct proxy-bypass HTTP rejected.
- whitelist mode에서 빈 whitelist는 reject.
- port rebind 실패 시 기존 listener 유지.

Auth:

- 브라우저 JWT로 MCP tool call 거부.
- session-bound token으로 whoami 성공.
- one-time claim code로 legacy 세션 claim 성공.
- claim code 재사용 거부.
- scope 부족 시 403.
- expired token 거부.
- replay jti 거부.
- wrong audience 거부.

Search:

- alias exact match.
- terminal-title display name은 낮은 우선순위.
- ambiguous target returns candidates.
- stopped/retired 제외.

Input:

- paste-only는 Enter를 보내지 않는다.
- newline 포함 prompt는 `mcp:message.submit` scope 없으면 거부.
- replay pending 상태에서 queue/reject 정책 검증.
- hidden terminal input 초기 정책 reject.
- AI TUI foreground 입력은 `idle` 유지.

Lifecycle:

- open_agent가 tab, binding, capability token을 생성한다.
- agent profile의 MCP client config mode별로 env/generated-file/manual 경로가 동작한다.
- explicit status update가 live session에 반영된다.
- reply_to_leader가 leader session에 메시지를 보낸다.
- close_self leader 없음 거부.
- close_self leader 있음 deferred close 성공.
- close_session 후 tab 제거와 process tree cleanup 완료.

Webhook:

- invalid key 거부.
- revoked key 거부.
- expired key 거부.
- GET prompt query 2048자 초과 거부.
- prompt size 초과 거부.
- source session id `"0"` 기록.
- query string redaction.
- rate limit.

### 프론트 단위 테스트

- Tools 메뉴 MCP 항목.
- dialog tab navigation.
- agent profile CRUD form validation.
- port rebind pending/success/failure 표시.
- security whitelist validation error 표시.
- webhook key masked 표시.
- live session alias 변경과 `tab:updated` 반영.

### 통합 테스트

- MCP `/mcp` endpoint에서 `tools/list`와 `tools/call` 동작.
- loopback 외 address 거부.
- whitelist address 허용.
- Origin mismatch 거부.
- `session.search -> message.send` 흐름.
- `open_agent -> update_status -> kickoff -> reply_to_leader -> close_self` 흐름.

### E2E 테스트

AGENTS.md 규칙에 따라 수동/Playwright 검증은 `https://localhost:2222`를 대상으로 한다.

시나리오:

1. Tools 메뉴에서 MCP 세션 제어 다이얼로그 열기.
2. MCP 포트 변경 후 listener 상태 active 확인.
3. 보안 탭에서 whitelist 저장 오류/성공 확인.
4. agent profile `claude`, alias `클로드` 생성.
5. 새 세션 열기.
6. 하단 상태 표시줄 왼쪽 alias를 `빌더 게이트`로 변경.
7. MCP search로 `빌더 게이트` resolve.
8. 메시지 paste.
9. follower가 leader에게 reply.
10. follower self-close.

## 완료 기준

- 모든 신규 서버 단위 테스트 통과.
- 관련 프론트 단위 테스트 통과.
- MCP 통합 테스트 통과.
- Playwright E2E 핵심 시나리오 통과.
- 보안 review에서 no critical/high finding.
- audit log에서 raw secret/prompt가 저장되지 않음 확인.
