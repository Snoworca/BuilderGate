# 팀장/팀원 세션 MCP 프롬프트 전송 및 완료 신호 연구

작성일: 2026-05-19 KST  
대상 프로젝트: `C:\Work\git\_Snoworca\ProjectMaster`  
요청 범위: 코드 변경 없이, 팀장/팀원 지정 시 MCP 기반 프롬프트 전송, `.buildergate` JSON 세션 키 저장, 특정 세션 메시지 전송, idle 단독 완료판정 금지, MCP/hook 완료 신호 설계를 추가 연구한다.

## 1. 결론

권장 구조는 다음과 같다.

```text
팀장 에이전트
  -> BuilderGate MCP tool
  -> TeamCoordinationService
  -> SessionKeyRegistry / AssignmentStore / AuditLog
  -> SessionInputService
  -> SessionManager / WsRouter 입력 안정성 계약
  -> 팀원 AI TUI 세션

팀원 에이전트
  -> MCP submit_assignment_result
     또는 buildergate-hook assignment-complete
  -> HookSignalService
  -> AssignmentStore + SessionManager external activity merge
```

핵심 정책:

1. 프롬프트 전송은 MCP tool로만 열고, raw WebSocket 조작이나 브라우저 창 조작으로 만들지 않는다.
2. 외부에서 지정할 대상은 volatile `sessionId`가 아니라 stable `sessionKey`여야 한다.
3. `.buildergate` JSON에는 `sessionKey -> currentSessionId` 매핑과 role/권한/assignment 상태를 저장하되, secret 원문은 같은 파일에 저장하지 않는다.
4. `idle`은 완료의 필요조건도 충분조건도 아니다. 완료는 `assignmentId`가 들어간 명시적 MCP 또는 hook 완료 신호가 있어야 한다.
5. 대화 종료 hook은 기본적으로 `turn-closed` 신호일 뿐이다. semantic 완료로 승격하려면 `assignmentId`, `status`, HMAC 서명 envelope, sequence, nonce가 있는 completion event가 필요하다.
6. `SessionManager`는 계속 status source of truth다. MCP/hook이 WebSocket `status`를 직접 emit하면 안 된다.

## 3. 권장 프로토콜

### 3.1 용어

| 이름 | 의미 | 안정성 | 보안 의미 |
|---|---|---:|---|
| `sessionId` | 현재 PTY runtime id | 낮음. 재시작/복구 시 바뀜 | 권한으로 쓰면 안 됨 |
| `sessionKey` | 탭/팀원에 붙는 stable id | 높음. tab lifecycle 동안 유지 | 식별자일 뿐 secret 아님 |
| `sessionNumber` | 사람이 보는 짧은 번호 | 낮음. 표시/선택 편의 | 권한으로 쓰면 안 됨 |
| `assignmentId` | 팀장이 팀원에게 배정한 작업 id | 높음 | 완료/감사 단위 |
| `capability` | MCP/hook write 권한 | 높음 | secret 또는 secret hash 필요 |

`sessionKey`는 `sess_` prefix + ULID/UUID 같은 랜덤 식별자가 적합하다. `sessionNumber`는 `1`, `2` 같은 UI 표시용 alias로만 둔다. MCP write tool은 `sessionNumber`를 직접 받더라도 서버가 `teamGeneration` 범위에서 정확히 하나의 `sessionKey`로 resolve할 수 있을 때만 허용해야 한다.

### 3.2 프롬프트 전송 흐름

```text
1. 팀장 MCP client가 list_team_sessions 호출
2. target sessionKey 확인
3. create_assignment 호출
4. preview_agent_prompt 호출
5. deliver_agent_prompt 호출
6. 서버가 role/scope/approval/newline 정책 검증
7. SessionInputService가 target currentSessionId로 입력 전달
8. 팀원 hook 또는 MCP가 assignment-start/progress/complete 제출
9. get_assignment_status 또는 read_audit_log로 확인
```

`deliver_agent_prompt`는 기본적으로 `submit=false` 또는 `mode="paste"`가 안전하다. Enter 포함 자동 제출은 다음 조건을 만족할 때만 허용한다.

- target이 `worker` role이고 `controllable=true`
- assignment가 `approved` 또는 policy상 write 허용 상태
- target agent 상태가 `waiting_input` 또는 명시적 attention 상태
- replay/snapshot/input barrier가 late Enter execution을 만들지 않는 경로
- audit log에 prompt hash, byte length, safe preview, actor, target, approval id 기록

### 3.3 팀원 프롬프트 envelope

팀원에게 보내는 프롬프트는 plain text만 보내지 말고 BuilderGate envelope를 붙인다.

```text
[BuilderGate Assignment]
assignment_id: asg_...
target_session_key: sess_...
worker_role: worker
lead_session_key: sess_...
completion_required: call MCP submit_assignment_result OR run buildergate-hook assignment-complete/assignment-failed

작업 내용:
...

완료 시:
1. 결과 요약을 작성한다.
2. BuilderGate MCP tool submit_assignment_result를 호출한다.
3. MCP가 불가능하면 성공 시 buildergate-hook assignment-complete, 실패 시 buildergate-hook assignment-failed를 호출한다.
```

이 envelope는 상태 판정의 source가 아니라 작업자가 완료 신호를 제출하도록 유도하는 instruction이다. 완료 판정은 반드시 서버가 받은 서명된 completion signal 기준으로 처리한다.

## 4. `.buildergate` JSON 설계

현재 프로젝트에는 `.buildergate` 디렉터리가 없고, 서버 runtime 데이터의 기본 위치는 `server/data`다. 그래서 권장 위치는 둘 중 하나다.

1. source runtime: `server/data/.buildergate/team-sessions.json`
2. packaged/runtime: `<runtime data dir>/.buildergate/team-sessions.json`

프로젝트 루트 `./.buildergate/team-sessions.json`을 쓰려면 `.gitignore` 대상이어야 하며, owner-only 권한을 적용해야 한다. 운영 데이터가 source tree에 섞이는 것을 피하려면 `workspace.dataPath`와 같은 runtime data root 아래가 더 낫다.

권장 파일:

```json
{
  "version": 1,
  "teamGeneration": "team_01J...",
  "lastUpdated": "2026-05-19T00:00:00.000Z",
  "sessions": [
    {
      "sessionKey": "sess_01J...",
      "sessionNumber": 1,
      "workspaceId": "workspace-id",
      "tabId": "tab-id",
      "currentSessionId": "pty-session-id",
      "role": "lead",
      "agentKind": "codex",
      "label": "Team Lead",
      "controllable": false,
      "keyVersion": 1,
      "createdAt": "2026-05-19T00:00:00.000Z",
      "updatedAt": "2026-05-19T00:00:00.000Z",
      "retiredAt": null
    }
  ],
  "assignments": [
    {
      "assignmentId": "asg_01J...",
      "leadSessionKey": "sess_01J...",
      "workerSessionKey": "sess_01J...",
      "status": "delivered",
      "promptHash": "sha256:...",
      "promptPreview": "safe redacted preview",
      "deliverySeq": 12,
      "createdAt": "2026-05-19T00:00:00.000Z",
      "updatedAt": "2026-05-19T00:00:00.000Z",
      "completion": {
        "required": true,
        "acceptedSignals": ["mcp.submit_assignment_result", "hook.assignment-complete", "hook.assignment-failed"],
        "terminalStatus": null,
        "terminalAt": null
      }
    }
  ]
}
```

별도 secret 파일:

```text
server/data/.buildergate/team-secrets.json
```

`completion.terminalStatus` 허용값은 `null | "completed" | "failed" | "cancelled" | "expired"`다. `assignment-complete`, `assignment-failed`, `submit_assignment_result`는 각각 `completed` 또는 `failed`로만 갱신할 수 있고, `cancelled`와 `expired`는 서버의 cancel/timeout 정책으로만 기록한다. `completion.terminalAt`은 assignment가 이 terminal status 중 하나로 처음 전환되는 시각이며, `turn-closed`, `idle`, `session:ready`만으로는 갱신하지 않는다.

여기에는 raw secret 대신 가능하면 다음 중 하나만 둔다.

- encrypted secret
- secret hash
- key id와 OS credential reference

`sessionKey` 자체를 secret으로 취급하면 감사 로그와 MCP result에 안전하게 노출하기 어렵다. 따라서 `sessionKey`는 식별자, 권한은 별도 capability로 분리한다.

## 5. MCP tool 설계

### Read-only tools

| Tool | 입력 | 출력 |
|---|---|---|
| `list_team_sessions` | optional role/filter | sessionKey, number, role, label, current status |
| `get_team_session` | `sessionKey` | workspace/tab/currentSessionId/role/agentKind/capability summary |
| `get_assignment_status` | `assignmentId` | state, last signal, completion policy |
| `read_assignment_audit` | `assignmentId`, cursor | redacted audit events |

### Write tools

| Tool | 입력 | 필수 정책 |
|---|---|---|
| `assign_session_role` | `sessionKey`, `role` | admin/lead scope, audit |
| `create_assignment` | lead/worker keys, title, prompt | prompt hash/preview, no PTY write |
| `preview_agent_prompt` | `assignmentId` | safe transform only |
| `deliver_agent_prompt` | `assignmentId`, `targetSessionKey`, `mode`, `submit` | approval, newline safety, SessionInputService |
| `request_worker_attention` | `targetSessionKey`, reason | no Enter by default |
| `cancel_assignment` | `assignmentId`, reason | active scope close, audit |
| `submit_assignment_result` | `assignmentId`, status, summary, evidence refs | worker scope or signed server-side actor |

`deliver_agent_prompt` 결과는 다음 필드를 포함해야 한다.

```ts
interface DeliverAgentPromptResult {
  accepted: boolean;
  assignmentId: string;
  targetSessionKey: string;
  resolvedSessionId: string;
  deliverySeq: number;
  mode: 'paste' | 'submit';
  inputAccepted?: boolean;
  rejectionReason?: string;
  auditId: string;
}
```

MCP spec 측면에서는 stdio와 Streamable HTTP가 모두 가능하다. 2025-06-18 MCP transport spec은 JSON-RPC 메시지가 UTF-8이어야 하고 표준 transport로 stdio와 Streamable HTTP를 정의한다. 로컬 Codex/IDE는 stdio 또는 localhost Streamable HTTP가 현실적이고, remote client는 인증된 HTTPS Streamable HTTP가 맞다. Streamable HTTP는 Origin 검증, localhost bind, 인증이 필수 보안 조건이다.

## 6. Hook 이벤트 설계

기존 hook 연구의 `SessionActivitySignal`에 assignment 필드를 추가한다.

```ts
type TeamHookEvent =
  | 'assignment-received'
  | 'assignment-start'
  | 'assignment-progress'
  | 'assignment-complete'
  | 'assignment-failed'
  | 'assignment-blocked'
  | 'tool-start'
  | 'tool-stop'
  | 'subagent-start'
  | 'subagent-stop'
  | 'turn-closed'
  | 'permission-prompt'
  | 'hook-error';

interface TeamHookSignal {
  version: 1;
  sessionKey: string;
  currentSessionId?: string;
  assignmentId?: string;
  event: TeamHookEvent;
  status?: 'received' | 'running' | 'completed' | 'failed' | 'blocked';
  sequence: number;
  timestamp: string;
  nonce: string;
  ttlMs?: number;
  source: 'codex-hook' | 'claude-code-hook' | 'hermes-hook' | 'buildergate-hook-cli';
  summaryHash?: string;
  summaryPreview?: string;
  metadata?: {
    scopeId?: string;
    parentScopeId?: string;
    toolCallId?: string;
    subagentId?: string;
    sourceSessionId?: string;
    sourceTurnId?: string;
    toolName?: string;
    stopReason?: string;
  };
}
```

서명은 `TeamHookSignal` 본문 필드가 아니라 canonical JSON body와 HTTP header 또는 CLI envelope로 전달한다.

```ts
interface SignedTeamHookEnvelope {
  body: TeamHookSignal;
  headers: {
    'X-BuilderGate-Session-Key': string;
    'X-BuilderGate-Timestamp': string;
    'X-BuilderGate-Nonce': string;
    'X-BuilderGate-Signature': string;
  };
}
```

수신 정책:

- HMAC, timestamp, nonce, sequence를 검증한다.
- `sessionKey`가 현재 retired 상태면 reject한다.
- `currentSessionId`가 있으면 registry의 current id와 일치해야 한다. 불일치하면 stale로 기록한다.
- `tool-start`, `tool-stop`, `subagent-start`, `subagent-stop`은 `metadata.scopeId`를 우선 key로 active running scope를 열고 닫는다. `scopeId`가 없으면 `sourceTurnId/toolCallId/subagentId` 조합으로 보수적으로 보정하고, 그래도 상관 ID가 없으면 low-confidence debug signal로 낮춘다.
- `turn-closed`는 assignment 완료가 아니다. `assignment-complete`, `assignment-failed`, 또는 MCP `submit_assignment_result`만 완료 후보가 된다.
- `assignment-complete`와 `assignment-failed`도 prompt 결과 전문을 저장하지 않고 hash, preview, evidence reference만 저장한다.

## 7. 완료판정 정책

### 금지

아래 조건만으로 assignment를 `done` 처리하면 안 된다.

- target session status가 `idle`
- PTY output quiet window가 지남
- `Stop` hook 수신
- 팀원 터미널에 "완료"로 보이는 문자열 출력
- WebSocket `session:ready`
- 브라우저 terminal snapshot 적용 완료

### 권장 상태기계

```text
draft
  -> approved
  -> delivered
  -> acknowledged
  -> running
  -> waiting_input | blocked | completing
  -> completed | failed | cancelled | expired
```

완료 승격 조건:

1. `assignmentId`가 일치한다.
2. signal source가 worker session capability로 검증된다.
3. sequence가 최신이다.
4. status가 `completed` 또는 `failed`처럼 terminal state다.
5. active running scope가 없거나, completion event가 해당 scope를 닫는다.
6. audit log write가 성공한다.

이 조건을 만족하면 assignment top-level `status`를 `completed` 또는 `failed`로 바꾸고, `.buildergate`의 `completion.terminalStatus`와 `completion.terminalAt`을 같은 transaction에서 기록한다. `cancelled`와 `expired`도 terminal state지만 명시적 취소/만료 정책에서만 기록하며, hook completion signal이 직접 만들 수 있는 값은 아니다.

`idle`은 보조 확인 신호로만 사용한다. 예를 들어 completion signal이 온 뒤 `idleDelayMs` 또는 짧은 settle window 동안 추가 running scope가 생기지 않으면 UI에 `completed`를 안정적으로 표시할 수 있다. 반대로 idle이 먼저 오면 `awaiting_completion_signal` 또는 `stalled_idle_without_completion`으로 둔다.

### 대화 종료 신호

팀원 대화가 끝날 때는 다음 중 하나가 필요하다.

- MCP: `submit_assignment_result({ assignmentId, status, summary, evidenceRefs })`
- hook/CLI: `buildergate-hook assignment-complete --assignment-id ... --status completed`
- hook/CLI failure: `buildergate-hook assignment-failed --assignment-id ... --status failed`

Agent lifecycle `Stop` hook은 `turn-closed`로 저장한다. 이것은 "이번 turn이 끝났다"는 신호이며 "작업이 완료됐다"는 신호가 아니다.

## 8. 세션키 저장/권한 정책

1. `sessionKey`는 tab 생성 시 만들고 tab restart에서는 유지한다.
2. orphan recovery로 `sessionId`가 바뀌면 `currentSessionId`만 갱신한다.
3. tab 삭제 시 sessionKey는 즉시 retired 처리하고 일정 기간 재사용하지 않는다.
4. `sessionNumber`는 teamGeneration 내에서만 unique하게 유지하고, 삭제 번호 재사용은 최소한 한 generation 안에서는 금지한다.
5. MCP write tool은 `sessionNumber`만 받는 입력을 허용하더라도 내부적으로 `sessionKey`로 resolve한 뒤 audit에 둘 다 남긴다.
6. role 변경은 `assign_session_role`로만 하고, workspace/tab 이름 변경과 분리한다.
7. lead가 worker에게 보낼 수 있는 권한은 sessionKey allowlist와 assignment scope로 제한한다.
8. hook capability는 세션별 HMAC secret을 사용하고, 브라우저 JWT와 공유하지 않는다.
9. `.buildergate` JSON은 UTF-8, atomic write, `.bak`, owner-only 권한을 사용한다.
10. raw prompt, hook payload 전문, JWT, hook secret은 저장하지 않는다.

## 9. 위험과 완화

| 위험 | 영향 | 완화 |
|---|---|---|
| `sessionId`를 장기 target으로 사용 | restart 후 엉뚱한 세션에 전송 | `sessionKey` stable mapping |
| `sessionNumber`를 권한처럼 사용 | 번호 재사용/혼동으로 오전송 | number는 표시 alias, write는 key resolve 후 수행 |
| MCP가 `writeInput()` 직접 호출 | replay/input queue 우회 | `SessionInputService` 공용 ingress |
| Enter 포함 prompt timeout 후 flush | 사용자가 보지 못한 시점에 명령 실행 | Enter timeout safety 유지 |
| idle을 완료로 오판 | 작업 미완료를 done 처리 | explicit completion signal 필수 |
| Stop hook을 완료로 오판 | turn 종료와 작업 완료 혼동 | Stop은 `turn-closed`로만 저장 |
| hook secret 유출 | 완료/상태 위조 | HMAC, nonce, sequence, TTL, rate limit |
| raw prompt 저장 | 민감 정보 노출 | hash, byte length, safe preview |
| remote MCP auth 약함 | 외부 제어면 노출 | Bearer/OAuth, audience/scope/expiry, Origin 검증 |
| 팀장 prompt injection | worker가 권한 밖 tool 실행 | tool allowlist, approval gate, destructive 금지 |
| workspace activeTab/viewMode 공유 | 팀장 조작이 팀원 화면 UX 방해 | 세션 제어와 브라우저 presence 분리 |

## 10. 테스트 포인트

### 서버 단위

- `team-sessions.json` schema validation
- atomic write, backup recovery, corrupt file recovery
- tab restart 후 `sessionKey` 유지와 `currentSessionId` 갱신
- orphan recovery 후 stale `sessionId` reject
- retired key로 deliver 시 reject
- `sessionNumber` ambiguity reject
- role/scope 없는 `deliver_agent_prompt` reject
- prompt hash/preview redaction
- newline/Enter approval 누락 reject
- server-origin input이 replay pending 정책을 우회하지 않음
- `idle` 단독으로 assignment completed 전환 금지
- `turn-closed` 단독으로 completed 전환 금지
- MCP `submit_assignment_result`로 completed 전환
- hook HMAC 정상/변조/만료/replay/역순 sequence
- tool/subagent start-stop scopeId 불일치, 누락, 중복 close 처리
- subagent active 중 parent turn close가 전체 completed 처리하지 않음

### WebSocket/SessionManager 회귀

- 기존 `status` payload는 `idle | running` 유지
- MCP/hook이 `status`를 직접 broadcast하지 않음
- AI TUI 사용자 입력, local echo, prompt redraw, cursor movement, waiting repaint는 계속 idle 유지
- hook `tool-start`는 running scope, `tool-stop`/TTL은 scope close
- permission prompt는 `idle + attention hint`

### 입력 안정성

- server-origin delivery에도 UTF-8 byte budget 적용
- sequence 또는 server delivery sequence가 audit/debug에 남음
- replay ACK 전 output queue 순서 보존
- replay timeout에서 Enter 포함 delivery는 실행되지 않음
- hidden/parked terminal 대상은 서버 PTY 기준으로 처리하되, 실패는 observable result로 반환
- IME/paste/command preset 기존 회귀 유지

### E2E

모든 Playwright/수동 검증 대상은 `https://localhost:2002`다.

- 두 브라우저 창에서 lead/worker session 지정
- lead MCP `create_assignment -> deliver_agent_prompt`
- worker terminal에서 prompt 수신
- worker MCP `submit_assignment_result`로 completed
- hook `assignment-complete` fallback으로 completed
- worker가 idle이 되어도 completion signal 없으면 `awaiting_completion_signal`
- target tab restart 후 같은 `sessionKey`로 새 `sessionId`에 deliver
- stale key/retired key/권한 부족/approval 없음 reject와 audit 확인
- reconnect/snapshot replay 중 prompt 중복/유실 없음

## 12. 구현 전 의사결정

구현에 들어가기 전 아래를 먼저 SRS로 고정해야 한다.

1. `.buildergate` JSON의 실제 위치: project root가 아니라 runtime data root 아래로 둘지 결정.
2. `sessionKey`를 `WorkspaceTab`에 넣을지, 별도 `TeamSessionRegistry`에 둘지 결정.
3. `deliver_agent_prompt` 기본 동작을 paste-only로 할지 submit까지 허용할지 결정.
4. worker completion은 MCP tool을 필수로 하고 hook을 fallback으로 둘지, 둘 다 동등하게 둘지 결정.
5. remote MCP를 1차 범위에 포함할지, local MCP만 먼저 할지 결정.
6. approval gate를 UI 기반으로 둘지, config policy allowlist 기반으로 둘지 결정.

권장 1차 범위는 local MCP + `.buildergate` session registry + read-only tools + `create_assignment` + paste-only `deliver_agent_prompt` + MCP `submit_assignment_result`다. Enter 자동 제출과 remote MCP는 보안/테스트 면적이 커서 2차로 미루는 편이 안전하다.
