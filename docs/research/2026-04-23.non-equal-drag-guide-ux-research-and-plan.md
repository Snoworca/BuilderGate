## 서브에이전트 결론 요약

## Halley: UX 관점

- 가장 싼 개선은 방향 affordance 강화다.
- 하지만 최고의 UX는 모드와 상관없이 drag를 slot move mental model로 통일하는 쪽이다.
- 다만 그 경로는 scope가 커지고 `focus/auto`의 post-drop 규칙을 다시 정의해야 한다.

## Hegel: 구현 구조 관점

- `layoutMode`와 mode별 tree 후처리는 앱이 이미 소유하고 있다.
  - [MosaicContainer.tsx](C:\Work\git\_Snoworca\ProjectMaster\frontend\src\components\Grid\MosaicContainer.tsx:163)
  - [mosaic.ts](C:\Work\git\_Snoworca\ProjectMaster\frontend\src\utils\mosaic.ts:292)
- vendor는 `reorderEnabled`만 알아서 사실상 `equal` vs `non-equal`만 구분한다.
  - [Mosaic.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\Mosaic.js:121)
- app-only CSS 강화는 가능하지만, active hover metadata를 React에서 안정적으로 쓰려면 thin vendor patch가 유리하다.
- 추천 구조는 “hover metadata 노출만 vendor patch, 가이드/색/카피/모드 후처리는 앱”이다.

## Boole: QA 관점

- 이번 범위에서 non-equal guide를 full-cell reorder처럼 보이게 만들면 안 된다. 시각과 의미가 다시 어긋난다.
- root edge drop zone도 유효 경로이므로 반드시 같이 시각화해야 한다.
- `focus/auto`는 drop 후 실제 레이아웃 재적용 규칙을 먼저 문서화하지 않으면 검증이 불가능하다.
- Playwright 기준으로 hover geometry, split commit 결과, equal 회귀까지 같이 잡아야 한다.

## 종합 판단

## 최종 UX 목표

가장 좋은 장기 UX는 `equal`과 non-equal을 모두 slot-based drag semantics로 통일하는 것이다.

이유:

- 사용자는 “어느 변으로 split하느냐”보다 “이 창이 어디로 가느냐”를 기대한다.
- 이미 `equal`에서는 full-cell reorder UX가 더 낫다는 것이 코드와 사용자 피드백으로 검증됐다.

## 현재 시점의 추천 구현 방향

하지만 이번 요구는 “non-equal에서 어디에 놓일지 미리 보이게 해 달라”는 문제를 푸는 것이고, 이를 가장 안전하게 해결하는 1차 구현은 아래다.

**추천안: split semantics는 유지하고, 최종 split 결과를 더 정확히 보여주는 predictive split overlay를 도입한다.**

즉:

- `none/focus/auto`는 계속 split drop이다.
- 대신 drag 중에는
  - 현재 hover target의 방향
  - 실제 drop 후 차지할 영역
  - root edge drop zone
를 명확하게 보여준다.

이 추천안을 고른 이유:

1. 사용자의 즉시 pain point를 바로 해결한다.
2. `equal`의 full-cell reorder UX와 충돌하지 않는다.
3. `focus/auto` semantics를 갈아엎지 않아도 된다.
4. thin vendor patch + app overlay 조합으로 구현 가능하다.

## 옵션 비교

### 옵션 A: CSS만 강화

- 내용:
  - 기존 `top/bottom/left/right` target의 색/불투명도/라벨만 키운다.
- 장점:
  - 가장 빠르다.
- 단점:
  - 현재처럼 hover 의미와 실제 결과가 계속 어긋날 수 있다.
  - root edge path와 cell 내부 path를 일관되게 다루기 어렵다.
  - DOM/class 의존성이 커진다.

### 옵션 B: predictive split overlay

- 내용:
  - split semantics는 유지하되, drag 중 실제 drop 결과 영역을 overlay로 직접 그린다.
- 장점:
  - 현재 요구를 가장 직접적으로 해결한다.
  - `equal` full-cell guide와 공존 가능하다.
  - `focus/auto`는 mode 후처리만 따로 다루면 된다.
- 단점:
  - hover metadata를 앱으로 끌어올 얇은 vendor patch가 필요하다.

### 옵션 C: 모든 모드 slot-reorder 통일

- 내용:
  - non-equal도 full-cell reorder UX로 바꾼다.
- 장점:
  - 최종적으로는 가장 좋은 UX다.
- 단점:
  - 이번 요구 범위를 넘어선다.
  - `none/focus/auto`의 drop 후 의미를 다시 설계해야 한다.
  - 회귀 범위가 커진다.

## 추천 아키텍처

### 원칙

- vendor patch는 최소화한다.
- app이 의미를 소유한다.
- `equal`과 non-equal guide selector를 명시적으로 분리한다.

### 권장 구조

1. `react-mosaic` thin patch
- active hover metadata를 앱이 읽을 수 있게 한다.
- 필요한 최소 정보:
  - `path`
  - `position`
  - root target 여부

2. 앱 레벨 drag guide state
- [MosaicContainer.tsx](C:\Work\git\_Snoworca\ProjectMaster\frontend\src\components\Grid\MosaicContainer.tsx:385) 부근에 non-equal drag guide state를 둔다.
- drag 중 active target metadata를 받아:
  - hovered cell rect
  - 방향별 final split rect
  - root edge rect
를 계산한다.

3. 앱 레벨 overlay 렌더
- `equal`에서는 기존 full-cell reorder guide 유지
- non-equal에서는 custom predictive split overlay 사용
- 둘은 selector와 data-attr를 명시적으로 분리

4. drop 후 mode 후처리
- `none`: 기본 split 결과 유지
- `focus`: split 결과 후 `applyFocusMode()` 재적용 여부를 명시적으로 고정
- `auto`: split 결과 후 `applyMultiFocusApprox()` 및 status-driven 재적용 계약을 고정

## 구현 전 고정해야 할 결정

### 1. 이번 범위의 semantics

이번 범위는 아래로 잠그는 것이 맞다.

- `equal`: reorder semantics 유지
- `none/focus/auto`: split semantics 유지

### 2. `focus/auto` post-drop invariant

구현 전에 아래를 문서로 못 박아야 한다.

- `focus`:
  - split 후 focus ratio를 즉시 다시 적용할지
  - 혹은 split 결과를 우선 보여줄지
- `auto`:
  - drag 중 status change가 오면 guide를 유지할지
  - cancel할지
  - drop 직후 auto rebalance를 언제 다시 적용할지

### 3. root edge guide 정책

- root edge drop zone도 반드시 시각화해야 한다.
- 셀 내부 guide와 다른 형태로 보여줘야 혼동이 줄어든다.

## 단계별 실행 계획

## Phase 1: scope와 계약 고정

- non-equal은 이번 범위에서 reorder로 확장하지 않는다.
- `focus/auto` post-drop invariant를 문서화한다.
- equal guide와 non-equal guide의 시각 의미를 분리한다.

## Phase 2: thin vendor patch

- `MosaicDropTarget`/`MosaicWindow`/`RootDropTargets` 쪽에서 active drop metadata를 앱이 읽을 수 있게 최소 patch를 넣는다.
- 여기서는 semantics를 바꾸지 않는다.

## Phase 3: app overlay 구현

- `MosaicContainer`에 non-equal predictive split overlay를 추가한다.
- 방향별 final rect를 계산한다.
- root edge overlay도 같이 렌더한다.

## Phase 4: mode-aware post-drop 정렬

- `focus/auto`의 actual post-drop behavior를 코드와 문서에 일치시킨다.
- overlay가 보여준 결과와 commit 후 layout이 일치해야 한다.

## Phase 5: 회귀 검증

- Playwright:
  - non-equal hover geometry
  - valid split drop result
  - outside/self/right-click/non-primary no-op
  - root edge guide
  - equal full-cell guide 회귀
- 수동:
  - `https://localhost:2002`
  - 각 모드별 top/right/bottom/left hover
  - root edge 시각 분리
  - focus/auto post-drop 체감

## 최종 결론

- **장기적으로 최고의 UX**는 모든 모드의 drag semantics를 slot move로 통일하는 것이다.
- **현재 요구를 해결하는 최적의 실행 계획**은 non-equal에서 split semantics는 유지하되, 최종 split 결과를 예측해서 보여주는 predictive split overlay를 도입하는 것이다.
- 구현 구조는 **thin vendor patch + app-owned overlay + 강한 E2E 회귀**가 가장 현실적이다.
