# BuilderGate TUI keyboard compatibility 연구

> **프로젝트**: BuilderGate / ProjectMaster  
> **연구 주제**: 브라우저 기반 xterm.js 터미널에서 Codex, Claude, Hermes 등 다양한 TUI 도구의 단축키를 안정적으로 전달하는 방법  
> **작성일**: 2026-05-09  
> **결론**: 단축키를 툴별로 하나씩 패치하는 방식은 장기적으로 취약하다. BuilderGate에는 `TerminalView` 앞단의 키보드 호환 계층과 사용자 등록형 shortcut capture modal이 필요하다. 단, 브라우저나 OS가 선점해 애플리케이션까지 도달하지 않는 키는 모달만으로 해결할 수 없으므로 fullscreen keyboard capture 또는 대체키 안내가 필요하다.

---

## 1. 배경

BuilderGate는 로컬 터미널이 아니라 다음 경로의 웹 터미널이다.

```text
Browser / WebView
  -> xterm.js
  -> WebSocket
  -> node-pty / ConPTY
  -> shell 또는 nested TUI app
```

이 구조에서는 사용자가 누른 키가 항상 TUI 애플리케이션까지 같은 의미로 전달되지 않는다.

최근 확인된 예시는 Codex TUI의 줄바꿈 단축키다.

- 로컬 터미널의 Codex에서는 `Ctrl+J`가 줄바꿈 입력으로 쓰일 수 있다.
- BuilderGate에서는 `Shift+Enter`가 xterm.js 기본 처리로 일반 Enter와 같은 `CR`(`\r`)로 접힌다.
- `Ctrl+J`는 xterm까지 도달하면 `LF`(`\n`)로 전달되지만, 실제 브라우저 또는 WebView가 단축키를 선점하면 BuilderGate까지 도달하지 않을 수 있다.

따라서 이 문제는 "Codex 전용 단축키 하나가 안 됨"이 아니라, 앞으로 사용될 여러 TUI 도구의 수정키 포함 단축키를 어떻게 다룰지에 대한 터미널 호환성 문제다.

## 2. 현재 코드 관찰

### 2.1 `TerminalView` 입력 처리

현재 키보드 입력의 핵심 경로는 다음 파일에 있다.

- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/contexts/WebSocketContext.tsx`
- `server/src/services/SessionManager.ts`

`TerminalView`는 `term.attachCustomKeyEventHandler()`를 사용한다.

현재 특별 취급되는 입력은 주로 다음이다.

- 선택 영역이 있는 `Ctrl+C`: 클립보드 복사
- `Ctrl+V`: xterm paste 이벤트와 중복되지 않도록 차단
- IME 조합 중 `Space`, `Backspace`, `Enter` 등: xterm 네이티브 경로 위임
- plain `Space`, `Backspace`: xterm 네이티브 경로 위임 및 debug capture

반면 `Ctrl+J`, `Shift+Enter`, `Ctrl+Enter`, `Alt+Enter` 같은 TUI별 특수 단축키는 명시적으로 처리하지 않는다. 대부분 xterm.js 기본 처리에 맡긴다.

### 2.2 xterm.js Enter 처리

현재 설치된 `@xterm/xterm` 6.0.0의 기본 키보드 매핑은 `Enter` keyCode 13을 다음처럼 처리한다.

```text
Enter -> CR
Alt+Enter -> ESC + CR
Shift+Enter -> CR
Ctrl+Enter -> CR
```

즉 `Shift+Enter`가 "줄바꿈"이나 "수정키 포함 Enter"로 보존되지 않는다.

### 2.3 실제 실행 관찰

BuilderGate debug capture를 켜고 Playwright synthetic key event로 확인한 결과는 다음과 같다.

```text
Control+J    -> xterm_data_emitted preview "\n"
Shift+Enter  -> xterm_data_emitted preview "\r"
```

이 결과는 다음을 뜻한다.

1. `Ctrl+J`가 앱까지 도달하면 BuilderGate는 `LF`를 PTY로 보낼 수 있다.
2. `Shift+Enter`는 현재 코드와 xterm 기본값 때문에 `CR`로 접히며, Codex 줄바꿈으로 동작하지 않는다.
3. Playwright synthetic key event는 브라우저 chrome-level 단축키 선점을 완전히 재현하지 못한다. 실제 사용자 환경에서 `Ctrl+J`가 브라우저 다운로드 shortcut 등으로 선점될 수 있는 문제는 별도 수동 검증이 필요하다.

## 3. 문제 분류

### 3.1 브라우저 또는 OS가 선점하는 키

예시:

- `Ctrl+L`
- `Ctrl+R`
- `Ctrl+W`
- `Ctrl+N`
- `Ctrl+T`
- `Alt+F4`
- 일부 환경의 `Ctrl+J`

이 키들은 웹 애플리케이션의 `keydown`까지 도달하지 않을 수 있다. 이 경우 BuilderGate의 JavaScript 코드로는 일반 모드에서 해결할 수 없다.

가능한 대응:

- fullscreen terminal mode 제공
- 지원 브라우저에서 Keyboard Lock API 시도
- desktop shell 또는 WebView host에서 accelerator 선점 완화
- 모달에서 "감지되지 않은 단축키"로 명시하고 대체키 등록 유도

### 3.2 xterm.js 기본 매핑이 수정키를 잃는 키

예시:

- `Shift+Enter`
- `Ctrl+Enter`
- `Alt+Enter`
- 일부 modified function key 또는 navigation key

이 키들은 브라우저 이벤트는 도달하지만 xterm.js 기본 변환 과정에서 TUI가 기대하는 의미와 달라질 수 있다.

가능한 대응:

- `TerminalView`에서 사용자 매핑을 먼저 적용
- 매핑이 없으면 xterm.js 기본 처리로 fallback
- AI TUI 호환 profile 또는 per-workspace/per-session profile 제공

### 3.3 입력 경로와 focus가 어긋나는 키

BuilderGate grid 모드에서는 여러 xterm helper textarea가 동시에 존재한다. 사용자가 보고 있는 pane과 실제 focus owner가 어긋나면 키가 다른 터미널로 들어가거나 입력이 무시된 것처럼 보일 수 있다.

가능한 대응:

- active terminal focus 상태를 debug capture와 UI 상태에 명확히 노출
- shortcut capture modal에서 현재 target session을 표시
- 키 등록 및 테스트 시 "어느 session으로 전송되었는지"를 보여준다.

### 3.4 IME와 paste

IME 조합, 한글 입력, paste는 일반 keydown 기반 처리와 충돌하기 쉽다.

현재 프로젝트에는 이미 다음 계열 회귀 테스트가 있다.

- `frontend/tests/e2e/terminal-keyboard-regression.spec.ts`
- `frontend/tests/e2e/terminal-korean-ime.spec.ts`
- `frontend/tests/e2e/terminal-paste.spec.ts`

새 keyboard compatibility layer는 이 경로를 깨지 않아야 한다.

## 4. 권장 접근: 키보드 호환 계층

단축키를 개별 TUI별로 하드코딩하지 말고, 다음 순서의 입력 처리 계층을 둔다.

```text
KeyboardEvent
  -> browser delivery check
  -> IME/composition guard
  -> user shortcut mapping
  -> built-in compatibility profile
  -> xterm.js native handling
  -> TerminalView onData
  -> TerminalContainer input sequencer
  -> WebSocket input
  -> SessionManager.writeInput()
```

핵심 원칙:

1. 기존 xterm 기본 동작을 기본값으로 유지한다.
2. 사용자가 등록한 mapping이 있으면 xterm 기본 처리보다 먼저 적용한다.
3. mapping 적용 시 `preventDefault()` 및 `return false`로 중복 전송을 막는다.
4. IME 조합 중에는 일반 shortcut mapping을 신중하게 제한한다.
5. mapping이 실패하거나 브라우저가 키를 전달하지 않으면 관측 가능한 오류/안내를 남긴다.

## 5. Shortcut capture modal 설계

### 5.1 목적

사용자가 앞으로 만나게 될 알 수 없는 TUI 단축키를 BuilderGate UI에서 직접 등록하도록 한다.

예시 사용 흐름:

1. 사용자가 "단축키 추가"를 연다.
2. 모달이 키 입력 대기 상태가 된다.
3. 사용자가 `Shift+Enter` 또는 `Ctrl+J`를 누른다.
4. BuilderGate가 실제 수신한 `KeyboardEvent`를 보여준다.
5. 사용자가 해당 키를 어떤 PTY 입력으로 보낼지 선택한다.
6. 저장 후 현재 session 또는 workspace에 적용한다.

### 5.2 모달에 표시할 정보

키가 감지된 경우:

```text
입력 감지됨
- key: Enter
- code: Enter
- modifiers: Shift
- location: Standard
- repeat: false
- browserDelivered: true
```

키가 감지되지 않은 경우:

```text
입력 감지 안 됨
이 단축키는 브라우저 또는 OS가 선점했을 수 있습니다.
fullscreen keyboard capture를 켜거나 다른 키를 등록하세요.
```

### 5.3 사용자가 선택할 action

최소 action:

| Action | 전송 데이터 | 용도 |
|---|---:|---|
| Send LF | `\n` | Codex newline, 일부 TUI line editor |
| Send CR | `\r` | 일반 Enter |
| Send ESC | `\x1b` | Escape |
| Send ESC+CR | `\x1b\r` | Alt+Enter legacy fallback |
| Send text/sequence | 사용자 입력 | 고급 escape sequence |
| Pass through | 없음 | xterm 기본 처리 |
| Block | 없음 | 브라우저 또는 xterm 전달 차단 |

고급 action:

- CSI-u sequence 전송
- Kitty keyboard protocol sequence 전송
- per-app preset 적용

단, 고급 프로토콜은 대상 TUI가 지원해야 의미가 있다. 기본 구현은 `LF`, `CR`, `ESC`, custom sequence 정도로 시작하는 것이 안전하다.

### 5.4 scope

mapping scope는 최소 다음을 지원할 수 있다.

| Scope | 의미 |
|---|---|
| Session | 현재 터미널 session에만 적용 |
| Workspace | 현재 workspace의 모든 터미널에 적용 |
| Global | 전체 BuilderGate에 적용 |
| Profile | `xterm-default`, `ai-tui-compat`, `custom` 등 profile 단위 |

초기 구현은 `Workspace` 또는 `Global` 중 하나로 단순화해도 된다.

## 6. 데이터 모델 초안

```ts
type ShortcutAction =
  | { type: 'send'; data: string; label?: string }
  | { type: 'pass-through' }
  | { type: 'block' };

interface TerminalShortcutBinding {
  id: string;
  scope: 'session' | 'workspace' | 'global';
  sessionId?: string;
  workspaceId?: string;
  profile?: string;
  key: string;
  code: string;
  ctrlKey: boolean;
  shiftKey: boolean;
  altKey: boolean;
  metaKey: boolean;
  location: number;
  action: ShortcutAction;
  enabled: boolean;
  description?: string;
  createdAt: string;
  updatedAt: string;
}
```

매칭에는 `event.key`만 쓰면 keyboard layout에 따라 흔들릴 수 있다. 권장 매칭 우선순위는 다음이다.

1. `code + modifiers + location`
2. 필요 시 `key + modifiers` fallback
3. IME/composition 중에는 mapping skip 또는 제한

## 7. 기본 preset 권장안

### 7.1 `xterm-default`

현재 xterm.js 기본 동작을 유지한다.

```text
Enter       -> xterm default CR
Shift+Enter -> xterm default CR
Ctrl+J      -> xterm default LF if delivered
```

### 7.2 `ai-tui-compat`

Codex, Claude, Hermes 등 line editor 성격의 TUI에서 우선 검토할 profile이다.

초기 권장:

```text
Shift+Enter -> LF
Ctrl+J      -> LF
Enter       -> CR
Ctrl+C      -> selection exists: copy, otherwise xterm default
Ctrl+V      -> paste event path only
```

주의:

- `Ctrl+J`가 브라우저에 선점되면 profile로 해결되지 않는다.
- `Ctrl+Enter`, `Alt+Enter`는 TUI별 의미가 다를 수 있으므로 기본 활성화보다 사용자 등록형으로 두는 편이 안전하다.

## 8. 구현 위치

### 8.1 프런트엔드

주요 변경 후보:

- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/Settings/SettingsPage.tsx`
- `frontend/src/types/settings.ts`
- `frontend/src/services/api.ts`

- 신규 후보: `frontend/src/components/Terminal/ShortcutCaptureModal.tsx`

`TerminalView`의 `attachCustomKeyEventHandler()` 초입에서 다음을 수행한다.

```text
1. keydown만 처리
2. IME active면 기존 IME guard 우선
3. 사용자 shortcut binding 조회
4. 매칭되면 action 수행
5. action이 send면 submitCapturedInput(mappedData, ...)
6. 중복 방지를 위해 preventDefault + return false
7. 매칭 없으면 기존 로직 유지
```

### 8.2 서버

초기 구현에서 서버 변경은 필수는 아니다. 기존 입력 전송 경로는 이미 arbitrary string을 PTY로 보낼 수 있다.

다만 전역 설정으로 저장하려면 다음 계층과 연결할 수 있다.

- `server/src/services/SettingsService.ts`
- `server/src/types/settings.types.ts`
- `server/src/schemas/config.schema.ts`
- `server/config.json5`

초기에는 localStorage 기반 client setting으로 시작하고, 안정화 후 서버 설정으로 승격할 수 있다.

## 9. 테스트 전략

### 9.1 단위 테스트

신규 matcher 유틸에 대해 다음을 검증한다.

- `code + modifiers` 매칭
- `key + modifiers` fallback
- scope 우선순위: session > workspace > global
- disabled binding 무시
- IME active일 때 skip 정책
- custom sequence escape 처리

### 9.2 Playwright E2E

기존 `terminal-keyboard-regression.spec.ts`에 추가 또는 별도 spec 생성:

```text
TC-KBD-MAP-01: Shift+Enter binding -> LF 전송
TC-KBD-MAP-02: plain Enter는 기존 CR 유지
TC-KBD-MAP-03: Ctrl+J binding -> LF 전송
TC-KBD-MAP-04: mapping 적용 시 xterm default와 중복 전송 없음
TC-KBD-MAP-05: binding disabled 시 xterm default로 fallback
TC-KBD-MAP-06: IME composition 중 mapping이 기존 IME guard를 깨지 않음
TC-KBD-MAP-07: modal capture가 key/code/modifiers/location을 저장
```

### 9.3 수동 검증

Playwright synthetic keyboard event는 브라우저 chrome-level shortcut 선점을 완전히 재현하지 못한다.

따라서 다음은 수동 검증이 필요하다.

- 실제 Chrome/Edge에서 `Ctrl+J`
- WebView packaged app에서 `Ctrl+J`
- fullscreen keyboard capture 사용 시 reserved shortcut 개선 여부
- Windows IME 환경에서 조합 중 shortcut 동작
- Codex, Claude, Hermes 각각의 multiline 입력

## 10. 리스크와 완화

| 리스크 | 설명 | 완화 |
|---|---|---|
| 브라우저 shortcut 선점 | 앱까지 event가 도달하지 않음 | 감지 실패 UI, fullscreen keyboard capture, 대체키 안내 |
| 중복 입력 | custom handler와 xterm native path가 둘 다 전송 | mapping 적용 시 `return false`, debug capture로 검증 |
| IME 회귀 | 한글 조합 중 keydown mapping이 조합을 깨뜨림 | IME guard 우선, IME E2E 유지 |
| 보안/오작동 | 사용자가 위험한 sequence를 global로 등록 | scope 제한, 고급 sequence 경고, disable/reset 제공 |
| TUI별 의미 충돌 | 같은 shortcut이 도구마다 다름 | profile/scope 제공, global 기본값 최소화 |
| 테스트 착시 | Playwright는 실제 브라우저 단축키 선점 미재현 | 수동 validation matrix 별도 유지 |

## 11. 권장 구현 순서

1. `terminalShortcutBindings` 유틸과 단위 테스트를 추가한다.
2. `TerminalView`에 mapping 적용 계층을 넣되 기본 mapping은 비활성으로 둔다.
3. `Shift+Enter -> LF`를 workspace 또는 profile 기반 opt-in으로 검증한다.
4. shortcut capture modal을 추가한다.
5. debug capture에 `shortcut_binding_matched`, `shortcut_binding_sent`, `shortcut_binding_skipped` 이벤트를 추가한다.
6. Playwright E2E로 LF/CR/중복 전송/IME guard를 고정한다.
7. fullscreen keyboard capture 또는 desktop host accelerator 조정은 별도 phase로 분리한다.

---

## 12. 결론

앞으로 여러 TUI 도구를 사용할 계획이라면, BuilderGate는 단축키를 개별 도구 기준으로 계속 패치하면 안 된다.

가장 안전한 방향은 다음 조합이다.

```text
1. xterm 기본 처리 유지
2. 사용자 등록형 shortcut capture modal
3. AI TUI compatibility profile
4. browser-reserved shortcut 감지 실패 안내
5. debug capture 기반 키 입력 매트릭스 테스트
```

이 접근은 현재 Codex의 `Ctrl+J`/`Shift+Enter` 문제를 해결하면서도, 미래에 등장할 TUI 단축키를 사용자가 직접 관측하고 매핑할 수 있는 확장 경로를 제공한다.

## 13. 기존 모달 기반 UI/UX 구성 연구

### 13.1 기존 모달 컴포넌트 판단

현재 코드베이스에는 크게 세 종류의 모달/다이얼로그 구현이 있다.

| 구현 | 위치 | 성격 | Shortcut manager 적합도 |
|---|---|---|---|
| `ConfirmModal`, `RenameModal`, `ShellSelectModal` | `frontend/src/components/Modal/*` | 단순 confirm, 단일 입력, 단일 선택 | 낮음. focus trap, portal, resize, 복합 layout 지원이 부족하다. |
| `FileOperationDialog` | `frontend/src/components/FileManager/FileOperationDialog.tsx` | mdir 패널 전용의 터미널풍 작업 모달 | 낮음. 파일 관리자 UX에 특화되어 있고 전체 앱 관리 UI와 시각 언어가 다르다. |
| `WindowDialog`, `MessageBox` | `frontend/src/components/dialog/*` | portal 기반 window-style dialog, modal/modeless, focus trap, inert, resize/drag, geometry persistence | 높음. shortcut capture와 binding 관리처럼 상태가 많은 도구형 UI에 가장 적합하다. |

단축키 등록 UI는 사용자가 "키 입력 감지", "전송 action 선택", "scope 결정", "저장된 mapping 관리", "테스트 전송"을 한 화면에서 반복해야 한다. 따라서 단순 `ConfirmModal` 계열보다 `WindowDialog`를 재사용해야 한다.

기준 구현은 `CommandPresetDialog`가 가장 가깝다.

- `WindowDialog mode="modal"`을 사용한다.
- tablist, form, list, inline edit, delete confirm을 한 dialog 안에 담는다.
- 삭제 확인은 `MessageBox`를 별도로 띄운다.
- dialog 내부는 어두운 운영 도구형 UI이며 터미널/설정 화면과 시각적으로 맞다.

### 13.2 추천 컴포넌트 구조

신규 컴포넌트는 다음처럼 분리한다.

```text
frontend/src/components/TerminalShortcutManager/
  TerminalShortcutDialog.tsx
  TerminalShortcutDialog.css
  ShortcutCapturePanel.tsx
  ShortcutBindingList.tsx
  ShortcutActionEditor.tsx
  shortcutBindingViewModel.ts
```

외부 wrapper:

```tsx
<WindowDialog
  dialogId="terminal-shortcut-manager"
  title="터미널 키보드"
  mode="modal"
  defaultRect={{ x: 120, y: 72, width: 820, height: 620 }}
  minSize={{ width: 620, height: 460 }}
  onClose={onClose}
  persistGeometry
>
  <TerminalShortcutDialogContent />
</WindowDialog>
```

권장 이유:

1. shortcut 관리는 단발성 confirm이 아니라 반복 관리 도구다.
2. 사용자는 TUI를 보면서 dialog 위치와 크기를 조정하고 싶을 수 있다.
3. key capture는 focus 제어가 핵심이므로 `WindowDialog`의 modal focus trap과 inert 처리가 필요하다.
4. 삭제, reset, dangerous custom sequence 확인은 `MessageBox`를 그대로 재사용할 수 있다.

### 13.3 정보 구조

Dialog는 3개 탭으로 구성하는 것이 가장 안정적이다.

| 탭 | 목적 | 주요 UI |
|---|---|---|
| Profiles | 기본 동작 선택 | `xterm default`, `AI TUI compatibility`, `Custom` segmented/tabs |
| Bindings | 저장된 mapping 관리 | list, enable toggle, scope badge, edit/delete buttons |
| Capture | 새 shortcut 등록 | capture target, detected event, action selector, save button |

`CommandPresetDialog`의 tablist 시각 패턴은 재사용할 수 있다. 단, 접근성 동작까지 그대로 충분하다고 보면 안 된다. Shortcut manager는 capture 상태와 설정 상태가 계속 바뀌므로, 구현 시에는 WAI-ARIA tabs keyboard behavior를 보강해야 한다.

필수 보강:

- 활성 tab만 `tabIndex=0`, 나머지는 `tabIndex=-1`로 두는 roving tabindex
- `ArrowLeft`/`ArrowRight`로 tab 이동
- `Home`/`End`로 첫/마지막 tab 이동
- 각 tab에 `aria-controls`
- 각 panel에 `role="tabpanel"`과 `aria-labelledby`

Shortcut dialog는 사용자가 현재 상태를 빠르게 스캔해야 하므로 각 탭의 내용은 card 중첩 없이 밀도 높은 list/form layout으로 구성한다.

### 13.4 Capture 탭 UX

Capture 탭은 "키를 누르면 바로 등록"이 아니라 "감지 -> 해석 -> action 선택 -> 저장" 흐름이어야 한다.

권장 화면 배치:

```text
[Target]
  Scope: Global / Workspace / Session
  Profile: AI TUI compatibility / Custom
  Target session: Terminal-3, Codex running

[Capture]
  상태: Waiting for shortcut...
  감지 결과:
    Key       Enter
    Code      Enter
    Modifiers Shift
    Location  Standard
    Delivered Yes

[Action]
  Send LF (\n)
  Send CR (\r)
  Send ESC
  Send ESC+CR
  Custom sequence
  Pass through xterm
  Block

[Preview]
  Shift+Enter -> LF (\n)
  중복 전송 방지: enabled

[Actions]
  Test in current terminal
  Save binding
  Cancel
```

중요 UX 원칙:

- 모달이 열렸다는 이유만으로 터미널에 키를 보내면 안 된다.
- capture 대기 상태에서는 모달이 keydown을 소비하고, 저장 또는 테스트 시에만 PTY로 전송한다.
- `Escape`는 기존 `WindowDialog`가 기본적으로 close를 직접 수행하지 않고 document capture 단계에서 전파를 막는다. capture 중 `Escape`는 "Esc 키 등록"과 "dialog 닫기"가 충돌하므로 명확한 모드가 필요하다.
- 권장 정책은 capture 중 첫 `Escape`를 shortcut 후보로 기록하고, 닫기는 close button 또는 Cancel button으로만 처리하는 것이다.
- capture 중 `Tab`은 focus trap과 충돌한다. `Tab` 자체를 등록하려면 "Capture next key" 버튼을 누른 직후 한 번만 Tab을 capture하고, 그 외에는 focus 이동으로 둔다.

### 13.5 브라우저 선점 키 UX

브라우저가 선점한 키는 JavaScript 이벤트가 오지 않는다. 사용자가 `Ctrl+J` 같은 키를 눌렀는데 아무 이벤트가 없으면 "등록 실패"가 아니라 "미감지 상태"를 보여줘야 한다.

권장 방식:

1. `Capture next key`를 누르면 3초 타이머를 시작한다.
2. keydown이 들어오면 감지 결과를 표시한다.
3. 타이머가 끝날 때까지 keydown이 없으면 다음 상태를 표시한다.

```text
감지된 키가 없습니다.
브라우저 또는 OS가 이 단축키를 먼저 처리했을 수 있습니다.
fullscreen keyboard capture를 켜거나 다른 키를 등록하세요.
```

이 문구는 오류처럼 보이되 사용자가 해결 가능한 다음 행동을 바로 고를 수 있어야 한다.

- `Try fullscreen capture`
- `Register alternative shortcut`
- `Open diagnostics`

단, fullscreen과 Keyboard Lock API는 브라우저 지원과 사용자 gesture가 필요하므로 첫 구현의 필수 요건으로 묶지 않는다.

### 13.6 Bindings 탭 UX

Bindings 탭은 command preset 목록과 비슷한 list UI가 적합하다.

각 row는 다음 정보를 한 줄에 담는다.

```text
[Enabled toggle] Shift+Enter  ->  LF (\n)     Scope: Workspace     Profile: AI TUI
                 Last tested: delivered       [Test] [Edit] [Delete]
```

상태 badge:

| Badge | 의미 |
|---|---|
| Delivered | 마지막 테스트에서 Browser -> xterm -> WS까지 도달 |
| Browser blocked | capture timeout 또는 실제 이벤트 미감지 |
| Duplicated | mapping과 xterm native가 중복 전송된 흔적 |
| IME guarded | IME 조합 중 mapping이 skip됨 |
| Disabled | 저장되어 있지만 적용하지 않음 |

삭제 확인은 `MessageBox`를 재사용한다.

위험한 custom sequence 저장 시에도 `MessageBox`를 사용한다.

```text
이 sequence는 제어 문자를 포함합니다. 현재 scope에 저장하시겠습니까?
```

### 13.7 Profiles 탭 UX

Profiles는 복잡한 설명 화면이 아니라 선택과 diff 중심이어야 한다.

권장 profile:

| Profile | 기본 정책 |
|---|---|
| xterm default | 기존 동작 유지 |
| AI TUI compatibility | `Shift+Enter -> LF`, `Ctrl+J -> LF if delivered`, paste/IME guard 유지 |
| Custom | 사용자 binding만 적용 |

Profile 선택 UI는 `<select>`보다 tab 또는 segmented button이 적합하다. 이미 `CommandPresetDialog`에 tablist 패턴이 있으므로 동일 스타일을 사용할 수 있다.

Profile을 바꾸면 바로 저장하지 말고 draft 상태로 두고 `Apply` 버튼을 둔다. 잘못된 keyboard mapping은 터미널 사용성을 즉시 망가뜨릴 수 있으므로 settings page와 같은 dirty/save 모델이 안전하다.

### 13.8 Terminal 화면과의 진입점

기본 진입점은 상단 Tools 버튼이 여는 context menu를 사용한다.

현재 `Header.tsx`에는 이미 다음 구조가 있다.

```text
header-tools-button
  -> ContextMenu
    -> 명령줄 관리
```

Shortcut manager는 새 top-level icon button을 추가하지 말고, 이 Tools menu에 항목을 추가하는 것이 가장 자연스럽다.

권장 메뉴 구조:

```text
Tools
  명령줄 관리
  터미널 키보드
```

구현 관점:

- `HeaderProps`에 `onOpenTerminalShortcutManager?: () => void`를 추가한다.
- `toolsMenuItems`에 `터미널 키보드` 항목을 추가한다.
- 메뉴 선택 시 첫 방문이거나 active terminal에서 TUI가 실행 중이면 `TerminalShortcutDialog`를 `Capture` 탭으로 연다.
- active terminal의 TUI 실행 여부를 확실히 판정할 수 없으면 `Capture` 탭으로 여는 것을 fallback으로 둔다. 관리 목록보다 진단 시작 화면이 사용자의 문제 해결 흐름에 더 가깝다.
- 사용자가 이미 shortcut manager를 사용한 적이 있으면 마지막으로 열었던 탭을 복원한다.
- `CommandPresetDialog`와 같은 방식으로 App 상위 상태에서 `open` boolean을 관리한다.
- 새 header button을 만들지 않으므로 상단 UI 밀도와 iconography를 유지할 수 있다.

이 진입점은 global/workspace binding 관리에 적합하다. 사용자는 "도구" 메뉴에서 명령줄 preset과 keyboard mapping을 같은 계열의 터미널 생산성 도구로 찾을 수 있다.

단, 현재 구현 기준으로 이 진입점은 desktop pointer UX에 한정된다.

- `Header.tsx`의 Tools 버튼은 `!isMobile` 조건에서만 렌더링된다.
- `ContextMenu` 항목은 현재 `div role="menuitem"` 중심이며, roving focus, Enter/Space 실행, Arrow navigation이 충분하지 않다.

따라서 "상단 Tools menu 진입점"을 정식 접근성 지원 진입점으로 선언하려면 다음 중 하나가 선행되어야 한다.

1. `ContextMenu`에 keyboard navigation을 추가한다.
2. 모바일에서도 Tools menu 또는 동등한 drawer action을 노출한다.
3. Settings page에 `TerminalShortcutDialog`를 여는 secondary action을 둔다.

초기 phase에서는 desktop pointer 진입점으로 제한하고, keyboard-only/mobile 진입점은 별도 phase로 분리하는 것이 안전하다.

보조 진입점은 후속 phase로 검토한다.

1. 터미널 pane context action
   - 목적: 현재 TUI에서 막힌 단축키를 바로 capture
   - 구현: 현재 active sessionId/workspaceId를 prefill하고 Capture 탭으로 바로 연다.
   - 우선순위: 낮음. 초기 구현은 상단 Tools menu만으로 충분하다.

2. Settings page 하위 section
   - 목적: 저장된 global 설정을 settings와 함께 확인
   - 구현: 직접 편집 화면을 중복 구현하지 않고 `TerminalShortcutDialog`를 여는 secondary action 정도로 제한한다.
   - 우선순위: 낮음. settings page가 이미 넓고 복잡하므로 shortcut manager 본문을 settings card 안에 넣으면 관리성이 떨어진다.

단, 기존 UI 규칙상 개인 판단만으로 아이콘/레이아웃을 바꾸면 안 된다. 구현 시에는 기존 header action 또는 context menu 확장 범위를 별도 계획에 명시해야 한다.

### 13.9 시각 디자인 지침

Shortcut manager는 운영 도구이므로 현재 `CommandPresetDialog`와 같은 조용한 밀도 높은 UI가 맞다.

권장 CSS 방향:

- 배경: `#252526`
- field 배경: `#1e1e1e`
- border: `#3c3c3c`, `#4c4c4c`
- 성공 상태: 기존 green 계열
- 경고 상태: settings warning banner 계열
- 오류 상태: `command-preset-error` 또는 `settings-banner-error` 계열
- border radius: 4~6px
- 키 캡은 pill보다 고정 폭 monospace token 형태가 낫다.

예시:

```text
Shift + Enter
Ctrl + J
ESC [ 13 ; 2 u
```

키 캡은 예쁘게 크게 보이는 장식 요소가 아니라, 매핑을 정확히 읽는 데이터 토큰이어야 한다.

### 13.10 접근성과 focus 정책

`WindowDialog`는 이미 modal일 때 body siblings에 `inert`와 `aria-hidden`을 적용하고, focus trap을 수행한다. 이를 유지해야 한다.

추가 정책:

- capture region은 `role="group"`과 `aria-live="polite"` 상태 영역을 가진다.
- 감지 성공/실패는 `role="status"` 또는 `role="alert"`로 전달한다.
- capture mode에서는 close shortcut보다 key capture가 우선한다.
- capture mode 종료 후 focus는 `Capture next key` 버튼으로 돌아간다.
- 저장 후 focus는 새로 생성된 binding row의 Test 또는 Edit 버튼으로 이동한다.

### 13.11 구현 시 주의할 점

1. `WindowDialog`의 document-level keydown trap은 `document.addEventListener('keydown', ..., true)`로 `Escape`와 `Tab`을 capture 단계에서 먼저 처리한다. DOM 이벤트 순서상 dialog surface 내부 capture handler는 document capture보다 늦게 실행되므로, surface capture만으로는 `Escape`/`Tab`을 선점할 수 없다. shortcut capture를 제대로 지원하려면 `WindowDialog`에 `keyboardTrapOptions`, `onDialogKeyDownCapture`, `captureMode` 같은 명시 prop을 추가하고, document-level trap 내부에서 capture mode를 우선 분기해야 한다.
2. `Ctrl+V`는 현재 paste 이중 입력 방지를 위해 별도 차단 로직이 있다. shortcut mapping에서 `Ctrl+V`를 임의로 send action에 매핑하면 기존 paste 회귀를 깨뜨릴 수 있으므로 기본적으로 reserved 처리한다.
3. `Ctrl+C`는 선택 영역이 있으면 copy, 없으면 SIGINT로 보내는 기존 UX를 유지해야 한다.
4. IME active 상태에서는 mapping보다 IME guard가 우선한다.
5. 저장된 mapping은 `TerminalView`에서 xterm native보다 먼저 적용되므로 debug capture에 matching 결과를 반드시 남겨야 한다.

### 13.12 최종 UI/UX 권고

최종 권고는 다음이다.

```text
기본 관리 UI:
  WindowDialog 기반 TerminalShortcutDialog

위험/삭제 확인:
  MessageBox 재사용

단순 ConfirmModal 계열:
  shortcut manager에는 사용하지 않음

FileOperationDialog:
  mdir 전용이므로 재사용하지 않음

초기 출시 범위:
  Profiles + Bindings + Capture 3탭
  AI TUI compatibility opt-in
  Shift+Enter -> LF preset
  사용자 등록 mapping
```

이 구성이 가장 중요한 이유는 사용자가 앞으로 어떤 TUI shortcut을 만날지 모른다는 점이다. BuilderGate는 정답 shortcut 목록을 미리 제공하려 하기보다, 사용자가 현재 환경에서 실제로 전달되는 키를 진단하고 필요한 mapping을 직접 만들 수 있는 작은 "keyboard lab"을 제공해야 한다.
