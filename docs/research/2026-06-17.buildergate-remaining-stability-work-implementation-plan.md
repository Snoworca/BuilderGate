# BuilderGate Remaining Stability Work Implementation Plan

작성일: 2026-06-17

대상 문서: `docs/research/2026-06-17.buildergate-stability-implementation-report.md`의 `남은 작업`

## 요약

남은 작업 5개는 모두 같은 문제를 다룬다. 브라우저와 서버 사이, 그리고 브라우저 내부의 xterm write 경로에 backpressure가 없으면 출력 폭주가 입력 지연과 메모리 증가로 이어진다. 설정으로 queue/cache 크기를 제한하는 방식은 효과가 있지만, 단순히 숫자만 두면 부족하다. 각 queue마다 "초과 시 무엇을 버리고, 무엇을 서버 authoritative snapshot으로 복구하며, 어떤 입력을 막을지"가 명확해야 한다.

구현 순서는 다음이 가장 안전하다.

1. 프론트엔드 runtime resource limits 파서를 일반화한다.
2. `WebSocketContext.send()`에 browser send `bufferedAmount` 기반 input backpressure를 넣는다.
3. visible terminal xterm write scheduler를 추가한다.
4. snapshot/localStorage budget과 tombstone TTL을 runtime config에 연결한다.
5. live runtime LRU eviction을 구현한다.
6. unified mode 안정화 뒤 split WebSocket을 feature flag로 추가한다.

원 문서의 순서에서는 LRU eviction이 snapshot budget보다 앞에 있지만, 실제 구현은 snapshot/localStorage budget을 먼저 하거나 같은 PR에서 함께 처리하는 편이 안전하다. LRU eviction은 terminal unmount/remount와 restore 빈도를 늘리므로 snapshot 저장소 제한과 tombstone TTL이 먼저 준비되어야 localStorage가 새로운 누수 지점이 되지 않는다.

## 조사 방식

사용한 서브에이전트 5개:

- Agent 1: browser WebSocket send backpressure와 input queue 연동 조사
- Agent 2: visible xterm write scheduler, overflow 복구 경로 조사
- Agent 3: live workspace/terminal runtime LRU eviction 조사
- Agent 4: snapshot/localStorage budget과 tombstone TTL 조사
- Agent 5: split WebSocket protocol/rollout 조사

실행하지 않은 작업:

- 서버 기동, Playwright 실검증, health check는 수행하지 않았다.
- `https://localhost:2002`는 사용하지 않았다.
- `node.exe` 전체 종료는 수행하지 않았다.

## 외부 근거

- MDN `WebSocket.bufferedAmount`: `send()`로 queue됐지만 아직 네트워크로 전송되지 않은 byte 수를 제공하며, closed 상태에서 계속 `send()`하면 값이 계속 증가할 수 있다. <https://developer.mozilla.org/en-US/docs/Web/API/WebSocket/bufferedAmount>
- MDN `WebSocket.send()`: `send()`는 데이터를 전송 queue에 넣고 `bufferedAmount`를 증가시키며, buffer가 가득 차면 socket이 자동으로 close될 수 있다. <https://developer.mozilla.org/en-US/docs/Web/API/WebSocket/send>
- WHATWG WebSockets Standard: `bufferedAmount`는 네트워크가 처리할 수 있는 속도보다 빠르게 보내지 않도록 monitoring하는 데 사용할 수 있다. <https://websockets.spec.whatwg.org/>
- xterm.js parser hooks guide: `term.write(data, callback)`의 callback은 chunk parse 완료 후 실행되지만, 그 시점에 screen update가 아직 일어나지 않았을 수 있다. <https://xtermjs.org/docs/guides/hooks/>
- xterm.js Terminal API: `onWriteParsed`는 write 후 terminal data parsing이 끝났을 때 발생하며, 많은 data가 있을 때 pending write가 남아 있어도 frame마다 한 번 발생할 수 있다. <https://xtermjs.org/docs/api/terminal/classes/terminal/>

## 현재 코드 근거

### 서버 resource limits는 이미 넓게 노출됨

- `server/src/schemas/config.schema.ts`는 `resourceLimits.clientWs`, `resourceLimits.terminal`, `resourceLimits.snapshots`, `resourceLimits.workspaceRuntime` schema와 기본값을 갖고 있다.
- `server/src/types/config.types.ts`는 같은 public config 타입을 갖고 있다.
- `server/src/services/RuntimeConfigStore.ts`는 `/api/runtime-config` public payload에 resource limits를 포함한다.

### 프론트엔드 runtime config 파서는 아직 좁음

- `frontend/src/utils/inputReliabilityMode.ts`는 현재 `inputReliabilityMode`와 terminal hidden output 일부만 파싱한다.
- `clientWs`, visible output limits, input queue limits, transport outbox limits, snapshots, workspaceRuntime은 서버에서 내려오지만 프론트에서 typed getter로 쓰기 어렵다.

### WebSocket send는 browser send buffer를 보지 않음

- `frontend/src/contexts/WebSocketContext.tsx`의 `SendResult` reason은 현재 `not-open`, `missing-token`, `stale-socket` 중심이다.
- `send(msg)`는 open socket이면 `ws.send(JSON.stringify(msg))`를 바로 호출한다.
- `ws.bufferedAmount` high-water, hard reconnect, send exception 처리, input-only gating이 없다.

### xterm visible write는 즉시 write 중심

- `frontend/src/components/Terminal/TerminalView.tsx`의 `writeOutput()`는 `term.write(data, callback)`를 즉시 호출한다.
- restore 중 buffered output은 `join('')` 후 한 번에 flush된다.
- visible output queue cap, chunk coalesce, per-frame budget, overflow 복구 요청이 없다.

### live runtime LRU scaffold는 비활성

- `frontend/src/App.tsx`에는 `MAX_ALIVE_WORKSPACES = 0`으로 비활성화된 workspace-only LRU scaffold가 있다.
- `TerminalRuntimeLayer`는 전달받은 non-disconnected tab을 모두 mount한다.
- runtime eviction은 persistent workspace/tab state가 아니라 mounted terminal runtime에만 적용해야 한다.

### snapshot/localStorage는 hard-coded budget 중심

- `frontend/src/utils/terminalSnapshot.ts`는 total budget과 per-snapshot cap이 상수다.
- removal tombstone은 `savedAt`을 저장하지만, TTL 없이 permanent marker처럼 동작한다.
- snapshot eviction은 char budget 중심이며 `maxEntries`, `tombstoneTtlMs` runtime config가 아직 연결되지 않았다.

### split WebSocket은 protocol/상태 모델 변경이 필요

- 현재 frontend는 하나의 `/ws?token=...` socket과 `wsRef`로 input, output, subscription, replay, repair, workspace event를 모두 처리한다.
- server `WsRouter`는 socket-local `WsClientMeta`, replay pending, repair pending, subscription state를 사용한다.
- split mode는 단순히 socket만 하나 더 여는 작업이 아니라 logical client group, connection id, pairing token, subscription epoch 설계가 필요하다.

## Phase 1: browser send `bufferedAmount` 기반 input backpressure

### 목표

브라우저 send buffer가 쌓이는 상태에서 키 입력이 계속 `ws.send()`로 들어가 메모리와 입력 지연을 증폭시키는 경로를 끊는다.

### 중요한 판단

backpressure gate는 `input` 메시지에만 우선 적용한다. `subscribe`, `unsubscribe`, `screen-snapshot:ready`, `screen-repair:ready`, `repair-replay` 같은 recovery/control 메시지를 막으면 snapshot/replay pending이 풀리지 않는 deadlock이 생길 수 있다.

### 구현 방법

`WebSocketContext.send()` 내부에서 다음 순서로 처리한다.

1. `JSON.stringify(msg)`를 한 번만 수행한다.
2. UTF-8 byte size를 계산한다.
3. `msg.type === 'input'`일 때만 `ws.bufferedAmount`를 읽는다.
4. `bufferedAmount >= hardReconnectBytes`이면 send하지 않고 hard backpressure result를 반환한 뒤 socket reconnect를 유도한다.
5. `bufferedAmount >= inputBackpressureBytes` 또는 `bufferedAmount + payloadBytes > inputBackpressureBytes`이면 send하지 않고 soft backpressure result를 반환한다.
6. 정상인 경우에만 `ws.send(serialized)`를 호출한다.
7. `send()` exception은 `send-error`류 reason으로 관찰 가능하게 남긴다.

`SendResult` 확장 예:

- `ok: false, reason: 'client-backpressure'`
- `ok: false, reason: 'client-hard-backpressure'`
- `ok: false, reason: 'send-error'`

`TerminalContainer`는 `client-backpressure`를 일반 실패와 구분해 bounded transport outbox에 넣는다. 이후 기존 retry loop가 drain을 시도한다. 단, browser WebSocket에는 drain event가 없으므로 timer 기반 retry가 필요하며 retry queue도 `transportOutboxMaxBytes`, `transportOutboxTtlMs`로 제한해야 한다.

`TerminalView` input barrier에는 `client-ws-backpressure` reason을 추가한다. 이 상태에서는 `captureAllowed=true`, `transportReady=false`가 되어야 한다. 즉 사용자의 key event는 잡되, 서버로 바로 보내지 않고 bounded input queue에 머물러야 한다.

backpressure barrier state machine:

1. `open`: `bufferedAmount < inputBackpressureBytes`이고 outbox retry가 정상 전송 중인 상태
2. `soft-blocked`: input send가 soft threshold에 걸린 상태. `TerminalContainer`가 `client-ws-backpressure` barrier를 set하고 retry timer를 시작한다.
3. `probing`: bounded timer가 outbox head를 재전송한다. 이때 `bufferedAmount`가 low-water threshold 아래로 내려갔는지도 함께 확인한다. low-water는 `inputBackpressureBytes / 2` 같은 derived value로 시작하되 config로 분리할 수 있다.
4. `recovered`: outbox head 전송이 성공하고 `bufferedAmount`가 low-water 아래이면 barrier를 clear하고 기존 `syncInputTransportState()` 경로를 호출한다.
5. `hard-reconnecting`: hard threshold에 걸리면 socket reconnect를 유도하고, reconnect 후 resubscribe와 outbox retry가 성공하기 전까지 barrier를 유지한다.

clear 조건은 반드시 "한 번의 retry 성공"만이 아니라 "retry 성공 + low-water 도달"이어야 한다. browser WebSocket에는 drain event가 없으므로 timer/probe가 barrier ownership을 가진다. 이 ownership은 `TerminalContainer`에 두고, `WebSocketContext.send()`는 threshold decision과 reconnect trigger만 반환하는 편이 책임 분리가 명확하다.

- `frontend/src/contexts/WebSocketContext.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/components/Terminal/TerminalView.tsx`

- `server/src/types/ws-protocol.ts`
- `frontend/tests/unit/runtimeConfig.test.ts`

### 테스트

- pure helper: below threshold sends
- pure helper: soft threshold returns `client-backpressure`
- pure helper: hard threshold returns `client-hard-backpressure`
- invalid/non-finite `bufferedAmount` is treated safely
- control/replay ACK 메시지는 backpressure gate를 우회
- E2E 또는 component-level: simulated backpressure에서 input이 outbox/TerminalView queue에 들어가고 retry 후 visible output까지 도달

### 위험과 대응

- 위험: 여러 queue가 중첩되어 오히려 memory cap이 늘어날 수 있다.

- 위험: ACK/control을 막아 recovery가 멈춘다.
  대응: input-only gate로 시작한다.

## Phase 2: visible terminal xterm write scheduler

### 목표

visible terminal에 출력이 폭주할 때 `term.write()` 호출과 xterm internal write buffer가 main thread를 장시간 점유하지 않도록 queue cap, coalesce, per-frame budget을 둔다.

### 중요한 판단

xterm `write` callback은 parse 완료 신호이지 실제 paint 완료 신호가 아니다. 따라서 callback만 보고 "화면에 안전하게 반영 완료"라고 간주하면 안 된다. snapshot ACK, screen repair readiness, restore release는 scheduler idle과 xterm parse callback을 함께 고려해야 한다.

### 구현 방법

새 utility를 추가한다.

- 후보 파일: `frontend/src/utils/terminalOutputScheduler.ts`
- 책임:
  - pending chunks 관리
  - UTF-8 byte accounting
  - `visibleOutputMaxChunks` 초과 시 coalesce
  - `visibleFlushBudgetBytes` 단위 drain
  - code point를 깨지 않는 split
  - per-write callback ordering 보장
  - reset/cancel 시 stale callback 무시

`TerminalView.writeOutput()`는 직접 `term.write()` 대신 scheduler `enqueue()`를 호출한다. 단, server-authoritative snapshot replacement와 screen repair apply는 일반 live output queue와 섞지 않는다. authoritative write 전에는 scheduler를 reset/cancel하고, pending live output은 server snapshot/replay로 복구한다.

overflow policy:

1. pending chunk 수가 `visibleOutputMaxChunks`를 넘으면 먼저 coalesce한다.
2. coalesce 후에도 pending bytes가 `visibleOutputQueueMaxBytes`를 넘으면 not-yet-written local visible queue를 drop한다.
3. terminal을 stale/recovery-pending으로 표시한다.
4. `TerminalContainer`에 `repair-replay` 요청을 보내 server snapshot으로 복구한다.
5. 복구 중 추가 live output은 local xterm에 쓰지 않는다. visible overflow 이후 local screen은 stale로 간주하고, authoritative replay/snapshot이 적용될 때까지 새 live output도 적용하지 않는다.
6. recovery send 실패 시 unbounded accumulation으로 돌아가지 않고 stale 상태를 유지한다.

visible overflow recovery state는 hidden output recovery와 분리해서 모델링한다. 현재 `repair-replay`는 hidden-output replay 경로와 강하게 연결되어 있으므로, 구현 시 recovery reason/token을 일반화한다.

권장 state shape:

```ts
type TerminalRecoveryReason = 'hidden-output' | 'visible-overflow' | 'screen-repair';

interface TerminalRecoveryState {
  reason: TerminalRecoveryReason;
  sessionId: string;
  token: string;
  requestedAt: number;
}
```

ACK와 clear 규칙:

- `hidden-output` recovery ACK는 hidden output skipped/tail 상태만 clear한다.
- `visible-overflow` recovery ACK는 scheduler stale 상태와 input barrier만 clear한다.
- `screen-repair` ACK는 repair pending state만 clear한다.
- 같은 session에서 recovery가 중첩되면 최신 token만 authoritative하고, stale ACK는 state를 clear하지 않는다.
- visible overflow 중 hidden replay나 screen repair가 이미 pending이면 새 요청을 무조건 덮어쓰지 말고, reason priority 또는 queueing 규칙을 둔다. 권장 priority는 `screen-repair` > `visible-overflow` > `hidden-output`이다.

### 주요 파일

- `frontend/src/utils/terminalOutputScheduler.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/tests/unit/terminalOutputScheduler.test.ts`
- `frontend/tests/unit/runtimeConfig.test.ts`

### 테스트

- flush budget 단위 drain
- chunk coalescing
- UTF-8 byte accounting
- multi-byte character split 방지
- callback ordering
- reset 후 stale callback 무시
- restore pending은 scheduler drain 전 release되지 않음
- screen repair는 scheduler non-idle 상태에서 apply되지 않음

### 위험과 대응

- 위험: pending live output 위에 screen repair를 적용하면 순서가 깨진다.
  대응: repair readiness check에 scheduler idle 조건을 추가한다.
- 위험: callback timing 변경으로 `screen-snapshot:ready` ACK가 빨라진다.
  대응: `replaceWithSnapshot()` resolve 조건에 scheduler drain 완료를 포함한다.
- 위험: output overflow 시 사용자가 local scrollback 일부를 잃는다.
  대응: server authoritative snapshot으로 최신 screen을 복구하고 debug event로 overflow를 관찰 가능하게 한다.

## Phase 3: snapshot/localStorage budget과 tombstone TTL

### 목표

localStorage snapshot이 장시간 사용 중 영구 증가하거나 tombstone이 영구 차단 상태로 남는 문제를 막는다.

### 구현 방법

`terminalSnapshot.ts`의 hard-coded 상수를 runtime limits 기반 option으로 바꾼다. 기존 상수는 fallback default로 유지한다.

연결할 config:

- `perSnapshotMaxChars`
- `totalStorageBudgetChars`
- `maxEntries`
- `tombstoneTtlMs`

변경 방향:

1. snapshot save/restore size guard에 `perSnapshotMaxChars` 적용
2. quota recovery와 eviction에 `totalStorageBudgetChars`, `maxEntries` 적용
3. tombstone marker의 `savedAt`을 읽고 TTL 만료 시 제거 또는 무시
4. legacy `'1'`, malformed JSON, missing date tombstone은 expired marker로 취급
5. 같은 page lifetime의 `pendingSnapshotRemovals`는 현재 session 의도를 나타내므로 TTL보다 우선

가능하면 snapshot index metadata를 도입해 매 save마다 localStorage 전체 scan을 줄인다. 단, 첫 구현에서는 기존 scan 방식을 유지하면서 `maxEntries`와 TTL만 추가해도 효과가 있다.

### 주요 파일

- `frontend/src/utils/terminalSnapshot.ts`
- `frontend/src/components/Terminal/TerminalView.tsx`
- `frontend/src/hooks/useWorkspaceManager.ts`
- `frontend/tests/unit/terminalSnapshot.test.ts`
- `frontend/tests/unit/runtimeConfig.test.ts`

### 테스트

- per-snapshot cap 초과 save/restore 방지
- total budget 초과 시 오래된 snapshot eviction
- `maxEntries` 초과 시 entry 수 기준 eviction
- fresh tombstone은 save 차단
- expired tombstone은 제거 또는 무시
- legacy/malformed tombstone은 permanent block이 되지 않음
- corrupt snapshot entry가 eviction을 막지 않음

### 위험과 대응

- 위험: custom config가 늦게 로드되어 첫 restore/save가 default로 처리된다.
  대응: `/api/runtime-config` 초기화 시점과 terminal mount 순서를 확인하고, 최소한 첫 save 전에 getter가 최신 config를 읽도록 한다.
- 위험: budget을 낮추면 오래된 snapshot이 즉시 많이 지워진다.
  대응: current visible/pinned session snapshot은 우선 보존하고, eviction debug event를 남긴다.

## Phase 4: live runtime LRU eviction

### 목표

workspace/session을 닫지 않아도 hidden terminal runtime과 xterm instance가 무제한 살아남지 않도록 mounted runtime 수를 제한한다.

### 중요한 판단

LRU eviction은 persistent model을 바꾸면 안 된다. `wm.tabs`, workspace 목록, tab bar, grid model은 그대로 두고, `TerminalRuntimeLayer`에 전달되는 resident runtime만 줄인다. 사용자가 보는 tab이 사라지거나 workspace state가 삭제되면 안 된다.

### 구현 방법

`App.tsx`의 disabled `MAX_ALIVE_WORKSPACES` scaffold를 제거하거나 대체하고, 작은 hook을 만든다.

- 후보: `frontend/src/hooks/useTerminalRuntimeResidency.ts`
- 입력:
  - `wm.tabs`
  - active workspace
  - active tab
  - current layout mode/grid visible tabs
  - `workspaceRuntime` limits
- 출력:
  - `residentTabs`
  - debug metadata

tracking metadata:

- `tab.id`
- `workspaceId`
- `sessionId`
- `lastAccessedAt`
- `hiddenSince`
- `workspaceLastAccessedAt`

policy:

- active visible tab은 항상 pinned
- desktop grid에서 visible pane은 모두 pinned
- cap보다 visible pane 수가 많으면 cap을 넘겨도 visible terminal은 evict하지 않음
- disconnected tab은 resident runtime 후보에서 제외
- `hiddenRuntimeTtlMs=0`이면 hidden 즉시 eviction 후보
- `maxLiveTerminals`는 hidden candidate에 대해 oldest first로 적용
- `maxLiveWorkspaces`는 active workspace를 제외하고 resident hidden terminal이 남은 workspace 기준으로 적용
- tab 삭제/재시작 시 metadata pruning/reset
- eviction hook에서 직접 unsubscribe하지 않고, unmount에 따른 `TerminalContainer` cleanup과 기존 `WebSocketContext` grace path에 맡김

`terminalRefsMap`은 closed tab에 대해서만 prune한다. LRU eviction만으로 ref를 삭제하면 remount/restore path가 불필요하게 흔들릴 수 있다.

## Phase 5: split WebSocket feature flag

### 목표

출력/replay traffic이 많은 socket 상태가 input/control 전송을 막는 상황을 줄인다. 단, split WebSocket은 main thread xterm parse/render stall 자체를 해결하지 못한다. 따라서 Phase 1-4로 unified mode의 bounded behavior를 만든 뒤 진행해야 한다.

### 권장 결론

보내는 WebSocket과 받는 WebSocket을 분리하면 키보드 입력 지연 완화에 도움이 될 수 있다. 특히 output socket의 send/receive backlog와 server-side slow-client handling이 input/control path에 미치는 영향을 줄인다. 하지만 브라우저 JS main thread가 xterm write로 막히는 문제는 socket을 나눠도 그대로 남는다. 그러므로 split은 visible output scheduler와 input backpressure 이후에 feature flag로 도입해야 한다.

### protocol 설계

기존 `/ws?token=<jwt>` unified mode는 유지한다.

split mode는 같은 endpoint에 opt-in query를 추가한다.

```text
/ws?token=<jwt>&mode=split&channel=control
/ws?token=<jwt>&mode=split&channel=output&clientGroupId=<id>&pairToken=<token>
```

control socket이 먼저 연결된다.

- server가 JWT를 검증한다.
- `clientGroupId`, `connectionId`, short-lived `pairToken`을 생성한다.
- 기존 `clientId` semantic은 logical group id로 유지한다.

output socket은 같은 JWT와 pair field로 연결된다.

- JWT subject/jti 일치 확인
- pair token TTL 확인
- group liveness 확인
- duplicate output socket 정책 확인

추가 protocol fields:

- `channel?: 'control' | 'output'`
- `connectionId?: string`
- `clientGroupId?: string`
- `subscriptionEpoch?: number`

control channel 담당:

- `subscribe`
- `unsubscribe`
- `input`
- `resize`
- `screen-snapshot:ready`
- `screen-repair` request
- `screen-repair:ready`
- `screen-repair:failed`
- `repair-replay`
- `ping`

output channel 담당:

- `connected` 이후 output-ready event
- `screen-snapshot`
- `screen-repair` response/rejected
- `output`
- `status`
- `cwd`
- `session:*`
- workspace/tab/grid broadcasts

server pending state는 `(WebSocket, sessionId)`에서 `(clientGroupId, sessionId, subscriptionEpoch)`로 이동한다. `subscribe`마다 epoch를 증가시키고, snapshot/repair ACK는 token뿐 아니라 epoch도 맞아야 queued output을 flush한다.

### migration phases

1. config/runtime-config에 `wsTransportMode: 'unified' | 'split-shadow' | 'split'` 추가, default는 `unified`
2. frontend `WebSocketContext` 내부를 `UnifiedTransport` adapter로 감싸 public hook API 유지
3. server에 split handshake와 client group registry 추가, shadow mode에서는 실제 terminal traffic 이동 없음
4. `sessionSubscribers`, replay pending, repair pending을 group/session/epoch 기반으로 옮기고 unified adapter도 같은 abstraction 사용
5. feature flag에서 control sends/output receives 활성화
6. split handshake/pair/output reconnect/epoch validation 실패 시 unified fallback
7. soak 이후 default 후보 검토, unified mode는 장기 유지

### rollback과 kill switch

split mode는 runtime-config kill switch를 전제로 한다. `wsTransportMode`가 `split`에서 `unified`로 내려오면 새 connection은 즉시 unified를 사용하고, active split group은 다음 순서로 정리한다.

1. control socket에 rollback reason을 기록하고 새 input send를 잠시 barrier 처리한다.
2. output socket을 close하고 server split group/pair token/pending output queue를 cleanup한다.
3. in-flight snapshot/repair/replay pending state는 stale epoch로 표시한다.
4. unified socket을 새로 열고 active subscription을 resubscribe한다.
5. server snapshot을 받은 뒤 input barrier와 outbox retry를 재개한다.

운영 kill switch 요구사항:

- default는 항상 `unified`
- runtime-config에서 `split`을 끄면 재배포 없이 fallback 가능
- split group registry에는 TTL cleanup이 있어야 함
- split-to-unified transition 중 raw input은 로그에 남기지 않음
- active session 중 toggle off regression test가 있어야 함

## 설정으로 queue/cache 크기 제한은 효과적인가

효과적이다. 다만 세 가지 조건을 만족해야 한다.

첫째, queue별 cap이 있어야 한다.

- browser send buffer: `clientWs.inputBackpressureBytes`, `clientWs.hardReconnectBytes`
- TerminalView input queue: `terminal.inputQueueMaxBytes`, `terminal.inputQueueTtlMs`
- TerminalContainer transport outbox: `terminal.transportOutboxMaxBytes`, `terminal.transportOutboxTtlMs`
- visible output scheduler: `terminal.visibleOutputQueueMaxBytes`, `terminal.visibleOutputMaxChunks`, `terminal.visibleFlushBudgetBytes`
- snapshot storage: `snapshots.perSnapshotMaxChars`, `snapshots.totalStorageBudgetChars`, `snapshots.maxEntries`, `snapshots.tombstoneTtlMs`
- runtime residency: `workspaceRuntime.maxLiveWorkspaces`, `workspaceRuntime.maxLiveTerminals`, `workspaceRuntime.hiddenRuntimeTtlMs`

둘째, 초과 정책이 있어야 한다.

- input queue 초과: 새 입력을 무제한 받지 않고 barrier/debug event 처리
- transport outbox 초과: stale input을 TTL로 reject하고 raw input을 로그에 남기지 않음
- visible output 초과: local pending output drop 후 server authoritative `repair-replay`
- snapshot 초과: 오래된 snapshot/tombstone eviction
- runtime cap 초과: hidden mounted runtime unmount, persistent tab/workspace state 보존

셋째, 관찰 가능해야 한다.

- cap hit count
- overflow count
- hard reconnect count
- dropped local output bytes
- snapshot eviction count
- runtime eviction/restore count

따라서 설정 기반 cap은 이번 문제의 핵심 완화책이다. 하지만 "캐시 크기만 줄이기"는 해결책이 아니며, 서버 snapshot/replay와 입력 barrier까지 이어지는 lifecycle 설계가 필요하다.

## split WebSocket은 키보드 입력 지연에 도움이 되는가

부분적으로 도움이 된다.

도움이 되는 경우:

- output/replay traffic이 같은 socket의 `bufferedAmount`와 server slow-client queue를 키워 input send가 같이 밀릴 때
- server가 느린 client socket을 terminate해야 하는데 input/control까지 같이 끊기는 경우
- replay/snapshot output이 input/control ACK와 같은 lane을 점유할 때

도움이 제한적인 경우:

- 브라우저 main thread가 xterm parse/render로 막혀 key event 자체가 늦게 처리될 때
- React render나 localStorage snapshot 작업이 main thread를 오래 점유할 때
- input queue/outbox cap 없이 split socket 양쪽에 queue만 늘어날 때

결론적으로 split WebSocket은 장기적으로 채택할 가치가 있지만, 1차 해결책은 아니다. 먼저 unified mode에서 `bufferedAmount` gate, visible xterm scheduler, runtime LRU, snapshot budget을 적용해야 split 이후에도 같은 문제가 두 socket으로 나뉘어 재발하지 않는다.

## 서브에이전트 반영 내역

| Agent | 조사 범위 | 핵심 발견 | 문서 반영 |
| --- | --- | --- | --- |
| Agent 1 | browser WebSocket input backpressure | `WebSocketContext.send()`가 `bufferedAmount`를 보지 않고 input을 즉시 send하며, TerminalContainer outbox와 TerminalView input barrier가 runtime cap과 연결되어야 함 | Phase 1에 input-only gate, soft/hard backpressure, outbox 연동, barrier state machine 추가 |
| Agent 2 | visible xterm write scheduler | `TerminalView.writeOutput()`가 즉시 `term.write()`를 호출하고, xterm callback은 paint 완료가 아니라 parse 완료 신호임 | Phase 2에 scheduler utility, frame budget, overflow 시 authoritative replay, recovery reason/token 추가 |
| Agent 3 | live runtime LRU eviction | `App.tsx`의 workspace-only LRU scaffold는 비활성이고, runtime eviction은 persistent tab/workspace state가 아니라 mounted runtime에만 적용해야 함 | Phase 4에 residency hook, visible pin, TTL/cap policy, unmount cleanup 위임 추가 |
| Agent 4 | snapshot/localStorage budget | snapshot budget은 hard-coded이고 tombstone은 TTL 없이 permanent marker처럼 동작함 | Phase 3에 runtime snapshot limits, `maxEntries`, tombstone TTL, legacy marker 처리 추가 |
| Agent 5 | split WebSocket | unified socket state가 physical socket 중심이라 split에는 logical group, pairing token, epoch, rollback이 필요함 | Phase 5에 split protocol, migration phases, kill switch/rollback 추가 |

## 완료 기준

남은 작업이 해결됐다고 볼 수 있는 기준:

- 장시간 visible output 폭주 후 browser memory가 cap 안에서 안정화된다.
- hidden workspace/session을 오래 유지해도 xterm instance와 snapshot storage가 무제한 증가하지 않는다.
- browser `bufferedAmount`가 high-water를 넘으면 input이 bounded queue/barrier로 이동하고 raw input leak 없이 회복된다.
- terminal visible output queue overflow 시 server snapshot/replay로 복구된다.
- workspace/session을 닫지 않아도 hidden runtime은 LRU 정책에 따라 unmount되고 재진입 시 snapshot restore가 된다.
- split WebSocket feature flag off에서 기존 unified behavior가 유지된다.
- split WebSocket feature flag on에서 output backlog가 input/control lane을 막지 않고, 실패 시 unified fallback이 가능하다.
