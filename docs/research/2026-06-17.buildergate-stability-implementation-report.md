## 결론

설정으로 chunk queue/cache 크기를 제한하는 방식은 효과가 있다. 단, "숫자를 둔다"만으로는 부족하고, 초과 정책이 명확해야 한다. 이번 구현에서는 서버 headless queue는 `degrade-headless`, 서버 WebSocket terminal output은 per-client queue/coalesce 후 slow client terminate, hidden terminal output은 client write 중단 후 snapshot restore로 처리한다.

보내는 WebSocket과 받는 WebSocket을 분리하는 방식은 키보드 입력 지연 완화에 도움이 될 가능성이 크다. 다만 바로 split을 넣기 전에 단일 WebSocket의 backpressure와 queue ordering을 먼저 안전화해야 한다. 이번 구현은 split 전 단계로 unified WebSocket에 `safeSend`와 terminal output backpressure를 적용했다.

## 구현된 내용

### Phase 0: resourceLimits 설정 골격

구현 파일:

- `server/src/types/config.types.ts`
- `server/src/schemas/config.schema.ts`
- `server/src/services/RuntimeConfigStore.ts`
- `server/src/index.ts`
- `server/src/utils/configTemplate.ts`
- `server/config.json5.example`

주요 결과:

- `resourceLimits` schema와 타입을 추가했다.
- legacy config에는 안전한 기본값이 적용된다.
- `/api/runtime-config`가 non-secret client resource limits를 반환한다.
- frontend는 hidden terminal output 정책과 tail byte cap을 runtime config에서 읽는다.
- `resourceLimits.headless.overflowPolicy`는 현재 구현 가능한 `degrade-headless`만 허용한다.

### Phase 1: SessionManager headless output queue cap

구현 파일:

- `server/src/services/SessionManager.ts`
- `server/src/services/SessionManager.test.ts`
- `server/src/test-runner.ts`

주요 결과:

- `pendingOutputChunks` 무제한 증가 경로를 byte/chunk bounded write pump로 바꿨다.
- `Array.shift()` 기반 소비를 cursor 기반 처리로 바꿨다.
- cap 초과 시 hidden authoritative state를 조용히 drop하지 않고 headless를 degraded로 전환한다.
- session delete/degrade 경로에서 pending output 참조를 정리한다.
- observability에 pending bytes, max pending bytes, overflow count를 추가했다.

### Phase 2: WsRouter safeSend와 server-side output backpressure

구현 파일:

- `server/src/ws/WsRouter.ts`
- `server/src/types/ws-protocol.ts`
- `server/src/test-runner.ts`

주요 결과:

- direct `ws.send()` 경로를 `safeSend`로 통합했다.
- terminal output은 socket `bufferedAmount` high-water 또는 기존 transport output queue 존재 시 per-client queue로 들어간다.
- 같은 session의 live output은 coalesce된다.
- replay/screen-repair protocol ordering은 `allowQueue: false`로 유지했다.
- hard limit 또는 queue overflow는 session이 아니라 느린 client socket만 terminate한다.
- observability에 transport queue, high-water, hard-limit disconnect, send error counters를 추가했다.

### Phase 3: hidden terminal live output pause

구현 파일:

- `frontend/src/components/Terminal/TerminalContainer.tsx`
- `frontend/src/utils/terminalHiddenOutput.ts`
- `frontend/src/utils/inputReliabilityMode.ts`
- `frontend/tests/unit/terminalHiddenOutput.test.ts`
- `frontend/tests/unit/runtimeConfig.test.ts`

주요 결과:

- hidden terminal은 `terminalRef.current?.write(data)`를 호출하지 않는다.
- hidden output은 기본적으로 client에 저장하지 않고 skipped byte count만 유지한다.
- `hiddenOutputPolicy=debug-tail`일 때도 `hiddenOutputTailBytes`만큼의 bounded UTF-8 tail만 유지한다.
- hidden 상태에서 출력이 발생한 terminal이 다시 visible이 되면 `{ type: 'repair-replay', sessionId }`를 보내 server snapshot restore를 요청한다.
- snapshot이 성공적으로 적용된 뒤에만 hidden output recovery state를 clear한다.
- recovery 중 입력 게이트는 replay-pending 상태가 되어 stale screen 위로 입력이 들어가는 것을 막는다.
- stale/duplicate/apply-skip/applied snapshot 경로 모두 `screen-snapshot:ready` ACK가 성공한 뒤에만 hidden recovery barrier를 해제한다. ACK 실패 시 server replay pending과 client input barrier가 서로 어긋나지 않게 유지한다.

## 남은 작업

이번 구현은 split WebSocket 이전의 안정화 단계다. 남은 장기 작업은 다음 순서가 적합하다.

1. browser send `bufferedAmount` 기반 input backpressure를 `WebSocketContext.send()`에 적용한다.
2. visible terminal xterm write scheduler에 queue cap/coalesce/frame budget을 적용한다.
3. `maxLiveWorkspaces`, `maxLiveTerminals`, `hiddenRuntimeTtlMs`로 live runtime LRU eviction을 구현한다.
4. snapshot/localStorage budget과 tombstone TTL을 runtime config로 연결한다.
5. unified mode가 안정화된 뒤 input/control socket과 output/replay socket을 feature flag로 분리한다.
