# BuilderGate agent hook status signal research

작성일: 2026-05-10

대상 프로젝트: `C:\Work\git\_Snoworca\ProjectMaster`

연구 주제: Codex, Claude Code, Hermes를 실행 중인 BuilderGate 터미널에서 `idle` / `running` 표시가 오판되는 문제를 줄이기 위해 agent hook CLI 신호를 서버 상태 판정에 병합할 수 있는지 검토한다.

연구 방식:

- 현재 BuilderGate 코드의 `SessionManager` 상태 파이프라인을 확인했다.
- 5개 서브에이전트가 Codex, Claude Code, Hermes, 현재 상태 파이프라인, 통합 아키텍처를 분리 조사했다.
- 공식 문서와 공개 저장소 근거를 교차 확인했다.
- 기존 AGENTS.md invariant를 최상위 제약으로 두고 충돌하는 제안을 정리했다.

## 1. 결론

가능하다. 다만 hook CLI는 기존 PTY/OSC/semantic detector를 대체하면 안 된다. BuilderGate의 상태 원본은 계속 `SessionManager` 하나로 유지하고, hook CLI 신호는 `SessionManager` 내부 derived state에 병합되는 외부 보강 신호로 다루어야 한다.

권장 구조는 다음과 같다.

```text
Codex / Claude Code / Hermes hook
  -> buildergate-hook CLI
  -> BuilderGate local endpoint
  -> HookSignalService 검증/정규화
  -> SessionManager.applyExternalActivitySignal()
  -> 기존 derived state, timers, detector 결과와 merge
  -> 기존 WebSocket status 메시지 유지
  -> 기존 frontend idle/running UI 유지
```

핵심 정책은 다음이다.

1. 기존 WS 계약 `{ type: "status", status: "running" | "idle" }`는 깨지 않는다.
2. `waiting`, `attention`, `blocked`, `stale` 같은 값은 1차 표시 상태가 아니라 보조 activity hint 또는 debug signal로 둔다.
3. 사용자 키 입력, local echo, prompt redraw, cursor movement, waiting-for-input repaint는 계속 `running` 전환 근거가 아니다.
4. hook `running` 신호도 단일 이벤트만 과신하지 않고 TTL, sequence, source scope, idle delay와 함께 처리한다.
5. hook이 없거나 실패하면 기존 detector 경로로 정상 fallback한다.

어려운 부분은 있다. 특히 `UserPromptSubmit`을 즉시 `running`으로 볼지 여부, `Stop` 누락, background subagent, permission prompt, tool batch 같은 상황이 까다롭다. 그래도 `activeScopes + TTL + source별 sequence`로 설계하면 구현 가능하다.

## 2. 현재 BuilderGate 상태 파이프라인

현재 서버 상태 원본은 `SessionManager`다.

확인된 주요 지점:

- `server/src/services/SessionManager.ts`: 세션 생성 시 기본 `status: "idle"`을 설정한다.
- `SessionManager`의 PTY `onData`가 모든 출력의 주 진입점이다.
- `OscDetector`는 OSC 133 shell integration marker를 처리한다. `command-start`는 일반 shell 명령이면 `running`, AI TUI foreground면 foreground process 시작으로 해석한다.
- `ForegroundAppDetectorRegistry`는 현재 `HermesForegroundDetector`를 등록한다.
- `ForegroundAppDetector.ts`의 `deriveDisplayStatus()`는 다음 규칙을 가진다.
  - `shell_prompt`는 `idle`
  - `foreground_app + waiting_input`은 `idle`
  - `foreground_app + repaint_only`는 `idle`
  - 그 외 foreground activity는 `running`
- semantic output은 바로 `running`으로 승격하지 않고 `runningDelayMs` 뒤에 `busy/running`으로 승격한다.
- `updateStatus()`가 최종 상태 변경과 WS `status` broadcast를 담당한다.
- 프론트엔드는 `WebSocketContext -> TerminalContainer -> App -> useWorkspaceManager` 경로로 탭 runtime status를 갱신한다.
- Grid, sidebar, workspace badge, breathing effect는 모두 기존 `idle | running | disconnected` 상태에 의존한다.

이 구조 때문에 외부 hook endpoint가 직접 WS를 보내거나 프론트 상태를 바꾸면 race condition이 생긴다. 외부 신호도 반드시 `SessionManager`로 들어가야 한다.

## 3. 지원 대상별 hook 근거

### 3.1 Codex

Codex CLI는 hook을 지원한다. 공식 개발자 문서에서 `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop` 같은 lifecycle event를 확인할 수 있다. 공개 `openai/codex` 저장소의 hook schema 기준으로는 `PreCompact`, `PostCompact`도 확인된다.

Codex hook 입력은 이벤트마다 다르지만 대체로 다음 필드를 포함한다.

- `session_id`
- `cwd`
- `hook_event_name`
- `model`
- `permission_mode`
- `transcript_path`
- turn-scoped event의 `turn_id`
- tool event의 `tool_name`, `tool_input`, `tool_use_id`, `tool_response`

Codex hook 출력은 stdout JSON으로 해석된다. `continue`, `stopReason`, `suppressOutput`, `systemMessage`, 일부 이벤트의 `decision: "block"` 등이 의미를 가진다. hook timeout, spawn 실패, invalid JSON, non-zero exit는 상태 신호 실패로 기록해야 한다.

Codex 이벤트 매핑 권고:

| Codex event | BuilderGate 내부 신호 | 표시 상태 권고 | 주의 |
| --- | --- | --- | --- |
| `SessionStart` | `waiting_input` 또는 debug heartbeat | `idle` | 작업 실행 신호가 아니다. |
| `UserPromptSubmit` | `turn-pending` 후보 | 기본 `idle`, 짧은 pending window | 기존 invariant 때문에 즉시 running 확정은 피한다. |
| `PreToolUse` | running scope open | `running` | 실제 tool execution 직전의 강한 신호다. |
| `PostToolUse` | running scope 유지 또는 tool scope close | 보통 `running` 유지 | 후속 model response가 남을 수 있다. |
| `PermissionRequest` | `attention` + `waiting` | `idle` + 보조 attention | 승인 대기 상태를 running으로 보지 않는다. |
| `Stop` | idle candidate | idle delay 후 `idle` | block/continuation 가능성이 있으므로 즉시 확정하지 않는다. |
| `PreCompact` | running 유지 또는 maintenance scope | 기존 상태 유지 | 세션 종료 신호가 아니다. |
| `PostCompact` | maintenance scope close | 기존 상태 복원 | idle 확정 신호로 쓰지 않는다. |

중요한 조정: Codex 담당 서브에이전트는 `UserPromptSubmit`을 `running` 후보로 제안했다. 그러나 BuilderGate의 현 invariant는 AI TUI 사용자 입력과 submit을 `idle`로 유지하도록 이미 구현되어 있다. 따라서 본 연구의 최종 권고는 `UserPromptSubmit`을 직접 `running` 확정으로 쓰지 않고 `turn-pending` 후보로 둔다. 실제 `running` 확정은 `PreToolUse`, semantic output, progress signal, 또는 explicit running hook이 뒤따를 때 수행한다.

### 3.2 Claude Code

Claude Code는 공식 hooks와 status line 기능을 제공한다. 상태 보강에는 status line보다 hooks가 적합하다.

Claude Code hook 입력은 공통적으로 다음 필드를 포함한다.

- `session_id`
- `transcript_path`
- `cwd`
- `hook_event_name`

공식 hook event 중 상태 판정에 유용한 것은 다음이다.

- `SessionStart`
- `UserPromptSubmit`
- `PreToolUse`
- `PostToolUse`
- `PostToolUseFailure`
- `PostToolBatch`
- `PermissionRequest`
- `PermissionDenied`
- `Notification`
- `Stop`
- `StopFailure`
- `SubagentStart`
- `SubagentStop`
- `SessionEnd`

Claude Code 이벤트 매핑 권고:

| Claude Code event | BuilderGate 내부 신호 | 표시 상태 권고 | 주의 |
| --- | --- | --- | --- |
| `SessionStart` | heartbeat/session attached | `idle` | `startup`, `resume`, `clear`, `compact` source를 구분한다. |
| `UserPromptSubmit` | turn-pending 후보 | 기본 `idle` | 사용자 입력 자체로 running 확정하지 않는다. |
| `PreToolUse` | running scope open | `running` | tool 실행 직전의 강한 신호다. |
| `PostToolUse` | running 유지 | `running` 유지 | tool 완료가 turn 종료는 아니다. |
| `PostToolUseFailure` | running 또는 attention | 상황별 | 실패 후 복구 응답이 이어질 수 있다. |
| `PostToolBatch` | batch scope update | `running` 유지 | batch 완료가 turn 종료는 아니다. |
| `PermissionRequest` | `attention` + `waiting` | `idle` + 보조 attention | 승인 대기 상태다. |
| `PermissionDenied` | attention | `idle` 또는 기존 상태 유지 | denial 유형을 metadata로 남긴다. |
| `Notification.permission_prompt` | `attention` + `waiting` | `idle` + 보조 attention | Notification은 block/modify 용도가 아니다. |
| `Notification.idle_prompt` | idle candidate | idle delay 후 `idle` | 세션 종료와 동일시하지 않는다. |
| `Stop` | main turn stop | idle delay 후 `idle` | user interrupt에서는 누락될 수 있다. |
| `StopFailure` | attention 또는 idle_error | 보조 attention | API 오류/turn failure로 기록한다. |
| `SubagentStart` | subagent scope open | `running` | parent와 별도 counter가 필요하다. |
| `SubagentStop` | subagent scope close | scope 상태에 따름 | subagent 1개 종료만으로 전체 idle 처리하지 않는다. |
| `SessionEnd` | session ended/offline | disconnected 후보 또는 cleanup | 현재 WS 계약과 별도 설계가 필요하다. |

Claude Code status line은 보조 heartbeat로는 쓸 수 있지만 lifecycle telemetry로 쓰기에는 약하다. status line은 표시 command이고 update trigger가 제한적이며, in-flight command가 취소될 수 있다. 따라서 `running/idle/waiting` 판정의 주 입력으로 사용하지 않는다.

### 3.3 Hermes

Hermes는 두 층을 분리해서 보아야 한다.

로컬 BuilderGate 코드 기준:

- 현재 Hermes는 외부 hook으로 감지되지 않는다.
- `HermesForegroundDetector`가 command hint와 출력 패턴을 기반으로 attach한다.
- welcome, tip, status frame, banner, cursor position request, CPR warning, ticker, cursor movement repaint는 `waiting_input` 또는 `repaint_only`로 분류한다.
- semantic output만 `busy`로 분류한다.
- 기존 테스트는 Hermes submit, command echo, bootstrap output, semantic output running delay, launch failure, zsh prompt return 등을 다룬다.

공식 Hermes 문서 기준:

- Hermes Agent 문서에는 CLI와 Gateway 공통으로 사용할 수 있는 shell hooks와 plugin hooks가 존재한다.
- shell hook은 `cli-config.yaml`의 `hooks`로 선언되며 특정 event에서 shell command를 실행할 수 있다.
- Hermes 문서의 hook은 Gateway-specific hook과 shell/plugin hook이 섞여 있으므로 BuilderGate는 CLI에서 동작하는 shell hook 또는 plugin hook만 대상으로 삼아야 한다.

따라서 Hermes의 1차 구현 권고는 다음이다.

1. 기존 `HermesForegroundDetector`를 계속 유지한다.
2. Hermes hook은 opt-in adapter로 둔다.
3. Hermes hook이 없거나 실패하면 detector fallback이 정상 동작해야 한다.
4. Hermes hook event 이름은 Codex/Claude와 1:1로 같지 않을 수 있으므로 `buildergate-hook --source hermes --event <raw>`에서 표준 schema로 변환한다.

Hermes 매핑 권고:

| Hermes signal | BuilderGate 내부 신호 | 표시 상태 권고 | 주의 |
| --- | --- | --- | --- |
| shell/plugin start 또는 model/tool start | running scope open | `running` | raw event 이름은 adapter에서 표준화한다. |
| waiting input / prompt ready | `waiting_input` | `idle` | 기존 detector와 같은 의미다. |
| repaint/ticker/cursor request | `repaint_only` | `idle` | running timer를 취소할 수 있다. |
| permission/approval prompt | `attention` + `waiting` | `idle` + 보조 attention | Hermes 환경별 확인 필요. |
| stop/done | idle candidate | idle delay 후 `idle` | detector semantic output과 충돌하면 running 우선. |
| hook failure | debug only | 기존 상태 유지 | detector fallback. |

## 4. 표준 signal schema

서버 내부 표준 신호는 agent별 raw hook payload를 직접 퍼뜨리지 않고 정규화한 뒤 사용한다.

권장 타입:

```ts
type HookSignalStatus =
  | 'idle'
  | 'running'
  | 'waiting'
  | 'attention'
  | 'unknown';

type HookSignalSource =
  | 'codex-hook'
  | 'claude-code-hook'
  | 'hermes-hook'
  | 'buildergate-hook-cli';

type HookSignalEvent =
  | 'session-start'
  | 'session-end'
  | 'turn-pending'
  | 'agent-start'
  | 'agent-stop'
  | 'tool-start'
  | 'tool-stop'
  | 'tool-batch-update'
  | 'permission-prompt'
  | 'permission-denied'
  | 'waiting-input'
  | 'notification'
  | 'subagent-start'
  | 'subagent-stop'
  | 'compaction-start'
  | 'compaction-stop'
  | 'heartbeat'
  | 'hook-error';

interface SessionActivitySignal {
  sessionId: string;
  status: HookSignalStatus;
  source: HookSignalSource;
  event: HookSignalEvent;
  confidence: 'low' | 'medium' | 'high';
  ttlMs: number;
  sequence: number;
  timestamp: string;
  scopeId?: string;
  sourceSessionId?: string;
  sourceTurnId?: string;
  metadata?: {
    toolName?: string;
    batchId?: string;
    commandId?: string;
    subagentId?: string;
    promptKind?: string;
    exitCode?: number;
    cwd?: string;
    rawHookEvent?: string;
  };
}
```

설계 원칙:

- `timestamp`는 ISO-8601 UTC로 고정한다.
- `sequence`는 `sessionId + source + sourceSessionId` 단위 monotonic 값으로 둔다.
- `ttlMs`는 stop 누락과 hook CLI 장애를 회복하기 위한 필수 필드다.
- `metadata`는 allowlist만 저장한다.
- raw payload 전체 저장은 debug capture가 켜진 경우에만 크기 제한 후 허용한다.

## 5. Merge policy

서버 내부에는 `activeScopes` 개념을 둔다.

```ts
interface ExternalActivityState {
  activityHint?: 'waiting' | 'attention';
  activeScopes: Map<string, {
    source: HookSignalSource;
    event: HookSignalEvent;
    expiresAt: number;
    confidence: 'low' | 'medium' | 'high';
  }>;
  lastAcceptedSequenceBySource: Map<string, number>;
}
```

병합 규칙:

1. `running` scope가 하나라도 살아 있으면 display status는 `running` 후보가 된다.
2. `waiting`과 `attention`은 기본 display status를 `running`으로 만들지 않는다.
3. `idle` signal은 전체 세션을 바로 idle로 만들지 않고 해당 source/scope를 닫는다.
4. 모든 running scope가 닫히고 기존 semantic/OSC 경로도 busy가 아니면 idle delay 후 `idle`로 확정한다.
5. `tool-start`, `agent-start`, `subagent-start`, `compaction-start`는 running scope를 연다.
6. `tool-stop`, `agent-stop`, `subagent-stop`, `compaction-stop`은 대응 scope를 닫는다.
7. `permission-prompt`, `notification`은 `attention` 또는 `waiting` hint를 갱신하되 display는 `idle`을 유지한다.
8. `heartbeat`는 기존 scope의 TTL을 연장한다.
9. low confidence 또는 unknown event는 상태 전환 없이 debug event로만 남긴다.
10. sequence가 오래되었거나 TTL이 만료된 stale signal은 reject한다.

충돌 처리:

| 충돌 | 권고 |
| --- | --- |
| Hook은 idle, detector는 busy | detector busy 우선. idle signal은 해당 scope만 닫는다. |
| Hook은 running, PTY는 prompt repaint | high-confidence running scope가 있으면 running 유지. repaint-only는 running scope를 닫지 않는다. |
| Permission prompt 중 | display는 idle, 보조 attention. 이후 tool-start가 오면 running. |
| Subagent active 중 parent Stop | subagent scope가 남아 있으면 running 유지. |
| Stop 누락 | TTL 만료 후 scope close, idle delay 적용. |
| 중복 start/stop | 같은 scopeId면 idempotent 처리. |

## 6. 보안 설계

hook CLI는 브라우저 JWT를 사용하지 않는 것이 맞다. hook은 사용자의 전체 BuilderGate API 권한이 아니라 "현재 세션에 활동 신호를 제출할 권한"만 가져야 한다.

권장 방식:

1. 세션 생성 시 `hookSecret`을 만든다.
2. 해당 PTY 환경에만 다음 환경변수를 주입한다.
   - `BUILDERGATE_SESSION_ID`
   - `BUILDERGATE_HOOK_URL`
   - `BUILDERGATE_HOOK_TOKEN` 또는 HMAC secret reference
3. `buildergate-hook` CLI는 stdin JSON을 읽고 표준 schema로 변환한다.
4. CLI는 body canonical JSON, timestamp, nonce, sequence를 HMAC-SHA256으로 서명한다.
5. 서버는 다음 헤더를 검증한다.
   - `X-BuilderGate-Session`
   - `X-BuilderGate-Timestamp`
   - `X-BuilderGate-Nonce`
   - `X-BuilderGate-Signature`
6. timestamp 허용 오차는 30초 내외, nonce cache TTL은 2분 내외로 둔다.
7. payload 크기는 8KB 수준으로 제한한다.
8. session/source 단위 rate limit을 둔다.
9. 세션 종료 시 hook secret을 즉시 폐기한다.

endpoint는 loopback 전용 local HTTPS 또는 별도 loopback HTTP receiver가 적합하다. BuilderGate의 일반 app endpoint가 self-signed HTTPS 뒤에만 있으면 Claude/Codex/Hermes hook에서 TLS 검증 문제가 생길 수 있다. 현실적인 1차 구현은 `buildergate-hook` CLI가 서버와 통신하고 agent hook 설정은 CLI만 호출하게 하는 방식이다.

## 7. Hook CLI 전략

agent hook에서 직접 HTTP를 호출하는 방식보다 `buildergate-hook` CLI를 두는 방식이 낫다.

장점:

- Codex, Claude Code, Hermes의 raw event 차이를 CLI에서 흡수할 수 있다.
- Windows PowerShell quoting과 UTF-8 stdin JSON 파싱을 통제할 수 있다.
- HMAC 서명, retry, timeout, payload redaction을 한 곳에서 처리할 수 있다.
- agent hook 설정은 짧고 안정적으로 유지된다.

CLI 기본 동작:

```text
buildergate-hook --source codex
buildergate-hook --source claude-code
buildergate-hook --source hermes --event <raw-event>
```

동작 순서:

1. stdin 전체를 UTF-8 JSON으로 읽는다.
2. `BUILDERGATE_SESSION_ID`, `BUILDERGATE_HOOK_URL`, `BUILDERGATE_HOOK_TOKEN`을 확인한다.
3. source별 adapter가 raw hook payload를 표준 `SessionActivitySignal`로 변환한다.
4. 1-5초 timeout으로 서버에 제출한다.
5. 실패해도 agent 작업을 막지 않는다.
6. 실패는 stderr에 과도하게 노출하지 않고 서버 debug 또는 local minimal log에 남긴다.

## 8. Frontend UX 정책

1차 구현에서는 UI를 바꾸지 않는다.

- 기존 sidebar running count 유지
- 기존 tab/grid `terminal-running` class 유지
- 기존 WS `status` 메시지 유지
- `waiting` / `attention`은 서버 debug capture 또는 나중의 optional `session:activity` WS 메시지로만 노출

2차 확장으로는 다음을 고려할 수 있다.

- permission prompt 또는 action required일 때 작은 보조 dot 표시
- tooltip에 `source=claude-code`, `event=PermissionRequest`, `confidence=high` 표시
- debug panel에 hook signal accepted/rejected/merged 이벤트 표시
- status flapping이 심한 세션에 stale hook warning 표시

주의: 기본 status enum을 `idle | running | waiting | attention`으로 즉시 확장하면 프론트 타입, grid layout, running count, breathing CSS, workspace badge가 모두 영향을 받는다. 따라서 1차 구현에서는 피한다.

## 9. 구현 단계 제안

Phase 1: SRS/계약

- hook signal 요구사항을 SRS에 추가한다.
- `idle/running` WS 계약 불변을 명시한다.
- session-scoped HMAC, TTL, sequence, nonce 요구사항을 명시한다.

Phase 2: 서버 타입과 검증

- `SessionActivitySignal` 타입 추가
- schema validator 추가
- HMAC verifier, nonce cache, sequence guard 추가
- payload/rate limit 추가

Phase 3: SessionManager 병합

- `SessionManager.applyExternalActivitySignal(sessionId, signal)` 추가
- `externalActivityState` 또는 active scope map 추가
- `running`, `idle candidate`, `waiting`, `attention`, `stale` 병합 정책 구현
- debug capture event 추가

Phase 4: route/service

- `HookSignalService` 추가
- local hook endpoint 추가
- 일반 auth endpoint와 hook signal endpoint 권한을 분리한다.

Phase 5: buildergate-hook CLI

- source adapter: Codex, Claude Code, Hermes
- PowerShell/Windows UTF-8 stdin JSON 처리
- HMAC 서명
- 짧은 timeout과 non-blocking failure

Phase 6: agent별 설정 문서/프리셋

- Codex config TOML 예시
- Claude Code settings.json 예시
- Hermes cli-config.yaml hook 예시
- BuilderGate 세션 환경변수 주입 방식 설명

Phase 7: optional frontend debug

- 기존 UI 변경 없이 debug event 확인
- 필요 시 feature flag 뒤에 `session:activity` additive WS 메시지 추가

## 10. 테스트 전략

서버 단위 테스트:

- schema validation: 필수 필드 누락, 잘못된 enum, ttl 범위 초과, payload size 초과 reject
- HMAC: 정상 서명 통과, body 변조 실패, timestamp 만료 실패, nonce replay 실패
- sequence: 증가 sequence 통과, 동일/감소 sequence reject, source별 독립 처리
- `tool-start -> running`
- `tool-stop -> idle delay 후 idle`
- `permission-prompt -> display idle + attention`
- `waiting-input -> idle 유지`
- `UserPromptSubmit/turn-pending`만으로 running 확정 금지
- `subagent-start` 후 parent stop이 와도 running 유지
- `subagent-stop`으로 마지막 scope 닫힌 뒤 idle delay
- TTL 만료로 누락 stop 회복
- duplicate start/stop idempotent
- stale idle이 최신 running을 덮지 못함
- hook 없음 상태에서 기존 Hermes/Codex/Claude idle invariant 테스트 유지

기존 회귀 테스트 보강 위치:

- `server/src/test-runner.ts`의 Codex/Claude/Hermes idle invariant 테스트 묶음
- `SessionManager` foreground harness
- route를 추가하면 session route HTTP 테스트 묶음

프론트 회귀:

- 기존 `status` WS payload shape 불변
- sidebar running count 유지
- grid auto layout이 기존 status에만 반응
- `waiting/attention` 보조 정보가 없을 때 기존 UI 변화 없음

E2E:

- Codex: prompt submit, tool use, permission request, stop, compaction
- Claude Code: PreToolUse/PostToolUse/Stop, permission prompt, SubagentStart/SubagentStop
- Hermes: hook 미설정 fallback, hook running, hook stop 누락 TTL, detector semantic output 우선
- 모든 E2E는 `https://localhost:2002`를 대상으로 한다.

## 11. 주요 위험

| 위험 | 영향 | 완화 |
| --- | --- | --- |
| `UserPromptSubmit`을 running으로 과신 | 입력만 해도 running 오판 | turn-pending 후보로만 처리 |
| `Stop` 누락 | running 고착 | TTL과 heartbeat |

## 12. 서브에이전트 취합 결과

Codex 연구:

- Codex hooks는 공식적으로 지원된다.
- 주요 event는 `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop`이고 공개 schema 기준 compaction event도 확인된다.
- `UserPromptSubmit`은 running 후보로 볼 수 있으나, 본 연구는 BuilderGate invariant를 우선해 turn-pending으로 낮췄다.

Claude Code 연구:

- status line보다 hooks가 상태 판정에 적합하다.
- `Stop`은 강한 idle 후보지만 interrupt나 failure에서는 누락/대체 event가 있을 수 있다.
- `SubagentStart/Stop`은 counter/scope로 처리해야 한다.
- HTTP hook은 local endpoint 연동에 유리하나 실패는 non-blocking으로 처리해야 한다.

Hermes 연구:

- 로컬 코드에는 Hermes 외부 hook 통합 근거가 없다.
- 현재 구현은 `HermesForegroundDetector` 기반이다.
- 공식 Hermes 문서에는 shell/plugin hooks가 있다.
- 따라서 Hermes hook은 opt-in adapter로 두고 기존 detector fallback을 반드시 유지해야 한다.

현재 파이프라인 연구:

- `SessionManager`가 상태 source of truth다.
- `ForegroundAppDetector.ts`의 derived state 모델이 hook signal을 병합하기 좋은 내부 표현이다.
- 기존 WS `status` payload는 유지하는 것이 가장 안전하다.

아키텍처 연구:

- active scope, TTL, sequence, HMAC, nonce, rate limit이 필요하다.
- `waiting/attention`은 보조 상태로 두고 display status는 기존 `idle/running`을 유지한다.
- hook CLI 실패는 작업 자체를 막지 않아야 한다.
