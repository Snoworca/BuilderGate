# Grid Cell Slot Reorder DnD Research

Date: 2026-04-22  
Scope: grid mode에서 기존 `react-mosaic` split DnD 대신, 셀 전체 target guide + slot reorder semantics로 바꿀 수 있는지 조사

---

## Requirement Understanding

사용자 요구를 다음처럼 해석했다.

- 현재처럼 `top/bottom/left/right` split 조각에 파란 guide가 뜨는 UX는 원하지 않는다.
- 드래그 중 guide는 **대상 세션 셀 전체**를 파란 반투명으로 덮어야 한다.
- drop 시 동작은 split 생성이 아니라 **reorder(move/insert)** 여야 한다.
- 예:
  - 현재 순서가 `[1,2,3,4,5]`
  - `1`을 `5`가 있는 위치로 보내면 결과는 `[2,3,4,5,1]`
  - 즉 swap이 아니라 source 제거 후 target 위치 삽입이다.
- 이 요구는 특히 `equal` 모드에서 자연스럽지만, 본질적으로는 “현재 보이는 셀 슬롯 기준 reorder”에 가깝다.

---

## 조사 방식

3개의 서브에이전트를 병렬 투입했다.

- `Hilbert`: `react-mosaic-component` 내부 DnD 모델 조사
- `Descartes`: 현재 앱 코드 결합 구조와 수정 지점 조사
- `Lorentz`: UX/알고리즘/명세/테스트 관점 조사

그 뒤 핵심 주장만 서로에게 다시 보내 짧은 2차 토론을 수행했다.

토론 결과:

- `Hilbert`: 현재 `path + position` drop model은 binary split 삽입 전용이라 slot reorder 의미를 직접 담지 못한다고 재확인
- `Descartes`: 앱 레벨에서 reorder 결과 흡수는 가능하지만, full-cell guide 자체는 라이브러리 기본 모델 밖이라고 재확인
- `Lorentz`: 요구사항은 split DnD보다 `move` 기반 slot reorder UX라고 재확인

즉, 세 관점 모두 큰 틀에서 같은 결론으로 수렴했다.

## Current Facts

### 1. `react-mosaic` 기본 DnD는 split semantics다

- 각 window는 4개의 `MosaicDropTarget`만 만든다.
  - [MosaicWindow.tsx](../../frontend/node_modules/react-mosaic-component/src/MosaicWindow.tsx)
- drop payload는 `{ path, position }` 뿐이다.
  - [MosaicDropTarget.tsx](../../frontend/node_modules/react-mosaic-component/src/MosaicDropTarget.tsx)
- drop 후 실제 업데이트는 `createDragToUpdates()`로 split parent를 만드는 흐름이다.
  - [MosaicWindow.tsx](../../frontend/node_modules/react-mosaic-component/src/MosaicWindow.tsx)
  - [mosaicUpdates.ts](../../frontend/node_modules/react-mosaic-component/src/util/mosaicUpdates.ts)

이 구조는 “타겟 셀 전체를 선택해 그 슬롯에 삽입”이 아니라, “타겟 leaf의 어느 방향으로 split할지”를 고르는 모델이다.

### 2. 현재 앱은 DnD 결과를 사후 흡수하는 구조다

- grid DnD 진입/사후 처리의 중심은 [MosaicContainer.tsx](../../frontend/src/components/Grid/MosaicContainer.tsx)다.
- 현재도 `equal` 모드에서는 drop 후 leaf 순서를 다시 읽어 `buildEqualMosaicTree()`로 재구성한다.
  - 즉, **결과 reorder를 앱 레벨에서 다시 해석하는 패턴 자체는 이미 존재한다.**
- tree 생성/복구/정규화 유틸은 [mosaic.ts](../../frontend/src/utils/mosaic.ts)에 몰려 있다.
- persistence는 [useMosaicLayout.ts](../../frontend/src/hooks/useMosaicLayout.ts)가 tree 자체를 저장/복원한다.

### 3. 현재 저장 모델은 “leaf order를 tree 안에 담는 방식”과 잘 맞는다

- 별도 `slotOrder` 스키마를 지금 당장 추가하지 않아도, reorder 결과를 새 tree로 만들어 저장하는 것은 가능하다.
- 다만 명시적 슬롯 메타를 따로 들고 싶다면 `schemaVersion` 확장이 필요하다.

### 4. 기존 선형 reorder UX 참고 구현이 이미 있다

- [useDragReorder.ts](../../frontend/src/hooks/useDragReorder.ts)는 탭바/사이드바에서 선형 목록 reorder를 처리한다.
- 다만 이 훅은 midpoint 기반이고, “셀 전체 target + head/tail sentinel + move semantics” 요구에는 그대로 맞지 않는다.

## What Is Feasible

### 가능한 것

- drop 결과를 **slot reorder 결과**로 다시 해석하고, 그 결과 leaf order로 tree를 재구성하는 것
- drag 중 **셀 전체 blue overlay guide**를 앱 레벨에서 직접 그리는 것
- `equal` 모드에서 reorder 후 canonical grid를 다시 적용하는 것
- `none/focus/auto` 모드에서도 reorder 후 각 모드 규칙을 다시 적용하는 것

### 바로 안 되는 것

- CSS만 만져서 기본 `react-mosaic` guide를 “의미까지” full-cell reorder guide로 바꾸는 것
- 기본 `{path, position}` drop result만으로 “앞뒤 밀기” semantics를 정확히 표현하는 것

## Implementation Options

## Option A: `react-mosaic` 내부 patch/fork

설명:

- `MosaicDropTarget`, `MosaicWindow`, `mosaicUpdates.ts`를 패치해
  - full-cell target
  - reorder payload
  - reorder update 생성
  를 라이브러리 내부에서 처리한다.

장점:

- guide hit-test와 drop semantics가 한 곳에서 일관된다
- 라이브러리 기본 DnD 프리뷰와 충돌이 적다

단점:

- vendor coupling이 크다
- 업데이트 추적과 patch 유지 비용이 생긴다
- binary tree 라이브러리를 reorder 엔진처럼 바꾸는 부담이 크다

판단:

- 가장 “정직한” 방법이지만 유지보수 비용이 높다.

## Option B: 앱 소유 reorder drag layer 추가

설명:

- grid 모드에서만 `react-mosaic` 기본 DnD를 사실상 우회한다.
- 앱이 직접:
  - drag start
  - 현재 보이는 cell rect 측정
  - hovered slot 계산
  - full-cell overlay guide 표시
  - drop 시 `move` 연산
  - leaf order 기반 tree 재구성
  을 처리한다.

장점:

- 사용자 요구 semantics를 가장 직접적으로 구현할 수 있다
- 라이브러리 fork를 피할 수 있다
- `equal` 전용으로 먼저 도입하고, 나중에 `none/focus/auto`로 확장하기 쉽다

단점:

- 기존 mosaic DnD 경로와 이중화된다
- 현재 그립 drag를 `react-dnd`에서 떼어내거나 별도 경로로 분기해야 한다
- cell rect hit-test, overlay, commit 경계를 앱에서 다 책임져야 한다

판단:

- **현재 코드베이스 기준 최우선 권장안**

## Option C: 현재 mosaic DnD 유지 + 사후 결과만 재해석

설명:

- 기본 split guide/target은 그대로 쓰고
- drop 후 결과 tree만 읽어 reorder 결과로 다시 바꾼다

장점:

- 코드 변경 범위가 가장 작다

단점:

- 사용자가 원하는 “셀 전체 blue guide”를 달성하지 못한다
- 드래그 중 UX와 최종 결과가 계속 불일치한다

판단:

- 요구사항 핵심을 충족하지 못하므로 비권장

---

## Recommended Direction

## 권장안: Option B, 단 `equal` 모드부터 시작

권장 이유:

- 요구사항의 본질은 split DnD 개선이 아니라 **slot reorder UX 도입**이다.
- 그러므로 binary split 전용인 `react-mosaic` 기본 DnD 의미를 억지로 늘리는 것보다,
  **앱이 직접 reorder drag를 소유하는 편이 더 명확하다.**
- 현재 코드도 이미 drop 결과를 사후 정규화하는 구조가 있어서, 이 방향과 잘 맞는다.

권장 단계:

1. `equal` 모드에서만 custom reorder drag를 도입  
2. full-cell blue guide overlay를 앱 레벨에서 그림  
3. drop 시 `move`로 leaf order를 재구성  
4. 그 결과를 `buildEqualMosaicTree()`로 다시 canonical grid로 변환  
5. 이후 필요하면 `none/focus/auto`에 확장

이 접근이면 사용자 요구를 가장 적은 vendor coupling으로 검증할 수 있다.

## Required Spec Clarifications

구현 전에 아래는 명시가 필요하다.

### 1. 실제 입력 장치

- 좌클릭 drag 기준인지
- 우클릭 drag/drop인지

현재 코드베이스와 UX 상식상 **좌클릭 drag**, 우클릭은 컨텍스트 메뉴로 분리하는 쪽이 자연스럽다.

### 2. target 셀 의미

- “셀 전체가 target”이면 그 셀 **앞에 삽입**인지
- 그 셀 **뒤에 삽입**인지
- 또는 target 셀 중앙 기준으로 before/after를 다시 나눌지

사용자 예시처럼 `1 -> 5 => [2,3,4,5,1]`을 정확히 만족하려면, 사실상 **tail sentinel** 개념이 필요하다.

### 3. 모드 범위

- 이 UX를 `equal`만 바꾸는지
- `none/focus/auto`까지 공통으로 바꾸는지

현실적으로는 **equal 전용 1차 도입**이 가장 안전하다.

### 4. source 시각화

- 드래그 중 source는 ghost/opacity로 남길지
- 즉시 비울지

권장안은 ghost/반투명 유지다.

## Suggested Algorithm

권장 알고리즘은 다음과 같다.

1. source cell의 current slot index를 구한다
2. 드래그 중 현재 pointer가 올라간 target slot index를 계산한다
3. commit 시 `move(sourceIndex, targetInsertionIndex)`를 수행한다
4. 결과 leaf order를 모드별로 tree에 반영한다

핵심은 `swap`이 아니라 `move`다.

예:

```ts
function move<T>(items: T[], from: number, to: number): T[] {
  const next = [...items];
  const [moved] = next.splice(from, 1);
  next.splice(to, 0, moved);
  return next;
}
```

그 다음 모드별 반영은 이렇게 분리하는 것이 맞다.

- `equal`: reordered leaf order -> canonical equal grid 재구성
- `none`: reordered leaf order -> 현재 topology 최대 보존 또는 별도 reorder tree rebuild
- `focus`: reordered tree -> `applyFocusMode()`
- `auto`: reordered tree -> `applyMultiFocusApprox()`

## Test Strategy

자동화와 수동 검증을 같이 가져가야 한다.

### 자동화 E2E

추가 권장 케이스:

- `1 -> 5`
- `5 -> 1`
- `middle -> middle`
- self drop no-op
- first sentinel / last sentinel
- equal 모드에서 reorder 후 canonical grid 유지
- none 모드에서 reorder 후 manual layout 의미 보존

### 수동 검증

- full-cell blue guide가 정확히 target cell 전체를 덮는지
- split guide 잔상이 남지 않는지
- 우클릭이 reorder를 시작하지 않는지
- 드래그 중 source ghost가 자연스러운지

### 디버그 포인트

- `sourceIndex`
- `hoverSlotIndex`
- `commitInsertionIndex`
- `finalLeafOrder`

## Final Assessment

최종 판단:

- **구현 가능**: 예
- **CSS만으로 해결 가능**: 아니오
- **현재 `react-mosaic` 기본 DnD를 거의 그대로 유지하면서 해결 가능**: 부분적으로만
- **요구사항에 가장 맞는 방향**:
  - 앱 소유 custom reorder drag layer
  - full-cell overlay guide
  - `move` semantics
  - `equal` 모드부터 단계 도입

즉, 이 기능은 “현재 split DnD를 조금 다듬는 작업”이 아니라,  
**grid 모드용 reorder interaction을 별도 설계로 도입하는 작업**으로 보는 것이 맞다.
