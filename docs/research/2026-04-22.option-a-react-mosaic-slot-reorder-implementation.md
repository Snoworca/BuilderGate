# Option A Implementation Research: React Mosaic Internal Patch For Slot Reorder

Date: 2026-04-22  
Scope: `react-mosaic-component` 내부 patch/fork를 통해 grid mode에서 full-cell guide + slot reorder semantics를 구현하는 방법 연구

---

## Goal

이 문서는 이전 조사에서 정의한 `Option A`를 실제로 구현한다면 어떤 방식으로 가야 하는지 정리한다.

Option A의 의미:

- `react-mosaic-component` 내부 DnD 모델을 패치한다
- 현재의 `split insertion` semantics 대신
  - 셀 전체 target
  - reorder payload
  - move/insert semantics
  를 library 내부에서 직접 다룬다

---

## Research Method

시니어 개발자 역할의 서브에이전트 3명을 병렬 투입했다.

- `Hilbert`
  - `react-mosaic-component` 내부 DnD 구조와 patch 지점 조사
- `Descartes`
  - 현재 앱 코드와의 결합 구조, 저장/복원/모드 영향 조사
- `Lorentz`
  - UX/알고리즘/검증 관점에서 Option A 구현 조건 정리

그 뒤 각 결과의 핵심 주장만 다시 서로에게 보내 2차 짧은 토론을 수행했다.

토론 결과:

- `react-mosaic`의 기본 `{path, position}` 모델은 binary split 삽입 전용이라는 점에 세 명 모두 동의
- 앱 레벨에서 결과 흡수는 가능하지만, **셀 전체 guide + reorder semantics**를 일관되게 만들려면 내부 patch 또는 별도 overlay가 필요하다는 점에 동의
- `swap`이 아니라 `move/insert` semantics를 명세로 고정해야 한다는 점에 동의

## Local Repository Facts

### 1. 현재 저장소에는 patch 관리 인프라가 없다

- [frontend/package.json](../../frontend/package.json)에는 `patch-package`가 없다.
- 저장소 전체에 `patch-package`, `patches/`, `pnpm patch`, `yarn patch` 흔적이 없다.

즉 Option A를 채택하면 먼저 **vendor patch 관리 방식**부터 정해야 한다.

### 2. 현재 `react-mosaic-component`는 소스/빌드 산출물을 같이 가진다

- [frontend/node_modules/react-mosaic-component/package.json](../../frontend/node_modules/react-mosaic-component/package.json)
- `main`: `lib/index.js`
- `typings`: `lib/index.d.ts`
- `src/`와 `styles/`도 같이 존재한다

즉 patch는 두 방식 중 하나다.

- `node_modules` 기준 `lib/`와 CSS 산출물에 직접 patch
- 또는 upstream source를 fork해 다시 build한 결과물을 사용

## Core Constraint

현재 `react-mosaic` DnD는 다음 고정 경로를 가진다.

1. 각 `MosaicWindow`가 `top/bottom/left/right` 4개의 `MosaicDropTarget`을 렌더링
2. `MosaicDropTarget`은 `{ path, position }`만 drop result로 반환
3. `MosaicWindow.end()`가 `createDragToUpdates(...)`를 호출
4. `createDragToUpdates()`가 split parent를 새로 만들며 tree를 갱신

관련 파일:

- [MosaicWindow.tsx](../../frontend/node_modules/react-mosaic-component/src/MosaicWindow.tsx)
- [MosaicDropTarget.tsx](../../frontend/node_modules/react-mosaic-component/src/MosaicDropTarget.tsx)
- [internalTypes.ts](../../frontend/node_modules/react-mosaic-component/src/internalTypes.ts)
- [mosaicUpdates.ts](../../frontend/node_modules/react-mosaic-component/src/util/mosaicUpdates.ts)
- [mosaic.less](../../frontend/node_modules/react-mosaic-component/styles/mosaic.less)

즉 Option A의 본질은 “guide CSS를 키운다”가 아니라,
**drop payload와 update semantics를 split에서 reorder로 바꾸는 일**이다.

## What Must Change In Option A

## 1. Drop target model

현재:

- 4방향 target
- partial split region
- payload: `{ path, position }`

Option A 후:

- full-cell target
- 필요 시 head/tail sentinel target 추가
- payload 예시:

```ts
type ReorderDropData = {
  path: MosaicPath;
  slotIndex: number;
  insertion: 'before' | 'after' | 'replace-slot';
};
```

주의:

- 사용자 예시 `1 -> 5 => [2,3,4,5,1]`를 정확히 만족하려면
  “셀 전체에 drop”만으로는 끝 삽입이 모호할 수 있다.
- 따라서 **마지막 셀 뒤 sentinel** 또는 명시적 `after` semantics가 필요하다.

## 2. Drag end update path

현재:

- `MosaicWindow.end()`가 바로 `createDragToUpdates()` 실행

Option A 후:

- `MosaicWindow.end()`는 reorder payload를 읽어
- split update가 아니라 `createReorderUpdates()` 또는 `createReorderedTree()`를 호출해야 한다

권장:

- 기존 `createDragToUpdates()`는 유지
- 새 함수 추가:

```ts
createReorderTree(root, sourcePath, targetPath, insertionMode)
```

이 함수는:

1. leaf order 추출
2. source leaf 제거
3. target slot 기준 삽입
4. 새 tree 재구성

을 수행해야 한다.

## 3. Guide rendering

현재:

- `.drop-target`는 `30%`
- hover 시 `50%`
- split semantics를 전제로 한 partial overlay

Option A 후:

- target cell 전체를 덮는 overlay
- source는 ghost/opacity 유지
- split zone 시각은 제거 또는 비활성화

주의:

- 이것도 CSS만으로는 부족하다
- hit-test와 payload도 같이 바뀌어야 한다

## Recommended Option A Shape

Option A 안에서도 구현 형태는 두 가지가 있다.

## Shape A1: patch-package 기반 vendor patch

구성:

- `patch-package` 도입
- `postinstall`에서 `react-mosaic-component` patch 적용
- 변경 대상:
  - `frontend/node_modules/react-mosaic-component/src/...`
  - 필요 시 `lib/...`
  - `react-mosaic-component.css`

장점:

- 현재 저장소에 가장 빨리 붙일 수 있다
- upstream 의존성 버전이 고정돼 있으면 관리가 단순하다
- 변경 diff를 저장소 안에 남길 수 있다

단점:

- node_modules patch 유지비용이 있다
- upstream minor update 때 patch 재적용 비용이 생긴다
- TypeScript source와 built lib 중 어느 쪽을 patch할지 명확히 정해야 한다

판단:

- **현재 저장소에서 Option A를 실제로 시작한다면 가장 현실적인 시작점**

## Shape A2: vendor fork/subtree

구성:

- `vendor/react-mosaic-component` 같은 내부 복사본 또는 fork를 둔다
- frontend가 그 포크를 직접 import 하게 바꾼다

장점:

- patch-package보다 장기 유지보수성이 낫다
- 테스트/빌드 파이프라인을 내부 통제로 가져올 수 있다

단점:

- 초기 전환 비용이 크다
- import 경로와 bundling 설정 손대는 범위가 넓다

판단:

- 장기적으로는 더 건강하지만, 시작 비용이 크다

## 최종 권장

Option A만 놓고 보면:

- **1차 구현은 A1 (`patch-package`)**
- 장기 유지 필요가 확정되면 **A2로 승격**

이 순서가 맞다.

## Concrete Patch Targets

## Patch target 1: `internalTypes.ts`

현재:

```ts
export type MosaicDropTargetPosition = 'top' | 'bottom' | 'left' | 'right';
```

Option A 방향:

- reorder 전용 payload 타입 추가
- split payload와 공존시키거나, grid 전용 branch로 분리

예:

```ts
export interface MosaicReorderDropData {
  path?: MosaicPath;
  slotIndex?: number;
  insertion?: 'before' | 'after';
}
```

## Patch target 2: `MosaicDropTarget.tsx`

현재:

- `drop()` returns `{ path, position }`
- full-cell이 아닌 4방향 target

Option A 방향:

- window별 full-cell drop target
- 필요 시 first/last sentinel
- hover 시 full-cell class 부여

핵심 질문:

- 기존 4개 target을 제거할지
- reorder mode일 때만 다른 target을 렌더링할지

권장:

- reorder mode prop/context를 추가하고
- 그 모드에서만 full-cell target 렌더링

## Patch target 3: `MosaicWindow.tsx`

현재:

- `renderDropTarget()`이 4방향 target 렌더링
- `end()`에서 `createDragToUpdates()` 호출

Option A 방향:

- reorder mode에서 다른 target set 렌더링
- `end()`에서 reorder payload 분기 처리

권장:

```ts
if (dropResult.kind === 'reorder') {
  mosaicActions.replaceRoot(createReorderedTree(...));
} else {
  mosaicActions.updateTree(createDragToUpdates(...));
}
```

## Patch target 4: `mosaicUpdates.ts`

현재:

- split insertion 전용 helper만 존재

Option A 방향:

- leaf order extraction
- move/insert
- tree rebuild helper 추가

권장 함수:

```ts
createReorderedTree(root, sourcePath, targetPath, insertionMode)
```

세부 규칙:

1. `root`에서 visible leaf order 추출
2. `sourceId` 제거
3. `target slot` 기준 삽입
4. policy에 맞는 tree rebuild

tree rebuild 정책은 두 갈래다.

- balanced rebuild
- 기존 topology 최대 보존 rebuild

Option A에서 가장 중요한 건 **semantics 일치**이므로,
초기 버전은 balanced rebuild가 더 단순하다.

## Interaction Model Recommendation

Option A를 쓰더라도 interaction 명세는 분명해야 한다.

권장 명세:

- 좌클릭 drag 또는 터치 long-press drag만 reorder 시작
- 우클릭은 컨텍스트 메뉴로 유지
- drop 의미는 `move`, never `swap`
- same-slot drop은 no-op
- 마지막 셀 뒤 삽입을 위해 tail sentinel 제공

예:

- `[1,2,3,4,5]`
- source `1`
- target `5-after`
- result `[2,3,4,5,1]`

## Integration With Current App

현재 앱에서 Option A가 들어오면 가장 크게 바뀌는 곳:

- [MosaicContainer.tsx](../../frontend/src/components/Grid/MosaicContainer.tsx)
- [mosaic.ts](../../frontend/src/utils/mosaic.ts)
- [useMosaicLayout.ts](../../frontend/src/hooks/useMosaicLayout.ts)

하지만 Option A의 핵심은 library 쪽이므로, 앱 쪽 책임은 아래 정도로 좁히는 게 맞다.

- mode별 후처리
  - `equal`: canonical equal grid 재적용
  - `none`: reorder 결과 유지
  - `focus`: reorder 후 `applyFocusMode`
  - `auto`: reorder 후 `applyMultiFocusApprox`
- persistence
- E2E verification

즉, reorder semantics를 library가 처리하고, mode semantics는 앱이 처리하는 분리가 가장 건강하다.

## Risks

## 1. Binary tree vs slot reorder mismatch

`react-mosaic`는 본질적으로 binary tree다.
slot reorder는 list semantics다.

즉, Option A는 라이브러리 본래 추상화를 확장하는 작업이라 항상 유지비가 따른다.

## 2. Upstream update cost

`react-mosaic-component` 버전이 바뀌면 patch 충돌 가능성이 높다.

## 3. Equal vs non-equal behavior divergence

사용자는 같은 drag gesture를 보지만, 실제 후처리는 모드별로 다를 수 있다.
이걸 문서/테스트로 명확히 관리해야 한다.

## 4. Sentinel semantics

“셀 전체 target”만으로는 끝 삽입/앞 삽입이 모호하다.
tail/head sentinel을 명세에 포함하지 않으면 UX 논쟁이 반복될 가능성이 높다.

## Test Strategy For Option A

Option A는 patch 자체 테스트와 앱 통합 테스트를 나눠야 한다.

## A. Library patch level

최소 확인:

- full-cell hover class가 켜지는지
- drop payload가 reorder payload로 생성되는지
- `createReorderedTree()`가 move semantics를 지키는지

핵심 케이스:

- `1 -> 5`
- `5 -> 1`
- `middle -> middle`
- same-slot no-op
- tail sentinel insert

## B. App integration level

현재 프로젝트 기준 E2E 추가 후보:

- equal 모드에서 reorder 후 canonical 2행/2열 유지
- none 모드에서 reorder 후 manual ratio 의미 유지
- focus/auto 모드에서 reorder 후 mode transform 재적용
- persistence/reload 후 leaf order 보존

---

## Final Recommendation

Option A는 구현 가능하다.

다만 “라이브러리 내부를 조금 손본다” 수준이 아니라,
**`react-mosaic`의 drop payload와 update semantics를 split insertion에서 reorder insertion으로 확장하는 작업**으로 봐야 한다.

현재 저장소 기준 최적 경로:

1. `patch-package` 도입
2. `internalTypes.ts`, `MosaicDropTarget.tsx`, `MosaicWindow.tsx`, `mosaicUpdates.ts` patch
3. full-cell target + tail sentinel + move semantics 구현
4. 앱은 mode 후처리와 persistence만 담당
5. 검증이 끝나면 필요 시 vendor fork로 승격

즉, Option A의 권장 구현 방법은:

- **단기**: `patch-package` 기반 internal patch
- **중기**: 안정화 후 vendor fork 전환 검토

이 방향이 가장 현실적이다.
