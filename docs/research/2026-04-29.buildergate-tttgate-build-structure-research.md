## 1. 요약 결론

BuilderGate를 TTTGate처럼 `dist/bin` 실행 파일과 `dist/web` 정적 자산 중심으로 정리하는 것은 가능하다.

다만 TTTGate도 완전한 단일 파일 배포가 아니다. 현재 TTTGate 산출물은 다음 구조다.

```text
dist/
  bin/
    TTTGate-alpine-x64
    TTTGate-linux-arm64
    TTTGate-linux-x64
    TTTGate-win-arm64.exe
    TTTGate-win-x64.exe
  web/
    index.html
    icon.png
    assets/
```

따라서 BuilderGate의 현실적인 목표는 다음이다.

```text
dist/
  bin/
    BuilderGate-win-x64.exe
    BuilderGate-linux-x64
    BuilderGate-win-arm64.exe
    BuilderGate-linux-arm64
    BuilderGate-macos-arm64
  web/
    index.html
    assets/
  config/
  runtime/
  native/        # node-pty 때문에 필요할 수 있음
```

또는 현재 릴리스 산출물의 target별 디렉터리를 유지한다면 다음처럼 정리할 수 있다.

```text
dist/bin/win-amd64-0.1.0/
  BuilderGate.exe
  web/
  config.json5
  runtime/
  native/        # 필요 시
```

핵심은 `server/dist/**/*.js`, `tools/daemon/**/*.js`, `server/node_modules/.bin/node.exe`가 사용자에게 그대로 노출되는 구조를 없애는 것이다.

## 2. TTTGate 빌드 구조 관찰

### 2.1 package.json

TTTGate는 `pkg`를 사용해 TypeScript 빌드 결과를 실행 파일로 묶는다.

```json
"bin": {
  "app": "build/src/app.js"
},
"pkg": {
  "scripts": "build/**/*.js",
  "targets": [
    "node18-linux-arm64",
    "node18-win-arm64",
    "node18-win-x64",
    "node18-linux-x64",
    "node18-alpine-x64"
  ],
  "outputPath": "dist/bin/"
},
"scripts": {
  "build": "tsc -b",
  "pkg": "pkg . --no-bytecode --public-packages \"*\" --public",
  "dist": "node deploy.js"
}
```

근거: `C:\Work\git\_Snoworca\TTTGate\package.json`

### 2.2 deploy.js

TTTGate `deploy.js`의 핵심 흐름은 다음이다.

1. `dist`, `dist.js` 제거
2. admin/frontend 빌드
3. `admin/dist`를 `dist/web`으로 복사
4. 루트 TypeScript 빌드
5. `dist.js` staging 디렉터리 생성
6. `npm run pkg` 실행

관찰상 최종 사용자 노출 산출물은 `dist/bin` 실행 파일과 `dist/web` 정적 자산이다. `package.json` 기준 실제 `pkg` entry는 `build/src/app.js`이며, `dist.js`는 별도 staging/legacy 성격으로 보인다.

근거: `C:\Work\git\_Snoworca\TTTGate\deploy.js`

### 2.3 런타임 경로 모델

TTTGate는 실행 파일 위치를 기준으로 root를 계산한다.

```ts
const DEFAULT_ROOT_DIR =
  (process.argv[0].includes('node') || process.argv[0].includes('npm'))
    ? process.cwd()
    : Path.join(process.argv[DEV_MODE ? 1 : 0], '..', '..');
```

실행 파일이 `dist/bin/TTTGate-win-x64.exe`에 있으면 root는 `dist`가 된다. 이후 주요 경로는 root 아래로 모인다.

```ts
logDir: Path.join(rootDir, 'logs'),
configDir: Path.join(rootDir, 'config'),
serverCacheDir: Path.join(rootDir, 'cache', 'server'),
clientCacheDir: Path.join(rootDir, 'cache', 'client'),
certDir: Path.join(rootDir, 'cert'),
webDir: Path.join(rootDir, 'web'),
binDir: !DEV_MODE ? Path.join(rootDir, 'bin') : Path.join(rootDir),
```

이 구조 덕분에 TTTGate는 실행 파일이 서버 엔트리 역할을 하면서, 웹 정적 파일과 쓰기 가능한 런타임 데이터만 외부 디렉터리에 둔다.

근거: `C:\Work\git\_Snoworca\TTTGate\src\Environment.ts`

## 3. BuilderGate 현재 빌드 구조

### 3.1 현재 산출물

현재 BuilderGate의 Windows amd64 로컬 산출물은 대략 다음 구조다.

```text
dist/bin/win-amd64-0.1.0/

  BuilderGate.icns
  BuilderGate.svg
  README.md

```

사용자가 지적한 `node.exe`와 여러 JS 파일 노출은 현재 설계상 의도된 결과다.

### 3.2 build-daemon-exe.js

현재 빌드 스크립트는 `pkg`로 전체 서버를 묶지 않고 `tools/start-runtime.js`만 런처 실행 파일로 만든다.

```js
run(npx, [
  '--yes',
  'pkg@5.8.1',
  path.join(ROOT, 'tools', 'start-runtime.js'),
  '--targets',
  target,
  '--output',
  path.join(outputDir, appExeName),
  '--no-bytecode',
  '--public-packages',
  '*',
  '--public',
], { label: 'pkg daemon launcher' });
```

그리고 별도로 다음 파일들을 산출물에 복사한다.

- `server/dist`
- `server/package.json`
- `server/package-lock.json`
- `tools/daemon/*.js`
- `README.md`
- `config.json5`
- icon assets

또한 대상 플랫폼의 Node 런타임을 `server/node_modules/.bin/node.exe` 또는 `node`로 복사한다.

```js
const runtimeBinDir = path.join(outputServerDir, 'node_modules', '.bin');
const runtimeNodeName = platform === 'win32' ? 'node.exe' : 'node';
const runtimeNodeTarget = path.join(runtimeBinDir, runtimeNodeName);
fs.copyFileSync(nodeRuntimePath, runtimeNodeTarget);
```

근거: `tools/build-daemon-exe.js`

### 3.3 runtime-paths.js

패키징된 BuilderGate는 root를 실행 파일 디렉터리로 본다.

```js
if (options.isPackaged ?? Boolean(process.pkg)) {
  return path.dirname(options.execPath ?? process.execPath);

```

그리고 서버 엔트리와 node runtime을 다음처럼 계산한다.

```js
serverDir: path.join(root, 'server'),

sentinelEntry: path.join(root, 'tools', 'daemon', 'sentinel-entry.js'),

```

### 3.4 launcher.js

현재 런처는 패키징 모드에서 bundled Node를 필수로 요구한다.

```js
function resolveAppNodeCommand(paths = resolveRuntimePaths()) {
  if (fs.existsSync(paths.nodeBin)) {
    return paths.nodeBin;
  }

  if (paths.isPackaged ?? Boolean(process.pkg)) {
    throw new Error(`Bundled Node runtime missing: ${paths.nodeBin}`);
  }

  return process.execPath;
}
```

foreground/app 실행도 `node server/dist/index.js` 형태다.

```js
return {
  command: resolveAppNodeCommand(paths),
  args: [paths.serverEntry],
  options: {
    cwd: paths.serverDir,
    ...
  },
};
```

## 4. TTTGate와 BuilderGate의 결정적 차이

| 항목 | TTTGate | BuilderGate 현재 |
|------|---------|------------------|
| 실행 파일 역할 | 앱 서버 엔트리 자체 | `tools/start-runtime.js` 런처 |
| 서버 실행 방식 | pkg 내부 코드 실행 | 외부 `node.exe server/dist/index.js` 실행 |
| 웹 정적 파일 | `dist/web` | `server/dist/public` |
| 노출 JS 파일 | 최종 산출물에 거의 없음 | `server/dist`, `tools/daemon` 노출 |
| bundled Node | 별도 `node.exe` 없음 | `server/node_modules/.bin/node.exe` 필수 |
| native addon | 주요 native 부담 없음 | `node-pty` native prebuild 필요 |
| 패키징 난이도 | 낮음 | 중간-높음 |

BuilderGate에서 `node.exe`가 보이는 근본 원인은 `pkg` 실행 파일이 실제 서버를 포함하지 않고, 서버를 별도 Node 프로세스로 실행하는 런처이기 때문이다.

## 5. 가능성 판단

### 5.1 가능한 것

다음은 가능하다.

1. `node.exe`를 최종 산출물에서 제거
2. `server/dist/**/*.js` 노출 제거 또는 최소화
3. `tools/daemon/**/*.js` 노출 제거
4. `dist/web` 중심 정적 자산 구조 도입
5. `BuilderGate.exe --foreground`, `BuilderGate.exe stop` 같은 기존 CLI 표면 유지
6. target별 release asset 이름 유지

### 5.2 제한적이거나 비추천인 것

다음은 기술적으로는 시도 가능하지만 추천하지 않는다.

1. 완전한 단일 exe 하나에 프론트엔드 assets까지 모두 내장
2. `node-pty` native addon까지 전부 exe 내부에서만 로딩
3. 쓰기 가능한 config/log/cert/runtime 데이터를 exe 내부에 저장

Node의 단일 실행 파일 방식이나 `pkg` virtual filesystem은 정적 읽기 자산에는 맞지만, 네이티브 addon과 쓰기 가능한 런타임 데이터까지 완전히 감추는 데는 제약이 있다.

## 6. 권장 목표 구조

### 6.1 1차 목표

1차 목표는 "진짜 단일 파일"이 아니라 "사용자에게 노출되는 배포 구조 단순화"다.

```text
BuilderGate-<target>-<version>/
  BuilderGate.exe          # 또는 buildergate
  BuilderGate.app          # macOS arm64 추가 제공
  web/
    index.html
    assets/
  config.json5
  config.json5.example
  runtime/
  README.md
  BuilderGate.ico
  BuilderGate.icns
  BuilderGate.svg
```

`node-pty` 문제로 필요한 경우에만 다음 디렉터리를 허용한다.

```text
  native/
    node-pty/
      prebuilds/
```

### 6.2 최종 이상형

TTTGate와 거의 같은 루트 구조를 목표로 할 수 있다.

```text
dist/
  bin/
    buildergate
    BuilderGate.exe
  web/
  config/
  runtime/
  native/
```

다만 현재 release workflow와 SRS가 target별 디렉터리/압축 파일을 전제로 하므로, 즉시 전환은 target별 디렉터리 내부를 정리하는 방식이 더 안전하다.

## 7. 필요한 설계 변경

### 7.1 동일 exe 내부 모드 실행

현재:

```text
BuilderGate.exe
  -> bundled node.exe
    -> server/dist/index.js
```

권장:

```text
BuilderGate.exe
  -> process.execPath --internal-app
```

sentinel도 동일하게 바꾼다.

현재:

```text
BuilderGate.exe
  -> bundled node.exe tools/daemon/sentinel-entry.js
```

권장:

```text
BuilderGate.exe
  -> process.execPath --internal-sentinel
```

이미 `launcher.js`에는 `BUILDERGATE_INTERNAL_MODE=app|sentinel` 개념이 있으므로, 방향성은 기존 구조와 충돌하지 않는다. 다만 현재 packaged mode의 command가 `paths.nodeBin`으로 고정되어 있어 이 부분을 바꿔야 한다.

### 7.2 서버 엔트리 패키징

현재 서버는 `server/package.json`에서 `"type": "module"`이고, `server/dist/index.js`를 직접 Node로 실행한다.

TTTGate형 구조로 가려면 서버를 `pkg`가 안정적으로 추적할 수 있는 패키징 엔트리로 모아야 한다.

후보:

1. `server/src/index.ts`에서 서버 시작 로직을 함수로 분리
2. `tools/start-runtime.js`에서 내부 app mode일 때 해당 함수 호출
3. 또는 `esbuild`/`ncc`로 서버를 CommonJS 단일 bundle로 만든 뒤 `pkg` entry에 포함

`pkg`는 정적 require 그래프에 강하므로, ESM/dynamic import가 많은 구조는 별도 패키징 entry를 두는 편이 안전하다.

### 7.3 정적 자산 경로 분리

현재:

```text
server/dist/public/
```

권장:

```text
web/
```

서버는 패키징 모드에서 `root/web`을 정적 자산 루트로 보아야 한다. 소스 개발 모드에서는 기존 `server/dist/public` 또는 Vite reverse proxy 흐름을 유지한다.

### 7.4 writable 데이터 유지

다음은 exe 내부에 넣으면 안 된다.

- `config.json5`
- daemon state
- logs
- TOTP secret
- TLS certificate
- shell integration cache/state

TTTGate도 `logs`, `config`, `cache`, `cert`, `web`, `bin`을 root 아래 외부 디렉터리로 유지한다. BuilderGate도 같은 원칙을 따라야 한다.

### 7.5 node-pty native 처리

BuilderGate의 가장 큰 차이는 `node-pty`다.

현재 `server/node_modules/node-pty/prebuilds`에는 다음 유형의 파일들이 존재한다.

- `*.node`
- Windows `winpty-agent.exe`
- Windows `winpty.dll`
- Windows `conpty/OpenConsole.exe`
- macOS `spawn-helper`
- debug symbol `*.pdb`

`pkg`가 `.node` native addon을 다룰 수는 있지만, 런타임에 파일 시스템으로 추출되거나 별도 경로가 필요할 수 있다. 따라서 완전 은닉보다 다음 중 하나가 현실적이다.

1. `pkg` assets로 `node-pty/prebuilds`를 포함하고 런타임 로딩 검증
2. `native/node-pty/prebuilds`만 외부 디렉터리로 유지
3. target별 필요한 prebuild만 선별 복사

권장안은 2 또는 3이다. 사용자에게 지저분하게 보이는 JS/Node 런타임은 없애되, native runtime만 작고 명확한 디렉터리로 남기는 방식이 안전하다.

## 8. 구현 옵션 비교

### 옵션 A: pkg 유지 + 동일 exe 내부 모드

가장 현실적인 1차안이다.

장점:

- 현재 빌드 체인과 가장 가깝다.
- TTTGate와 도구 선택이 같다.
- 기존 `build:*` target을 유지할 수 있다.
- `node.exe` 제거 가능성이 높다.

단점:

- `pkg`는 deprecated/archived 상태다.
- ESM 서버와 native addon 처리에 검증이 필요하다.
- 동적 require/import는 누락 가능성이 있다.

### 옵션 B: esbuild/ncc bundle + pkg

서버 코드를 먼저 단일 CommonJS bundle로 모은 뒤 `pkg`에 넣는 방식이다.

장점:

- 패키징 그래프가 명확해진다.
- 노출 JS 파일을 크게 줄일 수 있다.
- `pkg`가 놓치는 파일을 통제하기 쉽다.

단점:

- `node-pty`는 external 처리해야 한다.
- ESM/CJS 경계 테스트가 필요하다.
- sourcemap, __dirname, import.meta.url 경로 보정이 필요하다.

### 옵션 C: Node SEA

Node 공식 single executable applications 기능을 쓰는 방식이다.

장점:

- Node 공식 기능이다.
- 장기적으로 `pkg` 대체 후보가 될 수 있다.

단점:

- CommonJS/ESM/asset/require 제약을 직접 다뤄야 한다.
- 여러 플랫폼 릴리스 체인이 더 복잡하다.
- native addon 처리 난이도는 여전히 남는다.

### 옵션 D: Bun compile

Bun executable로 바꾸는 방식이다.

장점:

- 단일 executable 경험은 좋다.
- 프론트/백엔드 JS 번들링 속도가 빠르다.

단점:

- BuilderGate의 Node 런타임 계약을 Bun으로 바꾸는 큰 변경이다.
- `node-pty`, WebSocket, TLS, Express, 파일 경로 동작의 호환성 검증 범위가 크다.
- 현재 릴리스 안정화 목표에는 과하다.

## 9. 권장 단계별 계획

### Phase 0: POC 범위 고정

- `win-amd64`만 대상으로 실험
- 현재 릴리스 빌드는 유지
- 새 빌드 스크립트 또는 플래그로만 TTTGate형 산출물 생성

예:

```bash
npm run build:windows-amd64 -- --layout tttgate
```

또는 별도 스크립트:

```bash
node tools/build-daemon-exe.js --profile win-amd64 --clean-layout
```

### Phase 1: 서버 시작 함수 분리

- `server/src/index.ts`의 side effect start를 함수형 entry로 분리
- source mode와 packaged internal mode에서 동일 함수를 재사용
- 기존 `npm run server build/start` 계약은 유지

### Phase 2: 동일 exe internal app/sentinel 실행

- packaged mode에서 `resolveAppNodeCommand()`가 bundled Node를 요구하지 않도록 변경
- app 실행 command를 `process.execPath`로 변경
- sentinel 실행 command도 `process.execPath --internal-sentinel`로 변경
- 상태 파일의 `nodeBinPath` 의미를 재검토

### Phase 3: web root 전환

- packaged mode 정적 자산 루트를 `root/web`으로 변경
- build script는 `frontend/dist`를 `outputDir/web`으로 복사
- source/dev mode는 기존 reverse proxy와 `server/dist/public` 경로를 유지

## 11. 리스크

| 리스크 | 심각도 | 설명 | 대응 |
|--------|--------|------|------|
| `node-pty` native addon 로딩 실패 | High | `.node`, helper executable, DLL 경로 문제 | target별 PTY smoke test 필수 |
| ESM 서버 패키징 실패 | High | 현재 서버는 `"type": "module"` | CJS packaged entry 또는 bundle 도입 |
| daemon/sentinel 생명주기 회귀 | High | 현재는 외부 Node 프로세스 기준 | foreground, daemon, stop, restart 테스트 강화 |
| 정적 자산 경로 회귀 | Medium | `server/dist/public`에서 `web`으로 이동 | packaged/source mode 분기 테스트 |
| pkg 유지보수성 | Medium | `pkg`는 deprecated/archived | 단기 유지, 장기 SEA/esbuild 재검토 |
| macOS app bundle 경로 | Medium | `BuilderGate.app` 내부 runtime 경로와 충돌 가능 | macOS 전용 패키징 테스트 |

## 12. 최종 권고

BuilderGate는 TTTGate형 구조로 전환할 수 있다.

단, 목표를 "완전한 단일 exe"로 잡으면 `node-pty`와 writable runtime data 때문에 비용과 리스크가 커진다. 목표를 다음처럼 잡는 것이 맞다.

1. `node.exe` 제거
2. 사용자 노출 JS 파일 제거
3. `BuilderGate.exe`가 서버/센티널 내부 모드를 직접 실행
4. `web`, `runtime`, `config`, 필요 시 `native`만 외부 디렉터리로 유지

우선 `win-amd64` POC로 `process.execPath --internal-app` 구조를 검증한 뒤, 5개 릴리스 target으로 확장하는 순서를 권장한다.
