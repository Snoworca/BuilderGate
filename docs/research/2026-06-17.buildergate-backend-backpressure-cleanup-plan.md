## 핵심 결론

현재 장시간 사용 시 메모리 증가와 입력 지연을 서버 쪽에서 막으려면 세 가지를 먼저 닫아야 한다.

1. `SessionManager`의 headless output write backlog를 설정 기반 byte/chunk cap이 있는 구조로 바꿔야 한다.
2. `WsRouter`의 모든 송신 경로를 `safeSend`로 통합하고, client별 backpressure 정책을 둬야 한다.
3. 단일 `/ws`에 입력, 출력, snapshot, replay, workspace event가 모두 섞인 구조를 split WebSocket 구조로 점진 전환해야 한다.

사용자의 두 아이디어는 모두 server 설계에 직접 반영하는 것이 맞다.

- 설정 기반 chunk/cache cap: `pty`, `realtime`, `telemetry`, `logging` 설정으로 headless queue, replay/screen-repair queue, snapshot cache, per-client WS queue, debug/observability retention을 모두 byte 기준으로 제한한다.
- 송수신 WebSocket 분리: 기존 `/ws`는 호환 모드로 유지하고, 신규 `/ws/control`과 `/ws/output`을 도입해 입력/제어와 대량 출력/snapshot/replay의 backpressure를 분리한다.

## 문제 요약

### 1. Headless output queue가 unbounded에 가깝다

`SessionManager`는 PTY 출력마다 `queueHeadlessOutput()`을 호출한다. 현재 구조는 다음과 같다.

- `SessionData.pendingOutputChunks: string[]`
- `pendingOutputChunks.push(data)`
- `headlessWriteChain.then(...)`
- write 완료 후 `pendingOutputChunks.shift()`

위 구조의 문제는 세 가지다.

- 큐의 byte/chunk cap이 없다.
- `Array.shift()`는 큐가 커질수록 비용이 커진다.
- cleanup 시 `pendingOutputChunks`, `pendingHeadlessWrites`, `headlessWriteChain`의 의미 있는 종료 상태가 명시적으로 남지 않는다.

현재 `degradedReplayBuffer`, `unsnapshottedOutput`, snapshot serialize는 `pty.maxSnapshotBytes`로 tail cap이 있지만, headless write 대기열 자체는 별도 cap이 없다.

### 2. PTY exit가 heavy resource cleanup과 분리되어 있다

`ptyProcess.onExit()`는 현재 `session:exited`만 broadcast한다.

```ts
ptyProcess.onExit(({ exitCode }) => {
  this.broadcastWs(id, 'session:exited', { exitCode });
});
```

반면 `deleteSession()`은 PTY kill, detector destroy, headless dispose, CWD unwatch/unlink, WS replay state clear, debug capture clear, resize replay timer clear를 수행한다.

즉 shell 내부에서 `exit`하거나 PTY가 자연 종료되면 UI는 종료 이벤트를 받지만, `sessions` Map과 heavy resource의 수명은 사용자 삭제 경로와 같지 않다.

### 3. node-pty disposer를 저장하지 않는다

`node-pty`의 `onData`/`onExit`는 disposer를 반환할 수 있는 API로 쓰는 것이 안전하다. 현재 코드는 disposer를 `SessionData`에 저장하지 않는다. 따라서 공통 cleanup이 이벤트 listener 자체까지 idempotent하게 끊는 구조가 아니다.

### 4. `pendingRecoveryRestores`는 CWD-ready 경로 외 cleanup이 약하다

`WorkspaceService.pendingRecoveryRestores`는 restart/orphan recovery 후 새 세션이 CWD-ready가 되면 `runPendingRecoveryRestore()`에서 삭제된다. 하지만 탭 삭제, 워크스페이스 삭제, restart 중 취소/교체, stale session cleanup 경로에 공통 cancel API가 없다.

복구 restore는 작은 Map이지만, 장기 실행에서 session lifecycle과 맞지 않는 pending entry가 남는 것은 디버깅과 재시작 동작을 불안정하게 만든다.

### 5. WebSocket 송신 backpressure가 없다

`WsRouter`의 주요 송신 경로는 `readyState === OPEN`만 확인한다.

- `setupConnectionHandler()`의 `connected`
- `ping` 응답 `pong`
- `sendTo()`
- `broadcastAll()`
- `routeSessionOutput()`
- `rejectInput()`
- `SessionManager.broadcastWs()`

`ws.bufferedAmount`, send callback error, per-client queue cap, slow client termination 정책이 없다. 특히 `routeSessionOutput()`은 terminal output 고빈도 경로라 느린 브라우저 탭이 남아 있으면 서버 메모리와 JSON 직렬화 비용을 키울 수 있다.

### 6. 단일 `/ws`는 입력과 출력이 같은 backpressure 운명을 공유한다

현재 실시간 프로토콜은 단일 `/ws?token=<jwt>`가 다음을 모두 처리한다.

- input
- resize
- subscribe/unsubscribe
- screen-snapshot
- screen-snapshot ack
- screen-repair
- output
- status/cwd/session events
- workspace/tab/grid broadcast
- heartbeat

대량 output으로 브라우저 또는 TCP send buffer가 막히면 같은 소켓의 input 전송과 ack도 함께 늦어진다. 브라우저 input 지연의 직접 원인이 항상 서버 소켓인 것은 아니지만, server 설계상 격리가 없다.

### 7. daemon/dev 로그는 설정과 실제 writer가 분리되어 있다

설정에는 `logging.maxSize`, `logging.maxFiles`가 있다. 하지만 현재 runtime writer는 회전 정책을 적용하지 않는다.

- `tools/daemon/log.js`의 `appendLog()`는 `appendFileSync()`만 수행한다.
- `tools/start-runtime.js`의 daemon tee도 stdout/stderr write마다 `appendFileSync()`를 수행한다.
- `tools/daemon/launcher.js`는 child stdout/stderr를 같은 log file descriptor에 append로 연결한다.
- `server/src/index.ts`의 `[TerminalObs]`는 60초마다 session/ws snapshot 전체를 JSON stringify해 출력한다.

로그 회전이 없고, `TerminalObs`는 기본 info 출력으로는 너무 상세하다.

## 목표 아키텍처

### 전체 구조

```text
PTY process
  -> SessionManager.onData
    -> BoundedHeadlessOutputQueue
      -> HeadlessWriteScheduler
        -> @xterm/headless authoritative state
        -> bounded snapshot/cache state
    -> OutputEventBus
      -> WsOutputRouter
        -> per-client safeSend / queue / coalesce / terminate

Client input/control
  -> WsControlRouter
    -> SessionManager.writeInput / resize / ack / subscribe intent
```

핵심 원칙은 다음과 같다.

- 서버의 authoritative terminal state는 headless terminal이 유지한다.
- headless write backlog는 무제한 보존하지 않는다.
- 출력 송신은 per-client backpressure 정책을 통과해야 한다.
- input/control은 output congestion과 분리한다.
- 모든 cap은 UTF-8 byte 기준을 기본으로 한다.
- drop이 필요한 경우 raw terminal data를 조용히 버리지 않고, telemetry와 repair/snapshot 재동기화로 관측 가능하게 만든다.

### 채널 분리 모델

초기 구현은 기존 `/ws`를 유지하면서 신규 경로를 추가한다.

| 채널 | 경로 | 방향 | 책임 |
|---|---|---|---|
| 호환 단일 채널 | `/ws` | 양방향 | 기존 프로토콜 전체 |
| control 채널 | `/ws/control` | 양방향, 저용량 | auth 연결, input, resize, subscribe intent, snapshot/screen-repair ack, status/cwd/session lifecycle, workspace/tab/grid event |
| output 채널 | `/ws/output` | 주로 server -> client | output, screen-snapshot, screen-repair payload, replay/snapshot refresh, output-channel heartbeat |

`screen-snapshot:ready`와 `screen-repair:ready/failed`는 control 채널로 보낸다. output 채널이 막혀도 ack와 input이 같은 congestion에 갇히지 않게 하기 위해서다.

`status`, `cwd`, `session:exited`, `input:rejected`는 control 채널로 보낸다. 이 이벤트들은 UI state와 입력 안전성에 중요하고 output보다 훨씬 작다.

`output`, `screen-snapshot`, `screen-repair` payload는 output 채널로 보낸다. output 채널이 느려지면 해당 채널만 terminate/reconnect하고, control 채널은 살아 있도록 한다.

### 채널 pairing

split mode에서는 두 socket을 같은 logical client로 묶어야 한다.

권장 handshake:

1. 브라우저가 `/ws/control?token=<jwt>`로 연결한다.
2. 서버가 `connected`에 `clientId`, `splitSupported: true`, `outputPath: "/ws/output"`을 포함한다.
3. 브라우저가 `/ws/output?token=<jwt>&clientId=<clientId>`로 연결한다.
4. 서버는 output socket의 JWT와 `clientId`를 검증하고 같은 client registry entry에 붙인다.
5. output socket이 아직 없으면 control 채널은 input/control만 처리하고 output payload는 전송하지 않거나, 설정에 따라 single fallback을 사용한다.

기존 `/ws`에서는 `connected` payload를 확장하되 기존 client가 무시할 수 있는 additive field만 추가한다.

### Routing ownership

현재 `WsRouter` 하나가 모든 책임을 가진다. 장기 구조는 다음 중 한 단계로 나누는 것이 좋다.

1. 1차: `WsRouter` 내부에 `channelRole: "single" | "control" | "output"`을 추가하고 기존 class를 보존한다.
2. 2차: 송신 정책을 `WsSendManager` 또는 `WsTransportRegistry`로 추출한다.
3. 3차: `WsControlRouter`, `WsOutputRouter`로 분리하되 session subscription state는 registry가 소유한다.

처음부터 class를 크게 쪼개기보다, `safeSend`와 metadata 구조를 먼저 넣고 그 위에서 분리하는 편이 리스크가 낮다.

## 설정 기반 chunk/cache cap 설계

### 설계 원칙

- 설정은 기본값만으로 기존 사용자가 별도 수정 없이 안전한 bounded behavior를 얻어야 한다.
- legacy `pty.maxBufferSize`는 계속 `pty.maxSnapshotBytes`로 migrate한다.
- 새 cap은 raw terminal 내용이 아니라 byte 수, chunk 수, queue entry 수, session/client 수를 제한한다.
- UI settings에 바로 노출하지 않고, 먼저 `config.json5` + telemetry로 운영 검증한다.
- hot apply는 안전한 값만 적용한다. topology나 path 변경은 restart-required로 둔다.

## SessionManager 개선 계획

### 1. Headless pending queue를 bounded deque로 교체

대상:

- `server/src/services/SessionManager.ts`
- 신규 후보: `server/src/utils/boundedByteQueue.ts`
- 테스트: `server/src/test-runner.ts`

현재 `pendingOutputChunks: string[]` 대신 다음 상태를 둔다.

```ts
interface HeadlessPendingOutputQueue {
  entries: Array<{ seq: number; data: string; byteLength: number }>;
  headIndex: number;
  bytes: number;
  droppedBytes: number;
  droppedChunks: number;
  overflowCount: number;
}
```

구현 원칙:

- `push()`는 UTF-8 byte length를 계산한다.
- `shift()` 대신 `headIndex`를 증가시키고 일정 주기 compact한다.
- 아직 headless write가 시작되지 않은 pending chunks는 `maxHeadlessWriteBatchBytes` 이하에서 병합할 수 있다.
- cap 초과 시 기본 동작은 `headlessOverflowPolicy="degrade"`다.
- overflow 발생 시 headless를 degraded로 전환하고, pending queue를 bounded degraded replay seed로 합친 뒤 비운다.
- raw output을 silent drop하지 않는다. drop 또는 degrade는 `SessionManagerObservability`와 debug capture에 기록한다.

중요한 판단:

headless terminal은 authoritative state이므로 pending write를 임의로 tail-drop하면 이후 snapshot이 실제 PTY 화면과 달라질 수 있다. 따라서 첫 구현의 기본 정책은 drop이 아니라 degraded fallback이어야 한다. 나중에 output channel이 repair snapshot을 강제할 수 있는 구조가 충분히 안정화되면 `tail-drop-with-repair` 같은 정책을 추가할 수 있다.

### 2. Headless write chain lifecycle 정리

대상:

- `SessionData`
- `queueHeadlessOutput()`
- `applyHeadlessOutput()`
- `deleteSession()`
- `markHeadlessDegraded()`

개선 내용:

- session cleanup 시 `headlessCloseSignal.resolve()` 이후 pending queue를 명시적으로 clear한다.
- `pendingHeadlessWrites`는 gauge로 유지하되, queue bytes/chunks와 분리한다.
- `headlessWriteChain`은 cleanup 후 새 no-op resolved promise로 대체해 stale closure가 큰 string을 오래 잡지 않게 한다.
- `applyHeadlessOutput()`는 queue에서 꺼낸 entry를 기준으로 screenSeq와 unsnapshotted output을 갱신한다. 현재처럼 인자로 받은 `data`와 `pendingOutputChunks.shift()` 결과가 어긋날 여지를 줄인다.

### 3. PTY onExit cleanup을 공통 lifecycle로 통합

대상:

- `SessionManager.createSession()`
- `SessionManager.deleteSession()`
- 신규 후보: `cleanupSessionResources(sessionId, data, reason)`

권장 흐름:

1. `onExit`에서 `session:exited`를 control 채널로 broadcast한다.
2. heavy resource는 즉시 cleanup한다.
3. workspace/tab UI가 종료 상태를 표시할 수 있도록 bounded tombstone만 남긴다.
4. tombstone은 TTL과 최대 개수로 제한한다.

현재 `SessionStatus`는 `running | idle`뿐이다. 따라서 바로 `exited` status를 public DTO에 추가하면 frontend 계약 변경이 커진다. migration은 두 단계가 낫다.

- 1단계: `session:exited` 이벤트는 유지하고, `sessions` Map에서는 heavy resource를 제거하되 `exitedSessions` tombstone Map을 별도로 둔다.
- 2단계: 필요하면 `SessionStatus = "running" | "idle" | "exited"`로 확장하고 frontend 타입을 함께 갱신한다.

### 4. node-pty disposer 저장과 idempotent cleanup

대상:

- `SessionData`
- `createSession()`
- `deleteSession()`

후보 필드:

```ts
ptyDisposables: Array<{ dispose: () => void }>;
cleanupStarted: boolean;
exitMetadata?: { exitCode?: number; exitedAt: number };
```

적용:

- `const dataDisposable = ptyProcess.onData(...)`를 저장한다.
- `const exitDisposable = ptyProcess.onExit(...)`를 저장한다.
- cleanup은 disposer를 먼저 dispose하고, 그 다음 timers/headless/CWD/WS/debug state를 정리한다.
- `deleteSession()`이 `onExit`를 재진입으로 유발해도 한 번만 수행되도록 `cleanupStarted` guard를 둔다.

### 5. pendingRecoveryRestores cleanup

대상:

- `server/src/services/WorkspaceService.ts`
- 테스트: `server/src/test-runner.ts`

추가 API 후보:

```ts
private cancelPendingRecoveryRestore(sessionId: string, reason: string): void
private cancelPendingRecoveryRestores(sessionIds: string[], reason: string): void
private pruneExpiredPendingRecoveryRestores(now = Date.now()): void
```

`PendingRecoveryRestore`에는 `createdAt`과 `expiresAt`을 둔다. 기본 TTL 후보는 30초다.

적용 경로:

- `deleteTab()`에서 tab sessionId cancel
- `deleteWorkspace()`에서 workspace sessionIds cancel
- `restartTab()`에서 old session cancel, new session pending은 CWD-ready 또는 TTL로 정리
- `checkOrphanTabs()`에서 old orphan id cancel, new id pending 등록
- `handleSessionCwdReady()` 시작 전에 만료 pending prune
- server shutdown 또는 workspace save 전 optional prune

## WsRouter safeSend/backpressure 계획

### 1. 모든 송신을 `safeSend`로 통합

대상:

- `server/src/ws/WsRouter.ts`
- `server/src/services/SessionManager.ts`
- `server/src/types/ws-protocol.ts`

현재 직접 `ws.send()`하는 곳을 모두 `safeSend()`로 통과시킨다.

후보 return type:

```ts
type SafeSendResult =
  | { ok: true; mode: "sent" | "queued" | "coalesced" }
  | { ok: false; reason: "not-open" | "high-water" | "critical-water" | "queue-overflow" | "send-error" };
```

`safeSend()`가 확인할 항목:

- `ws.readyState`
- `ws.bufferedAmount`
- channel role
- message class
- current per-client queued bytes/messages
- critical backpressure terminate 여부

### 2. Message class별 정책

| class | 예시 | 정책 |
|---|---|---|
| critical-control | `input:rejected`, `session:error`, `session:exited`, auth/connected | control 채널에서 lossless 시도, 실패 시 socket terminate 및 telemetry |
| low-volume-state | `status`, `cwd`, workspace/tab/grid event | key 기준 last-wins coalesce 가능 |
| snapshot | `screen-snapshot`, `screen-repair` | sessionId 기준 최신 snapshot/repair가 이전 것을 대체 가능 |
| output | `output` | sessionId별 adjacent chunk coalesce, cap 초과 시 output channel reconnect + repair/replay 유도 |
| heartbeat | `pong`, ping/pong | backpressure 판단에 사용, 실패 시 terminate |

terminal output은 무한 queue하면 안 된다. 기본 정책은 다음 순서가 적절하다.

1. output 채널 `bufferedAmount < highWaterBytes`면 즉시 send.
2. high water 이상이면 per-client queue에 넣고 adjacent output을 sessionId별로 coalesce.
3. queue cap 초과 시 해당 client/session에 `needsReplay` 또는 `needsSnapshot` marker를 세운다.
4. output channel을 terminate하고 재연결 시 fresh snapshot을 보낸다.

이 정책은 출력 일부를 보존하려고 서버 메모리를 끝없이 쓰는 것보다 안전하다. terminal output은 최종적으로 server headless snapshot으로 재동기화할 수 있어야 한다.

### 3. Per-client queue 구조

후보:

```ts
interface WsClientSendState {
  queue: WsSendQueueEntry[];
  queuedBytes: number;
  queuedMessages: number;
  highWaterCount: number;
  criticalWaterCount: number;
  droppedOutputBytes: number;
  coalescedMessages: number;
  terminatedForBackpressure: number;
  drainTimer?: NodeJS.Timeout;
}
```

queue entry:

```ts
type WsSendQueueEntry = {
  class: "critical-control" | "low-volume-state" | "snapshot" | "output" | "heartbeat";
  sessionId?: string;
  key?: string;
  payload: string;
  byteLength: number;
  createdAt: number;
};
```

flush 규칙:

- timer는 client당 하나만 둔다.
- flush는 `drainBudgetBytesPerTick` 이하로 제한한다.
- send callback error는 client cleanup으로 연결한다.
- `bufferedAmount`가 critical 이상이면 flush하지 않고 terminate한다.
- `destroy()`는 모든 drain timer를 clear한다.

### 4. Replay/screen-repair queue와 safeSend 관계

현재 replay pending queue와 screen-repair pending queue는 session handshake correctness를 위한 queue다. 새 `safeSend` queue는 transport backpressure queue다. 둘은 섞으면 안 된다.

권장 계층:

```text
replay pending queue
  -> ready to emit message
    -> safeSend transport queue
```

즉 replay ack 전 output 보류는 protocol queue이고, socket high-water 보류는 transport queue다. telemetry도 분리한다.

### 5. UTF-8 byte 기준 통일

현재 일부 queue는 `data.length`를 쓰고, 일부는 `Buffer.byteLength(data, "utf8")`를 쓴다. cap은 모두 UTF-8 byte 기준으로 통일한다.

적용:

- replay queued output
- screen-repair queued output
- headless pending queue
- WebSocket send queue
- debug capture raw payload cap
- snapshot cache cap

## Split WebSocket 구현 계획

### Phase A: 단일 `/ws` 내부 safeSend 도입

목표:

- topology는 바꾸지 않고 backpressure를 먼저 관측 가능하게 만든다.

작업:

- `WsClientMeta`에 send state 추가
- `sendTo()`, `broadcastAll()`, `routeSessionOutput()`, `rejectInput()`, `handleMessageError()`, heartbeat pong을 `safeSend()`로 변경
- `SessionManager.broadcastWs()`가 직접 subscribers에 `ws.send()`하지 않고 `WsRouter.broadcastSessionEvent()` 같은 API를 호출하게 변경
- `/api/sessions/telemetry`에 WS send state summary 추가

성공 기준:

- slow fake ws에서 `bufferedAmount` high water가 telemetry에 잡힌다.
- critical water에서는 해당 client가 terminate된다.
- 기존 single socket tests는 유지된다.

### Phase B: Output message coalesce/drop/replay 정책

목표:

- 느린 client 때문에 서버 메모리가 무한 증가하지 않게 한다.

작업:

- output message class 도입
- 같은 client/session의 consecutive output을 `maxOutputCoalesceBytes`까지 병합
- cap 초과 시 output queue를 비우고 session/client에 `outputBackpressureReplayRequired` marker 기록
- marker가 있는 client가 다시 subscribe 또는 output channel reconnect하면 fresh `screen-snapshot`부터 보낸다.

성공 기준:

- output 폭주 중 per-client queued bytes가 cap 이하로 유지된다.
- drop/coalesce가 telemetry에 남는다.
- reconnect 후 snapshot이 먼저 오고 output이 이어진다.

### Phase C: `/ws/control`과 `/ws/output` 추가

목표:

- input/control path와 output/replay/snapshot path를 transport 수준에서 분리한다.

작업:

- `server/src/index.ts` upgrade dispatcher에 `/ws/control`, `/ws/output` 추가
- `WsRouter.handleUpgrade(req, socket, head, role)` 또는 role별 router instance 도입
- control 연결에서 `connected` payload에 `clientId`, split metadata 추가
- output 연결에서 `clientId` pairing 검증
- session subscription state를 logical client 기준으로 이동
- output socket이 없는 split client에는 output payload를 보내지 않고, control에 `output:unavailable` 상태를 보낼 수 있게 한다.

성공 기준:

- output socket을 의도적으로 느리게 만들어도 control socket input handling은 계속 동작한다.
- output socket termination이 control socket을 끊지 않는다.
- 기존 `/ws` client는 그대로 동작한다.

### Phase D: frontend opt-in과 runtime-config

이 문서는 backend/server 계획이지만 split WS는 frontend opt-in이 있어야 완성된다.

작업:

- `/api/runtime-config`에 `websocketTopology`, `controlPath`, `outputPath`를 노출
- frontend `WebSocketContext`는 `single`이면 기존 `/ws`, `split`이면 두 socket을 연다.
- subscribe/unsubscribe는 control로 보내고, output handlers는 output channel message를 dispatch한다.
- snapshot/screen-repair ack는 control로 보낸다.

성공 기준:

- `single` 모드에서 기존 e2e가 통과한다.
- `split` 모드에서 output socket 재연결 중 input outbox와 server replay가 일관된다.

### Phase E: 기본값 전환

초기 기본값은 `realtime.websocketTopology: "single"`로 둔다. 안정화 후 `"auto"` 또는 `"split"`을 검토한다.

전환 기준:

- 장시간 output 폭주 시 server heap plateau가 확인된다.
- output socket만 terminate/reconnect해도 control input이 유지된다.
- snapshot ack/replay race 회귀가 없다.
- 모바일/브라우저별 WebSocket 제한에서 문제가 없다.

목표:

- 실제 장시간 사용 시 heap, queue, latency가 plateau를 보이는지 확인한다.

작업:

1. slow WebSocket client 시뮬레이터 작성
2. 대량 PTY output session 시나리오 작성
3. split/single topology 비교
4. telemetry snapshot을 주기적으로 수집해 max/avg 추세를 확인
5. default cap 조정

성공 기준:

- 30분 이상 output 폭주 후 server heap과 queued bytes가 cap 근처에서 안정화된다.
- output channel 재연결 후 active terminal이 fresh snapshot으로 복구된다.
- control input path는 output backpressure 중에도 `input` handling latency가 낮게 유지된다.
- `session:exited`, deleteTab, deleteWorkspace, restart 반복 후 session/headless/ws/recovery pending count가 원래 수준으로 돌아온다.

## 위험과 결정 사항

### Headless overflow 정책

결정: 기본은 `degrade`가 맞다.

이유:

- authoritative terminal state는 중간 output 일부를 잃으면 신뢰할 수 없다.
- silent tail drop은 나중에 snapshot이 실제 PTY 화면과 달라지는 문제를 숨긴다.
- degraded fallback은 사용자에게 완벽한 화면 복구를 보장하지는 않지만, 불일치를 명시적으로 드러낸다.

### Split WebSocket 기본값

결정: 초기는 `single`.

이유:

- 현재 frontend와 문서는 단일 `/ws`를 source of truth로 설명한다.
- split은 browser/client pairing, reconnect, ack routing의 새 failure mode가 있다.
- safeSend/backpressure를 먼저 단일 채널에서 검증해야 split 문제와 기존 문제를 구분할 수 있다.

### Snapshot ack 위치

결정: split mode에서 ack는 control 채널.

이유:

- output 채널이 막힌 상황에서 ack까지 같은 채널에 두면 replay pending이 해소되지 않는다.
- ack는 작은 control message이고 input safety와 직접 연결된다.

### Logging

결정: `logging.maxSize/maxFiles`는 실제 daemon/dev writer에 연결한다.

이유:

- 설정이 이미 존재하므로 새 설정을 만들기보다 no-op에 가까운 기존 설정을 실제 동작으로 맞추는 것이 낫다.
- TerminalObs 상세 배열 출력은 기본 console log에서 제거한다.

## 최종 권장

가장 먼저 구현할 묶음은 "bounded headless queue + safeSend + telemetry"다. 이 묶음은 split WebSocket 없이도 서버의 무제한 누적 위험을 줄이고, 이후 split 설계의 효과를 측정할 기준선을 만든다.

그 다음 `/ws/control`과 `/ws/output`을 feature flag로 추가한다. split은 입력 지연을 구조적으로 줄일 수 있지만, backpressure 정책 없이 split만 먼저 넣으면 output socket 쪽 누수는 그대로 남는다. 따라서 순서는 cap과 cleanup이 먼저, split routing이 그 다음이다.
