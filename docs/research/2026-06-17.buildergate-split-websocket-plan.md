# BuilderGate Split WebSocket Long-Term Improvement Plan

작성일: 2026-06-17

## 목적

이 문서는 기존 누수/입력 지연 분석 문서인 `docs/research/2026-06-17.buildergate-memory-input-lag-research.md`를 바탕으로, 사용자의 아이디어 2번인 "보내는 WebSocket과 받는 WebSocket을 분리하여 키보드 입력 지연을 줄이는 방법"을 장기 개선안으로 조사한 결과다.

이번 작업은 문서 작성만 수행한다. 코드 수정은 하지 않는다.

## 핵심 결론

WebSocket을 `control/input` 채널과 `output/replay` 채널로 분리하면, 터미널 output 폭주나 느린 output socket의 송신 버퍼가 키보드 입력 전송 경로를 직접 압박하는 상황을 줄일 수 있다. 특히 현재처럼 `input`, `resize`, `screen-snapshot:ready`, `subscribe`, `output`, `screen-snapshot`, `workspace/tab/grid broadcast`가 모두 단일 `/ws` 연결을 공유하는 구조에서는 output 처리량이 커질수록 입력 전송 지연과 관측 불가능한 backpressure가 겹칠 수 있다.

다만 분리 WS는 만능 해결책이 아니다. 브라우저 메인 스레드에서 xterm write, snapshot 저장, hidden terminal 처리, React 재평가가 오래 걸리면 control socket이 분리되어도 `keydown -> xterm onData -> send()` 호출 자체가 늦어진다. 따라서 이 장기안은 기존 문서의 P0인 hidden terminal 구독/쓰기 제한, output scheduler, `bufferedAmount` 기반 backpressure, 서버 headless queue 제한과 함께 추진해야 효과가 크다.

권장 방향은 기존 단일 WS를 유지한 채, 기능 플래그로 split protocol을 추가하는 점진적 마이그레이션이다.

1. `/ws?token=...` 단일 연결은 그대로 legacy unified mode로 유지한다.
2. 신규 split mode는 같은 `/ws` path에 `channel=control|output` 또는 `mode=split` query를 추가해 시작한다. 현재 `server/src/index.ts`는 pathname이 `/ws`이면 `WsRouter.handleUpgrade()`로 넘기므로 path를 늘리지 않는 편이 초기 호환성이 좋다.
3. 서버는 `clientId`를 물리 socket ID가 아니라 browser provider 단위의 `clientGroupId`로 재정의하고, 각 물리 연결에는 별도 `connectionId`와 `channelRole`을 둔다.
4. `subscribe`와 replay pending state는 socket 단위가 아니라 `clientGroupId + sessionId + subscriptionEpoch` 단위로 옮긴다.
5. input, resize, snapshot/repair ACK, ping은 control socket으로 보내고, output, snapshot, repair patch, status/cwd/session event, workspace/tab/grid broadcast는 output socket으로 받는다.

### Protocol

현재 `ClientWsMessage`와 `ServerWsMessage`는 frontend/server 양쪽에 복사되어 있다.

- `frontend/src/types/ws-protocol.ts`
- `server/src/types/ws-protocol.ts`

현재 client -> server 메시지는 다음 범주가 단일 union에 섞여 있다.

- 구독: `subscribe`, `unsubscribe`
- replay/repair ACK: `screen-snapshot:ready`, `screen-repair:ready`, `screen-repair:failed`
- replay/repair 요청: `repair-replay`, `screen-repair`
- 입력과 geometry: `input`, `resize`
- heartbeat: `ping`

현재 server -> client 메시지도 다음 범주가 단일 union에 섞여 있다.

- terminal output/replay: `output`, `screen-snapshot`, `screen-repair`
- session state: `subscribed`, `session:ready`, `status`, `cwd`, `session:error`, `session:exited`, `input:rejected`
- workspace/tab/grid event
- connection: `connected`, `pong`

## 문제 모델

단일 WebSocket에서 input latency가 늦어지는 경로는 세 가지다.

1. 브라우저 메인 스레드 지연
   - output socket message를 받아 `TerminalContainer.onOutput -> terminal.write()`로 전달하고, `TerminalView.writeOutput()`이 xterm write callback과 snapshot save를 예약한다.
   - hidden terminal까지 계속 output을 쓰면 keydown, composition, `onData`, `send()` 호출이 같은 메인 스레드에서 밀릴 수 있다.

2. 단일 socket의 전송/backpressure 관측 부재
   - client `send()`와 server `routeSessionOutput()` 모두 `bufferedAmount`를 보지 않는다.
   - 느린 client 또는 막힌 socket에 output이 쌓여도 input path는 성공으로 오판할 수 있다.

3. 서버 이벤트 루프 공용화
   - split socket을 도입해도 Node.js 이벤트 루프는 같다.
   - `routeSessionOutput()`이 많은 subscriber에 대해 JSON send loop를 오래 돌면 control socket의 `message` handler도 지연될 수 있다.
   - 따라서 split WS와 함께 output send scheduler/backpressure가 필요하다.

결론적으로 split WS의 목표는 "output traffic과 input traffic의 물리 socket과 backpressure 상태를 분리해, output backlog가 input 전송 가능성 판단을 오염시키지 않게 하는 것"이다. 브라우저 main thread와 서버 event loop starvation은 별도 완화가 필요하다.

## 제안 아키텍처

### 채널 정의

```text
Browser WebSocketProvider
  ├─ Control/Input WebSocket
  │    client -> server: input, resize, subscribe, unsubscribe, replay ACK, repair ACK, ping
  │    server -> client: connected, control ack, input:rejected, protocol:error, pong, minimal transport state
  │
  └─ Output/Replay WebSocket
       server -> client: output, screen-snapshot, screen-repair, session:ready, status, cwd,
                         session:error, session:exited, workspace/tab/grid broadcast
       client -> server: no ordinary traffic after handshake
```

초기 구현은 WebSocket path를 늘리지 않고 query로 role을 구분하는 편이 안전하다.

```text
Legacy:
  /ws?token=<jwt>

Split:
  /ws?token=<jwt>&channel=control&protocol=2
  /ws?token=<jwt>&channel=output&protocol=2&clientId=<groupId>&pairToken=<serverIssuedToken>
```

향후 reverse proxy, 로그, 운영 지표상 endpoint 분리가 필요하면 `/ws/control`, `/ws/output` alias를 추가할 수 있다. 하지만 첫 단계에서는 현재 `server/src/index.ts`의 `/ws` upgrade dispatch를 최대한 유지하는 것이 마이그레이션 위험이 낮다.

### Client Group

현재 `clientId`는 물리 WebSocket마다 새로 발급된다. split mode에서는 다음 두 ID가 필요하다.

- `clientGroupId`
  - 브라우저의 한 `WebSocketProvider` 인스턴스를 나타낸다.
  - REST `x-client-id` self-exclusion에는 이 값을 사용한다.
  - control/output socket이 같은 그룹으로 묶인다.
- `connectionId`
  - 물리 WebSocket 연결 하나를 나타낸다.
  - heartbeat, stale socket guard, observability, cleanup에 사용한다.

권장 handshake:

1. control socket이 먼저 연결한다.
2. 서버가 JWT를 검증하고 `clientGroupId`, `controlConnectionId`, 짧은 TTL의 `pairToken`을 발급한다.
3. frontend는 `setWsClientId(clientGroupId)`로 REST self-exclusion 값을 설정한다.
4. output socket은 같은 JWT와 `clientGroupId`, `pairToken`을 사용해 연결한다.
5. 서버는 pair token이 맞을 때만 output socket을 해당 group에 연결한다.

client가 임의로 `clientGroupId`를 주장하는 방식은 피한다. self-exclusion은 보안 경계는 아니지만, 다른 탭의 broadcast 수신을 의도치 않게 억제할 수 있으므로 서버 발급 group ID가 낫다.

### Server Registry

현재 `WsRouter`의 핵심 state는 socket 객체 중심이다.

```text
clients: Map<WebSocket, WsClientMeta>
sessionSubscribers: Map<sessionId, Set<WebSocket>>
replayPendingSessions: WsClientMeta 내부 Map<sessionId, ReplayPendingState>
screenRepairPendingSessions: WsClientMeta 내부 Map<sessionId, ScreenRepairPendingState>
```

split mode에서는 다음 구조로 확장한다.

```text
connectionMeta: Map<WebSocket, ConnectionMeta>
clientGroups: Map<clientGroupId, ClientGroup>
sessionSubscribers: Map<sessionId, Set<clientGroupId>>

ClientGroup
  clientGroupId
  control?: WebSocket
  output?: WebSocket
  activeSubscriptions: Set<sessionId>
  subscriptionEpochBySession: Map<sessionId, number>
  replayPendingBySession: Map<sessionId, ReplayPendingState>
  screenRepairPendingBySession: Map<sessionId, ScreenRepairPendingState>
  lastControlSeenAt
  lastOutputSeenAt
```

중요한 변화는 replay pending과 subscription이 더 이상 물리 socket 하나에 종속되지 않는다는 점이다. snapshot은 output socket으로 나가지만 ACK는 control socket으로 들어올 수 있으므로, `replayToken`과 `subscriptionEpoch`를 통해 group/session pending state를 찾아야 한다.

### Frontend Transport

`WebSocketContext`를 바로 두 socket으로 키우기보다, 내부 transport abstraction을 먼저 둔다.

```text
WebSocketProvider
  -> RealtimeTransport
       UnifiedTransport
       SplitTransport
  -> 기존 context API
       status
       clientId
       send(msg)
       subscribeSession(sessionId, handlers)
       setWorkspaceHandlers(handlers)
```

외부 API는 가능한 한 유지한다.

- `TerminalContainer`는 계속 `send()`와 `subscribeSession()`을 사용한다.
- `send()`는 split mode에서 message type에 따라 control/output/unified를 선택한다.
- `subscribeSession()`은 handler 등록과 active subscription 유지 API로 남긴다.
- `handleMessage()`는 output socket message를 기존 session/workspace handler로 라우팅한다.

`SendResult`는 backpressure를 표현할 수 있어야 한다.

```ts
type SendResult =
  | { ok: true; channel?: 'unified' | 'control' | 'output'; bufferedAmount?: number }
  | {
      ok: false;
      reason:
        | 'not-open'
        | 'missing-token'
        | 'stale-socket'
        | 'backpressure'
        | 'channel-unavailable'
        | 'protocol-not-ready';
      channel?: 'unified' | 'control' | 'output';
      bufferedAmount?: number;
    };
```

`TerminalContainer`는 기존 transport outbox 정책을 재사용하되, `backpressure`와 `channel-unavailable`을 canonical reject/barrier reason으로 매핑해야 한다. 장기적으로 `TerminalInputBarrierReason`에 `control-backpressure`, `output-replay-pending`, `output-disconnected` 같은 명시 reason을 추가하는 것이 관측에 유리하다.

## 메시지 라우팅 설계

### Client -> Server

| 메시지 | Split 채널 | 이유 |
|---|---|---|
| `input` | control | 키보드 입력의 최우선 저지연 경로 |
| `resize` | control | PTY geometry 변경은 output backlog와 분리되어야 함 |
| `subscribe` / `unsubscribe` | control | subscription state는 client group 소유 |
| `screen-snapshot:ready` | control | snapshot은 output으로 받지만 ACK는 저지연 control로 보냄 |
| `screen-repair` | control | repair 요청은 현재 화면 상태 기반 제어 메시지 |
| `screen-repair:ready` / `screen-repair:failed` | control | repair ACK/NACK |
| `repair-replay` | control | 서버 replay 시작 요청 |
| `ping` | control 중심, output 별도 heartbeat 가능 | input transport 상태를 빠르게 확인 |

output socket은 handshake 이후 일반 메시지를 보내지 않는 방향이 좋다. 다만 브라우저 WebSocket API상 pong은 내부 처리이고, app-level output heartbeat를 별도로 둔다면 output socket에서 `output:pong`을 보낼 수 있다. 이 경우도 terminal input과 섞이지 않는 저빈도 메시지로 제한한다.

### Server -> Client

| 메시지 | Split 채널 | 이유 |
|---|---|---|
| `connected` | 각 채널 | channel role, group ID, connection ID 확인 |
| `subscribed` 또는 `subscription:ack` | control | subscription 요청 결과는 제어 응답 |
| `input:rejected` | control | 입력 outbox/reject telemetry와 직접 연결 |
| `pong` | control | control RTT 측정 |
| `output` | output | 고빈도 terminal output |
| `screen-snapshot` | output | 대형 replay payload |
| `screen-repair` / `screen-repair:rejected` | output 또는 control+output 혼합 | patch payload는 output, 거절 사유는 control도 가능 |
| `session:ready` | output, control mirror 선택 | 기존 handler 호환은 output, 입력 gate 해제는 control mirror가 유리 |
| `status`, `cwd`, `session:error`, `session:exited` | output, critical event는 control mirror 고려 | 고빈도는 아니지만 기존 session event stream과 묶임 |
| `workspace:*`, `tab:*`, `grid:updated` | output | UI 상태 broadcast, input 경로와 분리 |
| `protocol:error`, `transport:backpressure`, `output:degraded` | control | 사용 가능한 입력 경로 판단에 필요 |

`session:ready`는 특히 중요하다. 기존에는 snapshot ACK 이후 같은 socket에서 `session:ready`가 도착하고, 프론트가 입력을 열 수 있다. split mode에서는 ACK가 control로 가고 snapshot/output은 output으로 오므로 ordering이 흔들릴 수 있다. 해결책은 다음 중 하나다.

1. `session:ready`를 control에 보내 input gate를 열고, output에는 기존 호환용으로도 보낸다.
2. `session:ready`를 output에만 보내되, frontend input gate는 output handler 수신 전까지 열지 않는다.

입력 안전성을 우선하면 2번이 보수적이다. 입력 latency를 더 줄이려면 1번을 택하되, `subscriptionEpoch`와 `replayToken`이 일치하는 경우에만 gate를 열어야 한다.

## Subscription과 Replay State

현재 `handleSubscribe()`는 WebSocket 객체를 session subscriber set에 넣고, snapshot을 같은 socket으로 보낸다. split mode에서는 control socket이 subscribe를 받고 output socket으로 snapshot을 보내야 한다.

권장 흐름:

```text
Client control -> subscribe(sessionIds)
Server
  -> clientGroup.activeSubscriptions.add(sessionId)
  -> sessionSubscribers[sessionId].add(clientGroupId)
  -> subscriptionEpochBySession[sessionId] += 1
  -> control: subscribed/session status ACK
  -> output: screen-snapshot(replayToken, subscriptionEpoch) 또는 session:ready

Client control -> screen-snapshot:ready(sessionId, replayToken, subscriptionEpoch)
Server
  -> group replayPendingBySession 검증
  -> queued output flush는 output socket으로 전송
  -> queued input flush는 SessionManager.writeInput()으로 처리
  -> session:ready emit
```

output socket이 아직 없거나 reconnect 중이면 subscribe는 성공시키되 `outputPending` 상태로 남긴다. 이때 input은 즉시 열지 않는다. output socket이 연결되면 active subscription에 대해 snapshot replay를 다시 시작한다.

## 실패와 재연결 시나리오

### Control socket만 끊김

상태:

- output socket은 계속 output을 받을 수 있다.
- input, resize, ACK는 보낼 수 없다.

정책:

- `WebSocketContext.status`는 입력 관점에서 `reconnecting`으로 둔다.
- `TerminalContainer`는 기존 reconnect TTL outbox를 재사용한다.
- TTL 안에 control이 복구되면 queued input을 sequence 순서대로 flush한다.
- TTL을 넘으면 Enter 포함 입력은 `timeout-enter-safety` 또는 `transport-closed`로 reject한다.
- output은 계속 표시할 수 있지만, control ACK가 필요한 replay/repair 중이면 input gate를 열지 않는다.

### Output socket만 끊김

상태:

- control socket은 살아 있어 input을 서버로 보낼 수 있다.
- 사용자는 echo/output을 보지 못하므로 blind input 위험이 있다.

정책:

- output socket이 짧게 reconnect 중이면 input을 queue한다.
- output이 복구되어 snapshot replay와 `session:ready`가 끝나면 queue를 flush한다.
- output socket이 backpressure로 terminate된 경우도 같은 replay 복구 경로를 탄다.
- 이미 shell에 전달된 input은 되돌릴 수 없으므로, output unavailable 상태에서는 새 input을 보내지 않는 보수 정책이 기본값이어야 한다.
- 추후 power user mode에서 output unavailable 중 printable input만 허용하는 옵션을 검토할 수 있지만 기본값으로 두지 않는다.

### 두 socket 모두 끊김

정책:

- 현재 unified reconnect와 동일하게 exponential backoff를 적용한다.
- control을 먼저 복구하고, output을 붙인 뒤 subscription replay를 완료한다.
- 둘 중 하나라도 auth 실패하면 group 전체를 auth-expired로 닫는다.

### Output backpressure

상태:

- 서버의 output socket `bufferedAmount`가 high-water mark를 넘는다.

정책:

- warning 구간에서는 output send를 coalesce하고 observability에 기록한다.
- severe 구간에서는 output socket을 terminate하고, 해당 group/session을 replay 필요 상태로 전환한다.
- control socket은 닫지 않는다.
- input gate는 output replay 완료 전까지 transient-blocked로 둔다.
- hidden/unsubscribed terminal에는 output을 보내지 않는 정책을 병행해야 한다.

### Control backpressure

상태:

- client control socket `bufferedAmount` 또는 server control socket `bufferedAmount`가 high-water mark를 넘는다.

정책:

- input 전송은 즉시 `{ ok: false, reason: 'backpressure', channel: 'control' }`를 반환한다.
- frontend outbox가 TTL과 byte budget 안에서 queue한다.
- high-water가 지속되거나 critical mark를 넘으면 control socket을 닫고 reconnect한다.
- control channel은 저용량이어야 하므로 backpressure는 정상 상태가 아니다. 반드시 telemetry와 debug capture에 남긴다.

### Stale socket과 React StrictMode

현재 frontend는 `wsRef.current !== ws` guard로 stale single socket message를 무시한다. split mode에서는 channel별 ref와 generation이 필요하다.

- `controlRef.current !== ws`이면 control message 무시.
- `outputRef.current !== ws`이면 output message 무시.
- `clientGroupGeneration` 또는 `transportGeneration`이 다른 message는 무시.
- 서버도 같은 group에 두 번째 control이 붙으면 기존 control을 graceful close하거나 새 control을 거절해야 한다. output도 동일하다.

### Server restart

서버 restart 후 기존 pair token과 group state는 사라진다.

정책:

- client는 control reconnect에서 새 `clientGroupId`를 받는다.
- REST `x-client-id`를 새 값으로 갱신한다.
- output도 새 group에 pair한다.
- active subscription은 frontend `activeSubscriptionsRef`를 source로 재전송한다.

## Auth, ClientId, Heartbeat 설계

### Auth

- 두 채널 모두 현재처럼 JWT를 검증한다.
- 초기 구현은 기존 query token 방식을 유지한다.
- pair token은 인증 토큰이 아니라 같은 browser provider의 두 socket을 묶기 위한 서버 발급 nonce다.
- pair token TTL은 짧게 둔다. 예: 30초.
- output reconnect를 위해 group이 살아 있는 동안 pair token을 rotate하거나, control socket이 output reconnect용 token을 재발급할 수 있어야 한다.

### ClientId

- 기존 `connected.clientId`는 split mode에서 `clientGroupId` 의미로 유지하면 frontend REST self-exclusion 변경이 작다.
- 새 `connectionId`와 `channel` 필드를 optional로 추가한다.
- legacy client는 추가 필드를 무시한다.

예시:

```json
{
  "type": "connected",
  "clientId": "group-uuid",
  "connectionId": "connection-uuid",
  "channel": "control",
  "protocolVersion": 2,
  "pairToken": "short-lived-token"
}
```

### Heartbeat

현재 서버는 30초마다 모든 client socket에 `ping()`을 보내고 `pong`으로 `isAlive`를 갱신한다.

split mode에서는 role별 상태를 따로 기록한다.

- control heartbeat 실패
  - input transport를 `reconnecting`으로 전환한다.
  - output이 살아 있어도 input은 queue/reject 정책을 탄다.
- output heartbeat 실패
  - output transport를 reconnect한다.
  - active session은 output replay 필요 상태로 둔다.
- group cleanup
  - control과 output이 모두 닫히고 grace TTL이 지나면 subscriptions, replay pending, repair pending을 정리한다.

## Backpressure 정책

### 공통 `safeSend`

모든 server send 경로를 공통 `safeSend()`로 통합해야 한다.

현재 직접 send 지점:

- `WsRouter.routeSessionOutput()`
- `WsRouter.broadcastAll()`
- `WsRouter.sendTo()`
- `SessionManager.broadcastWs()`
- `WsRouter.handleMessage()`의 `pong` 직접 send
- `WsRouter.rejectInput()`의 `input:rejected` 직접 send

`safeSend()`는 다음을 반환해야 한다.

```text
sent
closed
backpressure-warn
backpressure-critical
send-error
```

기본 정책 후보:

| 채널 | warn | critical | 처리 |
|---|---:|---:|---|
| control server -> client | 64 KiB | 256 KiB | critical이면 terminate 후 reconnect |
| output server -> client | 1 MiB | 4 MiB | critical이면 output terminate 후 snapshot replay |
| unified legacy | 1 MiB | 4 MiB | 기존 동작 보호를 위해 terminate 전 observe 기간 필요 |
| frontend control client -> server | 64 KiB | 256 KiB | `send()`가 backpressure 실패 반환 |

수치는 초기값이다. 실제 default 승격 전에는 telemetry로 조정해야 한다.

### Output coalescing

output은 drop보다 replay 기반 회복이 안전하다.

- active visible session output은 frame 단위 또는 짧은 interval 단위로 coalesce한다.
- output socket이 severe backpressure에 걸리면 socket을 끊고 snapshot replay로 복구한다.
- hidden terminal은 구독 자체를 줄이거나 output write를 중단하는 기존 P0 개선과 결합한다.
- replay pending output queue는 현재 `SessionManager.getReplayQueueLimit()` 기반 tail 유지와 충돌하지 않게 한다.

### Frontend incoming backlog

브라우저는 incoming message backlog를 `WebSocket.bufferedAmount`로 볼 수 없다. 따라서 output socket 수신 측은 별도 지표가 필요하다.

- output message receive count/bytes
- `onOutput` 호출부터 xterm write callback까지 latency
- `inFlightOutputRef` depth/bytes/oldest age
- snapshot save duration
- main thread long task count

이 지표가 나쁘면 split WS만으로 input latency가 좋아지지 않을 수 있다.

## 기존 단일 WS와 호환/마이그레이션 전략

### Compatibility 원칙

- `/ws?token=<jwt>`는 계속 unified mode로 동작한다.
- `ClientWsMessage`와 `ServerWsMessage`의 기존 message type은 제거하지 않는다.
- split mode 필드는 optional additive로 시작한다.
- 서버는 `channel` query가 없으면 기존 `WsClientMeta`와 동일한 unified connection으로 처리한다.
- frontend feature flag가 꺼져 있으면 기존 `WebSocketContext` 동작을 유지한다.

### 단계별 rollout

1. Observe
   - single WS에 `safeSend()`와 `bufferedAmount` telemetry만 추가한다.
   - split channel은 만들지 않는다.
   - input latency와 output backlog baseline을 확보한다.

2. Shadow control
   - control socket을 추가로 열지만 실제 input은 기존 unified socket으로 보낸다.
   - control ping RTT, handshake, reconnect 안정성만 검증한다.
   - input duplicate 전송은 금지한다.

3. Split output receive
   - output socket이 subscription output을 받는다.
   - control socket은 subscribe/ACK/input을 담당한다.
   - legacy unified path는 fallback으로 유지한다.

4. Split default candidate
   - localhost/manual validation에서 split mode를 기본 후보로 켠다.
   - 실패 시 unified mode로 자동 fallback하되, fallback은 명시 telemetry로 남긴다.

5. Unified deprecation은 별도 판단
   - 단일 WS는 오래 유지하는 것이 좋다. packaged runtime, 오래된 frontend bundle, test harness, 외부 automation이 `/ws` legacy를 쓸 수 있다.

### Fallback 조건

자동 fallback은 조용히 일어나면 안 된다.

- control handshake 실패
- output handshake 실패
- pair token mismatch
- protocol version mismatch
- split mode에서 연속 reconnect 실패

이 경우 `transport:fallback` debug event와 `/api/sessions/telemetry` 지표에 남기고 unified로 전환한다.

### Phase 0: 기준 지표와 문서화

완료 조건:

### Phase 1: Unified mode 안전화

작업:

- server send를 `safeSend()`로 통합하는 계획을 구현한다.

- legacy unified mode에서도 behavior flag를 `observe`로 두고 terminate 정책은 바로 켜지 않는다.

완료 조건:

- single WS만으로도 backpressure가 관측 가능하다.
- 기존 input outbox는 `backpressure` 실패를 queue/reject로 처리할 수 있다.

### Phase 2: Server split channel infrastructure

작업:

- `WsRouter.handleUpgrade()`에서 `channel` query를 파싱한다.

- unified connection과 split connection을 동시에 지원한다.
- `clientGroupId`, `connectionId`, `channelRole`, `pairToken` handshake를 추가한다.
- group-level heartbeat cleanup을 구현한다.

완료 조건:

- `/ws?token=...` legacy test가 그대로 통과한다.
- `/ws?token=...&channel=control`과 output pair handshake가 인증/만료/중복 연결 케이스를 통과한다.

### Phase 3: Frontend transport abstraction

작업:

- `WebSocketContext` 내부에 `UnifiedTransport`와 `SplitTransport`를 둔다.
- context 외부 API는 유지한다.
- split feature flag를 추가한다.
- channel별 stale socket guard와 reconnect state를 둔다.
- `setWsClientId()`는 control handshake의 `clientGroupId`로 갱신한다.

완료 조건:

- feature flag off에서 기존 동작 diff가 없다.
- shadow control mode에서 control socket RTT와 reconnect telemetry가 수집된다.

### Phase 4: Subscription/replay group migration

작업:

- `sessionSubscribers`를 group 기반으로 확장한다.
- `subscribe/unsubscribe`는 control에서 받고 output socket으로 snapshot/output을 보낸다.
- replay pending과 screen repair pending을 group/session/epoch 기반으로 이동한다.
- `screen-snapshot:ready`와 repair ACK는 control에서 받아 output pending을 해제한다.
- `session:ready` ordering 정책을 확정한다.

완료 조건:

- output reconnect 중 stale snapshot ACK가 input gate를 열지 않는다.
- replay pending 중 input queue가 group-level pending state를 올바르게 참조한다.
- 기존 `input:rejected` 경로가 split에서도 debug capture에 남는다.

### Phase 5: Backpressure enforcement

작업:

- role별 high-water/critical threshold를 적용한다.
- output critical backpressure 시 output socket terminate 후 replay 복구한다.
- control backpressure 시 input send failure를 반환하고 outbox policy를 탄다.
- `/api/sessions/telemetry`의 `WsRouterObservabilitySnapshot`에 channel별 지표를 추가한다.

완료 조건:

- 느린 output client가 server heap과 input latency를 무한히 악화시키지 않는다.
- control socket bufferedAmount가 high-water 이상 지속되는 상황이 관측 및 복구된다.

### Phase 6: Rollout과 회귀 검증

작업:

- split mode를 localhost/manual validation에서 켠다.
- 기존 terminal 입력 안정화 E2E와 paste/IME/keyboard regression을 모두 돌린다.
- output flood와 hidden terminal scenario를 추가한다.
- 문서화와 rollback flag를 확정한다.

완료 조건:

- unified fallback이 가능하다.
- split mode에서 output flood 중 입력 누락/중복이 없다.
- input latency p95가 baseline보다 개선된다.

## 리스크

1. 복잡도 증가
   - 단일 socket state machine이 control/output/group/epoch state machine으로 늘어난다.
   - replay pending이 socket 객체에서 group/session으로 이동하므로 stale ACK 버그 위험이 있다.

2. ordering hazard
   - snapshot은 output으로 받고 ACK는 control로 보내면 message ordering이 단일 socket 때와 달라진다.
   - 모든 replay/repair message에 `replayToken`, `repairToken`, `subscriptionEpoch` 검증이 필요하다.

3. blind input 위험
   - output socket이 죽은 상태에서 control socket만 살아 있으면 input은 서버에 들어가지만 사용자는 결과를 보지 못한다.
   - 기본 정책은 output unavailable 중 input queue/reject여야 한다.

4. 브라우저 main thread 병목 미해결
   - split WS가 있어도 xterm write와 snapshot save가 main thread를 오래 점유하면 입력 capture가 늦다.
   - hidden terminal unmount/unsubscribe, output scheduler, snapshot storage 개선을 병행해야 한다.

5. 서버 event loop 병목 미해결
   - output send loop가 길면 control message 처리도 같은 Node event loop에서 지연될 수 있다.
   - server-side output batching/yielding과 safeSend가 필요하다.

6. pair token과 clientGroupId 보안
   - clientGroupId를 client가 임의 지정하면 broadcast self-exclusion이나 group state가 오염될 수 있다.
   - 서버 발급 group ID와 TTL pair token이 필요하다.

7. 테스트 난이도
   - 실제 input latency 개선은 환경 의존성이 크다.
   - Playwright의 `insertText`와 실제 Windows Korean IME 수동 입력은 다르므로 둘 다 필요하다.

8. socket 수 증가
   - 브라우저 탭마다 WebSocket이 2개가 된다.
   - 현재는 로컬 앱이라 부담이 작지만, 장시간 다중 탭/다중 workspace 사용에서는 heartbeat와 cleanup 검증이 필요하다.

- 입력 중복 0회.
- Enter 포함 queued input이 timeout/reconnect 실패 후 뒤늦게 실행되지 않는다.
- output flood 중 `captureToSendMs`와 `sendToServerReceiveMs` p95가 baseline보다 개선된다.
- control socket `bufferedAmount`가 high-water를 지속적으로 넘지 않는다.
- output socket backpressure가 server heap 증가로 이어지지 않고 terminate/replay로 회복된다.
- debug capture에서 reject/queue 이유가 `transport-closed`, `backpressure`, `output-disconnected`, `timeout-enter-safety` 등으로 명시된다.

## 성공 기준

장기적으로 split WS를 기본 후보로 보려면 다음 조건이 필요하다.

1. legacy unified mode가 계속 통과한다.
2. split mode에서 terminal input, paste, IME, replay, repair, workspace broadcast 회귀가 없다.
3. output flood 중 input path latency가 유의미하게 개선된다.
4. 느린 output client가 server/client memory를 무한히 증가시키지 않는다.
5. output socket 장애가 input blind execution으로 이어지지 않는다.
6. fallback이 조용히 발생하지 않고 telemetry에 남는다.

## 최종 권고

split WebSocket은 장기 개선안으로 타당하다. 하지만 바로 물리 socket을 나누기 전에 unified mode의 `safeSend`, `bufferedAmount` 관측, frontend output backlog telemetry를 먼저 넣어야 한다. 그래야 분리 후 개선 폭을 측정할 수 있고, 분리 자체가 만든 replay/order/reconnect 버그도 구분할 수 있다.

구현 순서는 다음이 가장 안전하다.

1. unified mode backpressure 관측과 `safeSend()` 공통화.
2. transport abstraction 도입.
3. 서버 client group/connection registry 추가.
4. split control/output handshake를 feature flag 뒤에 추가.
5. subscription/replay state를 group-level로 이전.
6. output backpressure terminate/replay와 input latency 검증.

이 방식이면 기존 단일 WS 계약을 깨지 않으면서, output이 많을 때 keyboard input path를 더 독립적으로 운용할 수 있다.
