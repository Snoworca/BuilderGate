# BuilderGate Memory Growth and Browser Input Lag Research

작성일: 2026-06-17

## 요약

BuilderGate를 오래 사용할수록 서버 메모리와 브라우저 메모리가 증가하고, 브라우저에서 키보드 입력 반응이 느려지며, 새로고침하면 일시적으로 회복되는 증상은 단일 원인보다 여러 누적 경로가 겹친 결과로 보인다.

가장 가능성이 높은 핵심 원인은 다음 네 가지다.

1. 프론트엔드가 숨겨진 워크스페이스와 탭의 xterm 런타임, WebSocket 구독, observer, scrollback을 제한 없이 유지한다.
2. 보이지 않는 터미널도 서버 output을 계속 xterm에 쓰고 snapshot 저장을 예약하므로 브라우저 메인 스레드 부하가 계속 증가한다.
3. 서버와 브라우저 양쪽 WebSocket 송신 경로에 `bufferedAmount` 기반 backpressure 처리가 없어 느린 클라이언트나 막힌 소켓에 큐가 쌓일 수 있다.
4. 서버 headless terminal 출력 큐와 일부 세션 종료/복구 상태가 명시적으로 정리되지 않아 서버 프로세스 메모리와 처리 비용이 증가할 수 있다.

## 조사 범위와 방식

- 메인 에이전트가 코드 경로를 직접 검색하고 라인 단위로 확인했다.
- 최초 요청에 따라 5개 서브에이전트를 파견해 backend session/terminal lifecycle, WebSocket/router lifecycle, frontend terminal runtime, workspace/store/cache, dev server/daemon/log 경로를 독립 분석했다.
- 추가 요청에 따라 3개 서브에이전트를 더 파견해 브라우저 입력 파이프라인, React/WebSocket 렌더링 압력, 브라우저 DOM/observer/storage 보존 경로를 독립 분석했다.
- 이 문서는 읽기 전용 정적 분석 결과다. 코드 수정이나 성능 프로파일링 실행은 수행하지 않았다.

## 증상별 결론

### 서버 메모리 증가

서버 쪽에서 가장 위험한 지점은 `SessionManager`의 headless 출력 처리다. `queueHeadlessOutput()`는 PTY 출력마다 `pendingOutputChunks.push(data)`를 수행하지만 큐 크기나 바이트 제한이 없다.

- 위치: `server/src/services/SessionManager.ts:2252`
- 관련 위치: `server/src/utils/headlessTerminal.ts:120`
- 영향: PTY 출력 속도가 xterm-headless write 처리 속도보다 빠르면 chunk가 계속 쌓인다. 이후 `shift()`로 처리하므로 큐가 커질수록 처리 비용도 증가한다.
- 해결 방향: byte cap이 있는 deque/ring buffer나 coalesced tail buffer로 바꾸고, `Array.shift()`를 제거한다. 세션 삭제 시 `pendingOutputChunks`, `pendingHeadlessWrites`, `headlessWriteChain` 참조를 명시적으로 정리한다.

PTY가 자연 종료될 때 heavy session state가 남는 경로도 있다.

- 위치: `server/src/services/SessionManager.ts:393`, `server/src/services/SessionManager.ts:504`
- 근거: `ptyProcess.onExit()`가 `session:exited`만 broadcast하고 `deleteSession(id)` 또는 공통 cleanup을 호출하지 않는다.
- 영향: 사용자가 shell에서 `exit`를 실행하거나 PTY가 스스로 종료되면 `sessions` Map에 `SessionData`, headless terminal, snapshot cache, detector, CWD watcher 관련 참조가 남을 수 있다.
- 해결 방향: onExit에서는 heavy resource cleanup을 즉시 수행하고, UI에 필요한 exit metadata만 bounded tombstone으로 남긴다.

복구 상태도 세션 삭제 경로에서 일부 누락된다.

- 위치: `server/src/services/WorkspaceService.ts:66-67`, `server/src/services/WorkspaceService.ts:553-562`, `server/src/services/WorkspaceService.ts:587-592`
- 근거: `pendingRecoveryRestores`는 CWD-ready 경로에서만 삭제되고, tab/workspace 삭제 경로에서 삭제되지 않는다.
- 해결 방향: tab 삭제, workspace 삭제, restart old/new session cleanup에서 `pendingRecoveryRestores.delete(sessionId)`를 공통 호출한다. timeout도 둔다.

### 브라우저 메모리 증가와 새로고침 회복

프론트엔드는 현재 숨겨진 워크스페이스의 터미널 런타임을 무제한 유지한다.

- 위치: `frontend/src/App.tsx:41-42`, `frontend/src/App.tsx:571-574`, `frontend/src/App.tsx:604-647`
- 위치: `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx:148-159`
- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1515-1542`
- 근거: `MAX_ALIVE_WORKSPACES = 0`이 제한 없음이며, tab mode에서 모든 워크스페이스의 모든 탭이 render되고 비활성 탭만 `display: none` 처리된다. Runtime layer는 `status !== disconnected` 탭마다 `TerminalContainer`를 유지한다.
- 영향: xterm 인스턴스, 10000 line scrollback, addon, DOM, ResizeObserver, WebSocket subscription이 워크스페이스 전환 후에도 계속 살아 있다. 브라우저 새로고침은 이 상태를 한 번에 dispose하므로 입력 지연이 일시적으로 회복된다.
- 해결 방향: active/focused 터미널과 최근 N개 workspace만 live xterm으로 유지한다. 나머지는 unmount/unsubscribe하고 서버 headless snapshot과 screen repair로 복원한다.

숨겨진 터미널도 output을 계속 처리한다.

- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:1479-1484`
- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1123-1131`, `frontend/src/components/Terminal/TerminalView.tsx:1115-1120`
- 근거: `onOutput`이 visibility와 무관하게 `terminalRef.current?.write(data)`를 호출한다. `writeOutput()`은 매 chunk마다 `term.write()`를 호출하고 완료 후 snapshot save debounce를 예약한다.
- 영향: 사용자가 보지 않는 터미널도 xterm buffer 증가, xterm write callback, snapshot serialization, localStorage 저장 비용을 만든다. 출력이 많은 hidden session이 있으면 현재 활성 터미널의 키 입력 이벤트 처리도 메인 스레드에서 밀릴 수 있다.
- 해결 방향: `isVisible=false` 터미널은 브라우저 xterm write를 중단한다. hidden 중 output을 클라이언트 문자열 버퍼로 쌓지 말고 visible 전환 시 서버 snapshot/repair를 요청한다.

HostSlot observer와 window listener도 숨겨진 탭 수만큼 늘어난다.

- 위치: `frontend/src/components/Terminal/TerminalHostSlot.tsx:31-57`, `frontend/src/components/Terminal/TerminalHostSlot.tsx:82-90`
- 근거: slot마다 `ResizeObserver`가 host/root를 observe하고 `window.resize`, capture `scroll` listener를 등록한다. `measure()`는 `getBoundingClientRect()`를 두 번 호출한다.
- 해결 방향: Provider 단위의 단일 observer와 단일 scroll/resize listener로 합치고, visible host만 RAF batch로 측정한다.

### 키보드 입력 지연

일반 키 입력 자체는 React state에 직접 누적되는 구조가 아니다. 주요 경로는 xterm `onData` -> input sequencer -> WebSocket send이며 ref 기반이다.

- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1785`
- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:1492`
- 위치: `frontend/src/utils/terminalInputSequencer.ts:55`

그러나 입력 지연을 만들 수 있는 병목은 여러 개 있다.

첫째, 브라우저 WebSocket 송신도 backpressure를 보지 않는다.

- 위치: `frontend/src/contexts/WebSocketContext.tsx:472-475`
- 근거: `ws.send(JSON.stringify(msg))` 후 즉시 `{ ok: true }`를 반환하며 `ws.bufferedAmount`를 확인하지 않는다.
- 영향: 브라우저 송신 버퍼가 커져도 입력은 "전송됨"으로 처리된다. echo/output이 점점 늦게 보이고, 새로고침 시 socket과 버퍼가 초기화되어 회복될 수 있다.
- 해결 방향: `SendResult`에 `backpressure` reason을 추가하고, input 전송 전후 high-water mark를 확인한다. 초과 시 bounded queue 또는 명시적 reject/재연결 상태로 전환한다.

둘째, 서버 WebSocket 송신도 backpressure를 보지 않는다.

- 위치: `server/src/ws/WsRouter.ts:1496-1539`, `server/src/ws/WsRouter.ts:1708-1719`, `server/src/ws/WsRouter.ts:705-716`
- 위치: `server/src/services/SessionManager.ts:1976-1983`
- 근거: `routeSessionOutput()`, `broadcastAll()`, `broadcastWs()`가 `readyState === OPEN`만 보고 `send()`한다. `bufferedAmount`, send callback error, per-client queue cap이 없다.
- 영향: 느린 브라우저 탭이나 잔존 socket마다 output buffer가 누적된다. terminal output은 고빈도라 서버 메모리와 JSON 직렬화 비용이 같이 증가한다.
- 해결 방향: 모든 WS send를 `safeSend()`로 통합하고 `bufferedAmount` high-water mark 초과 client는 terminate한다. terminal output은 drop/coalesce 또는 bounded per-client queue를 둔다.

셋째, xterm output write backlog가 관측도 제한도 없이 쌓일 수 있다.

- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:1479`
- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1123`
- 근거: `inFlightOutputRef`는 push/shift만 하고 throttle, coalescing, queue depth cap, write latency telemetry에 쓰이지 않는다.
- 해결 방향: output 전용 write scheduler를 두고 write callback 완료 전 chunk를 병합하거나 frame 단위로 flush한다. queue bytes, depth, oldest age, write latency를 기록하고 임계값 초과 시 hidden output write를 중단한다.

넷째, 입력 디버그 메타데이터 계산이 키 입력마다 중복 수행된다.

- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1785`
- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:1492`, `frontend/src/components/Terminal/TerminalContainer.tsx:624-654`
- 위치: `frontend/src/utils/terminalDebugCapture.ts:318-353`
- 근거: `buildTerminalInputDebugPayload()`는 regex, `Array.from`, `TextEncoder`, `Intl.Segmenter` 생성/segment를 수행한다. debug capture enabled 여부와 무관하게 입력 경로에서 여러 번 호출된다.
- 해결 방향: 최초 입력 지점에서 lightweight metadata를 한 번만 생성해 전달한다. grapheme/preview 등 비싼 계산은 debug capture enabled일 때 lazy 계산한다. `TextEncoder`와 `Intl.Segmenter`는 재사용한다.

다섯째, 입력 queue timer fan-out이 가능하다.

- 위치: `frontend/src/components/Terminal/TerminalView.tsx:473-479`
- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:490-518`, `frontend/src/components/Terminal/TerminalContainer.tsx:737-739`, `frontend/src/components/Terminal/TerminalContainer.tsx:801-803`
- 근거: transient-blocked/reconnect 상태에서 입력이 queue될 때마다 expiry timer가 새로 생성된다. queue flush 시 기존 expiry timer를 즉시 clear하지 않는다. 일부 retry `setTimeout`은 별도 ref/set으로 추적되지 않는다.
- 해결 방향: queue별 단일 expiry timer로 바꾸고 earliest expiry 기준으로 재스케줄한다. flush/reject 시 timer를 즉시 clear하고 retry timer도 추적한다.

### Snapshot과 localStorage 비용

snapshot 저장은 브라우저 메인 스레드에서 동기 localStorage scan을 수행한다.

- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1063-1113`
- 위치: `frontend/src/utils/terminalSnapshot.ts:192-224`, `frontend/src/utils/terminalSnapshot.ts:248-293`, `frontend/src/utils/terminalSnapshot.ts:303-323`
- 근거: `setTerminalSnapshotWithQuotaRecovery()`가 저장 전 eviction을 수행하고, 이 과정에서 `localStorage.length` 전체 순회, snapshot/removal key 수집, 일부 JSON parse, 정렬을 수행한다.
- 영향: output이 많은 live terminal이 여러 개면 2초 debounce마다 동기 storage scan이 반복된다.
- 해결 방향: 평상시에는 바로 `setItem`하고 `QuotaExceededError`에서만 eviction한다. 또는 snapshot index와 총량 캐시를 별도 유지한다. 가능하면 IndexedDB 같은 async storage로 옮긴다.

삭제 marker 정리도 cleanup에 의존한다.

- 위치: `frontend/src/utils/terminalSnapshot.ts:353-391`
- 위치: `frontend/src/components/Terminal/TerminalView.tsx:1988-1990`
- 위치: `frontend/src/hooks/useWorkspaceManager.ts:271-276`, `frontend/src/hooks/useWorkspaceManager.ts:432-439`
- 근거: `terminal_snapshot_remove_*` tombstone과 module-level Set 정리가 `TerminalView` unmount cleanup에 의존한다. 이미 unmounted였거나 retained 밖 세션이면 회수 경로가 약하다.
- 해결 방향: tombstone에 TTL을 두고 startup/snapshot-save 전에 만료 정리한다. close/delete 완료 후 mounted runtime이 없으면 removal request를 즉시 clear한다.

Mosaic layout도 stale localStorage를 남길 수 있다.

- 위치: `frontend/src/hooks/useMosaicLayout.ts:22`, `frontend/src/hooks/useMosaicLayout.ts:110-126`, `frontend/src/hooks/useMosaicLayout.ts:227-235`
- 근거: `mosaic_layout_<workspaceId>`는 workspace 삭제 시 제거되지 않고, unmount cleanup이 다시 save할 수 있다.
- 해결 방향: `removeMosaicLayout(workspaceId)`를 만들고 삭제 전에 호출한다. 삭제로 인한 unmount에서는 cleanup save를 skip한다.

### React 렌더링과 재생성 압력

WebSocket 상태 변화가 xterm 재생성까지 전파될 수 있다.

- 위치: `frontend/src/contexts/WebSocketContext.tsx:434-496`, `frontend/src/contexts/WebSocketContext.tsx:559-563`
- 위치: `frontend/src/components/Terminal/TerminalContainer.tsx:910-937`, `frontend/src/components/Terminal/TerminalView.tsx:2012-2024`
- 근거: `send` callback이 `status`를 dependency로 가진다. status 변화가 actions context identity 변경을 만들고, resize/request callback 체인을 통해 `TerminalView` 생성 effect dependency까지 도달할 수 있다.
- 해결 방향: `send`는 stable callback으로 두고 status는 ref에서 읽는다. xterm 생성 effect는 `sessionId`와 불변 setup만 의존하게 줄인다.

status/cwd 이벤트도 App 전체 평가를 유발한다.

- 위치: `frontend/src/hooks/useWorkspaceManager.ts:527`
- 위치: `frontend/src/App.tsx:506`, `frontend/src/App.tsx:571`
- 위치: `frontend/src/components/Workspace/WorkspaceSidebar.tsx:47`
- 근거: 실제 값이 바뀌면 `tabs` 배열 전체가 새로 생성되고 App, sidebar, tabbar, runtime layer가 다시 평가된다. Sidebar는 workspace마다 `tabs.filter`를 반복한다.
- 해결 방향: runtime status/cwd를 `tabs` 배열에서 분리해 sessionId keyed external store/selectors로 구독하게 한다. Sidebar counts는 memoized aggregate로 넘긴다.

`TerminalHostSlot`의 inline props도 listener churn을 만든다.

- 위치: `frontend/src/components/Terminal/TerminalHostSlot.tsx:31-57`, `frontend/src/components/Terminal/TerminalHostSlot.tsx:75-97`
- 위치: `frontend/src/App.tsx:640-647`
- 근거: `measure`가 `style`, `onContextMenu`, `onPointerDown`에 의존하고 App은 inline style/callback을 매 render 새로 만든다. 이로 인해 ResizeObserver/window listener effect가 상위 render마다 재구독될 수 있다.
- 해결 방향: tab color style과 host interaction callback을 안정화한다. interaction은 ref/context에 저장하고 measurement effect dependency에서 제거한다.

### Daemon, dev server, 로그

로그 누적은 heap leak은 아니지만 장시간 사용 시 I/O와 편집기/터미널 처리 비용을 키울 수 있다.

- 위치: `server/src/index.ts:606-614`
- 위치: `server/src/ws/WsRouter.ts:1663-1681`, `server/src/ws/WsRouter.ts:1691-1705`
- 위치: `tools/start-runtime.js:260-299`
- 위치: `tools/daemon/launcher.js:426-433`
- 근거: `[TerminalObs]`가 60초마다 session/ws snapshot을 JSON 직렬화해 출력하고, daemon tee는 stdout/stderr write마다 동기 `appendFileSync()`를 호출한다. rotation도 없다.
- 현재 관측된 대형 로그 예:
  - `.devjs.2002.codexp-recovery.out.log`: 약 531 MB
  - `.snoworca/dev-2222.out.log`: 약 546 MB
  - `runtime/buildergate-daemon.log`: 약 297 MB
  - `.devjs.log`: 약 39.5 MB
- 해결 방향: `TerminalObs`는 기본 비활성 또는 debug level로 내리고, event 전문 대신 count/max/last id만 기록한다. daemon log는 size/date rotation과 retention을 둔다. tee는 `appendFileSync` 대신 write stream과 backpressure를 사용한다.

프로세스 정리 경로도 위험하다.

- 위치: `tools/daemon/state-store.js:275-288`, `tools/daemon/launcher.js:488-490`, `tools/daemon/stop-client.js:118-120`, `tools/daemon/stop-client.js:290-295`
- 근거: fatal daemon state가 `appPid`, `sentinelPid`를 보존하지만 stop/start 경로가 fatal state를 stop 대상으로 보지 않는다.
- 해결 방향: fatal 상태라도 PID가 있으면 validation 후 stop 가능하게 만들고, start 전 stale fatal PID를 검사/정리한다.

- 위치: `dev.js:54-70`, `dev.js:74-107`
- 근거: cleanup은 `taskkill` spawn 후 대기하지 않고, `process.on('exit')`에서는 async cleanup 보장이 없다. frontend start timer도 signal 처리 시 clear되지 않는다.
- 해결 방향: idempotent async cleanup을 만들고 child process tree 종료를 기다린다. server/frontend 중 하나가 close되면 sibling도 정리한다.

## 수정 우선순위

### P0: 증상 완화와 누적 차단

1. `MAX_ALIVE_WORKSPACES`를 유한값으로 바꾸고 hidden workspace/tab의 `TerminalContainer`와 `TerminalView`를 unmount한다.
2. hidden terminal은 WS subscription과 xterm write를 중단하고 visible 전환 시 서버 snapshot/repair로 복원한다.
3. 클라이언트와 서버 WS send를 `safeSend()`로 통합하고 `bufferedAmount` high-water mark, send callback error 처리, terminate 정책을 추가한다.
4. 서버 headless output queue를 bounded 구조로 바꾸고 `shift()`를 제거한다.
5. PTY onExit에서 heavy session resource cleanup을 수행한다.
6. `TerminalObs` 로그를 기본 비활성화하거나 축약하고 daemon/dev log rotation을 추가한다.

### P1: 입력 지연과 브라우저 메인 스레드 부하 감소

1. xterm output write scheduler를 추가해 chunk 병합, frame 단위 flush, queue cap, write latency telemetry를 구현한다.
2. snapshot localStorage eviction을 quota error fallback 중심으로 바꾸거나 IndexedDB로 이동한다.
3. `TerminalHostSlot` observer/listener를 Provider 단위로 합치고 visible host만 측정한다.
4. `WebSocketContext.send`를 stable callback으로 만들고 `TerminalView` 생성 effect dependency를 축소한다.
5. input debug payload 계산을 lazy/one-pass로 바꾸고 `TextEncoder`, `Intl.Segmenter`를 재사용한다.
6. 입력 queue와 transport outbox timer를 queue별 단일 timer로 바꾼다.

### P2: 상태 정리와 장기 안정성

1. `terminalRefsMap`, `workspaceVisitOrder`, `aliveWorkspaceIds`를 `wm.tabs`/`wm.workspaces` 기준으로 prune한다.
2. `pendingRecoveryRestores`를 모든 세션 삭제/restart 경로에서 삭제한다.
3. node-pty `onData`/`onExit` disposer를 저장하고 session cleanup에서 dispose한다.
4. `terminal_snapshot_remove_*` tombstone TTL과 startup cleanup을 추가한다.
5. workspace 삭제 시 `mosaic_layout_<workspaceId>`를 삭제하고 삭제 unmount에서는 cleanup save를 skip한다.
6. 직접 `/api/sessions/:id` DELETE가 workspace tab lifecycle과 분리된 문제를 정리한다.

## 검증 계획

구현 후 다음 지표로 재현/성공을 확인하는 것이 좋다.

1. 브라우저 Performance profile에서 30분 사용 후 keydown -> xterm echo latency를 측정한다.
2. Chrome Memory heap snapshot에서 detached DOM node, xterm Terminal, ResizeObserver, terminal container 수가 workspace/tab close 후 감소하는지 확인한다.
3. 서버에서 session count, headless pending bytes/chunks, WS client bufferedAmount, per-session output write lag를 로그 또는 debug endpoint로 관측한다.
4. hidden workspace에서 대량 output을 발생시켜도 active terminal 입력 latency가 증가하지 않는지 확인한다.
5. shell 내부 `exit`, workspace/tab delete, restart 직후 close, browser refresh, WS reconnect를 각각 반복해 resource count가 원래 수준으로 돌아오는지 확인한다.
6. daemon/dev runtime restart 후 stale node/vite/tsx process와 대형 로그 증가가 없는지 확인한다.

## 무이슈 또는 낮은 위험으로 확인한 부분

- `TerminalView`의 주요 DOM listener, `ResizeObserver`, timer cleanup은 unmount 시 존재한다.
- WebSocket `subscribeSession()`은 cleanup과 300ms grace 후 unsubscribe 경로가 있다.
- 클라이언트 debug event 배열은 400개로 bounded라 그 자체가 무한 heap leak은 아니다.
- `frontend/src/services/api.ts`는 thin wrapper이고 자체 cache/retention은 보이지 않는다.
- `authMiddleware.ts`는 요청 단위 user 설정 외 장기 참조를 보관하지 않는다.

## 결론

현재 증상은 "닫아도 메모리가 바로 OS에 반환되지 않는 정상 현상"만으로 설명하기 어렵다. 코드상 실제 retention/backpressure 포인트가 여러 개 있다. 특히 브라우저 입력 지연은 active terminal의 키 입력 처리 자체보다, hidden terminal을 계속 살려 둔 상태에서 output write, snapshot, observer, WS buffer가 누적되는 구조가 더 직접적인 원인으로 보인다.

가장 먼저 hidden terminal live 유지 정책을 제한하고, WS/xterm/headless output backpressure를 넣어야 한다. 그 다음 snapshot/storage, observer/listener, debug metadata, stale state cleanup을 정리하면 장시간 사용 시 점진적 slowdown과 메모리 증가를 크게 줄일 수 있다.
