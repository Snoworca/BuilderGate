# BuilderGate 장시간 사용 지연 및 세션 프로세스 누수 후속 연구

작성일: 2026-06-24

대상: BuilderGate 서버/프론트엔드 터미널 런타임, 세션 종료, 워크스페이스 종료, 데몬 종료 수명주기

## 결론

현재 증상인 "쓰면 쓸수록 점점 느려짐"은 프론트엔드 단독 문제보다 서버 런타임/세션 수명주기 문제가 더 강하게 지목된다. 다만 이번 조사는 controlled before/after 재현, queue growth 계측, 입력 지연 상관관계 측정을 완료한 것은 아니므로 확정 인과가 아니라 정적 분석과 현재 실행 상태가 가리키는 우선 가설로 봐야 한다. 프론트엔드는 숨은 터미널 런타임을 많이 살려 두고 계속 구독/렌더링하는 구조라 서버 누적 상태의 비용을 증폭할 수 있다.

세션 닫기를 눌러도 PowerShell, bash, WSL, 하위 dev server 같은 프로세스가 남을 수 있다는 의심은 타당하다. UI의 세션 닫기와 워크스페이스 삭제 경로는 서버의 `SessionManager.deleteSession()`까지 도달하지만, 현재 구현은 ProjectMaster 레벨의 프로세스 트리 종료가 아니라 `data.pty.kill()` 호출에 의존한다. 종료 완료 대기, PID 소유권 재검증, 재귀 하위 프로세스 종료, WSL Linux-side 프로세스 종료 확인이 없다.

사용자 정정에 따라 이전 표현은 "세션 종료"의 오타로 본다. 본 문서는 BuilderGate의 tab/session 삭제, workspace 삭제, 앱/데몬 shutdown 관점에서 세션 종료가 실제 OS 프로세스 종료까지 보장하는지 확인했다. 워크스페이스 삭제는 세션 레지스트리 제거를 호출하지만 실제 OS 프로세스 종료를 검증하지 않는다. 앱/데몬 종료는 세션 삭제가 아니라 CWD snapshot/flush와 process exit에 기대며, 저장된 탭은 다음 시작 시 orphan recovery로 다시 세션을 만들 수 있다.

## 조사 방식

사용자 요청에 따라 진단용 5개 서브에이전트와 해결 계획용 5개 서브에이전트를 분리했다.

진단 에이전트:

- 프론트엔드 런타임/구독/스냅샷 누적 분석
- 서버 `SessionManager`, headless, WebSocket 누적 분석
- 세션 닫기/워크스페이스 삭제/프로세스 트리 정적 분석
- 워크스페이스 삭제, 앱 shutdown, 데몬 stop 수명주기 분석
- 운영 증거: 현재 PID, 프로세스 트리, 저장된 workspace 상태, telemetry/log 분석

해결 계획 에이전트:

- 세션 소유 프로세스 트리 종료 설계
- 서버 headless/WebSocket backpressure 설계
- 프론트엔드 runtime residency/output backpressure 설계
- workspace/session lifecycle 및 orphan recovery 정합성 설계
- 검증, rollout, telemetry, rollback 계획

추가로 메인 에이전트가 핵심 경로를 직접 교차 확인했다. 파일 읽기는 UTF-8 기준으로 수행했고, 조사 중 프로세스 종료나 코드 수정은 하지 않았다.

## 서버 원인 분석

### 1. Headless output queue가 무제한에 가깝다

`SessionManager`는 PTY 출력마다 headless 처리 큐에 데이터를 넣는다.

- `server/src/services/SessionManager.ts:392`: PTY `onData`에서 출력 처리.
- `server/src/services/SessionManager.ts:2224`: `pendingOutputChunks.push(data)`.
- `server/src/services/SessionManager.ts:2269`: 완료 시 `pendingOutputChunks.shift()`.
- `server/src/services/SessionManager.ts:2300`: degraded 전환 시 pending chunk 전체를 join한 뒤 truncate.

문제:

- byte/chunk cap이 없다.
- `Array.shift()`는 큐가 커질수록 비용이 증가한다.
- headless write가 밀리면 live output도 같이 느려진다.
- degraded 경로도 큰 큐를 한 번에 합칠 수 있어 메모리 spike가 가능하다.

### 2. 서버 WebSocket 송신에 backpressure가 없다

`WsRouter`와 `SessionManager.broadcastWs()`는 `ws.bufferedAmount`나 send callback 실패 정책 없이 `ws.send()`를 직접 호출한다.

- `server/src/ws/WsRouter.ts:1538`: terminal output direct send.
- `server/src/ws/WsRouter.ts:1712`: `broadcastAll()` direct send.
- `server/src/ws/WsRouter.ts:1717`: `sendTo()` direct send.
- `server/src/services/SessionManager.ts:1948`: session event broadcast direct send.

느린 브라우저, hidden tab, stale socket, 대량 terminal output이 있을 때 서버와 브라우저 양쪽 큐가 계속 커질 수 있다.

### 3. PTY 자연 종료와 full cleanup이 분리되어 있다

- `server/src/services/SessionManager.ts:503`: PTY `onExit`는 `session:exited`만 broadcast한다.
- `server/src/services/SessionManager.ts:823`: heavy cleanup은 `deleteSession()`에만 있다.

따라서 shell 내부에서 `exit`하거나 PTY가 자연 종료되는 경우, `deleteSession()`과 같은 cleanup이 보장되지 않는다. CWD watcher, WS state, debug/replay state, headless state가 session registry와 함께 남을 수 있는 구조다.

### 4. CWD watcher와 debug/replay cleanup이 delete 경로에 몰려 있다

- `server/src/services/SessionManager.ts:1676`: session마다 `fs.watchFile` 기반 CWD polling.
- `server/src/services/SessionManager.ts:849`: CWD unwatch/unlink는 `deleteSession()` 내부.
- `server/src/services/SessionManager.ts:855`: WS replay/debug cleanup도 `deleteSession()` 내부.

정상 삭제가 아닌 자연 종료/비정상 종료/shutdown 경로와 cleanup 의미가 다르다.

## 프론트엔드 원인 분석

프론트엔드는 현재 증거만으로는 단독 원인보다 증폭 요인으로 보는 편이 더 보수적이다. 다만 오래 열어둔 workspace/tab이 많으면 브라우저 측에서도 충분히 느려질 수 있다.

### 1. live workspace cap이 꺼져 있다

- `frontend/src/App.tsx:41`: `MAX_ALIVE_WORKSPACES = 0`.
- `frontend/src/App.tsx:570`: runtime layer에 전달되는 tabs가 무제한 조건을 따른다.
- `frontend/src/App.tsx:603`: tab mode도 모든 workspace tab을 렌더링 후보로 유지한다.

SRS의 terminal runtime residency 요구는 존재하지만, 현재 앱은 실제로 제한을 적용하지 않는다.

### 2. 숨은 터미널도 mounted/subscribed 상태로 유지된다

- `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx:148`: disconnected가 아닌 모든 runtime entry mount.
- `frontend/src/components/Terminal/TerminalContainer.tsx:1468`: mount된 container는 session을 subscribe.
- `frontend/src/components/Terminal/TerminalContainer.tsx:1479`: output 수신 시 visibility와 무관하게 xterm write 호출.
- `frontend/src/components/Terminal/TerminalView.tsx:1123`: xterm write callback이 snapshot 저장 경로를 예약한다.

숨은 session output은 active terminal과 같은 브라우저 main thread, xterm buffer, snapshot/localStorage 경로를 공유한다.

### 3. 신규 안정화 유틸이 완전히 연결되어 있지 않다

- `frontend/src/hooks/useTerminalRuntimeResidency.ts`: residency hook은 존재하지만 `App.tsx`에 연결되지 않았다.
- `frontend/src/utils/terminalHiddenOutput.ts`, `terminalOutputScheduler.ts`, `webSocketBackpressure.ts`, `splitWebSocketLifecycle.ts`: 관련 유틸/테스트가 있으나 앱 핵심 경로에 완전 적용되지 않았다.
- 프론트 해결 계획 에이전트의 focused 확인에서 `useTerminalRuntimeResidency.test.ts`는 `inputReliabilityMode.ts`의 누락 export 때문에 실패했고, `npm run typecheck`도 미통합 타입 때문에 실패한다고 보고했다.

### 4. 작은 retention 버그

`TerminalRuntimeLayer`는 `terminalRefsMap`에 tab id별 ref를 추가하지만 삭제/eviction 시 명시적으로 prune하는 경로가 확인되지 않았다.

- `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx:157`: ref map set.

이 자체는 대형 slowdown의 1차 원인으로 보기 어렵지만, tab close/eviction 반복 시 작은 누적 메모리 원인이 된다.

### `deleteSession()`이 실제로 하는 일

`deleteSession()`은 ProjectMaster 내부 상태를 꽤 많이 정리한다.

- idle/running timer clear.
- `data.pty.kill()`.
- OSC/terminal title/foreground detector destroy.
- headless terminal dispose.
- CWD watcher/temp file cleanup.
- WS replay/debug state cleanup.
- resize replay timer cleanup.
- `this.sessions.delete(id)`.

그러나 OS 프로세스 종료 관점에서는 `node-pty`의 `kill()`에 의존한다. ProjectMaster 레벨에서 다음을 하지 않는다.

- 종료 완료까지 await.
- root PID identity 재검증.
- process tree leaf-first 종료.
- Windows `taskkill /T /F` fallback.
- POSIX process group 종료.
- WSL 내부 Linux PID/process group 종료.
- 종료 후 남은 descendant 검증.

### node-pty 한계

현재 의존성은 `node-pty`이며 Windows는 ConPTY 설정이다.

- `server/config.json5:14`: `useConpty: true`.
- `server/node_modules/node-pty/lib/unixTerminal.js:226`: Unix kill은 기본적으로 PTY child PID에 signal.
- `server/node_modules/node-pty/lib/windowsPtyAgent.js:133`: Windows ConPTY kill은 console process list를 사용.
- `server/node_modules/node-pty/lib/windowsPtyAgent.js:181`: console process list 수집 실패 시 shell PID fallback.

즉 console-attached shell은 대체로 죽을 수 있지만, detached child, `Start-Process`, `cmd start`, background/nohup/disown/setsid, WSL 내부 child, timeout/fallback 상황은 남을 수 있다.

## 세션 종료와 워크스페이스 삭제 확인

사용자 정정에 따라 확인 대상은 세션 종료다. 관련 경로는 BuilderGate tab/session 삭제와 workspace 삭제다.

### workspace delete

- `server/src/services/WorkspaceService.ts:203`: `deleteWorkspace`.
- `server/src/services/WorkspaceService.ts:216`: workspace의 session ids를 `deleteMultipleSessions()`로 삭제.
- `server/src/services/SessionManager.ts:1457`: `deleteMultipleSessions`는 `deleteSession()` 반복 호출.

결론: workspace delete는 `SessionManager` registry에서 세션을 제거한다. 그러나 실제 프로세스 트리 종료는 `deleteSession()`과 같은 한계를 가진다.

### tab delete

- `server/src/services/WorkspaceService.ts:303`: `deleteTab`.
- `server/src/services/WorkspaceService.ts:312`: tab session 삭제.

결론: tab delete도 session registry는 제거하지만 OS process exit 검증은 하지 않는다.

### tab restart

- `server/src/services/WorkspaceService.ts:336`: `restartTab`.
- `server/src/services/WorkspaceService.ts:345`: 새 session 생성.
- `server/src/services/WorkspaceService.ts:347`: old session 삭제.

새 session을 먼저 만든 뒤 old session을 삭제하는 순서는 사용자 작업 보존 측면에서 안전하다. 다만 old session process tree가 실제로 종료됐는지는 검증하지 않는다.

### app shutdown / daemon stop

- `server/src/index.ts:622`: SIGINT/SIGTERM shutdown 진입.
- `server/src/services/gracefulShutdown.ts:73`: `performGracefulShutdown`.
- `server/src/services/gracefulShutdown.ts:84`: `stopAllCwdWatching()`.
- `server/src/services/gracefulShutdown.ts:89`: `snapshotAllCwds()`.
- `server/src/services/gracefulShutdown.ts:91`: `forceFlush()`.

shutdown은 세션 삭제가 아니다. CWD watcher를 멈추고 workspace state를 flush한 뒤 process exit에 의존한다.

daemon stop도 외부 app PID identity와 flush evidence를 확인하는 구조는 비교적 안전하지만, 내부 session process tree를 직접 종료하는 lifecycle은 없다.

### orphan recovery

- `server/src/index.ts:444`: startup 후 `checkOrphanTabs()`.
- `server/src/services/WorkspaceService.ts:595`: missing session id를 orphan으로 보고 새 session 생성.

직접 `DELETE /api/sessions/:id`처럼 workspace tab metadata를 갱신하지 않는 삭제는 다음 startup에서 orphan recovery가 session을 다시 만들 수 있다. "의도적으로 종료한 session"과 "서버 재시작으로 사라진 recoverable session"을 구분하는 persisted lifecycle metadata가 필요하다.

## 문서/SRS와 현재 코드의 불일치

`docs/spec/30.buildergate-stability.srs.md`에는 runtime resource limit, terminal residency, WebSocket transport 안정화 요구가 `implemented`로 표시된 항목이 있다. 기존 `docs/research/2026-06-17.*` 문서들도 backend backpressure, frontend runtime lifecycle, split WebSocket 계획을 상세히 다룬다.

그러나 현재 소스와 실행 상태에는 다음이 남아 있다.

- `pendingOutputChunks`는 여전히 array push/shift 기반이다.
- `WsRouter` direct send 경로가 남아 있다.
- `MAX_ALIVE_WORKSPACES = 0`으로 live runtime cap이 꺼져 있다.
- `useTerminalRuntimeResidency()`는 실제 앱 경로에 연결되지 않았다.
- session close는 process-tree verified termination이 아니다.
- shutdown은 session lifecycle cleanup이 아니라 snapshot/flush 중심이다.

따라서 기존 안정화 문서는 설계 방향으로는 유효하지만, 현재 실행 증거 기준으로 "완료"라고 보기 어렵다.

## 해결 계획 합의안

### Phase 0. 타입/설정 drift 정리와 기준선 확보

observe flag를 추가하기 전에 현재 runtime-config/resource-limit 관련 drift를 먼저 정리한다. `/api/runtime-config`는 현재 제한된 public 설정만 노출하고 있고, 서브에이전트/리뷰 결과 기준으로 프론트 typecheck는 runtime config export, WebSocket protocol type, recovery-option export 등에서 실패한다. 서버 쪽도 resource limit 및 `WsRouterOptions` 계열 타입 drift가 보고되었으므로, 구현 착수 전 `server`/`frontend` build/typecheck가 어떤 기준에서 깨지는지 한 번 고정해야 한다.

우선 작업:

- 현재 `docs/spec/30.buildergate-stability.srs.md`의 implemented 상태와 실제 source/typecheck 상태 차이를 이슈로 기록한다.
- runtime config schema, public payload, frontend getter/export, tests의 이름과 타입을 맞춘다.
- 새 observe flag는 기존 drift를 정리한 뒤 additive field로 넣는다.
- 이 단계에서는 behavior 변경 없이 telemetry surface와 타입 계약만 회복한다.

### Phase 0.5. Observe mode 확장

기본 동작을 바로 바꾸지 말고 관측 가능한 guard를 먼저 넣는다. 단, 이 flag들은 Phase 0의 타입/설정 drift 정리 이후에 추가한다.

권장 flag/config:

- `headlessQueueGuard=observe`
- `serverWsSafeSend=observe`
- `clientWsBackpressure=observe`
- `frontendRuntimeResidency=off`
- `visibleOutputScheduler=observe`
- `wsTransportMode=unified`

대상:

- `server/src/index.ts:259`의 `/api/runtime-config`.
- `server/src/index.ts:288`의 `/api/sessions/telemetry`.
- `server/src/index.ts:597`의 `[TerminalObs]` 로그를 summary 중심으로 축소.

추가 telemetry:

- session별 PTY root PID, descendant count, cleanup reason/result.
- headless pending bytes/chunks, oldest age, overflow/degrade count.
- WS max `bufferedAmount`, queued/coalesced bytes, slow-client close count.
- frontend resident/evicted/restored tab count.
- visible output queue bytes, overflow recovery count.
- shutdown duration, terminated session count, skipped/unverified process count.

### Phase 1. 세션 소유 process-tree terminator 도입

새 유틸 후보:

- `server/src/utils/processTreeTerminator.ts`

`SessionManager.createSession()`에서 session 소유 metadata를 저장한다.

- root PID
- shell command/args/shell type
- platform
- Windows creation time / executable / command line
- POSIX `/proc/<pid>/stat` start ticks
- WSL Linux-side PID/process group metadata

소유권 정책:

- 기본 정책은 "BuilderGate 세션이 직접 만든 PTY root와 그 descendant"만 종료하는 것이다.
- PID 재사용 가능성을 고려해 PID만으로 force kill하지 않고 start time, executable, command line, cwd 또는 session token을 함께 검증한다.
- 사용자가 terminal 안에서 의도적으로 detached/background service를 띄운 경우는 제품 정책 결정이 필요하다. 초기 구현은 descendant tree 안에 남아 있는 프로세스만 종료 대상으로 삼고, breakaway/detached로 소유권 검증이 불가능한 프로세스는 force kill하지 않는다.
- unverified descendant가 남으면 UI/API/log에 `cleanupStatus=degraded`와 남은 PID count, reason을 남긴다. 사용자가 수동 정리해야 할 수 있음을 운영 로그와 telemetry에서 볼 수 있어야 한다.
- 절대 `taskkill /IM node.exe`, `killall`, process name 기반 broad kill을 사용하지 않는다.

삭제 전략:

1. 입력 수신을 중단하고 cleanup 시작 tombstone을 남긴다.
2. `pty.kill()`을 먼저 호출해 PTY/ConPTY 측 종료를 시도한다.
3. 짧은 timeout 후 root identity를 재검증한다.
4. Windows native는 검증된 root에 대해서만 `taskkill.exe /PID <pid> /T /F`를 fallback으로 사용한다. `shell: true`나 process name kill은 금지한다.
5. POSIX는 process group이 검증되면 `SIGTERM`, timeout 후 `SIGKILL` 순서로 종료한다. group 검증이 안 되면 소유권이 검증된 descendant에 한해 leaf-first 종료를 사용한다.
6. WSL은 Windows `wsl.exe` wrapper 종료와 별도로 Linux-side PID/process group을 기록해 `wsl.exe -e sh -lc` 기반으로 종료한다. 사용자 입력 문자열을 shell command에 직접 삽입하지 않는다.
7. 검증 실패 시 force kill하지 않고 degraded cleanup으로 기록한다.

수정 대상:

- `server/src/services/SessionManager.ts`
- `server/src/routes/sessionRoutes.ts`
- `server/src/services/WorkspaceService.ts`
- `server/src/services/gracefulShutdown.ts`

### Phase 2. Session lifecycle service로 의미 통합

새 서비스 후보:

- `server/src/services/SessionLifecycleService.ts`

목표:

- workspace/tab/direct session/shutdown/orphan recovery가 같은 lifecycle 의미를 사용한다.
- `deleteSession(id): boolean` 호환 wrapper는 유지하되, 새 경로는 `terminateSession(id, options)`처럼 async 결과와 reason을 돌려준다.
- PTY `onExit`도 full cleanup 공통 경로를 호출한다.
- node-pty `onData`/`onExit` disposer를 `SessionData`에 저장하고 cleanup에서 끊는다.

workspace tab에 additive lifecycle metadata를 둔다.

- `state: active | stopped`
- `recoverable: boolean`
- `reason: tab-delete | workspace-delete | tab-restart | direct-session-delete | internal-shutdown | daemon-stop | process-exit`
- `updatedAt`
- optional `previousSessionId` / `generation`

주의:

- `sessionId`를 nullable로 바꾸지 않는다.
- 기존 `workspaces.json`은 optional field로 migration한다.
- direct `DELETE /api/sessions/:id`가 workspace-owned session을 만나면 tab 삭제가 아니라 stopped/non-recoverable 상태로 표시하고 broadcast한다.

### Phase 3. 서버 headless queue와 WebSocket backpressure

`SessionManager`:

- `pendingOutputChunks: string[]`를 byte-counted deque로 교체한다.
- `Array.shift()` 제거.
- `pendingOutputMaxBytes`, `pendingOutputMaxChunks`, `writeLagWarnMs`, `overflowPolicy` 설정 추가.
- overflow 기본 정책은 silent drop이 아니라 headless degraded 전환과 telemetry 기록.
- degraded seed는 bounded tail만 사용하고 unbounded join을 금지한다.

`WsRouter`:

- 모든 `ws.send()`를 `safeSend()`로 통합한다.
- `WsClientMeta`에 send state를 둔다.
- output은 queue/coalesce 가능하게 한다.
- control message는 lossless-or-close 정책을 둔다.
- snapshot/replay ACK timer는 실제 send accept 이후에 시작한다.
- replay output queue도 UTF-8 byte 기준으로 통일한다.

split WebSocket은 나중 단계다. 단일 `/ws`에서 safeSend/backpressure를 먼저 검증한 뒤 `split-shadow`, 이후 `split`로 간다.

### Phase 4. 프론트엔드 runtime residency와 output backpressure

선행 작업:

- 현재 미통합 안정화 유틸의 타입 export/import를 정리해 `npm --prefix frontend run typecheck`를 통과시킨다.

그 다음:

- `useTerminalRuntimeResidency()`를 `App.tsx`에 실제 연결한다.
- active/visible tab은 항상 pin한다.
- hidden workspace/tab은 `maxLiveWorkspaces`, `maxLiveTerminals`, `hiddenRuntimeTtlMs`에 따라 evict한다.
- evicted tab은 `TerminalRuntimeLayer`에 전달하지 않는다.
- `terminalRefsMap`, `aliveWorkspaceIds`, `workspaceVisitOrder`를 `wm.tabs`/`wm.workspaces` 기준으로 prune한다.
- hidden output은 xterm write와 snapshot save를 계속 수행하지 않고 dirty/stale flag만 남긴다.
- visible 전환 시 screen snapshot/repair로 복원한다.
- browser `WebSocket.send()`도 `bufferedAmount`를 확인해 backpressure reason을 명시한다.

초기 권장값:

- `maxLiveWorkspaces=2`
- `maxLiveTerminals=12`
- `hiddenRuntimeTtlMs=60000`
- `wsTransportMode=unified`

### Phase 5. Shutdown과 orphan recovery 정합화

`performGracefulShutdown()`은 CWD snapshot/flush만 하지 말고 lifecycle policy를 받아야 한다.

권장 순서:

1. live tab CWD snapshot.
2. workspace flush.
3. session lifecycle에 `internal-shutdown` 또는 `daemon-stop` reason 기록.
4. owned session process tree 종료.
5. 종료 evidence count 반환.
6. 최종 flush.
7. process exit.

`checkOrphanTabs()`는 다음만 recover해야 한다.

- lifecycle metadata가 없거나 legacy active인 tab.
- `state=active`이고 `recoverable=true`인 tab.

다음은 recover하면 안 된다.

- direct session delete.
- tab delete/workspace delete.
- internal shutdown/daemon stop으로 stopped 처리된 tab.
- process-exit 후 사용자가 명시적으로 stopped 처리한 tab.

### Phase 6. 검증과 soak

우선 type/build recovery gate:

```powershell
npm --prefix server run build
npm --prefix frontend run typecheck
```

이 gate가 실패하면 process termination, WebSocket backpressure, frontend runtime residency 구현 전에 실패 항목을 분류한다. 특히 runtime config/resource limit, WebSocket protocol, recovery-option, split transport 관련 타입 drift를 별도 선행 수정으로 다룬다.

그 다음 automated gate:

```powershell
npm --prefix server test
node --experimental-strip-types --test frontend/tests/unit/runtimeConfig.test.ts frontend/tests/unit/webSocketBackpressure.test.ts frontend/tests/unit/terminalOutputScheduler.test.ts frontend/tests/unit/useTerminalRuntimeResidency.test.ts frontend/tests/unit/visibleOutputRecovery.test.ts
node --test tools/daemon/process-info.test.js tools/daemon/launcher.test.js tools/daemon/stop-client.test.js tools/daemon/sentinel.test.js
```

safe live scenario:

- shell 목록은 `/api/sessions/shells`로 확인한 뒤 지원 shell만 테스트한다.
- 테스트가 시작한 root PID만 기록한다.
- process name 기반 kill은 금지한다.
- app shutdown/CTRL+C/internal shutdown을 먼저 사용하고, fallback kill도 검증된 PID에만 적용한다.

검증 시나리오:

- PowerShell tab에서 marker PID 기록 후 tab close, PID 및 descendants 소멸 확인.
- bash/native tab에서 marker PID 기록 후 tab close, PID 및 descendants 소멸 확인.
- WSL tab에서 Linux PID 기록 후 tab close, `/proc/<pid>` 소멸 확인.
- workspace delete 후 해당 workspace의 모든 session id와 marker PID 소멸 확인.
- direct `DELETE /api/sessions/:id` 후 workspace tab이 stopped/non-recoverable로 남고 orphan recovery가 재생성하지 않는지 확인.
- restart tab에서 old session tree가 종료되고 new session만 남는지 확인.
- app/daemon shutdown 후 세션 process tree가 남지 않는지 확인.
- slow WebSocket fake client에서 server buffered/queued bytes가 cap을 넘지 않고 slow client만 close되는지 확인.
- hidden workspace output이 active terminal keydown-to-echo latency를 올리지 않는지 확인.

## 우선순위

1. 타입/설정 drift 정리와 build/typecheck 기준선 회복.
2. 관측과 telemetry 확장.
3. 세션 소유 process-tree terminator와 idempotent cleanup.
4. PTY `onExit`와 `deleteSession()` cleanup 통합.
5. headless queue bounded deque와 WebSocket `safeSend`.
6. workspace/session lifecycle metadata와 orphan recovery gate.
7. `useTerminalRuntimeResidency()` 실제 연결.
8. hidden output write 중단 및 visible repair.
9. split WebSocket shadow/preview.
10. soak 후 기본값 hardening.

이 순서는 서버에서 실제 누적되는 세션/프로세스를 먼저 막고, 그 다음 프론트가 누적된 서버 상태를 과도하게 구독/렌더링하지 않도록 낮추는 방향이다.

## 남은 질문

- WSL session에서 Linux-side root PID를 어떻게 안전하게 기록할지는 bootstrap 스크립트 설계가 필요하다.
- 현재 라이브 PID `92300` 아래 남은 프로세스들을 실제로 언제/어떻게 정리할지는 구현/검증과 별도 운영 결정이 필요하다.
- BGSTAB SRS의 non-2002 validation 지침과 현재 AGENTS.md의 `https://localhost:2002` validation rule이 충돌한다. live validation 전에 기준을 정해야 한다.

## 최종 판단

사용자 의심은 정적 분석과 현재 실행 증거 기준으로 강하게 지지된다. 세션 닫기와 워크스페이스 삭제는 ProjectMaster 내부 registry cleanup까지는 수행하지만, PowerShell/bash/WSL 및 그 하위 프로세스가 실제로 종료됐는지 보장하지 않는다. 또한 장시간 사용할수록 느려지는 현상은 서버의 세션/프로세스/출력 큐 누적을 우선 가설로 두고 조사/수정하는 것이 가장 정합하며, 프론트엔드의 hidden runtime 유지와 무제한 구독/렌더링은 이를 증폭하는 구조다.
