# Mosaic Drag Source Shrink Research

Date: 2026-04-22
Scope: grid drag start behavior in `react-mosaic-component`
Status: research only

## Question

When the user starts dragging a window, the source pane immediately shrinks. This happens regardless of mode. Why does it happen, and is it necessary?

## Conclusion

It happens because the vendor drag-start path still calls `hide()` on the source node, which mutates the live mosaic tree before drop completes.

This is not technically required.

It is an inherited `react-mosaic` drag behavior that was left in place while Step26 changed only the drop semantics.

## Root Cause

### 1. Drag start still hides the source node

In the patched vendor file, drag start still runs:

- [MosaicWindow.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\MosaicWindow.js:275)

That line creates:

- `defer(() => mosaicActions.hide(props.path))`

So the source tile is not only shown as a drag preview. The actual live layout tree is changed during drag start.

### 2. `hide()` collapses the parent split percentage

The hide implementation still comes from:

- [mosaicUpdates.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\util\mosaicUpdates.js:150)

`createHideUpdate()` sets the parent `splitPercentage` to `0` or `100`.

That is why the source pane appears to collapse or become extremely thin.

### 3. Step26 changed drop semantics, not drag-start semantics

Step26 added:

- reorder payloads
- full-cell targets
- root snapshot restore
- equal-only reorder gating

But the drag-start hide behavior was preserved.

So today the flow is:

1. drag starts
2. source pane is hidden by tree mutation
3. preview/targets are shown
4. drop or restore path runs later

That is why the pane shrinks before the user even completes a move.

## Is It Necessary?

No.

The old library comment only explains why the source element must still exist at drag start for HTML DnD to initialize correctly. It does not prove that the source must be hidden immediately afterward.

The current shrink is therefore a legacy implementation choice, not a hard requirement.

## Fix Options

## Option A: Equal-mode-only fix

Change the vendor path so `reorderEnabled === true` does not call `mosaicActions.hide(props.path)` at drag start.

### Pros

- smallest patch
- directly fixes the new equal-mode reorder UX
- low blast radius

### Cons

- does not fix shrink in non-equal modes
- user explicitly reported the problem across all modes

## Option B: Remove drag-start hide for all window drag paths

Do not mutate the mosaic tree on drag start at all.

Instead:

1. keep the source tile in place
2. keep using drag preview / blue targets
3. apply split or reorder tree changes only on successful drop
4. make invalid drop/reset a true no-op or root-snapshot restore

### Pros

- fixes the user-reported problem in all modes
- aligns better with modern DnD UX
- removes the main reason panes visually jump before commit

### Cons

- broader vendor change than Option A
- split-mode invalid-drop path must also be updated, because the old reset logic assumes the tree was hidden first

## Option C: Keep layout geometry, but hide only visual content

Instead of tree mutation, mark the drag source visually:

- lower opacity
- outline
- ghost overlay
- `visibility: hidden` on content only

### Pros

- preserves spatial stability
- still gives the user a clear “this is the dragged pane” cue

### Cons

- more UI state plumbing
- still requires replacing the current hide/reset logic
- more moving parts than Option B

## Recommended Direction

Option B is the correct target.

Reason:

- the user complaint applies to all modes
- the shrink is caused by tree mutation, not by reorder logic
- removing drag-start tree mutation is the cleanest structural fix

Recommended implementation shape:

1. In vendor `ConnectedInternalMosaicWindow`, stop calling `mosaicActions.hide(props.path)` on drag start.
2. Keep `rootSnapshot` capture.
3. On successful drop:
   - split modes use `createDragToUpdates(...)`
   - reorder mode uses `createReorderedTreePreservingShape(...)`
4. On invalid drop:
   - restore `rootSnapshot` for both branches, or simply no-op if no tree mutation occurred
5. Add a non-destructive drag-source visual state, if desired:
   - opacity
   - outline
   - subtle ghost class

## Risk Notes

### 1. Split-mode reset path must be reworked

The current split reset path still assumes drag start changed the tree:

- [MosaicWindow.js](C:\Work\git\_Snoworca\ProjectMaster\frontend\node_modules\react-mosaic-component\lib\MosaicWindow.js:341)

That branch clears `splitPercentage` on the parent.

If drag-start hide is removed, this reset path should be unified with the explicit restore strategy instead of keeping the old split-only behavior.

### 2. Drag preview may now show both source and preview

That is usually acceptable, but if it looks too noisy, apply a source visual state instead of shrinking the layout.

### 3. Tests should verify geometry stability at drag start

Current tests validate drop results well.

For this change, an extra regression should be added:

1. measure source tile rect
2. start drag from handle
3. assert source width/height stay effectively unchanged before drop

## Practical Judgment

The current behavior is explainable, but not defensible as the desired UX.

The source shrink is not required by the browser, not required by equal-mode reorder, and not required by the user requirement.

It is best treated as a carry-over from upstream drag implementation and removed deliberately.
