# Terminal Screen Repair V2 권장 구현 연구

작성일: 2026-05-08  
대상: BuilderGate 웹 터미널에서 Codex 실행 중 workspace 전환 또는 창 크기 변경 후 전체 scrollback이 느리게 재출력되는 현상  
결론: 품질 저하 없이 해결하려면 `full snapshot restore`와 `screen repair`를 프로토콜과 적용 경로에서 분리해야 한다.

## 1. 문제 정의

현재 grid/workspace repair는 `repair-replay`를 보내고, 서버는 `screen-snapshot`을 반환하며, 프런트는 `replaceWithSnapshot()`에서 `term.reset()` 후 snapshot 전체를 다시 쓴다.

관련 경로:

- `frontend/src/components/Terminal/TerminalContainer.tsx`
  - workspace 복귀 후 `runGridLayoutRepair('workspace')`
  - `repair-replay` 전송
  - `screen-snapshot` 수신 후 `replaceWithSnapshot()`
- `frontend/src/components/Terminal/TerminalView.tsx`
  - `replaceWithSnapshot()`
  - `applySnapshotReplacement()`
  - `term.reset()`
- `server/src/ws/WsRouter.ts`
  - `handleRepairReplay()`
  - `sendSnapshotReplay()`
- `server/src/services/SessionManager.ts`
  - `getScreenSnapshot()`
- `server/src/utils/headlessTerminal.ts`
  - `serializeHeadlessTerminal()`

`serializeHeadlessTerminal()`은 `SerializeAddon.serialize()`를 옵션 없이 호출한다. xterm serialize addon의 기본 동작은 사용 가능한 normal buffer scrollback을 포함한다. 따라서 repair 목적의 snapshot도 scrollback 전체를 포함할 수 있고, 웹 xterm은 그 전체 문자열을 다시 파싱한다.

네이티브 터미널에서도 Codex TUI는 resize/SIGWINCH에 전체 repaint를 수행할 수 있다. 다만 웹에서는 BuilderGate의 snapshot replay, WebSocket, xterm write queue, DOM/canvas 렌더링이 추가되어 체감 속도가 크게 느려진다.

## 2. 제외해야 할 품질 저하 방식

다음 방식은 문제를 줄일 수는 있지만 기능 또는 품질을 떨어뜨리므로 권장하지 않는다.

| 방식 | 제외 이유 |
| --- | --- |
| `pty.scrollbackLines` 축소 | 복원 가능한 히스토리 양이 줄어든다. |
| workspace/grid 자동 repair 비활성화 | 기존 TUI 깨짐 복구 품질이 떨어진다. |
| Codex `--no-alt-screen` 또는 특정 CLI 옵션 의존 | Codex/xterm 조합에서 scrollback 문제가 별도 보고되어 있고 환경 의존적이다. |
| repair에서 `serialize({ scrollback: 0 })`만 사용하고 `term.reset()` 유지 | 현재 scrollback을 날리거나 화면 상태를 불완전하게 복원할 수 있다. |
| synthetic resize를 기본 복구 수단으로 사용 | SIGWINCH를 추가 발생시켜 Codex 같은 TUI가 다시 전체 repaint할 수 있다. |
| 오류를 숨기고 repair skip | 화면 깨짐이 남아도 원인을 추적하기 어렵다. |

## 3. 권장 아키텍처

권장안은 `full snapshot restore`와 `screen repair`를 별도 계약으로 분리하는 것이다.

### 3.1 Full Snapshot Restore

목적:

- WebSocket 재연결
- 브라우저 새로고침 후 복원
- 새 subscribe
- 서버 headless 상태와 클라이언트 xterm 상태가 크게 어긋난 경우의 최후 복구

특징:

- 기존 `screen-snapshot` 프로토콜 유지
- 기존 full scrollback snapshot 유지
- `replaceWithSnapshot()` 경로 유지
- scrollback 보존 기능과 reconnect 복원 품질 유지

### 3.2 Screen Repair

목적:

- workspace 복귀 후 grid TUI 화면 교정
- 사용자가 수동 repair를 요청한 경우
- resize 이후 visible viewport 재동기화
- Codex/Hermes/Claude 같은 interactive TUI의 줄바꿈, status line, cursor 위치 보정

특징:

- 새 프로토콜 `screen-repair` 사용
- scrollback 미포함
- `term.reset()` 금지
- 전체 snapshot write 금지
- 현재 보이는 화면 영역만 row 단위로 덮어쓰기
- 기존 scrollback과 selection, reconnect restore 기능은 그대로 유지

## 4. Screen Repair V2 프로토콜

### 4.1 Client Request

```ts
type ScreenRepairRequest = {
  type: 'screen-repair';
  sessionId: string;
  reason: 'workspace' | 'manual' | 'resize';
  cols: number;
  rows: number;
  clientAtBottom: boolean;
};
```

요청 조건:

- terminal이 visible이어야 한다.
- IME composition 중이면 defer한다.
- 사용자 입력 capture가 열려 있으면 defer한다.
- 사용자가 scrollback을 올려 보고 있으면 defer한다.
- 마지막으로 서버에 전송한 `cols/rows`와 현재 `cols/rows`가 다르면 먼저 resize를 처리하고 quiet window 이후 요청한다.

### 4.2 Server Response

```ts
type ScreenRepairMessage = {
  type: 'screen-repair';
  sessionId: string;
  repairToken: string;
  seq: number;
  cols: number;
  rows: number;
  bufferType: 'normal' | 'alternate';
  cursorX: number;
  cursorY: number;
  rowsData: Array<{
    y: number;
    ansi: string;
  }>;
  source: 'headless';
};
```

서버 생성 조건:

- `headlessWriteChain`이 drain된 뒤 생성한다.
- `pendingHeadlessWrites > 0`이면 짧은 quiet window 후 재시도한다.
- `cols/rows`가 클라이언트 요청과 다르면 repair 대신 `screen-repair:rejected`를 보내고 프런트가 재측정한다.
- headless가 degraded이면 full snapshot으로 자동 전환하지 말고 명시적으로 `screen-repair:rejected`를 보낸다. 자동 전환은 전체 재출력 문제를 되살릴 수 있으므로 사용자가 체감 가능한 repair 경로에서는 관측 가능해야 한다.

### 4.3 Client Ack

```ts
type ScreenRepairReady = {
  type: 'screen-repair:ready';
  sessionId: string;
  repairToken: string;
};
```

repair pending 중 live output은 기존 replay queue와 같은 원칙으로 보류하고, ack 후 정확히 한 번 flush한다.

## 5. 서버 구현 방식

### 5.1 새 API 추가

`SessionManager`에 `getScreenRepair(sessionId, cols, rows)`를 추가한다.

반환값은 full snapshot 문자열이 아니라 active viewport의 row patch 모델이어야 한다.

필요 정보:

- `screenSeq`
- `cols`, `rows`
- active buffer type
- cursor position
- visible rows
- row별 ANSI styled text

### 5.2 active buffer 기반 row 추출

`@xterm/headless`는 `allowProposedApi: true`로 생성되어 있다. 따라서 headless terminal의 active buffer에서 row/cell 정보를 읽을 수 있다.

권장 방식:

1. `terminal.buffer.active`를 읽는다.
2. visible viewport 범위를 계산한다.
3. 각 row를 cell 단위로 순회한다.
4. cell char, width, foreground/background/style attribute를 ANSI SGR로 변환한다.
5. row 끝의 의미 없는 blank는 유지 정책을 명확히 한다.
   - 화면 교정 정확성을 우선하면 `cols`까지 채운다.
   - 전송량을 줄이려면 `CSI 2K` 후 trailing blank를 생략한다.

주의:

- `SerializeAddon.serialize({ scrollback: 0 })`만으로는 충분하지 않다.
- 그 문자열을 그대로 현재 xterm에 write하면 cursor 이동, line feed, alt buffer 전환이 섞일 수 있다.
- repair payload는 "복원용 VT stream"이 아니라 "화면 row patch 모델"이어야 한다.

### 5.3 output queue 경계

repair 중 output이 오면 다음 순서를 보장한다.

```text
repair requested
  -> server creates repair payload after headless drain
  -> live output is queued for that client/session
  -> client applies repair rows
  -> client sends screen-repair:ready
  -> server flushes queued live output once
  -> session:ready
```

이 경계가 없으면 repair payload와 live output이 서로 덮어쓰거나 중복될 수 있다.

## 6. 프런트 구현 방식

### 6.1 새 handle 추가

`TerminalHandle`에 다음 메서드를 추가한다.

```ts
applyScreenRepair: (repair: ScreenRepairPayload) => Promise<boolean>;
```

기존 `replaceWithSnapshot()`은 full restore 전용으로 남긴다.

### 6.2 applyScreenRepair 규칙

금지:

- `term.reset()`
- `term.clear()`
- full snapshot write
- `scrollToBottom()` 강제 호출
- `CSI 2J` 또는 `CSI 3J` 기반 전체 화면 clear

허용:

- cursor save/restore
- cursor home/absolute move
- row별 `CSI 2K`
- row별 styled text write
- 최종 cursor 위치 복원
- repair 완료 후 `fit()`은 하되 `cols/rows`가 바뀌지 않으면 resize 전송 생략

권장 VT patch 형태:

```text
ESC 7
ESC [?25l
ESC [1;1H ESC [2K <row 0 ansi>
ESC [2;1H ESC [2K <row 1 ansi>
...
ESC [cursorY+1;cursorX+1H
ESC [?25h
ESC 8
```

단, cursor visibility는 실제 headless state가 있으면 그 값을 따라야 한다. `ESC 7/ESC 8`과 최종 cursor 위치 복원을 함께 쓸 때 충돌하지 않도록 테스트에서 고정한다.

중요한 점은 line feed를 사용하지 않는 것이다. absolute cursor movement와 erase-in-line만 사용하면 scrollback이 증가하지 않는다.

### 6.3 사용자 스크롤 보호

repair 직전 `term.buffer.active.baseY === term.buffer.active.viewportY` 또는 이에 준하는 "at bottom" 상태를 확인한다.

- at bottom이면 repair 적용
- 사용자가 scrollback을 보고 있으면 repair defer
- 일정 시간이 지나도 계속 at bottom이 아니면 repair skip이 아니라 `repair_deferred_user_scrollback` debug event를 남긴다

이 정책은 사용자의 scrollback 탐색을 방해하지 않으면서 기능을 보존한다.

### 6.4 buffer type mismatch

서버 repair payload의 `bufferType`과 클라이언트 xterm active buffer type이 다르면 row patch를 적용하지 않는다.

권장 동작:

1. `screen_repair_rejected_buffer_mismatch` debug event 기록
2. 즉시 full snapshot으로 자동 전환하지 않음
3. 다음 subscribe/reconnect 또는 명시적 full restore 경로에서만 full snapshot 사용

자동 full fallback은 현재의 느린 전체 재출력 문제를 다시 만들 수 있으므로 repair 경로에서는 관측 가능한 실패로 남기는 것이 낫다.

## 7. resize dedupe 보강

서버에는 이미 no-op resize skip이 있다. 그래도 프런트에서 같은 `cols/rows` 전송을 줄여야 한다.

권장:

- `TerminalView.emitResize()` 또는 `TerminalContainer.handleResize()`에서 마지막 전송 `cols/rows`를 기억한다.
- 같은 값이면 WebSocket `resize`를 보내지 않고 `resize_deduped_client` debug event만 남긴다.
- `reason`이 `manual`, `workspace`, `visible`, `resize-observer`여도 값이 같으면 생략한다.

이 변경은 기능을 줄이지 않고 SIGWINCH와 replay refresh 가능성을 줄인다.

## 8. 구현 단계

### Phase 1: 관측성 및 회귀 기준 고정

- debug capture에 다음 이벤트 추가
  - `screen_repair_requested`
  - `screen_repair_sent`
  - `screen_repair_applied`
  - `screen_repair_deferred_user_scrollback`
  - `screen_repair_rejected_geometry_mismatch`
  - `screen_repair_rejected_buffer_mismatch`
  - `resize_deduped_client`
- 기존 `screen_snapshot_authoritative_applied`가 workspace repair에서 발생하지 않는 것을 새 성공 기준으로 둔다.

### Phase 2: 서버 `ScreenRepairPayload` 생성

- `SessionManager.getScreenRepair()` 추가
- active buffer row extraction helper 추가
- headless pending write drain 보장
- degraded/headless missing/geometry mismatch는 명시적 reject

### Phase 3: WS protocol 분리

- `ClientWsMessage`에 `screen-repair`, `screen-repair:ready` 추가
- `ServerWsMessage`에 `screen-repair`, `screen-repair:rejected` 추가
- repair pending queue를 기존 replay pending과 같은 수준으로 관리
- ack timeout과 stale token 처리 추가

### Phase 4: 프런트 row patch 적용

- `TerminalView.applyScreenRepair()` 추가
- `TerminalContainer`의 workspace/manual repair 요청을 `repair-replay`에서 `screen-repair`로 전환
- full restore 경로는 기존 `screen-snapshot`으로 유지
- user scrollback, IME, input gate와 충돌하지 않게 defer

### Phase 5: 기존 repair-replay 정리

- `repair-replay`는 호환 기간 동안 유지하되 workspace/manual repair 기본 경로에서는 사용하지 않는다.
- 호환 기간 이후 제거하거나 `full-repair-replay`처럼 이름을 바꿔 의미를 명확히 한다.

## 9. 테스트 계획

### 9.1 서버 단위 테스트

- `getScreenRepair()`는 scrollback 전체를 포함하지 않는다.
- 500줄 scrollback + 24 rows viewport에서 payload rows는 24개만 생성된다.
- repair 생성 중 `pendingHeadlessWrites > 0`이면 drain 이후 생성된다.
- geometry mismatch는 reject된다.
- degraded headless는 repair reject를 반환한다.
- repair pending 중 output은 queue되고 ack 후 한 번만 flush된다.

### 9.2 프런트 단위 테스트

- `applyScreenRepair()`는 `term.reset()`을 호출하지 않는다.
- repair 적용 전후 scrollback length가 증가하지 않는다.
- 사용자가 scrollback을 보고 있으면 repair가 defer된다.
- same `cols/rows` resize는 client에서 dedupe된다.
- IME composition 중 repair는 적용되지 않는다.

### 9.3 Playwright E2E

필수 시나리오:

1. 긴 출력 300줄 생성
2. grid workspace 전환 왕복
3. debug capture 확인
   - `screen_repair_applied` 발생
   - `screen_snapshot_authoritative_applied`는 workspace repair 때문에 발생하지 않음
   - `replaceWithSnapshot()` 경로 미사용
4. 화면 marker 중복 없음
5. scrollback 탐색 가능
6. Codex 실행 중 workspace 왕복 후 전체 히스토리 재출력 없음

### 9.4 수동 검증

대상 URL:

- `https://localhost:2002`

검증 항목:

- grid mode에서 Codex 실행
- 긴 대화 또는 긴 출력 이후 workspace 전환
- 창 크기 변경
- 수동 repair
- 입력 중 repair defer
- scrollback을 올려 본 상태에서 workspace 복귀

성공 기준:

- 화면 깨짐은 repair된다.
- 전체 scrollback이 느리게 다시 흐르지 않는다.
- 기존 scrollback은 유지된다.
- 입력 손실이 없다.
- debug capture로 repair와 full restore가 구분된다.

## 10. 구현 시 주의사항

1. repair payload는 full restore 대체물이 아니다.
2. row patch는 line feed 없이 absolute cursor movement로만 적용한다.
3. `CSI 2J`, `CSI 3J`는 사용하지 않는다.
4. headless와 client geometry가 다르면 repair하지 않는다.
5. repair 실패를 숨기지 않는다.
6. full snapshot fallback은 subscribe/reconnect 같은 restore 경로에만 남긴다.
7. Codex 전용 처리로 만들지 않는다. 모든 TUI에 적용 가능한 terminal contract로 구현한다.

## 12. 최종 권고

품질 저하 없이 문제를 해결하려면 `screen-repair` 전용 row patch 프로토콜을 추가해야 한다.

기존 `screen-snapshot`은 full restore 전용으로 유지하고, workspace/grid/manual repair에서는 더 이상 전체 snapshot을 적용하지 않는다. 이 방식만이 scrollback 보존, reconnect 복원, grid TUI 자동 복구를 모두 유지하면서 웹에서 전체 히스토리가 느리게 재출력되는 병목을 제거한다.
