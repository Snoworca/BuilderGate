# BuilderGate ConPTY 출력 복구 및 입력 지연 연구

날짜: 2026-06-30

범위: ConPTY 모드는 화면 출력이 미려하고 정확해 보이지만, Codex 같은 TUI 세션은 복구가 느리고 에이전트가 화면을 반복 갱신하는 동안 입력이 어려워질 수 있다. 이 문서는 읽기 전용 서브에이전트 10개의 조사와 로컬 정적 분석을 취합한 결과다. 이 연구 과정에서 코드는 변경하지 않았다.

## 요약

ConPTY 자체만이 유일한 문제는 아니다. 가장 설득력 있는 설명은 여러 계층이 겹친 병목이다.

1. ConPTY와 AI TUI는 고빈도 화면 재그리기 출력을 만든다. Codex 스타일 화면은 커서 이동, 줄 지우기 시퀀스, title/progress 업데이트, local echo, ticker/status redraw, 때때로 큰 resume 출력 폭주를 내보낸다.
2. 프로젝트에는 이미 많은 보호 장치가 있다. bounded headless output queue, 서버 safe-send/backpressure 경로, viewport-only snapshot, hidden output skip/recovery, 프론트엔드 visible output scheduler, 브라우저 input backpressure, AI TUI idle heuristic이 존재한다.
3. 일부 보호 장치는 기본 경로에서 활성화되어 있지 않다. 로컬 설정은 ConPTY를 사용하지만, schema/env 기본값은 명시 설정이 없으면 여전히 `stabilityModes.wsSendMode = "direct"`, `session.processCleanup.mode = "observe"`, input reliability observe-style 동작을 유지한다. `direct` 모드에서는 서버 WebSocket output coalescing과 backpressure queueing이 느린 client를 실질적으로 보호하지 못한다.
4. 현재 정상 복구 경로는 full-log replay가 아니다. 서버와 로컬 snapshot은 viewport-only(`scrollback: 0`)다. 느린 부분은 큰 snapshot apply, ACK-pending tail flush, headless drain 지연, Codex resume 출력, 또는 output/control이 하나의 transport 경로를 공유하는 구조일 가능성이 더 높다.
5. 입력은 frontend/server transport gate를 통과하면 `SessionManager.writeInput()`과 `pty.write()`까지 직접 도달한다. 사용자가 느끼는 "입력이 안 됨"은 그 이전 단계에서 발생한다. 즉 브라우저 main thread 렌더링, restore/recovery barrier, unified WebSocket backpressure, replay ACK 대기, 서버 output/control FIFO 동작이 원인이다.

효과적인 해결책은 ConPTY를 기본으로 끄는 것이 아니다. 대화형 품질을 위해 ConPTY는 유지하되, output delivery를 bounded, chunked, measurable, configurable하게 만들어야 한다. input/control은 output보다 우선되어야 한다. recovery는 live output과 같은 write scheduler 원칙으로 snapshot과 tail을 적용해야 한다.

## 현재 관련 구현

### PTY 및 backend 선택

- `server/src/services/SessionManager.ts`는 `node-pty.spawn(...)`으로 PTY session을 만들고 Windows에서 `useConpty`를 전달한다.
- `server/src/utils/ptyPlatformPolicy.ts`는 Windows 기본값을 ConPTY로 두고, non-Windows에서는 ConPTY 전용 설정을 정규화해 제거한다.
- `server/config.json5`는 현재 `useConpty: true`, `windowsPowerShellBackend: "conpty"`를 사용한다.
- PowerShell backend는 `inherit`, `conpty`, `winpty`가 가능하지만, winpty는 서버가 host 지원 여부를 probe하므로 항상 가능한 fallback은 아니다.

결론: backend 선택은 의도적인 계약이다. 명령 이름만 보고 Codex 세션을 pipe/winpty로 조용히 전환하면 안 된다.

### Output path

현재 live output path:

```text
ConPTY / node-pty output
  -> SessionManager pty.onData
  -> headlessOutputQueue
  -> headless xterm write commit
  -> WsRouter.routeSessionOutput
  -> server WebSocket transport send
  -> frontend WebSocketContext
  -> TerminalContainer
  -> TerminalView
  -> terminalOutputScheduler
  -> xterm.write
```

이미 존재하는 중요한 보호 장치:

- `SessionManager`는 bounded headless output queue를 사용하며, 조용히 무한 증가시키지 않고 degrade한다.
- `WsRouter`에는 safe-send/backpressure 필드가 있다. buffered amount, queued output/control bytes, slow-client close counter, send-error counter, output coalescing count가 있다.
- `TerminalView` live output은 `createTerminalOutputScheduler()`를 거치며, `xterm.write` callback을 기다리고 budget 단위로 flush한다.
- `TerminalContainer`는 hidden output을 skip하고 hidden 상태에서 계속 output을 write하는 대신 snapshot/repair로 복구할 수 있다.

중요한 gap:

- Schema 기본 `wsSendMode`는 `direct`다. 이 모드에서 `WsRouter`는 즉시 send하고 output coalescing을 수행하는 queue 경로를 사용하지 않는다.
- 서버 transport state에는 output/control byte counter가 있지만 실제 queue는 단일 FIFO다. control message가 이미 queued된 output 뒤에 남을 수 있다.
- Snapshot restore와 screen repair는 일반 visible output scheduler를 우회하고 큰 payload를 `xterm.write`에 직접 밀어 넣을 수 있다.

### Recovery path

정상 복구는 full history replay가 아니다.

- 서버 headless snapshot serialization은 viewport-only semantics를 사용한다.
- client local snapshot도 viewport-only semantics를 사용한다.
- subscribe/reconnect 시 서버는 `screen-snapshot`을 보내고, `screen-snapshot:ready`를 기다린 다음 ACK-pending tail output과 queued input을 flush한다.

주요 recovery 병목:

- authoritative snapshot이 viewport-only여도 payload가 크면 browser를 block할 수 있다.
- ACK-pending tail output은 replay queue limit까지 커질 수 있고, 큰 chunk로 flush될 수 있다.
- Codex `resume`은 자체 high-output replay를 생성할 수 있으며, 이는 terminal replay와 다르다.
- server WS mode가 `direct`이면 output flood가 ACK/ready/control traffic을 지연시킬 수 있다.

### Input path

현재 input path:

```text
xterm input
  -> TerminalView.submitCapturedInput
  -> TerminalContainer.handleInput
  -> TerminalInputSequencer
  -> WebSocketContext.send({ type: "input" })
  -> WsRouter.handleInput
  -> SessionManager.writeInput
  -> pty.write
```

입력이 `SessionManager.writeInput`에 도달하면 headless output queue 뒤에 놓이지 않는다. 지연은 그 전에 발생한다.

- 브라우저 main thread가 output 또는 snapshot/repair 렌더링으로 바쁘다.
- `restorePending`, visible-output recovery, session-ready gate가 활성 상태다.
- browser `bufferedAmount`가 input backpressure를 만든다.
- unified `/ws` 때문에 output/control/input이 하나의 transport를 공유한다.
- replay pending mode는 ACK/timeout 전까지 input을 queue하거나 reject한다.
- server safe-send enforce mode도 control/output queue가 물리적으로 분리되지 않으면 단일 FIFO를 drain한다.

## Root Cause Assessment

### 사실일 가능성이 높은 것

- ConPTY는 legacy/simple output보다 fidelity가 높은 stream을 만든다. 여기에는 많은 redraw noise가 포함된다.
- Codex 같은 TUI output은 line-oriented log output이 아니다. 화면 mutation stream이다.
- Browser xterm rendering은 여전히 main thread에서 실행된다. 서버 queueing만으로 oversized `xterm.write` 작업을 완전히 숨길 수 없다.
- 현재 기본값과 reserved setting 때문에 구현된 guardrail이 실제 사용자 경로에 적용되지 않을 수 있다.

### 주요 원인이 아닐 가능성이 높은 것

- 단순 duplicate input bug가 아니다. 기존 xterm input path와 debug event는 input이 대체로 하나의 path를 탄다는 점을 보여준다.
- 정상 full scrollback replay가 아니다. 현재 정상 snapshot은 viewport-only다.
- browser worker만으로 해결되지 않는다. worker는 WebSocket/JSON parsing/batching을 맡을 수 있지만 실제 xterm rendering은 여전히 main thread에서 일어난다.

## 권장 해결 전략

### 1. 서버 output 보호를 활성화하고 priority-aware하게 만든다

권장 변경:

- ConPTY high-output validation에서는 `safe-send-enforce`를 사용하고, soak evidence 이후 기본값 후보로 검토한다.
- 서버 transport queue를 물리적으로 `controlQueue`와 `outputQueue`로 분리한다.
- lifecycle, ACK, input rejection, session ready, ping/pong, workspace/tab metadata, status message는 control을 먼저 flush한다.
- terminal payload lane 내부에서는 payload ordering을 보존한다. replay correctness를 깨는 방식으로 snapshot/output/repair를 reorder하지 않는다.
- high/hard water 판단은 현재 `bufferedAmount`만 보지 말고 projected size인 `bufferedAmount + message.byteLength` 기준으로 한다.
- 긴 queue drain에서 `Array.shift()` 대신 head-index/deque 방식을 사용한다.
- replay pending tail을 string slicing이 아니라 byte-aware seq ring metadata로 바꾼다.
- screen repair overflow 시 큰 payload를 이어 붙여 보내지 말고 abort/refresh로 전환한다.

### 2. Snapshot과 repair를 recovery write gate로 보낸다

Live output은 이미 scheduler를 사용한다. Snapshot과 repair도 같은 원칙을 사용해야 하지만 반드시 같은 queue instance를 공유할 필요는 없다. 더 안전한 설계는 독립적인 recovery write gate 또는 recovery lane이다. 그래야 live output scheduler의 stale/overflow 상태가 recovery ordering을 오염시키지 않는다.

- authoritative snapshot apply는 recovery lane을 통해 chunking 또는 scheduling한다.
- ACK-pending tail flush는 snapshot-before-tail ordering을 보존하면서 chunking한다.
- screen repair `ansiPatch`는 repair/write queue를 통해 직렬화한다.
- `screenRepairInFlightRef`는 session id뿐 아니라 request token/nonce를 포함해야 한다.
- visible output recovery가 input gate를 필요 이상 오래 붙잡지 않게 한다.

목표는 recovery 화면을 stale하게 만드는 것이 아니다. 하나의 큰 `term.write`가 main thread를 독점하지 못하게 하는 것이다.

### 3. Input/control을 output보다 앞세운다

권장 frontend/server 동작:

- input, resize, subscribe, unsubscribe, ACK, ready, ping/pong, close/reject는 control이다.
- output과 큰 snapshot/repair payload는 terminal payload다.
- resize는 latest-value-wins이며 retry 가능해야 한다.
- replay ACK는 recovery ordering을 보존해야 한다. snapshot과 해당 ACK-pending terminal tail이 완료된 뒤에만 `session:ready`와 queued replay input을 release한다.
- 그 다음 queued replay input은 unrelated live output backlog에 굶주리지 않고 `SessionManager.writeInput()`까지 도달해야 한다.
- frontend는 control send를 하나의 observable path로 중앙화해야 한다.

Control priority를 active snapshot/repair replay를 reorder해도 된다는 의미로 해석하면 안 된다. Recovery transaction 안에서는 terminal payload ordering이 우선이다. Control priority는 해당 transaction 밖의 unrelated live output backlog와 lifecycle/control message에 적용한다.

장기 옵션:

- Web Worker가 WebSocket, JSON parsing, output batching, high-priority control postMessage를 소유한다.
- Split socket은 control/output channel을 분리한다. repo에는 이미 split URL/lifecycle utility가 있지만 active path는 여전히 unified `/ws`다.

### 4. Output coalescing interval을 Settings에 노출한다

사용자 요구사항: coalescing/throttling 단위를 Settings에서 조정 가능해야 한다.

현재 상태:

- `resourceLimits.ws.outputCoalesceWindowMs`는 schema에 이미 존재하며 범위는 `1..1000ms`, 기본값은 `16`이다.
- `ConfigFileRepository`가 저장한다.
- `SettingsService` schema가 수용한다.
- `WsRouter` runtime config에 적용된다.
- 현재 `RuntimeConfigStore` capability에서 reserved/unavailable이다.
- `frontend/src/components/Settings/settingsDraftHelpers.ts` 목록에 없다.
- output이 safe-send queue path에 들어갈 때, 주로 `safe-send-enforce`에서 실질적 효과가 있다.

권장 Settings 변경:

- reserved Settings set에서 `resourceLimits.ws.outputCoalesceWindowMs`를 제거한다.
- "Server Backpressure" resource group에 `Output coalesce window (ms)`로 추가한다.
- 제약은 `1..1000ms`를 유지한다.
- apply scope는 immediate로 표시한다.
- 이 값이 safe-send mode의 queued server output에 적용된다는 점을 문서화한다.
- 가능하면 `stabilityModes.wsSendMode` rollout field와 함께 노출한다. 그렇지 않으면 사용자가 `direct` mode에서 값을 바꿔도 효과가 없는 상황이 생긴다.

추천 preset:

| Preset | Value | Use case |
|---|---:|---|
| Low latency | `16ms` | 대화형 입력과 일반 terminal 사용 |
| Balanced | `33ms` | noisy TUI redraw와 적당한 batching |
| High compression | `50ms` | freshness보다 redraw flood 완화가 중요한 심한 상황 |
| Custom | `1..1000ms` | debugging 및 host별 tuning |

### 5. ConPTY는 기본으로 유지하고 fallback은 명시적으로만 사용한다

Codex를 조용히 다른 backend로 전환하면 안 된다. 권장 정책:

- Windows interactive terminal은 ConPTY를 기본으로 유지한다.
- PowerShell backend override는 user-visible이어야 하며 new session부터 적용한다.
- winpty fallback은 지원되고 사용자가 명시 선택한 경우에만 허용한다.
- Pipe mode를 추가한다면 implicit TUI backend가 아니라 별도 non-interactive command runner mode여야 한다.
- Headless는 backend replacement가 아니다. mirror/recovery layer이며 실패 시 관측 가능한 방식으로 degrade해야 한다.

## Test And Soak Plan

### Server unit/integration

- `WsRouter` safe-send enforce는 output보다 control을 먼저 flush해야 한다.
- `WsRouter`는 `direct`, `safe-send-observe`, `safe-send-enforce`를 각각 분리 테스트해야 한다.
- hard limit을 초과하는 단일 output payload가 보호 장치를 우회하면 안 된다.
- Output coalescing은 같은 session의 인접 compatible output만 병합해야 한다.
- Replay pending tail truncation은 명시적이고 관측 가능해야 한다.
- Screen repair overflow는 giant direct flush가 아니라 refresh/recovery를 유발해야 한다.
- reserved 제거 후 Settings PATCH가 `resourceLimits.ws.outputCoalesceWindowMs`를 허용하고 `0`/`1001`을 거부해야 한다.
- Runtime config update가 `WsRouter` coalescing behavior를 즉시 바꿔야 한다.

### Frontend unit

- Snapshot apply와 repair는 write gate 또는 dedicated serialized repair queue를 거쳐야 한다.
- Hidden output은 hidden 상태에서 xterm write를 호출하지 않아야 한다.
- Visible output scheduler는 queue와 chunk budget을 지켜야 한다.
- Browser input backpressure는 실제 send 성공이 아니면 `ws_input_sent`를 남기면 안 된다.
- Settings page는 output coalescing interval을 렌더링하고 저장해야 한다.

### E2E

프로젝트 규칙에 따라 `https://localhost:2222`를 대상으로 실행한다.

시나리오:

- high-output visible ConPTY session 중 active terminal에 입력
- hidden high-output tab이 있는 상태에서 다른 visible tab에 입력
- Codex-like resume output 이후 reload/resubscribe
- noisy TUI redraw 중 workspace/tab switch
- output pending 중 resize flood
- coalescing interval과 safe-send mode Settings 변경

권장 pass 기준:

- input echo p95가 host별로 정한 기준 이하
- recovery-to-input-ready p95가 기준 이하
- 정상 경로에서 `input_write_failed` 없음
- 설명 불가능한 `server_input_rejected` 없음
- coalescing 이후 sentinel output loss, reordering, duplication 없음
- soak 이후 output 및 WS queued bytes가 baseline으로 복귀
- close/restart/shutdown test 이후 verified descendant가 0

### Soak

`start.bat --port 2222`와 `stop.bat`를 사용한다. 모든 `node.exe`를 종료하지 않는다.

시나리오:

- PowerShell ConPTY create/delete 30~100회 반복
- 같은 tab restart loop
- high-output server soak
- hidden runtime soak
- ASCII, Hangul/IME, Backspace, Enter input latency soak
- shutdown evidence soak

Strict soak는 verified descendant가 남는 degraded/skipped cleanup을 실패로 처리해야 한다. WSL 또는 detached descendant가 안전하게 identity 검증되지 않아 skipped되는 경우, cleanup 완료라고 주장하지 말고 명시적으로 보고한다.

## Implementation Roadmap

### Phase A: Evidence first

- 누락된 backend/input/frontend runtime metrics를 추가한다.
- 반복 가능한 ConPTY latency 및 high-output soak harness를 추가한다.
- 현재 settings로 baseline을 기록한다.

Rollback: runtime behavior 변경 없음.

### Phase B: Server transport priority

- safe-send enforce path를 priority-aware하게 만든다.
- control/output queue를 분리한다.
- projected buffered size를 사용한다.
- 긴 queue `shift()` drain을 교체한다.
- control starvation과 coalescing test를 추가한다.

Rollback: `stabilityModes.wsSendMode`를 `direct` 또는 `safe-send-observe`로 되돌린다.

### Phase C: Frontend write-gate hardening

- snapshot, repair, ACK-tail flush를 chunked/serialized recovery write gate로 보낸다.
- repair token/nonce matching을 추가한다.
- snapshot apply와 repair apply 주변 latency debug measurement를 추가한다.

Rollback: setting 뒤에 legacy direct snapshot/repair write path를 복원한다.

### Phase D: Settings exposure

- `resourceLimits.ws.outputCoalesceWindowMs`를 unreserve한다.
- Settings UI field를 추가한다.
- 가능하면 `stabilityModes.wsSendMode`도 함께 추가/제어하거나 dependency를 명확히 표시한다.
- PATCH, persistence, runtime, frontend Settings test를 추가한다.

Rollback: field를 다시 reserve하고 config-file-only tuning으로 유지한다.

### Phase E: Default rollout

Soak evidence 이후에만 제품 기본값 변경을 검토한다.

- `wsSendMode`: `direct` -> `safe-send-enforce`
- `inputReliabilityMode`: 필요한 경우 observe-style behavior -> queue/reject behavior
- frontend runtime residency bounded mode가 아직 제품 기본값이 아니라면 기본값 후보로 검토

Rollback: git reset이나 process-name kill이 아니라 setting switch로 한다.

## Decision

ConPTY 문제는 ConPTY의 시각적 품질을 포기하지 않고도 해결 가능하다.

가장 확신도 높은 fix set:

1. server safe-send를 활성화하고 control priority로 보강한다.
2. snapshot과 repair write를 chunk/schedule한다.
3. unified socket이 허용하는 한 input/control을 output과 독립시킨다.
4. output coalescing interval을 Settings에 노출한다.
5. 기본값을 바꾸기 전에 ConPTY-specific input/recovery latency를 측정한다.

사용자가 요청한 configurable coalescing unit은 적절한 방향이다. repo에는 이미 backend setting이 있다. 남은 작업은 reserved-capability 제거, Settings UI 노출, test 추가, 그리고 safe-send queue path를 사용해 해당 설정이 실제 의미를 갖게 만드는 것이다.
