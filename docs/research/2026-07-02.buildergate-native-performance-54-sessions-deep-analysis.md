## 1. 목표 정의 — "네이티브 동등"의 정량 기준

사용자 목표: **웹 브라우저에서 54개 이상 셸을 동시에 띄워도 네이티브 PowerShell/터미널과 동등한 체감 성능.** 주 워크로드는 코딩 에이전트(codex, claude 등)의 고빈도 TUI 재그리기.

| 지표 | 목표 | 현재 (분석 추정) |
|---|---|---|
| 키 입력 → 로컬 에코 지연 | p99 < 50ms (부하 중 포함) | 대량 출력 중 수 초~무응답 |
| 대량 출력 처리량 (codex 전체 재출력) | 네이티브와 체감 동등 | **~960KB/s 상한** (16KB × 60fps) |
| 그리드 접속 직후 셸 크기 | 즉시 정확 | 작게 렌더, window resize 후 교정 |
| 장시간 사용 후 입력 지연 | 열화 없음 | 누적 열화, 새로고침 시 회복 |

| 세션 종료 시 잔존 프로세스 | 0 (검증 가능) | 트리 종료 휴면(observe), 가시성 0 |
| 동시 세션 수 | 54+ | config 상한 32 (모순) |

## 2. 증상 → 근본 원인 매핑 (교차 검증 결과)

| 사용자 증상 | 근본 원인 (발견 ID) |
|---|---|
| 화면이 빠릿하지 않음 | **F1**(DOM 렌더러) + **F2**(16KB/frame 스로틀) + F5(서버 headless 직렬화) + F12(transport 직렬 전송) |
| codex 대량 스크롤이 네이티브보다 훨씬 느림 | **F2**(주범: 가시 처리량 960KB/s 상한) + F1 + F5 |
| 쓸수록 키 입력이 느려짐, 새로고침 시 회복 | **F8**(residency soft-cap 허점 + scrollback 10000×N) + F11(localStorage 전체 스캔) + F3(고아 프로세스 누적, 서버 측) + F19(dev 핫리로드 고아) |

| 종종 화면이 다 깨짐 | **F10**(라이브 출력 vs repair/snapshot 이중 write 경로 순서 역전) + F16(서버 resize가 headless write 체인 밖) |
| 세션 닫아도 하위 프로세스 잔존 의심 | **F3**(정확한 의심: enforce 휴면 + stub 인스펙터 + node-pty ConPTY kill의 detached-child 한계) |
| 54개 동시 셸 확장성 | F13(maxTotalSessions=32 모순 + headless 54개 파싱 CPU) + F4(세션 생성마다 ~1초 동기 블로킹) + F14(DOM 렌더러×N 상시 비용) |

핵심 통찰: **단일 원인이 아니라 "프레임당 16KB 스로틀 × CPU 전용 DOM 렌더러 × 서버 직렬화 게이트"가 곱해진 구조적 상한**이 존재하고, 그 위에 입력 경로의 고정 비용과 누적 비용이 겹친다. 순수 메모리 누수는 5개 축 모두에서 발견되지 않았다 — 열화는 leak이 아니라 **상주 비용의 스케일링 실패**다.

## 3. 발견 상세 (심각도순)

### CRITICAL

#### F1. xterm 렌더러 애드온 부재 — 가장 느린 DOM 렌더러로 동작
- 근거: `frontend/package.json`(의존성에 `@xterm/addon-webgl`/`addon-canvas` 없음, fit·serialize만), `frontend/src/components/Terminal/TerminalView.tsx:1642-1645`(fit/serialize만 loadAddon), `:2011`(주석이 "DOM 렌더러 모드" 명시).
- 영향: WebGL 대비 최대 ~9배 느린 렌더링(xterm.js 공식 수치 기준 WebGL이 canvas 대비 최대 900%, canvas > DOM). 대량 출력 시 행 재구성·스타일 재계산이 전부 메인 스레드 CPU에서 발생 → 출력 버벅임 + keydown 밀림의 공통 토대. 그리드에서 N개 동시 표시 시 비용 N배.
- 개선: `@xterm/addon-webgl` 도입. 단 **브라우저 WebGL context 상한(데스크톱 ~16개)** 때문에 54셸 전부 WebGL은 불가 — **"보이는/포커스 터미널만 WebGL 부착, 파킹 터미널은 addon dispose(DOM 폴백)"** 라이프사이클 관리 필수. `webglcontextlost` 시 canvas→DOM 자동 폴백.

#### F2. 가시 출력 스로틀: `visibleFlushBudgetBytes=16KB` + rAF당 1청크 flush
- 근거: `server/config.json5:79`(16384 — 스키마 기본 262144의 1/16로 하향, `server/src/schemas/config.schema.ts:150`), `frontend/src/utils/terminalOutputScheduler.ts:52-105`(flush는 프레임당 예산 1청크만 write, 콜백에서 다음 rAF 예약).
- 영향: 가시 출력 처리량 상한 ≈ **16KB × 60fps ≈ 960KB/s**(낙관치). codex 200KB 재그리기 = 약 13프레임(216ms+), 수 MB 덤프 = 수 초. "네이티브 PowerShell은 빠른데 여기선 한참 기다린다"의 직접 원인. backlog가 4MB(visibleOutputQueueMaxBytes) 도달 시 stale→overflow→복구 배리어(F6)로 연쇄.
- 개선: (1) config 값을 스키마 기본 256KB 수준으로 복원, (2) 스케줄러를 "rAF당 1청크"에서 **"프레임당 시간 예산(예: 6~8ms)까지 다중 청크 flush"**로 변경(입력 yield 유지). 이 항목은 설정+국소 수정만으로 즉효가 있는 최상위 leverage.

#### F3. 프로세스 트리 종료가 휴면 상태 (observe) + 관측 스텁 + node-pty kill 한계
- 근거: `config.json5`에 `session.processCleanup` 미지정 → 스키마 기본 `mode:'observe'`(`config.schema.ts:84-89`). enforce 분기만 `DefaultProcessTreeTerminator`(taskkill `/PID <pid> /T /F`, `server/src/utils/processTreeTerminator.ts:592-605`) 호출(`server/src/services/SessionManager.ts:1149-1171`) — 현재 미실행. 프로덕션 인스펙터는 아무것도 샘플링하지 않는 스텁(`SessionManager.ts:389-392, 536, 3280`). node-pty v1.1.0의 ConPTY `kill()`은 콘솔 attach 프로세스만 개별 kill + 열거가 비동기라 레이스 존재 + **detached 자식(MCP 서버, 데몬, `Start-Process`, WSL)은 생존**(`node_modules/node-pty/lib/windowsPtyAgent.js` 직접 확인).
- 영향: "세션 닫아도 하위 프로세스가 안 닫히는지 모르겠다"는 의심이 코드상 정확. 에이전트가 자식을 반복 spawn하는 워크로드에서 고아 누적 → 서버 "쓸수록 느려짐"에 직접 기여. 관측조차 스텁이라 서버가 스스로 알 수 없음.
- 개선: (1) `processCleanup.mode: "enforce"` 활성화(코드·테스트 기구현), (2) 종료 배치 병렬화 — 54세션 shutdown 시 세션당 PowerShell CIM 신원조회(1.5s 타임아웃)+graceful/force 대기가 순차면 shutdown 3s 타임아웃 초과, (3) observe 유지 구간이라도 스텁 인스펙터를 실제 descendant 샘플러로 교체, (4) WSL Linux-side 종료는 별도 트랙, (5) 장기적으로 Job Object(네이티브 애드온) 검토 — taskkill /T는 종료 시점의 트리만 잡고 re-parented 프로세스는 놓침.

#### F4. 세션 생성마다 동기 PowerShell로 이벤트 루프 최대 ~1초 블로킹 (신규 발견)
- 근거: `createSession` 동기 경로에서 `readProcessStartIdentitySync` → `execFileSync('powershell.exe', ..., {timeout:1000})`(`processTreeTerminator.ts:328-359`, 호출 `SessionManager.ts:2345, 566`).
- 영향: 세션 생성/시작 복구 일괄 생성 시 N×1초 순차 블로킹 — 그동안 전체 세션의 WS/HTTP/PTY I/O 정지. observe 모드에선 이 값을 쓰지도 않음.
- 개선: 비동기화 또는 생성 크리티컬 패스에서 제거(신원 조회를 종료 시점으로 지연). observe에선 호출 자체 생략.

### HIGH

#### F5. 서버 headless write 직렬화 게이트
- 근거: PTY 출력이 세션당 단일 promise 체인에서 `await writeHeadlessTerminal`(xterm-headless 파싱 완료 대기) 후에야 `routeSessionOutput`으로 브라우저에 전달(`SessionManager.ts:2988, 3022, 3038`, `server/src/utils/headlessTerminal.ts:120-127`).
- 영향: 청크 N+1은 N의 headless 파싱 완료 후 전송 → 고빈도 출력에서 처리율 상한+지연 추가. 54세션 고출력 시 단일 이벤트 루프에서 54개 체인이 인터리브되어 CPU 경쟁.
- 개선: 브라우저 라우팅을 headless write와 분리(병렬/선행 전달, headless는 snapshot 용도로 비동기 반영). seq/ack 정합성 유지 설계 필요.

#### F6. 대량 출력 중 입력 프리즈의 결합 구조
- 근거: 스케줄러 큐 4MB 초과 → stale → `visible_output_overflow` → 복구 배리어(`terminalOutputScheduler.ts:109-128`, `frontend/src/components/Terminal/TerminalContainer.tsx:1770-1799`). 배리어 중 입력은 fast-path 불가, 기본 observe 모드에선 transient-blocked 입력을 **드롭**(`TerminalView.tsx:667-683`). 배리어 해제 신호·복구 페이로드가 소켓/레인에서 출력 뒤에 정렬 — `session:ready` 등 sessionId 보유 메시지는 terminal-control 레인(출력과 순서 보존, `server/src/ws/wsSendPolicy.ts:160-168`).
- 영향: F2의 낮은 스로틀 탓에 배리어 해제(복구 repair 왕복+출력 배압 소진)가 오래 걸려 "키 입력 안 됨" 체감이 길어짐.
- 개선: 활성 snapshot/repair 트랜잭션이 없을 때 `session:ready`를 control 우선 레인으로 승격해 입력 조기 해제. F2 해소 시 배리어 진입 빈도 자체가 급감.

#### F7. 키 입력 1회당 고비용 디버그 페이로드 3회 계산 (캡처 off여도 실행)
- 근거: `buildTerminalInputDebugPayload`가 키당 3회 호출(`TerminalView.tsx:1894`, `TerminalContainer.tsx:1697`, `:708`) + sequencer flush에서 재계산(`frontend/src/utils/terminalInputSequencer.ts:99, 166-172`). 내부에서 매번 `new Intl.Segmenter`(`frontend/src/utils/terminalDebugCapture.ts:516`)·`new TextEncoder`(`:501`)·regex 5종.
- 영향: 키당 수백 µs 고정 비용 — keydown 핸들러를 길게 만들어 대량 출력 중 프리즈 악화. (선행 연구 지적 후 오히려 악화된 유일 항목.)
- 개선: 최초 지점에서 lightweight 메타데이터 1회 생성 후 전달, Segmenter/TextEncoder 모듈 싱글턴화, grapheme/preview는 캡처 활성 시 lazy.

#### F8. residency soft-cap 허점 — "쓸수록 느려짐"의 프론트 측 주 요인
- 근거: `bounded` residency(기본, maxLiveTerminals=12)가 pinned+**protected-hidden(최근 60초 내 접근) 탭을 cap 적용 전에 무조건 resident 유지**(`frontend/src/hooks/useTerminalRuntimeResidency.ts:122-155`). 54개를 번갈아 쓰는 에이전트 병렬 운용 패턴에서는 대부분 protected → resident 최대 ~54까지 팽창. 탭당 xterm 인스턴스+**scrollback 10000줄**(`TerminalView.tsx:1637`)+observer+구독 상주. 새로고침이 mounted 집합을 초기화 → "회복" 증상과 정확히 일치.
- 개선: protected-hidden에도 hard-cap 적용(초과 시 LRU evict), 그리드 워크로드용 scrollback 프로파일(예: 2000줄) 검토 — xterm.js 실측상 5000줄 가득 찬 버퍼 ≈ 34MB/세션.

#### F9. 그리드 초기 크기 버그 — 두 개의 확정된 비대칭 결함
- 근거: ① 레이아웃 변경용 임퍼러티브 `fit()` 핸들이 `fitAddon.fit()`만 하고 `emitResize` 미호출 + `term.onResize` 미구독이라 서버 PTY는 이전(작은) 크기 유지(`TerminalView.tsx:1532-1537, 798-801`). ② rect 불변 시 `sameRect` 비교로 오버레이 재렌더/ResizeObserver 재발동이 억제되어 재교정 경로 차단(`frontend/src/contexts/TerminalRuntimeContext.tsx:80-89, 115-117, 144-149`). 최초 visible-fit(single rAF)이 레이아웃 정착 전 작은 cols/rows를 서버로 전송하면 이후 교정 불가. **window resize만 항상 `.terminal-view` 박스를 바꿔 교정 경로를 태움** — "창 크기 바꾸면 괜찮아짐"과 정합.
- 개선: `term.onResize` 구독으로 cols/rows 변경 시 항상 서버 통보 + 가시 전환 fit을 "2프레임 연속 동일 크기 확인 후 fit"으로 강화 + `document.fonts.ready` 후 1회 재fit(환경 의존 보조).

#### F10. 화면 깨짐 — 이중 write 경로의 순서 역전
- 근거: 라이브 출력은 스케줄러(rAF 비동기) 경유, screen-repair(`TerminalView.tsx:1445`)·snapshot 교체(`:1311, 1318`)·snapshot 복원(`:1285`)은 `term.write` 직접 호출로 스케줄러 우회. 둘 사이 드레인/플러시 조율 없음. snapshot 교체(`term.reset()`) 시 `outputScheduler.reset()` 미호출 — 리셋된 화면 위로 옛 청크가 쏟아질 수 있음. repair 게이트는 스케줄러 큐 잔량을 확인하지 않음(`TerminalView.tsx:1337-1386`).
- 메커니즘: resize 150ms 후 repair 요청(`TerminalContainer.tsx:1719-1722`) → repair 패치가 즉시 write → 뒤늦게 스케줄러 rAF가 **더 오래된 라이브 청크를 repair 위에 덮어씀**.
- 개선: repair/snapshot도 단일 write 파이프라인(스케줄러 우선순위 항목)으로 통일하거나 직접 write 전 동기 드레인 + snapshot 교체 시 scheduler reset 의무화.

#### F11. localStorage snapshot eviction의 동기 전체 스캔
- 근거: snapshot 저장(2초 debounce, resident마다) 시 eviction이 localStorage 전체 순회 ×2 + 항목별 최대 2MB `JSON.parse`(`frontend/src/utils/terminalSnapshot.ts:194-227, 262, 269, 72-91`). tombstone 정리는 config 로드 시에만.
- 영향: 세션 open/close 반복 → 스캔 대상 증가 → 시간 비례 열화(새로고침으로도 회복 안 되는 유일한 누적 축).
- 개선: 평상시 즉시 setItem + `QuotaExceededError` 시에만 evict, 또는 인덱스/총량 캐시. 장기적으로 IndexedDB.

#### F12. 서버 transport 1메시지/콜백 직렬 전송
- 근거: `flushTransportQueue`가 dequeue 1건 전송 후 send 콜백에서 다음 flush(`server/src/ws/WsRouter.ts:2199-2261`) — 메시지당 최소 이벤트루프 1틱.
- 개선: 콜백당 바이트 예산까지 다중 dequeue drain.

#### F13. 54세션 규모 정합성 — 상한 모순 + 서버 파싱 예산 부재
- 근거: `workspace.maxTotalSessions: 32`(`config.json5:73`) vs 실사용 54. 서버는 구독 여부와 무관하게 54개 PTY 출력 전부를 headless 파싱(단일 스레드).
- 개선: 상한 상향과 동시에 비구독/비활성 세션의 headless 파싱 경감(raw tail 유지 + lazy rehydrate) 설계. 메모리는 세션당 1~3MB+캐시로 54셸 ≈ 100~300MB 수준(관리 가능) — 병목은 CPU.

#### F14. 그리드 상시 렌더 비용 — cursorBlink × N + `:has()` 재계산
- 근거: `cursorBlink: true`(`TerminalView.tsx:1611`)가 터미널마다 점멸 리페인트. 활동 표시가 출력 write마다/keydown마다 클래스 토글 + `:has()` 선택자 매칭(`frontend/src/styles/breathing.css:13-64`, `TerminalView.tsx:1178-1184, 1721-1727`).
- 개선: 포커스 터미널만 blink, `:has()` 대신 부모 직접 클래스 부여.

### MEDIUM

- **F15. JSON 텍스트 프레이밍**: 출력이 `{"type":"output",...,"data":"<ANSI>"}` JSON — ESC가 `` 6바이트로 팽창, 메시지당 서버 stringify+브라우저 parse(`wsSendPolicy.ts:39`, `frontend/src/contexts/WebSocketContext.tsx:221`). 개선: 출력 전용 binary frame(ArrayBuffer) 또는 Web Worker 수신 파싱. 압축(permessage-deflate)은 54연결에서 메모리 폭탄이므로 off 또는 window 11/mem 4 확인.
- **F16. 서버 resize가 headless write 체인 밖에서 동기 적용**(`SessionManager.ts:1367-1380`): 큐잉된 옛-크기 출력이 새-크기 headless에 write되어 mirror divergence → 깨진 snapshot/repair의 서버 측 원인. 개선: resize도 `headlessWriteChain`에 큐잉.
- **F17. PTY onData 청크별 동기 detector 4종+디버그 캡처**(`SessionManager.ts:679-786`): 고빈도 출력 시 서버 CPU 가중. 개선: 버스트 중 샘플링.
- **F18. onOutput 메시지당 할당**: 매번 `new TextEncoder`·limits 객체 spread(`TerminalContainer.tsx:96-98, 1661-1663`) — 싱글턴/캐시화.
- **F19. dev 핫리로드 고아**: `tsx watch`가 서버 프로세스를 교체할 때 기존 PTY 트리가 정리 보장 없음(`server/package.json:7`, `dev.js:54-66`은 dev.js 종료 시에만 taskkill). F3 enforce + shutdown 완주 보장으로 함께 해소.
- **F20. HostSlot 불안정 props**: 렌더마다 observer/listener 재구독 churn(`frontend/src/App.tsx:599-601`, `TerminalHostSlot.tsx:57`).
- **F21. 스키마 기본 `wsSendMode='direct'` footgun**: 현 인스턴스는 config로 enforce 활성(`config.json5:84`)이나, config 누락/재생성 시 backpressure 없는 direct로 원복(`config.schema.ts:198`). 개선: 스키마 기본값을 enforce로 승격(soak 증거 확보 후).
- **F22. WSL 세션은 enforce에서도 트리 종료 skip**(`processTreeTerminator.ts:419-421`).

### LOW

- node-pty flowControl 미사용(현행 유지 무방 — ConPTY 품질 우선, 애플리케이션 레벨 ACK가 더 안전).
- `isInputPending` 양보가 Chromium 전용.
- `takeUtf8Prefix`가 코드포인트 단위 encode(대량 flush 시 오버헤드, `terminalOutputScheduler.ts:186-193`).
- ANSI escape의 청크 경계 절단: 동일 파서에 순차 write되므로 스케줄러 경로 내에서는 무해(깨짐은 F10의 경로 간 문제).
- 크기 불일치 snapshot 적용: geometry 게이트로 방어됨(`SessionManager.ts:2274-2283`, `TerminalView.tsx:1409-1430`).

### 로드맵 주의사항

- **P0-1과 P1-1이 최상위 leverage 쌍**이다: P0-1이 병목의 뚜껑(스로틀)을 열고, P1-1이 열린 뚜껑을 감당할 렌더 처리량을 제공한다. P0-1만 하면 DOM 렌더러가 다음 병목으로 드러날 수 있으므로 순차 적용 후 각각 실측할 것.

## 6. 검증 계획 (수용 기준 예시)

1. **출력 처리량**: 5MB ANSI 덤프를 활성 터미널에 재생 → 완료까지 벽시계 시간, P0-1 전/후 및 P1-1 전/후 비교. 목표: 전 구간 합산 10배 이상 개선.
2. **부하 중 입력 지연**: codex급 재그리기 시뮬레이션 중 keydown→에코 p99 측정(기존 telemetry + `/api/sessions/telemetry` 확장). 목표 p99 < 50ms.
3. **그리드 초기 크기**: Playwright E2E — 그리드 워크스페이스 접속 직후 cols/rows가 서버 PTY와 일치(창 크기 변경 없이). 회귀 테스트로 고정.
4. **프로세스 트리**: E2E — 세션에서 자식 spawn(detached 포함) 후 세션 삭제 → descendant 0 검증(enforce). observe 텔레메트리로 잔존 카운트 노출.
5. **누적 열화**: 54세션 round-robin 입력/출력 soak(≥2h) — heap/입력 지연 추세 플랫 확인.
6. **화면 무결성**: resize 폭풍 + 대량 출력 동시 주입 반복 → snapshot/실화면 diff 0.

## 8. 방법론 및 한계

- 서브에이전트 5개가 독립 분석 후 메인 세션이 상충 판정을 교차 검증했다. 대표 상충: 한 축이 `wsSendMode='direct'` 활성으로 판정했으나 `config.json5:84` 직접 확인으로 enforce 활성이 맞음을 확정(스키마 기본값 위험은 F21로 유지).

- 발생 빈도 미실측 항목은 본문에 추정으로 표시했다(F10의 repair 시점 큐 잔량 등).
