# MCP 세션 조회 및 세션 간 메시지 전달 가능성 1차 조사

작성일: 2026-07-08 KST  
대상: `C:\Work\git\_Snoworca\ProjectMaster`  
질문:

1. MCP를 통하여 세션 터미널에 올라간 자신의 세션 ID와 다른 세션 ID 및 이름 리스트를 가져올 수 있는가?
2. MCP와 연결된 코딩 에이전트가 세션 ID를 통하여 다른 세션에 메시지를 전달할 수 있는가?

## 결론

현재 구현 그대로는 두 기능 모두 "MCP를 통해서는 아직 불가능"하다.

다만 기존 BuilderGate 서버 구조 위에 MCP tool layer를 추가하면 구현 가능하다. 정확히 나누면 다음과 같다.

| 항목 | 현재 MCP로 가능 여부 | 기존 코드 기반 구현 가능성 | 핵심 이유 |
| --- | --- | --- | --- |
| 다른 세션 ID/이름 목록 조회 | 불가능 | 높음 | `GET /api/sessions`와 `SessionManager.getAllSessions()`가 이미 세션 목록을 제공한다. 이를 MCP read-only tool로 감싸면 된다. |
| 자신의 세션 ID 식별 | 불가능에 가까움 | 중간~높음 | 현재 PTY 안의 에이전트에게 `sessionId`가 환경변수/세션 키로 주입되지 않는다. 구현하려면 `BUILDERGATE_SESSION_ID` 같은 자기 식별 수단 또는 hook 기반 등록이 필요하다. |
| sessionId로 다른 세션에 메시지 전달 | 불가능 | 가능하지만 고위험 | `SessionManager.writeInput()`와 WebSocket input path가 있으나, 현재 MCP tool이 없고 REST input endpoint는 제거됐다. 안전한 server-origin input service가 필요하다. |

따라서 "가능/불가능"만 판단하면, **현재는 불가능, 설계·구현을 추가하면 가능**이다.

## 현재 코드 근거

### 1. 세션 목록 조회 기반은 이미 있다

서버에는 전체 세션 목록 API가 있다.

- `server/src/routes/sessionRoutes.ts:15` - `GET /api/sessions` 주석
- `server/src/routes/sessionRoutes.ts:17` - `sessionManager.getAllSessions()` 호출
- `server/src/services/SessionManager.ts:1152` - `getAllSessions(): SessionDTO[]`
- `server/src/types/index.ts:95` - `SessionDTO`는 `id`, `name`, `status`, `createdAt`, `lastActiveAt`, `sortOrder`를 포함한다.

단, 이 API는 일반 REST API이며 MCP tool이 아니다. 또한 `server/src/index.ts:332`에서 `/api/sessions`가 `authMiddleware` 뒤에 mount되므로, 외부 MCP 서버가 이를 직접 감싸려면 별도 인증 모델이 필요하다.

### 2. 현재 REST input endpoint는 제거됐다

프론트엔드 API와 서버 route에 모두 같은 방향의 흔적이 있다.

- `frontend/src/services/api.ts:228` - `sendInput and resize removed — now handled via WebSocket`
- `server/src/routes/sessionRoutes.ts:138` - `POST /:id/input`, `POST /:id/resize`, `GET /:id/stream` 제거 주석

즉 현재 공식 입력 경로는 WebSocket이다. MCP tool이 단순히 HTTP POST로 `/api/sessions/:id/input`을 호출하는 방식은 현재 코드에 없다.

### 3. 내부적으로는 세션 입력 주입이 가능하다

서버 내부에는 PTY에 입력을 쓰는 메서드가 있다.

- `server/src/services/SessionManager.ts:1250` - `writeInput(id, input, ...)`
- `server/src/ws/WsRouter.ts:514` - WebSocket `handleInput`
- `server/src/ws/WsRouter.ts:515` - `validateInputMessage`
- `server/src/ws/WsRouter.ts:528` - replay pending 확인
- `server/src/ws/WsRouter.ts:586` - 검증 후 `sessionManager.writeInput(...)`

따라서 "sessionId를 알고 있을 때 서버가 해당 PTY에 입력을 전달할 수 있는가"는 가능하다. 하지만 MCP write tool이 `writeInput()`을 직접 호출하면 WebSocket의 replay/input queue, Enter 안전 정책, rejection telemetry 일부를 우회할 수 있다. 그래서 직접 호출보다 `SessionInputService` 같은 공용 ingress를 추가하는 편이 맞다.

### 4. 자기 세션 ID 식별은 현재 부족하다

현재 `WorkspaceTab`은 `sessionId`를 저장한다.

- `server/src/types/workspace.types.ts:19` - `WorkspaceTab`
- `server/src/types/workspace.types.ts:22` - `sessionId: string`
- `server/src/services/WorkspaceService.ts:331` - tab 생성
- `server/src/services/WorkspaceService.ts:357` - tab에 `sessionDTO.id` 저장
- `server/src/services/WorkspaceService.ts:519` - tab restart
- `server/src/services/WorkspaceService.ts:540` - restart 시 새 `sessionId`로 교체
- `server/src/services/WorkspaceService.ts:1349`~`1352` - orphan recovery 시 새 session 생성 및 `tab.sessionId` 갱신

하지만 PTY 안에서 실행되는 코딩 에이전트가 "내가 어느 BuilderGate sessionId에 있는지" 알 수 있는 안정적 주입 경로는 검색되지 않았다. `SessionManager.createSession()`은 ID 생성 후 PTY를 spawn하지만, `buildShellEnv(shellType)`에 sessionId를 넘기지 않는다.

관련 근거:

- `server/src/services/SessionManager.ts:572` - session id 생성 시작
- `server/src/services/SessionManager.ts:583` - `buildShellEnv(shellType)` 호출
- `server/src/services/SessionManager.ts:1770` - `buildShellEnv(...)`

즉 현재는 "서버가 모든 세션을 알고 있음"과 "터미널 안의 에이전트가 자기 세션을 알고 있음" 사이에 연결고리가 없다.

## MCP 사양 관점

MCP 자체는 이 기능을 표현하기에 충분하다. 공식 사양상 서버는 tool을 노출할 수 있고, client는 `tools/list`, `tools/call`로 도구를 발견하고 호출한다. HTTP 기반 transport를 쓰면 JSON-RPC 메시지는 UTF-8이어야 하며, Streamable HTTP는 단일 MCP endpoint를 통해 POST/GET을 처리한다.

중요한 보안 요구도 있다. MCP tool 서버는 입력 검증, access control, rate limiting, output sanitization을 해야 하며, client는 민감 작업에 사용자 확인, timeout, audit logging을 고려해야 한다. Streamable HTTP를 로컬에서 열 경우 Origin 검증, localhost bind, 인증이 필요하다.

따라서 BuilderGate MCP 서버를 만든다면 읽기 tool과 쓰기 tool을 분리하고, session input 전달은 민감 작업으로 취급해야 한다.

## 권장 구현 형태

### 1차 read-only MCP

먼저 다음 tool만 구현하는 것이 안전하다.

```text
list_sessions
get_session
get_workspace_tabs
whoami_session
```

`list_sessions`는 기존 `SessionManager.getAllSessions()`와 `WorkspaceService`의 tab 정보를 합쳐 다음 정도를 반환하면 된다.

```json
{
  "sessions": [
    {
      "sessionId": "runtime-pty-session-id",
      "name": "Tab or Session name",
      "status": "idle",
      "workspaceId": "workspace-id",
      "tabId": "tab-id",
      "recoverable": true
    }
  ]
}
```

`whoami_session`은 현재 상태로는 바로 만들기 어렵다. 권장안은 PTY spawn 시 다음 환경변수를 주입하는 것이다.

```text
BUILDERGATE_SESSION_ID
BUILDERGATE_TAB_ID
BUILDERGATE_WORKSPACE_ID
```

단순 ID만으로 write 권한을 주면 안 된다. write capability는 별도 secret이나 signed capability로 분리해야 한다.

### 2차 controlled write MCP

그 다음에 다음 tool을 추가한다.

```text
preview_session_message
deliver_session_message
request_session_attention
```

`deliver_session_message`는 기본을 paste-only로 두고, Enter 포함 자동 제출은 승인/정책이 있을 때만 허용해야 한다.

필수 정책:

- 대상 session 존재 확인
- actor session과 target session의 role/scope 확인
- payload UTF-8 byte limit
- control character와 newline policy
- replay/snapshot pending 시 queue 또는 reject 명시
- Enter 포함 late flush 금지
- audit log 기록
- prompt 전문 저장 금지, hash/preview만 저장

### 장기 안정 식별자

`sessionId`는 restart/orphan recovery 때 바뀐다. 장기 팀원 식별에는 `sessionKey`가 필요하다.

권장:

- `sessionId`: 현재 PTY runtime id
- `sessionKey`: tab/team member에 붙는 stable id
- `currentSessionId`: sessionKey가 현재 가리키는 runtime session id

이 판단은 기존 연구 `docs/research/2026-05-19.team-session-mcp-completion-protocol-research.md`와도 일치한다.

## 위험 요약

| 위험 | 영향 | 완화 |
| --- | --- | --- |
| MCP write tool이 `writeInput()` 직접 호출 | replay/input safety 우회 | `SessionInputService`로 공용 입력 경로 통합 |
| `sessionId`를 장기 식별자로 사용 | restart 후 잘못된 세션에 전달 | `sessionKey -> currentSessionId` mapping |
| 자기 sessionId를 환경변수만으로 권한 처리 | 세션 내부 탈취 시 권한 상승 | 식별자와 write capability 분리 |
| Enter 포함 자동 제출 | 원치 않는 명령 실행 | paste-only 기본, submit은 승인 필요 |
| raw prompt/audit 저장 | 민감 정보 노출 | hash, byte length, safe preview 저장 |
| local MCP HTTP 노출 | DNS rebinding/외부 제어 | Origin 검증, localhost bind, 인증 |

## 최종 판정

1. 자신의 세션 ID와 다른 세션 ID/name 목록 조회:
   - 다른 세션 목록 조회는 기존 REST/서비스 기반이 있으므로 MCP tool 추가만으로 가능하다.
   - 자신의 세션 ID 식별은 현재 구현에는 없다. PTY env 주입 또는 hook/session registry가 필요하다.

2. sessionId를 통한 다른 세션 메시지 전달:
   - 현재 MCP로는 불가능하다.
   - 내부적으로 PTY 입력 전달 기능은 있으므로 구현은 가능하다.
   - 단, 직접 `writeInput()`을 MCP에 노출하면 위험하다. replay/input queue와 audit/approval을 통과하는 별도 server-origin input layer가 필요하다.

따라서 연구 질문의 답은 다음이다.

> 현재 상태: 불가능.  
> 구현 가능성: 높음. 단, read-only 조회부터 시작하고, write 전달은 `sessionKey`, 권한, 승인, audit, input safety를 붙인 뒤 열어야 한다.
