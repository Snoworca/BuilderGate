# 터미널 제어 시퀀스 기반 idle/running 판정 연구

작성일: 2026-04-28

대상 프로젝트: `C:\Work\git\_Snoworca\ProjectMaster`

분석 방식: Windows Terminal 공식 문서, Codex CLI 공개 소스, Claude Code 공식 문서, 서브에이전트 2개 연구 결과를 취합했다.

## 1. 분석 목적

BuilderGate는 Codex, Claude, Hermes 같은 interactive AI TUI 세션을 그리드 터미널에서 관찰하고 `idle`/`running` 상태를 판단한다.

기존 핵심 원칙은 다음과 같다.

> 사용자가 Codex, Claude, Hermes 같은 interactive AI TUI에 키보드를 입력할 때 해당 세션은 `idle` 상태를 유지해야 한다. 사용자 키보드 입력, local echo, prompt redraw, cursor movement, ticker output, waiting-for-input repaint는 `running`으로 전환하면 안 된다.

이 문서는 Windows Terminal에서 Codex/Claude 사용 중 보이는 탭 spinner, progress ring, bell 아이콘 같은 터미널 제어 시퀀스를 BuilderGate의 `idle`/`running` 판정에 활용할 수 있는지 연구한다.

## 2. 결론 요약

터미널 제어 시퀀스는 `idle`/`running` 판정에 활용 가능하다. 단, 단독 주 판단 로직으로 쓰면 위험하며 기존 semantic output classifier를 보강하는 보조 신호로 사용해야 한다.

가장 강한 신호는 `OSC 9;4` progress sequence다. `state=1`, `2`, `3`, `4`는 작업 진행 또는 비정상 진행 상태를 뜻하므로 `running` 신호로 볼 수 있다. `state=0`은 progress clear 신호이므로 `idle 후보`로 볼 수 있다.

Codex CLI는 공개 소스 기준으로 터미널 title에 activity spinner를 기록한다. 이 title update는 `OSC 0` sequence로 출력되며 Windows Terminal 탭 제목에 반영될 수 있다. 따라서 Codex title spinner 반복 갱신은 `running` 보조 신호가 될 수 있다.

독립 `BEL` 문자는 완료, 권한 요청, 사용자 주의 필요 같은 상태를 알리는 데 쓰일 수 있다. 그러나 `BEL`은 임의 프로그램도 출력할 수 있고, `OSC` 문자열의 종료자로도 쓰이므로 단독으로 `idle` 확정 신호로 쓰면 안 된다.

## 3. 터미널 측 메커니즘

### 3.1 OSC sequence

`OSC`는 Operating System Command의 약자로 `ESC ]`로 시작한다. 일반적인 ANSI cursor/color 제어인 `CSI`(`ESC [`)와 다르다.

`OSC` sequence는 보통 `BEL`(`0x07`) 또는 `ST`(`ESC \`)로 종료된다.

중요한 점은 `OSC` 내부의 `BEL`은 sequence terminator일 뿐, 독립적인 terminal bell 이벤트로 해석하면 안 된다는 것이다.

### 3.2 Windows Terminal progress sequence

Windows Terminal은 ConEmu 계열 progress sequence인 `OSC 9;4`를 지원한다.

형식은 다음과 같다.

```text
ESC ] 9 ; 4 ; <state> ; <progress> BEL
```

상태값은 다음처럼 해석된다.

| state | 의미 | BuilderGate 판정 후보 |
| --- | --- | --- |
| `0` | progress 숨김/초기화 | `idle 후보` |
| `1` | normal progress | `running` |
| `2` | error progress | `running` 또는 `attention` |
| `3` | indeterminate progress | `running` |
| `4` | warning/paused progress | `running` 또는 `attention` |

Windows Terminal은 이 sequence를 수신하면 탭 헤더에 progress ring을 표시하고, 작업표시줄에도 진행 상태를 표시할 수 있다.

### 3.3 Terminal bell

독립 `BEL`(`0x07`) 문자는 terminal bell 이벤트다.

Windows Terminal의 `bellStyle` 설정에 따라 소리, 창/작업표시줄 점멸, 탭 bell indicator 등이 발생할 수 있다.

BuilderGate 관점에서는 `BEL`을 다음처럼 다뤄야 한다.

| 상황 | 의미 |
| --- | --- |
| `OSC ... BEL` 내부 terminator | bell 아님 |
| 독립 `BEL` | 주의 필요 또는 완료 후보 |
| prompt 입력 중 사용자가 만든 bell | 상태 판정 근거로 부적합 |

## 4. Codex CLI 관찰 결과

Codex CLI는 공개 Rust 소스 기준으로 다음 상태 표현을 직접 구현한다.

| 항목 | 확인 내용 |
| --- | --- |
| TUI 작업 표시 | `Working` spinner, elapsed time, `Esc to interrupt` 등을 표시한다. |
| compact 상태 | `Starting`, `Ready`, `Working`, `Waiting`, `Undoing`, `Thinking` 상태 문자열을 갖는다. |
| terminal title | `OSC 0` sequence로 title을 쓴다. |
| title activity | 기본 title item은 `activity`, `project-name`이며, activity는 작업 중 spinner로 갱신될 수 있다. |
| notification backend | `auto`, `osc9`, `bel`을 지원한다. Windows Terminal은 Codex 테스트상 `OSC 9` desktop notification 대상이 아니므로 `auto`에서는 `BEL`로 fallback된다. |

Codex title write 형식은 다음과 같다.

```text
ESC ] 0 ; <sanitized title> BEL
```

따라서 BuilderGate가 PTY raw output에서 `OSC 0` title을 파싱할 수 있다면 다음을 보조 신호로 사용할 수 있다.

| title 패턴 | 판정 후보 |
| --- | --- |
| Braille spinner 프레임 반복 | `running` |
| `Working`, `Thinking`, `Undoing` | `running` |
| `Ready` | `idle 후보` |
| `Action Required` | `idle` 또는 `attention-required` 후보 |

단, title 문자열은 사용자 설정으로 바뀔 수 있으므로 Codex 전용 강한 신호가 아니라 보조 신호로만 사용해야 한다.

## 5. Claude Code 관찰 결과

Claude Code 본체 TUI 구현은 공식 공개 저장소로 확인되지 않았다. 따라서 내부 escape sequence 생성 코드는 사실로 단정할 수 없다.

공식 문서로 확인 가능한 범위는 다음과 같다.

| 항목 | 확인 내용 |
| --- | --- |
| notification event | 작업 완료 또는 permission prompt에서 notification event가 발생한다. |
| desktop notification | Ghostty, Kitty, iTerm2는 직접 desktop notification 대상이다. 그 외 터미널은 Notification hook을 권장한다. |
| hooks | `Notification` hook과 `Stop` hook을 제공한다. |
| status line | 사용자 정의 status line 명령을 실행하고 stdout을 렌더링한다. |
| terminal progress bar | `terminalProgressBarEnabled` 설정이 있다. 공식 문서상 지원 대상은 ConEmu, Ghostty 1.2.0+, iTerm2 3.6.6+로 기재되어 있다. |

Windows Terminal은 자체적으로 `OSC 9;4` progress sequence를 지원하지만, Claude Code 공식 문서가 Windows Terminal을 progress bar 지원 대상으로 명시하지는 않는다. 따라서 Claude에서 `OSC 9;4`가 실제로 출력되는지는 환경과 버전에 따라 별도 캡처가 필요하다.

## 6. BuilderGate 적용 가능성

### 6.1 사용 가능한 raw signal

BuilderGate가 PTY raw output stream을 받을 수 있다면 다음 신호를 파싱할 수 있다.

| 신호 | 예 | 권장 판정 |
| --- | --- | --- |
| `OSC 9;4;3` | indeterminate progress | `running` 강한 신호 |
| `OSC 9;4;1` | progress value | `running` 강한 신호 |
| `OSC 9;4;2` | error progress | `running` 또는 `attention` |
| `OSC 9;4;4` | warning/paused progress | `running` 또는 `attention` |
| `OSC 9;4;0` | progress clear | `idle 후보` |
| `OSC 0` title spinner 반복 | Codex activity spinner | `running` 보조 신호 |
| `OSC 0` title `Ready` | Codex ready title | `idle 후보` |
| `OSC 0` title `Action Required` | Codex blocked state | `idle` 또는 `attention-required` 후보 |
| 독립 `BEL` | terminal bell | `attention-required` 또는 `idle 후보` |

### 6.2 권장 판정 흐름

권장 구조는 다음과 같다.

```text
PTY raw output
  -> terminal control sequence parser
  -> OSC 9;4 progress event
  -> OSC 0 title event
  -> standalone BEL event
  -> 기존 semantic output classifier와 결합
  -> running delay / idle delay 적용
  -> session status 확정
```

권장 정책은 다음과 같다.

```text
running 확정:
  - OSC 9;4 state 1/2/3/4 감지
  - 또는 Codex title spinner가 짧은 시간 안에 반복 갱신됨
  - 또는 기존 semantic classifier가 실질 agent output/tool execution을 감지

idle 후보:
  - OSC 9;4 state 0 감지
  - 또는 title Ready/Action Required 감지
  - 또는 standalone BEL 감지

idle 확정:
  - idle 후보 감지 후 idle delay 경과
  - 그 사이 새 running 신호 없음
  - user input/local echo/prompt redraw만 관찰됨

running 금지:
  - 사용자 키보드 입력
  - local echo
  - prompt redraw
  - cursor movement
  - waiting-for-input repaint
```

### 6.3 running delay와의 관계

`running delay`는 사용자가 입력을 시작하거나 prompt가 redraw되는 것만으로 `running`이 풀리는 문제를 줄이는 데 필요하다.

터미널 제어 시퀀스는 running delay의 보조 입력으로 사용할 수 있다.

| 상황 | 권장 동작 |
| --- | --- |
| 사용자 입력 직후 local echo만 발생 | `idle` 유지 |
| 사용자 입력 후 Codex title spinner 또는 `OSC 9;4;3` 발생 | running delay 후 `running` |
| `OSC 9;4;0` 또는 `BEL` 발생 | idle delay 시작 |
| idle delay 중 agent output 재개 | `running` 유지 |

## 7. 구현 시 주의사항

### 7.1 BEL 오판 방지

`OSC` sequence terminator로 쓰인 `BEL`을 독립 bell로 처리하면 안 된다.

잘못된 예:

```text
ESC ] 9 ; 4 ; 3 ; 0 BEL
```

위 문자열의 마지막 `BEL`은 progress sequence 종료자다. 이것을 완료 알림으로 처리하면 `running` 시작과 동시에 `idle 후보`가 발생하는 버그가 생긴다.

### 7.2 title 문자열 의존성

`OSC 0` title은 사용자가 설정으로 끌 수 있거나 구성 항목을 바꿀 수 있다. Codex에는 terminal title 설정이 있고, Claude Code도 `CLAUDE_CODE_DISABLE_TERMINAL_TITLE` 같은 환경 변수로 title update가 달라질 수 있다.

따라서 title 기반 판정은 항상 best-effort로 둬야 한다.

### 7.3 terminal emulator별 차이

Windows Terminal은 `OSC 9;4` progress를 지원하지만 `OSC 9` desktop notification 지원은 공식 근거가 약하다. Codex 공개 코드도 Windows Terminal을 `OSC 9` notification 지원 대상으로 보지 않고 `BEL` fallback을 사용한다.

즉, `OSC 9`와 `OSC 9;4`를 혼동하면 안 된다.

| sequence | 의미 |
| --- | --- |
| `OSC 9;4` | progress state |
| `OSC 9` | 일부 터미널의 desktop notification |

### 7.4 tmux/screen/SSH

tmux나 screen 안에서는 escape sequence가 outer terminal까지 전달되지 않을 수 있다. Claude 공식 문서도 tmux에서 notification/progress가 outer terminal에 도달하려면 passthrough 설정이 필요하다고 설명한다.

BuilderGate는 직접 PTY stream을 받는 구조라면 outer terminal 렌더링 여부와 무관하게 raw sequence를 볼 수 있다. 반대로 이미 렌더링된 DOM이나 terminal title UI만 보고 판단하면 terminal emulator 설정에 크게 의존하게 된다.

### 7.5 xterm.js 처리 위치

그리드 터미널이 xterm.js를 사용한다면, 제어 시퀀스는 xterm.js가 소비하기 전에 raw PTY stream 단계에서 파싱하는 것이 가장 안전하다.

xterm.js 렌더 결과나 DOM 텍스트에서 spinner/title/progress를 역추적하면 다음 문제가 생긴다.

| 문제 | 영향 |
| --- | --- |
| OSC sequence가 화면 텍스트로 남지 않음 | 감지 불가 |
| title/progress는 terminal addon 내부 상태로만 반영될 수 있음 | 일관성 부족 |
| local echo와 agent output 구분이 늦어짐 | idle/running 오판 가능 |

## 8. 권장 결론

BuilderGate에 이 로직을 도입한다면 다음 원칙이 적절하다.

1. `OSC 9;4`는 강한 상태 신호로 사용한다.
2. `OSC 0` title spinner는 Codex/Claude 보조 신호로 사용한다.
3. 독립 `BEL`은 `attention-required` 또는 `idle 후보` 신호로만 사용한다.
4. `OSC` 종료자 `BEL`과 독립 `BEL`을 반드시 구분한다.
5. 사용자 keyboard input, local echo, prompt redraw는 절대 `running` 전환 근거로 쓰지 않는다.
6. 최종 상태 확정은 기존 semantic classifier, running delay, idle delay와 결합한다.

이 방식은 “사용자가 입력하면 idle이어야 한다”는 원칙을 강화하는 데 도움이 된다. 특히 Codex에서 사용자가 텍스트를 입력하기만 해도 `running`이 풀리는 문제는, 입력 이벤트 자체가 아니라 agent가 실제로 출력한 `OSC 9;4`, title spinner, semantic output을 기준으로 running을 판단하도록 바꾸면 완화할 수 있다.
