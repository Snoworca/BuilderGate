# BuilderGate 기본 데몬 모드 도입 PRD 선행 분석

작성일: 2026-04-25

참고 프로젝트: `C:\Work\git\_Snoworca\TTTGate`

분석 방식: 서브에이전트 2개 병렬 분석 결과와 메인 에이전트의 코드 대조 결과를 취합했다.

## 1. 분석 목적

이 문서는 BuilderGate에 PM2 없는 네이티브 데몬 모드를 도입하기 위한 PRD 작성 전 분석이다.

핵심 요구사항은 다음과 같다.

| 항목 | 요구사항 |
| --- | --- |
| 기본 실행 모드 | 빌드된 `BuilderGate.exe` 또는 OS별 실행파일은 별도 `--daemon` 없이 기본 데몬 모드로 동작해야 한다. |
| 포그라운드 실행 | `--forground` 요청이 있었으나 표준 철자는 `--foreground`이다. PRD에서 canonical flag와 오탈자 alias 정책을 결정해야 한다. |
| 2D 바코드 출력 | 데몬 모드로 실행하더라도 TOTP QR 같은 2D 바코드는 사용자 콘솔에 먼저 출력되고, 이후 백그라운드로 전환되어야 한다. |
| PM2 제거 | 현재 PM2 기반 데몬 구현은 제거하고 자체 데몬 런처로 대체해야 한다. |
| 배포 기준 | `dist/bin` 산출물과 EXE 옆 `config.json5` 정책은 유지해야 한다. |

## 2. 결론 요약

TTTGate의 데몬 모드는 OS 서비스 등록 방식이 아니라, 실행파일이 자기 자신을 detached child로 다시 실행하고 별도 sentinel 프로세스가 app PID를 감시하는 구조다. BuilderGate에 이 패턴은 일부 재사용할 수 있지만 그대로 복사하면 안 된다.

가장 중요한 차이는 BuilderGate가 서버 시작 시 콘솔에 TOTP QR을 출력한다는 점이다. TTTGate 방식처럼 데몬 child의 `stdio`를 `ignore`로 버리면 QR 출력이 사용자에게 보이지 않는다. 따라서 BuilderGate의 새 런처는 데몬 child를 완전히 분리하기 전에 QR 출력이 보장되는 별도 단계가 필요하다.

현재 BuilderGate의 PM2 의존은 서버 본체가 아니라 `tools/start-runtime.js`, `stop.js`, `tools/build-daemon-exe.js`에 집중되어 있다. 서버 본체는 `NODE_ENV=production`, `PORT`, `BUILDERGATE_CONFIG_PATH`, `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS` 환경변수만 올바르게 받으면 PM2 없이도 실행 가능하다.

## 3. TTTGate 데몬 모드 분석

### 3.1 실행 흐름

TTTGate의 최상위 엔트리포인트는 `src/app.ts`의 `app()`이다.

| 단계 | 코드 근거 | 내용 |
| --- | --- | --- |
| CLI 파싱 | `TTTGate/src/app.ts:12-24`, `TTTGate/src/util/CLI.ts:40-55` | `server`, `client`, `stop` 중 첫 non-option 토큰을 실행 모드로 판정한다. |
| 데몬 판정 | `TTTGate/src/Sentinel.ts:20-30` | `-daemon`이 있으면 데몬 모드로 분기한다. `--daemon`은 CLI 옵션으로는 파싱되지만 sentinel 판정에는 걸리지 않는다. |
| foreground 기본값 | `TTTGate/src/Sentinel.ts:72-76` | `-daemon`이 없으면 `.pid_foreground`를 쓰고 본 프로세스가 그대로 실행된다. |
| stop | `TTTGate/src/app.ts:18-21`, `TTTGate/src/Sentinel.ts:121-125` | `stop` 모드는 app, sentinel, foreground PID 파일을 읽어 종료한다. |

TTTGate에는 `--foreground` 또는 `--forground` 플래그가 없다. 포그라운드는 `-daemon`이 없을 때의 기본값이다.

### 3.2 데몬화 방식

TTTGate는 `pkg`로 만든 OS별 실행파일을 `dist/bin`에 생성한다. 관련 설정은 `TTTGate/package.json:9-19`에 있다.

데몬 모드 실행 시 흐름은 다음과 같다.

| 단계 | 코드 근거 | 내용 |
| --- | --- | --- |
| 런타임 루트 계산 | `TTTGate/src/Environment.ts:5-22` | Node/npm 실행은 `process.cwd()`, 패키징 실행파일은 실행파일 위치 기준 상위 루트를 계산한다. |
| 부모 boot | `TTTGate/src/Sentinel.ts:130-154` | 기존 app, sentinel, foreground PID를 정리한 뒤 새 app과 sentinel을 시작한다. |
| app child | `TTTGate/src/Sentinel.ts:246-247` | `EXECUTE_MODE=execute`로 자기 자신을 detached child로 실행한다. |
| sentinel child | `TTTGate/src/Sentinel.ts:242-243` | `EXECUTE_MODE=sentinel`, `APP_PID=<app pid>`로 감시 프로세스를 실행한다. |
| child detach | `TTTGate/src/Sentinel.ts:250-267` | `spawn(..., { detached: true, stdio: ['ignore','ignore','ignore'] })` 후 `child.unref()`한다. |
| app 감시 | `TTTGate/src/Sentinel.ts:217-238` | 3초마다 `find-process`로 app PID를 확인하고 죽었으면 재시작한다. |

이 방식은 PM2 없이도 백그라운드 실행과 재시작 감시를 제공한다. 다만 stdout/stderr를 버리기 때문에 초기 QR 출력 같은 사용자 전달 정보가 필요한 BuilderGate에는 그대로 적용할 수 없다.

### 3.3 PID와 종료 처리

TTTGate는 `Environment.path.binDir` 아래 PID 파일을 사용한다.

| PID 파일 | 용도 | 코드 근거 |
| --- | --- | --- |
| `.pid_app` | 실제 서버 또는 클라이언트 app child | `TTTGate/src/Sentinel.ts:157-205` |
| `.pid_sentinel` | app PID를 감시하는 sentinel child | `TTTGate/src/Sentinel.ts:157-205` |
| `.pid_foreground` | 데몬이 아닌 foreground 실행 프로세스 | `TTTGate/src/Sentinel.ts:93-95` |

주의할 점이 있다.

| 문제 | BuilderGate 반영 시 처리 |
| --- | --- |
| PID 파일이 append 방식이다. | 단일 JSON pid state 또는 lock 파일 기반으로 바꿔 stale PID와 PID 재사용을 검증해야 한다. |
| `process.kill(1)` 같은 위험 코드가 있다. | 재사용 금지. 종료 대상은 app, sentinel, foreground PID 또는 process group으로 제한해야 한다. |
| graceful shutdown 보장이 약하다. | BuilderGate는 workspace/CWD flush가 있으므로 SIGTERM 기반 graceful shutdown을 반드시 유지해야 한다. |
| stdout/stderr가 `ignore`다. | QR 출력, 오류 노출, 초기 health 실패 진단을 별도 경로로 보장해야 한다. |

### 3.4 QR/2D 바코드 관련 차이

TTTGate에는 QR 또는 2D 바코드 출력 구현이 없다. 유사한 초기 사용자 전달 정보는 관리자 bootstrap token이며, 값 자체가 아니라 파일 위치를 로그로 남긴다.

BuilderGate는 이와 다르다. TOTP가 활성화되면 서버 시작 중 콘솔 QR을 출력한다. 따라서 새 데몬 런처는 TTTGate와 달리 “분리 전에 QR을 사용자에게 보여준다”는 요구를 별도 기능으로 가져야 한다.

## 4. BuilderGate 현재 구조 분석

### 4.1 실행 엔트리포인트

| 실행 유형 | 코드 근거 | 현재 동작 |
| --- | --- | --- |
| 개발 실행 | `dev.js:8-99` | 서버는 `npm run dev`, 프론트는 Vite로 실행한다. 기본 HTTPS 포트는 2002다. |
| 소스 런처 | `start.bat:5`, `start.sh:6` | `node tools/start-runtime.js`를 호출한다. |
| 서버 본체 | `server/src/index.ts:280-546` | Express, HTTPS, HTTP redirect, WebSocket, static assets, graceful shutdown을 초기화한다. |
| 중지 | `stop.bat:5`, `stop.sh:6`, `stop.js:105-137` | PM2 앱을 찾아 stop/delete한다. |

`tools/start-runtime.js`의 현재 CLI는 `-p/--port`, `--reset-password`, `--bootstrap-allow-ip`, `--help`만 처리한다. `--daemon`, `--foreground`, `--forground`는 없다.

### 4.2 현재 PM2 기반 데몬 구현

PM2 의존은 세 파일에 집중되어 있다.

| 파일 | PM2 역할 |
| --- | --- |
| `tools/start-runtime.js` | PM2 설치 확인, 없으면 글로벌 설치, 기존 PM2 앱 삭제, `pm2 start dist/index.js --name buildergate --cwd server --time` 실행 |
| `stop.js` | `pm2 jlist`로 `buildergate`와 legacy `projectmaster`를 찾아 `pm2 stop`, `pm2 delete` 실행 |
| `tools/build-daemon-exe.js` | `dist/bin/server`에 production dependencies와 `pm2@latest`를 설치하고 `BuilderGate.exe`, `BuilderGateStop.exe`를 pkg로 생성 |

현재 `tools/start-runtime.js`는 PM2 child에 다음 환경변수를 주입한다.

| 환경변수 | 용도 |
| --- | --- |
| `NODE_ENV=production` | 서버가 Vite proxy 대신 정적 assets를 서빙하게 한다. |
| `PORT` | `-p/--port`, config, 기본값 우선순위로 결정된 HTTPS 포트다. |
| `BUILDERGATE_CONFIG_PATH` | EXE 옆 `config.json5`를 서버가 읽게 한다. |
| `BUILDERGATE_BOOTSTRAP_ALLOWED_IPS` | `--bootstrap-allow-ip` 실행 한정 허용 IP 목록이다. |

PM2 제거 후에도 이 환경변수 계약은 유지해야 한다.

### 4.3 빌드 산출물 구조

현재 루트 `package.json`의 `build:daemon-exe`는 `tools/build-daemon-exe.js`를 실행한다.

현재 기본 출력 위치는 `dist/bin`이다.

| 산출물 | 현재 의미 |
| --- | --- |
| `BuilderGate.exe` | PM2를 통해 백그라운드 서버를 시작하는 런처 |
| `BuilderGateStop.exe` | PM2 앱을 중지하고 제거하는 stop 유틸리티 |
| `server/` | 서버 `dist`, package files, production dependencies, 로컬 PM2, 번들 Node runtime |
| `config.json5` | EXE와 같은 위치의 런타임 설정 파일 |
| `config.json5.example` | 설정 예시 |
| `README.md` | 배포본 실행 문서 |

PM2 제거 시 `BuilderGate.exe`와 `BuilderGateStop.exe` 이름은 유지할 수 있지만 내부 의미는 바뀌어야 한다.

### 4.4 TOTP QR 출력 흐름

BuilderGate의 2D 바코드는 현재 TOTP QR이다.

| 단계 | 코드 근거 | 내용 |
| --- | --- | --- |
| 서버 시작 시 TOTP 초기화 | `server/src/index.ts:318-320` | `config.twoFactor.enabled`이면 `applyTwoFactorRuntime(config)`를 호출한다. |
| TOTPService 생성 | `server/src/services/twoFactorRuntime.ts:31-35` | `TOTPService`를 만들고 `initialize()`를 호출한다. |
| secret 생성/로드 | `server/src/services/TOTPService.ts:54-69` | secret 파일이 없으면 생성하고, 있으면 로드한다. 두 경우 모두 QR을 출력한다. |
| 콘솔 QR 출력 | `server/src/services/TOTPService.ts:114-125` | `qrcode-terminal`로 콘솔 QR과 수동 입력 키를 출력한다. |
| 브라우저 QR API | `server/src/services/TOTPService.ts:132-140` | `qrcode`로 data URL을 만들어 Settings 화면에 제공한다. |

중요한 경로 문제가 있다. `TOTPService`의 기본 secret 위치는 `path.join(process.cwd(), 'data', 'totp.secret')`다. 현재 PM2는 `--cwd server`로 서버를 띄우기 때문에 secret은 `dist/bin/server/data/totp.secret`에 생긴다. 새 native daemon도 서버 child의 `cwd`를 `SERVER_DIR`로 유지하거나, TOTP secret 위치를 명시적으로 고정해야 한다.

## 5. 주요 갭 분석

| 갭 | 현재 | 목표 |
| --- | --- | --- |
| 데몬 기본값 | `BuilderGate.exe`는 PM2 백그라운드 시작을 수행한다. | 빌드된 실행파일은 PM2 없이 기본 데몬 모드로 동작한다. |
| 포그라운드 플래그 | 없음 | `--foreground` 또는 요청 원문 `--forground`로 포그라운드 실행한다. |
| QR 출력 | 서버 child stdout에 의존한다. PM2 로그로 흘러가면 실행 콘솔에 직접 보장되지 않는다. | 데몬 모드에서도 사용자에게 QR이 출력된 뒤 parent가 종료되어야 한다. |
| 프로세스 감시 | PM2가 재시작과 상태 관리를 담당한다. | native sentinel 또는 동등한 watchdog이 필요하다. |

| graceful shutdown | PM2가 SIGINT/SIGTERM 전달 | stop 유틸이 서버의 `setupGracefulShutdown()`을 타도록 SIGTERM 또는 동등 신호를 전달해야 한다. |

## 6. PRD에 반영할 목표 동작 제안

### 6.1 실행 모드 계약

| 명령 | 목표 동작 |
| --- | --- |
| `BuilderGate.exe` | 기본 데몬 모드. QR 출력이 필요한 경우 콘솔에 출력한 뒤 서버를 백그라운드에서 계속 실행하고 parent는 종료한다. |
| `BuilderGate.exe --foreground` | 서버를 현재 콘솔에서 직접 실행한다. stdout/stderr와 QR 출력은 그대로 콘솔에 보인다. |
| `BuilderGate.exe --forground` | 결정 필요. 요청 원문을 반영하려면 alias로 허용하되 canonical은 `--foreground`로 문서화하는 것이 안전하다. |
| `BuilderGate.exe -p 2002` | 기존처럼 HTTPS 포트를 override한다. 데몬과 foreground 모두 동일해야 한다. |
| `BuilderGate.exe --reset-password` | 기존처럼 실행 전 `config.json5`의 `auth.password`만 비운다. |
| `BuilderGate.exe --bootstrap-allow-ip <ip[,ip]>` | 기존처럼 해당 실행에만 bootstrap allowlist를 환경변수로 전달한다. |
| `BuilderGateStop.exe` | PM2 없이 pid state를 읽어 app과 sentinel을 graceful stop한다. |

### 6.2 QR 출력 보장 계약

데몬 모드에서 QR 출력은 반드시 detach 이전 또는 parent가 아직 콘솔을 붙잡고 있는 동안 완료되어야 한다.

PRD 후보는 두 가지다.

| 후보 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| Preflight QR 단계 | parent가 서버와 동일한 config, cwd, crypto 경로로 TOTP secret을 초기화하고 QR을 출력한 뒤 child를 detached로 시작한다. | QR 출력이 확실하고 child stdout을 버릴 수 있다. | 런처가 서버 TOTP 로직을 재사용해야 해서 CJS/ESM/pkg 경계 설계가 필요하다. |
| Child handshake 단계 | parent가 server child를 pipe로 시작해 QR/ready 이벤트를 읽고 콘솔에 미러링한 뒤 parent가 종료한다. | 서버의 기존 초기화 로직을 그대로 사용한다. | parent 종료 뒤 child stdout pipe 처리와 detach 안정성이 까다롭다. 로그/파이프 단절 위험이 있다. |

권장안은 Preflight QR 단계다. 단, TOTP secret 경로가 현재 `process.cwd()/data/totp.secret`에 의존하므로 preflight와 server child가 동일한 cwd 또는 동일한 secret path를 사용하도록 먼저 고정해야 한다.

### 6.3 데몬 프로세스 구조

PM2 대체 수준을 맞추려면 다음 native 구조가 필요하다.

| 프로세스 | 역할 |
| --- | --- |
| launcher parent | CLI 파싱, config path 결정, reset-password 처리, QR preflight, 기존 pid 정리, app/sentinel 시작 후 종료 |
| app child | `server/dist/index.js`를 `cwd=server`, `NODE_ENV=production`, `PORT`, `BUILDERGATE_CONFIG_PATH`로 실행 |
| sentinel child | app PID를 감시하고 비정상 종료 시 app child를 재시작 |
| stop utility | pid state를 읽고 sentinel을 먼저 중지한 뒤 app에 SIGTERM을 보내 graceful shutdown을 유도 |

sentinel이 필수인지 PRD에서 명시해야 한다. PM2의 “프로세스 모니터링”을 대체한다는 관점에서는 sentinel을 포함하는 것이 맞다.

| `server/src/services/twoFactorRuntime.ts` | 런처 preflight와 서버 runtime의 TOTP 초기화 중복 방지 정책 검토. |
| `server/src/utils/config.ts` | EXE 옆 config 경로 정책은 유지. preflight에서도 동일한 config loader를 사용할지 결정. |
| `README.md` | PM2 문구 삭제, 기본 데몬/`--foreground`/중지 방식 갱신. |
| `docs/srs` 또는 신규 PRD | 기본 데몬 모드, QR 출력, PM2 제거, pid/graceful shutdown 요구사항 추가. |

## 8. 설계 리스크와 PRD 결정 필요사항

| 항목 | 리스크 | PRD 결정 |
| --- | --- | --- |
| `--forground` 오탈자 | 요청 원문대로 구현하면 표준 CLI와 어긋난다. 표준 철자만 구현하면 요청과 다를 수 있다. | `--foreground` canonical, `--forground` alias 허용 여부 결정 |
| QR preflight 중복 초기화 | parent와 app child가 모두 TOTP initialize를 수행하면 QR이 두 번 출력되거나 secret 파일 race가 생길 수 있다. | parent preflight 후 child에서는 콘솔 QR 억제 옵션을 둘지 결정 |
| TOTP secret 위치 | 현재 `process.cwd()/data/totp.secret`라서 cwd가 바뀌면 secret 위치가 바뀐다. | server child cwd를 `server/`로 고정하거나 secret path를 config 기준으로 명시 |
| stale PID | PID 재사용으로 엉뚱한 프로세스를 종료할 수 있다. | pid state에 pid, executable, argv hash, cwd, start time을 저장하고 검증 |
| Windows 종료 | POSIX signal과 Windows process tree 종료 방식이 다르다. | Windows는 app PID와 child tree 종료 정책을 별도 정의 |
| sentinel 재시작 루프 | 잘못된 config로 app이 즉시 죽으면 무한 재시작할 수 있다. | backoff, max restart, fatal config error 감지 정책 필요 |
| 로그 경로 | daemon stdout/stderr를 버리면 진단성이 떨어진다. | daemon child stdout/stderr를 `logs/buildergate-daemon.log` 같은 파일로 redirect |
| source 실행 정책 | 요구는 “빌드된 exe 또는 OS별 실행파일” 중심이다. | `node tools/start-runtime.js`도 기본 daemon으로 맞출지, 개발 편의를 위해 foreground로 둘지 결정 |

## 9. 테스트 전략 제안

모든 버그 수정과 기능 추가에는 회귀 테스트가 필요하므로 PRD에는 다음 테스트를 포함해야 한다.

| 테스트 | 검증 내용 |
| --- | --- |
| CLI parser unit | `--foreground`, `--forground` alias 여부, `-p`, `--reset-password`, `--bootstrap-allow-ip` 조합 파싱 |
| daemon launch smoke | `BuilderGate.exe` 실행 후 parent가 종료되고 background app이 `/health`를 제공 |
| foreground smoke | `BuilderGate.exe --foreground`가 현재 콘솔에서 서버를 실행하고 Ctrl+C 또는 SIGTERM에 정상 종료 |
| QR daemon smoke | TOTP enabled 상태에서 기본 daemon 실행 시 콘솔에 QR/수동 키가 출력된 뒤 background 실행 |
| QR no-duplicate | daemon preflight와 app child에서 secret race 또는 QR 중복 출력이 없는지 확인 |
| config path | EXE 옆 `config.json5`가 `BUILDERGATE_CONFIG_PATH`로 서버에 전달되는지 확인 |
| TOTP secret path | 기존 `server/data/totp.secret` 호환 또는 새 명시 경로가 유지되는지 확인 |
| stop smoke | `BuilderGateStop.exe`가 PM2 없이 app과 sentinel을 종료하고 `/health`가 닫히는지 확인 |
| graceful shutdown | stop 시 workspace/CWD flush 로그 또는 상태 파일 반영 확인 |
| build output | `npm run build:daemon-exe` 후 `dist/bin`에 PM2 없는 런타임과 실행파일 생성 확인 |

## 11. 최종 권장 방향

BuilderGate는 TTTGate의 self-reexec와 sentinel 패턴을 참고하되, 다음 원칙으로 새 데몬 런처를 설계하는 것이 적합하다.

| 원칙 | 이유 |
| --- | --- |
| PM2는 완전히 제거한다. | 요구사항이 PM2 기반 구현 제거와 대체를 명시한다. |
| 기본 데몬, 명시 foreground로 전환한다. | 빌드 실행파일의 사용 경험을 단순화한다. |
| QR 출력은 daemon detach 전 parent 책임으로 보장한다. | detached child의 stdout을 기대하면 QR이 보이지 않는 문제가 재발한다. |
| 서버 child cwd와 config path를 명시한다. | 현재 TOTP secret과 static asset 경로가 cwd와 config env에 민감하다. |
| PID state는 append pidfile이 아니라 검증 가능한 상태 파일로 둔다. | stale PID와 무관 프로세스 kill 위험을 줄인다. |
| stop은 graceful shutdown을 우선한다. | BuilderGate는 workspace/CWD flush가 기능 요구사항에 포함된다. |

이 분석을 바탕으로 다음 단계에서는 PRD에 실행 모드 계약, QR 출력 순서, native watchdog 범위, PID state 데이터 모델, 중지/로그/테스트 요구사항을 명확히 정의해야 한다.
