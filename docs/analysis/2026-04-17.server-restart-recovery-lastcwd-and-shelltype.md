# Server Restart Recovery LastCwd And ShellType

작성일: 2026-04-17
상태: 분석만 기록, 구현 보류

## 문제

사용자 관찰:

- 서버를 다시 기동하면 세션 개수는 복구된다.
- 그러나 마지막 디렉터리 위치는 기대대로 복구되지 않는다.
- 추가 질문: 세션 형태도 복구되는가.

## 결론

현재 코드 기준 결론은 다음과 같다.

1. 세션 개수는 복구된다.
2. 셸 형태(`shellType`)는 복구된다.
3. 마지막 디렉터리(`lastCwd`)도 복구 대상으로 설계되어 있다.
4. 다만 `lastCwd`는 "마지막으로 저장된 값"만 복구되므로, 실제 체감상 복구되지 않을 수 있다.

즉, 현재 현상은

- "복구 로직이 전혀 없다"가 아니라
- "복구 시도는 하지만 최신 값이 디스크에 안 남았거나, 재기동 시 spawn 가능한 경로로 해석되지 못했다"

로 보는 편이 맞다.

## 근거

### 1. 서버 시작 시 세션 개수는 탭 메타데이터 기준으로 복구된다

서버 시작 경로:

- [server/src/index.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/index.ts:306)
  - `workspaceService.initialize()`
  - `workspaceService.checkOrphanTabs()`

복구 핵심:

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:449)
  - 저장된 각 `tab`에 대해
  - `sessionManager.hasSession(tab.sessionId)`가 false이면 orphan로 간주
  - `createSession(tab.name, tab.shellType, tab.lastCwd)`로 새 세션 생성
  - 새 `sessionId`로 탭 메타데이터 갱신

따라서 디스크에 저장된 `tabs.length` 기준으로 세션 수는 다시 만들어진다.

### 2. 셸 형태는 복구된다

탭 메타데이터에는 `shellType`가 저장된다.

- [server/src/types/workspace.types.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/types/workspace.types.ts:25)

탭 생성 시에도 저장된다.

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:223)

재시작 복구와 탭 재시작 모두 같은 값을 사용한다.

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:309)
- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:456)

따라서 `powershell`, `wsl`, `bash`, `cmd`, `auto` 같은 셸 종류는 복구 대상이 맞다.

## lastCwd 복구가 체감상 실패할 수 있는 이유

### 1. `lastCwd`는 즉시 디스크에 확정되지 않는다

`lastCwd` 저장 경로:

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:37)
  - `SessionManager.onCwdChange(...)`
  - 세션 CWD가 바뀌면 `tab.lastCwd = cwd`
  - 이후 `save()` 호출

문제는 `save()`가 기본적으로 debounce 저장이라는 점이다.

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:382)
- 기본 debounce: 5초
  - [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:33)

즉:

- 디렉터리 이동 직후
- 5초 안에 서버가 죽거나 재부팅되면
- 최신 `lastCwd`가 `workspaces.json`에 아직 안 써졌을 수 있다.

### 2. crash / reboot 시에는 마지막 CWD snapshot이 유실될 수 있다

보완 장치가 있긴 하다.

- 30초 주기 snapshot
  - [server/src/index.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/index.ts:315)
- graceful shutdown 시 최종 snapshot + flush
  - [server/src/index.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/index.ts:457)

하지만 이건 다음 조건에서만 완전하다.

- 30초 주기 snapshot 이전에 죽지 않을 것
- 혹은 SIGINT/SIGTERM 기반 graceful shutdown이 실행될 것

갑작스런 재부팅, 강제 종료, 프로세스 crash라면 최신 `lastCwd`는 유실될 수 있다.

### 3. 저장된 경로가 재기동 시 spawn 가능한 경로가 아닐 수 있다

세션 복구는 결국 `createSession(..., cwd)`에 저장된 `lastCwd`를 넘긴다.

- [server/src/services/WorkspaceService.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/WorkspaceService.ts:456)

하지만 실제 spawn 직전에는 경로 정규화/검증을 거친다.

- [server/src/services/SessionManager.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/SessionManager.ts:828)

여기서 다음 경우 fallback이 일어난다.

1. 경로가 현재 파일시스템에 존재하지 않음
2. Windows에서 `/home/...` 같은 Linux 경로인데 `/mnt/x/...`로 변환할 수 없음
3. WSL/PowerShell 혼합 경로가 현재 셸/호스트 기준으로 유효하지 않음

이 경우 최종적으로 홈 디렉터리로 떨어진다.

따라서 디스크에는 `lastCwd`가 남아 있어도, 재기동 후에는 다른 디렉터리로 시작할 수 있다.

### 4. `workspaces.json`에 저장된 값은 "최근 저장본"이지 "현재 메모리 값"이 아니다

현재 `server/data/workspaces.json`를 보면 각 탭에 `lastCwd`와 `shellType`가 실제로 저장되어 있다.

- [server/data/workspaces.json](/C:/Work/git/_Snoworca/ProjectMaster/server/data/workspaces.json)

즉 데이터 모델 자체는 준비되어 있다.

하지만 서버 재기동 복구가 읽는 값은 이 파일에 마지막으로 flush된 값이다.

그래서 사용자가 기대하는 "방금 직전 위치"와 실제 복구되는 "마지막 디스크 저장 위치" 사이에 차이가 생길 수 있다.

## 세션 형태의 의미별 답변

`세션 형태`가 무엇을 뜻하는지에 따라 답이 조금 다르다.

### 1. 셸 종류를 뜻하면

복구된다.

- `shellType` 저장됨
- orphan recovery 시 그대로 재사용됨

### 2. 워크스페이스 view mode (`tab` / `grid`)를 뜻하면

복구된다.

`workspaces.json`에 저장된 값이 그대로 로드된다.

- [server/src/types/workspace.types.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/types/workspace.types.ts:10)

### 3. 터미널 내부 실행 상태(예: 이미 실행 중이던 장기 프로세스)를 뜻하면

복구되지 않는다.

현재 복구는 "새 PTY 세션 재생성"이지, 프로세스 체크포인트/복원이 아니다.

## 현재 판정

이번 이슈의 판정은 다음과 같다.

- 세션 수 복구: 정상
- 셸 타입 복구: 정상
- `lastCwd` 복구: best-effort

실사용에서 `lastCwd`가 안 맞는 것은 충분히 가능한 현재 동작이다.

## 나중에 다룰 때 볼 포인트

1. `onCwdChange` 저장을 debounce가 아니라 더 공격적으로 flush할지
2. 서버 종료 시점 외에도 탭 전환/워크스페이스 전환 때 `snapshotAllCwds()`를 호출할지
3. `lastCwd`가 spawn fallback 되었을 때 원인 로그를 더 명확히 남길지
4. WSL/Linux path 복구 정책을 셸 종류별로 더 정밀하게 나눌지
