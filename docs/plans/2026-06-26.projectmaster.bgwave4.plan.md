# BuilderGate Wave 4 Bounded Output Queues And WebSocket Backpressure Plan

## §1 Overview

### 1.1 Goal

Implement `FR-BGSTAB-012` by bounding server output memory growth in the headless terminal path and WebSocket send path. Wave4 must remove unbounded pending headless output arrays, add byte/chunk caps, and route server WebSocket sends through a safe-send policy that respects `bufferedAmount`, send callback errors, and per-client queue limits.

### 1.2 Scope

- Add reusable bounded byte deque and headless output queue utilities.
- Wire Wave0 `resourceLimits.headless`, `resourceLimits.ws`, `stabilityModes.headlessQueueMode`, and `stabilityModes.wsSendMode` into `SessionManager` and `WsRouter`.
- Replace `pendingOutputChunks` hot/degrade paths with a bounded headless queue and drain pump.
- Route server WebSocket sends through one safe-send policy and per-client output/control queues.
- Preserve replay and screen-repair protocol queues as separate protocol state.
- Add telemetry for headless pending output and WebSocket transport backpressure.
- Add regression tests for caps, ordering, overflow degrade, slow-client closure, send errors, and replay/repair behavior.

### 1.3 Out Of Scope

- Split WebSocket transport changes.
- Frontend runtime residency or UI visual changes.
- Shutdown/daemon stop policy changes, reserved for Wave5.
- Broad process-name killing or process cleanup mode changes.
- Defaulting `headlessQueueMode` to `bounded` or `wsSendMode` to `safe-send-enforce` before Wave4 verification passes.

### 1.4 Assumptions

- Wave0 config contracts are present and validated.
- Wave1-Wave3 process cleanup and lifecycle tests must remain green.
- Slow clients are isolated at the WebSocket client level and must not terminate sessions.
- `safe-send-observe` can collect telemetry without changing direct-send behavior; enforce behavior is gated by `wsSendMode`.
