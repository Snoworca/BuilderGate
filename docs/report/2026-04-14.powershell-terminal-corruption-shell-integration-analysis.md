### Medium — `convertEol: true`가 raw PTY 스트림에 과하게 개입하고 있습니다

- 파일:
  - [frontend/src/components/Terminal/TerminalView.tsx](C:/Work/git/_Snoworca/ProjectMaster/frontend/src/components/Terminal/TerminalView.tsx:416)
  - [frontend/node_modules/@xterm/xterm/src/common/InputHandler.ts](C:/Work/git/_Snoworca/ProjectMaster/frontend/node_modules/@xterm/xterm/src/common/InputHandler.ts:716)
- 관찰:
  - 현재 xterm은 `convertEol: true`로 생성됩니다.
  - xterm 내부 구현상 이 옵션은 `LF`를 받을 때 커서를 강제로 열 시작으로 돌립니다.
- 왜 이 현상과 맞는가:
  - PowerShell/ConPTY는 이미 `CR/LF`와 cursor control을 포함한 raw terminal stream을 보냅니다.
  - 여기에 프런트가 한 번 더 EOL 변환을 적용하면 PowerShell이 의도한 cursor 위치와 xterm 내부 cursor 위치가 어긋날 수 있습니다.
  - 특히 PSReadLine redraw, 긴 에러 메시지, 한글이 섞인 wrap 상황에서 손상이 더 잘 보입니다.
- 가능성:
  - 이 옵션 하나가 유일 원인이라고 단정하긴 이르지만, High 1번과 결합되면 손상 확률을 크게 높입니다.
- 제안 수정:
  - live PTY 스트림에는 `convertEol: false`가 더 안전합니다.
  - 만약 특정 fallback payload만 LF 정규화가 필요하면 그건 snapshot/fallback 경로에서만 별도로 처리해야 합니다.

## 결론

PowerShell shell integration 관점에서 보면, 현재 코드에서 화면을 직접 망가뜨릴 만한 것은 `OSC 주입`이 아니라 `ConPTY와 xterm 설정의 계약 불일치`입니다.

정리하면:

1. 서버는 ConPTY를 사용합니다.
2. 프런트 xterm은 Windows PTY라는 사실을 모릅니다.
3. 추가로 `convertEol: true`가 raw stream 해석에 개입합니다.
4. PowerShell/PSReadLine의 redraw가 여기에 걸리면서 줄 wrap/cursor 위치가 틀어지고 출력이 겹칩니다.

즉, shell integration의 직접 원인은 약하고, PowerShell 전용 PTY 계약을 프런트가 모르는 것이 근본 원인에 가깝습니다.

## 권장 수정 순서

1. `TerminalView`에 `windowsPty` 설정을 넣고, 서버가 `conpty/winpty + build number`를 프런트에 전달하도록 계약을 추가합니다.
2. live PTY 경로에서 `convertEol`을 끄고 증상 재현 여부를 다시 확인합니다.
3. 그 다음에 PowerShell prompt hook의 `Out-File` 기반 CWD 추적을 경량화합니다.
4. 마지막으로 PowerShell 전용 회귀 E2E를 추가합니다.

## 세 갈래 조사 요약

### 조사 축 1 — Shell Integration / Prompt Hook

- PowerShell은 OSC 133을 안 씁니다.
- prompt override는 존재하지만 화면에 직접 ANSI/OSC를 주입하지 않습니다.
- 따라서 주범은 아닙니다.

### 조사 축 2 — PTY / xterm 계약

- 이 축이 가장 강합니다.
- ConPTY를 쓰는데 xterm `windowsPty`가 비어 있습니다.
- PowerShell corruption과 가장 잘 맞는 설명입니다.

### 조사 축 3 — CWD 추적 / WS / 리마운트 경로

- `cwd` 이벤트는 metadata 갱신에 그칩니다.
- 현재 memoization 구조상 직접적인 터미널 재생성 트리거로 보이지 않습니다.

## 최종 판단

- 1순위 원인: `ConPTY + missing windowsPty contract`
- 2순위 증폭 요인: `convertEol: true`
- 3순위 부수 요인: PowerShell `prompt` hook의 무거운 `Out-File`
- 가능성 낮음: OSC integration, CWD watcher 자체, `cwd` WS 이벤트 자체
