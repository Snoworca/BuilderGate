# 창 제어형 팀장-팀원 MCP/Hook 구조 구현 가능성 보고서

작성일: 2026-05-19 KST  
대상 프로젝트: `C:\Work\git\_Snoworca\ProjectMaster`  
요청: 창 하나가 팀장이 되고 다른 창의 코딩 에이전트들이 팀원처럼 협업하는 구조를 MCP 또는 hook 기반으로 구현하고, 이곳의 도구로 붙일 수 있는지 조사

## 결론

구현 가능하다. 다만 "한 브라우저 창이 다른 브라우저 창을 직접 조작"하는 방식으로 설계하면 현재 구조와 충돌하고 보안 위험이 커진다. 이 프로젝트에는 이미 다중 workspace/tab/session, 단일 WebSocket, headless terminal snapshot, debug capture, session status 판정 파이프라인이 있으므로, 권장 구조는 다음처럼 분리하는 것이다.

```text
팀장 창 / 팀장 에이전트
  -> MCP tool 호출
  -> BuilderGate Team Control Plane
  -> SessionInput/Assignment/Approval/Audit 서비스
  -> 기존 SessionManager / WsRouter / WorkspaceService
  -> 팀원 창의 코딩 에이전트 터미널 세션

팀원 에이전트 hook
  -> buildergate-hook CLI
  -> HookSignalService
  -> SessionManager external activity merge
  -> 기존 idle/running WS status 유지
```

핵심 판단은 다음이다.

- `MCP`는 팀장이 세션 목록 조회, 작업 배정, 승인 요청, 프롬프트 전달, 상태/감사 로그 조회를 수행하는 도구면으로 둔다.
- `hook`은 Codex/Claude/Hermes 같은 팀원 코딩 에이전트의 생명주기와 상태 신호를 서버에 제출하는 보조 신호로 둔다.
- 최종 상태 원본은 계속 `SessionManager`여야 한다.
- 기존 `status: idle | running` WebSocket 계약은 1차 구현에서 깨지 않는 편이 안전하다.
- 자동 입력/명령 실행 도구는 human approval, 감사 로그, 권한 scope, newline 안전 정책 없이는 열면 안 된다.

즉, "MCP 서버 추가만으로 창들이 알아서 팀처럼 협업"하는 것은 부족하다. `TeamCoordinationService`, `HookSignalService`, `AuditLogService`, 입력 권한 정책, 승인 게이트를 함께 추가해야 한다.

## 이곳의 도구로 붙일 수 있는가

조건부로 가능하다.

- Codex CLI/IDE 쪽은 MCP 서버를 설정 파일 또는 CLI로 추가하는 흐름이 공식 문서에 있다. OpenAI Docs MCP 문서는 Codex에서 `codex mcp add ... --url ...` 형태로 MCP 서버를 연결할 수 있다고 설명한다.
- Responses API는 remote MCP 서버를 `type: "mcp"` tool로 사용할 수 있다.
- ChatGPT custom MCP app은 현재 로컬 서버 직접 연결이 아니라 remote server 중심이다. OpenAI Help 문서는 2026-05-19 확인 기준으로 local MCP server 연결은 지원하지 않는다고 안내한다.

따라서 이 프로젝트에서 만든 MCP를 "현재 실행 중인 이 대화 세션에 즉시 동적으로 주입"하는 것은 보장할 수 없다. 그러나 BuilderGate가 MCP 서버를 제공하고, 사용자가 Codex/IDE/지원되는 ChatGPT connector 설정에 등록하면 도구로 붙이는 구조는 가능하다. 로컬 전용 제어를 원하면 Codex/IDE용 local/HTTP MCP가 현실적이고, ChatGPT connector까지 고려하면 remote HTTPS endpoint와 별도 인증/승인 모델이 필요하다.

공식 확인 출처:

- OpenAI Docs MCP: https://developers.openai.com/learn/docs-mcp
- OpenAI MCP/connectors guide: https://developers.openai.com/api/docs/guides/tools-connectors-mcp
- OpenAI ChatGPT developer mode FAQ: https://help.openai.com/en/articles/12584461
- MCP transport spec: https://modelcontextprotocol.io/specification/2025-06-18/basic/transports

## 현재 프로젝트 근거

### 백엔드

현재 백엔드는 구현 기반을 이미 상당 부분 갖고 있다.

- 서버 핵심 서비스는 `SessionManager`, `WorkspaceService`, `AuthService`, `FileService`, `WsRouter`로 분리되어 있다. 관련 설명은 `docs/wiki/03.backend-architecture.md`에 정리되어 있다.
- REST는 로그인/설정/workspace/tab/file/debug capture를 맡고, WebSocket은 터미널 입출력, snapshot, status, cwd를 맡는다. 관련 설명은 `docs/wiki/05.realtime-protocol-and-state.md`에 있다.
- `SessionManager.writeInput()`은 PTY 입력, AI TUI foreground 상태, echo tracker, debug capture, status 전이를 함께 처리한다.
- `WsRouter.handleInput()`은 `input` 메시지를 검증한 뒤 `SessionManager.writeInput()`으로 전달한다.
- `WsRouter.routeSessionOutput()`은 동일 세션의 여러 WebSocket 구독자에게 output을 fan-out할 수 있다.
- `WorkspaceTab`은 `sessionId`를 보유하지만 팀장/팀원, role, assignment, controller 개념은 아직 없다.

중요한 제약:

- `AGENTS.md`의 invariant에 따라 Codex/Claude/Hermes 같은 AI TUI에서 사용자 키 입력, local echo, prompt redraw, cursor movement, waiting-for-input repaint는 세션을 `running`으로 바꾸면 안 된다.
- `server/src/types/ws-protocol.ts`와 프론트 복사본의 status 메시지는 `running | idle`에 맞춰져 있다.
- `ForegroundAppDetector.deriveDisplayStatus()`는 `waiting_input`과 `repaint_only`를 `idle`로 해석한다.
- 기존 입력 안정성은 replay pending, snapshot ack, transport outbox, sequencer와 연결되어 있으므로 MCP가 `SessionManager.writeInput()`을 직접 호출해 기존 큐 정책을 우회하면 위험하다.

### 프론트엔드

프론트엔드는 "여러 세션을 팀원처럼 나란히 관리"하는 UI 기반이 있다.

- `Workspace`, `WorkspaceTab`, `GridLayout` 모델이 이미 존재한다.
- `WebSocketContext`가 단일 WebSocket 연결, 세션 구독, workspace event handler를 관리한다.
- `useWorkspaceManager`는 workspace/tab/grid 변경을 WebSocket 브로드캐스트로 반영한다.
- `TerminalRuntimeLayer`는 disconnected가 아닌 tab에 대해 terminal runtime을 유지한다.
- `TerminalView -> TerminalContainer -> WebSocket` 경로가 input gate, sequencer, debug capture를 탄다.

중요한 제약:

- 현재 `activeTabId`와 `viewMode`는 workspace 상태에 포함되어 있어 한 창의 조작이 다른 창 UI까지 움직일 수 있다.
- "팀장 창이 팀원 세션에 명령을 보낸다"와 "팀장 창이 팀원 브라우저 창의 화면 상태를 바꾼다"는 다른 기능이다.
- 숨겨진/parked terminal에 대한 `TerminalHandle.sendInput()`은 실패하거나 silent no-op처럼 보일 수 있으므로, 서버 중심 입력 경로와 명시적 실패 telemetry가 필요하다.
- UI 변경은 프로젝트 규칙상 최소화해야 하며, 새 대형 패널보다 기존 workspace/tab/grid 표면에 role/status/action을 얹는 방향이 안전하다.

### 기존 연구 재사용

`docs/research/2026-05-10.agent-hook-status-signal-research.md`가 이번 기능의 상태 신호 부분에 직접 관련된다. 이 문서의 핵심 결론은 다음과 같다.

- hook CLI는 기존 PTY/OSC/semantic detector를 대체하면 안 된다.
- BuilderGate 상태 원본은 `SessionManager` 하나로 유지한다.
- hook 신호는 `SessionManager` 내부 derived state에 병합되는 외부 보강 신호로 다룬다.
- `UserPromptSubmit`은 `running` 확정 신호가 아니라 turn-pending 후보로 둔다.
- `PreToolUse`, semantic output, explicit running signal만 running 근거로 신뢰한다.
- HMAC, nonce, sequence, TTL, rate limit이 필요하다.

`docs/research/cmux-research.md`도 참고 가치가 있다. CMUX 계열은 socket/CLI API로 여러 AI 코딩 에이전트를 다루는 패턴을 제공하며, BuilderGate는 웹 기반 원격 접근과 파일/작업 관리 쪽에서 차별화할 수 있다.

## 권장 아키텍처

### 1. TeamCoordinationService

새 도메인 서비스로 추가한다.

책임:

- 팀장/팀원 role 관리
- 세션 ownership과 controllability 관리
- 작업 assignment 생성/갱신/완료/취소
- 승인 상태 관리
- MCP tool 요청을 내부 command로 정규화
- audit log 기록

초기 데이터 모델 예시:

```ts
type TeamRole = 'lead' | 'worker' | 'observer';

interface AgentSessionRef {
  sessionId: string;
  workspaceId: string;
  tabId: string;
  role: TeamRole;
  agentKind?: 'codex' | 'claude-code' | 'hermes' | 'shell' | 'unknown';
  controllable: boolean;
}

interface Assignment {
  id: string;
  leadSessionId: string;
  workerSessionId: string;
  status: 'draft' | 'pending_approval' | 'sent' | 'running' | 'blocked' | 'done' | 'cancelled' | 'failed';
  promptHash: string;
  promptPreview: string;
  createdAt: string;
  updatedAt: string;
}
```

### 2. HookSignalService

기존 연구 문서의 hook schema와 merge policy를 구현한다.

책임:

- `buildergate-hook` CLI가 보낸 표준 신호 검증
- per-session HMAC, nonce, timestamp, sequence, TTL 검증
- source별 adapter: Codex, Claude Code, Hermes
- `SessionManager.applyExternalActivitySignal()`로 병합
- debug capture/audit log 기록

표시 정책:

- `tool-start`, `agent-start`, `subagent-start`는 running scope를 연다.
- `tool-stop`, `agent-stop`, `subagent-stop`, TTL 만료는 scope를 닫는다.
- `permission-prompt`, `waiting-input`, `repaint-only`는 display status를 `running`으로 만들지 않는다.
- hook이 없거나 실패하면 기존 detector 경로가 정상 동작해야 한다.

### 3. SessionInputService

MCP/팀장 창/프론트 UI가 모두 같은 입력 안정성 정책을 쓰도록 별도 서비스로 빼는 것이 안전하다.

책임:

- target session 존재 여부 검증
- controllable/approval/scope 검증
- newline/Enter 포함 payload 정책 적용
- replay pending, snapshot ack, input sequence, debug capture 정책과 통합
- 성공/거부 결과를 명시적으로 반환

주의:

- 1차 구현에서 `SessionManager.writeInput()` 직접 호출은 최소화한다.
- 기존 `WsRouter`의 replay queue와 충돌하지 않게 공용 입력 큐 또는 내부 server-origin input 경로를 설계해야 한다.

### 4. MCP Server

권장 transport:

- Codex/IDE 로컬 사용: stdio 또는 localhost Streamable HTTP
- ChatGPT/API remote 사용: public/remote HTTPS Streamable HTTP

권장 tool 목록:

| Tool | 성격 | 기본 승인 |
|---|---|---|
| `list_agent_sessions` | read-only | 불필요 |
| `get_agent_status` | read-only | 불필요 |
| `get_assignment` | read-only | 불필요 |
| `create_assignment` | additive | 필요할 수 있음 |
| `request_worker_attention` | additive | 필요할 수 있음 |
| `preview_agent_prompt` | read-only/safe transform | 불필요 |
| `deliver_agent_prompt` | write/destructive 가능 | 필요 |
| `cancel_assignment` | write | 필요 |
| `read_audit_log` | read-only 민감 | scope 필요 |

의도적으로 1차에서 제외할 도구:

- 임의 shell command 실행
- 파일 삭제/이동
- 세션 강제 종료/재시작
- raw key stroke stream
- newline 포함 payload 자동 제출
- 브라우저 JWT 재사용

### 5. AuditLogService

모든 제어 작업은 추적 가능해야 한다.

기록 대상:

- MCP tool call request/result
- actor, target session, assignment id
- 승인/거부 이유
- hook accepted/rejected/merged
- prompt hash, byte length, safe preview
- destructive action approval id

기본 저장 금지:

- 브라우저 JWT
- hook secret
- raw OAuth token
- raw prompt 전문
- raw hook payload 전문

## 필수 보안 정책

- MCP 인증과 브라우저 JWT 인증을 분리한다.
- remote MCP는 Bearer/OAuth, audience, scope, expiry를 검증한다.
- local hook은 per-session HMAC secret을 사용한다.
- query string token은 금지한다.
- localhost 전용 endpoint는 loopback bind와 Origin 검증을 둔다.
- destructive tool은 human approval 또는 명시적 policy allowlist 없이는 실행하지 않는다.
- tool schema는 엄격하게 닫는다.
- rate limit과 payload size limit을 둔다.
- 실패는 silent fallback하지 말고 tool result, debug capture, audit log에 남긴다.

서버 단위:

- HMAC 정상/변조/만료/replay
- sequence 증가/중복/역순
- TTL 만료로 running scope 정리
- permission prompt는 `idle + attention hint`
- UserPromptSubmit만으로 running 전환 금지

- MCP scope 부족 403

- audit log redaction

프론트 단위:

- `TerminalHandle` stale target 방지
- hidden/parked target 입력 실패가 observable
- command preset/paste 안전 정책 유지
- `activeTabId/viewMode` 공유/로컬 정책에 따른 동기화

E2E:

- 두 창에서 workspace/tab/grid 동기화 확인
- 팀장 창 assignment 생성/전송
- 팀원 terminal에서 프롬프트 수신
- Codex/Claude/Hermes AI TUI 입력 idle 유지
- tool-start hook running, stop/TTL idle
- reconnect/snapshot replay 중 입력 유실/중복 없음

## 주요 리스크

| 리스크 | 영향 | 완화 |
|---|---|---|
| MCP를 raw terminal control로 개방 | 팀장 에이전트가 로컬 운영자 권한을 사실상 획득 | tool allowlist, approval, scope, audit |

| raw prompt/log 저장 | 민감 정보 노출 | hash/preview/redaction |

## 3개 서브에이전트 취합

### 백엔드 담당

판정: 가능. 단, 현재 백엔드에는 팀장/팀원 의미론과 MCP 서버가 없다.

핵심 지적:

- 상태 원천은 `SessionManager`로 유지해야 한다.
- MCP/hook이 WebSocket `status`를 직접 보내면 invariant가 깨질 수 있다.
- 서버 입력 주입은 replay/input queue를 우회하지 않는 별도 입력 경로가 필요하다.
- role, assignment, approval, audit log가 필요하다.

### 프론트엔드 담당

판정: 같은 앱 안에서 여러 세션/에이전트를 관리하는 것은 가능성이 높다. 다른 브라우저 창 자체를 제어하는 의미라면 presence/controller 모델이 필요하다.

핵심 지적:

- workspace/tab/grid 구조는 협업 UI 기반으로 재사용 가능하다.
- `activeTabId/viewMode`가 전역 workspace 상태라 창 간 UX 충돌 위험이 있다.
- hidden/parked terminal 대상 입력은 stale handle과 silent failure 위험이 있다.
- 제어 입력도 sequencer/transport/debug capture를 타야 한다.

### 아키텍처/보안 담당

판정: 조건부 승인.

핵심 지적:

- `MCP = 승인/지시/조회 제어면`, `hook = 생명주기/상태 신호`, `SessionManager = 최종 상태 원본`으로 나눠야 한다.
- 브라우저 JWT 재사용, query token, raw payload 저장, newline 자동 실행 도구는 금지해야 한다.
- HMAC, nonce, sequence, TTL, rate limit, audit log, approval gate가 SRS 수용 기준에 들어가야 한다.

## 최종 권고

1차 목표는 "팀장 창이 팀원 창 화면을 물리적으로 조종"이 아니라 "팀장 에이전트가 MCP tool로 팀원 세션에 작업을 배정하고, 승인된 입력만 서버가 안전 경로로 전달하며, hook으로 팀원 상태를 보강하는 구조"로 잡는 것이 맞다.

이 방식이면 현재 BuilderGate의 강점인 웹 기반 다중 터미널, WebSocket fan-out, headless snapshot, debug capture, workspace persistence를 살릴 수 있다. 반대로 브라우저 창 간 DOM/화면 상태를 직접 제어하는 모델은 구현은 가능해도 active workspace state와 보안 정책을 크게 흔들기 때문에 1차 구현 대상으로는 부적합하다.

구현을 시작하려면 먼저 SpecKiwi/SRS로 요구사항을 추가하고, 그 다음 `TeamCoordinationService + HookSignalService + MCP read-only tools` 순서로 작게 들어가는 것이 안전하다.
