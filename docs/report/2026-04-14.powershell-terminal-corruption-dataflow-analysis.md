# PowerShell Terminal Corruption Dataflow Analysis

## 범위

- 대상 코드만 검토했다.
- 수정은 하지 않았다.
- 관점은 세 갈래로 분리했다.
  - 트랙 A: 서버 PTY 입력/출력 순서와 live output 전달 경로
  - 트랙 B: 프런트 `screen-snapshot` / `restorePending` / xterm write 큐 경로
  - 트랙 C: PowerShell 부트스트랩, 초기 resize, prompt hook 타이밍

## 전제

- 재현은 "새 PowerShell 세션을 연 직후" 또는 "새로고침/재구독 직후"에 빠르게 타이핑하고 Enter 하는 경우라고 가정한다.
- 이 전제가 맞다면 현재 코드에서 가장 그럴듯한 원인은 `steady-state live output duplication` 이 아니라 `interactive-ready barrier 부재` 다.
- 만약 세션이 완전히 안정화된 뒤에도 같은 수준으로 계속 재현된다면, 2차로는 ConPTY/PSReadLine 의 폭 재계산과 line redraw 경로를 추가 추적해야 한다.

가능한 실제 순서는 이렇다.

1. 서버 PTY 는 80x24 로 먼저 PowerShell prompt 를 만든다.
2. 프런트는 아직 `fit/onResize` 를 끝내지 않았다.
3. 사용자가 바로 타이핑하고 Enter 를 친다.
4. PowerShell/PSReadLine 은 carriage return 기반 redraw 를 내보낸다.
5. 그 사이 서버 PTY 가 실제 크기로 resize 되거나 snapshot 복원이 끝난다.
6. redraw 시퀀스가 "옛 화면/옛 폭" 기준 상태와 "새 폭/새 화면" 기준 상태 위에 섞여 보인다.

이 증상은 사용자가 보여준 "문장 중간 삽입", "남은 꼬리 조각", "겹쳐 보이는 줄"과 맞는다. PowerShell 은 입력 중에도 전체 라인을 다시 그리는 편이라, 이런 race 가 Bash 보다 훨씬 눈에 띈다.

제안 수정:

- 터미널에 `bootstrap barrier` 를 둔다.
- 최소 조건이 충족되기 전에는 `term.onData -> onInput` 을 막는다.
  - 조건 1: 최초 `screen-snapshot` 적용 완료
  - 조건 2: 최초 `fit` 결과가 서버 PTY 에 적용 완료
  - 조건 3: 가능하면 최초 prompt 재등장까지 확인
- 더 근본적으로는 세션 생성 API 단계에서 클라이언트 cols/rows 를 함께 전달해 PTY 를 처음부터 올바른 크기로 띄운다.

문제는 이 주입이 다음과 무관하다는 점이다.

- 사용자가 이미 타이핑 중인지
- snapshot 복원이 끝났는지
- 최초 resize 가 끝났는지
- PowerShell 이 prompt 상태인지

즉 새 세션에서 사용자가 빨리 입력하면, 내부 bootstrap 명령과 사용자 명령이 같은 interactive stream 에 섞일 수 있다. 이 경로는 "처음 몇 초 동안"만 심하게 재현되는 증상과 잘 맞는다.

제안 수정:

- 500ms `setTimeout` 주입을 버리고, PowerShell 시작 인자나 초기 bootstrap command 로 prompt hook 을 먼저 적용한 뒤 interactive handoff 하도록 바꾼다.
- 또는 hook 적용 완료 전에는 입력을 잠근다.

## 세 트랙 의견 취합

- 트랙 A 결론: steady-state server output ordering 은 비교적 잘 직렬화되어 있다. 핵심 결함은 output duplication 이 아니라 입력이 너무 일찍 열리는 점이다.
- 트랙 B 결론: `screen-snapshot` 과 `restorePending` 은 recovery 에는 도움이 되지만, interactive-ready barrier 없이 쓰면 PowerShell redraw 와 충돌할 수 있다.
- 트랙 C 결론: PowerShell 은 PSReadLine redraw, 긴 localized error line, delayed prompt hook 때문에 race 가 가장 잘 드러나는 셸이다.

세 트랙은 모두 같은 결론으로 수렴했다.

> 가장 가능성이 큰 근본 원인은 "사용자가 interactive terminal 로 보기 시작한 시점"과 "서버/클라이언트가 실제로 interactive-ready 상태가 된 시점"이 다르다는 점이다.

## 권장 해결 순서

1. 입력 차단 기반 bootstrap barrier 추가
   - 새 세션 / 재구독 / 새로고침 복원 동안 입력 잠금
   - snapshot 적용 + 최초 resize 적용 완료 후 해제
2. 세션 생성 시 client cols/rows 전달
   - PTY 를 default 80x24 로 먼저 띄우지 않도록 변경
3. PowerShell prompt hook 을 delayed write 가 아니라 startup bootstrap 으로 이동
4. 그 다음에야 PowerShell 빠른 타이핑 회귀 테스트 추가
   - "새 세션 열기 -> 바로 `dasf` + Enter"
   - "새로고침 직후 -> 바로 `dasf` + Enter"
   - "워크스페이스 재진입 직후 -> 바로 `dasf` + Enter"

## 결론

현재 코드에서 가장 그럴듯한 설명은 다음 조합이다.

- PTY 가 기본 크기로 먼저 시작됨
- 프런트가 나중에 fit/resize 함
- `screen-snapshot` 복원과 interactive open 이 분리되어 있음
- PowerShell prompt hook 이 지연 주입됨

그래서 사용자가 빠르게 입력하면 PowerShell redraw 가 "아직 안정화되지 않은 화면 상태" 위에 적용되고, 결과가 겹쳐 보인다.
