# 한글 IME Space 분리 현상 조사 및 수정 계획

작성일: 2026-04-17 (KST)

## 결론 요약

현재 증상:

- `안녕하세요 ` 입력 시 `안녕하세  요`
- `코딩을 합시다` 입력 시 `코딩  을 합시다`

이 현상의 **1차 원인**은 프런트엔드 `TerminalView`가 `Space` 키를 xterm 네이티브 IME 파이프라인으로 보내지 않고, `keydown` 단계에서 직접 `onInput(' ')`로 우회 전달하는 회귀 코드다.

즉:

1. 한글 조합의 마지막 글자가 아직 IME 내부에서 확정되기 전
2. `Space`가 먼저 PTY로 전달되고
3. 그 다음 tick에 마지막 한글 음절이 뒤늦게 확정되면서
4. 화면상 공백이 단어 중간에 끼어든 것처럼 보인다

이 현상은 PowerShell과 WSL bash 모두에서 재현되므로, shell-specific 문제라기보다 **브라우저/xterm/IME 입력 경계 문제**로 보는 것이 맞다.

---

## 조사 범위

병렬 조사 3축:

1. 프런트엔드 입력 경로
2. 백엔드/PTY/WS 경로
3. 최근 터미널 입력/런타임 변경 이력

추가로 로컬 코드 직접 대조와 git 이력 분석을 수행했다.

### 2. xterm 내부 구조상 실제 race가 가능한 구조다

xterm 내부 참고:

- [frontend/node_modules/@xterm/xterm/src/browser/CoreBrowserTerminal.ts](C:/Work/git/_Snoworca/ProjectMaster/frontend/node_modules/@xterm/xterm/src/browser/CoreBrowserTerminal.ts:1021)
- [frontend/node_modules/@xterm/xterm/src/browser/input/CompositionHelper.ts](C:/Work/git/_Snoworca/ProjectMaster/frontend/node_modules/@xterm/xterm/src/browser/input/CompositionHelper.ts:85)

핵심:

- xterm은 custom key handler를 먼저 호출한다.
- 우리 코드가 `false`를 반환하면 xterm의 composition-safe path가 그 keydown을 처리하지 못한다.
- 게다가 `compositionend()`는 최종 한글 음절 확정을 `setTimeout(0)` 뒤로 미룬다.

따라서 실제 순서는 아래처럼 될 수 있다.

1. 사용자: 한글 마지막 음절 조합 중 `Space` 입력
2. 우리 코드: `Space`를 즉시 전송
3. xterm IME 최종 확정: 마지막 음절을 뒤늦게 전송
4. 결과: `안녕하세  요`, `코딩  을`

## 가능성이 낮은 가설

### 1. PowerShell 고유 문제

아니다.

- WSL bash에서도 발생
- 공통점은 shell보다 browser/xterm/Windows PTY 경계다

### 2. 서버 인코딩/문자셋 깨짐

가능성 낮다.

- 서버는 문자열을 거의 그대로 `pty.write()`에 전달한다
- 증상도 깨진 글자보다 “공백 위치가 밀리는” 형태다

### 3. 단순 snapshot/replay 버그만의 문제

가능성 낮다.

- 평상시 steady-state 입력에서도 재현되는 보고와 맞지 않는다
- 다만 직후 상태에서는 증폭 가능성이 있다

## 수정 계획

### 권장안 A. 가장 안전한 1차 수정

`TerminalView`에서 plain `Space` 수동 전달을 제거하고, xterm 네이티브 IME 파이프라인에 맡긴다.

구체:

1. [frontend/src/components/Terminal/TerminalView.tsx](C:/Work/git/_Snoworca/ProjectMaster/frontend/src/components/Terminal/TerminalView.tsx:583) 제거 또는 비활성화
2. 조합 중/조합 종료 직후 `Space`는 `return true`로 xterm 기본 처리 허용
3. `Backspace`도 동일 원칙으로 재검토

이유:

- 과거 `6bcf8aa`의 수정 원칙과 일치
- 구조적으로 가장 단순
- IME를 custom key handler로 우회하지 않게 됨

### 권장안 B. 최소 핫픽스

만약 `Space`/`Backspace` 수동 전달을 당장 완전히 없애기 어렵다면:

1. helper textarea에 `compositionstart` / `compositionend` 리스너 추가
2. `isComposingRef.current` 도입
3. 가드 조건을 아래처럼 강화

예시 개념:

- `ev.isComposing`
- `isComposingRef.current`
- 필요 시 `keyCode === 229` 계열 조합 상태 힌트

그리고 이 조건 중 하나라도 true면:

- `Space` 수동 전달 금지
- xterm 기본 처리 허용

이 방식은 이미 `fix/korean-ime-space-race-guard` 브랜치의 방향과 맞는다.

### 권장안 C. 2차 하드닝

1차 수정 후에도 아래는 점검 필요:

1. replay pending 동안 input gating이 IME 조합과 충돌하지 않는지 확인
2. resize/workspace-switch 직후 IME 조합 입력 수동 검증
3. Windows ConPTY redraw 증폭 여부 확인

즉, root cause fix와 backend hardening은 분리하는 것이 맞다.

## 검증 계획

### 수동 검증 필수

Playwright는 IME composition 재현이 제한적이므로 수동 검증이 반드시 필요하다.

필수 시나리오:

1. PowerShell
   - `안녕하세요 `
   - `코딩을 합시다`
   - `한글 입력 후 Backspace`
2. WSL bash
   - 동일 시나리오
3. resize 직후
4. workspace switch 직후
5. reconnect/refresh 직후

### 자동 검증 보강

다음 자동화는 추가할 가치가 있다.

1. `TerminalView` unit/integration 수준에서 `compositionstart/end` state guard 테스트
2. existing keyboard regression spec에 IME-safe guard 조건 검증 추가

단, 실제 한글 IME 조합 완전 재현은 수동 검증이 최종 게이트여야 한다.

## 최종 판단

가장 가능성이 높은 설명은 아래 한 문장으로 정리된다.

> 최근 `TerminalView`에서 재도입된 `Space` 수동 전달이 xterm의 네이티브 한글 IME 조합 확정보다 먼저 실행되면서, 마지막 한글 음절과 공백의 순서가 뒤집히는 race가 발생했다.

따라서 우선순위는:

1. `TerminalView`의 `Space` 수동 전달 제거 또는 IME guard 강화
2. 수동 한글 입력 검증
3. 필요 시 replay/ConPTY 경계 추가 하드닝
