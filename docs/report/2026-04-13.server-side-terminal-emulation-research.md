# 서버 사이드 터미널 에뮬레이션 연구 보고서

작성일: 2026-04-13  
대상 프로젝트: `ProjectMaster`  
관점: 새로고침, 재연결, 워크스페이스 전환 시 터미널 상태를 서버가 권위적으로 보존하는 방식

## 1. 연구 결론

이 프로젝트에서 가장 먼저 검토해야 할 서버 사이드 에뮬레이션 방식은 `@xterm/headless` 입니다.

이유는 단순합니다.

- 현재 프런트가 이미 `@xterm/xterm` 계열을 사용하고 있습니다.
- 서버는 이미 `node-pty` 기반입니다.
- 따라서 서버와 프런트의 시퀀스 해석기를 가능한 한 같은 계열로 맞추는 것이 가장 작은 리스크로 가장 큰 효과를 냅니다.

반대로 `libvterm` 계열은 기술적으로 강력하지만, 이 저장소의 현재 스택(Node.js/TypeScript/Windows 중심 운영)과는 통합 비용이 훨씬 큽니다.

최종 권장안:

1. 1차 PoC는 `node-pty + @xterm/headless + 현재 xterm 프런트` 조합으로 간다.
2. 브라우저 `localStorage snapshot`은 보조 수단으로 내리고, 서버의 headless terminal state를 1차 권위 데이터로 삼는다.
3. PoC에서 성능, 메모리, alt-screen/TUI 동작이 만족스럽지 않으면 그 다음 대안으로 `libvterm` 사이드카를 검토한다.

## 3. 서버 사이드 에뮬레이션이 필요한 이유

현재의 replay buffer 또는 문자열 snapshot 방식은 본질적으로 한계가 있습니다.

- 출력 스트림을 다시 흘려보내는 것과
- 실제 터미널의 화면 상태를 복원하는 것은
  서로 다릅니다.

문제가 되는 것은 다음입니다.

- 줄바꿈과 carriage return
- soft wrap
- cursor 위치
- clear screen / erase in line
- alternate screen buffer
- resize 이후 재배치
- TUI가 짧은 주기로 화면을 다시 그리는 패턴

이런 항목은 “로그 재생”으로는 정확히 복원되지 않고, “에뮬레이터가 유지한 상태”로만 비교적 정확하게 복원됩니다.

## 4. 후보 1: `@xterm/headless`

### 4.1 공식 자료에서 확인된 사실

공식 README는 `@xterm/headless` 를 이렇게 설명합니다.

- Node.js 에서 실행 가능한 headless terminal
- 원격 서버에서 호스팅되는 프로세스의 terminal state 를 추적하는 용도로 유용
- 일반 xterm 과 같은 API 계열
- 단, 패키지 자체는 `experimental`

또 xterm.js 공식 시퀀스 문서는 xterm.js가 C0/C1/CSI/DCS/ESC/OSC 범주의 많은 시퀀스를 해석한다고 설명합니다.  
즉, 서버와 프런트가 같은 xterm 계열이면 “어떤 시퀀스를 어떻게 해석하는가”의 불일치가 줄어듭니다.

추가로 xterm.js 공식 릴리스 노트에는 `xterm-headless` 환경에서 `serialize` 관련 보정이 들어간 기록이 있습니다.  
이건 `headless + serialize` 조합이 단순 아이디어가 아니라 실제 사용 경로로 고려되고 있다는 신호로 해석할 수 있습니다.

이 마지막 해석은 릴리스 노트로부터의 추론입니다.

### 4.2 장점

- 가장 중요한 장점은 프런트와 해석기 계열을 맞출 수 있다는 점입니다.
- 현재 프로젝트는 이미 xterm 프런트를 쓰므로, 서버까지 xterm 계열이면 parser mismatch가 최소화됩니다.
- TypeScript/Node 스택 안에서 바로 붙일 수 있습니다.
- 현재 `node-pty -> SessionManager -> WsRouter -> xterm 프런트` 구조에 가장 자연스럽게 들어갑니다.
- alt-screen, cursor, erase 류의 상태를 “문자열 tail”이 아니라 에뮬레이터 상태로 다룰 수 있습니다.

### 4.3 단점

- 공식적으로 `experimental` 입니다.
- 공식 README는 Node 전용으로 바로 쓸 수 있는 addon 패키지가 없다고 적고 있습니다.
- 즉, `serialize` 나 상태 덤프를 실제로 어떤 방식으로 꺼낼지 PoC 검증이 필요합니다.
- 세션 수가 많아지면 PTY 하나당 headless terminal 하나를 유지해야 하므로 메모리 비용이 늘어납니다.

### 4.4 이 프로젝트에 대한 적합성

적합성이 가장 높습니다.

이 프로젝트는 이미 xterm 프런트를 쓰고 있고, 브라우저 snapshot과 raw replay 혼합 때문에 문제가 커졌습니다.  
서버 쪽도 xterm 계열로 맞추면 “브라우저가 보던 화면 모델”과 “서버가 기억하는 화면 모델”의 차이가 가장 작아집니다.

## 5. 후보 2: `libvterm`

### 5.1 공식 자료에서 확인된 사실

공식 upstream 사이트는 libvterm을 다음처럼 설명합니다.

- VT220 / xterm / ECMA-48 계열 터미널 에뮬레이터 구현
- C99 기반
- 특정 그래픽 툴킷에 묶이지 않음
- 임베더가 callback 으로 그리기/상태 처리를 받는 구조
- 정상 동작 중에는 `malloc()` 호출을 피하도록 설계

즉, “진짜 에뮬레이터 코어”로서의 성격은 분명합니다.

또 `neovim/libvterm` 저장소는 현재 Neovim에서 직접 쓰는 저장소가 아니라, 실제 사용 코드는 Neovim 본 저장소에 번들되어 있다고 명시합니다.  
이 말은 libvterm 계열이 완전히 버려졌다는 뜻은 아니고, 실제 적용 사례는 있지만 Node/TS 친화적인 배포 형태와는 거리가 있다는 뜻에 가깝습니다.

### 5.2 장점

- 에뮬레이터로서의 정체성이 분명합니다.
- 렌더링 툴킷에 종속되지 않아 상태 모델 중심 설계에 적합합니다.
- C 기반이므로 성능과 제어 가능성 측면에서 강력할 수 있습니다.

### 5.3 단점

- 이 저장소는 Node.js/TypeScript 중심인데, libvterm은 C 라이브러리입니다.
- 따라서 직접 바인딩하거나, 별도 네이티브 모듈을 두거나, 사이드카 프로세스를 둬야 합니다.
- Windows 배포/빌드 파이프라인이 복잡해집니다.
- 프런트는 xterm 계열인데 서버는 libvterm 계열이 되므로 parser behavior mismatch가 다시 생길 수 있습니다.
- “정확한 에뮬레이터”를 얻는 대신 “현재 프로젝트와의 통합 위험”이 커집니다.

### 5.4 이 프로젝트에 대한 적합성

장기적으로는 검토 가치가 있지만, 지금 바로 1차 해법으로 선택하기엔 무겁습니다.

특히 지금 문제는 “기술적으로 가능한 가장 강한 엔진”보다 “현재 코드베이스에 가장 적은 충돌로 권위 상태를 도입하는 것”이 더 중요합니다.

## 6. `node-pty`가 여전히 맡아야 할 역할

공식 README 기준으로 `node-pty` 는 PTY 바인딩입니다.  
즉, 터미널을 에뮬레이션하는 것이 아니라 “프로세스를 PTY에 붙여 control sequence를 내보내게 만드는 역할”입니다.

중요한 사실도 명시돼 있습니다.

- Windows는 ConPTY 기반이며, `winpty` 지원은 제거됨
- 프로세스는 부모와 같은 권한 수준에서 실행됨
- 서버에 노출된 환경에서는 container 격리를 권장
- `node-pty` 는 thread safe 하지 않음

이 뜻은 다음과 같습니다.

- 서버 사이드 에뮬레이터를 도입해도 `node-pty` 자체의 보안/격리 문제는 사라지지 않습니다.
- 결국 “PTY 실행 레이어”와 “화면 상태 에뮬레이션 레이어”는 분리해서 봐야 합니다.

## 7. 이 프로젝트에 맞는 권장 아키텍처

권장 아키텍처는 아래입니다.

```text
Shell Process
  -> node-pty
    -> server-side terminal emulator
      -> authoritative screen state / scrollback state
      -> websocket snapshot + incremental updates
        -> browser xterm renderer
```

핵심은 이겁니다.

- `node-pty` 는 입력/출력 운반
- headless emulator 는 상태 해석
- 브라우저 xterm 은 최종 렌더러

즉, 브라우저 snapshot이 아니라 서버 emulation state가 권위 데이터가 되어야 합니다.

## 8. 구현 방향 제안

### 8.1 1단계 PoC

1. 세션마다 `node-pty` 외에 `@xterm/headless` 인스턴스를 하나 둡니다.
2. PTY 출력이 들어오면:
   - 클라이언트로 브로드캐스트
   - 동시에 headless terminal 에도 `write()`
3. 새 구독/재연결 시:
   - headless terminal 상태에서 권위 snapshot을 만든다
   - 브라우저는 먼저 clear
   - 그 다음 snapshot 적용
   - 그 뒤 live output 으로 전환

### 8.2 권장 프로토콜

- `history` 또는 `screen-snapshot`
- `history:ready`
- 이후 `output`

여기서 중요한 점은 기존처럼 raw replay tail을 보내는 것이 아니라,  
“서버 에뮬레이터가 가진 현재 상태”를 먼저 보내는 쪽이 중심이어야 한다는 것입니다.

### 8.3 snapshot 형태

여기서는 두 갈래가 있습니다.

1. ANSI/VT 시퀀스 형태 snapshot
2. 구조화된 screen model snapshot

현재 스택에서는 1번이 더 현실적입니다.  
다만 `@xterm/headless` 에서 어떤 직렬화 경로가 현재 버전에서 가장 안정적인지는 PoC로 확인해야 합니다.

이 문장은 공식 자료와 현재 코드 구조를 바탕으로 한 권장입니다.
