# Workspace Switch Scrollback Root Cause

Date: 2026-04-13
Branch: `refactor/buffer`
Scope: workspace switch, long PowerShell input, Codex terminal scrollback loss, browser refresh reset

## Executive Summary

The observed symptom is two issues layered together:

1. Browser refresh clears terminal history because current code does not persist xterm scrollback anywhere.
2. "Only recent few lines remain" after switching workspaces is not explained by a normal tab-mode switch alone. The strongest explanation in the current code is:
   - a WebSocket reconnect or equivalent session re-subscription event occurs,
   - all `TerminalView` instances are recreated,
   - the server can only replay the recent tail from `SessionManager.outputBuffer`,
   - older scrollback is gone because it existed only inside the disposed xterm instance.

A secondary explanation exists if one of the workspaces is in `grid` mode: grid mode unmounts terminal trees across workspace switches, which also destroys xterm scrollback.

## Reproduction Interpretation

User scenario:

1. Open a Codex terminal in workspace A and build up long history.
2. Move to workspace B.
3. Open PowerShell and type a very long command.
4. Return to workspace A.
5. Both A and B now show only recent lines.
6. Browser refresh resets everything.

The important detail is "very long command". In PowerShell, long interactive input still produces PTY output because the line editor echoes and redraws the command line. In current server code, that traffic is treated as normal PTY output and is subject to the same live-broadcast/buffer-tail rules as command output.

## Finding 1: Refresh Reset Is Current Behavior

This part is straightforward.

- `TerminalView` creates a fresh xterm instance on mount and disposes it on cleanup. There is no serialize/restore path in current code.
  - `frontend/src/components/Terminal/TerminalView.tsx:121`
  - `frontend/src/components/Terminal/TerminalView.tsx:275`
- Frontend restore logic only reconstructs workspace metadata and current CWD.
  - `frontend/src/hooks/useWorkspaceManager.ts:87`
  - `frontend/src/hooks/useWorkspaceManager.ts:93`
  - `frontend/src/hooks/useWorkspaceManager.ts:101`
- Server persistence stores `workspaces`, `tabs`, `gridLayouts`, and `lastCwd`, not terminal output.
  - `server/src/routes/workspaceRoutes.ts:32`
  - `server/src/services/WorkspaceService.ts:417`
  - `server/src/services/WorkspaceService.ts:477`

Conclusion: refresh-induced scrollback loss is currently by design, not an incidental bug.

## Finding 3: The Strongest Current Root Cause Is WS Reconnect Causing Global Terminal Remount

This is the key frontend defect pattern.

- `WebSocketProvider` stores `status` and `clientId` in state and rebuilds the context value object on every provider render.

  - `frontend/src/contexts/WebSocketContext.tsx:67`
  - `frontend/src/contexts/WebSocketContext.tsx:276`
- `status` changes on open/close/reconnect and `clientId` changes on `connected`.
  - `frontend/src/contexts/WebSocketContext.tsx:87`
  - `frontend/src/contexts/WebSocketContext.tsx:88`
  - `frontend/src/contexts/WebSocketContext.tsx:153`
  - `frontend/src/contexts/WebSocketContext.tsx:157`
  - `frontend/src/contexts/WebSocketContext.tsx:177`
  - `frontend/src/contexts/WebSocketContext.tsx:180`
- `TerminalContainer` depends on the whole `ws` context object, not only stable functions.

- `TerminalView` recreates the xterm instance whenever `onInput` or `onResize` changes, because they are in the effect dependency list.
  - `frontend/src/components/Terminal/TerminalView.tsx:109`
  - `frontend/src/components/Terminal/TerminalView.tsx:204`

### Why this matches the symptom

If the WebSocket reconnects while the user is in workspace B:

1. provider state changes,
2. all `TerminalContainer` consumers re-render,
3. their `handleInput` and `handleResize` callbacks change,
4. every `TerminalView` effect cleans up,
5. every xterm instance is disposed and recreated,
6. all preexisting in-browser scrollback disappears at once.

That explains why both the hidden Codex terminal and the currently visible PowerShell terminal end up reduced together. It also explains why the damage looks global rather than workspace-specific.

## Finding 4: The Server Only Replays Tail Data After Subscriber Loss

Once the frontend recreates terminals, the server cannot reconstruct full history.

- PTY output is only appended to `outputBuffer` when the session currently has no WS subscribers.
  - `server/src/services/SessionManager.ts:194`
  - `server/src/services/SessionManager.ts:196`
  - `server/src/services/SessionManager.ts:737`
- `outputBuffer` is truncated with `slice(-maxBufferSize)`, so it preserves only the recent tail.
  - `server/src/services/SessionManager.ts:739`
  - `server/config.json5:26`
- On first subscribe, the server flushes that buffer and then clears it.
  - `server/src/ws/WsRouter.ts:168`
  - `server/src/ws/WsRouter.ts:172`
  - `server/src/services/SessionManager.ts:675`
  - `server/src/services/SessionManager.ts:679`
  - `server/src/services/SessionManager.ts:682`
- On WS disconnect, session subscriptions are removed for that client.
  - `server/src/ws/WsRouter.ts:211`
  - `server/src/ws/WsRouter.ts:216`

Conclusion: after any reconnect/remount window, the best the system can do is replay the tail accumulated during the subscriber gap. Old scrollback that lived only inside xterm is already gone.

## Finding 5: Long PowerShell Input Is Sufficient To Create "Recent Few Lines" Tail

The scenario uses a very long PowerShell command. That matters because:

- PowerShell line editing still generates PTY output as the line is echoed/redrawn.
- Server buffering is byte/string-tail based, not line-history based.
  - `server/src/services/SessionManager.ts:737`
  - `server/src/services/SessionManager.ts:739`
- Therefore large interactive line editing traffic can easily dominate the replay buffer and visually collapse the restored terminal to "recent few lines".

This also explains why the symptom can appear even if the user did not run a long producing command, but only typed a very long one.

## Secondary Candidate: Grid Mode Workspace Switch

One subagent found a legitimate alternate path:

- grid mode renders only the active workspace tree through `MosaicContainer`,
  - `frontend/src/App.tsx:399`
- so switching away from a grid workspace unmounts that terminal tree,
- and remounting recreates xterm from scratch.

This is a valid explanation if either workspace was in `grid` mode. It is less consistent with the emphasis on "very long command" because grid-mode loss would occur on the switch itself, even without the long-input trigger.

## Subagent Synthesis

Three subagents were used:

- `Dewey`: confirmed that refresh reset is expected because only workspace metadata and `lastCwd` persist.
- `Hooke`: confirmed that server replay logic is tail-only and destructive after flush.
- `Tesla`: confirmed that normal tab-mode switching should preserve hidden terminals, with grid mode as a valid alternate unmount path.

Combined conclusion:

- refresh reset is a design limitation of the current architecture,
- server-side tail replay explains why only recent lines remain,
- the missing link for the workspace-switch symptom is a frontend remount trigger,
- the strongest such trigger in the present code is WebSocket reconnect propagating through unstable context dependencies into `TerminalView` disposal/recreation.

## Confidence and Gaps

Confidence is high on the architectural explanation and medium-high on the exact runtime trigger.

High-confidence facts:

- full scrollback is not persisted,
- server replay is tail-only,
- normal tab-mode hide/show is supposed to preserve terminals,
- grid mode can unmount terminals,
- WebSocketProvider state changes can force `TerminalView` recreation through current dependencies.

Remaining runtime uncertainty:

- whether the user's exact repro passed through a WS reconnect,
- or whether one of the involved workspaces was in grid mode.
