# 설정 페이지 2FA 저장 직후 QR 즉시 표시 가능성 연구

## 결론

가능하다. 다만 **현재 구조에서는 프런트엔드만 수정해서는 안 되고, 백엔드의 TOTP 런타임 반영 방식부터 보강해야 한다.**

지금 구현은 다음 상태다.

- 설정 페이지는 진입 시 한 번만 QR 코드를 가져온다.
- 저장 성공 후에는 QR 코드를 다시 요청하지 않는다.
- 더 큰 문제는 `twoFactor.enabled`, `issuer`, `accountName` 변경이 **실행 중인 `TOTPService` 인스턴스에 반영되지 않는다**는 점이다.

따라서 현재 코드 그대로는:

1. 2FA를 비활성 상태에서 활성화해도 저장 직후 QR이 바로 나오지 않는다.
2. 2FA가 이미 활성화된 상태에서 `issuer`/`accountName`을 바꿔도 QR이 즉시 새 값으로 갱신되지 않는다.

## 실질적 판단

### 현재 코드로 가능한 것

- 설정 페이지 저장 후 프런트에서 `totp-qr`를 다시 호출하는 것 자체는 가능하다.
- 사용자는 이미 로그인된 상태라 `/api/auth/totp-qr` 접근 권한도 있다.

### 현재 코드로 불가능한 것

- `twoFactor.enabled: false -> true` 저장 직후 새 QR 표시
  - 이유: 서버에 `totpService`가 아직 없어서 QR API가 `404`를 반환한다.
- `issuer`/`accountName` 저장 직후 새 QR 반영
  - 이유: 기존 `totpService`가 예전 config를 계속 들고 있다.

## 권장 구현 방향

### 권장안

백엔드에 **TOTP 런타임 재구성 단계**를 추가한 뒤, 프런트에서 저장 성공 후 QR을 재조회한다.

필요 작업:

1. `SettingsService.savePatch()` 성공 경로에서 `twoFactor.enabled / issuer / accountName` 변경을 감지한다.
2. 서버 런타임의 `totpService`를 다음 규칙으로 재구성한다.
   - `enabled=false`면 인스턴스 제거
   - `enabled=true`면 인스턴스 생성 또는 재생성
   - 필요 시 `initialize()` 수행
3. 저장 성공 응답 후 프런트에서 `/api/auth/totp-qr`를 다시 호출한다.
4. 응답을 `setTotpQR(...)`로 반영해 화면을 즉시 갱신한다.

### 장점

- 사용자가 저장 직후 QR을 바로 확인할 수 있다.
- `issuer`/`accountName` 수정도 즉시 눈으로 검증 가능하다.
- 서버 재시작 없이 설정 페이지 UX가 완결된다.

### 주의점

- 현재 README는 [README.md](/C:/Work/git/_Snoworca/ProjectMaster/README.md:243) 에서 “다음 로그인부터 적용”이라고 설명한다.
- 즉시 QR 표시를 넣으려면:
  - 문서 문구를 “QR은 즉시 갱신, 로그인 적용도 즉시 또는 다음 로그인”으로 정확히 정리해야 한다.
  - 실제 로그인 플로우에서도 `totpService` 재구성 결과를 일관되게 보도록 만들어야 한다.

## 최종 판단

**구현 가능하다.**

하지만 현재 구조 기준으로는 단순 프런트 패치가 아니라 아래 두 단계를 같이 해야 한다.

1. 백엔드 `TOTPService` hot-swap / reconfigure
2. 프런트 저장 성공 후 QR 재조회

즉, “바로 화면이 갱신되어 2D 바코드가 나오게” 하는 것은 현실적으로 가능하지만, **현재 코드에는 그 기반이 아직 없다**는 결론이다.
