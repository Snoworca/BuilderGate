# start.js 운영 실행 보강 연구: 공식 문서 교차검증

작성일: 2026-04-17 (KST)

## 목적

로컬 코드 조사로 도출한 `start.js` 운영 실행 방향을 공식 문서로 교차검증한다.

대상:

- Vite 공식 문서
- Express 공식 문서

## 핵심 확인 사항

### 1. Express 정적 서빙은 절대 경로가 안전하다

Express 공식 문서는 `express.static(root)` 사용 시, 상대 경로는 **Node 프로세스를 어디서 실행했는지**에 따라 달라질 수 있으므로 절대 경로를 쓰는 편이 안전하다고 안내한다.

출처:

- Express, "Serving static files in Express"
  - https://expressjs.com/en/starter/static-files.html

의미:

- `start.js`가 서버를 루트에서 직접 띄우는 구조보다, `cwd=server`를 고정하거나
- 더 안전하게는 `server/dist/index.js` 내부에서 `path.join(__dirname, 'public')` 같은 절대 경로 기반 정적 서빙을 써야 한다.

### 2. Vite `build.outDir`은 프로젝트 루트 기준이며, 루트 밖 출력은 주의가 필요하다

Vite 공식 문서는:

- `build.outDir`은 프로젝트 루트 기준 출력 디렉터리라고 설명한다.
- `outDir`이 root 밖에 있으면 실수로 중요한 파일을 지우지 않도록 경고를 낸다고 설명한다.
- `publicDir` 내용은 기본적으로 `outDir`에 복사된다고 설명한다.

출처:

- Vite, "Build Options"
  - https://vite.dev/config/build-options

의미:

- `frontend`에서 바로 `../server/dist/public`으로 출력하는 구성은 가능하더라도, Vite의 안전장치/경고 맥락상 조심해서 다뤄야 한다.
- 따라서 초기 구현은:
  - `frontend/dist`로 기본 빌드
  - 후처리 복사로 `server/dist/public` 구성
  가 더 보수적이고 안전하다.

### 3. Vite manifest는 서버 템플릿 렌더링이 필요할 때 주로 의미가 크다

Vite 공식 문서는 `build.manifest`를 "해시된 자산 파일명을 서버 프레임워크가 올바른 링크로 렌더링할 수 있게 해주는 매핑"으로 설명한다.

출처:

- Vite, "Build Options" -> `build.manifest`
  - https://vite.dev/config/build-options

의미:

- 현재 ProjectMaster는 SSR이나 서버 템플릿 렌더링이 아니라, 정적 `index.html`을 그대로 내려주면 충분하다.
- 따라서 이번 작업의 1차 범위에서는 `manifest` 도입이 필수는 아니다.

## 강화된 결론

공식 문서 기준으로도 다음 방향이 가장 합리적이다.

1. `frontend`는 기본 `dist` 빌드를 유지한다.
2. 빌드 후 `frontend/dist`를 `server/dist/public`으로 복사한다.
3. `server`는 production에서 `express.static(absolutePathToPublic)`으로 정적 자산을 서빙한다.
4. 일반 HTML 진입 요청만 `index.html` fallback을 주고, 자산 404는 자산 404로 남긴다.
5. 루트 `start.js`는 `cwd=server`로 `node dist/index.js`를 실행한다.

## 구현 계획에 미치는 영향

- 직접 `outDir=../server/dist/public` 방식은 2차 최적화 후보로 둔다.
- 1차 구현은 "기본 빌드 + 후처리 복사 + 절대 경로 정적 서빙"으로 계획한다.
