# Move Handle Root Cause Report

Date: 2026-04-22
Method: 4 independent sub-agent investigations + local code/runtime evidence review
Scope: “무브 버튼을 누르고 드래그해도 창 이동이 시작되지 않는다” 이슈의 정확한 원인 규명

## Final Conclusion

현재 기준의 정확한 원인은 **stale Vite cache가 아니라, equal 모드에서 실제 drag source가 지나치게 좁고 제한적으로 연결된 구조**입니다.

정확히는:

1. equal 모드에서 vendor의 기본 toolbar-wide drag wrapper가 꺼집니다.
2. app 쪽 custom toolbar에서는 오직 작은 grip 하나만 `connectDragSource`로 연결되어 있습니다.
3. 그 grip은 hover 기반이며 기본 상태에서 `opacity: 0`이고,
4. toolbar 자체는 `height: 0`로 접혀 있습니다.

그래서 사용자는 “무브 버튼/그 주변 chrome”을 누른다고 느끼지만, 실제로는 **drag source가 아닌 영역**을 잡기 쉬워지고, 그 경우 drag가 시작되지 않습니다.

즉, 이 현상은 “드래그 코드가 완전히 끊어졌다”기보다는 **실제 드래그 가능 히트 영역이 너무 작고 엄격하게 제한된 결과**입니다.

## What It Is Not

- 현재 기준으로는 **stale optimized dependency cache가 1순위 원인이라는 증거는 부족**합니다.
- 현재 소스/patch/optimized dep를 기준으로 보면 drag wiring 자체는 존재합니다.

## Sub-Agent Synthesis

### Reviewer A

- drag wiring은 살아 있음
- 현재 cache mismatch를 주원인으로 보기 어렵다

### Reviewer B

- stale cache보다 `draggable={layoutMode !== 'equal'}` 와 `reorderEnabled` 조합이 더 직접적인 위험

### Reviewer C

- 원인은 작은 hidden grip + zero-height toolbar + toolbar-wide drag 비활성화

### Reviewer D

- 사용자 체감의 직접 원인은 “버튼처럼 보이는 영역”과 “실제 drag source”가 다르기 때문

## Final Judgment

정확한 원인은 다음 한 문장으로 요약할 수 있습니다.

**equal 모드에서 drag 시작 권한을 작은 grip 하나에만 남겨 두고, toolbar 전체 drag를 꺼 버린 현재 구조 때문에 사용자가 실제 drag source를 정확히 잡지 못해 drag가 시작되지 않는다.**

즉:

- “기능이 완전히 끊긴 것”은 아님
- “현재 1순위 원인 = stale cache”도 아님
- **실제 hit target 설계와 조건 분기 구조가 문제의 본체**입니다

## Recommended Next Step

이후 수정은 다음 둘 중 하나를 명확히 선택해야 합니다.

1. grip hit area를 크게 하고 항상 명확히 보이게 만든다
2. equal 모드에서도 사용자가 기대하는 “move button 영역 전체”를 drag source로 다시 연결한다

현재 사용자 보고를 기준으로는 2번 쪽이 더 자연스럽습니다.
