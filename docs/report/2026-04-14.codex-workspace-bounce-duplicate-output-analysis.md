# Codex Workspace 왕복 중 중복 출력 분석

작성일: 2026-04-14  
대상 환경: `https://localhost:2002`

## 결론

이 문제의 1순위 원인은 `중복 subscribe`가 아니라 `workspace 왕복 시 발생하는 resize 폭주`입니다.

- 프런트는 workspace 전환 때 터미널을 `display: none/flex`로 숨기고 다시 보여 줍니다.
- 활성 탭에서만 `MetadataRow`를 붙여서, 복귀 순간 터미널 높이가 추가로 흔들립니다.
- `TerminalView`는 이 변화를 `ResizeObserver + fit + onResize`로 서버에 계속 보냅니다.
- 서버 `SessionManager.resize()`는 `cols/rows`가 실제로 바뀌지 않아도 항상 `pty.resize()`, `screenSeq++`, `refreshReplaySnapshots()`를 수행합니다.
- `codex` 같은 TUI는 resize/SIGWINCH에 매우 민감해서 하단 프레임과 상태 줄을 다시 그립니다. 사용자 눈에는 이것이 `중복 출력`과 `공백 증가`로 보입니다.

보조 원인도 있습니다.

- `@xterm/headless` snapshot은 현재 `headless write queue`와 프레임 단위로 원자적이지 않습니다.
- 그래서 replay가 실제로 개입하는 경로에서는 같은 TUI 프레임이 `snapshot`과 `queued live output`으로 갈라져 겹쳐질 위험이 남아 있습니다.

즉, 현재 관찰한 증상은 아래 두 축이 합쳐진 결과로 보는 것이 맞습니다.

1. `resize-driven redraw`
2. `TUI-safe 하지 않은 snapshot/live output 경계`

## 5개 서브에이전트 검토 요약

### 1. Bacon

서버 축을 다시 파고들어, `no-op resize`도 실제 resize처럼 처리되는 점을 지적했습니다.

- [frontend/src/App.tsx](../../frontend/src/App.tsx)
- [frontend/src/components/MetadataBar/MetadataRow.tsx](../../frontend/src/components/MetadataBar/MetadataRow.tsx)
- [frontend/src/components/Terminal/TerminalView.tsx](../../frontend/src/components/Terminal/TerminalView.tsx)
- [server/src/services/SessionManager.ts](../../server/src/services/SessionManager.ts)
- [server/src/ws/WsRouter.ts](../../server/src/ws/WsRouter.ts)

핵심 판단:

- `duplicate subscribe`보다 `resize-driven redraw`가 훨씬 유력하다.
- `SessionManager.resize()`가 no-op resize를 무시하지 않는 것이 핵심 결함이다.

### 2. Hegel

프런트 레이아웃 관점에서 `MetadataRow 조건부 렌더 + display 토글 + 다중 fit/ResizeObserver` 조합을 주원인으로 평가했습니다.

핵심 판단:

- workspace 복귀 시 `fit()` 출처가 여러 개다.
- 터미널 높이가 한 번이 아니라 여러 번 흔들린다.
- `codex` TUI는 이 resize burst에 반응해 전체 화면을 반복 redraw한다.

### 3. Halley

TUI/headless 관점에서 `snapshot`과 `live output` 경계가 TUI 프레임 단위로 안전하지 않다고 평가했습니다.

핵심 판단:

- `serialize/restore`는 상태 덤프이지 TUI 프레임 트랜잭션이 아니다.
- alt-screen, footer redraw, absolute cursor move가 많은 프로그램에서는 replay overlap이 중복처럼 보이기 쉽다.
- `getScreenSnapshot()`은 `headless write drain` 이후로 fence 되어야 한다.

### 4. Plato

구독 생명주기 관점에서 `workspace 왕복 = 중복 subscribe` 경로는 약하다고 봤습니다.

핵심 판단:

- tab mode에서는 모든 터미널이 살아 있고 `display`만 바뀐다.
- 따라서 일반적인 workspace 왕복은 `unsubscribe -> subscribe`를 직접 만들지 않는다.
- 다만 grid mode나 consumer handoff 경계에서는 snapshot ownership ambiguity가 남아 있다.

### 5. Hume

테스트/관측성 관점에서 현재 회귀망이 이 문제를 제대로 증명하지 못한다고 평가했습니다.

핵심 판단:

- 지금 E2E는 주로 `visible DOM`만 읽는다.
- 현재 구조에서 단순 workspace 클릭 왕복은 replay를 잘 안 타므로, replay 버그 증명 시나리오로는 약하다.
- `sessionId / replayToken / snapshotSeq` 단위의 ring buffer 텔레메트리가 필요하다.

## 현재 구조에서 실제로 무슨 일이 일어나는가

### 1. workspace 왕복 자체는 보통 재구독이 아니다

tab mode에서 앱은 모든 탭을 계속 렌더하고, 보이지 않는 탭은 숨기기만 합니다.

- [frontend/src/App.tsx](../../frontend/src/App.tsx)
  - `display: isVisible ? 'flex' : 'none'`
  - 활성 탭만 `MetadataRow` 렌더
- [frontend/src/components/Terminal/TerminalContainer.tsx](../../frontend/src/components/Terminal/TerminalContainer.tsx)
  - `subscribeSession()` effect는 `sessionId`가 바뀌지 않으면 다시 돌지 않음

따라서 지금 사용자가 보는 현상은 대체로 `재구독 중복`보다 `같은 세션이 계속 살아 있는 상태에서 resize에 의해 TUI가 다시 그려지는 현상`으로 설명하는 편이 맞습니다.

### 2. 프런트가 복귀 시 resize를 과하게 발생시킨다

관련 위치:

- [frontend/src/App.tsx](../../frontend/src/App.tsx)
  - visible 토글
  - `MetadataRow` 조건부 렌더
- [frontend/src/components/MetadataBar/MetadataRow.tsx](../../frontend/src/components/MetadataBar/MetadataRow.tsx)
  - 높이 `28px`
- [frontend/src/components/Terminal/TerminalContainer.tsx](../../frontend/src/components/Terminal/TerminalContainer.tsx)
  - `isVisible`가 `false -> true`일 때 `fit()`
- [frontend/src/components/Terminal/TerminalView.tsx](../../frontend/src/components/Terminal/TerminalView.tsx)
  - mount 직후 `fitAddon.fit(); onResize(term.cols, term.rows);`
  - `ResizeObserver`에서 다시 `fitAddon.fit(); onResize(term.cols, term.rows);`

현재는 `직전 cols/rows와 같으면 skip`하는 억제가 없습니다.

### 3. 서버가 no-op resize도 새 replay 경계처럼 취급한다

관련 위치:

- [server/src/services/SessionManager.ts](../../server/src/services/SessionManager.ts)
  - `resize()`가 무조건 `pty.resize()`
  - `screenSeq += 1`
  - `markSnapshotDirty()`
  - `refreshReplaySnapshots()`
- [server/src/ws/WsRouter.ts](../../server/src/ws/WsRouter.ts)
  - `refreshReplaySnapshots()`가 pending replay가 있으면 새 토큰으로 snapshot을 다시 보냄

이건 일반 셸에서는 티가 덜 나지만, `codex`처럼 하단 상태줄과 진행 UI를 자주 redraw하는 TUI에서는 눈에 띄게 드러납니다.

### 4. replay 경계도 아직 TUI-safe하지 않다

관련 위치:

- [server/src/services/SessionManager.ts](../../server/src/services/SessionManager.ts)
  - `getScreenSnapshot()`
  - `queueHeadlessOutput()`
  - `applyHeadlessOutput()`
- [server/src/ws/WsRouter.ts](../../server/src/ws/WsRouter.ts)
  - `markReplayPending()`
  - `routeSessionOutput()`
  - `handleScreenSnapshotReady()`
- [server/src/utils/headlessTerminal.ts](../../server/src/utils/headlessTerminal.ts)

`headless` snapshot과 queued output이 충분히 좁은 경계에서 만나지 않으면, 같은 TUI 프레임 일부가 snapshot에 들어가고 나머지가 live output으로 다시 붙을 수 있습니다.

이건 `refresh`, `reconnect`, `replay refresh` 경로에서 특히 심각합니다.

## 왜 Codex에서 특히 심하게 보이는가

`codex`는 단순 line-oriented shell 출력이 아닙니다.

- 하단 프롬프트/상태줄을 자주 갱신합니다.
- cursor move, carriage return, redraw를 많이 사용합니다.
- resize가 발생하면 현재 프레임을 다시 그릴 가능성이 높습니다.

그래서 같은 텍스트를 실제로 두 번 "출력"하지 않아도, resize가 여러 번 들어오면 사용자는 같은 UI 블록이 여러 번 쌓인 것처럼 보게 됩니다.

사용자가 본 예시의 반복되는 아래 블록은 이 해석과 잘 맞습니다.

- `› Run /review on my current changes`
- 하단 모델/컨텍스트/status line
- 중간의 큰 공백

이건 전형적인 `TUI redraw가 scrollback에 쌓여 보이는 패턴`입니다.

## 우선순위 해결안

### 1순위: no-op resize 차단

가장 먼저 막아야 합니다.

- 프런트:
  - 마지막으로 전송한 `cols/rows`를 기억하고 같으면 `onResize`를 보내지 않기
  - `TerminalContainer`의 visible-effect `fit()`와 `TerminalView` 내부 `ResizeObserver`/initial fit 중복 정리
- 서버:
  - `SessionManager.resize()`에서 `data.cols === cols && data.rows === rows`면 즉시 return
  - no-op인 경우 `screenSeq++`, `markSnapshotDirty()`, `refreshReplaySnapshots()`도 모두 금지

### 2순위: 터미널 높이 흔들림 제거

- `MetadataRow`를 active일 때만 mount/unmount 하지 말고, 항상 고정 높이 슬롯을 유지
- 또는 terminal viewport와 metadata 영역을 확실히 분리해서 workspace 왕복 때 terminal height가 바뀌지 않게 만들기

이 작업이 없으면 resize dedupe를 해도 실제 `rows` 변화가 계속 발생합니다.

### 3순위: snapshot을 headless write drain 뒤로 fence

`getScreenSnapshot()`은 최소한 다음 계약을 가져야 합니다.

- 현재 `headlessWriteChain`이 drain된 뒤에만 authoritative snapshot 생성
- snapshot 생성 시점의 `seq`와 pending output 경계를 명확히 분리
- resize refresh가 들어와도 이미 snapshot에 포함된 출력이 다시 queued live output으로 flush되지 않도록 보장

### 4순위: TUI-safe fallback/replay 정책 분리

- oversized snapshot에서 `empty fallback`보다 `last good authoritative snapshot` 우선 사용
- TUI/alt-screen 감지 시 local snapshot fallback을 제한하거나 비활성화
- replay refresh와 restore를 분리하여, 단순 resize만으로 새 replay cycle이 남발되지 않게 조정

## 필요한 테스트

### 서버 단위 테스트

- `no-op resize` 시:
  - `pty.resize()`가 호출되지 않아야 함
  - `screenSeq` 증가 금지
  - `refreshReplaySnapshots()` 호출 금지
- `headlessWriteChain`이 밀린 상태에서 snapshot 생성 시:
  - drain 이전 snapshot 금지
  - 동일 marker 중복 flush 금지
- `replay refresh + stale ack + fresh ack` 혼합 시:
  - marker line이 정확히 1회만 전달되는지 검증

### 프런트 단위/통합 테스트

- visibility 토글만으로 동일 `cols/rows` resize가 여러 번 전송되지 않는지 검증
- `MetadataRow` 존재 여부가 terminal viewport를 흔들지 않는지 검증

### Playwright E2E

지금처럼 visible DOM만 읽으면 부족합니다.

- `localStorage`의 `terminal_snapshot_${sessionId}` 또는 별도 debug surface를 사용해 전체 snapshot에서 marker line occurrence를 세어야 함
- 시나리오:
  - `codex` 또는 synthetic TUI가 `BG_SEQ_001..BG_SEQ_300` 출력
  - workspace A/B 빠른 왕복
  - 필요 시 refresh 또는 reconnect
  - 최종 snapshot에서 각 marker가 정확히 1회인지 확인

## 필요한 관측성

`/api/sessions/telemetry`만으로는 부족합니다.

추가 필요 항목:

- 최근 replay event ring buffer
  - `timestamp`
  - `clientId`
  - `sessionId`
  - `replayToken`
  - `snapshotSeq`
  - `event`
    - `subscribe`
    - `snapshot_sent`
    - `snapshot_refreshed`
    - `ack_ok`
    - `ack_stale`
    - `output_queued`
    - `output_flushed`
    - `unsubscribe`
- 프런트 debug surface
  - `replaceWithSnapshot` 호출 횟수
  - 마지막 `replayToken`
  - 마지막 `seq`
  - 마지막 전송 `cols/rows`

이게 있어야 `서버가 두 번 보냈는지`, `클라이언트가 같은 snapshot을 두 번 적용했는지`, `실제 원인은 resize redraw인지`를 분리할 수 있습니다.

## 최종 판단

심각도는 `High`입니다.

다만 해결 방향은 이미 좁혀졌습니다.

- 즉시 고칠 것:
  1. 프런트/서버 `no-op resize` 차단
  2. `MetadataRow`로 인한 viewport height oscillation 제거
- 구조적으로 고칠 것:
  3. `headless snapshot` 생성 시 write-drain fence 추가
  4. TUI 전용 replay/fallback 정책 정교화

현재 상태를 한 문장으로 요약하면 이렇습니다.

`workspace 왕복 시 발생하는 resize burst가 Codex TUI redraw를 유발하고, replay 경계가 아직 TUI-safe하지 않아 그 redraw가 중복 출력과 공백으로 보이고 있다.`
