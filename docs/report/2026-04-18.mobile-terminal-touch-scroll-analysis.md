# 모바일 터미널 세로 스크롤 불능 분석

작성일: 2026-04-18
대상: 모바일 브라우저에서 터미널 scrollback이 위아래 스와이프로 움직이지 않는 현상
분석 방식: 메인 에이전트 코드 분석 + 독립 서브에이전트 2인 교차 검토

## 요약

- 두 서브에이전트 결론은 거의 동일했다.
- 1순위 원인은 `single-touch` 세로 pan을 scrollback 이동으로 연결하는 경로가 현재 프런트엔드에 없다는 점이다.
- 현재 모바일 터미널은 `pinch`만 앱이 직접 구현했고, 한 손가락 세로 스와이프는 xterm에도 앱에도 처리 경로가 없다.
- 여기에 상위 레이아웃과 터미널 래퍼가 `overflow: hidden` 위주로 닫혀 있어 브라우저 기본 세로 스크롤 fallback도 없다.
- 추가로 현재 앱 CSS는 `.xterm-viewport`를 스크롤 표면으로 가정하지만, 실제 번들된 `@xterm/xterm@6.0.0`은 `Viewport -> SmoothScrollableElement`와 custom scrollbar 중심으로 scrollback을 관리한다.

## 서브에이전트 공통 결론

### 1. 가장 유력한 원인

- `TerminalView`는 모바일에서 `touchstart/touchmove/touchend`를 핀치 줌 용도로만 등록한다.
  - 근거: `frontend/src/components/Terminal/TerminalView.tsx:896-911`
- `usePinchZoom`는 두 손가락일 때만 `preventDefault()` 후 폰트 크기를 조정한다.
  - 근거: `frontend/src/hooks/usePinchZoom.ts:31-59`
- 한 손가락 세로 pan을 `term.scrollLines()`나 `scrollToLine()`로 연결하는 코드가 없다.
- 반면 실제 xterm 6 scrollback 경로는 `Viewport`와 `SmoothScrollableElement`에 있다.
  - 근거: `frontend/node_modules/@xterm/xterm/src/browser/Viewport.ts:53-76`, `:108-113`
  - 근거: `frontend/node_modules/@xterm/xterm/src/vs/base/browser/ui/scrollbar/scrollableElement.ts:220-229`, `:258-259`
- xterm browser source를 `touchstart|touchmove|touchend|pointerdown|pointermove|pointerup`로 검색했을 때, scrollback을 위한 모바일 touch pan 처리 경로는 확인되지 않았다.

### 2. 왜 pinch는 되고 scroll은 안 되는가

- pinch는 앱이 직접 구현했기 때문에 동작한다.
  - `usePinchZoom`가 2-touch일 때만 `preventDefault()` 후 폰트 크기를 바꾼다.
- scroll은 앱이 구현하지 않았고, xterm 6도 현재 통합 방식에서는 `wheel`과 custom scrollbar 중심으로 scrollback을 처리한다.
- 따라서 현재 모바일에서는:
  - `2-touch` → 앱 커스텀 pinch 경로 존재
  - `1-touch vertical pan` → 경로 부재

## 코드 근거

### 앱 측

- scrollback 자체는 충분히 켜져 있다.
  - `frontend/src/components/Terminal/TerminalView.tsx:522`
  - `scrollback: 10000`
- 모바일 터치 리스너는 pinch 전용이다.
  - `frontend/src/components/Terminal/TerminalView.tsx:896-911`
- pinch 훅은 2-touch만 가로챈다.
  - `frontend/src/hooks/usePinchZoom.ts:31-59`
- 터미널 런타임 레이어는 별도 상위 터치 레이어를 두고 long-press를 감시한다.
  - `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx:87-109`
- long-press 훅은 이동량을 보고 취소만 할 뿐, 세로 스크롤 위임 로직은 없다.
  - `frontend/src/hooks/useLongPress.ts:39-75`
- 앱 전체와 터미널 컨테이너는 `overflow: hidden` 위주다.
  - `frontend/src/styles/globals.css:7-10`
  - `frontend/src/styles/globals.css:20-24`
  - `frontend/src/styles/globals.css:41-59`
  - `frontend/src/App.tsx:65`
  - `frontend/src/components/Terminal/TerminalView.css:1-20`
- 현재 커스텀 CSS는 `.xterm-viewport`만 스크롤 surface처럼 다룬다.
  - `frontend/src/components/Terminal/TerminalView.css:31-34`

### xterm 측

- 실제 scrollback 구현은 `Viewport -> SmoothScrollableElement` 경로다.
  - `frontend/node_modules/@xterm/xterm/src/browser/Viewport.ts:53-76`
  - `frontend/node_modules/@xterm/xterm/src/browser/Viewport.ts:108-113`
- `SmoothScrollableElement`는 wheel/scrollbar 중심으로 구성돼 있다.
  - `frontend/node_modules/@xterm/xterm/src/vs/base/browser/ui/scrollbar/scrollableElement.ts:220-229`
  - `frontend/node_modules/@xterm/xterm/src/vs/base/browser/ui/scrollbar/scrollableElement.ts:258-259`
- 즉, 앱이 믿고 있는 `.xterm-viewport` CSS와 실제 scroll truth가 어긋나 있다.

## 보조 관찰

- `TerminalRuntimeLayer`는 `onPointerDownCapture`에서 포커스를 바로 주고, `onTouchStart/Move/End`에서 long-press 경로를 먼저 탄다.
  - `frontend/src/components/Terminal/TerminalRuntimeLayer.tsx:91-109`
- 이 코드는 주원인이라기보다 2차 악화 요인으로 보인다.
  - 특히 모바일에서 세로 pan이 확정될 때 long-press를 즉시 취소하거나, touch 입력에 대해서는 포커스를 지연해야 더 안전하다.

## 백엔드 가능성 평가

- 백엔드가 주원인일 가능성은 낮다.
- 이유:
  - 프런트 xterm은 `scrollback: 10000`으로 충분한 버퍼를 가진다.
  - 서버는 snapshot/scrollback 계층을 유지하고 있으며, 이번 현상은 “버퍼가 없다”가 아니라 “모바일 제스처가 scrollback 탐색으로 연결되지 않는다”는 형태다.

## 통합 결론

### 1순위 원인

- 모바일 `single-touch vertical pan -> terminal scrollback` 연결 부재

### 2순위 원인

- `.xterm-viewport`를 스크롤 표면으로 가정한 CSS/구현 가정이 xterm 6 실제 구조와 어긋남

### 3순위 악화 요인

- `overflow: hidden` 중심 레이아웃
- `TerminalRuntimeLayer`의 상위 touch/long-press/focus 경로

## 권장 수정 방향

### 권장안 A: xterm custom scroll 유지

- `TerminalView`에 모바일 `single-touch vertical pan` 상태기를 추가한다.
- `1-touch`에서 Y축 우세가 확정되면 `preventDefault()` 후 `term.scrollLines()` 또는 `scrollToLine()`로 scrollback을 직접 움직인다.
- `2-touch`가 되면 기존 pinch 로직으로 전환한다.
- `TerminalRuntimeLayer`의 long-press는 세로 pan이 일정 임계치를 넘는 즉시 취소한다.
- touch 입력에서 즉시 `focus()`하는 현재 경로는 지연 또는 조건부로 바꾸는 편이 안전하다.

### 권장안 B: native pan 위임

- 실제 스크롤 surface를 모바일에서 노출한다.
- 그 surface에 `touch-action: pan-y`, `overscroll-behavior: contain`, `-webkit-overflow-scrolling: touch`를 적용한다.
- 다만 현재 xterm 6 통합은 custom scroll 구조이므로, 이 방향은 구조 변경 비용이 더 크다.

## 현재 시점의 판단

- 가장 안전한 1차 수정은 권장안 A다.
- 이유:
  - 현재 구조를 가장 덜 흔든다.
  - pinch와 scroll을 같은 터치 상태기 안에서 명시적으로 분기할 수 있다.
  - `.xterm-viewport` CSS 가정에 덜 의존한다.

## 미확인 항목

- 실제 iOS Safari / Android Chrome 원격 디버깅으로 최종 차단점까지는 아직 찍지 못했다.
- 따라서 “정확히 어느 DOM 레이어가 마지막으로 pan을 삼키는지”는 실기기 디버깅으로 한 번 더 고정할 가치가 있다.
- 다만 현재 코드상으로는 수정 우선순위를 정하기에 충분한 근거가 모였다.

## 최종 판단

- 이 이슈는 백엔드보다 프런트엔드 터치 입력 설계 문제다.
- 특히 모바일 터미널에서 pinch만 구현되고 scroll pan이 구현되지 않은 것이 핵심이다.
- 수정은 가능하며, 1차 타깃은 `TerminalView.tsx`의 모바일 터치 처리와 `TerminalRuntimeLayer.tsx`의 long-press/focus 경계다.
