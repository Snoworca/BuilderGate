# 터미널 스크롤백 보존 (브라우저 새로고침 시)

**날짜**: 2026-04-13  
**커밋**: `5d0fdf6` feat: 브라우저 새로고침 시 터미널 스크롤백 보존

## 이슈 설명

브라우저를 새로고침하면 모든 터미널 세션의 스크롤백(히스토리)이 사라지는 문제.
`maxBufferSize`를 90MB로 설정해도 효과 없음.

## 문제 원인

`SessionManager.flushBufferToWs()`가 클라이언트에 버퍼를 전송한 후 `data.outputBuffer = ''`로 초기화.
이로 인해 버퍼 크기 설정과 무관하게 재접속 시 항상 빈 버퍼를 받음.

이전 시도에서 버퍼 클리어를 제거했으나, TUI 앱(Codex = ratatui + crossterm)의 Alternate Screen Buffer 시퀀스(`\x1b[?1049h`)가 replay되어 "빈줄" 아티팩트 발생.

## 해결 방법

### 1. 버퍼 영속화 (SessionManager.ts)
- `flushBufferToWs()`에서 `data.outputBuffer = ''` 제거
- 버퍼가 `maxBufferSize` 범위 내에서 영구 유지

### 2. Alternate Screen 스트리핑 (SessionManager.ts)
- `stripAlternateScreen()`: `\x1b[?1049h` ~ `\x1b[?1049l` 구간 제거
- 미종료 alternate screen은 마지막 진입점부터 끝까지 제거
- CLI 히스토리만 깨끗하게 복원, TUI 앱 렌더링 아티팩트 방지

### 3. 서버측 bufferFilter 적용 (SessionManager.ts)
- `applyBufferFilter()`: flush 시 ED 시퀀스 필터링 (실시간 broadcast는 미적용)
- 클라이언트측과 동일한 필터 레벨 적용 (none/minimal/standard/aggressive)

### 4. Escape 경계 안전 트림 (SessionManager.ts)
- `safeEscapeBoundary()`: `slice(-maxBufferSize)` 후 잘린 CSI 시퀀스 감지 및 복구
- 불완전 escape 시퀀스로 인한 깨짐 방지

### 5. WS Subscribe 순서 변경 (WsRouter.ts)
- flush BEFORE subscribe: 히스토리 버퍼 먼저 전송, 이후 실시간 구독 등록
- 출력 순서 보장 (히스토리 → 실시간)

### 6. 프론트엔드 scrollToBottom (TerminalView.tsx)
- 10KB 초과 write 시 `term.write(data, () => term.scrollToBottom())`
- 대량 버퍼 replay 후 최신 출력 위치로 자동 스크롤

## Codex 분석 결과

Codex CLI는 Rust(ratatui + crossterm) 기반 TUI로 다음 시퀀스를 사용:
- `\x1b[?1049h/l` (Alternate Screen Buffer)
- `\x1b[2J\x1b[H` (화면 지우기 + 커서 홈)
- `\x1b[?25l/h` (커서 숨기기/보이기)

이 시퀀스들이 raw 버퍼 replay 시 빈줄/아티팩트의 직접적 원인이었으며, `stripAlternateScreen()`이 이를 해결.
