# `start.js` 기반 운영 실행 연구

작성일: 2026-04-17 (KST)

## 결론

`dev.js` 대신 `start.js`로 실행하려면 빌드 단계는 필요하다.  
다만 지금 코드베이스에서는 **루트 단일 `dist/`로 모두 몰아넣는 방식보다, `server/dist`를 운영 런타임의 기준으로 유지하고 프런트 산출물을 `server/dist/public`에 넣는 방식이 가장 안전하다.**

권장안:

1. `server`는 지금처럼 `server/dist`로 빌드한다.
2. `frontend`는 운영 빌드 후 결과물을 `server/dist/public`으로 복사한다.
3. `server/src/index.ts`에 `production` 전용 정적 서빙을 추가한다.
4. 루트의 `start.js`는 **반드시 `cwd=server`로** `node dist/index.js`를 실행한다.

핵심 이유는 현재 서버 런타임이 일부 경로를 `process.cwd()` 또는 `server` 루트 기준으로 해석하고 있어서, 서버 번들을 루트 `dist/`로 옮기면 설정/데이터 경로가 쉽게 어긋나기 때문이다.

---

## 현재 상태 조사

### 1. 빌드 스크립트는 이미 반쯤 준비되어 있다

- 프런트는 `vite build` 기반 빌드가 있다: `frontend/package.json:7-10`
- 서버는 `tsc` 후 `dist/index.js`를 실행하는 구조가 있다: `server/package.json:6-9`

즉, 서버는 이미 운영 실행 진입점이 있고, 부족한 것은:

- 프런트 정적 파일 운영 서빙
- 루트 레벨 오케스트레이션 (`start.js`)
- 빌드 산출물 배치 전략

### 2. 현재 서버는 운영 모드에서 프런트를 서빙하지 않는다

`server/src/index.ts:327-352`를 보면 비개발 환경에서는 Vite 프록시가 비활성화된다.  
그런데 그 자리를 대체하는 `express.static(...)` 또는 `index.html` fallback은 없다.

실측 검증:

- 2026-04-17 KST, 임시 포트 `2106`에서 `NODE_ENV=production`으로 `server/dist/index.js` 실행
- `GET /health` -> `200 OK`
- `GET /` -> `404 Not Found`, 본문 `Cannot GET /`

즉, **운영 서버는 뜨지만 UI는 열리지 않는다.**

### 3. 현재 프런트 빌드는 정상 동작한다

실행 결과:

- `frontend` 빌드 성공
- 산출물:
  - `frontend/dist/index.html`
  - `frontend/dist/assets/index-Bd4tS8MZ.js`
  - `frontend/dist/assets/index-CnXJ6O8z.css`
  - `frontend/dist/logo.svg`

따라서 프런트 자체는 이미 운영 정적 자산으로 만들 수 있다.

### 4. 경로 해석이 아직 "서버 루트 + 현재 작업 디렉토리"에 묶여 있다

다음 지점이 중요하다.

- `server/src/utils/config.ts:25-27`
  - `config.json5`를 서버 루트 기준으로 찾는다.
- `server/src/utils/config.ts:238-239`
  - 서버 루트를 `server` 디렉토리 기준으로 계산한다.
- `server/src/services/WorkspaceService.ts:29-35`
  - `workspace.dataPath` 기본값은 `./data/workspaces.json`
  - 실제 해석은 `path.resolve(...)`, 즉 `process.cwd()` 기준
- `server/src/services/TOTPService.ts:29`
  - TOTP secret 기본 경로가 `path.join(process.cwd(), 'data', 'totp.secret')`

이 말은 곧:

- `node server/dist/index.js`를 **루트에서 직접 실행**하면 일부 파일은 루트 기준으로,
- `cwd=server`에서 실행하면 일부 파일은 `server` 기준으로 해석된다.

따라서 `start.js`는 단순 실행 스크립트가 아니라, **런타임 작업 디렉토리까지 보장하는 스크립트**여야 한다.

## 왜 "루트 단일 dist"가 지금은 최선이 아닌가

사용자 질문대로 "빌드해서 dist 디렉토리에 모으는" 생각 자체는 맞다.  
하지만 **그 `dist`를 루트 하나로 통합하는 것**은 현재 구조에서 비용이 크다.

이유:

1. 서버가 `server` 루트를 기준으로 설정 파일과 보조 파일을 찾는다.
2. 일부 서비스는 `process.cwd()` 기준 상대경로를 사용한다.
3. 셸 통합 스크립트도 현재 `server/dist/...` 배치를 전제로 움직인다.
4. 루트 `dist/`로 재배치하려면 설정, 데이터, 인증서, secret, workspace 파일, shell integration 경로까지 같이 정리해야 한다.

즉, 지금 바로 루트 `dist/`로 가면 `start.js` 하나 만드는 일이 아니라 **배포 경로 정규화 작업**이 된다.

## 대안 비교

### 대안 A. 권장: `server/dist`를 운영 런타임 루트로 유지

구조:

- `server/dist/**` : 서버 JS
- `server/dist/public/**` : 프런트 정적 자산
- `server/config.json5`, `server/data/**`, `server/certs/**` 유지
- 루트 `start.js`는 `cwd=server`로 서버 시작

장점:

- 현재 서버 구조와 가장 잘 맞는다.
- `config.json5`, `certs`, `shell-integration`, `data` 경로를 거의 건드리지 않아도 된다.
- `start.js` 도입 범위가 명확하다.
- 운영 실행 시 실제로 필요한 파일이 `server` 하위에 모인다.

단점:

- 엄밀한 의미의 "루트 단일 dist"는 아니다.
- 프런트/서버 빌드 파이프라인을 한 번 묶어줘야 한다.

평가:

- **지금 코드베이스에서 가장 현실적이고 안전한 선택**

### 대안 B. `frontend/dist`와 `server/dist`를 따로 유지하고 서버가 sibling 경로를 읽게 만들기

구조:

- `frontend/dist/**`
- `server/dist/**`
- 서버 production static dir -> `../frontend/dist`

장점:

- 프런트 기본 빌드 구조를 유지하기 쉽다.
- 복사 단계가 없어도 된다.

단점:

- 운영 산출물이 둘로 나뉜다.
- 배포/압축/이동 시 실수 가능성이 크다.
- `start.js`가 아니라 운영 배포 문서가 더 복잡해진다.

평가:

- 동작은 가능하지만, 장기적으로 덜 깔끔하다.

### 대안 C. 루트 `dist/` 하나로 완전 통합

구조 예시:

- `dist/index.js`
- `dist/public/**`
- `dist/config.json5`
- `dist/data/**`
- `dist/certs/**`

장점:

- 보기에는 가장 단순하다.

단점:

- 현재 코드 기준으로 경로 관련 회귀 위험이 가장 크다.
- `process.cwd()`와 `server` 루트 가정이 섞여 있어서 바로 옮기기 어렵다.
- 사실상 "운영 패키징 리팩터링" 과제다.

평가:

- **지금 바로 선택할 방법은 아님**

## 권장 구현 방식

### 1. 서버가 production에서 정적 파일을 서빙하게 만든다

필요 변경:

- `server/src/index.ts`

개념:

- `NODE_ENV === 'production'`일 때
  - `server/dist/public`를 `express.static`으로 서빙
  - `/api`, `/health`, `/ws` 외 일반 요청은 `public/index.html`로 fallback

이 변경이 들어가야 `https://localhost:2002/`에서 실제 UI가 열린다.

### 2. 프런트 빌드 결과를 `server/dist/public`에 넣는다

권장 방법은 두 가지가 있다.

#### 방법 2-A. 안전 우선: `frontend/dist` 빌드 후 복사

흐름:

1. `cd frontend && npm run build`
2. `server/dist/public` 비우기
3. `frontend/dist` -> `server/dist/public` 복사

장점:

- 기존 Vite 설정을 거의 건드리지 않는다.
- 프런트 개발/프리뷰 흐름을 유지하기 쉽다.
- 배포용 조립 책임이 루트 build 스크립트에 모인다.

평가:

- **초기 도입에 가장 안전**

#### 방법 2-B. 단순화 우선: Vite outDir을 직접 `server/dist/public`으로

예:

- production 빌드 시만 `outDir=../server/dist/public`

장점:

- 복사 단계가 없다.

단점:

- Vite 설정이 운영 배치와 더 강하게 결합된다.
- `outDir` 정리 정책을 잘못 잡으면 원치 않는 삭제 위험이 생긴다.

평가:

- 안정화 후 고려 가능

### 3. 루트 `start.js`는 서버를 `cwd=server`로 실행한다

중요 포인트:

- `process.cwd()` 의존 코드가 있기 때문에 `start.js`는 서버 child process를 `server` 디렉토리에서 띄워야 한다.

의도하는 형태:

```js
spawn(process.execPath, ['dist/index.js'], {
  cwd: path.join(__dirname, 'server'),
  env: { ...process.env, NODE_ENV: 'production', PORT: '2002' },
  stdio: 'inherit',
});
```

이렇게 해야 다음이 기대한 위치를 유지한다.

- `server/data/workspaces.json`
- `server/data/totp.secret`
- `server/config.json5`

### 4. 장기적으로는 상대경로 해석을 서버 루트 기준으로 통일하는 것이 좋다

당장 필수는 아니지만, 운영 경로를 더 튼튼하게 만들려면 다음은 정리하는 편이 맞다.

- `WorkspaceService`의 `dataPath` 해석을 `process.cwd()`가 아니라 `server root` 기준으로
- `TOTPService` secret 경로도 `process.cwd()`가 아니라 `server root` 기준으로

이 작업이 끝나면 나중에 루트 `dist/` 통합이나 패키징도 쉬워진다.

## 추천 실행 플로우

### 빌드

권장 초기안:

1. `server` 빌드
2. `frontend` 빌드
3. `frontend/dist`를 `server/dist/public`으로 복사

### 시작

1. 루트에서 `node start.js`
2. `start.js`가 내부적으로 `cwd=server`, `NODE_ENV=production`으로 `node dist/index.js` 실행

### 검증

운영 검증 포인트:

1. `curl -k https://localhost:2002/health` -> `200`
2. `https://localhost:2002/` 접속 시 로그인 화면 렌더링
3. 정적 자산 `/assets/*.js`, `/assets/*.css` 응답 확인
4. 로그인 후 `/ws` 연결 확인
5. workspace 저장이 `server/data/workspaces.json`에 기록되는지 확인

## 최종 권고

질문에 대한 직접 답:

- **예, `start.js` 기반 운영 실행을 하려면 빌드가 필요하다.**
- 하지만 **지금은 루트 단일 `dist/`보다 `server/dist` 중심 구조가 더 좋다.**
- 가장 좋은 방법은:
  - 서버는 `server/dist`
  - 프런트는 `server/dist/public`
  - 서버는 production에서 정적 서빙
  - 루트 `start.js`는 `cwd=server`로 서버 시작

즉, 지금 단계의 최적해는 **"하나의 운영 런타임 기준점은 `server`로 두고, 그 안의 `dist`에 프런트 자산까지 모으는 방식"**이다.

## 부록: 검증 중 확인한 운영상 주의점

- production 서버 기동 검증은 런타임 데이터 파일을 실제로 건드린다.
- 특히 workspace 복구 로직 때문에 `server/data/workspaces.json` 계열 파일이 갱신될 수 있다.
- 따라서 이후 실제 구현/검증 때는 전용 테스트 데이터 경로나 임시 설정을 두는 편이 안전하다.
