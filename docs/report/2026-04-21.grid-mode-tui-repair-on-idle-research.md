# Grid Mode TUI 깨짐과 Idle 시 1회 복구 아이디어 연구

작성일: 2026-04-21  
대상: Grid mode에서 `codex`, `hermes` 같은 TUI가 줄바꿈/라인 배치가 깨졌다가 창 크기 조절 후 정상화되는 현상

## 결론 요약

- 현상은 단순 React 재렌더 문제라기보다 `fit -> PTY resize -> authoritative snapshot replay` 경로와 더 강하게 연결된다.
- 사용자가 그리드 셀 크기를 살짝 조절하면, 프런트가 새 cols/rows를 측정하고 서버 PTY에 resize를 보내며, 서버는 quiet window 뒤 authoritative snapshot 재전송을 유도한다.
- 프런트는 이 authoritative snapshot을 받을 때 기존 화면을 `term.reset()` 후 통째로 다시 써서, 깨졌던 TUI가 깨끗하게 복구된다.
- 따라서 `idle 상태에 들어갔을 때 한 번씩 렌더링`이라는 아이디어는 방향은 맞지만, 단순 렌더나 단순 React 리렌더로는 약하다.
- 효과가 있는 방식은 “리사이즈가 유발하는 복구 경로와 동등한 복구”여야 한다.

## 문제 현상 정리

- Grid mode에서만 TUI의 줄바꿈, 라인 정렬, status line 배치가 종종 깨진다.
- 같은 세션이 tab mode에서는 상대적으로 덜 깨지거나 티가 적다.
- Grid 셀의 경계를 살짝 움직여 크기를 다시 계산하게 만들면 화면이 정상화된다.

이 패턴은 CSS repaint만으로 설명되기보다, 터미널 geometry와 authoritative replay의 재동기화로 설명하는 편이 더 타당하다.

## 현재 코드 기준 리사이즈 복구 경로

### 1. Grid mode는 절대 배치 runtime layer 위에 터미널을 렌더링한다

- Host slot이 실제 그리드 셀의 `getBoundingClientRect()`를 측정한다.
- 측정 결과는 runtime layer에 전달되어 절대 좌표 기반으로 터미널이 다시 배치된다.

관련 코드:

- [TerminalHostSlot.tsx](../../frontend/src/components/Terminal/TerminalHostSlot.tsx)
- [TerminalRuntimeLayer.tsx](../../frontend/src/components/Terminal/TerminalRuntimeLayer.tsx)
- [TerminalRuntimeContext.tsx](../../frontend/src/components/Terminal/TerminalRuntimeContext.tsx)

즉, Grid mode는 tab mode보다 한 단계 더 많은 layout indirection이 있다.

## 2. TerminalView는 ResizeObserver에서 `fit()`을 수행한다

핵심 경로:

- [TerminalView.tsx](../../frontend/src/components/Terminal/TerminalView.tsx) `ResizeObserver`
- `fitAddon.fit()`
- `emitResize(term.cols, term.rows, 'resize-observer')`

요약:

1. DOM 크기가 바뀌면 `fitAddon.fit()`이 새 cols/rows를 계산
2. 100ms debounce 뒤 서버로 resize 메시지 전송

이 단계에서만 해도 클라이언트 쪽 줄바꿈/viewport 계산이 다시 맞춰질 수 있다.

## 3. 서버는 PTY 자체를 resize하고 replay refresh를 예약한다

핵심 경로:

- [SessionManager.ts](../../server/src/services/SessionManager.ts) `resize()`
- [SessionManager.ts](../../server/src/services/SessionManager.ts) `scheduleResizeReplayRefresh()`

요약:

1. `pty.resize(cols, rows)`
2. headless terminal도 같은 geometry로 resize
3. pending replay 상태를 만들고 quiet window 이후 snapshot refresh 예약

특히 중요:

- 서버는 output이 잠잠해지고 headless write가 drain될 때까지 기다린 뒤 refresh를 보낸다.
- 즉 “resize 직후의 어수선한 중간 상태”가 아니라, 어느 정도 안정화된 화면을 snapshot으로 다시 보낸다.

## 4. WsRouter는 authoritative snapshot을 다시 보낸다

핵심 경로:

- [WsRouter.ts](../../server/src/ws/WsRouter.ts) `refreshReplaySnapshots()`

주의할 점:

- 이 함수는 **이미 replay pending 상태인 세션**에만 동작한다.
- 즉, 현재 구현상 `status=idle`이 되었다고 그냥 호출한다고 동작하지 않는다.
- resize가 먼저 replay pending 상태를 만들고, 그 위에서만 refresh가 성립한다.

이 점은 “idle 때 한 번 렌더” 아이디어를 바로 넣기 어려운 구조적 이유다.

## 5. 프런트는 snapshot을 받을 때 화면을 통째로 재구성한다

핵심 경로:

- [TerminalContainer.tsx](../../frontend/src/components/Terminal/TerminalContainer.tsx) `handleScreenSnapshot()`
- [TerminalView.tsx](../../frontend/src/components/Terminal/TerminalView.tsx) `replaceWithSnapshot()`

특히 `replaceWithSnapshot()`는:

1. `restorePendingRef`를 켠다
2. `term.reset()`을 호출한다
3. snapshot 전체를 다시 `term.write()` 한다
4. 완료 후 `requestViewportSync(term, true)`로 `fit()`까지 동반한다

즉, 사용자가 체감하는 “창 크기 조절 후 복구”는 사실상:

- geometry 재측정
- PTY resize
- authoritative snapshot 전체 재적용

의 복합 효과다.

## 왜 Grid mode에서 더 잘 터지나

가능성이 높은 이유:

1. Host slot 측정과 runtime layer 절대 배치가 추가된다.
2. 초기/중간 레이아웃이 잠깐 흔들릴 때 `fit()`이 최종 크기가 아닌 중간 크기로 수행될 수 있다.
3. TUI는 status line, wide char, wrapped line, alt-screen에 민감해서 geometry mismatch가 즉시 눈에 띈다.

즉, tab mode보다 grid mode가 geometry race를 더 잘 드러내는 구조다.

## `idle 때 한 번 렌더링` 아이디어 평가

### 1. 단순 React 재렌더

효과 낮음.

이 문제의 주된 복구 경로는 React rerender가 아니라:

- xterm `fit()`
- PTY resize
- authoritative snapshot replay

이기 때문이다.

### 2. `fit()`만 한 번 더 실행

중간 정도.

클라이언트 측 측정 오차가 원인이라면 효과가 있을 수 있다.  
하지만 서버 PTY와 headless snapshot이 이미 잘못된 geometry 기준으로 진행된 상태면 부족하다.

### 3. authoritative snapshot만 한 번 다시 적용

효과 높음.

현재 코드에서 실제 화면 재구성은 `replaceWithSnapshot()`에서 일어나므로, 이 경로는 매우 유력하다.  
다만 현재 구조상 resize 없이 snapshot refresh만 보내는 경로가 없다.

### 4. 리사이즈 복구 과정을 의도적으로 재현

가장 유력.

사용자가 실제로 문제를 해결하는 행위가 “살짝 창 크기 조절”이라면, 그 과정을 동일하게 구현하는 것이 가장 자연스럽다.

다만 현재 구조에서는 몇 가지 선택지가 있다.

## 구현 후보 비교

### 후보 A. Idle 진입 후 `fit()`만 1회 실행

장점:

- 변경 범위가 작다
- 서버 계약을 거의 건드리지 않는다

단점:

- authoritative snapshot replay가 없어서 복구력이 약할 수 있다
- 서버 PTY가 계속 잘못된 geometry로 그리는 경우 해결되지 않는다

평가:

- 1차 실험용으로는 가능
- “사용자 수동 리사이즈와 동일”한 복구는 아니다

### 후보 B. Synthetic resize를 강제로 발생시킨다

의미:

- 실제 사용자 리사이즈와 같은 효과를 내기 위해 cols/rows 변화를 의도적으로 만들어 server `resize()` 경로를 타게 함

장점:

- 현재 문제를 해결하는 실제 경로와 가장 비슷하다

단점:

- geometry를 진짜로 바꾸는 부작용이 있다
- TUI가 SIGWINCH에 민감하게 반응할 수 있다
- 1칸 증감 같은 인위적 resize는 또 다른 flicker를 만들 수 있다

평가:

- “완전히 같은 과정”을 복제하려면 가장 직접적
- 하지만 안정성 리스크가 있다

### 후보 C. Idle 진입 시 “repair replay” 전용 경로를 만든다

의미:

- 실제 geometry는 유지
- replay pending을 별도로 만들고
- 현재 authoritative snapshot을 다시 보내
- 프런트가 `replaceWithSnapshot()`으로 화면을 재구성하도록 함

장점:

- 사용자가 체감하는 복구 효과는 크다
- geometry를 인위적으로 흔들지 않는다
- 서버와 프런트 모두 현재 snapshot 기반 구조를 재사용할 수 있다

단점:

- 현재 `WsRouter.refreshReplaySnapshots()`는 pending replay가 있는 경우에만 동작하므로, 별도 repair protocol이 필요하다
- 단순 “idle 이벤트 왔다”만으로는 현재 구조에 끼워 넣을 수 없다

평가:

- 실무적으로는 가장 안정적인 방향
- “리사이즈와 동일한 복구 효과”를 geometry mutation 없이 재현하는 대안

## 중요한 구조적 제약

### 제약 1. 같은 cols/rows로는 server resize 경로가 사실상 skip된다

[SessionManager.resize()](../../server/src/services/SessionManager.ts)는 cols/rows가 같으면 `resize_skipped`로 끝난다.

즉:

- 그냥 `fit()`을 다시 해서 결과 cols/rows가 동일하면
- 서버는 PTY resize도 안 하고
- replay refresh도 안 예약한다

이 점 때문에 단순 idle fit은 수동 리사이즈와 동일하지 않다.

### 제약 2. 현재 replay refresh는 “pending replay” 상태가 필요하다

[WsRouter.refreshReplaySnapshots()](../../server/src/ws/WsRouter.ts)는 subscriber가 있어도 replay pending이 없으면 그냥 지나간다.

즉:

- idle 상태 진입
- status event 수신

만으로는 현재 snapshot 재적용이 일어나지 않는다.

## 추천 방향

사용자 가설:

> 그리드모드에서 창 크기를 살짝 조절하여 다시 랜더링 되는 과정을 심층 분석하여 이것을 동일하게 구현하면 될 것 같다

이 판단은 타당하다.

다만 “동일하게 구현”의 의미를 아래처럼 잡는 것이 안전하다.

### 추천 1순위

**Idle 진입 후 quiet window를 둔 뒤, geometry mutation 없는 repair replay를 수행한다.**

즉:

1. `idle` 진입 감지
2. 짧은 quiet window 대기
3. grid 모드 + TUI 세션에 한해 repair replay 시작
4. 현재 authoritative snapshot을 다시 적용
5. 적용 후 `fit()` 동반

이 방향은 실제 수동 리사이즈의 “화면 재구성 효과”를 가장 안전하게 재현한다.

### 추천 2순위

**1차 실험으로는 `idle` 진입 후 `fit()` 1회만 수행해본다.**

만약 이 정도로도 복구 비율이 높다면 구현이 단순하다.

다만 geometry mismatch가 근본 원인이면 한계가 있을 가능성이 크다.

### 추천 3순위

**Synthetic resize는 마지막 수단으로 검토한다.**

이건 사용자 수동 리사이즈를 가장 그대로 복제하지만, TUI side effect 리스크가 가장 크다.

## 다음 단계 제안

코드 작성 전에 아래 실험을 먼저 설계하는 것이 좋다.

1. 깨진 상태에서 `fit()`만 실행했을 때 복구되는가
2. 깨진 상태에서 authoritative snapshot만 재적용했을 때 복구되는가
3. 위 둘로 안 되면 실제 1칸 resize가 필요한가
4. 서버 snapshot 자체가 이미 깨져 있는지, 아니면 클라이언트 렌더만 깨져 있는지 확인

이 네 가지가 분리되면 구현 방향이 명확해진다.

## 최종 판단

- `idle 진입 시 한 번 손본다`는 아이디어는 유효하다.
- 하지만 “손본다”의 실체는 단순 렌더가 아니라, 현재 사용자가 수동 리사이즈로 유발하는 복구 경로와 가까워야 한다.
- 가장 현실적인 목표는:

**Grid mode TUI 세션에서 idle 진입 후 1회 authoritative repair replay를 수행해, 수동 리사이즈가 주는 복구 효과를 geometry mutation 없이 재현하는 것**

이다.
