# 탭/그리드 터미널 재활용 및 `react-mosaic` 통합 연구 보고서

작성일: 2026-04-14  
작성자: Codex  
조사 참여: Bacon, Hegel, Halley

## 1. 연구 목적

이번 조사의 질문은 두 가지였다.

1. 탭 모드 터미널을 그리드 모드에서, 혹은 그 반대로 **같은 터미널 인스턴스로 재활용**할 수 있는가
2. 이 문제를 풀기 위해 **탭 모드까지 `react-mosaic` 위에 통일**하는 것이 현실적인가

결론부터 말하면:

- **가능은 하다.**
- 다만 핵심은 "`react-mosaic`를 쓰느냐"가 아니라 **누가 터미널 인스턴스를 소유하느냐**다.
- `react-mosaic`를 탭 모드까지 통일하는 방향은 의미가 있지만, **탭 모드를 단순히 single-pane mosaic로 바꾸는 것만으로는 문제를 해결하지 못한다.**
- 가장 현실적인 방향은 **세션당 단일 터미널 런타임 + layout host layer 분리**이고, 그 위에서 `react-mosaic`를 **canonical layout engine**으로 쓰는 것이다.

## 2. 현재 구조 요약

### 2.1 탭 모드와 그리드 모드는 서로 다른 소유 트리다

현재 `App.tsx`는 `viewMode === 'grid'`일 때 `MosaicContainer`만 렌더하고, `viewMode === 'tab'`일 때는 전혀 다른 절대배치 터미널 스택을 렌더한다.

- `grid` 분기: `frontend/src/App.tsx:399-415`
- `tab` 분기: `frontend/src/App.tsx:418-472`

탭 모드에서는 모든 탭의 `TerminalContainer`를 한 번에 렌더한 뒤, 활성 탭만 `visibility: visible`로 두고 나머지는 숨긴다.

- `frontend/src/App.tsx:420-470`

반면 그리드 모드에서는 `MosaicContainer -> MosaicTile -> renderTerminal(tab)` 경로로 다시 `TerminalContainer`를 생성한다.

- `frontend/src/App.tsx:303-320`
- `frontend/src/components/Grid/MosaicContainer.tsx:357-390`

즉 현재는 같은 `tab.id/sessionId`라도 탭 모드와 그리드 모드가 **서로 다른 React subtree에서 따로 소유**한다.

### 2.2 `TerminalView`는 mount/unmount 수명주기에 강하게 묶여 있다

실제 xterm 인스턴스는 `TerminalView` 안에서 생성되고 해제된다.

- 생성: `frontend/src/components/Terminal/TerminalView.tsx:504-557`
- `ResizeObserver`/focus/paste binding: `frontend/src/components/Terminal/TerminalView.tsx:610-666`
- dispose: `frontend/src/components/Terminal/TerminalView.tsx:668-698`

즉 현재 구조에서 모드 전환은 "같은 터미널을 옮긴다"가 아니라, 실제로는 **터미널을 새로 만들고 snapshot/replay로 복원한다**에 가깝다.

### 2.3 프런트는 세션당 UI consumer 하나를 사실상 가정한다

서버는 세션당 여러 subscriber를 허용한다.

- `server/src/ws/WsRouter.ts:157-165`
- `server/src/ws/WsRouter.ts:422-452`

하지만 프런트 `WebSocketContext`는 `sessionId -> handlers`를 단일 `Map`으로 가진다.

- `frontend/src/contexts/WebSocketContext.tsx:88-91`
- `frontend/src/contexts/WebSocketContext.tsx:127-152`
- `frontend/src/contexts/WebSocketContext.tsx:264-312`

따라서 같은 브라우저 안에서 같은 세션의 UI 인스턴스를 탭/그리드 양쪽에 **동시에 두 개 살아 있게 두면**, 마지막으로 등록한 핸들러가 이전 핸들러를 덮어쓴다.

이 점 때문에 현재 구조에서는 "동시 다중 UI"보다 **단일 mounted consumer를 이동/재배치**하는 쪽이 더 맞다.

### 2.4 서버 계약은 `session -> screen snapshot -> ack -> queued output flush`다

서버의 권위 상태는 브라우저 xterm이 아니라 `SessionManager`의 headless terminal이다.

- 세션 headless 상태 보유: `server/src/services/SessionManager.ts:207-236`
- snapshot 생성: `server/src/services/SessionManager.ts:941-1015`
- headless 출력 적용 및 `screenSeq` 증가: `server/src/services/SessionManager.ts:1144-1190`

구독 시 `WsRouter`는:

1. `screen-snapshot` 전송
2. 클라이언트 ack(`screen-snapshot:ready`) 대기
3. 그동안 live output은 `queuedOutput`에 적재
4. ack 이후 flush

- subscribe snapshot 전송: `server/src/ws/WsRouter.ts:182-238`
- ack 소비 및 flush: `server/src/ws/WsRouter.ts:260-302`
- replay pending 중 output queue: `server/src/ws/WsRouter.ts:422-452`

이 계약 때문에 UI 인스턴스를 새로 만들면, 단순 visibility 변경이 아니라 **subscribe/replay 경계 전체를 다시 밟게 된다.**

## 3. 현재 구조에서 재활용이 어려운 직접 원인

### 3.1 React가 같은 인스턴스로 보지 않는다

`TerminalContainer`는 `memo`이지만, 같은 부모 subtree 안에서만 의미가 있다.

- `frontend/src/components/Terminal/TerminalContainer.tsx:20-22`

지금은 `tab` subtree와 `grid` subtree가 구조적으로 갈라져 있으므로, key가 같아도 모드 전환 시 React는 **같은 인스턴스의 이동**이 아니라 **한쪽 unmount + 다른 쪽 mount**로 처리한다.

### 3.2 레이아웃 엔진이 터미널 인스턴스를 모르고 leaf id만 안다

`MosaicContainer`와 `useMosaicLayout`은 `tabId` 기준으로 tree를 복원하고 고친다.

- `frontend/src/hooks/useMosaicLayout.ts:61-117`
- `frontend/src/utils/mosaic.ts:231-273`

즉 그리드 시스템은 "터미널 런타임"이 아니라 "leaf id를 어디에 놓을 것인가"만 안다.

그래서 현재 구조는 **layout-first**이고, **terminal-runtime-first**가 아니다.

### 3.3 지금의 안정화 장치들도 사실상 handoff 보정이다

최근 넣은 `subscribeSession`의 unsubscribe grace, duplicate snapshot ignore, hidden resize 차단 등은 모두 "재생성은 하되 덜 깨지게 하는" 장치다.

- unsubscribe grace: `frontend/src/contexts/WebSocketContext.tsx:62`, `264-312`
- duplicate snapshot ignore: `frontend/src/components/Terminal/TerminalContainer.tsx:104-118`
- hidden fit/resize 차단: `frontend/src/components/Terminal/TerminalView.tsx:105-162`, `702-719`

즉 현재 버그 수정들은 유효하지만, **근본적으로 same-instance reuse를 구현한 것은 아니다.**

## 4. `react-mosaic`를 탭 모드에도 통일하는 발상 평가

사용자 제안의 핵심은 합리적이다.  
탭 모드와 그리드 모드가 서로 다른 렌더 트리이기 때문에 계속 handoff 버그가 나왔으므로, 둘 다 같은 layout system 위에 올리면 문제가 줄어들 수 있다.

다만 이 아이디어는 두 방식으로 나뉜다.

### 4.1 나쁜 버전: 탭 모드를 "single-pane mosaic"로 매번 바꿔치기

예를 들어:

- 그리드 모드에서는 실제 split tree 사용
- 탭 모드로 가면 tree를 active tab 하나만 남긴 single leaf로 바꿈
- 다시 그리드 모드로 가면 이전 tree를 복원

이 방식은 추천하지 않는다.

이유:

- 모드 전환 때마다 canonical tree를 파괴하거나 별도 보관해야 한다
- 현재 `useMosaicLayout`/`restoreLayoutWithSessionRecovery`는 `workspaceId + currentTabIds` 기준 persisted tree를 다시 읽고 고친다  
  `frontend/src/hooks/useMosaicLayout.ts:61-117`, `frontend/src/utils/mosaic.ts:231-273`
- 탭 모드가 tree를 single leaf로 바꾸면, 그리드 레이아웃을 잃거나 동기화 비용이 커진다
- 본질적으로는 여전히 "모드 전환 때 다른 tree"를 다룰 뿐이라, 터미널 인스턴스 소유 문제를 해결하지 못한다

정리하면, **mosaic를 탭 UI에 억지로 맞추는 것**은 큰 이득이 없다.

### 4.2 좋은 버전: `react-mosaic`를 canonical layout model로 두고, 탭 모드는 그 tree의 projection으로 본다

이 방식은 다르다.

- workspace마다 canonical split tree는 계속 유지
- 그리드 모드는 canonical tree를 그대로 렌더
- 탭 모드는 canonical tree를 파괴하지 않고, active tab만 전면에 보이게 하는 **projection layer**로 본다
- 탭 바는 여전히 `WorkspaceTabBar`가 담당
- `react-mosaic`는 "layout state 저장소"와 "grid host 생성기" 역할을 맡는다

이 방식의 장점:

- 탭/그리드가 서로 다른 세계가 아니라 같은 workspace layout model 위의 두 가지 표현이 된다
- 그리드 레이아웃 persisted state를 모드 전환이 파괴하지 않는다
- focus history, close semantics, leaf identity를 한 군데에서 유지할 수 있다

하지만 이것만으로는 충분하지 않다.

**결정적으로 필요한 것은 터미널 런타임의 소유권 분리다.**

## 5. 핵심 설계 옵션 비교

### 옵션 A. 현재 구조 유지 + handoff 계속 개선

설명:

- 지금처럼 탭/그리드에서 각자 `TerminalContainer`를 렌더
- snapshot/replay/subscribe grace/resize dedupe를 계속 보강

장점:

- 가장 작은 변경
- 지금까지 쌓은 안정화 코드와 잘 맞음

단점:

- 터미널 재생성이 계속 남음
- TUI, scrollback, input race, snapshot/ack 경계 문제가 구조적으로 재발 가능

평가:

- 단기 대응은 가능
- 이번 요구사항의 본질적 답은 아님

### 옵션 B. 세션당 단일 `TerminalContainer` registry + tab/grid는 host slot만 제공

설명:

- `tab.id` 또는 `sessionId` 기준 registry가 **유일한 live terminal runtime**을 소유
- `App` 상단 또는 별도 layer가 모든 `TerminalContainer`를 유지
- tab mode와 grid mode는 terminal을 직접 렌더하지 않고, "이 terminal이 지금 붙어야 할 DOM host"만 제공
- 필요하면 terminal surface를 absolute overlay로 옮기거나 host rect 기준으로 배치

장점:

- 프런트의 "세션당 consumer 하나" 계약과 맞다
- subscribe/snapshot/ack 경계를 모드 전환 때 다시 밟지 않는다
- 이번에 반복된 black screen, duplicate replay, resize churn의 구조적 원인을 줄인다

단점:

- host registry, slot registration, rect sync, focus handoff 구현이 필요
- React 선언형 렌더링만으로 끝나지 않고 약간의 imperative layer가 필요

평가:

- **가장 현실적인 권고안**

### 옵션 C. `react-mosaic`를 workspace canonical layout engine으로 승격 + tab mode는 projection

설명:

- `gridTree`는 항상 존재
- tab mode는 tree를 파괴하지 않고 active tab projection만 표시
- 옵션 B의 registry/host layer와 결합

장점:

- layout model이 하나로 정리된다
- grid persistence와 tab selection이 충돌하지 않는다
- 장기적으로 tab/grid 전환 코드를 가장 깨끗하게 만들 수 있다

단점:

- `react-mosaic`가 workspace 표현의 핵심 의존성이 된다
- 현재도 React 19 경고(`element.ref`)가 있어 라이브러리 blast radius가 커진다
- tab UI 특성(순서, active tab bar, 빠른 전환, 색상, rename)이 mosaic 개념과 1:1로 맞지 않는다

평가:

- **좋은 장기 방향**
- 단, 옵션 B 없이 옵션 C만으로는 부족하다

### 옵션 D. pure DOM reattach

설명:

- xterm wrapper DOM 자체를 `appendChild`로 옮김
- React는 host만 제공하고 terminal DOM 실체는 imperative하게 이동

장점:

- same-instance reuse가 가장 직접적이다

단점:

- `TerminalView`가 현재 `term.open`, observer, focus/paste handler를 mount lifecycle에 묶고 있어서 분리가 크다  
  `frontend/src/components/Terminal/TerminalView.tsx:552-698`
- 디버깅 난이도가 높다

평가:

- 가능하지만 초기 권고안으로는 과하다

## 6. `react-mosaic`를 탭 모드까지 쓰는 것이 주는 실제 이점

이 질문에 대한 답은 "예, 다만 어디에 쓰느냐가 중요하다"이다.

### 유의미한 이점

1. **workspace layout state의 단일화**
   - 지금은 탭 모드와 그리드 모드가 구조적으로 다르다
   - mosaic를 canonical tree로 두면 "현재 workspace가 가진 pane topology"를 한 곳에서 관리할 수 있다

2. **모드 전환을 tree 파괴가 아니라 view projection으로 바꿀 수 있다**
   - 이것이 터미널 재사용 전략과 잘 맞는다

3. **close/remove/add/restart가 leaf identity 기준으로 정리된다**
   - 현재도 `removeFromMosaicTree`, `restoreLayoutWithSessionRecovery` 같은 유틸이 이미 존재한다  
     `frontend/src/utils/mosaic.ts:24-42`, `231-273`

### 함정

1. **탭 UI는 split layout과 다른 문제를 가진다**
   - 탭 순서
   - active tab bar
   - 드래그 없는 빠른 전환
   - color badge / rename / shell add UX

2. **mosaic 하나로 모든 UX를 대체하려고 하면 오히려 탭 UX가 나빠진다**

3. **라이브러리 리스크가 커진다**
   - 현재 `react-mosaic-component`는 React 19 경고가 이미 관측됐다
   - 지금은 grid 경로에 국한되지만, tab mode까지 올리면 전체 workspace shell이 그 리스크를 공유한다

정리하면:

- **탭 UI를 mosaic처럼 보이게 만드는 것**은 별 이득이 없다
- **workspace layout state를 mosaic 계열 모델로 통일하는 것**은 이득이 있다

## 7. 권고안

### 최종 권고

가장 현실적인 구조는 아래 조합이다.

1. **세션당 단일 terminal runtime registry를 도입한다**
2. **tab/grid는 terminal을 직접 생성하지 않고 host slot만 제공한다**
3. **workspace canonical layout state는 mosaic tree로 유지한다**
4. **tab mode는 canonical tree를 파괴하지 않는 projection으로 구현한다**
5. **`WorkspaceTabBar`는 계속 유지한다**

즉, 한 문장으로 쓰면:

> `react-mosaic`를 탭 모드에 "UI로 재사용"하기보다, workspace의 단일 layout state로 승격하고, 실제 터미널 재사용은 registry/host-layer가 담당하게 만드는 것이 맞다.

### 왜 이 방향이 현재 코드와 가장 잘 맞는가

- 현재 `terminalRefsMap`이 이미 터미널 핸들 registry의 씨앗 역할을 하고 있다  
  `frontend/src/App.tsx:55`
- 현재 `TerminalView`는 `isVisible`과 `fit/resize` 분리를 이미 어느 정도 갖고 있다  
  `frontend/src/components/Terminal/TerminalView.tsx:105-162`, `702-719`
- 현재 서버/WS 계약은 세션당 권위 상태 하나, ack 책임 하나에 맞춰져 있다  
  `server/src/ws/WsRouter.ts:205-302`

즉 큰 틀을 뒤집는 것이 아니라, **지금 코드가 이미 가진 경향을 명확한 아키텍처로 완성하는 방향**이다.

## 8. 실행 관점의 제안

구현 순서는 아래가 맞다.

1. `TerminalRuntimeRegistry` 설계
   - `tabId/sessionId -> runtime entry`
   - WS consumer와 xterm lifecycle을 여기로 이동

2. `TerminalHostSlot` 설계
   - tab active host
   - grid pane host
   - hidden parking host

3. `App.tsx`에서 tab/grid가 `TerminalContainer`를 직접 렌더하지 않도록 변경

4. `MosaicContainer`는 terminal JSX를 받지 않고, leaf host descriptor만 그리도록 축소

5. `useMosaicLayout`는 grid canonical tree 전용으로 유지
   - tab mode는 tree를 바꾸지 않고 active projection만 바꿈

## 9. 최종 판단

질문에 대한 직접 답을 짧게 적으면 이렇다.

- **예, `react-mosaic`를 활용해 탭 모드까지 하나의 체계로 묶는 것은 충분히 검토할 가치가 있다.**
- **하지만 그 자체가 해답은 아니다.**
- 진짜 핵심은 **터미널 인스턴스를 layout tree 밖으로 끌어내어 세션당 하나만 살리고, tab/grid는 그걸 어디에 보여줄지만 바꾸는 구조**다.
- 그 위에서 `react-mosaic`를 workspace의 canonical layout model로 쓰면, 지금의 반복적인 handoff/replay/black-screen 계열 문제를 가장 설계적으로 줄일 수 있다.
