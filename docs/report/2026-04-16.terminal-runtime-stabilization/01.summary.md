# Summary

## Problem History

The work started from three user-visible failures:

1. Workspace switch or reconnect could collapse terminal scrollback to a recent tail.
2. `tab <-> grid` mode switches could recreate or visually corrupt live terminals during active output.
3. Grid mode side effects appeared after stabilization work, especially missing pane focus behavior and broken splitter interaction.

During repeated real-browser validation, the team also found Windows-specific redraw issues involving `PowerShell`, `Codex`, `ConPTY`, and `xterm` behavior under repeated resize and remount conditions.

## Main Technical Changes

### 1. Server-side replay authority hardening

The server replay path was moved away from naive client-local restoration and toward server-owned screen recovery.

Relevant code:

- [SessionManager.ts](../../../server/src/services/SessionManager.ts)
- [WsRouter.ts](../../../server/src/ws/WsRouter.ts)
- [ws-protocol.ts](../../../server/src/types/ws-protocol.ts)

Key outcomes:

- replay refresh now accounts for resize, redraw, and headless-write drain timing
- no-op resize does not trigger unnecessary replay work
- replay-pending output is better protected from duplication or premature refresh

### 2. Terminal runtime reuse across tab and grid

The biggest structural frontend change was separating terminal runtime ownership from presentation.

Relevant code:

- [App.tsx](../../../frontend/src/App.tsx)
- [MosaicContainer.tsx](../../../frontend/src/components/Grid/MosaicContainer.tsx)
- [TerminalRuntimeContext.tsx](../../../frontend/src/components/Terminal/TerminalRuntimeContext.tsx)
- [TerminalRuntimeLayer.tsx](../../../frontend/src/components/Terminal/TerminalRuntimeLayer.tsx)
- [TerminalHostSlot.tsx](../../../frontend/src/components/Terminal/TerminalHostSlot.tsx)
- [TerminalContainer.tsx](../../../frontend/src/components/Terminal/TerminalContainer.tsx)
- [TerminalView.tsx](../../../frontend/src/components/Terminal/TerminalView.tsx)

Key outcomes:

- tab/grid no longer rely on fully separate terminal instances for the same session
- hidden-host and hostless states are handled more carefully
- terminal focus and host attachment are more stable than the earlier direct-remount model

### 3. Grid interaction recovery

Grid mode lost splitter usability during runtime-layer work, so interaction and layering had to be restored.

Relevant code:

- [MosaicOverrides.css](../../../frontend/src/components/Grid/MosaicOverrides.css)

Key outcomes:

- resize cursor is visible again on the splitter
- splitter drag changes pane geometry again

## Investigation Path

The work followed this progression:

1. Diagnose scrollback loss and snapshot/replay ordering problems.
2. Evaluate server-side canonical replay buffer and then headless-terminal authority approaches.
3. Investigate `Codex`/TUI redraw behavior under repeated resize and workspace bounce.
4. Replace tab/grid terminal duplication with a runtime registry approach.
5. Narrow the remaining issue to `post-resize replay timing`.
6. Harden deadline, quiet-window, drain, and near-deadline rearm behavior.

## Current Assessment

The project is no longer in the original failure state.

What is now in good shape:

- workspace switching
- dedicated PowerShell validation path for `tab -> grid`
- dedicated PowerShell validation path for `grid -> tab`
- grid splitter hover/drag
- server replay timing regression coverage

What is still not claimed as fully solved:

- certain real interactive workloads can still show imperfect first-frame presentation during mode switch
- React 19 still emits an upstream `element.ref` warning from the mosaic stack during browser validation

## Recommended Next Steps

1. Keep using dedicated clean-room workspaces for future replay regressions.
2. If the remaining first-frame issue resurfaces in real workloads, capture paired browser/server logs again and compare against the new scheduler behavior.
3. Separate the React 19 mosaic warning cleanup from the terminal replay work so the remaining runtime noise does not mask real regressions.
