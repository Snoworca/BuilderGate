# Cross-Platform ConPTY Startup Analysis

## 요약

- 문제의 1차 원인은 `ConPTY` 기능 자체가 아니라, 저장소 공용 설정 파일이 윈도우 전용 PTY 옵션을 기본값처럼 들고 있는 점입니다.
- `macOS`/`Linux`에서는 세션 생성 전, 서버 부팅 단계에서 바로 설정 검증이 실행되고 실패합니다.
- `start.sh`/`start.bat`/`dev.js` 자체는 대체로 플랫폼 분기를 하고 있습니다. 실제로 서버를 죽이는 것은 `server/config.json5` 및 `server/config.json5.example`의 값과 부팅 검증의 조합입니다.
- 따라서 핵심 해결책은 `repo default`를 플랫폼 중립으로 되돌리고, 비윈도우에서는 파일 기반 설정을 한 번 더 방어적으로 정규화하는 것입니다.

## 분석 범위

이번 분석은 3개 서브에이전트와 메인 분석을 통합한 결과입니다.

- Agent 1: 기본 설정 파일과 서버 부팅 경로 분석
- Agent 2: `SessionManager`의 PTY capability 검증 및 backend 분기 분석
- Agent 3: `dev.js` / `start.sh` / `start.bat` / runtime launcher의 플랫폼 의존성 분석

## 정확한 원인

### 1. 저장소 기본 설정이 윈도우 전용 값으로 고정됨

현재 실제 설정 파일과 예제 설정 파일이 모두 윈도우 전용 값을 담고 있습니다.

- [server/config.json5](/C:/Work/git/_Snoworca/ProjectMaster/server/config.json5:23): `useConpty: true`
- [server/config.json5](/C:/Work/git/_Snoworca/ProjectMaster/server/config.json5:44): `windowsPowerShellBackend: "conpty"`
- [server/config.json5.example](/C:/Work/git/_Snoworca/ProjectMaster/server/config.json5.example:25): `useConpty: true`

반면 스키마 기본값은 이미 플랫폼 중립입니다.

- [server/src/schemas/config.schema.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/schemas/config.schema.ts:60): `useConpty` 기본값 `false`
- [server/src/schemas/config.schema.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/schemas/config.schema.ts:61): `windowsPowerShellBackend` 기본값 `"inherit"`

즉, 코드 레벨 기본값은 안전한데, 파일 기본값이 그것을 덮어써서 회귀를 만들고 있습니다.

### 2. 설정 파일 자동 생성 경로가 잘못된 기본값을 그대로 복사함

따라서 새 macOS/Linux 사용자는 별도 수정 없이 실행해도 즉시 같은 문제를 밟습니다.

### 3. 부팅 단계 검증이 세션 생성 이전에 전역 설정을 바로 차단함

서버는 실제 셸을 띄우기 전에 전역 PTY 설정을 검사합니다.

이 구조 때문에 실제로 PowerShell을 쓰지 않더라도, 설정 파일 안에 윈도우 전용 PTY 값이 들어 있으면 서버가 기동 자체를 거부합니다.

## 왜 macOS / Linux 에서 실패하는가

핵심은 `shell: "auto"`가 문제가 아니라, 전역 PTY backend 설정이 문제라는 점입니다.

`SessionManager.resolveShell()`은 이미 플랫폼별 기본 셸을 잘 고릅니다.

- Windows: PowerShell
- macOS: 가능하면 `zsh`
- 그 외 Unix: `bash`

관련 코드:

- [server/src/services/SessionManager.ts](/C:/Work/git/_Snoworca/ProjectMaster/server/src/services/SessionManager.ts:860)

하지만 이 셸 분기보다 먼저 `assertRuntimePtyCapabilities()`가 실행되고, 여기서 `useConpty`와 `windowsPowerShellBackend`를 전역 값으로 검사합니다. 그래서 macOS/Linux는 셸 fallback까지 도달하지 못하고 죽습니다.

## 2차 관찰 사항

### A. `dev.js`가 서버 치명 오류 후에도 프런트엔드를 띄움

현재 `dev.js`는 서버가 이미 실패했는지와 무관하게 3초 뒤 프런트엔드를 시작합니다.

- [dev.js](/C:/Work/git/_Snoworca/ProjectMaster/dev.js:89)

이 때문에 사용자는 "프런트는 뜨는데 서버만 이상한가?"처럼 원인을 더 헷갈리게 볼 수 있습니다.

이것은 1차 원인은 아니지만 개발 경험 측면에서 별도 개선 가치가 있습니다.

### B. 플랫폼별 셸 지원 표면이 서로 일치하지 않음

코어 `SessionManager`는 `zsh`, `sh`까지 어느 정도 고려하고 있지만, API / Settings / Frontend 타입은 더 좁게 제한하고 있습니다.

## 권장 해결 방향

### 권장안 1. 저장소 기본 설정을 플랫폼 중립으로 되돌리기

가장 안전하고 즉효인 수정입니다.

대상:

- `server/config.json5.example`
- 저장소에 포함할 기본 `server/config.json5` 정책
- README 예시

권장값:

- `pty.useConpty: false`
- `pty.windowsPowerShellBackend: "inherit"`
- `pty.shell: "auto"`

이렇게 하면:

- Windows: 여전히 실행 가능
- macOS/Linux: 부팅 차단 해소
- 기존 fail-fast validation 유지 가능

단점:

- Windows에서 "기본값이 무조건 ConPTY"는 아니게 됩니다.

### 권장안 2. 파일 로드 시 비윈도우용 정규화 레이어 추가

중장기적으로는 이 방어막이 필요합니다.

의도:

- 저장소의 공용 설정 파일이 플랫폼 특수값을 품고 있어도 비윈도우에서 서버가 벽돌이 되지 않게 함

정규화 규칙 예:

- non-win32면 `pty.useConpty = false`
- non-win32면 `pty.windowsPowerShellBackend = "inherit"`
- non-win32에서 `pty.shell`이 `powershell`/`wsl`/`cmd`면 `auto` 또는 `bash`로 치환하거나 명확히 거절

적용 위치 후보:

- `server/src/utils/config.ts`의 `loadConfig()`
- 또는 `SessionManager` 초기 runtime config 정규화 경계

주의:

- UI/API patch는 계속 엄격하게 거절해도 괜찮지만, 파일 bootstrap만큼은 방어적으로 처리하는 편이 cross-platform repo에는 더 적합합니다.

### 권장안 3. `SessionManager.createSession()` 자체를 방어적으로 만들기

현재는 상위 검증이 막아줄 것을 가정하고 있습니다.

보강 방향:

- non-win32에서는 `node-pty.spawn()`에 `useConpty`를 실어 보내지 않기
- non-win32에서 `powershell` / `wsl` / `cmd` 요청 시 명확한 예외 또는 안전한 fallback

이 수정은 1차 근본 원인 해결 이후의 hardening 단계로 적합합니다.

### 권장안 4. `dev.js`가 backend 실패 시 frontend를 시작하지 않도록 수정

직접 원인은 아니지만, 다음 이유로 추천합니다.

- 치명 오류가 더 빨리 드러남
- 사용자 혼란 감소
- CI / 로컬 디버깅 신뢰도 향상

## 우선순위 제안

1. `server/config.json5.example`과 README 예시를 플랫폼 중립으로 수정
2. 저장소 기본 `server/config.json5` 정책도 플랫폼 중립으로 정리
3. `loadConfig()` 또는 초기 runtime config 경계에 non-Windows 정규화 추가
4. `SessionManager.createSession()` 방어 로직 추가
5. `dev.js` backend 실패 시 frontend 시작 차단
6. 셸 capability / validation / frontend 타입 정합성 정리

## 결론

이번 장애는 `ConPTY` 기능 버그가 아니라, `Windows-only runtime preference`를 저장소 공용 기본 설정으로 커밋한 설계 문제입니다.

따라서 가장 좋은 해결책은 다음 두 축입니다.

- 공용 설정 파일은 플랫폼 중립으로 유지
- 플랫폼 특수 최적화는 런타임에서 조건부 적용

이 방향이면 Windows에서는 계속 `ConPTY` 선호 정책을 살릴 수 있고, macOS/Linux에서는 부팅 실패 없이 동일 저장소를 사용할 수 있습니다.

## 검증 한계

- 이번 분석은 Windows 워크스페이스에서 코드 경로를 기준으로 수행했습니다.
- 실제 macOS/Linux 실행 검증은 아직 하지 않았습니다.
- 다만 예외 발생 경로와 설정 복사 경로가 명확하므로, 원인 판정의 신뢰도는 높습니다.
