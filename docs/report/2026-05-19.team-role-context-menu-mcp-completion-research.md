# 팀장/팀원 지정 컨텍스트 메뉴와 MCP 완료 신호 추가 연구 보고서

대상 프로젝트: `C:\Work\git\_Snoworca\ProjectMaster`  
요청: 우클릭 컨텍스트 메뉴로 세션을 팀장/팀원으로 지정하고, MCP를 사용해 역할 프롬프트와 세션 번호를 전달하며, `.buildergate` JSON에 상태를 저장하고, `idle`이 아니라 명시 MCP/hook 완료 신호로 대화 종료를 판단하는 구조를 2개 서브에이전트로 추가 연구

## 결론

구현 가능하다. 다만 현재 프로젝트에는 팀 역할, 안정 세션키, 작업 배정, 완료 신호 모델이 없으므로 기존 `WorkspaceTab`/`WorkspaceService`에 단순 필드만 추가하는 것보다 별도 팀 세션 레지스트리와 완료 프로토콜을 함께 두는 것이 안전하다.

권장 1차 범위는 다음이다.

```text
터미널 우클릭
  -> 팀장으로 지정 / 팀원으로 지정
  -> WorkspaceTab 또는 TeamSessionRegistry role 갱신
  -> 세션 우측 상단 역할 아이콘 표시
  -> .buildergate/team-sessions.json 저장
  -> 팀원에게 MCP 사용 지시 프롬프트 paste-only 전달
  -> 팀원은 작업 종료 시 submit_assignment_result MCP 호출
  -> hook은 fallback/보강 신호
```

핵심 정책:

- `sessionId`는 PTY runtime id라 재시작/복구 시 바뀔 수 있으므로 장기 식별자로 쓰지 않는다.
- 외부에서 메시지를 보낼 대상은 `sessionKey`로 식별하고, `sessionNumber`는 UI 표시용 alias로만 쓴다.
- `idle`은 완료 신호가 아니다. 완료는 `assignmentId`가 들어간 `submit_assignment_result` MCP tool 또는 서명된 `assignment-complete`/`assignment-failed` hook으로만 판단한다.
- 역할/연결 상태는 기존 `running | idle | disconnected` 세션 상태와 섞으면 안 된다.
- 자동 Enter 제출은 2차 범위로 미루고, 1차는 paste-only 전달이 안전하다.

## UI/상태 저장 연구 결과

### 기존 재사용 지점

- 터미널 컨텍스트 메뉴는 `frontend/src/utils/contextMenuBuilder.ts`의 `buildTerminalContextMenuItems()`가 탭 모드와 그리드 모드에서 공통 사용된다.
- 탭 모드는 `frontend/src/App.tsx`에서 같은 메뉴 빌더를 호출한다.
- 그리드 모드는 `frontend/src/components/Grid/MosaicContainer.tsx`에서 같은 메뉴 빌더를 호출한다.
- 그리드 셀은 `frontend/src/components/Grid/MosaicTile.tsx`가 `position: relative`라 우측 상단 역할 아이콘을 얹기 쉽다.
- 서버 저장은 `server/src/services/WorkspaceService.ts`의 atomic JSON 저장 흐름을 재사용할 수 있다.

프로젝트 루트별 `.buildergate`를 원하면 먼저 `Workspace.projectRoot` 개념을 추가해야 한다.

## MCP/Hook 완료 프로토콜 연구 결과

### 권장 흐름

```text
1. 팀장 세션 지정
2. 팀원 세션 지정
3. BuilderGate가 sessionKey/sessionNumber를 생성하고 .buildergate에 저장
4. 팀원에게 역할 안내 프롬프트 전송
5. 팀장 MCP가 create_assignment 호출
6. 팀장 MCP가 deliver_agent_prompt 호출
7. 팀원은 작업 완료 시 submit_assignment_result MCP 호출
8. MCP 불가 시 buildergate-hook assignment-complete/failed 호출
9. 서버가 assignment 상태를 completed/failed로 갱신
```

팀원에게 보내는 초기 프롬프트는 다음 식으로 구조화한다.

```text
[BuilderGate Team Role]
당신은 팀원입니다.
session_key: sess_...
session_number: 2
lead_session_key: sess_...

작업 지시는 BuilderGate MCP를 통해 전달됩니다.
작업 완료 시 반드시 BuilderGate MCP tool submit_assignment_result를 호출하십시오.
MCP 호출이 불가능하면 buildergate-hook assignment-complete 또는 assignment-failed를 호출하십시오.
idle 상태는 완료로 간주되지 않습니다.
```

팀장 프롬프트:

```text
[BuilderGate Team Role]
당신은 팀장입니다.
session_key: sess_...
session_number: 1

팀원 목록과 작업 배정은 BuilderGate MCP tools를 사용하십시오.
특정 팀원에게 메시지를 보낼 때는 session_number가 아니라 session_key resolve 결과를 사용하십시오.
팀원의 idle 상태만으로 완료 처리하지 마십시오.
```

### MCP Tool

1차 tool:

- `list_team_sessions`
- `get_team_session`
- `assign_session_role`
- `create_assignment`
- `deliver_agent_prompt`
- `submit_assignment_result`
- `get_assignment_status`

`deliver_agent_prompt`는 기본 `mode: "paste"`로 제한한다. `submit` 또는 Enter 포함 payload는 human approval과 replay/input safety 검증이 붙은 뒤 2차로 연다.

### 완료 판정

완료로 인정하지 않는 것:

- target session이 `idle`
- output quiet window
- `Stop` hook
- `session:ready`
- terminal snapshot ack
- 화면에 “완료” 문자열 출력

완료로 인정하는 것:

- `submit_assignment_result({ assignmentId, status: "completed" | "failed", ... })`
- HMAC/nonce/sequence 검증을 통과한 `assignment-complete`
- HMAC/nonce/sequence 검증을 통과한 `assignment-failed`

권장 assignment 상태:

```text
draft -> approved -> delivered -> acknowledged -> running
  -> waiting_input | blocked | completing
  -> completed | failed | cancelled | expired
```

`Stop` hook은 `turn-closed`로만 저장한다. 작업 완료가 아니다.

## 보안 정책

- `.buildergate/team-sessions.json`에는 secret 원문을 저장하지 않는다.
- `sessionKey`는 식별자이고 권한이 아니다.
- write 권한은 별도 capability/HMAC secret으로 분리한다.
- MCP remote 사용 시 Bearer/OAuth token, audience, scope, expiry를 검증한다.
- MCP token은 query string에 넣지 않는다.
- Streamable HTTP MCP는 Origin 검증, localhost bind 또는 HTTPS 인증이 필요하다.
- 모든 프롬프트 전송과 완료 신호는 audit log에 남긴다.
- raw prompt 전문, JWT, hook secret, raw hook payload 전문은 기본 저장하지 않는다.

## 2개 서브에이전트 취합

### UI/상태 저장 담당

판정: 구현 가능. 최소 변경은 `WorkspaceTab`에 역할/번호/연결 상태를 추가하고 기존 PATCH tab 경로와 `WorkspaceService` 저장을 재사용하는 것이다.

핵심 지적:

- `.buildergate` 기준 위치가 모호하다.
- 현재 workspace root 개념이 없어 프로젝트 루트별 저장은 바로 정의하기 어렵다.
- 팀 역할 상태는 `running/idle/disconnected`와 분리해야 한다.
- 팀장 1명 규칙은 서버에서 강제해야 한다.
- 그리드 아이콘은 쉽지만 탭 모드 아이콘은 wrapper 조정이 필요하다.

### MCP/hook 완료판정 담당

판정: `idle` 단독 완료판정은 금지해야 하며, `sessionKey + assignmentId + 명시 완료 신호`가 필요하다.

핵심 지적:

- `sessionId`는 runtime id라 장기 target으로 쓰면 안 된다.
- `sessionNumber`는 UI alias이고 권한으로 쓰면 안 된다.
- MCP `submit_assignment_result`를 1차 완료 경로로 둔다.
- hook `assignment-complete/failed`는 fallback 또는 보강 경로로 둔다.
- `Stop` hook은 turn 종료이지 작업 완료가 아니다.
- 별도 연구 문서: `docs/research/2026-05-19.team-session-mcp-completion-protocol-research.md`

## 공식 문서 대조

- MCP 2025-06-18 transport spec은 stdio와 Streamable HTTP를 표준 transport로 정의하고, JSON-RPC 메시지는 UTF-8이어야 한다.
- Streamable HTTP MCP는 단일 endpoint를 쓰며, Origin 검증, localhost bind, 인증이 필요하다.
- MCP tools는 모델이 자동 호출할 수 있으므로 human-in-the-loop, 입력 검증, access control, rate limit, audit log가 필요하다.
- Codex hooks는 `SessionStart`, `PreToolUse`, `PermissionRequest`, `PostToolUse`, `UserPromptSubmit`, `Stop` 등을 지원한다.
- Codex `Stop` hook은 turn continuation/stop 제어용이며 작업 완료의 안정 신호가 아니다.

- https://modelcontextprotocol.io/specification/2025-06-18/basic/transports
- https://modelcontextprotocol.io/specification/2025-06-18/server/tools
- https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization
- https://developers.openai.com/codex/hooks

## 최종 권고

1차 구현은 `local MCP + .buildergate team registry + 역할 컨텍스트 메뉴 + 역할 아이콘 + paste-only prompt delivery + MCP submit_assignment_result 완료 신호`로 제한하는 것이 맞다.

자동 Enter 제출, remote MCP, 프로젝트 루트별 `.buildergate`, 팀장/팀원 화면 presence 제어는 모두 2차 범위가 안전하다. 특히 완료 판정은 반드시 `idle`에서 분리해야 한다. `idle`은 “현재 터미널이 조용하다”는 표시일 뿐이며, 팀 작업 완료 여부는 `assignmentId`가 포함된 MCP/hook 완료 신호로만 확정해야 한다.
