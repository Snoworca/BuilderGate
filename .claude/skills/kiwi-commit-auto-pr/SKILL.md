> Kiwi MCP rule: normal target-scoped SRS reads, mutations, validation, status/stability updates, acceptance-criteria changes, evidence, trace links, and completed-work logging require working `speckiwi mcp`. CLI is diagnostic/remediation only and is not a normal replacement for MCP mutations.

# kiwi-commit-auto-pr

Git 변경사항을 **사용자 확인 없이** 자동 커밋·push 한 뒤 **Pull Request 까지 자동 생성·갱신**한다. `kiwi-commit-auto-push` 의 차별점:

1. **보호 브랜치 자동 우회** — main/master/develop/release/* 에서 호출 시 issue/REQ 컨텍스트로 feature branch 자동 생성
2. **PR 생성·갱신 양면 처리** — 현재 브랜치에 PR 가 없으면 신규 생성, 있으면 본문 보존 + 변경 요약 코멘트 등록
3. **PR body 품질 게이트** — 신규 PR 생성 시 Haiku 평가자가 본문(요약 + Test plan + Closes/Refs trailer) A+ 까지 평가-개선
4. **speckiwi verification evidence 의 PR variant** — `add_verification_evidence(type:"pr", reference=PR_URL)` + `add_trace_link(type:"PullRequest", reference=PR_URL)` 사용 (commit 측 `type:"Code"` 와 명확히 구분)

`kiwi-commit-auto-push` 의 Step 1~7 (변경사항 수집 → 의도 분석 → issue/REQ 매칭 → 커밋 메시지 작성 → 2개 Haiku A+ 루프 → 자동 스테이징·커밋 → 시그니처 검증) 은 **그대로 계승**. 본 스킬은 push 이후를 PR-aware 흐름으로 확장한다.

### Step 1~7: kiwi-commit-auto-push Step 1~7 동일

**본 스킬에서 재정의 없음.** 부모 스킬의 다음 Step 을 그대로 적용:

- 부모 Step 1 변경사항 수집 (`git status` / `git diff` / `git log` / `git rev-parse --abbrev-ref HEAD`)
- 부모 Step 2 코드 의도 분석 (type / 영향 범위 / WHY)
- 부모 Step 3 Issue 후보 감지 + 매칭 평가 (브랜치명 패턴 → assignee → trailer → 수동 hint → Haiku 매칭). **등급 정의**: Haiku 평가자가 scope_match / intent_match / ac_coverage / no_partial 4축 모두 A+ 일 때 `A+`. 1축 이상 A 이하 + 후보 자체는 유효한 경우 `A 이하`. 후보 자체가 0건이거나 평가자가 매칭 근거 부재로 거부한 경우 `매칭 없음` (부모 §3.3 참조 — 등급 변화 시 부모 SSOT 갱신)
- 부모 Step 4 커밋 메시지 초안 작성 (`Closes #N` / `Refs #N` trailer 결정)
- 부모 Step 5 2개 Haiku 평가자 A+ 루프 (MessageQualityReviewer + SpecComplianceChecker, `MAX_EVAL_ITERATIONS = 10`)
- 부모 Step 6 자동 스테이징 + 커밋 (민감 파일 필터)
- 부모 Step 7 시그니처 검증

부모 스킬 SKILL.md 의 해당 Step 이 SSOT. 부모 Step 이 갱신되면 본 스킬은 별도 변경 없이 자동 계승.

**본 스킬 한정 추가 규칙** (Step 1~7 적용 중 + Step 8 진입 전):
- branch 명 자동 생성 단계에서 사용할 메타데이터(issue 번호, REQ-ID, type, slug)를 §3·§4 결과에서 채집하여 Step 8 입력으로 보관
- **시그니처 검증 4축 확장**: 부모 Step 7 은 commit message 1축만 검증 (부모 Step 내부 수행). 본 스킬은 commit message 1축은 부모 Step 7 결과를 인수하고, branch 명·PR body·PR 코멘트 3축은 본 스킬 후속 Step 에서 추가 검증 — 시점별: (1) commit message [부모 Step 7 결과 인수], (2) branch 명 [§8.3 slug 생성 직후], (3) PR body [§11.2 Haiku 루프 + Step 12 호출 직전], (4) PR 코멘트 [§12.2 FINAL_COMMENT 호출 직전]. 한 축이라도 시그니처 발견 시 해당 축의 산출물 재생성 (Haiku 재호출 또는 slug fallback). 4축 모두 통과해야 후속 Step 진입

### Step 8: 브랜치 결정 + 자동 생성 (★ 신규)

#### 8.1 보호 브랜치 판정

```bash
CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)
DEFAULT_BRANCH=$(gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name' 2>/dev/null || echo "main")
```

보호 브랜치 정규식 (default + 명시적 보호 패턴):

```
^(main|master|develop|release/.*|hotfix/.*)$
```

추가로 `DEFAULT_BRANCH` 값도 보호 대상에 포함 (default branch 가 비표준명일 수도 있으므로).

#### 8.2 브랜치 분기 결정

| 현재 브랜치 | `--allow-direct` | `--branch=<name>` | standalone 동작 | child 모드 동작 |
|---|---|---|---|---|
| 비보호 | — | 없음 | 현재 브랜치 유지 (Step 9 push 직행) | 동일 |
| 비보호 | — | 있음 | `--branch` 값으로 rename (`git branch -m "$OPTION_BRANCH"`) 후 push. **반드시 `NEW_BRANCH="$OPTION_BRANCH"` + `RENAMED=true` 설정** (§9 push 시 `-u` 재설정 트리거) | 동일 |
| 보호 | 없음 | 없음 | **신규 feature branch 자동 생성** (§8.3) | 동일 (KIWI_PM_CONTEXT.branch 없을 때) |
| 보호 | 없음 | 있음 | `--branch` 값으로 분기 (§8.3 자동 명명 생략) | 동일 |
| 보호 | **있음** | — | WARN 출력 후 보호 브랜치 직접 push (위험, 사용자 명시 책임) | **NEEDS_USER bubble-up** (reason: `protected_branch_direct_push_requested`, severity: `business-decision`) — 자동 진행 차단. 호출자(kiwi-pm) 가 `--auto` 모드일 때는 severity `business-decision` 가드레일에 따라 호출자가 강제 중단 (본 스킬은 NEEDS_USER 반환만, 중단 판단은 호출자 책임) |

**보호 브랜치 자동 분기 절차** (이미 부모 Step 6 에서 commit 된 상태가 정상 경로):

```bash
# 0) 사전 확인: origin/$CURRENT_BRANCH 가 존재해야 안전하게 복원 가능
if ! git rev-parse --verify "origin/$CURRENT_BRANCH" >/dev/null 2>&1; then
  # origin tracking 부재 → 위험 (어디로 복원할지 모름). 사용자에게 보고:
  WARN "보호 브랜치 '$CURRENT_BRANCH' 의 origin tracking 없음 — 자동 복원 불가. NEW_BRANCH 분기는 진행하되 보호 브랜치 복원은 skip"
  SKIP_RESTORE=true
fi

# 1) 새 branch 결정 (§8.3)
NEW_BRANCH=$(determine_branch_name)

# 2) 새 branch 로 분기 (현재 HEAD = 새 커밋 포함된 보호 브랜치)
git switch -c "$NEW_BRANCH"

# 3) 보호 브랜치 로컬 포인터를 origin 기준으로 복원 (multi-commit 안전)
#    origin/$CURRENT_BRANCH 사용 이유: 사용자가 push 안 한 다른 로컬 commit 이 있을 때
#    "$NEW_BRANCH"^ 는 직전 1개 commit 만 되돌리므로 잔류 commit 발생 위험
if [ "${SKIP_RESTORE:-false}" = "false" ]; then
  git branch -f "$CURRENT_BRANCH" "origin/$CURRENT_BRANCH"
fi

# 4) 새 branch 가 HEAD, 보호 브랜치 로컬 포인터는 origin 기준으로 정확히 복원됨
#    origin/$CURRENT_BRANCH 는 영향받지 않음 (push 는 §9 에서 NEW_BRANCH 만 대상)
```

`git branch -f` 는 로컬 ref 만 이동한다. origin 보호 브랜치는 push 대상 아님.

**중요**: `origin/$CURRENT_BRANCH` 기준 복원은 사용자가 보호 브랜치에 미push 로컬 commit 이 여러 개 있어도 안전. `$NEW_BRANCH^` (first parent) 방식은 부모 Step 6 직전 1개 commit 만 처리하므로 잔류 commit 발생 — 본 절은 origin tracking 부재 시 SKIP_RESTORE 로 안전망 작동.

#### 8.3 자동 branch 명명 규칙

우선순위 순으로 첫 매칭 적용 (slug 는 영문 소문자 + hyphen 만, 32자 cap):

| 조건 | 명명 패턴 | 예 |
|---|---|---|
| REQ-ID 매칭 (§11.2 결과) | `<type>/<req-id-lower>-<slug>` | `feat/fr-auth-001-token-refresh` |
| Issue 매칭 (§3 결과 A+ / A 이하 모두 사용, 매칭 등급 무관) | `<type>/<issue-num>-<slug>` | `fix/42-login-timeout` |
| 둘 다 없음 또는 부모 Step 3 결과 "매칭 없음" | `<type>/<slug>-<yyyymmdd>` | `chore/readme-update-20260523` |

- `<type>`: 부모 Step 4 에서 결정된 conventional commit type (`feat` / `fix` / `refactor` / `chore` / `docs` / `test` / `perf` / `style`)
- `<slug>`: commit subject 에서 한글·특수문자 제거, 영문 소문자화, 공백 → hyphen, 32자 cap. ASCII 변환 불가 글자만 있을 때는 commit hash 의 short(`git rev-parse --short HEAD`) 사용
- `<yyyymmdd>`: `date -u +%Y%m%d`

**시그니처 차단**: `<slug>` 생성 시 부모 Step 7 시그니처 차단 정책과 동일 규칙 적용 — `claude`, `gpt`, `copilot`, `ai`, `bot` 등 AI 식별 토큰이 slug 에 포함되면 제거.

**충돌 회피 알고리즘** (생성된 branch 명이 이미 origin 또는 로컬에 존재할 때):

```
# 후보 순서: base_name, base_name-2, base_name-3, ..., base_name-9 (총 9개)
# 모두 충돌 시 commit short hash suffix 로 fallback
#
# exists_local(name)  := `git rev-parse --verify "refs/heads/$name"` exit 0
# exists_remote(name) := `git ls-remote --heads origin "$name"` 출력 비어있지 않음
#   ※ `git rev-parse --verify "refs/remotes/origin/$name"` 는 로컬 캐시 기준이라 stale
#     위험 → ls-remote 권장 (네트워크 1회). 두 함수 모두 인자는 짧은 branch 이름만
#     (origin/ prefix 절대 부착 금지 — `git ls-remote --heads origin "origin/foo"` 는
#     항상 빈 출력 = false-negative → 충돌 회피 무력화)
candidate = base_name
if not exists_local(candidate) and not exists_remote(candidate):
    return candidate

for i in 2..9:   # 8회 추가 시도 (-2 ~ -9)
    candidate = base_name + "-" + i
    if not exists_local(candidate) and not exists_remote(candidate):
        return candidate

# base_name + (-2 ~ -9) 9개 후보 모두 충돌 시 hash suffix (무한 루프 방지)
return base_name + "-" + git_short_hash(HEAD)
```

후보 검사 횟수: base_name 1회 + suffix `-2`~`-9` 8회 = **총 9회**. 그래도 충돌 시 commit short hash suffix 로 fallback (예: `feat/42-login-timeout-a1b2c3d`). 동일 prefix branch 가 10개 이상 누적된 경우만 fallback 진입.

(child 모드 정의는 본 SKILL.md 상단 **용어 사전** 및 §14.6 참조)

KIWI_PM_CONTEXT 입력에 `branch` 가 명시되어 있으면 §8.2·§8.3 건너뛰고 명시 branch 채택. 명시된 branch 가 보호 브랜치이면 §8.2 표 마지막 행 적용 — `--allow-direct` 가 KIWI_PM_CONTEXT 에 함께 명시되지 않은 한 NEEDS_USER bubble-up (reason: `protected_branch_direct_push_requested`).

### Step 9: git push

```bash
# Step 8 결과에서 push 대상 branch 결정:
#   비보호 + --branch 미명시: CURRENT_BRANCH 그대로
#   비보호 + --branch 명시   : --branch 값 (이미 git branch -m 으로 rename 완료)
#   보호 + 자동 분기         : NEW_BRANCH (§8.3)
#   보호 + --allow-direct    : CURRENT_BRANCH (위험)
BRANCH=${NEW_BRANCH:-$CURRENT_BRANCH}

# remote 결정 (origin 하드코딩 안 함):
#   1. 현재 branch 의 tracking remote 우선 (git config branch.<BRANCH>.remote)
#   2. 미설정 시 fallback 으로 origin
REMOTE=$(git config --get "branch.$BRANCH.remote" 2>/dev/null || echo "origin")

# upstream 설정 여부 분기:
#   git rev-parse --verify 는 ref 존재 시 exit 0, 미존재 시 exit 128 (>/dev/null 2>&1 으로 suppress)
#   추가: `git branch -m` 으로 rename 된 branch 는 기존 tracking 이 옛 이름을 가리키므로 -u 로 재설정 필요
if git rev-parse --verify "$REMOTE/$BRANCH" >/dev/null 2>&1 && [ "${RENAMED:-false}" = "false" ]; then
  git push "$REMOTE" "$BRANCH"
else
  git push -u "$REMOTE" "$BRANCH"   # 신규 branch 또는 rename 된 branch
fi
```

`RENAMED` 변수는 §8.2 분기표의 "비보호 + `--branch` 있음" 케이스에서 `git branch -m` 실행 시 `true` 로 설정한다 (rename 직후 origin tracking 갱신 필요).

성공 시 Step 10. 실패 시 Step 13. PR base repo 식별은 §10.1 `gh repo view --json isFork` 가 처리 (본 Step 에서 별도 변수 불필요).

**`--force` push 절대 금지** (자동 branch / feature branch / 보호 브랜치 모두 동일).

### Step 11: PR body 초안 작성 + Haiku A+ 루프

**`EXISTING_PR` 존재 + `--update-pr-body` 부재 일 때 본 Step 전체 skip.**

#### 11.1 PR body 초안 형식 (한글 서술형)

```markdown
## Summary

<2~5문장. 변경의 WHY 중심. diff 가 해결하는 문제 + 핵심 접근법>

## Changes

- <핵심 변경 1~5 bullet — 파일/함수 레벨 구체성>

## Test plan

- [ ] <검증 항목 1>
- [ ] <검증 항목 2>
- [ ] (자동 테스트가 있다면) `./gradlew :<module>:test -Pbuild.skipTests=false` PASS

Closes #N
```

**trailer 부착 규칙** (부모 Step 3 결과 그대로 적용, `Closes` / `Refs` 상호배타 — 정확히 하나만 선택):

| 부모 Step 3 결과 | trailer (PR body + commit message 양쪽 동일) |
|---|---|
| A+ 매칭 (4축 모두 A+) | `Closes #N` (GitHub PR merge 시 auto-close) |
| A 이하 매칭 (B/C/D/F 등) | `Refs #N` (참조만, 닫지 않음) |
| 매칭 없음 (후보 0건 또는 평가 거부) | trailer 없음 |
| `--issue=N` 사용자 명시 | `Closes #N` (사용자 신뢰 — A+ 평가 결과 무관) |

**상호배타**: 동일 issue 에 대해 `Closes` 와 `Refs` 둘 다 부착 금지 — 정확히 하나만 (또는 둘 다 없음). 다중 issue 일 때는 issue 별로 독립 결정.

PR body 의 trailer 와 commit message 의 trailer 는 **양쪽 모두 동일 trailer 부착** (PR merge 후 commit 만 남았을 때도 issue link 유지). 중복 부착은 GitHub 가 멱등 처리.

**Squash merge 주의**: GitHub PR 을 squash merge 하면 commit message 가 PR title + body 로 대체된다. 따라서 commit trailer 가 사라지더라도 PR body trailer 는 보존되므로 issue auto-close 는 정상 동작. 단, merge 방식이 "Rebase" 인 경우에만 commit trailer 가 그대로 유지됨. 결론: **PR body trailer 가 SSOT** (Squash/Merge/Rebase 어느 방식이든 GitHub 가 PR body 의 `Closes` 키워드를 인식).

#### 11.2 Haiku A+ 루프

부모 Step 5 와 동일 구조 (동일 `MAX_EVAL_ITERATIONS` 상수 — 부모 갱신 시 자동 계승), 평가자 1개 (PR body 전용):

```
MAX_EVAL_ITERATIONS = 10   # 부모 Step 5 와 동일. 부모 값 변경 시 본 스킬은 자동 계승

best_body = None
best_score = -1   # A+ 받은 축 수가 많을수록 높은 점수 (초기 -1 → 첫 iteration 무조건 갱신)
iterations = 0

WHILE iterations < MAX_EVAL_ITERATIONS:
    body_result = Agent(haiku, PRBodyQualityReviewer)
    score = count_axes_at_or_above_grade(body_result, "A+")  # 0~7

    # >= 사용: 동점 시 더 늦은 iteration 채택 (개선 누적 효과 가정)
    IF score >= best_score:
        best_score = score
        best_body  = body

    IF body_result.all_axes_A+:
        return body   # 정상 종료

    body = IMPROVE(body, body_result.feedback)
    iterations += 1

# 10회 후 A+ 미달 → best-of-N (가장 많은 축에서 A+ 받은 iteration 산출물) + 경고
return best_body
```

"최고 등급 body" 정의: **10회 iteration 중 PRBodyQualityReviewer 7개 평가축 중 A+ 받은 축 수가 가장 많은 산출물**. 동점 시 더 늦은 iteration 우선 (개선 누적 효과 가정).

##### 서브에이전트: PRBodyQualityReviewer

| 기준 | A+ 조건 |
|---|---|
| 의도 반영 | Summary 가 diff 의 WHY 를 정확히 전달, 100% 일치 |
| Changes 구체성 | bullet 이 모호 표현 없고 파일/함수/모듈 레벨 명시 |
| Test plan 검증가능성 | 각 항목이 실행 가능한 명령 또는 명확한 사용자 액션 |
| 시그니처 무결성 | `Co-Authored-By` / `Generated by` 등 0건 |
| 형식 준수 | Markdown 헤더 정확, trailer 가 §11.1 규칙 일치 |
| 누락/과장 없음 | diff 대비 PR body 가 정확 (없는 기능 광고 금지) |
| issue trailer 정확성 | Step 3 결정 (Closes #N / Refs #N / 없음) 과 일치 + 번호 유효 |

판정 근거는 `.kiwi/sessions/pr-{timestamp}/pr-body-review.json` 에 보존.

### Step 12: gh pr create 또는 기존 PR 갱신

#### 12.1 신규 PR 생성

`BASE_BRANCH` / `DRAFT_FLAG` 결정 후 단일 명령:

- `BASE_BRANCH`: `--base=<X>` 명시 → 값 사용 / 미명시 → `gh repo view --json defaultBranchRef --jq '.defaultBranchRef.name'`
- `PR_TITLE`: commit message subject 라인 그대로 (이미 부모 Step 5 에서 A+ 검증됨)
- `PR_BODY`: §11.1·§11.2 산출물

```bash
# USE_DRAFT: 사용자가 --draft 인자 명시했을 때 메인 에이전트가 1 로 설정 (인자 파싱 단계)
USE_DRAFT=${USE_DRAFT:-0}   # 기본 0 (draft 비활성)

# 배열 expansion 으로 빈 플래그 제거 (USE_DRAFT=0 일 때 --draft 추가 안 됨)
PR_FLAGS=(--base "$BASE_BRANCH" --head "$BRANCH" --title "$PR_TITLE" --body "$PR_BODY")
if [ "$USE_DRAFT" = "1" ]; then
  PR_FLAGS+=(--draft)
fi

gh pr create "${PR_FLAGS[@]}"
```

생성 성공 시 PR URL 캡쳐 + 보고.

#### 12.2 기존 PR 코멘트 등록 (body 보존)

Step 10 에서 추출한 `EXISTING_PR_NUMBER` 사용. 본 절 의사코드 흐름:

```
COMMIT_HASH = git rev-parse HEAD
COMMIT_URL  = "{repo_url}/commit/{COMMIT_HASH}"   # gh repo view --json url --jq .url

# (1) Haiku 1회 호출하여 변경 요약 본문 생성
COMMENT_BODY_TEXT = Agent(
  subagent_type="general-purpose",
  model="haiku",
  description="PR 추가 commit 요약 코멘트 작성",
  prompt=PR_COMMENT_PROMPT(pr_title, pr_body, commit_message, diff_summary)
)

# (2) Commit URL trailer 자동 부착 (Haiku 출력에는 포함하지 않음)
FINAL_COMMENT = COMMENT_BODY_TEXT + "\n\nCommit: " + COMMIT_URL

# (3) gh pr comment 호출 — 메인 에이전트가 FINAL_COMMENT 를 --body 인자로 전달
gh pr comment "$EXISTING_PR_NUMBER" --body "$FINAL_COMMENT"
```

> **주**: 위는 의사코드. `{변경 요약 본문}` / `{COMMIT_URL}` 같은 placeholder 는 메인 에이전트가 실제 값으로 치환한 후 `gh pr comment` 를 호출한다. 단순 shell heredoc 으로는 변수 치환과 시그니처 검증을 동시에 보장하기 어렵기 때문에 의사코드 형태로 명시.

**Haiku 출력 형식 규칙** (PR_COMMENT_PROMPT 가 강제):
- 한글 서술형, 2~5줄
- "이 commit 으로 PR 에 무엇이 추가/변경되었는지" 리뷰어 관점 서술
- diff 핵심 변경 1~3건 bullet
- 시그니처 금지 (CLAUDE.md §6 — `Co-Authored-By` / `Generated by` / `🤖` 등 0건)
- `Commit:` trailer 는 메인 에이전트가 자동 부착 (Haiku 출력에 포함 금지)

**시그니처 사후 검증**: `FINAL_COMMENT` 를 `gh pr comment` 에 전달하기 전 메인 에이전트가 정규식으로 시그니처 패턴 0건 확인 (부모 Step 7 와 동일 규칙). 발견 시 Haiku 재호출.

### Step 13: push 실패 / PR 작업 실패 처리

부모 Step 8 (push 충돌 — non-fast-forward / rebase / merge) 그대로 계승. **본 스킬에서 PR 전용 케이스 추가**:

#### 13.1 push 충돌 (부모 Step 10 동일)

- non-fast-forward → AskUserQuestion (rebase / merge / 중단)
- rebase 충돌 → AskUserQuestion (직접수정 / merge전환 / 중단)
- merge 충돌 → AskUserQuestion (직접수정 / 중단)
- 중단 선택 시 로컬 커밋·branch 유지, push·PR 없이 종료

**충돌 발생 시 Step 10~12 미실행** (push 미완료이므로 PR 작업 보류).

**중단 선택 시 상태 보존** (재시도 가능성을 위해):
- 로컬 commit hash 그대로 유지 (`git rev-parse HEAD` 로 후속 호출에서 재추출 가능)
- child 모드 NEEDS_USER 반환 JSON 의 `context.commit_hash` 필드에 commit hash 명시 (§14.6.1 NEEDS_USER 스키마 참조)
- 사용자가 충돌 해결 후 본 스킬 재호출 시 동일 commit 이 재사용됨 (재커밋 안 함)

#### 13.2 보호 브랜치 직접 push 시도 시 origin reject

`--allow-direct` 로 보호 브랜치에 push 했는데 GitHub branch protection rule 이 reject 한 경우:
- AskUserQuestion (자동 feature branch 분기 / 중단)
- 자동 분기 선택 시 §8.3 명명 규칙 적용하여 §8.2 보호 브랜치 분기 절차 수행 후 재시도

#### 13.3 PR 작업 실패는 FAILED 아님

§12.4 의 PR 작업 실패는 commit·push 가 이미 완료된 부분 성공 상태. child 모드 (§14.6) 에서도 `TASK_DONE` + warnings 채움.

**재시도 시 복구 분기** (실패 원인 별):
- **gh CLI 정상이고 `gh pr create / comment / edit` 자체가 일시적 실패** (API timeout, rate limit 등): 동일 branch 에서 본 스킬 재호출 시 §10 의 `EXISTING_PR_JSON` 비어있음 → Step 12.1 신규 PR 생성으로 자연스럽게 복구
- **`gh` CLI 미설치 / 미인증** (§12.4 마지막 2행): 재호출해도 Step 10~12 전체 skip — **환경 복구 선행 필수** (`brew install gh` 또는 `gh auth login`). child 모드에서는 FAILED (`gh_cli_not_installed` / `gh_cli_not_authenticated`) 반환되므로 호출자가 사용자에게 환경 복구 안내
- 사용자 수동 PR 생성도 항상 가능 (push 는 이미 완료된 상태)

## 완료 보고 양식

본 절은 **기본 3종 보고 형식** (신규 PR / 기존 PR 코멘트 / PR 작업 실패) 의 SSOT. speckiwi MCP 연동 시 추가되는 REQ/Task/stability 필드는 **§14.7 보고 양식 확장** 참조 (본 양식의 superset).

신규 PR 생성:
```
커밋: <hash> <message 첫 줄>
branch: <branch> (보호 브랜치 main 에서 자동 분기)
push: origin/<branch> ← 성공
PR: #N <title> (신규 생성)
PR URL: <url>
issue: #M close (PR merge 시 자동, Closes trailer 부착)
```

기존 PR 코멘트:
```
커밋: <hash> <message 첫 줄>
push: origin/<branch> ← 성공
PR: #N <title> (기존, body 보존)
PR URL: <url>
코멘트: 등록 완료
```

PR 작업 실패 (부분 성공):
```
커밋: <hash> <message 첫 줄>
push: origin/<branch> ← 성공
PR: 생성 실패 (gh CLI 인증 오류) ⚠️
수동 작업 필요: gh pr create --base main --head <branch>
```

## 안전 규칙

- **사용자 확인 없이 commit + push + PR 작업** 자율 진행 (본 스킬 핵심)
- **자율 결정 원칙 예외 6종 카테고리** (reason enum 8개 — §14.6.1·§14.6.2 SSOT): (1) frozen REQ 변경 (부모 계승, reason: `stability_frozen`), (2) push 충돌 — 3 sub-reason (부모 계승, reason: `push_conflict_non_fast_forward` / `_rebase` / `_merge`), (3) fork repo 신규 PR (§10.1, reason: `pr_target_ambiguous_fork`), (4) 보호 브랜치 push reject (§13.2, reason: `protected_branch_push_rejected`), (5) `--allow-direct` + 보호 브랜치 명시 (§8.2, reason: `protected_branch_direct_push_requested`), (6) issue 매칭 후보 모호 (부모 계승, reason: `issue_candidate_ambiguous`). 합산: 6종 카테고리 → 8개 reason enum
- `--force` push **절대 금지** (보호 브랜치 / feature branch 동일)
- 민감 파일 자동 제외 (부모 Step 6 계승)
- 시그니처 0건 강제 — **commit message + PR body + PR 코멘트 + branch 명** 4축 모두 (CLAUDE.md §6)
- 기본 동작은 **기존 PR body 보존** — destructive 갱신은 `--update-pr-body` 명시 시에만

## 14. kiwi-* 시리즈 통합 (speckiwi MCP 연동)

본 스킬은 kiwi-commit-auto-push 의 §11 정책을 **그대로 계승하되 PR variant 를 추가**한다. commit ↔ REQ/Task ↔ PR 3축 trace 를 영속화한다.

**부모 §11 ↔ 본 스킬 §14 매핑** (부모 누락 절 명시):

| 부모 §  | 본 스킬 §  | 비고 |
|---|---|---|
| §11.1 입력 컨텍스트 감지 | §14.1 | 그대로 계승 |
| §11.2 REQ/Task 매칭 평가 | §14.2 | 그대로 계승 |
| §11.3 Stability 가드 | §14.3 | 그대로 계승 |
| §11.4 trailer 순서 SSOT | §14.4 | PR body 양쪽 적용 + squash merge 주의 추가 |
| §11.5 speckiwi MCP mutation | §14.5 | PR variant 추가 (`type:"PullRequest"` 신규) |
| §11.6 kiwi-coder/pm 인계 | §14.6 | 4종 NEEDS_USER reason + PR 필드 추가 |
| §11.7 옵션 매트릭스 | (인자 표로 흡수) | 본 스킬 상단 `## 인자` 표 참조. 부모 `--no-comment` 불계승 (PR 코멘트는 `--no-pr-comment` 로 별도 정의) |
| §11.8 보고 양식 확장 | §14.7 | 동일 패턴, PR 필드 추가 |
| §11.9 changelog 없음 선언 | §14.8 | 본 스킬도 동일 선언 (CLAUDE.md §7) |
| 부모 §12 Pipeline event emit | §15 | 그대로 계승. next_hint 동일 (`kiwi-pipeline`). `skill` 만 본 스킬 값으로 |

**핵심 설계 결정** (이후 변경 시 본 문단도 함께 갱신):
- **trailer 키 5종 화이트리스트** 부모와 동일 (`Closes` / `Refs` / `REQ` / `Task` / `STABILITY-OVERRIDE`). 본 스킬은 commit message + PR body 양쪽에 동일 trailer 부착.
- **branch 명 시그니처 차단** — 본 SKILL.md 상단 **시그니처 완전 차단 정책** 절 + §8.3 `<slug>` 생성 규칙에서 AI 식별 토큰 제거. 본 §14 도입부에서 cross-reference 로 재확인.
- **MCP type 컨벤션 PR 추가** — `add_verification_evidence.type="pr"` 신규 도입 (commit variant 와 구분). `add_trace_link.type="PullRequest"` 신규 도입 (`type:"Code"` 는 commit reference 전용으로 부모와 분담 — §14.5.1 참조).
- **자율 결정 원칙 예외 6종 카테고리** (`## 안전 규칙` SSOT 참조 — frozen REQ / push 충돌 / fork repo / 보호 브랜치 push reject / 보호 브랜치 직접 push 요청 / issue 후보 모호). 모두 비가역·고위험.
- **speckiwi 연동 best-effort** — MCP 부재 / mutation 실패 시 commit·push·PR 은 정상 완료.
- **commit 측 add_trace_link 호출 분담** — 본 스킬이 단독 실행될 때만 commit 측 호출. 부모(kiwi-commit-auto-push) 가 chain 호출되는 경우는 부모 책임이고 본 스킬은 PR 측만 호출 (§14.5.1 §분담 규칙).

### 14.1 입력 컨텍스트 자동 감지

부모 §11.1 (Step 3 확장) 그대로. 추가 출처 없음.

### 14.2 REQ-ID / Task-ID 매칭 평가

부모 §11.2 그대로. PR body 작성 (§11.1) 시 REQ/Task trailer 도 commit message trailer 와 동일하게 PR body 마지막 줄에 부착 (순서는 §14.4).

### 14.3 Stability 가드

부모 §11.3 그대로. standalone 모드에서 frozen REQ → AskUserQuestion 3옵션. child 모드 → `NEEDS_USER` bubble-up.

### 14.4 trailer 다중 부착 순서 SSOT

부모 §11.4 와 **동일 순서**, commit message + PR body 양쪽 모두 적용:

```
{type}: {subject}            ← commit message (PR body 에는 없음)

{body / PR Summary 등}

Closes #N              ← Step 3 GitHub issue 결정
Refs #N                ← Step 3 결정 (A 이하)
REQ: FR-XXX-001        ← §14.2 REQ 매칭
Task: T-PHnnn-mm       ← §14.2 task 매칭
STABILITY-OVERRIDE: <reason>  ← §14.3 frozen 변경 시
```

PR body 의 trailer 는 §11.1 형식의 마지막 빈 줄 다음에 배치. PRBodyQualityReviewer 의 형식 준수 평가 축에 본 순서 + 화이트리스트 + 상호배타성 검증 포함.

### 14.5 speckiwi MCP mutation (★ Step 12 확장)

`--no-speckiwi` 명시 시 본 절 skip. **commit push 성공 + PR 작업 완료 후** (Step 12.1 / 12.2 / 12.3 어느 분기든 PR URL 확보된 다음):

#### 14.5.1 `add_trace_link` — commit / PR 호출 분담

**type 컨벤션 분리** (HIGH 결함 해소):
- `type:"Code"` — commit reference (부모 §11.5.1 점유, 본 스킬은 단독 실행 시에만 호출)
- `type:"PullRequest"` — PR reference (본 스킬이 신규 점유)

이로써 trace 그래프에서 `type` 으로 commit-link 와 PR-link 를 명확히 구분 가능.

**호출 분담 규칙** (중복 방지):

| 본 스킬 호출 컨텍스트 | commit 측 호출 (`type:"Code"`) | PR 측 호출 (`type:"PullRequest"`) |
|---|---|---|
| 단독 실행 (사용자 직접) | 본 스킬이 호출 | 본 스킬이 호출 |
| child 모드 + 부모 스킬과 chain (kiwi-pm 등) | 부모 책임 (skip) | 본 스킬이 호출 |

**실제 chain 시나리오 빈도**: 본 스킬은 부모(kiwi-commit-auto-push) 의 commit+push 동작을 **완전 포함**한다 (Step 1~9 는 부모 Step 1~7 + 본 스킬 Step 8~9 로 구성). 따라서 "부모가 commit+push 만 처리하고 본 스킬을 chain 호출"하는 시나리오는 **드문 케이스** — kiwi-pm 이 동일 task 에 부모 + 본 스킬을 모두 spawn 하는 미래 시나리오 (예: Phase 1 commit only → Phase 2 PR open) 를 대비한 예약 메커니즘이다. 현재 표준 실행 경로는 단독 모드.

**KIWI_PM_CONTEXT 협의 메커니즘**:
- 부모(kiwi-commit-auto-push) 가 chain 호출의 선행 단계로 실행되어 commit_trace_link 호출까지 완료한 경우, kiwi-pm 이 본 스킬 spawn 시 `KIWI_PM_CONTEXT.commit_trace_owner: "kiwi-commit-auto-push"` 필드를 명시 주입한다 (kiwi-pm 책임)
- 본 스킬은 KIWI_PM_CONTEXT 입력에서 `commit_trace_owner` 필드 읽고 — 존재 + 값이 부모 스킬명이면 commit 측 호출 skip, 부재 또는 다른 값이면 본 스킬이 commit 측도 호출 (단독 실행 동등 안전 디폴트)
- 필드 미정의 (kiwi-pm 신구 버전 혼재) 시에도 안전 — 중복 호출이 멱등이므로 데이터 무결성 유지 (speckiwi MCP `add_trace_link` 가 동일 id+type+reference 조합에 대해 idempotent 보장)
- **notes 컨벤션 충돌 처리** — chain 시 commit 측 호출은 부모(`kiwi-commit-auto-push`) 가 수행하므로 부모의 notes 값 (`"kiwi-commit-auto-push by run-id {run_id}"`) 이 기록됨. 본 스킬은 PR 측 notes (`"kiwi-commit-auto-pr PR by run-id {run_id}"`) 만 추가. 호출자 (kiwi-pm) 가 mcp_calls 로그 비교 시 notes 의 skill 명 prefix 로 호출자 식별 가능 (commit-측=부모/본 스킬 분담 책임이며, PR-측=항상 본 스킬)

REQ trailer 가 있을 때:

```
# commit 측 (부모 §11.5.1 동일, 분담 규칙에 따라 호출 또는 skip)
mcp__speckiwi__add_trace_link({
  id: "FR-XXX-001",
  type: "Code",
  reference: "{repo}@{commit_hash}:{primary_file_path}",
  relation: "implements",
  notes: "kiwi-commit-auto-pr commit by run-id {run_id}"
})

# PR 측 (★ 신규 — 항상 호출)
mcp__speckiwi__add_trace_link({
  id: "FR-XXX-001",
  type: "PullRequest",
  reference: "{pr_url}",
  relation: "implements",
  notes: "kiwi-commit-auto-pr PR by run-id {run_id}"
})
```

다중 REQ trailer 시 각각 1~2회 호출 (분담 규칙 적용).

#### 14.5.2 `add_verification_evidence` — REQ verification 으로 PR 등록

REQ trailer 가 있을 때:

```
# commit 측 (부모 §11.5.2 동일)
mcp__speckiwi__add_verification_evidence({
  id: "FR-XXX-001",
  type: "commit",
  reference: "{commit_url}",
  notes: "{commit_message_subject}"
})

# PR 측 (★ 신규 type "pr")
mcp__speckiwi__add_verification_evidence({
  id: "FR-XXX-001",
  type: "pr",
  reference: "{pr_url}",
  notes: "{pr_title}"

```

`type="pr"` 는 본 스킬이 점유. 다른 kiwi 스킬이 동일 type 도입 시 본 SKILL.md 만 갱신.

#### 14.5.3 plan task 갱신 (sidecar `mcp_call_log` append)

Task trailer 가 있고 활성 sidecar 가 감지된 경우 부모 §11.5.3 그대로 적용. 추가로 sidecar `tasks[].pr_url` 필드가 정의되어 있으면 PR URL 도 채움 (kiwi-planner sidecar schema 가 해당 필드 지원 시).

#### 14.5.4 MCP 부재 / mutation 실패

부모 §11.5.4 그대로. commit·push·PR 작업 결과는 보존, mutation 만 skip + WARN.

### 14.6 `/kiwi-coder` / `/kiwi-pm` 와의 인계 프로토콜

부모 §11.6 그대로. KIWI_PM_CONTEXT 입력 시 child 모드 전환. 본 스킬 child 모드 차이점:

- AskUserQuestion 비활성 — 4종 예외 모두 `NEEDS_USER` bubble-up
- `branch` 가 KIWI_PM_CONTEXT 에 명시되면 §8 자동 생성 건너뛰고 명시 branch 채택
- `.kiwi/sessions/{run_id}/pr-{task_id}.json` 에 PR 결과 영속화 (commit / push / PR URL / mutation 결과)

#### 14.6.1 child 모드 3상태 반환 JSON SSOT

**TASK_DONE** (정상 완료):

```json
{
  "state": "TASK_DONE",
  "task_id": "T-PH001-02",
  "commit_hash": "abc123def",
  "commit_url": "https://github.com/.../commit/abc123def",
  "push_branch": "feat/fr-auth-001-token-refresh",
  "pr": {
    "number": 101,
    "url": "https://github.com/.../pull/101",
    "action": "created" | "commented" | "body_updated" | "skipped"
  },
  "trailers": { "Closes": ["#42"], "REQ": ["FR-AUTH-001"], "Task": ["T-PH001-02"] },
  "mcp_calls": [
    { "tool": "add_trace_link",           "id": "FR-AUTH-001", "type": "Code",        "reference_kind": "commit", "ok": true },
    { "tool": "add_trace_link",           "id": "FR-AUTH-001", "type": "PullRequest", "reference_kind": "pr",     "ok": true },
    { "tool": "add_verification_evidence","id": "FR-AUTH-001", "type": "commit",      "ok": true },
    { "tool": "add_verification_evidence","id": "FR-AUTH-001", "type": "pr",          "ok": true }
  ],
  "warnings": []
}
```

`mcp_calls[].type` 은 MCP tool 호출 시 사용한 `type` 인자 값 (`"Code"` / `"PullRequest"` / `"commit"` / `"pr"`). `reference_kind` 은 호출자 분석용 보조 라벨.

**NEEDS_USER** (사용자 결정 필요 — `reason` enum 8종, 카테고리는 6종):

```json
{
  "state": "NEEDS_USER",
  "task_id": "T-PH001-02",
```

- `pr_target_ambiguous_fork` `decision_options`: §10.1 4옵션 직렬화 (upstream PR / fork PR / skip / 중단)
- `protected_branch_push_rejected` `decision_options`: §13.2 2옵션 (자동 분기 / 중단)
- `protected_branch_direct_push_requested` `decision_options`: 3옵션 (`--allow-direct` 명시 진행 / 자동 분기로 전환 / 중단)
- `stability_frozen` `decision_options`: §14.3 3옵션 (override / 중단 / feasibility 선행)
- `push_conflict_*` `decision_options`: 부모 §11.6.2 의 rebase/merge/중단 옵션 직렬화. `context.commit_hash` 가 채워져 있으면 재시도 시 본 commit 이 재사용됨

**FAILED** (복구 불가):

```json
{
  "state": "FAILED",
  "task_id": "T-PH001-02",
  "error": "git_push_authentication_failed"
         | "speckiwi_mcp_unavailable_required"
         | "gh_cli_not_installed"
         | "gh_cli_not_authenticated"
         | "gh_cli_not_installed_required"     // **inbound 호환 alias** (deprecated). 본 스킬은 반환 시 항상 분리값(`gh_cli_not_installed` / `gh_cli_not_authenticated`) 만 사용 — 본 alias 는 부모 또는 구버전 호출자가 본 값으로 쿼리/매칭할 때 본 스킬이 동등 처리하기 위한 수신용 enum. §14.6.2 게이트 매핑도 분리값 기준
         | "commit_signature_check_failed_post_amend",
  "details": "한 줄 설명 (원본 stderr 1줄 포함)",
  "partial_state": {
    "commit_hash": "abc123def | null",
    "pushed": false,
    "pr_created": false
  }

```

`error` enum 분리 이유 (HIGH 결함 해소): 호출자가 `gh_cli_not_installed` 면 설치 안내, `gh_cli_not_authenticated` 면 `gh auth login` 안내를 자동 제시 가능.

PR 작업 실패 (§12.4) 는 FAILED 아님 — TASK_DONE + warnings 로 처리 (commit·push 는 완료된 부분 성공).

#### 14.6.2 child 모드 게이트 매핑

| standalone 동작 | child 모드 동작 |
|---|---|
| frozen REQ → AskUserQuestion 3옵션 | NEEDS_USER (reason: `stability_frozen`) |
| push 충돌 (3종) | NEEDS_USER (reason: `push_conflict_*`) |
| fork repo 신규 PR → AskUserQuestion 4옵션 | NEEDS_USER (reason: `pr_target_ambiguous_fork`) |
| 보호 브랜치 push reject → AskUserQuestion 2옵션 | NEEDS_USER (reason: `protected_branch_push_rejected`) |
| **`--allow-direct` + 보호 브랜치 명시** (§8.2 표 마지막 행) | NEEDS_USER (reason: `protected_branch_direct_push_requested`) |
| **issue 후보 모호** (Step 3 매칭 평가 모호 — 부모 §11.6.2 계승) | NEEDS_USER (reason: `issue_candidate_ambiguous`) |
| `gh` CLI 미설치 | FAILED (error: `gh_cli_not_installed`) |
| `gh` CLI 미인증 | FAILED (error: `gh_cli_not_authenticated`) |
| speckiwi MCP 호출 모두 실패 | TASK_DONE + warnings (best-effort) |
| PR 작업만 실패 (gh pr create 등) | TASK_DONE + warnings (commit·push 는 완료) |

reason enum 8종 모두 본 표에서 standalone ↔ child 매핑이 명시됨 (§14.6.1 NEEDS_USER reason enum 과 1:1 대응).

### 14.7 보고 양식 확장

본 절은 본 SKILL.md 상단 `## 완료 보고 양식` (기본 3종) 의 **speckiwi 확장 superset**. 기본 양식의 모든 라인을 포함하고 REQ / Task / stability 라인을 추가한다. `--no-speckiwi` 명시 시 본 양식은 적용 안 함 (기본 3종 양식 그대로).

```
커밋: <hash> <message 첫 줄>
branch: <branch> (보호 브랜치 main 에서 자동 분기)
push: origin/<branch> ← 성공
PR: #N <title> (action: created | commented | body_updated)
PR URL: <url>
issue: #M close (Closes trailer, PR merge 시 자동 close)
REQ: FR-XXX-001 (add_trace_link ×2 [commit+pr] + add_verification_evidence ×2 [commit+pr] 등록)
Task: T-PH001-01 (sidecar mcp_call_log + trace_links 갱신)
stability: stable (정상)
```

stability 가 `frozen` 이고 override 사용 시:
```
stability: frozen ⚠️ override (reason: "hotfix-CVE-2026-xxxx")
```

### 14.8 Changelog 없음 (CLAUDE.md §7 — 부모 §11.9 와 동일)

부모 §11.9 와 동일. 본 SKILL.md 본문에 변경 이력 섹션 (`## 변경 이력` / `## Changelog` / `### v0.x`) 을 작성하지 않는다. 연혁 SSOT 는 `git log ~/.claude/skills/kiwi-commit-auto-pr/SKILL.md`. 부가 정책 — CLAUDE.md §7 은 "헤더의 버전 표기 (예: `# foo v0.1.9`) 유지 가능" 을 허용하지만, 본 스킬은 사용자 명시 정책에 따라 헤더에도 버전명을 포함하지 않는다 (본 SKILL.md 본문 `# kiwi-commit-auto-pr` 한 줄로 종결).

### 14.9 `--auto` 옵션 SSOT

본 스킬의 standalone 모드는 `_shared/kiwi/auto-option.md` v1.0 을 따른다. child 모드(KIWI_PM_CONTEXT 주입) 는 호출자(kiwi-pm 등) 의 `--auto` 가드레일을 따르며 본 스킬은 §14.6 의 NEEDS_USER bubble-up 만 담당 (본 절은 standalone 한정).

**자연어 매핑**: "자동", "묻지 말고", "확인 없이", "auto" → `--auto`. `## 인자` 표의 `--auto` 행에서 활성. **자율 결정 원칙 SSOT 인 `## 안전 규칙` 의 "예외 6종 카테고리 (reason enum 8개)" 와 일대일 매핑** — 본 §14.9 의 critical_gates 는 그 6종 카테고리를 SSOT §5 인터페이스 형식으로 직렬화한 것이다.

본 스킬의 `critical_gates[]` (§14.6.1 NEEDS_USER reason enum 8종 ↔ `## 안전 규칙` 6종 카테고리 ↔ critical_gates 매핑):

| gate_id | reason (8종 enum 매핑) | 안전 규칙 카테고리 | 발생 위치 | critical 사유 |
|---|---|---|---|---|
| `stability-frozen-violation` | `stability_frozen` | (1) frozen REQ 변경 | §14.3 (부모 §11.3 계승) | frozen REQ 본문 변경은 비가역, SSOT §5.1 표준 카탈로그 등재 |
| `push-conflict-non-fast-forward` | `push_conflict_non_fast_forward` | (2) push 충돌 — 3 sub | §13 (부모 Step 8 계승) | rebase/merge 자동 선택은 비가역 (SSOT §5.1 `push-conflict-rebase-merge-choice`) |
| `push-conflict-rebase` | `push_conflict_rebase` | (2) push 충돌 — 3 sub | §13 (부모 Step 8 계승) | rebase 충돌 자동 해결 비가역 |
| `push-conflict-merge` | `push_conflict_merge` | (2) push 충돌 — 3 sub | §13 (부모 Step 8 계승) | merge 충돌 자동 해결 비가역 |
| `fork-repo-pr-create` | `pr_target_ambiguous_fork` | (3) fork repo 신규 PR | §10.1 | 외부 fork repo 영향, SSOT §5.1 표준 카탈로그 등재 |
| `protected-branch-push-rejected` | `protected_branch_push_rejected` | (4) 보호 브랜치 push reject | §13.2 | 보호 브랜치 영향 비가역 |
| `protected-branch-direct-push` | `protected_branch_direct_push_requested` | (5) `--allow-direct` + 보호 브랜치 | §8.2 표 마지막 행 | SSOT §5.1 표준 카탈로그 등재 (`protected-branch-direct-push`) |
| `issue-candidate-ambiguous` | `issue_candidate_ambiguous` | (6) issue 매칭 후보 모호 | 부모 §11.6.2 계승 | clarification 이지만 trailer 부착 결정이 commit 본문 영속화로 이어지므로 critical 격상 (오부착 시 GitHub auto-close 오동작) |

**자율 결정 원칙 보존**: 본 절은 `## 안전 규칙` 의 "예외 6종 카테고리 → reason enum 8개" 매핑을 변경하지 않고, SSOT critical_gates 인터페이스에 직렬화만 한다. 6종 카테고리 → 8개 reason enum 정합은 §14.6.1 §14.6.2 와 일치하며, 본 §14.9 가 신규 매핑을 추가하지 않는다 (1:1 매핑 검증 SSOT).

**critical_gates 외 게이트** (severity `clarification` / `business-decision` 자동 결정 대상): commit 메시지 평가자 A+ 미달 후 강제 진행, branch 명 자동 명명 충돌 회피 hash fallback (§8.3), 기존 PR 본문 보존 vs `--update-pr-body`, `--no-pr-comment` skip 결정 등 — `--auto` 활성 시 §2 서브에이전트 결정 적용.

**호환**:
- `--auto --max` / `--auto --model` 합성은 SSOT §2 / §8 그대로 적용
- `--model` 와 동시 명시 시 본 스킬은 Haiku no-op (기존 `--model` 행 동일)
- `--allow-direct` + `--auto` 동시 명시 시 critical_gates `protected-branch-direct-push` 가 우선 → HALT

## 15. Pipeline event emit (의무)

`~/.claude/skills/_shared/kiwi/pipeline-event.md` 의 §2 schema 와 §5 emit 패턴을 따라 본 스킬 1회 실행 종료 직전 `./kiwi/pipeline.jsonl` 에 정확히 1줄 append. 멱등성: 동일 `run_id` 의 이벤트가 이미 존재하면 skip.

**호출 컨텍스트별 정책**:
- **단독 호출 (사용자 직접)**: 본 스킬이 emit. `next_hint`: 통상 `"kiwi-pipeline"` (다음 plan 또는 종료).
- **kiwi-pm 자식 모드**: 부모(`kiwi-pm`) 의 통합 이벤트에 흡수. 본 스킬 자체 emit 하지 않음.

**필수 필드 매핑** (pipeline-event.md §2.1 — 9종 필수 필드 모두 포함):

| 필드 | 값 |
|---|---|
| `ts` | ISO-8601 UTC 타임스탬프. 예: `"2026-05-25T13:45:12.345Z"` (emit 직전 `date -u +%Y-%m-%dT%H:%M:%S.%3NZ`) |
| `schema_version` | `"1.0.0"` (pipeline-event.md SSOT 버전) |
| `skill` | `"kiwi-commit-auto-pr"` (pipeline-event.md §3 skill enum 에 등재됨) |
| `run_id` | 호출 스킬 run_id (예: `"2026-05-25.skillfactory.add-auth"`). standalone 모드에서는 `"{date}.kiwi-commit-auto-pr.{short_hash}"` 자동 생성 |
| `status` | commit + push + PR 작업 성공 = `"TASK_DONE"`; 사용자 결정 보류 = `"NEEDS_USER"`; push 또는 인증 실패 = `"FAILED"`; dry-run 모드 = `"DRY_RUN"` (pipeline-event.md §5.3 — `dry_run=true` 와 짝). pipeline-event.md §2.1 enum (`TASK_DONE`/`NEEDS_USER`/`FAILED`/`DRY_RUN`/`CORRECTION`) 중 본 스킬은 `CORRECTION` 미사용 |
| `summary` | 1~3문장. 예: 신규 PR `"{branch} → PR #{number} 생성 (Closes #{issue})"`, 기존 PR 코멘트 `"{branch} → PR #{number} 코멘트 등록"`, 충돌 `"push 충돌 — 사용자 결정 대기 ({reason})"` |
| `next_hint` | pipeline-event.md §4 Table T1 따름 — TASK_DONE: `"kiwi-pipeline"`; NEEDS_USER/FAILED: `null`; DRY_RUN: `"kiwi-commit-auto-pr"` (실제 실행 권장). child 모드: emit 안 함 (부모가 통합 이벤트 emit) |
| `artifacts` | 빈 객체 `{"spec_files":[], "plan_file":null, "sidecar_file":null, "analysis_dir":null}` (PR 은 GitHub 가 SSOT, 별도 파일 산출물 없음) |
| `dry_run` | `--dry-run` 또는 `KIWI_DRY_RUN=1` 시 `true`, 아니면 `false`. `dry_run=true` 일 때 `status` 는 `"DRY_RUN"` 권장 (pipeline-event.md §5.3) |

**선택 필드** (pipeline-event.md §2.3): `req_ids` (trailer 또는 commit/PR 본문에 명시된 REQ-ID 배열), `notes` (commit hash + branch + PR number + PR URL + closed issue ids 권장), `target` (speckiwi active target), `duration_sec`.

**next_hint Table T1 등록** — 메타 스킬(`kiwi-pipeline`) 의 결정표 (`pipeline-event.md §4`) 에 `kiwi-commit-auto-pr | TASK_DONE | kiwi-pipeline` 행이 등재됨 (pipeline-event.md §4 갱신 완료, 본 스킬 이벤트는 Table T1 친화적으로 처리됨).

emit 실패는 best-effort.
