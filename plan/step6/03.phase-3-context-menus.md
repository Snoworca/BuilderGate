# Phase 3 Plan

## Objective

Extend the existing `ContextMenu` component to support nested submenus, then implement three new context menus: Pane area menu, Resizer (split boundary) menu, and TabBar menu extension for layout preset and save/load functionality. All menus must integrate with `usePaneManager` actions from Phase 2.

- Phase 2 completed (usePaneManager, SplitPane, PaneResizer, PaneRenderer fully functional)

- `usePaneDB` preset methods operational (savePreset, loadPresets, deletePreset from Phase 1)
- `paneTree.ts` utility functions available (countPanes, getTreeDepth, flattenPaneTree)

## Detailed Guidance

### 3.1: ContextMenu Submenu Extension

**File**: `frontend/src/components/ContextMenu/ContextMenu.tsx`

Extend the existing `ContextMenuItem` interface to support nested submenus. The current interface has `label`, `icon`, `onClick`, `disabled`, `destructive`, and `shortcut` fields. Add the following:

```typescript
export interface ContextMenuItem {
  label: string;
  icon?: string;
  onClick?: () => void;         // optional when children present
  disabled?: boolean;
  destructive?: boolean;
  shortcut?: string;
  children?: ContextMenuItem[];  // NEW: submenu items
  separator?: boolean;           // NEW: render as divider line
}
```

**Submenu rendering requirements:**

1. **Arrow indicator**: Items with `children` display a `▶` character right-aligned within the menu item row. The arrow replaces the shortcut position for submenu items.

2. **Hover open**: When the user hovers over a submenu item, the submenu panel opens after a brief delay (immediate on enter, 300ms delay on leave). Use `onMouseEnter` to open and start a 300ms timer on `onMouseLeave`. If the user re-enters the item or its submenu before the timer fires, cancel the timer.

3. **Screen boundary detection**: Measure the submenu panel dimensions after rendering. If the submenu would overflow the right edge of the viewport (`x + submenuWidth > window.innerWidth`), position it to the left of the parent menu instead. Apply the same logic for bottom overflow.

4. **300ms close delay**: When the pointer leaves both the parent item and the submenu panel, a 300ms timer starts. If the pointer does not re-enter either element within 300ms, the submenu closes. This prevents accidental closure when moving diagonally from the parent item to the submenu.

5. **Shortcut display**: The existing `shortcut` field is already rendered in a right-aligned `<span className="context-menu-shortcut">`. Ensure this renders for leaf items (no children) and is replaced by the `▶` arrow for submenu parent items.

6. **Separator support**: When `separator: true`, render a horizontal divider (`<div className="context-menu-separator" />`) instead of a button. Skip `onClick` handling for separators.

**CSS additions** to `ContextMenu.css`:

- `.context-menu-item.has-children` layout for the arrow indicator
- `.context-menu-submenu` positioning (absolute, left offset equal to parent width)
- `.context-menu-submenu.left` for left-side positioning when overflowing right
- `.context-menu-separator` styling (1px line, margin 4px 0, border-color var(--border-color))

**Implementation approach**: Render submenu items recursively. Each `ContextMenu` item with `children` renders a nested `<div className="context-menu-submenu">` containing the child items. Track the currently open submenu index in local state. Only one submenu can be open at a time within a single menu level.

### 3.2: Pane Area Context Menu (FR-6201)

**Trigger**: Right-click on the terminal area within any Pane (`SplitPane` or `PaneRenderer` wrapping a `TerminalContainer`).

**Integration point**: Add `onContextMenu` handler to the Pane wrapper element in `SplitPane.tsx` or `PaneRenderer.tsx`. The handler calls `e.preventDefault()` and opens the ContextMenu at `(e.clientX, e.clientY)`.

**Menu items** (in order):

| Label | Shortcut | Action | Disabled When |
| --- | --- | --- | --- |
| 수평 분할 (위/아래) | `Ctrl+B, "` | `splitPane(paneId, 'horizontal')` | `!canSplit` (paneCount >= 8 or treeDepth >= 4) |
| 수직 분할 (좌/우) | `Ctrl+B, %` | `splitPane(paneId, 'vertical')` | `!canSplit` |
| --- separator --- | | | |
| 줌 토글 / 줌 해제 | `Ctrl+B, z` | `toggleZoom(paneId)` | paneCount === 1 |
| Pane 교환 | | `startSwap(paneId)` | paneCount === 1 |
| --- separator --- | | | |
| 출력 복사 | | Copy terminal selection or full screen text to clipboard | never |
| --- separator --- | | | |
| Pane 닫기 | `Ctrl+B, x` | `closePane(paneId)` | paneCount === 1 (last pane) |
| 다른 Pane 모두 닫기 | | `closeOtherPanes(paneId)` | paneCount === 1 |

**Zoom state behavior**: When `layout.zoomedPaneId` is set:
- Change "줌 토글" label to "줌 해제"
- Disable "수평 분할", "수직 분할", "Pane 닫기", "다른 Pane 모두 닫기"
- "Pane 교환" remains disabled (cannot swap while zoomed)

**Output copy**: Use the xterm.js selection API (`terminal.getSelection()`) to get selected text. If no selection exists, use `terminal.buffer.active` to read all visible lines. Write to clipboard via `navigator.clipboard.writeText()`.

**Pane ID passing**: The context menu must know which Pane was right-clicked. Pass `paneId` through the `onContextMenu` callback from `SplitPane.tsx`. Store the target pane ID alongside the menu position in state.

### 3.3: Resizer Context Menu (FR-6202)

**Trigger**: Right-click on a `PaneResizer` element (the split boundary drag handle).

**Integration point**: Add `onContextMenu` handler to `PaneResizer.tsx`. The handler receives the `splitId` of the parent `PaneSplit` node.

**Menu items:**

| Label | Action |
| --- | --- |
| 균등 분할 | `equalizePanes(splitId)` |
| 방향 전환 (horizontal ↔ vertical) | `toggleDirection(splitId)` |
| --- separator --- | |
| {첫 번째 Pane 방향 label} 닫기 | `closePane(firstChildPaneId)` |
| {두 번째 Pane 방향 label} 닫기 | `closePane(secondChildPaneId)` |

**Direction-aware labels**: The close item labels depend on the current split direction:
- Vertical split (좌/우): "왼쪽 Pane 닫기", "오른쪽 Pane 닫기"
- Horizontal split (상/하): "위쪽 Pane 닫기", "아래쪽 Pane 닫기"

**Child identification**: Use `findSplit(root, splitId)` to get the `PaneSplit` node, then access `children[0]` and `children[1]`. If a child is a `PaneSplit` (not a leaf), closing it requires recursively closing all leaves within that subtree. Delegate to `closePane()` for leaf children, or show "서브트리 닫기" label for split children with a note "(N개 Pane)" indicating the count.

**Disabled states**: If either child is the last remaining Pane in the entire tree (paneCount === 1), disable its close item. This should not normally occur for a Resizer menu since a Resizer implies at least 2 panes.

### 3.4: TabBar Menu Extension (FR-6203)

**File**: `frontend/src/components/TabBar/TabBar.tsx`

Extend the existing tab right-click context menu with three new items appended after the existing items. Add a separator before the new section.

**New menu items:**

| Label | Type | Action |
| --- | --- | --- |
| --- separator --- | separator | |
| 레이아웃 저장 | leaf | Open layout name input modal (Task 3.5) |
| 레이아웃 불러오기 ▶ | submenu | Show saved layouts submenu |
| 프리셋 레이아웃 ▶ | submenu | Show preset layouts submenu |

**"레이아웃 불러오기" submenu items** (dynamic, loaded from `usePaneDB.loadPresets()`):

For each saved layout (`isBuiltIn === false`):
```
{name} ({paneCount}개)    [삭제]
```
- Clicking the item: calls `loadLayout(layoutId)` which terminates current sessions and applies the saved layout
- The `[삭제]` affordance (a small icon or text button on the right): opens delete confirmation modal (Task 3.6)
- If no custom layouts exist, show a disabled item: "저장된 레이아웃 없음"

**"프리셋 레이아웃" submenu items** (static, 6 presets):

| Label | Preset ID |
| --- | --- |
| 단일 | `preset-single` |
| 좌우 분할 | `preset-vertical-2` |
| 상하 분할 | `preset-horizontal-2` |
| 4분할 | `preset-quad` |
| 1+2 (메인+보조) | `preset-main-side` |
| 에이전트 모니터 | `preset-agent-monitor` |

Clicking a preset item calls `applyPreset(presetType)` which:
1. Terminates all current Pane PTY sessions
2. Creates the required number of new PTY sessions
3. Builds the preset layout with new session IDs via `buildPresetLayout()`
4. Sets the new layout

**Preset loading**: Load the preset list once when the submenu is opened (or cache from the initial `initBuiltInPresets()` call). Built-in presets are always present. Custom layouts are fetched from IndexedDB.

**Layout application confirmation**: Before applying a preset or loading a saved layout, if the current tab has more than 1 Pane, show a confirmation dialog: "현재 레이아웃이 초기화됩니다. 계속하시겠습니까?" with "확인" and "취소" buttons. Use the existing `ConfirmModal` pattern.

### 3.5: Layout Name Input Modal

**New component**: Create within the existing modal pattern (reference `ConfirmModal` in `frontend/src/components/Modal/`).

**Behavior:**
- Triggered from "레이아웃 저장" menu item
- Displays a modal overlay with:
  - Title: "레이아웃 저장"
  - Text input field with placeholder "레이아웃 이름 입력"
  - Validation: 1~30 characters, allows Korean, English, numbers, spaces, hyphens, underscores
  - "저장" button (disabled when input is empty or invalid)
  - "취소" button
- On submit:
  - Check if a custom layout with the same name already exists (via `loadPresets()` filtering)
  - If duplicate name exists, show overwrite confirmation: "'{name}' 레이아웃이 이미 존재합니다. 덮어쓰시겠습니까?"
  - On confirm, call `usePaneDB.savePreset(name, layout)` which saves the current PaneLayout with sessionIds replaced by placeholders
- Auto-focus the input field on open
- Enter key submits, Escape key cancels

**Validation regex**: `/^[\w가-힣\s\-]{1,30}$/u` (word characters, Korean syllables, spaces, hyphens, 1-30 chars).

### 3.6: Layout Delete Confirmation

**Behavior:**
- Triggered from the delete affordance in the "레이아웃 불러오기" submenu
- Uses the existing `ConfirmModal` pattern
- Message: "'{name}' 레이아웃을 삭제하시겠습니까?"
- "삭제" button (destructive style) and "취소" button
- On confirm: calls `usePaneDB.deletePreset(id)` then refreshes the submenu list
- Built-in presets (`isBuiltIn === true`) never show the delete affordance, so this modal is never triggered for them

## Regression Checklist

- Existing TabBar tab right-click menu items (rename, close, close others) still function correctly.
- Existing ContextMenu usage elsewhere in the app (if any) is not broken by the interface extension (backward-compatible since `children` and `separator` are optional).
- Menu items without `children` still render and behave identically to before the extension.
- Context menu positioning logic (viewport boundary clamping) still works for root-level menus.
- `ConfirmModal` component behavior is unchanged for other uses.
- Mobile layout is not affected (context menus are desktop-only; mobile uses long-press bottom sheet in Phase 4).
- Keyboard shortcut display strings in menu items match the actual bindings defined in Phase 5.
- Pane focus state is preserved after menu interaction (opening and closing a menu does not change the focused Pane unless an action explicitly changes it).
- Tab switching and session creation/deletion still work after TabBar menu extension.
- usePaneManager actions (split, close, zoom, swap, resize, equalize, toggleDirection) trigger correct state updates and IndexedDB persistence.

## Definition Of Done

- ContextMenu supports nested submenus with hover open, 300ms close delay, screen boundary detection, and arrow indicators.
- Right-clicking a Pane terminal area shows the full Pane context menu with correct disabled states.
- Right-clicking a split boundary shows the Resizer context menu with direction-aware labels.
- TabBar right-click menu includes layout save, load submenu, and preset submenu with all 6 presets.
- Layout name input modal validates input and handles duplicate names.
- Layout delete confirmation works for custom layouts and is hidden for built-in presets.
